01
Vision & Thesis

One platform. One truth.
One north star.

Every tool you're replacing was built to optimize its own metric. ShopCX is built to optimize yours: subscriber retention.

The $3,000–5,000/month you're spending across Gorgias, Siena AI, Appstle, and Klaviyo isn't just expensive — it's fragmented. Each tool has its own customer record, its own definition of success, and no way to see the whole picture. A customer who opened a support ticket, ignored two emails, had a failed payment, and then cancelled is just a churn event in your analytics. It shouldn't be.

ShopCX unifies every signal — support history, subscription health, purchase behavior, social engagement, email activity — into a single customer record. The result is a Retention Score: a living number that tells you exactly how at-risk each subscriber is, and why.

The Retention Score — Core Concept

Every customer in your Shopify store has a Retention Score from 0–100, synthesized in real time from: open support tickets and resolution history, subscription payment health and tenure, email/SMS open and click engagement, behavioral signals from the storefront and subscription portal, and social interactions with your brand. This score drives prioritization across every surface of the app — from ticket routing to marketing segmentation.

What ShopCX replaces — and why you can build it

Tool Monthly Cost What it does Why we can build it
Gorgias $300–$1,500 Helpdesk & ticketing — email/chat/social unified inbox Well-understood domain. Core value is integration breadth, which we control directly.
Siena AI $2,000+ AI customer service — autonomous ticket resolution General-purpose AI. Ours will be purpose-trained on our specific products, orders, and subscription data — strictly better.
Appstle $30–$100 Subscription management — pause/skip/swap/cancel portal Essentially a Stripe Billing wrapper. Complex to get right but not complex to understand. We migrate incrementally.
Klaviyo $400–$2,000 Email & SMS marketing — flows, campaigns, segmentation Hardest to replace (deliverability). Build last, using Resend + Twilio. Our segmentation will be richer because we own all behavioral data.
Build Strategy

Internal first, SaaS later. Phase 1 is building for Superfoods Company only. The multi-tenant workspace architecture ships from day one, but we don't open it up. Once the product is proven, the path to SaaS is adding billing, onboarding flows, and a Shopify App Store listing. The real moat is the product intelligence layer — no competitor can replicate product-specific AI training without the underlying product data.

Founding Principle
"Every feature in ShopCX exists to answer one question: does this help us keep the customer subscribed? If it doesn't, we don't build it."
02
System Architecture

How the pieces
fit together

A Next.js monorepo on Vercel, backed by Supabase Postgres, with a clear separation between the agent layer and the data layer.

Frontend

Next.js 14+ App Router · Vercel

The main dashboard, all feature UIs, and the embeddable chat widget. App Router gives us Server Components for fast data fetching, and Vercel Edge for low-latency widget delivery. The chat widget ships as a standalone compiled bundle hosted on a CDN — installed on Shopify via a script tag.

Backend / Database

Supabase · Postgres · pgvector

Supabase gives us Postgres (structured data), Row Level Security for multi-tenant isolation, Realtime subscriptions for live ticket updates and chat, Storage for attachments, and Edge Functions for webhook processing. pgvector stores AI embeddings for the product knowledge base and semantic search.

AI Layer

Anthropic Claude · RAG · Embeddings

Claude (via the Anthropic API) powers ticket deflection, draft generation, confidence scoring, and nightly knowledge gap analysis. Product intelligence is stored as embeddings in pgvector and retrieved via semantic similarity search before each AI call — giving the agent accurate, up-to-date product context without fine-tuning.

Background Jobs

Inngest · Vercel Cron

Inngest handles durable background jobs with retry logic: nightly AI analysis, subscription dunning sequences, email/SMS send queues, and event-driven automation flows. Vercel Cron triggers scheduled jobs. This separates long-running work from request/response cycles cleanly.

Integrations

Shopify · Meta · Stripe · Twilio · Resend

Shopify OAuth for customer data and orders. Meta Graph API for DMs and page comments. Stripe Billing for subscription management (replacing Appstle). Twilio for SMS. Resend for transactional and marketing email with excellent deliverability defaults and React Email templates.

Multi-Tenancy

Workspace Model · RLS · Roles

Every resource belongs to a workspace. Supabase Row Level Security enforces isolation at the database level — not the application level. Each workspace has an owner, and can invite members with roles: Admin, Agent, Social Manager, Marketing, Read-Only. Built for Superfoods Co. first; the architecture supports adding new workspaces from day one.

Key Architectural Decision — Data Isolation

Multi-tenancy via Supabase RLS means workspace isolation is enforced by the database, not by application code. Every query automatically filters to the current workspace via a JWT claim. This is not optional — it must be architected from migration #1. Retrofitting RLS onto an existing schema is extremely painful. The rule: every table that holds workspace-specific data has a workspace_id column with an RLS policy that checks auth.jwt() ->> 'workspace_id'.

03
Customer Identity

One customer.
Every signal.

Identity resolution is the most important architectural decision in the platform. Get this wrong and every other feature is weaker.

When a ticket arrives — whether via email, chat widget, Meta DM, or SMS — ShopCX resolves it to a single customers record. That record aggregates the customer's full history across every surface. The goal is that when a support agent opens a ticket, they see everything: recent orders, subscription status, behavioral events, past tickets, email engagement, and the Retention Score.

Primary Key

Email Address

The canonical identifier. Every system — Shopify, Stripe, Twilio, Meta (when available) — resolves to email. When a new contact arrives, we check email first. If matched, we link. If not, we create a new customer record.

Shopify Link

shopify_customer_id

Matched at ticket creation via Shopify Customer API lookup by email. Gives us access to full order history, tags, metafields, and any active Appstle subscription data during the migration window.

Session Link

Storefront Session Token

The chat widget captures the Shopify storefront session token when a logged-in customer opens chat. This enables real-time order lookup mid-conversation without the customer having to identify themselves.

Retention Score — Calculation

The Retention Score (0–100) is recalculated on each significant event and stored on the customers table. Component weights (tunable): Subscription health — tenure, payment history, skip/pause frequency (30%). Support history — open tickets, unresolved issues, sentiment of past interactions (25%). Purchase behavior — recency, frequency, order value trend (20%). Email/SMS engagement — recent open/click rate, unsubscribe signals (15%). Behavioral signals — login frequency to portal, page views, add-to-cart events (10%). Score below 40 = at-risk, flagged for proactive outreach.

04
Feature — AI Support & Ticketing

An AI agent that
actually knows your products.

Replaces Gorgias + Siena AI. Ticket ingestion via email, live chat widget, and Meta DMs. AI attempts deflection before routing to humans.

Ticket Ingestion Channels

Email

Inbound Email Parsing

Use Resend Inbound or a dedicated inbox via SendGrid Inbound Parse. Incoming emails to support@yourdomain.com POST to a webhook that creates a ticket. Thread matching by In-Reply-To header groups replies to the same ticket. Attachments stored in Supabase Storage.

Live Chat

Embeddable Chat Widget

A small JS bundle (shopcx-widget.js) installed via Shopify theme script tag. Opens a chat window that connects to ShopCX via WebSocket (Supabase Realtime). Captures storefront session token automatically for identity resolution. Not crazy to build — this is a weekend-scale feature.

Meta DMs

Instagram & Facebook DMs

Via the Meta Messenger API (Pages scope). Incoming DMs create tickets automatically. Outgoing replies send back through the Graph API. Requires pages_messaging permission — include in Meta app review submission.

AI Agent — Confidence Tiers

90%+
Auto-resolve

Agent responds autonomously

AI sends the response directly to the customer, marks ticket as pending customer reply. If no reply in 48hrs, auto-closes. CSAT survey sent on close.

60–89%
Human review

AI drafts, agent approves

AI generates a draft response and surfaces relevant knowledge base articles. Ticket assigned to available agent with "Review & Send" action. Agent can edit before sending.

<60%
Human only

Routes to agent queue

AI surfaces context only — customer history, order details, subscription status, related past tickets. Agent works the ticket. AI learns from the resolution for future tickets.

AI Knowledge Base — RAG Architecture

Product intelligence is stored as document chunks with embeddings in Supabase pgvector. Before each AI call, we perform a semantic similarity search against the customer's message to retrieve the top 5–10 most relevant chunks. These are injected into the system prompt alongside the customer's full profile (orders, subscription status, past tickets). The AI then generates a response grounded in real product knowledge — not hallucinated. The knowledge base is managed via a simple admin UI where workspace members can add, edit, and delete product documents, FAQs, and policies.

Nightly Knowledge Gap Analysis

An Inngest cron job runs nightly at 2am. It sends the day's manually-resolved and escalated tickets to Claude in batch with the prompt: "Identify which of these tickets could have been automatically resolved with additional product or policy knowledge. Cluster them by topic. For each cluster, suggest the knowledge base entry that would enable future auto-resolution." Output is a structured Gaps Report surfaced in the dashboard each morning. Workspace admins review and one-click add suggested knowledge base entries.

05
Feature — Social Command Center

Ads vs. organic.
Context that matters.

Meta comments and DMs, split by placement. Reply, delete, and block — all from one queue. Ad comments show the destination URL so you know exactly what product is being discussed.

Organic Posts

Page Comments — Standard

Pull from the Pages API (GET /{page-id}/feed). Comments show post content, timestamp, user, and existing replies. Actions: Reply (posts via Graph API), Hide, Delete, Block user. Sentiment analysis run on each comment to flag potentially negative or harassing content for priority review.

Paid Posts

Ad Comments — With Context

Pull from the Ads API (GET /{ad-id}/comments). Each comment is enriched with: ad name, ad set name, campaign name, destination URL, and creative preview. This tells your social manager exactly which product a comment refers to without opening Ads Manager. Critical for high-volume ad campaigns.

Meta API — Permissions Required

Budget 2–4 weeks for Meta's app review process. Required permissions: pages_read_engagement (read comments), pages_manage_engagement (reply/delete), pages_messaging (DMs), ads_read (ad comment context), instagram_basic + instagram_manage_comments (Instagram). Submit for review early — this is the longest external dependency in the build.

Moderation Rules Engine

Workspace admins define moderation rules that run automatically on every incoming comment. Rules consist of a trigger (keyword match, sentiment score threshold, or AI classification) and an action (auto-hide, auto-delete, flag for review, or block user). Examples: Keyword rule — any comment containing a list of slurs → auto-delete + block. Sentiment rule — AI sentiment score below 0.2 → flag for human review. Competitor rule — mentions of competitor brands → flag for response. Rules are ordered and the first matching rule wins.

06
Feature — Subscription Manager

Replace Appstle.
Own the subscription layer.

Built on Stripe Billing. Pause, skip, swap, cancel — plus a rules engine for automated responses to subscription events.

Migration Strategy — Appstle → ShopCX

Do not migrate all at once. Run both systems in parallel for 60 days. New subscribers go onto ShopCX subscriptions. Existing Appstle subscribers migrate in cohorts — start with the lowest-risk segment (most recent subscribers). Appstle stores subscription data in Shopify metafields; export these and import into ShopCX before switching each cohort. Zero-downtime cutover requires Stripe to have the payment method on file before the Appstle subscription is cancelled.

Customer Portal Actions

Self-Service

What customers can do

Pause subscription (1–12 weeks), skip next order, swap product variant, change frequency, update shipping address, update payment method, cancel subscription (with save offer). All actions log a behavioral event to the customer timeline.

Retention Saves

Cancel flow with save offers

When a customer initiates cancel, present a targeted save offer based on their cancellation reason and Retention Score: payment issue → offer pause, too much product → offer frequency change or skip, too expensive → offer discount. Save offer acceptance/decline logged and used to refine future offers.

Subscription Webhook Events & Automated Actions

EventTriggerAutomated Actions
payment_failed Stripe invoice.payment_failed 1. Create support ticket (type: billing, auto-tagged). 2. Send dunning email #1 with payment update link. 3. Reduce Retention Score. 4. Queue dunning email #2 at +3 days if still unpaid.
payment_recovered Stripe invoice.paid (after failure) Close any open billing tickets. Send recovery confirmation. Restore Retention Score adjustment.
subscription_cancelled Stripe customer.subscription.deleted Tag customer as churned. Trigger win-back flow (email at +1d, +7d, +30d). Suppress from active subscriber segments.
subscription_paused Custom (pause action in portal) Log event. Suppress marketing sends during pause. Queue re-engagement email 1 week before pause end.
subscription_created Stripe customer.subscription.created Create customer record if new. Trigger onboarding flow. Set initial Retention Score (60 — new customer baseline).
order_shipped Shopify fulfillment webhook Send shipping confirmation with tracking. Log to customer timeline. Increment Retention Score slightly.
Rules Engine

Beyond the default event-to-action mappings above, workspace admins can build custom rules in a no-code interface: When [event] AND [conditions] THEN [actions]. Conditions can reference any customer attribute (Retention Score, subscription tenure, total orders, last CSAT score). Actions include: create ticket, send email, send SMS, add tag, assign to agent, trigger flow. This is ShopCX's automation backbone — the equivalent of Klaviyo flows but with subscription and support data as first-class triggers.

Feature — Behavioral Tracking

Know everything
before they write in.

A lightweight event tracker on the Shopify storefront and subscription portal feeds the customer timeline in real time.

When a support ticket arrives, the agent should never have to ask "what's going on with this customer?" ShopCX surfaces the customer's last 30 days of activity automatically: which products they viewed, what they added to cart, how often they log into the subscription portal, whether they've tried to cancel before.

This is implemented via a lightweight JavaScript SDK (shopcx-track.js) installed alongside the chat widget on Shopify. Events POST to a Supabase Edge Function that appends to the customer_events table and updates the Retention Score.

08
Feature — Email & SMS Marketing (Phase 9)

Marketing that knows
everything Klaviyo doesn't.

Built last. Powered by Resend + Twilio. Segmentation backed by full behavioral, subscription, and support data.

Build This Last

Email/SMS marketing is the most complex feature to operate reliably. Deliverability is a craft — domain reputation, DKIM/SPF/DMARC, bounce handling, unsubscribe compliance, and IP warming all require careful attention. Build this after the retention engine is proven. Use Resend for email (excellent developer experience, great deliverability defaults, React Email for templates) and Twilio for SMS. Do not attempt to build a sending infrastructure from scratch — use these services as the transport layer and own the product layer above them.

Segmentation Advantage

Richer than Klaviyo

Because ShopCX owns behavioral, subscription, support, and social data, segments can reference any combination: "Active subscribers with Retention Score below 50 who haven't opened an email in 30 days and have viewed the product page more than twice." No integration required — it's all in the same database.

Flows & Campaigns

Event-driven automations

Flows are triggered by any event in the Rules Engine: subscription events, ticket events, behavioral events, or scheduled time-based triggers. Campaigns are one-time sends to a segment. Both use React Email templates rendered server-side and sent via Resend. SMS flows via Twilio Messaging Service with opt-in/opt-out compliance built in.

Risks & Mitigations

What can go wrong
and how to handle it.

Known risks across technical, operational, and external dependency categories.

Risk Likelihood Impact Mitigation
RLS misconfiguration
Tenant data leaking across workspaces
Med High
Write RLS policy tests as part of CI/CD. Use Supabase's built-in RLS testing tools. Audit all new tables before merging. Never trust application-level filtering alone.
Meta App Review delay
Social feature blocked for weeks
High Med
Submit the Meta app for review at the START of Phase 5, not when Phase 6 begins. The review can take 2–4 weeks. Social feature is Phase 6 — 5 phases of runway if submitted early.
Appstle subscription migration
Customer double-billed or subscription lost
Med High
Never cancel Appstle subscriptions before Stripe has the payment method. Migrate in small cohorts (50 customers). Run parallel for 60 days. Build a rollback path before migrating anyone.
AI hallucination in customer responses
Incorrect info sent to customers at 90%+ confidence
Med High
Ground all AI responses in retrieved knowledge base chunks — never answer from model memory alone. Start confidence threshold at 95%, not 90%. Log every AI-sent response. Review weekly in early operation. Lower threshold gradually as accuracy is validated.
Email deliverability degradation
Marketing emails landing in spam post-Klaviyo
Med High
This is why Phase 9 is last. Warm the sending domain gradually. Use Resend's dedicated IP if volume warrants. Maintain proper DKIM, SPF, DMARC. Monitor bounce rates daily in the first 30 days of migration. Keep Klaviyo running in parallel for the first 30 days.
AI token cost overrun
Anthropic API costs exceed savings
Low Med
Track token usage per ticket from day one. Set workspace-level monthly limits. Use Claude Haiku for classification/triage, Claude Sonnet for draft generation. Cache RAG results for identical queries within a time window.
Shopify API rate limits
Bulk operations throttled during initial sync
Low Low
Use Shopify's bulk operations GraphQL API for initial customer sync. Implement exponential backoff on all Shopify API calls. The initial sync is a one-time event — throttling is acceptable if it takes a few hours.
