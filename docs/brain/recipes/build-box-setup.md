# Recipe: the self-hosted build box

The box that runs autonomous spec builds on the **Max subscription** for the [[../specs/roadmap-build-console]] (the "do it" button). Reachable only over Tailscale. This page is the reproducible runbook + the day-to-day ops.

## What it is

- **Hetzner CCX33** ‚Äî Ubuntu 26.04, 8 vCPU, 30 GiB RAM, name `claude-server`. Public IPv4 `178.156.246.235` (firewalled), tailnet IP `100.75.99.7`.
- Runs `systemd` service **`shopcx-builder`** = `scripts/builder-worker.ts`, which polls [[../tables/agent_jobs]], claims a job via `claim_agent_job()`, runs `claude -p` (Max) ‚Äî or `codex exec` (ChatGPT plan) for the Codex-routed kinds, see [¬ß Second runtime: Codex](#second-runtime-codex-on-a-chatgpt-plan-box-codex-runner) ‚Äî tsc-gates, and opens a `claude/*` PR. See [[../dashboard/branches]] to merge.

## Access (SSH) ‚Äî how to get on the box

**You always forget this ‚Äî it lives here now.** The box is Tailscale-only. SSH lands as **root**, but the worker + every build run as the **`builder`** user, so drop into that user to inspect anything:

```bash
ssh root@claude-server     # host alias in ~/.ssh/config ¬∑ tailnet claude-server.tail298416.ts.net / 100.75.99.7
sudo -iu builder           # become the worker user ‚Äî EVERYTHING lives under /home/builder
```

As `builder`:
- **Repo (the worker's OWN checkout):** `~/shopcx` (`/home/builder/shopcx`) ‚Äî the SHA the dashboard/heartbeat reports (`git -C ~/shopcx rev-parse --short HEAD`).
- **Build worktrees:** `~/builds/build-<slug>` ‚Äî named for the **spec slug**, stable across all of a spec's phases (see the `--resume` invariant below). plan/pr-resolve/fold lanes still use `~/builds/<job-id>` (dir name = `agent_jobs.id`); spec-chat/dev-ask/director-* use stable per-session names. A running `claude` whose `cwd` is a `~/builds/*` with **no matching active `agent_jobs` row** (no active build for that slug, or a deleted job) is a ZOMBIE ‚Äî safe to `kill <pid>` that PID *only*.
- **Max creds:** `~/.claude` (the round-robin login config dirs).

Service ops (as **root**): `systemctl {status,restart,stop} shopcx-builder` ¬∑ live logs `journalctl -u shopcx-builder -f`. List build sessions + their worktrees: `for p in $(pgrep -f claude); do echo "$p $(readlink /proc/$p/cwd)"; done`. (Service `User=builder`; a hard `systemctl restart` orphans in-flight builds ‚Äî prefer the dashboard "Queue restart" which drains first.)

## Core invariants

- **Max billing.** Builds run `env -u ANTHROPIC_API_KEY claude -p ‚Ä¶`. The box must have **no `ANTHROPIC_API_KEY`** in the environment ‚Äî `/home/builder/shopcx/.env.local` keeps it commented out. Claude is logged in as the **Max** account (`dylan@superfoodscompany.com`); creds live in `/home/builder/.claude`. Verify: claude.ai/settings/usage moves, the API console stays flat.
- **No inbound.** The box only reaches *out* (to Supabase + GitHub). Public SSH (:22) is firewalled; the trigger is a DB row the worker polls, never an HTTP call in.
- **One worker.** Only the `shopcx-builder` service runs the worker (don't also start a `nohup` copy).
- **`claude --resume` is DOUBLY scoped ‚Äî pin BOTH the account AND the cwd.** A resumed session resolves only when run under (1) the **same Max account** that created it (`CLAUDE_CONFIG_DIR` ‚Äî a cross-account resume is a guaranteed "No conversation found"; every resume pins to its session's owning account, see the round-robin/failover section below) **AND** (2) the **same working directory** ‚Äî claude keys its transcript store by cwd at `$CLAUDE_CONFIG_DIR/projects/<cwd-slug>/`, which lives **outside** the worktree. That's why a **build worktree is named for the spec slug** (`builds/build-<slug>`), stable across every phase: a chained Phase 2 is a *new* `agent_jobs.id` but must resume Phase 1's session from the *same* cwd. Keying the build worktree on `job.id` silently broke resume on **every** multi-phase spec ("No conversation found with session ID" ‚Äî the 2026-07-02 [[../archive.d/chained-phase-session-resume]] regression, fixed in PR #980); the `spec-chat`/`dev-ask` lanes use the same stable-path trick for the same reason. Corollary: reaping the worktree dir *between* phases is harmless ‚Äî the transcript survives outside it, so `--resume` still resolves once the stable path is recreated on the next dispatch. The orphan reaper therefore only reaps a `build-<slug>` worktree when **no** build job for that slug is still active.

## Multi-account round-robin + usage-cap failover (box-multi-account-failover)

A single Max account hits a **5-hour usage wall**; with one account a burst of builds caps it and *everything* fails until the reset (the 2026-06-22 "12 builds failed at once"). So the box keeps a **pool of `CLAUDE_CONFIG_DIR`s** ‚Äî each an isolated, once-logged-in Max account ‚Äî round-robins **new** sessions across them (~2√ó sustained throughput, each account hits its wall ~2√ó slower), pulls a capped account from rotation until its estimated reset, **pins every resume to the account that created its session** (a cross-account `claude --resume` is a guaranteed "No conversation found"), and parks `blocked_on_usage` (auto-resumes, never a hard fail) only when **all** accounts are capped. The worker logic is in `scripts/builder-worker.ts` (the `ACCOUNT_POOL` / `accounts` block). The pool is `CLAUDE_CONFIG_DIRS` (comma-separated; set explicitly in `/root/shopcx-worker.env`, four dirs as of 2026-07-01; the `builder-worker.ts` fallback lists the same four for an env-unset box).

**Set up the second account (one-time, owner op ‚Äî over the tailnet):**

```bash
# Log the SECOND Max account into its own isolated config dir (so two accounts coexist on the box).
sudo -u builder CLAUDE_CONFIG_DIR=/home/builder/.claude-personal claude   # ‚Üí /login ‚Üí sign in with account #2; creds persist in ~/.claude-personal
# A convenience alias for ad-hoc MANUAL runs / debugging under account #2 (the manual escape hatch):
echo "alias claude2='CLAUDE_CONFIG_DIR=/home/builder/.claude-personal claude'" >> /home/builder/.bashrc
# Verify each account independently (each is its OWN Max login, never an API key):
claude2 -p "say hi"      # runs under account #2
```

The **alias never reaches the worker** ‚Äî it `spawn`s `claude` directly (no interactive shell), so the worker selects the config dir in the `env` object it builds for each lane (`runClaude(..., configDir)`). The alias is only for *your* manual runs. As with account #1: **no `ANTHROPIC_API_KEY`** in either login (still Max, never per-token).

**Third+ account (same pattern ‚Äî added 2026-06-24 after a weekly cap on account #2).** Log account #3 into its own dir, one-time:

```bash
sudo -u builder CLAUDE_CONFIG_DIR=/home/builder/.claude-third claude   # ‚Üí /login ‚Üí sign in with account #3
echo "alias claude3='CLAUDE_CONFIG_DIR=/home/builder/.claude-third claude'" >> /home/builder/.bashrc
sudo -u builder CLAUDE_CONFIG_DIR=/home/builder/.claude-third claude -p "say hi"   # verify
```

**Fourth account (`invoices@superfoodscompany.com` = Round Robin 4, added 2026-07-01).** Config dir `/home/builder/.claude-fourth` is created; the pool env is flipped to include it only AFTER the login (gated to avoid drawing work onto an unlogged account). Log it in, one-time:

```bash
sudo -u builder CLAUDE_CONFIG_DIR=/home/builder/.claude-fourth claude   # ‚Üí /login ‚Üí sign in with invoices@superfoodscompany.com
echo "alias claude4='CLAUDE_CONFIG_DIR=/home/builder/.claude-fourth claude'" >> /home/builder/.bashrc
sudo -u builder CLAUDE_CONFIG_DIR=/home/builder/.claude-fourth claude -p "say hi"   # verify
```

Then activate: add `/home/builder/.claude-fourth` to the `CLAUDE_CONFIG_DIRS` line in `/root/shopcx-worker.env` and `systemctl restart shopcx-builder` ‚Äî the box then round-robins across all four.

A config dir that isn't logged in yet just fails its lanes (the failover pulls it / parks `blocked_on_usage`) ‚Äî it never blocks the healthy accounts, so adding the dir before the login is authorized is safe.

### ‚öÝÔ∏è 2026-06-24 incident ‚Äî the primary repo got `rm -rf`'d (worktree guard)

The worker once **deleted its own live repo** (`/home/builder/shopcx`): a build resume passed a branch that resolved the **primary** checkout's worktree, `removeWorktreeForBranch` fed that path to `removeWorktreeDir`, and its `rm -rf` nuked the repo out from under the running process (cwd went `‚Ä¶/shopcx (deleted)`; every dynamic `import()` + `git worktree add` then failed with "Cannot find module" / "worktree add failed"). Fix in `scripts/builder-worker.ts`: `removeWorktreeDir` now **refuses any path outside `BUILDS_DIR`** (only `builds/<uuid|build-<slug>|spec-chat-*>` worktrees are removable), and `removeWorktreeForBranch` **bails on an empty/falsy branch** (the primary shows as a detached worktree, branch `null` ‚Äî matching it was the trap). **Recovery** when the live repo is missing: stop the worker, re-clone fresh, restore `node_modules`, restart (below).

### Primary checkout self-healing (box-primary-checkout-branch-wedge-self-heal)

The 2026-07-05 commerce-sdk-display-operations build wedge exposed a **recurring class**: when the primary checkout (`/home/builder/shopcx`) gets parked on a feature branch with uncommitted changes, every build's `git worktree add -B <branch>` fails with "already used by worktree at /home/builder/shopcx", creating a **silent wedge** that blocks all builds of that branch until a human manually resets the primary. The worktree guard above refuses the unsafe `rm -rf` ‚Äî **correct** ‚Äî but that converts the failure into a silent stall. The fix is three-part **auto-recovery**:

1. **Build-claim precondition** ‚Äî Before claiming a worktree and starting a build, the worker asserts the primary checkout is on `main` and clean. If it's on a feature branch OR dirty, the worker stashes/cleans loose files, `git switch main`, `git reset --hard origin/main`, and deletes the stale local build branch. This makes a wedged primary auto-recover on the next build claim, not fail the worktree-add. Implemented in `scripts/builder-worker.ts` as a **precondition before every worktree setup** (composes with the existing periodic self-update).

2. **Worktree-add recovery** ‚Äî When `git worktree add -B <branch>` fails because the branch is held by the PRIMARY checkout (path == REPO_DIR, the original worktree guard that refuses `rm -rf`), the reaper/worktree-add precondition **switches the primary to main** (freeing the branch) and retries ‚Äî instead of failing the build terminally. The safety guard that refuses to delete the primary remains intact; this adds a **switch-based recovery for the one path that guard was dead-ending**.

3. **Primary-off-main audit** ‚Äî A lightweight startup check + periodic reaper tick logs and heals if REPO_DIR is ever found off `main`, so a regression surfaces immediately rather than as a silent wedge. All branch checkouts and PR diffs in the worker run in an **isolated worktree** (the existing `investWt` pattern), never in the primary, so the primary stays anchored to `main` by design. Code in `scripts/builder-worker.ts` at `claimBuildJob` (precondition) and the startup reaper, with a periodic confirmation in the reaper tick.

**Universal rollover via ONE foreman helper (universal-account-failover, 2026-06-27).** EVERY box `claude -p` session now routes through a single failover helper so a usage wall on one account **hops to the next** instead of killing the job ‚Äî no session type starts on a non-failover path:

- **The classifier was the load-bearing bug.** Anthropic's wall wording **drifted**: the current shape is a structured `rate_limit_event ¬∑ rateLimitType:"seven_day" ¬∑ status:"rejected" ¬∑ overageStatus:"rejected" ¬∑ overageDisabledReason:"org_level_disabled"` (the **7-day** account/org wall). The old `isUsageCapError` matched only the prose "usage/session limit" phrasings and deliberately ignored bare `429`/`rate_limit`, so this `seven_day:rejected` rejection slipped through **as if it were a transient 429** ‚Äî and the cap-hop never fired, even on paths that ARE wired for failover (a fold died on RR1 with no hop). `isUsageCapError` now classifies on the rate-limit/usage **signal** (robust to re-wording, not brittle phrases): a **hard rejection** (`status`/`overageStatus` "rejected", a window `rateLimitType` like `seven_day`/`weekly`/`*_limit`, or `org_level_disabled`) ‚Üí the wall; an explicit wall phrase ("usage limit reached", "N-hour/N-day limit", "limit will reset" ‚Ä¶) ‚Üí the wall; **529/overloaded with no wall signal stays a transient** (existing in-account retry); a **bare 429 with none of that vocabulary stays a transient too** (no false cap on a healthy build's transcript). `isHardRateLimitRejection` is the structured tell.
- **One helper, every lane.** `runBoxClaude` wraps the proven `withAccountFailover` in-call hop; `runBoxLane` is its single-shot variant (self-selects a healthy account, hops on the wall, **always returns a `RunResult`** ‚Äî synthesizing an all-capped marker the lane handles as a cap). The fold / goal-fold / spec-chat lanes (which used to run on the **default account with NO failover** ‚Äî the actual death) plus every single-shot lane ‚Äî repair, regression, security-review (its parse-repair resume pins to the landed account), ticket-improve, spec-test (repair pins), spec-review, migration-fix, pr-resolve, storefront-optimizer, triage (one account pinned per ticket ‚Äî its revise resumes the solver session), product-seed, and the platform-director's passes ‚Äî now route through `runBoxLane`. The interactive dev-ask/coach (Ask-Ada) paths already used `withAccountFailover`. The **feature build + plan DISPATCH** keeps its own richer, session-pinned `resolveAccountForJob` + `handlePoolUsageCap` re-dispatch (it was always wired for failover ‚Äî it just needed the classifier fix), so it is NOT double-wrapped. A hop **starts fresh** on the next account and rebuilds context from the transcript (the established pattern); resume **pins** to its session's owning account; **all-accounts-capped ‚Üí park `blocked_on_usage`** (auto-resumes at the soonest reset).
- **Reset-time parse (auto-recovery).** `parseResetTime` feeds the capped account's `cappedUntil` from, in priority order: (1) the `resetsAt` **unix-seconds epoch** inside the structured `rate_limit_info` JSON, (2) a "resets at &lt;ISO/time&gt;" / "try again at" phrase, (3) the legacy UTC clock-hour phrase ("resets 7pm (UTC)"); no parseable time ‚Üí the default ~5h cooldown (never caps forever). The capped account auto-returns at its real reset and logs `Round Robin N recovered ‚Äî usage wall reset ‚Äî back in rotation`.

**DB-driven health:** `restoreAccountCapsOnBoot()` restores each account's `cappedUntil` from the last heartbeat on boot, so a restart no longer resets accounts to healthy + re-probes the wall (the cap/failover-event spam after every restart is gone).

**Surfaced (box-multi-account-failover Phase 2):** the worker writes a per-account snapshot onto its [[../tables/worker_heartbeats|heartbeat]] (`accounts` column ‚Äî per-account in-flight load + capped state, healthy/all-capped flags, soonest-reset, and a recent cap/failover/recovery event ring). [[../dashboard/roadmap]] `/dashboard/roadmap/box` renders a **Max accounts** panel (per-account load + an all-capped banner + recent events) and the [[../dashboard/control-tower]] box tile flips **amber** "all Max accounts capped ‚Äî builds parked, auto-resume" so a silent everything's-capped state is visible. See [[../libraries/control-tower]] (`evalWorker`).

## Second runtime: Codex on a ChatGPT plan (box-codex-runner)

The box runs a **second agent runtime** ‚Äî OpenAI's `codex exec` on a **ChatGPT-plan device-code login** (NOT an API key ‚Äî same "session billing, no per-token `$`" shape as `claude -p` on Max). A whitelist of job kinds runs on **Codex primary**; **Claude is the fallback** when Codex is capped or errors (reusing the existing account-failover machinery). This **offloads Max** ‚Äî directly relieving the usage wall ‚Äî while keeping customer-facing voice + the core build/plan on Claude.

**The split (owner decision 2026-07-01).**

| Runtime | Kinds |
|---|---|
| **Claude** (kept) | `build` (Bo), `plan` (Pia), Ada's chat/voice + her `director` standing pass, and the customer-voice kinds: `ticket-improve`, `product-seed`, `spec-chat`, `storefront-optimizer`, `triage-escalations` |
| **Codex** (primary, Claude fallback) | grading + coaching (`agent-grade`, `agent-coach`, `director-grade`, `campaign-grade`, `gap-grade`), `spec-review`, `spec-test`, `repair`, `regression`, `security-review`, `pr-resolve`, `fold` |

The routing lives entirely inside `runBoxSession` (`CODEX_KINDS` / `runtimeForKind` / `runCodexSession` in `builder-worker.ts`) ‚Äî a Codex-routed kind runs `codex exec` first and, on a Codex cap/error/throw, **falls through to the Claude path** on the healthy account the failover wrapper already picked. Because a Codex run returns the **same `RunResult`** (session/text/isError/usage/model), caller metering (`meterAgentJob`, model tagged `codex/‚Ä¶`) and the failover wrappers are untouched ‚Äî **no call site changed**.

**Deferred:** the two *resumable* Codex-candidate kinds ‚Äî `dev-ask` and `migration-fix` (answer-resume) ‚Äî **stay on Claude** for now, because a Codex thread id can't `--resume` on Claude and vice-versa; moving them needs per-job **runtime pinning** (store the runtime beside `claude_session_config_dir`). Everything in the Codex column above is single-shot, so the fallback is trivially correct (start fresh on Claude). The in-app **Sonnet-API** agents (`db_health`, `coverage-register`, `deploy-guardian`, `spec-drift`, `monitor`) don't route through the box runner at all, so they're out of scope for this seam.

**Set up Codex (one-time, owner op ‚Äî over the tailnet):**

```bash
# 1. Install the CLI globally (as root, so `builder` gets it on PATH).
sudo npm i -g @openai/codex        # ‚Üí /usr/bin/codex

# 2. Device-code login on the ChatGPT plan whose Codex credits the box should burn (NOT an API key).
sudo -iu builder bash -lc 'CODEX_HOME=/home/builder/.codex codex login --device-auth'
#   ‚Üí prints a URL (auth.openai.com/codex/device) + a one-time code; approve in a browser.
#   ‚Üí writes /home/builder/.codex/auth.json. Verify: `codex login status` ‚Üí "Logged in using ChatGPT".

# 3. Provision headless full-auto ‚Äî REQUIRED. In `codex exec` the sandbox modes (read-only /
#    workspace-write) AUTO-DENY model-generated commands non-interactively, so Codex can't run
#    git/cat/node and HALLUCINATES instead of reading the repo. approval_policy="never" +
#    danger-full-access = run commands unattended (the box is itself the isolation boundary, same
#    rationale as Claude's --dangerously-skip-permissions; per-lane secret stripping happens in the
#    worker env). Written to CODEX_HOME/config.toml:
cat > /home/builder/.codex/config.toml <<'TOML'
approval_policy = "never"
sandbox_mode = "danger-full-access"
TOML
```

The worker also passes `--dangerously-bypass-approvals-and-sandbox` on every `codex exec` (belt-and-suspenders if the config is ever missing). Each lane's env drops **both** `ANTHROPIC_API_KEY` and `OPENAI_API_KEY` (an API key in env would flip Codex to per-token billing instead of the ChatGPT plan) and points Codex at `CODEX_HOME`.

**Skill parity.** The box lanes are authored for Claude Code and several say "use the `<name>` skill" (auto-loaded from `.claude/skills/`). Codex has no skill system, so `runCodexSession` prepends a **preamble** teaching Codex the convention ‚Äî it reads the referenced `.claude/skills/<name>/SKILL.md` itself (the skills are git-committed), honors the read-only contract, and ends with the single-JSON final message the worker parses. Verified 2026-07-01: Codex reads the real skill, runs `git`/`cat`, and emits the exact expected JSON.

**Controls.** `BOX_CODEX_ENABLED=0` (in `/root/shopcx-worker.env`) is the kill-switch ‚Üí every kind reverts to Claude. `BOX_CODEX_MODEL` overrides the model (default: codex's `gpt-5-codex`). On a Codex usage wall, `markCodexCapped` cools Codex down (re-probe clamped like a session account) and routes to Claude until it clears ‚Äî logged in **AST** (`[runtime] codex usage wall ‚Äî routing to Claude until ‚Ä¶`).

## Bo agent's durable mandate ([[../specs/agent-mandate-hardening-build]] ‚Äî shipped)

The Bo build agent runs with a **DURABLE MANDATE** ‚Äî five pieces of accumulated coaching that were baked into the permanent prompt in `scripts/builder-worker.ts` and the agent's personality in `src/lib/agents/personas.ts` (see [[../functions/platform]] "Autonomous build platform"). The mandate replaces ephemeral per-job `agent_instructions` for these five classes of mistakes ‚Äî Bo now **always** behaves this way, and any new coaching flows through the ephemeral path only (closing the grade‚Üícoach‚Üíapply loop, moving recurring issues from coaching triage to durable code).

**The five mandate items (lines ~14850‚Äì14900 in `scripts/builder-worker.ts`):**

1. **START-STATE RECONCILE** ‚Äî Before writing a single line, verify what a prior session or sibling PR already carries. `git log --oneline origin/main..HEAD` on `claude/build-<slug>` and `gh pr list --head claude/build-<slug>` ‚Äî if commits exist for THIS phase, DO NOT rebuild from scratch; make only the follow-up edits needed to green tsc/next-build. If a PR already exists, ADOPT it (push follow-up commits only, or return no_changes_reason). Rebuild churn from ignoring an existing head is the #1 recurring build mistake.

2. **PHASE-SCOPED COMMIT** ‚Äî Read the assigned `Phase N` and commit ONLY that phase's files. Deliver a SINGLE-PHASE branch ‚Äî never bundle Phase N+1 silently. If a later phase's file is unavoidable for THIS phase to compile, name it explicitly in the summary. The worker enforces one-phase-per-session and stamps from the branch's `Phase:` trailer; a bundled commit collides with the separately-queued later phase and produces a dirty/failed PR.

3. **TEST-FIRST for a NAMED FAILING STATE** ‚Äî When the spec names a concrete failing state (a gate omitting a disposition, an empty-body accepted via title/summary), write the smallest test that asserts the CORRECT state first, watch it fail, then change ONLY the predicate needed to make it pass. Skip only when the spec's verification is code inspection / grep-only and say so in the summary.

4. **DERIVED-PHASE ALIGNMENT** ‚Äî The phase you build MUST be the FIRST nonterminal `public.spec_phases` row whose `build_sha` is null. If the scope names `Phase N`, verify your commit trailer says `Phase: N` ‚Äî the worker's `stampPhaseBuilt` reads it. A trailer mismatch leaves later phases' `build_sha` unstamped. Return no_changes_reason if the code is already on main.

5. **ONE PHASE PER SESSION IS THE MANDATE** ‚Äî The worker enforces this and AUTO-QUEUES the next phase's build. Do NOT power through Phase 2/3 in the same session. Ship THIS phase's diff, return completed, and let the auto-chain take the next ‚Äî a clean single-phase PR merges faster than an ambitious bundled one that collides at merge.

Bo's `personality` string in `src/lib/agents/personas.ts` summarizes the mandate so it's discoverable in the org chart. The permanent prompt block replaces the recurring coaching that was reaching Bo through 13+ ephemeral sessions; durable behavior now frees the coaching lane to handle fresh mistake classes instead of re-teaching the same five items every build.

## Queue restart (drain-for-update, shipped 2026-06-24)

Devops constantly queues builds, so the box rarely idles ‚Äî and the idle self-update only fires when no lane is running, so a merged worker fix can sit unloaded for a long time. The **Queue restart** button on `/dashboard/roadmap/box` stages a clean restart without force-killing in-flight builds: it sets `worker_controls.drain_for_update` (POST `/api/roadmap/box/drain`, owner-only). While draining the worker **claims no new work** ‚Äî in-flight lanes finish, the box reaches idle, the idle self-update fires (SHA advances), and the fresh worker's **boot clears the flag** (`requested_at_sha != HEAD`), with a 30-min max-age safety clear so a no-op drain (nothing to pull) can't strand the box. New builds just pile up `queued` and run once the box is current. Toggle it off to cancel. The button + an amber "draining" banner show the state; the worker logic is `readDrainControl` / `reconcileDrainOnBoot` in `scripts/builder-worker.ts`.

## Worker self-update (worker-self-update, shipped 2026-06-19)

The worker keeps **its own code** current ‚Äî a merged worker fix goes live within one idle cycle, **zero manual redeploy** (closes the gap that left #77's `markReady` fix inert until a human ran the command above, so PRs kept coming out draft).

- **Idle self-update loop.** Each poll tick, **only when `active.size === 0`** (no build/fold lane ‚Äî in-flight work is sacrosanct), the worker `git fetch origin main` and compares boot-time `RUNNING_SHA` to `origin/main` (the canonical staleness signal ‚Äî the SHA the *process is executing*, not a fresh disk read). If behind: checks disk `HEAD` vs. `origin/main` ‚Äî if already equal, skips the redundant `git reset --hard` and proceeds directly to `process.exit(0)` (the fresh on-disk code gets loaded); if disk is behind, runs `git reset --hard origin/main`, `npm ci` **if `package-lock.json` changed**, then exits. `systemd Restart=always` relaunches on the fresh `builder-worker.ts`. A clean exit + restart is the safe re-exec (never hot-reload in-process). Worktrees live in the sibling `/home/builder/builds/`, so resetting the main repo never touches a running build. The journal logs the `from‚Üíto` short SHA.
- **No thrash.** Self-update only fires when `HEAD != origin/main`, at most once per `SELF_UPDATE_MIN_INTERVAL_MS` (60s).
- **Honors the watchdog quarantine (worker-self-update-respects-quarantine).** Before advancing, `maybeSelfUpdate` reads the watchdog's marker (`readQuarantinedSha()` ‚Üí `quarantinedSha` in `/home/builder/.box-watchdog.json`; same `builder` user owns the file). If `quarantinedSha` is set **and matches the SHA `origin/main` currently points at**, the worker **does NOT update** ‚Äî it stays on the current (good) checkout and logs loudly (`[self-update] origin/main <sha> is QUARANTINED by the watchdog ‚Äî holding on <good-sha>; fix main to release`). This closes the **oscillation** the watchdog flagged in ¬ß Box crash-loop watchdog: the worker no longer re-pulls a known-bad tip every idle cycle for the watchdog to re-roll back. Compare is **prefix-tolerant** (the watchdog stores the SHORT sha, the worker holds the full `origin/main` sha; a `--short` sha is ‚â•7 hex so `startsWith` is unambiguous). **Fail-open for availability:** a missing / unreadable / malformed marker returns `null` ‚Üí self-update behaves exactly as before; only a clearly-set `quarantinedSha` matching `origin/main`'s tip holds. **Self-releasing:** the moment `main` advances to a different (fixed) SHA the pin no longer matches ‚Üí update proceeds, and the watchdog clears `quarantinedSha` once the box is healthy on the new SHA. `WORKER_PIN_SHA` env is an additional manual operator hold in the same shape (a SHA the box must refuse to advance onto); the watchdog-file read is the primary mechanism.
- **Crash-loop guard.** A per-SHA startup counter persists in `/home/builder/.worker-startup.json`. If a freshly-pulled worker keeps dying on startup (same SHA, ‚â• `CRASH_LOOP_MAX = 3`), it writes a `needs_attention` [[../tables/worker_heartbeats|heartbeat]] (the breadcrumb) and `exit(1)`s so systemd's `StartLimit` stops the flapping instead of churning a broken commit. The counter zeroes on the first clean poll tick. **Falls back to manual redeploy** (above).
- **Heartbeat.** Every tick the worker upserts [[../tables/worker_heartbeats]] (running SHA ¬∑ `active_builds` ¬∑ `last_poll_at`); [[../dashboard/branches]] shows a **Build box** banner so "is the box behind?" is answerable from the UI.

**Required systemd config for the crash-loop guard** (in `/etc/systemd/system/shopcx-builder.service`): `Restart=always` (relaunch on the self-update `exit(0)`) **plus** a start-limit so a flapping worker is eventually parked rather than restarted forever:

```ini
[Unit]
StartLimitIntervalSec=300
StartLimitBurst=5
[Service]
Restart=always
RestartSec=5
```

**Bootstrapping:** self-update ships in `builder-worker.ts` itself, so the **one last manual redeploy** (the command above) activates it; every redeploy after that is automatic.

## Box crash-loop watchdog (box-crash-loop-watchdog)

**The incident this exists for:** the worker crash-looped for **~5 HOURS** on a syntax error, **completely unnoticed**. `systemctl restart` returns success when it *issues* the restart, **not** when the worker comes up healthy ‚Äî so every restart looked fine while the service `activating ‚Üí failed ‚Üí auto-restart` looped (restart counter climbed to **11**). The worker's OWN crash-loop guard (`recordStartupAttempt`, ¬ß Worker self-update) **can't catch a parse error**: a syntax/esbuild error kills the module *before* `main()` runs, so the per-SHA counter never increments and nothing parks it. A bad commit to `main` reaches the box via the idle self-update (`git reset --hard origin/main`), so any merge can crash-loop the box.

The watchdog is a **separate, dead-simple monitor that runs from OUTSIDE the worker** (its own systemd timer, **every 60s**) so it **survives a worker crash**. It does NOT modify the worker (monitors it from outside). Entrypoint: **`scripts/box-watchdog.ts`** (parse-gated in [scripts/_check-box-parses.ts](../../../scripts/_check-box-parses.ts), like every box tsx entrypoint). State persists in **`/home/builder/.box-watchdog.json`**.

**1. Detect ‚Äî crash-loop vs. a normal self-update restart.** Each tick reads `systemctl is-active shopcx-builder` + the last **6 min** of `journalctl -u shopcx-builder` and counts **FAILURE exits** (`Main process exited, code=exited, status=N/FAILURE`, N‚âÝ0) + the max `restart counter is at N` + the latest `builder-worker up ‚Äî ‚Ä¶ @ <sha>` line + a captured Node/esbuild **error line** (the cause). The discriminator:
- A **self-update restart** is a CLEAN `exit(0)` ‚Üí systemd logs "Deactivated successfully"/"Stopped", **never** `status=1/FAILURE`, then a fresh `up ‚Äî` line + `is-active=active`. So `failureCount` stays **0** and the tick reads **healthy** ‚Äî never acted on.
- A **crash-loop** is repeated `status=1/FAILURE` + a climbing restart counter with **no** `up ‚Äî` line after the failures (it dies before booting). **Confirmed** = `‚â• CRASH_FAILURES (3)` FAILURE exits within 6 min, OR `‚â• UNHEALTHY_CONSEC (3)` consecutive unhealthy watchdog ticks, OR the box is back on the quarantined bad SHA. A **single** clean restart / transient blip is "unhealthy but UNCONFIRMED" ‚Üí it waits, does not alert/roll back.

**2. Alert the CEO LOUDLY (fires even if the worker is down ‚Äî own process + DB).** On a confirmed crash-loop it opens ONE critical **`loop_alerts`** row (`loop_id='box-crash-loop'`, `owner='platform'`, `kind='worker'`, `reason='crash_loop'`, `signature='box_crash_loop:<sha>'` ‚Äî distinct `loop_id` from the registry `'box'` loop, so the [[../inngest/control-tower-monitor]] never fights it, same self-managed pattern as kpi-drift) carrying the **failing SHA + the esbuild/Node error**, AND a CEO-routed **`dashboard_notifications`** row (`type='system'`, de-duped per failing SHA via `metadata.dedupe_key`, workspace resolved from the latest `agent_jobs` row ‚Äî the build-console workspace). The alert refreshes while open and **auto-resolves** when the box is healthy again.

**3. Auto-roll-back to last-known-good.** **Known-good** = the SHA from the most recent `builder-worker up ‚Äî ‚Ä¶ @ <sha>` line (fallback: live `HEAD`), persisted **only** on a tick the box is confirmed healthy ‚Äî so even a 5h crash-loop window with no `up ‚Äî` line still has a durable rollback target. On a confirmed crash-loop the watchdog does `git reset --hard <good-sha>` (in `/home/builder/shopcx`, as the `builder` user via `runuser`) + `systemctl restart shopcx-builder`, then **VERIFIES** it came up healthy (`is-active=active` + a fresh `up ‚Äî` line within ~15s). It does **NOT revert `main`** ‚Äî only the box's checkout ‚Äî so the CEO still sees + fixes the bad commit. The rollback runs **before** the DB alert (survival first; a Supabase outage never blocks it).
- **Self-update race ‚Äî FIXED, the worker now honors the quarantine (worker-self-update-respects-quarantine).** The worker self-updates its checkout back to `origin/main` when idle (`maybeSelfUpdate`). The watchdog **QUARANTINES** the bad SHA on rollback (sets `state.quarantinedSha = failingSha` in `/home/builder/.box-watchdog.json`, persisted as the **short** sha) and re-asserts the rollback on any tick `HEAD` returns to it. **The worker now READS that marker** (`readQuarantinedSha()`) before advancing and **refuses to self-update onto a quarantined SHA** ‚Äî so it no longer re-pulls the bad tip each idle cycle. This is the **marker-file contract** between the two: the watchdog WRITES `quarantinedSha`, the worker READS it (well-known path `/home/builder/.box-watchdog.json`, both run as `builder`). Net (now): the box holds steadily on good code ‚Äî **no oscillation** ‚Äî and stays loudly alerted until the CEO fixes `main`; the moment `main` advances past the quarantined SHA the worker updates, and the watchdog **clears quarantine + resolves the alert** when the box is healthy on the new SHA. The watchdog is unchanged by this fix (it already wrote the marker); only `maybeSelfUpdate` gained the read + hold. `WORKER_PIN_SHA` env is an additional manual operator hold in the same shape. See ¬ß Worker self-update.

**4. Verify-after-restart helper (the discipline gap that hid the 5h outage).** `npx tsx scripts/box-watchdog.ts --verify` confirms the worker actually came up healthy (`is-active=active` + the `up ‚Äî` heartbeat within ~15s) and exits `0`/`1` (prints `HEALTHY`/`UNHEALTHY`). Run it after ANY manual restart/redeploy so a deploy can't "succeed" while the worker is dead:
```bash
sudo -u builder git -C /home/builder/shopcx fetch origin && sudo -u builder git -C /home/builder/shopcx reset --hard origin/main && systemctl restart shopcx-builder && \
  tsx /home/builder/shopcx/scripts/box-watchdog.ts --verify   # ‚Üê exits non-zero if the worker didn't come up healthy
```

### Install on the box (deploy step ‚Äî owner op, run once as root over the tailnet)

The watchdog ships in-repo (`scripts/box-watchdog.ts` + the units under `ops/systemd/`) but the systemd **timer must be installed on the box** ‚Äî it is NOT auto-deployed by the worker self-update. One-time:
```bash
# tsx + the repo are already present (the worker uses them). Install the unit + timer:
cp /home/builder/shopcx/ops/systemd/shopcx-box-watchdog.service /etc/systemd/system/
cp /home/builder/shopcx/ops/systemd/shopcx-box-watchdog.timer   /etc/systemd/system/
systemctl daemon-reload
systemctl enable --now shopcx-box-watchdog.timer    # enable the TIMER (the .service is Type=oneshot ‚Äî never enable it directly)
# Verify it fires + reads healthy:
systemctl start shopcx-box-watchdog.service && journalctl -u shopcx-box-watchdog -n 20 --no-pager   # expect "[watchdog] healthy ‚Äî active, 0 failures‚Ä¶"
systemctl list-timers shopcx-box-watchdog.timer
```
The `.service` uses the **same `EnvironmentFile=/root/shopcx-worker.env`** as the worker (it needs the Supabase service role to write `loop_alerts`/`dashboard_notifications`). It runs as **root** (for `systemctl restart` + `journalctl`); the repo `git reset` drops to the `builder` user via `runuser` (correct git ownership). Code changes to `scripts/box-watchdog.ts` reach the box via the worker's normal self-update of the repo, so only a change to the **unit files themselves** needs a re-`cp` + `daemon-reload`.

### How it's verified (detect / known-good / no-false-positive)

- **Healthy worker / single clean restart ‚Üí no action.** A self-update is a clean `exit(0)` (no FAILURE line) so the tick reads healthy; a lone transient failure is "UNCONFIRMED" (1 < 3 failures, consec < 3) and only waits. Neither alerts nor rolls back.
- **Simulated crash-loop ‚Üí alert + rollback fire.** ‚â•3 FAILURE exits in 6 min (or 3 consecutive unhealthy ticks) ‚Üí confirmed ‚Üí it opens the critical `loop_alerts` row + the CEO notification with the failing SHA + error, rolls the checkout to the persisted known-good SHA, restarts, and verifies. A safe simulation: temporarily point the watchdog at a throwaway unit (`WATCHDOG_SERVICE`/`WATCHDOG_REPO_DIR` envs) that runs a script which `exit 1`s on boot, or merge a known-bad commit to a test branch the box tracks ‚Äî never test against live `main`.
- **Known-good is durable + correct.** It's only ever recorded on a *confirmed-healthy* tick (active + 0 failures), parsed from the `up ‚Äî` line the worker logs at boot, so it can't be poisoned by the crash-looping SHA (which never logs `up ‚Äî`).

## Startup orphan-reaper (worker-orphan-reaper, shipped 2026-06)

A restart ‚Äî self-update (`git reset --hard origin/main` + `exit(0)`), deploy, or crash ‚Äî abandons any job the old instance had **in flight** (`building`/`claimed`/`queued_resume`). The new worker only claims `queued`, so those rows sit in `building` **forever**, never completing and tripping the Control Tower's "N jobs stuck in building past 60m" alert (observed live: 7 spec-test jobs piled up across several deploys in one session). So **`reapOrphans()` runs in `main()` before the poll loop** ‚Äî best-effort (wrapped so a reaper failure never blocks startup), idempotent (a clean boot with nothing orphaned is a no-op), and **cutoff-gated**: it only touches jobs `claimed_at < WORKER_STARTED_AT` (the heartbeat `started_at`), so nothing the *current* instance owns is ever mid-build-killed.

Reap is **by kind**:
- **Re-runnable / idempotent kinds** ‚Äî `RERUNNABLE_KINDS` = `spec-test`, `triage-escalations`, `migration-fix`, `dev-ask`, `pr-resolve` ‚Üí **reset to `queued`** (clear `claimed_at`) so they simply re-run. No work lost.
- **Work-producing kinds** ‚Äî `build`, `plan`, `fold`, `product-seed`, `spec-chat`, `ticket-improve` ‚Üí **mark `failed`** with `error="orphaned by worker restart"` (NOT re-queued ‚Äî a restart may have left a half-pushed branch, and a blind re-run could double-push). The failed-builds callout on [[../dashboard/roadmap]] then surfaces it; if a branch *was* pushed, [[build-recover-pr-create]]'s **Create PR** recovers the completed work ‚Äî no rebuild.
- Logs the reap counts by kind (`[reaper] reaped N orphan(s) ‚Ä¶ re-queued: ‚Ä¶; failed: ‚Ä¶`, or `0 reaped ‚Äî no orphaned in-flight jobs from a previous instance` on a clean boot) so a restart's cleanup is visible, not silent.

`RERUNNABLE_KINDS` is the **single shared source of truth** ‚Äî the self-update `sacrosanctActive` check (lanes a restart may interrupt) reads the same constant, so "safe to re-run" can't drift between reaper and self-update. No migration (free-text `kind`, existing [[../tables/agent_jobs]] columns). This is the standing replacement for the manual `requeue-stale` ([[manage-the-build-queue]]) after a restart, and removes the stuck-jobs false-positive source the [[../dashboard/control-tower]] flagged.

## Stale-session reaper (stale-session-reaper, shipped 2026-08)

The startup orphan-reaper above only fires on a **clean restart**. It misses the zombie class where a session dies *while the worker keeps running* ‚Äî a Max usage cap mid-run, a session crash/disconnect, or a box **crash-loop that never reaches `reapOrphans`**. The row stays `building` with `updated_at` frozen at `claimed_at`, the lane is held, and the work stalls **indefinitely** (observed live: a `plan` job for goal `growth-director` claimed in `building` never updated again for ~5h). The fix is a **heartbeat the live session bumps + a periodic in-loop reaper keyed on its staleness**, all in `scripts/builder-worker.ts`, **lane-agnostic** (every session-bearing kind ‚Äî build/plan/spec-test/director/‚Ä¶ ‚Äî runs through `runBoxSession`, so one mechanism covers all). Thresholds:

- **M = `HEARTBEAT_INTERVAL_MS` = 2 min** ‚Äî `runBoxSession` bumps [[../tables/agent_jobs]] `last_heartbeat_at` on **each** stream-json line (a session emits these continuously while alive ‚Äî the reason the runner uses `stream-json`), throttled to once per M via `makeHeartbeatBumper`. It's a **direct `last_heartbeat_at`-only write**, NOT the `update()` helper, so a heartbeat never bumps `updated_at` or runs the needs_attention classifier. `launch` also stamps an initial heartbeat at claim, so a just-claimed job that hasn't emitted its first line yet isn't prematurely reaped. This is how **"alive but long" is distinguished from "dead"**: a live session's heartbeat is at most M old.
- **N = `REAP_STALE_MS` = 20 min** ‚Äî `reapStaleSessions(active)` runs **~every minute** in the poll loop (`REAP_SWEEP_INTERVAL_MS`), in-flight-guarded + throttled, placed **before the claim block** so a freed lane re-fills the same tick. It selects in-flight rows (`building`/`claimed`/`queued_resume`) whose `last_heartbeat_at` (fallback `updated_at` for a pre-migration row with a NULL heartbeat, so an old zombie is still caught) is stale > N. **N >> M + the 10-min build idle-kill window**, so a legitimately-long-but-alive build is never reaped. A reaped job is **re-queued FRESH** (`status='queued'`, `claimed_at`/`claude_session_id`/`claude_session_config_dir`/`log_tail` cleared ‚Äî a dead session's account can't be resumed, "No conversation found") and `reap_count` incremented, so it re-runs on whatever account the dispatcher picks next and the lane frees.
- **Live-vs-dead guard** ‚Äî a job **this** worker process is actively running is in the in-memory `active` Map and is **never reaped** (its session is alive ‚Äî its heartbeat is at most M old; a transient DB hiccup that delayed a bump must not kill a live build). The reaped zombie is a *dead* process's row, so its account `inFlight` counter was incremented in that process, not this one ‚Äî there's nothing to decrement here (this process's live in-flight sessions are exactly the `active` ones we exempt).
- **K = `REAP_MAX` = 3** ‚Äî a job with `reap_count ‚â• K` stops being re-queued and **escalates to `needs_attention`** (a CEO-visible park, error = "session died mid-run K√ó; escalating instead of re-queuing") instead of infinite-looping the reaper on a structurally-doomed job.
- Every reap/escalate is **logged** (`[reaper] REAP stale session ‚Ä¶ ‚Üí re-queued (reap n/K)` / `[reaper] ESCALATE zombie ‚Ä¶ ‚Üí needs_attention` / a per-sweep summary) ‚Äî never silent.

Migration `20260802120000_agent_jobs_heartbeat_reaper.sql` (apply `scripts/apply-agent-jobs-heartbeat-reaper-migration.ts`) adds `last_heartbeat_at` + `reap_count` + a partial index on `last_heartbeat_at` over the in-flight statuses. This complements the startup reaper (mid-run deaths it can't see) and the build idle-kill (a hang *within* a live session) ‚Äî together they cover every "row stuck building" failure mode.

**Stale-worktree reap + idempotent worktree-add (worker-orphan-reaper-worktree-prune).** The reaper handles orphaned *jobs* but the box never cleaned git **worktrees** ‚Äî two failures, one root. (1) A **resume** re-ran `git worktree add <branch>` while a worktree for that branch still existed from the first run ‚Üí `fatal: '‚Ä¶' is already used by worktree at ‚Ä¶` and the build failed (the work was fine; the infra wasn't). (2) **Stale worktrees accumulated** ‚Äî a completed/merged/failed build's `builds/<uuid>` dir lingered (`git worktree prune` alone won't remove it while the dir exists), piling up until they collided or filled disk. The fix, all in `scripts/builder-worker.ts`:
- **Idempotent worktree-add** ‚Äî `removeWorktreeForBranch(branch)` force-removes ANY worktree currently holding `<branch>` (`git worktree remove --force` + prune), called immediately **before** `git worktree add -B <branch>` on BOTH the build job's fresh-build and resume paths in `runBuildJob`. So a resume can never fatal with "already used by worktree at ‚Ä¶".
- **Worktree reaping** ‚Äî `reapOrphanWorktrees()` (wired into `reapOrphans`, so it runs at startup / every self-update restart, and always sweeps even when 0 jobs were orphaned) lists `git worktree list --porcelain` and removes every `builds/<job.id>` worktree (UUID basename; the `spec-chat-*`/`dev-ask-*` lanes are excluded) whose backing [[../tables/agent_jobs]] row is **terminal** (`completed`/`failed`/`needs_attention`) or **no longer exists** (how a merged-and-deleted branch surfaces). Worktrees for jobs in `ACTIVE_JOB_STATUSES` (`queued`/`claimed`/`building`/`needs_input`/`needs_approval`/`queued_resume`) are **never** touched. Logs `[reaper] worktrees: removed N terminal/orphaned, kept M active`.
- Helpers: `listWorktrees()` (parses the porcelain output ‚Üí `[{path, branch}]`, `refs/heads/` stripped, null when detached) and `removeWorktreeDir(path)` (`git worktree remove --force` + `rm -rf` any lingering dir; tolerates an already-gone dir ‚Äî a later `git worktree prune` reconciles the admin entry; never throws). Runs as the worker's own user (correct git ownership ‚Äî no "dubious ownership").

**Archived-spec orphan reap (fold-guard-live-build).** Right after `reapOrphans()`, `main()` also runs `reapArchivedSpecJobs()` ‚Äî it calls `cancelJobsForArchivedSpecs` ([[../libraries/agent-jobs]]) to cancel every non-terminal `build`/`spec-test` job whose `spec_slug` now lives in `docs/brain/archive.d/`. A fold moves the spec markdown out from under any build still running, so that build's spec page 404s and its paused/active card is a dead link; this flips each such job `status='completed'` `error='spec archived ‚Äî build auto-cancelled ‚Ä¶'` (questions cleared). Logs `cancelled N orphaned job(s) for archived spec(s): ‚Ä¶` (or `0 archived-spec orphans ‚Ä¶` on a clean boot). The preventive guard (the fold path refusing a live slug via `getLiveJobForSlug`) makes this rare; this is the belt-and-suspenders for a fold that raced a build. Best-effort, idempotent. The same `cancelJobsForArchivedSpecs` also fires on the `kind='fold'` merge reconcile (board load after a fold PR merges).

## Gotchas (learned in the live bring-up 2026-06-18)

- **A fresh box has no git identity** ‚Üí `git commit` silently fails ‚Üí empty branch ‚Üí GitHub rejects the PR ("No commits between‚Ä¶"). The worker now sets a repo-local identity itself (`ensureGitIdentity`) and fails loudly on commit/push/PR errors.

- **`pkill -f builder-worker.ts` matches its own shell** (the pattern is in the command's argv) ‚Äî kill by process group / pidfile, or let `systemd` manage it.

## Non-root builder + sandboxed builds (build-approval-gates, 2026-06-18)

`shopcx-builder` now runs as the **non-root `builder` user** from `/home/builder/shopcx` (its own clone + `/home/builder/.claude` Max login), so builds run `--dangerously-skip-permissions` (refused as root) with no per-tool prompts. Prod stays safe via two secret stores:
- **`/root/shopcx-worker.env`** (root-only `0600`, systemd `EnvironmentFile=`) holds the worker's prod creds (Supabase service role, DB password, `GITHUB_TOKEN`, ‚Ä¶); systemd injects them into the worker process.
- The builder repo has **no `.env.local`**, and the worker **strips secrets from the spawned build's env** (`SECRET_RE` in `builder-worker.ts`) ‚Äî so the `claude -p` build can't reach prod. The worker (which holds the creds) executes only **owner-approved** gated actions.

Update worker code: `sudo -u builder git -C /home/builder/shopcx fetch origin && git reset --hard origin/main && systemctl restart shopcx-builder`.

## Parallel builds ‚Äî 8 worktree-isolated lanes (parallel-builds, shipped 2026-06-18; pool 5‚Üí8)

The worker runs **up to `MAX_CONCURRENT = 8` builds at once**, each in its own **git worktree** under `/home/builder/builds/{job-id}` (isolated dir + branch) so concurrent builds never clobber each other. The pool claims (`claim_agent_job()`) up to 8 jobs per poll tick and tops up as lanes free; a resume recreates a worktree on the job's pushed branch (force-removing any prior tree for that branch first ‚Äî see ¬ß Stale-worktree reap, so a resume can't fatal with "already used by worktree"), and every worktree is torn down (`git worktree remove --force`) on any outcome; a worktree that a crash/restart leaves behind is swept by the startup reaper. Cap = **8** because the real ceiling is **Max rate limits**, not the 8-core box (tune `MAX_CONCURRENT`).

**‚≠ê The claim gate counts RUNNING lanes only ‚Äî parked ‚âÝ running (box-fill-8-lanes).** The build/plan fill loop's "is there a free lane?" check is `while (countOther() < MAX_CONCURRENT)` where `countOther()` reads the **in-memory `active` map** ‚Äî only the jobs THIS worker is actually running on a lane (`claimed`/`building`). A build that **parks on a human** (`needs_input`/`needs_approval` ‚Äî "1 build paused ‚Äî storefront-optimizer, awaiting owner action") **ends its session, so `active.delete` frees its lane** and it does NOT count against the 8. So the claim path correctly computes free lanes as `8 ‚àí running` and fills ALL idle lanes from the queue each tick. The multi-account round-robin is **account-spread ONLY, never a concurrency gate**: `pickNewSessionAccount` returns the least-loaded HEALTHY account (per-account `inFlight` is just the LRU sort key), so **one healthy Max account can back all 8 lanes** ‚Äî it only blocks a claim when EVERY account is genuinely usage-capped (‚Üí `blocked_on_usage`, auto-resumes). The matching ENQUEUE-side rule lives in the director: `idleBuildCapacity` (the saturation target that decides how many specs to groom/initiate into the queue) also **excludes parked builds** from its in-flight count ‚Äî a parked build must never read the pool "full" and starve the queue. See [[../libraries/platform-director]] `idleBuildCapacity`. **The bug this fixed:** before, `idleBuildCapacity` counted `needs_approval`/`needs_input`, so a couple of builds parked awaiting the CEO read the pool full ‚Üí the director enqueued nothing ‚Üí queued builds sat 30+ min with lanes wide open.

**Idempotent worktree-add (worker-orphan-reaper-worktree-prune, shipped 2026-06).** Both the fresh-build and the **resume** path call `removeWorktreeForBranch(branch)` **before** `git worktree add -B <branch>` ‚Äî it force-removes ANY worktree currently holding `<branch>` (via `removeWorktreeDir`), then prunes. Without it, a build that paused for approval and was then re-claimed hit `fatal: '<branch>' is already used by worktree at ‚Ä¶` on resume (the work was fine; the infra collided with the first run's leftover tree). Now the add can never fatal on a pre-existing tree ‚Äî a resume always re-establishes its worktree cleanly. The matching cleanup of *terminal/orphaned* trees is the ¬ß Startup orphan-reaper worktree sweep.

**Three gotchas that bit us in bring-up (permanent notes):**
- **Async exec is mandatory.** `runClaude` / `tsc` / approved-action `bash` use `shAsync` (`spawn`, non-blocking), **not `spawnSync`**. `spawnSync` blocks Node's event loop for the *entire* (long-running, up to the 60-min hard cap) build ‚Üí the 5-lane pool silently collapses to ~1 real lane. Any new long-running step must use `shAsync`.
- **`node_modules` is gitignored** ‚Üí absent in a fresh worktree ‚Üí `tsc` fails. The worker force-symlinks the main clone's `node_modules` into each worktree (`ln -sfn`).
- **`.env.local` is absent on the box** (secrets come from the `EnvironmentFile`) ‚Üí an apply-script that hard-reads it crashes with ENOENT and the migration silently "fails" through the approval gate. The `write-migration` skill now guards the read (`existsSync`) + falls back to `process.env`.
- **Draft-PR trap:** a build that paused (needs_input/needs_approval) opens its PR as a **draft**; on completion the worker now calls `markReady` (GraphQL `markPullRequestReadyForReview`) or it stays unmergeable.

### Hang-based build timeout (build-all-phases-chain Phase 2, shipped 2026-06-22)

A build is killed only when it **hangs**, not on a wall-clock guillotine. The old hard `BUILD_TIMEOUT_MS = 30 min` is **deleted**, replaced by two limits in `scripts/builder-worker.ts`: `BUILD_IDLE_TIMEOUT_MS = 10 min` (no-output idle kill) + `BUILD_HARD_CAP_MS = 60 min` (backstop). `shAsync` gained an **opt-in `idleTimeout`** option ‚Äî a per-chunk idle timer reset on every `stdout`/`stderr` data event (`bumpIdle()`); if the subprocess emits nothing for `idleTimeout` it's hung ‚Üí `SIGKILL` (`killed = "idle"`), while `timeout` stays the absolute hard cap (`killed = "hardcap"`). Only `runClaude` passes `idleTimeout`, so every other `shAsync` caller (tsc, apply-scripts, seed/improve/triage/spec-test runners) is unchanged. For the idle signal to mean anything, `runClaude` switched the build subprocess from `--output-format json` (buffers the whole run, prints once at the end ‚Üí zero liveness) to **`--output-format stream-json --verbose`** (an event per message/tool-use as work happens); the parser reads the newline-delimited events and takes the final `{type:"result"}` for `result`/`session_id`/`is_error`. A hang/cap kill yields no result event ‚Üí an errored build with a `[build killed: ‚Ä¶]` note, so a hung phase never reaches `merged` and the [[../specs/build-all-phases-chain|phase chain]] **stops** rather than barreling on. Net: a phase actively working at 35‚Äì40 min finishes; one silent for >10 min dies fast; nothing runs past 60 min.

**Separate per-kind lanes (beyond the 5-lane build pool):** `MAX_FOLD = 1` (fold-builds), `MAX_SEED = 2` (product-seeds), and `MAX_SPEC_CHAT = 1` (box-spec-chat authoring-chat turns). Each is filled by its own `claim_agent_job(['<kind>'])` call before the build/plan pool, and excluded from the build-pool count, so an interactive chat turn or a long seed never steals a feature-build lane (and vice-versa). The **spec-chat** lane ([[../specs/box-spec-chat]]) runs each authoring-chat turn as a resumable `claude -p` on Max in a **stable per-chat worktree** (`builds/spec-chat-{chat_id}`, recreated on `origin/main` each turn) ‚Äî stable because `claude --resume` is cwd-scoped; concurrency-1 keeps two turns of the same chat off the same dir.

**Recovery / fleet ops:** [[manage-the-build-queue]] (`queue-control`) ‚Äî pause by stopping the worker, `reset-all` to a clean slate, `requeue-stale` after a restart orphans in-flight lanes.

**Interactive Max lanes (no PR).** Beyond the 5 build/plan worktree lanes + the concurrency-1 fold lane, the worker fills five more dedicated lanes each poll tick: `['product-seed']` (`MAX_SEED=2`, [[../specs/box-product-seeding]]), `['ticket-improve']` (`MAX_TICKET_IMPROVE=1`, [[../specs/box-ticket-improve]] ‚Äî one resumable Max turn per ticket, reads brain/`src/` in the main checkout read-only, never mutates / opens a PR), `['triage-escalations']` (`MAX_TRIAGE=1`, [[../specs/box-escalation-triage]] ‚Äî the hourly escalation sweep, `runEscalationTriageJob`), `['spec-test']` (`MAX_SPEC_TEST=1`, [[../specs/spec-test-agent]] ‚Äî the daily QA pass over one shipped-but-unverified spec, `runSpecTestJob`, in the main checkout read-only), and `['migration-fix']` (`MAX_MIGRATION_FIX=1`, [[../specs/migration-fix-agent]] ‚Äî `runMigrationFixJob`, in the main checkout, diagnoses a failed migration read-only then proposes a gated billing repair). All run as top-level `claude -p` on Max (`ANTHROPIC_API_KEY` unset) and keep the DB/crypto secrets so their deterministic CLI tools can reach prod ‚Äî product-seed writes; ticket-improve only reads (via `scripts/improve-box-tools.ts`) and spec-test only reads (via `scripts/spec-test-db-probe.ts`, `gh`, `vercel`, GET hits); **migration-fix reads for the diagnosis but its mutations are gated to the server-side approval path** (the worker runs [[../libraries/migration-fix]] `applyMigrationFix` only on owner approval), with any mutation gated to the server-side approval path.

**Event-fired dirty-PR resolve (dirty-pr-resolver-agent).** Also an **event, not a cron**: the [[../integrations/github-webhook|GitHub webhook]] (`POST /api/webhooks/github`, on a `push` to `main` / a `pull_request` event) detects a dirty (`CONFLICTING`) `claude/*` build PR via [[../libraries/github-pr-resolve]] and enqueues a `kind='pr-resolve'` [[../tables/agent_jobs]] job. The worker claims it on its **concurrency-1 `MAX_PR_RESOLVE=1` lane** (`runPrResolveJob`) and ‚Äî UNLIKE the read-only Max lanes above ‚Äî **does use a throwaway worktree** (`builds/{job-id}` on the PR branch): a `claude -p` on Max (git available, prod secrets stripped ‚Äî **no prod creds, and `GITHUB_TOKEN` stripped so the LLM can't push**) runs `git merge origin/main` + resolves the (usually additive) conflicts. The **worker** then enforces the gate deterministically ‚Äî `git merge-base --is-ancestor origin/main HEAD` (the merge really happened) ¬∑ no `git ls-files -u` (no unmerged paths) ¬∑ no leftover conflict markers ¬∑ **`npx tsc --noEmit`** ‚Äî and only then `git push`es ‚Üí the PR flips `CONFLICTING ‚Üí MERGEABLE`. On a heavy parallel-rewrite divergence or a merge that can't compile it does NOT force a wrong merge: it **rebuilds-on-main** (close the PR + re-queue the originating `build` off a clean `main`, deduped) or ‚Äî if it can't identify the spec ‚Äî **surfaces** "PR #N needs a human merge: {why}" via [[../libraries/notify-ops-alert]] (job ‚Üí `needs_attention`). Capped, idempotent, deduped (one resolve per PR). `claude/*` branches ONLY ‚Äî never a human PR or `main`. No PR of its own.

**Event-fired Repair Agent (repair-agent).** Also an **event, not a cron**: the moment the Control Tower records a NEW problem ([[../libraries/control-tower]] `recordError` on a new [[../tables/error_events]] signature, or `runControlTowerMonitor` on a newly-opened [[../tables/loop_alerts]]), [[../libraries/repair-agent]] `enqueueRepairJob` inserts a `kind='repair'` [[../tables/agent_jobs]] job (deduped by signature). The worker claims it on its **`MAX_REPAIR` (default 2) lane** (`runRepairJob`, main checkout read-only) and runs a `claude -p` on Max (web search on, no `ANTHROPIC_API_KEY`, keeps read-only DB/crypto secrets): INVESTIGATE the signature read-only ‚Üí CLASSIFY (`real-bug`/`monitor-false-positive`/`foreign-app-noise`/`transient`/`needs-human`) ‚Üí ACT ‚Äî author a single-phase fix spec to main + **surface** it for one-tap owner Build (`needs_approval` + a `repair_build` `pending_actions` entry, actioned via `POST /api/developer/control-tower/repair`), or **auto-queue** the build for a narrow mechanical allow-list (`REPAIR_AUTOBUILD_KINDS` = foreign-app-noise + monitor-false-positive), or no-op-**resolve** a transient error row, or **surface needs-human** (`needs_attention`). Surface-don't-auto-build (North star): it NEVER edits product code / opens a PR / applies a migration ‚Äî building stays owner-gated. No PR of its own; deduped by signature.

**Event-fired migration-fix (migration-fix-agent, shipped 2026-06-20).** UNLIKE the cron-enqueued spec-test/triage lanes, the migration-fix lane is fired by an **event, not a cron**: [[../libraries/migration-audit]] `verifyMigration` enqueues a `kind='migration-fix'` [[../tables/agent_jobs]] job the moment a [[../tables/migration_audits]] row transitions to `failed` (a renewal at risk). The worker claims it on the **concurrency-1 `MAX_MIGRATION_FIX=1` lane** and runs the `migration-fix` skill: diagnose the failing checks read-only over a baked-in brief (audit + sub + catalog + engine pricing + the live Appstle contract) ‚Üí propose a typed fix (`price_reconcile`/`variant_backfill`/`appstle_cancel`) parked in `pending_actions`. The owner approves on [[../dashboard/migrations]] (`/api/roadmap/approve` ‚Üí `queued_resume`); the worker executes via [[../libraries/migration-fix]] `applyMigrationFix` then re-runs `verifyMigration` ‚Äî only a re-`passed` clears the row; unfixable (no billable card) stays `failed` with the box's written diagnosis. No PR.

**Spec-review (Vale) lane (spec-review, shipped 2026-06-26).** An **Inngest cron enqueues the sweep**: `enqueueSpecReviewIfDue` ([[../inngest/spec-review-cron]]) inserts one `kind='spec-review'` [[../tables/agent_jobs]] row when specs enter `in_review` state, deduped against an in-flight job. **Serialized ‚Äî one at a time** (`MAX_SPEC_REVIEW=1`), since the cron's own in-flight-dedup enforces single-scan sequencing. The worker claims it on the **concurrency-1 `MAX_SPEC_REVIEW=1` lane** (`runSpecReviewJob`, Vale the spec-reviewer agent) and applies her review checklist (domain, syntax, ownership, cross-links, etc.), stamping a `vale_pass` on the spec row (`approved` / `needs_fix`). **No PR; read-only review**. On `approved`, the board unblocks Build; on `needs_fix`, the spec owner is notified. A safety guard ‚Äî `scripts/_check-worker-lanes.ts` ‚Äî runs in CI/predeploy to verify every `Job.kind` in the union has both a claim lane and a dispatcher entry, so a missing lane (like Vale's was, pre-spec-review-restore) fails red, preventing silent regressions.

**Daily spec-test QA (spec-test-agent, shipped 2026-06-20).** Like triage, an **Inngest cron enqueues the sweep**: `spec-test-cron` ([[../inngest/spec-test-cron]], `45 10 * * *`, concurrency-1) inserts one `kind='spec-test'` [[../tables/agent_jobs]] row per shipped-but-not-archived spec (`deriveStatus==shipped` AND no `archive.d/{slug}.md`), per build-console workspace, deduped against an in-flight job or a fresh (<20h) run. The worker claims each on the **concurrency-1 `MAX_SPEC_TEST=1` lane** and runs the `spec-test` skill ‚Äî classify each `## Verification` bullet (auto / needs-human / mutating‚Üíneeds-human) and run only the **non-destructive** checks (repo/`tsc`, `gh` CI, `vercel` deploy+logs+env, read-only DB probes via `scripts/spec-test-db-probe.ts`, GET endpoints) ‚Äî writing one [[../tables/spec_test_runs]] row with an `agent_verdict` stamp. **No PR; never mutates prod; never marks a spec verified/archived** (that owner-only gate stays untouched). Surfaced on **Developer ‚Üí Spec Tests**, the board card chip, and the spec's VerificationCard.

**Vercel CLI provisioning (spec-test-agent).** The spec-test agent needs the **`vercel` CLI** + a **read-scoped `VERCEL_TOKEN`** to confirm the deploy is READY, read build/runtime logs, and check `vercel env ls`. Install on the box (`npm i -g vercel`) and add `VERCEL_TOKEN` to the worker env (`/root/shopcx-worker.env`, the systemd `EnvironmentFile`) ‚Äî **owner op** (token minting is the owner's; surface it as a gated action if missing). `gh`/`GITHUB_TOKEN` are already present. If `vercel`/`VERCEL_TOKEN` is absent the agent degrades gracefully: it marks Vercel-dependent bullets `inconclusive` (with that reason) rather than failing them.

**Vercel preview builds are SPEC-BUILDS ONLY (`vercel-skip-non-spec-build-refs`).** The worker auto-heals the Vercel project's Ignored-Build-Step to [[../libraries/vercel-project]] `CLAUDE_PREVIEW_IGNORE_COMMAND` on **every tick** (`patchIgnoredBuildStep`, startup + each poll loop). That command whitelists **`^claude/build-` ONLY** ‚Äî so a per-build preview deploys for a SPEC-BUILD branch (`runBuildJob` pushes `claude/build-${slug}-${hash}`, the only lane pre-merge spec-tested against its preview), while EVERY other foreman lane (`claude/fold-*`, `claude/goal-fold-*`, `claude/plan-*`, `claude/spec-chat-*`, `claude/dev-ask-*`, `claude/director-coach-*`) and production-only deploys behave correctly (folds skip; prod always builds). **Consequence for ops:** because the auto-heal re-PATCHes every tick, a bare Vercel-dashboard/API change to the Ignored-Build-Step is **reverted within ~5s** to whatever the RUNNING worker exports. To change it for real, change `CLAUDE_PREVIEW_IGNORE_COMMAND` in code and let the box self-update (or `Queue restart`) onto the new SHA. (History: the original M1 command whitelisted all `claude/*`, which built fold previews; a fold preview FAILS and its failing `Vercel` check left fold PRs `UNSTABLE` ‚Üí the Gate-A auto-merge refused them. The `claude/build-` prefix + `^claude/build-` whitelist fixed it.)

**Playwright / chromium provisioning (spec-test-deep-verification Phase 1).** The spec-test agent's **headless-browser check** (`scripts/spec-test-browser-check.ts`) needs the chromium browser binary + its system libs. Provision it on the box **as root** (owner op): `npx playwright install --with-deps chromium`. The `playwright` npm package is in `package.json` (picked up by the worker's `npm ci` on self-update), so only the browser/system-deps install is a manual root step ‚Äî run it once (and after a major Playwright bump). No new token: the tool mints the **owner Supabase session server-side** from the service-role key already in the worker env (no human creds). Screenshots land in the private **`spec-test-evidence`** Storage bucket, which the tool creates idempotently on first run (`ensureSpecTestEvidenceBucket`). If chromium is missing, browser-check bullets surface as `inconclusive`/error (the launch throws) rather than passing silently.

**Spec-test sandbox fixtures (spec-test-deep-verification Phase 2).** The spec-test agent's **sandbox check** (`scripts/spec-test-sandbox.ts`) drives INTERNAL-ONLY behavioral flows against a dedicated **`is_test` workspace** + test customer / comp subscription / ticket / migration_audit. Two one-time **gated owner ops** (the box has no prod creds ‚Äî surface them as `needs_approval` actions): (1) apply the sentinel-column migration ‚Äî `npx tsx scripts/apply-workspaces-is-test-migration.ts` (adds [[../tables/workspaces]] `is_test`, idempotent); (2) seed the fixtures ‚Äî `npx tsx scripts/seed-spec-test-fixtures.ts` (idempotent upsert on stable UUIDs; re-runnable, also resets fixtures to baseline). The seed adds the owner as an `owner` `workspace_member` of the test tenant so owner-gated POST flows pass when scoped there, and leaves **all credential columns null** so no external API call can fire. No new token: owner-gated POSTs are driven with a server-side-minted owner session (`mintOwnerCookieHeader`, same mechanism as the browser check). The toolkit refuses any non-`is_test` workspace (`assertTestWorkspace`); see [[../libraries/spec-test-sandbox]]. If the fixtures aren't seeded, sandbox-check bullets surface as `inconclusive`/error (the tool throws "run the fixture seed first") rather than passing silently.

**Hourly escalation triage (triage-escalations, shipped 2026-06-20).** The box has no internal ticker, so an **Inngest cron enqueues the sweep** (the [[../inngest/portal-auto-resume]] precedent): `triage-escalations-cron` ([[../inngest/triage-escalations]], `30 * * * *`, concurrency-1) inserts one `kind='triage-escalations'` [[../tables/agent_jobs]] row per workspace with a routine-owned escalated ticket. The worker claims it on the **concurrency-1 `MAX_TRIAGE=1` lane** and sweeps up to `TRIAGE_CAP` (default 5, env `AGENT_TODO_TRIAGE_CAP`) tickets, running a **solver‚Üískeptic‚Üíquorum** loop as 2‚Äì4 separate `claude -p` Max sessions per ticket. On quorum it materializes (customer fix ‚Üí `pending` [[../tables/agent_todos]] ¬∑ rule ‚Üí `proposed` [[../tables/sonnet_prompts]] ¬∑ code/analyzer fix ‚Üí a `docs/brain/specs/` file committed straight to main); every ticket per sweep gets a [[../tables/triage_runs]] audit row. No PR (specs commit via the Contents API like the planner). This **replaced the retired Anthropic-cloud agent-todo routine** ([[../lifecycles/agent-todo-system]]).
