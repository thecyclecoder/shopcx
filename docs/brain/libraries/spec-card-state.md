# libraries/spec-card-state

The read/write layer over the [[../tables/spec_card_state]] table — the live, instant project-management mirror behind the [[../dashboard/roadmap|Roadmap board]] ([[../specs/spec-card-db-companion]]). Supersedes the disabled `roadmap-reads-specs-from-git`.

**File:** `src/lib/spec-card-state.ts`

## Why this exists

The board parses a card's status from the spec markdown's `⏳/🚧/✅` phase emojis **as bundled in the deployed build** ([[brain-roadmap]]). So a status change — a build merging, a [[spec-drift]] flip, an owner mark — didn't show until a markdown edit + commit + Vercel **deploy** (the deploy-lag that made the board feel stale). This module is the DB companion: the merge / drift / owner / build paths write it the moment the event happens, and the board reads it **DB-first** (markdown fallback when no row exists). **No GitHub API calls for status** — the retired git-read approach burned the quota.

**Canonical-source rule:** markdown stays canonical for content + the durable phase record; this is only the board mirror + transient flags. `resolveBoardStatus` takes whichever of (markdown, mirror) is **further along**, so the mirror only ever moves a card forward and a markdown that's ahead wins.

## Types

- **`SpecCardState`** — `{ workspace_id, spec_slug, status: SpecStatus, phase_states: SpecCardPhaseState[], flags: SpecCardFlags, last_merge_sha, updated_at }`. `status` is the whole-spec `SpecStatus` (`Phase | "deferred"`); per-phase `status` stays `Phase`.
- **`SpecCardPhaseState`** — `{ index, title, status: Phase, pr?: number, merge_sha?: string }`. The optional `pr` + `merge_sha` ([[../specs/spec-status-phase-pr-provenance]]) record the PR # + merge commit SHA that SHIPPED this phase — so "shipped" is provable/auditable, not inferred. Stamped by the merge hook ([[agent-jobs]] `applyMergedBuildEffects`) and the one-time backfill (`scripts/backfill-phase-pr-provenance.ts`). Absent on a planned phase.
- **`SpecCardFlags`** — `{ deploy_pending?, blocked?, critical?, deferred?, short_circuit?, short_circuit_reason?, merged_pr?, vale_pass?, vale_review_passed?, ada_disposition?, intended_status?, vale_disposition?, vale_disposition_reason? }` (transient). `merged_pr` is the card-level shipping PR for a **one-shot spec** (zero `## Phase` sections) — multi-phase specs carry their PR per-phase via `phase_states[i].pr` instead. `vale_pass` is Vale's TRANSIENT verdict (consumed by Ada's disposition); **`vale_review_passed`** (build-gate-durable-review-signal) is the DURABLE counterpart that mirrors to `specs.vale_review_passed_at` and is NOT consumed — the claim-time build gate reads it. **`vale_disposition`** + **`vale_disposition_reason`** ([[../specs/vale-reasons-the-disposition]] Phase 1) carry Vale's reasoned planned/deferred recommendation on a PASS (mirror to the same-named `specs` columns); cleared alongside `vale_pass` on send-back.
- **`DeployState`** — `"deploying" ｜ "live"`.

## Exports

- **`getSpecCardStates(workspaceId)`** → `Record<slug, SpecCardState>` — the board's one DB read.
- **`resolveBoardStatus(markdownStatus, state?)`** → `SpecStatus` — forward-merge of the markdown parse and the mirror (DB-first for the deploy-lag; markdown wins when it's already ahead). `rejected` (phase-level) passes through as the markdown value. A **`deferred`** markdown status is markdown-owned ([[../specs/director-drives-all-specs-and-deferred-status]] Phase 1): a deferred spec stays deferred and a once-deferred mirror never overrides an un-deferred (now Planned) markdown — only the CEO removing the marker un-defers it.
- **`deploymentState(state?, markdownStatus, deployedSha)`** → `DeployState | null` — the `shipped · deploying` → `shipped · live` signal. `live` when the deployed `VERCEL_GIT_COMMIT_SHA` **is** the card's `last_merge_sha`, or a later deploy already carries the flipped emoji (`markdownStatus === "shipped"`); else `deploying`. `null` for a card that isn't shipped / has no row / no merge SHA.
- **`markSpecCardStatus(workspaceId, slug, status, phaseStates?)`** — mirror a derived status + per-phase snapshot (drift reconciler, owner flips). No deploy flag. The mirror `spec_card_state.status` keeps the derived value, but the dual-write to [[../tables/specs]] is OVERRIDE-ONLY (see below) — a derived `status` clears `specs.status` to `NULL`.
- **`isOverrideStatus(status)`** → `boolean` — `specs-status-overrides-only` predicate: true iff `status ∈ {deferred, folded}` (the two NON-DERIVABLE lifecycle overrides that may live on `specs.status`). Every other value — derived `planned`/`in_progress`/`shipped`/`rejected`/`in_testing`, AND `in_review` (now derived from the rollup + `vale_review_passed_at`) — must be `NULL` on the column. `dualWriteSpecRow` calls this: a non-override `patch.status` is mapped to `NULL` at the single dual-write choke point every `markSpecCard*` writer funnels through — so `markSpecCardForReview('in_review')` lands as `NULL` and the spec DERIVES `in_review` (fresh row: no phases + `vale_review_passed_at IS NULL`). Kills the stored-`in_review` drift (an all-phases-shipped spec pinned in In Review) and the `noop-pipeline-test-4` stored-`planned` bug.
- **`rollupPhaseStatus(phaseStates)`** → `Phase` — roll the per-phase states up to one board status, driven purely by the phases (never the H1 emoji): all ✅ → `shipped`; any ✅/🚧 but not all → `in_progress`; else `planned`. `rejected` (cut) phases are ignored; an empty set → `planned`. Used by the merge-write (Bug A fix below).
- **`markSpecCardMergeShipped(workspaceId, slug, { status, mergeSha, phaseStates?, prNumber? })`** — mirror a just-merged build: `flags.deploy_pending = true` + `last_merge_sha`, and **status = `rollupPhaseStatus(phaseStates)`** (not the caller's title-derived `status`). chain-and-cardstate-under-automerge **Bug A:** a multi-phase spec whose first phase shipped but whose H1 is still ⏳ derives `planned` from the markdown (the title wins in `deriveStatus`), which parked a part-shipped card in Planned — the phase rollup reads it as `in_progress`. `status` is the fallback only when `phaseStates` is absent/empty (a spec with no parsed phases). [[../specs/spec-status-phase-pr-provenance]] Phase 1: for a one-shot spec (no phases) the `prNumber` is stored on `flags.merged_pr` so the card carries its shipping PR; multi-phase specs ignore `prNumber` here (the per-phase PR is stamped on `phase_states[i].pr` by `applyMergedBuildEffects`).
- **`markSpecCardBlocked(workspaceId, slug, blocked)`** — set/clear the `blocked` transient flag (spec-blockers).
- **`markSpecCardForReview(workspaceId, slug, intendedStatus, audit)`** — the spec-creation entry point ([[../specs/spec-review-agent]] Phase 3). Sets the card to `in_review` AND records the author's `flags.intended_status` (a SUGGESTION for Ada's disposition lane, not binding). Every NEWLY-authored spec-creation surface calls this on first commit. **Dual-write since [[../specs/spec-authoring-writes-db-and-worker-materialize]] Phase 1:** the author's suggestion ALSO lands on the typed [[../tables/specs]] `intended_status` column ([[author-spec]] `authorSpecRowFromMarkdown` → [[specs-table]] `upsertSpec`). The `flags.intended_status` mirror is kept until [[../specs/spec-readers-from-db-retire-parser]] (M3) retires the markdown-first readers, then the typed column is the sole source.
- **`markSpecCardValePassed(workspaceId, slug, audit, disposition?)`** — Vale's pass leg ([[../specs/spec-review-agent]] Phase 3). Sets `flags.vale_pass=true` so [[agents-spec-dispose]]'s `runAdaDispositionSweep` can pick it up. Does NOT flip the status — a passed spec stays in `in_review` until Ada disposes. **build-gate-durable-review-signal** — ALSO sets the DURABLE `flags.vale_review_passed=true` (→ `specs.vale_review_passed_at=now()`); unlike the transient `vale_pass`, this is NOT consumed by Ada's disposition, so the claim-time build gate can still tell at build time that the spec passed review (the old gate read `vale_pass` and deadlocked every disposed spec). **[[../specs/vale-reasons-the-disposition]] Phase 1** — the optional `disposition` argument (`{ disposition: 'planned' | 'deferred', disposition_reason }`) writes Vale's reasoned recommendation onto `flags.vale_disposition` + `flags.vale_disposition_reason` (→ same-named `specs` columns) for Ada's Phase-2 sweep to consume. Omit when the pass carried no recommendation (legacy path — the sweep falls back to `intended_status`).
- **`markSpecCardValeNeedsFix(workspaceId, slug, audit)`** — Vale's needs_fix leg (vale-instant-per-spec-review). Sets `flags.vale_pass=false` (→ `specs.vale_pass=false`), the durable "reviewed → malformed" marker that takes the spec OUT of Vale's queue (`selectUnreviewedInReviewSpecs` = `vale_pass IS NULL`) until `markSpecCardBackToReview` NULLs it on re-author. Does NOT flip status — the spec stays `in_review` (the build hard-stop holds). Ada's disposition lane treats `false` like `null` (`if (!r.vale_pass) continue`), so a needs_fix spec is correctly NOT disposed. Without this marker the ~30s instant-per-spec poll would re-review the same malformed spec every cycle.
- **`applyAdaDisposition(workspaceId, slug, decision, kind, audit)`** — Ada's disposition writer ([[../specs/spec-review-agent]] Phase 3). Flips a Vale-passed `in_review` spec OUT of the override into its final column and CONSUMES the lane flags (`intended_status`, `vale_pass`, `ada_disposition`). **`specs-status-override-only`:** `planned` is a DERIVED state, so the disposition CLEARS the `specs.status` override to `NULL` (the rollup then derives `planned` from the phases) — it NEVER writes the literal `'planned'` (the bug that left `noop-pipeline-test-4` stored as `planned`); `deferred` is a true override and IS stored (via `setSpecStatus`, plus `flags.deferred=true`). The override clear goes straight to [[../tables/specs]] via `setSpecStatus`, NOT through the `spec_card_state.status` mirror (whose CHECK can hold neither `deferred` nor `NULL`). **Does NOT touch `vale_review_passed`** — the durable review-passed stamp survives the disposition (build-gate-durable-review-signal), so the disposed spec still passes the claim-time build gate after `vale_pass` is consumed.
- **`markSpecCardPendingUpgrade(workspaceId, slug, audit)`** — Ada's UPGRADE park ([[../specs/spec-review-agent]] Phase 3). Sets `flags.ada_disposition='pending_upgrade'`; the CEO's approval resolves to either `applyAdaDisposition('planned', …)` or `markSpecCardDeferred(true, …)`.
- **`markSpecCardBackToReview(workspaceId, slug, audit)`** — the SHARED back-to-review writer ([[../specs/spec-review-agent]] Phase 4). Any agent that spots a malformed/off spec (Vale on a re-check, Bo refusing to build on an empty/phaseless body, Ada's `spec-status` action with `status:'in_review'`, repair/regression noticing an authored spec doesn't pass the CHECKLIST, the CEO via the board control) flips the card back to `in_review` so it returns to Vale's queue (the build pipeline refuses an in_review spec). Consumes the prior lane signals: `vale_pass`, `ada_disposition`, `intended_status`, and — [[../specs/vale-reasons-the-disposition]] Phase 1 — `vale_disposition` + `vale_disposition_reason` are cleared so the next Vale + dispose pass start clean — and **build-gate-durable-review-signal** also clears the DURABLE `vale_review_passed` (→ `specs.vale_review_passed_at=null`), so a materially-changed / bounced spec must pass Vale AGAIN before the build gate releases it. Caller is responsible for recording the matching `director_activity` row (`spec_sent_back_to_review`) with their actor. **The claim-time build gate calls this** to route an unreviewed `planned` spec (authored directly, never in_review) INTO review — flip to `in_review` + enqueue a Vale pass — instead of deadlocking it.

All writers are **best-effort** — `upsertCardState` swallows its own error so a mirror-write failure never breaks the underlying merge / flip / build path. `flags` is read-modify-write **merged** (a merge's `deploy_pending` doesn't clobber a `blocked`).

**Dual-write to `public.specs`** ([[../specs/spec-authoring-writes-db-and-worker-materialize]] Phase 3): every `upsertCardState` patch ALSO writes the corresponding typed column on the future-canonical [[../tables/specs]] row (status, deferred, priority, intended_status) via the inline `dualWriteSpecRow` helper. The mirror stays the read-path until [[../specs/spec-readers-from-db-retire-parser]] (M3) cuts readers over; the dual-write keeps the row honest with the mirror so callers reading either source see the same answer. Field mapping: `patch.status → specs.status`, `flags.deferred → specs.deferred` (the `specs_deferred_rollup_trigger` rolls status to `'deferred'` when set, restores phase rollup when unset), `flags.critical → specs.priority` (`'critical'`/null), `flags.intended_status → specs.intended_status` (cleared on Ada's disposition), `flags.vale_pass → specs.vale_pass`, **`flags.vale_review_passed → specs.vale_review_passed_at`** (true → `now()`, false/cleared → null — the DURABLE review-passed stamp; build-gate-durable-review-signal), `flags.ada_disposition → specs.ada_disposition`, `flags.vale_disposition → specs.vale_disposition` (`'planned'`/`'deferred'`/null), `flags.vale_disposition_reason → specs.vale_disposition_reason` ([[../specs/vale-reasons-the-disposition]] Phase 1), `flags.{short_circuit,short_circuit_reason,merged_pr} → specs.{…}`, `last_merge_sha → specs.last_merge_sha`. A flag key with value `undefined` is treated as a CLEAR (writes null/false) — matching the writers' convention of consuming a flag by setting it `undefined`. A 0-row UPDATE (no specs row yet for this slug) is silently fine; the row catches up on the next author/edit pass via [[specs-table]] `upsertSpec`. Per-phase PR provenance dual-writes via [[agent-jobs]] `applyMergedBuildEffects` — the typed [[../tables/spec_phases]] row is the canonical surface for phase progress; `spec_card_state.phase_states[i].{pr,merge_sha}` mirrors it.

## Callers

- **Writers:** [[agent-jobs]] `applyMergedBuildEffects` (the shared merge-write run by both `reconcileMergedJobs` and the auto-merge path `handleAutoMergedBuildBranch`) · [[spec-drift]] `reconcileSpecDrift` (drift flip) · `src/app/api/roadmap/status/route.ts` + `src/app/api/roadmap/spec-drift/route.ts` (owner flips).
- **Reader:** `src/app/dashboard/roadmap/page.tsx` (the board — `getSpecCardStates` + `resolveBoardStatus` + `deploymentState`).
- `phaseStatesFromRaw(raw)` lives in [[spec-drift]] (it already parses phases with line numbers) and feeds the per-phase snapshot to the writers.

## Gotchas

- **`deploy_pending` is cleared at READ time, not by a webhook** — `deploymentState` derives `live` from the SHA compare. The stored flag staying `true` is harmless; a merge whose SHA is already live still shows `shipped · live`.
- **No circular import** — this module imports only `supabase/admin` + the `Phase`/`SpecStatus` *types* from [[brain-roadmap]]; [[spec-drift]] / [[agent-jobs]] / the routes import *it* (one direction).
- **`building` stays on the live-job overlay.** A mid-build card shows In progress via the board's existing active-`agent_jobs` overlay (instant + self-clearing on terminal status), which [[../specs/spec-card-db-companion]] generalizes — this mirror persists the *shipped* progression + deploy flag, the parts that actually had deploy-lag.
