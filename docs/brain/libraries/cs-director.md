## Why this exists

CS's tools already work â€” the ticket-improve Max session, the analyzer/grader, the box-hosted solverâ†’skeptic quorum sweep â€” but nothing supervises them **as a director**. > **Operate + author, never build (CEO directive 2026-06-29).** The CS Director OPERATES its own software (its `function_autonomy` is *operational* autonomy) and AUTHORS specs for the tools it needs. ## Scope (what the CS Director owns)

- **The ticket-improve loop** ([[../specs/box-ticket-improve]]) â€” supervises the customer-action plans the Improve agent proposes, approves the in-leash ones, escalates the rest.
- **The escalation-triage quorum** ([[../specs/box-escalation-triage]]) â€” sits **above** the hourly solverâ†’skepticâ†’quorum sweep in the escalation ladder: a no-quorum disagreement, an ambiguous verdict, or a proposal outside the CS leash routes to the CS Director instead of straight to the CEO.
- **Ticket-derived product specs** â€” a code recommendation from a ticket becomes a `owner=cs`, `Derived-from-ticket:` spec authored on the roadmap; the CS Director owns the authoring quality gate, then hands the build to [[../functions/platform|Ada]].
- **The conversation-rule library** ([[../tables/sonnet_prompts]] + grader-rule proposals) â€” reviews proposed rule changes surfaced by the analyzer/grader before they reach `admin`.

## How it decides (the leash, structural + soundness gates)

Two-gate pattern shared with [[platform-director]] + [[growth-director]]: a **structural** gate (which action class) *and* a **soundness** gate (a read-only investigation â€” *never rubber-stamps*). Concrete leash categories land in the M5 behavior spec; the identity+placement scaffold reserves the seat and the runtime guard.

- **Runtime guard.** Until [[../tables/function_autonomy]] `('cs','cs-director')` is `live + autonomous`, the CS Director is **dormant**: the router never enqueues a `cs-director` job and every downstream surface no-ops. Phase 2 of this spec seeds the `function_autonomy` row at the **safest available leash** ("dormant" / the enum's `off` label) so the seat exists but nothing acts autonomously until the CEO flips it live.
- **Always escalates** (never auto-approves): destructive/irreversible actions, a non-binary multi-CHOICE decision, a customer-refund action that exceeds the CS refund ceiling, a proposed `sonnet_prompts` / `grader_prompts` change with high blast radius, or anything the read-only investigation cannot confirm sound. Escalations route to the CEO via `escalateApprovalRequestToCeo` (the same plumbing Ada + Max use).

### Phase 1 â€” the `cs-director-call` box lane (third rung of the escalation ladder)

The session's brief bakes in:

1. The **ticket** â€” subject / channel / status / escalation reason + full conversation (via `loadTriageBrief`).
2. The **customer, subscriptions, and last 5 orders** â€” the same commerce/* slice the triage lane uses.
3. The **[[../tables/ticket_resolution_events]] ledger** â€” every prior orchestrator turn's `problem` / `confidence` / `verified_outcome` / `reasoning`. 4. The **[[../tables/triage_runs]] row** that dispatched the call â€” solver decision + skeptic verdict + no-quorum reasoning (why the box quorum missed).
5. **Live `sonnet_prompts`** â€” the rules the orchestrator reads every turn (so June sees what the system already tried to enforce).

The session emits ONE JSON object â€” the **verdict**:

- **`decision: 'approve_remedy'`** â€” the right customer-facing fix is clear AND in leash (reversible OR trivially bounded â€” a coupon / a partial refund inside the CS refund ceiling / a sub pause / a resend). Payload: `remedy: RemedyPlan` (action_type / summary / payload / confidence). `applyBoxCsDirectorCall`'s `approve_remedy` handler fires it through `executeSonnetDecision` (the same real executor prod uses; see [[../recipes/run-orchestrator-action]]) and â€” ONLY after the executor returns without escalation â€” delivers the RemedyPlan's `customer_message` via `deliverTicketMessage`. - **`decision: 'author_spec'`** â€” the ledger shows a recurring `drifted` / `unbacked` outcome the code / rules keep failing on; the right fix is code, not a customer-side patch. Payload: `spec_seed: SpecSeed` (`slug` / `title` / `intent` / `problem` / `target?`). `applyBoxCsDirectorCall`'s `author_spec` handler hands it to the [[specs-table]] SDK (`authorSpecRowStructured`) with `owner='cs'` and a `**Derived-from-ticket:** {ticket_id}` header on the summary; the BUILD is always Ada's ([[../functions/platform]] â€” the CEO directive 2026-06-29), so the authored spec lands with `autoBuild: false` and commissions from the Roadmap.
- **`decision: 'escalate_founder'`** â€” a real judgment the CEO must make: destructive / irreversible / out-of-leash / non-binary / storyline-shaped / the read-only investigation could not confirm sound. Payload: `reasoning` (always â€” the 2-4 sentence diagnosis) + `recommended_remedy` (OPTIONAL RemedyPlan-shaped `{kind, summary}` â€” omit only when the call is a policy/storyline judgment with no concrete action to propose). The runner records the verdict to [[../tables/director_activity]] as `{ director_function:'cs', action_kind:'cs_director_call', spec_slug: verdict.spec_seed.slug ?? null, reason: verdict.reasoning, metadata:{ job_id, ticket_id, triage_run_id, decision, remedy, spec_seed, recommended_remedy, autonomous:true, phase:2 } }` â€” the audit trail the CEO / recap / #directors board sees WHAT the CS Director decided and WHY. `autonomous:true` / `phase:2` reflect that the runner then IMMEDIATELY hands the SAME normalized verdict to the Phase-2 executor `applyBoxCsDirectorCall` (in [`src/lib/cs-director.ts`](../../../src/lib/cs-director.ts)) â€” no CEO wait, no manual re-drive.

### Phase-2 executor â€” `applyBoxCsDirectorCall`

[[../specs/cs-director-call-phase-2-executor-fires-june-verdicts]] (all four phases shipped) â€” the deterministic mutator in [`src/lib/cs-director.ts`](../../../src/lib/cs-director.ts) that materializes a recorded verdict into a real action. Same shape as `applyBoxDeployReview` in [[deploy-guardian]] (Reva's Phase-3 mutator): the box session decides read-only + returns a typed verdict; this writer routes it to the per-decision handler. Called once per `kind='cs-director-call'` job by `runCsDirectorCallJob` (scripts/builder-worker.ts) IMMEDIATELY after the `director_activity` audit row above â€” the mutator sees the SAME normalized verdict the audit trail carries.

The derived-from state â€” ticket `115350d5` (portal changedate escalation), where June ruled `approve_remedy: change_next_date -> 2026-10-06` at 06:35 and NOTHING fired until a human ran it by hand â€” is the concrete failure this executor closes.

**Verdict â†’ handler routing:**

| `verdict.decision`   | Handler in `cs-director.ts`   | What it writes                                                                                                                                                             |
|----------------------|-------------------------------|-----------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| `approve_remedy`     | `handleApproveRemedy`         | Fires `verdict.remedy.action_type` via `executeSonnetDecision`, THEN delivers `verdict.remedy.customer_message` via `deliverTicketMessage` â€” only on a clean executor return |
| `author_spec`        | `handleAuthorSpec`            | `authorSpecRowStructured` (the [[specs-table]] SDK) with `owner:'cs'`, `autoBuild:false`, `**Derived-from-ticket:** {ticket_id}` in the summary                             |
| `escalate_founder`   | `handleEscalateFounder`       | Resolves `ticket_id` + `triage_run_id` from `job.instructions` and returns them as the LINKAGE-BACK payload â€” the runner is the SOLE writer of the CEO card (single-writer)  |
| anything else        | logged no-op (`ok:true`)      | `normalizeCsDirectorVerdict` already defends against shape drift; a decision that leaks past the guard never crashes the runner                                              |

**Execute-then-message rule** (the derived-from ticket's whole failure mode â€” never promise a fix you didn't ship). The `approve_remedy` handler enforces ordering explicitly:

1. Builds the `SonnetDecision` with `action_type:'direct_action'` and NO `response_message` â€” the executor has nothing to deliver via its own send path.
2. Passes a NO-OP `send` fn so any executor-internal message (verify-failure / escalate holding text) is SUPPRESSED.
3. Only AFTER `executeSonnetDecision` returns without escalation, calls `deliverTicketMessage` OURSELVES with the RemedyPlan's `customer_message`.

A failed action never sends a customer message â€” `handleApproveRemedy` returns `needs_attention:true` and the runner parks the job so a human eyeballs the `log_tail`. Failure paths: missing/malformed remedy Â· `ticket_id` unresolvable from `job.instructions` Â· ticket has no `customer_id` Â· executor threw Â· executor `escalated=true` on return Â· delivery threw after a successful action.

**Escalate-founder â†’ CEO inbox** (single-writer principle). The runner mints the CEO `dashboard_notifications` card (see Â§ [Escalate to founder â†’ CEO card contract](#escalate-to-founder--ceo-card-contract) below) â€” `handleEscalateFounder` NEVER mints a second card (a duplicate would page the CEO twice). Its Phase-3 role is to RESOLVE the linkage-back payload (`linkage_ticket_id` + `linkage_triage_run_id` from `job.instructions`) and surface it on the result so the runner's `log_tail` names the linkage in a machine-readable form a bounce-back handler / audit join can pick up without re-parsing the CEO card's metadata JSON.

**Author-spec** goes through the specs SDK chokepoint (`authorSpecRowStructured`) â€” NEVER a raw `.from('specs').insert` per CLAUDE.md Â§ "PM data WRITES go through the specs-table SDK" (enforced by `_check-pm-sdk-compliance.ts`). The summary carries `**Derived-from-ticket:** {ticket_id}` as the first line so a Roadmap reader can trace the spec to the ticket in one grep. `autoBuild:false` per the 2026-06-29 CEO directive â€” Ada builds every spec, all functions; a director-authored spec commissions on the Roadmap.

**Never throws.** The executor returns a structured `ApplyBoxCsDirectorCallResult` (`{ ok, handler, reason?, needs_attention?, error?, message_delivered?, spec_slug?, linkage_ticket_id?, linkage_triage_run_id? }`) â€” the runner logs it on `log_tail` and decides `needs_attention` vs `completed` from the flag. ### Loop closure â€” internal note + ticket state per verdict

Every verdict CLOSES THE TICKET LOOP so a ruled-on ticket is never left in the `open+escalated+no-owner` limbo (spec [[../specs/cs-director-call-closes-the-ticket-loop-note-and-resolution-per-verdict]] Phase 2 invariant). 1. **Internal note.** `runCsDirectorCallJob` writes ONE internal `ticket_messages` row (`visibility='internal'`, `author_type='system'`) naming June as the reviewer, the decision, the 2â€“4-sentence reasoning, and the concrete per-verdict output â€” the authored spec slug for `author_spec`, the RemedyPlan summary for `approve_remedy`, the founder-escalation reason for `escalate_founder`. ### Tier-ladder-before-escalation rule (out-of-policy refunds/returns)

[[../specs/cs-director-treats-tier-eligible-out-of-policy-refund-as-playbook-offer-not-escalation]] pins the hard rule: **an out-of-policy refund/return from a customer who clears a playbook exception tier (with no active disqualifier) is a Tier-1/Tier-2 SAVE routed through the playbook's `offer_exception` step â€” NOT a founder escalation.** June MUST evaluate the matched playbook's exception ladder BEFORE emitting `escalate_founder` on any out-of-policy refund/return; escalating a tier-eligible customer wastes a sanctioned save the playbook was designed to make and burns customer trust while the ladder still had a legal move.

The rule is enforced at three points (spec, brief, rubric) so it can't drift:

1. **In the brief** (Phase 1). `runCsDirectorCallJob` (scripts/builder-worker.ts) calls [[cs-director-playbook-tier-eligibility]] `loadPlaybookTierEligibility` for the ticket's `customer_id` and bakes the resulting `PLAYBOOK EXCEPTION-TIER ELIGIBILITY` section into June's prompt. Per playbook it lists each matched (`âś…`) / unmatched (`âś—`) tier with the concrete condition (`Tier 1 "Return for Store Credit" â†’ store_credit_return â€” LTV â‰Ą $300.00 OR orders â‰Ą 3 Â· LTV $1569.00 â‰Ą $300.00`), plus the disqualifier state (`previous_exception`, `has_chargeback`, `has_chargeback_on_order`), and a single `eligible_for_offer` flag per playbook â€” TRUE only when the customer clears â‰Ą1 tier AND no disqualifier is active. Thresholds + disqualifier types are pulled from the `playbook` / `playbook_exceptions` rows VERBATIM (never hardcoded); the pure evaluator is unit-tested against a fixture mirroring ticket `87ce35a1`.
2. **In the prompt** (Phase 1). The `csDirectorCallPrompt` block names the RULE explicitly: BEFORE emitting `escalate_founder` on an out-of-policy refund/return, consult the tier-eligibility section; if any matching playbook shows `eligible_for_offer=true`, return `approve_remedy` routing back into the playbook's `offer_exception` step (the Tier-1 `store_credit_return` or Tier-2 `refund_return` sanctioned save) and cite the specific tier (`Tier N "<exception_name>" â†’ <resolution_type>`) in `reasoning`. `escalate_founder` is reserved for: clears NO tier, a disqualifier applies, or a genuine policy/authority gap the playbook can't resolve.
3. **In the CEO's grading rubric** (Phase 2). [[director-grader]] `buildDirectorGraderSystemPrompt` dimension `cs_director_call` REWARDS an `approve_remedy` routing a tier-eligible refund back into `offer_exception` when the verdict cites the specific tier by name (scores HIGH on soundness) and PENALIZES an `escalate_founder` on an out-of-policy refund/return the brief showed `eligible_for_offer=true` on (scores LOW; reasoning MUST cite the missed tier). The anti-Goodhart cascade explicitly cuts BOTH directions â€” a CS Director that escalates a call the sanctioned tier ladder was designed to save is failing the same objective as one that refund-everyones. The over-correction guard: no penalty when the customer clears NO tier, a disqualifier applies, or the call is a genuine out-of-leash / storyline / precedent judgment (that IS what the third rung exists for).

**Motivating case â€” ticket `87ce35a1`.** June originally escalated an out-of-policy renewal-refund on a customer with LTV $1,569 / 19 orders / no disqualifier to the founder â€” even though the Refund playbook's Tier-1 `store_credit_return` + Tier-2 `refund_return` were DESIGNED to save exactly that high-LTV / tenured customer. See [[../playbooks/refund]] for the tier definitions in the operator's voice.

**What DOESN'T change.** `escalate_founder` remains the right verdict when the customer clears NO tier, a disqualifier fires (`previous_exception`, `has_chargeback`, `has_chargeback_on_order`), OR the call is a genuine out-of-leash / storyline / precedent judgment (full refund past the CS ceiling, cancel-with-refund on a legacy sub, identity merge, changing a `sonnet_prompts` rule, comping a promoter). The rule narrows a specific miss (sanctioned tier-eligible saves), never the third rung's real judgment surface.
