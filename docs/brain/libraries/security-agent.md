# libraries/security-agent

The queue plumbing + autonomy policy behind the **Security / Dependency Agent** box worker ([[../specs/security-dependency-agent]]) — a worker the [[../specs/platform-director-agent|Platform/DevOps Director]] supervises. Persona: **Vault — Security Guardian** (🔒, [[agent-personas]]). It is **the supervisor on the auto-merge proxy**: auto-merge optimizes "ship the fix"; its degenerate state is shipping a fix that introduces an injection / secret-leak / authz hole. Vault gives **every merged diff an autonomous security pass** and **watches dependencies for CVEs** — review read-only + author/surface a fix, **never auto-mutate**.

**File:** `src/lib/security-agent.ts` · the box runner is `scripts/builder-worker.ts` `runSecurityReviewJob` (see [[../recipes/build-box-setup]]).

## North star — review + escalate; the owner-gated build disposes

The agent **reviews + classifies** (a bounded proxy: "did this diff/dep introduce a vulnerability + here's the fix"); the **owner-gated build applies any fix** and is graded on whether it held ([[../operational-rules]] § North star). It NEVER edits product code, opens its own PR, or bumps a dependency. Mirrors [[repair-agent]] / [[regression-agent]]: diagnose + author/surface, the disposer builds. Ambiguity escalates (`needs-human`). Every detect/escalate/author writes a [[../tables/director_activity|director_activity]] row (`director_function: 'platform'`). **Secret findings are referenced by location, never echoed in plaintext** into logs / `director_activity` / the surfaced spec.

## Three modes (one lane, `agent_jobs.kind='security-review'`)

- **`diff`** (post-merge) — a **per-merged-diff** security pass. Reviews the merged `claude/*` commit for injection, secret/credential leak, authz / RLS-policy regressions, unsafe `createAdminClient()` exposure, and `_encrypted` handling (the [[../specs/security-dependency-agent]] / `/security-review` checklist + CLAUDE.md invariants).
- **`branch`** (pre-merge — [[../specs/archive.d/security-test-on-preview-pre-merge]] Phase 1) — the **per-branch** security pass that runs WHILE the build is still unmerged. Same checks as `diff`, but the box reads the UNMERGED diff (`git diff main...claude/<branch>`) and any runtime probe lands on the **per-build Vercel preview origin** (from [[archive.d/per-build-vercel-preview-deploys]]), not prod. The terminal verdict feeds the per-branch clean signal (`isSecurityGreenForBranch` / `getSecurityStateForBranch` — Phase 3) the **M4 promote gate** reads to refuse to auto-merge an unreviewed branch. The clean definition is the SAME `completedClean` rule the post-ship fold gate uses (`getSecurityStateBySlug`), so the two gates can never disagree. **fused-premerge-security-authoritative-drop-standalone Phase 2 (supersedes isolate-premerge-security-verdict Phase 1) — the fused spec-test session is now AUTHORITATIVE for pre-merge branch mode:** `runSpecTestJob` emits BOTH the spec-test verdict AND a structured, evidence-backed security envelope in ONE Max session (fused; halves pre-merge Max sessions + skips a full cold diff re-hydration). Phase 1's pure validator ([[../src/lib/security-envelope]] `classifyFusedSecurityEnvelope`) enforces per-check evidence — a bare/self-declared `clean` DOWNGRADES to `needs_human` (advisory — NEVER an auto-clean gate pass), so a rubber-stamp strands the PR for a human, exactly as a missing dedicated review would. Phase 2 wiring (`applyFusedSecurityAsBranchVerdict` in [[../../../scripts/builder-worker]]) INSERTS a synthetic `security-review` `agent_jobs` row for the branch (kind='security-review', spec_branch=branch, mode='branch', pre-claimed via status='claimed' so the standalone `claim_agent_job` RPC never picks it), then reuses `applySecurityVerdictToJob` to record the terminal state — `isSecurityGreenForBranch` / `getSecurityStateForBranch` resolve it exactly like a standalone row. Only a clean-with-evidence classification lands a `completed` row that satisfies the M4 promote gate; any Phase-1 downgrade or finding blocks it. The standalone `runSecurityReviewJob` still runs for `diff` (post-merge) + `dep-watch` — only branch-mode is fused.
- **`dep-watch`** — the daily **`npm audit`** CVE scan (enqueued by [[../inngest/security-dep-watch]]). On a vulnerable dep it authors a scoped upgrade-fix spec + surfaces a Build card — never auto-bumps.

## Exports

- `type SecurityVerdict = "real-vuln" | "needs-human" | "false-positive"` — the box's per-diff classification (`clean` is a no-action terminal alongside `false-positive`).
- `SECURITY_DIRECTOR_FUNCTION = "platform"` — the function whose objective supervises this worker.
- `SECURITY_DEP_UPGRADE_SLUG = "security-dep-upgrades"` — the stable slug of the dep-upgrade fix spec (find-or-update, never N of them).
- `SECURITY_DEP_WATCH_SLUG = "security-dep-watch"` — the sentinel `spec_slug` a dep-watch scan job carries (so it dedups distinctly from per-diff jobs).
- `SECURITY_RECENT_FIX_WINDOW_MS` (24h) — a fix authored within this window is "pending deploy"; don't re-scan/re-surface.
- `LIVE_SECURITY_STATUSES` — the statuses that mean a security-review job is still live (working or surfaced).
- `securitySha12(mergeSha)` / `depFindingSignature(findings)` — the per-diff dedup key (the merge SHA) and the advisory-set signature (sorted `pkg@severity`, sha1).
- `enqueueSecurityReviewJob(admin, input)` → enqueue ONE `security-review` [[agent_jobs]] job. **Best-effort, never throws.** Two input shapes (one function — discriminated by the presence of `branch`):
  - **post-merge `diff`** `{ mergeSha, specSlug, prNumber?, workspaceId? }` — **deduped by the merge SHA** so a re-run (the two merge-hook paths firing for one merge) never double-files.
  - **pre-merge `branch`** `{ branch, previewOrigin, specSlug, prNumber?, workspaceId? }` ([[../specs/security-test-on-preview-pre-merge]] Phase 1) — **two-part dedup** (vault-security-review-loop-fix): (1) **ONE OPEN review per branch** (any live/surfaced security-review job with `spec_branch === branch`) AND (2) **ONE CLEAN review per UNCHANGED branch** — a `completed` review for the branch blocks re-enqueue UNLESS a newer build-job push landed since (compare the latest `completed` review's `created_at` vs the latest `build` job's `updated_at` on the branch). A `failed` review never blocks (no verdict produced). This stopped the **Vault re-review loop**: the standing-pass backstop re-fired branch-mode security every pass and the old dedup (live-only) never saw the prior CLEAN review, so a fully-built branch got a fresh security-review every ~5 min forever. We re-review only when the branch's diff actually advanced. The `securitySha12`/`depFindingSignature` signatures still converge same-finding recurrences inside the box's review of the branch's diff.
- `enqueueDepWatchJob(admin, { workspaceId? })` → enqueue the daily dep-watch scan. Deduped to ≤1 live scan + a recent-window guard. **Best-effort, never throws.**
- `enqueueSecurityDiffIfDue(admin, { sinceMs?, workspaceId? })` → `{ enqueued, scanned, resolved }` — the STANDING-PASS **backstop for the post-merge `diff` pass** ([[../specs/archive.d/fix-vault-post-merge-diff-backstop-7fbde0]]). Symmetric with `backstopPreMergeChecks` on the pre-merge leg: the reactive `enqueueSecurityReviewJob` inside `applyMergedBuildEffects` is fire-and-forget (Vercel reap / box down / transient error drops the send → the merged commit never gets its `diff` review). This walks `spec_status_history` for `actor='merge:<sha>'` rows in the last 14d — the **audit-authoritative source** that survives `fold` (append-only, never deleted; the SAME source `audit-spec-shipped-state` walks to re-stamp `phase_states` provenance) — and idempotently calls `enqueueSecurityReviewJob` for each unique SHA. **Why history + not `getSpec()`/`spec_phases.merge_sha` (fix-vault-post-merge-diff-backstop-7fbde0-0845a3 regression fix):** the earlier iteration resolved SHAs by iterating `agent_jobs kind='build' status='merged'` + `getSpec` per row; that gapped on FOLDED specs (their brain pages moved to `docs/brain/archive.d/`) — the observed orphans included `acquisition-research-hub` and `ad-creative-scout`, both folded, whose `spec_phases.merge_sha` was NULL or the `getSpec` `(workspace_id, slug)` lookup was stale. History rows were written by the SAME hook (`markSpecCardMergeShipped` → `upsertCardState`) that stamped the phases, so every SHA the phase-provenance path could resolve, the history can too — plus the ones fold has since made unreachable. Idempotency piggybacks on `enqueueSecurityReviewDiff`'s 14d SHA dedup — safe to run every pass. **Best-effort, never throws.**
- `getOpenSecurityReviews(admin, workspaceId)` → `SecuritySurfaceItem[]` — READ-ONLY: open security items awaiting the owner (`needs_approval` = a routed fix with a `security_build` Build action, or `needs_attention` = needs-human). Clean reviews complete silently.
- `getSecurityStateBySlug(admin, workspaceId)` → `Record<slug, SecurityStateBySlug>` — READ-ONLY per-spec rollup (`live`/`surfaced`/`completedClean`) for the build-card lifecycle timeline's Security node + the fold gate (they MUST agree). Excludes infra dep-watch jobs by **identity** (the `security-dep-watch` sentinel slug OR a `dep-watch`-MODE row), NOT by slug name — the dep-watch agent's authored `security-dep-upgrades` fix spec is a REAL shippable spec whose `diff`/`branch` reviews DO roll up (excluding that slug blanked its Security node forever, so its fold gate could never clear).
- `getSecurityStateForBranch(admin, branch)` → `SecurityStateForBranch` ([[../specs/archive.d/security-test-on-preview-pre-merge]] Phase 3) — READ-ONLY **per-branch** rollup keyed to `spec_branch === branch` (the pre-merge `branch`-mode jobs from Phase 1). Same shape + same `completedClean` definition as `getSecurityStateBySlug` (a `completed` job exists AND no live/surfaced sibling) so the **M4 pre-merge promote gate** and the post-ship fold gate ([[../specs/build-card-lifecycle-timeline]] Phase 3 / [[spec-test-runs]] `getAutoFoldEligibleSlugs`) can never disagree on what "security green" means. A branch with no security-review record yet returns all-false (absence ≠ clean, same as the post-ship gate).
- `isSecurityGreenForBranch(admin, branch)` → `boolean` ([[../specs/security-test-on-preview-pre-merge]] Phase 3) — convenience boolean the **M4 pre-merge promote gate** reads. Green iff `getSecurityStateForBranch(...).completedClean === true` — the SAME `completedClean` definition `getSecurityStateBySlug` uses, so the pre-merge gate and the post-ship fold gate stay in lockstep.
- `listSecurityReviews(admin, workspaceId, limit?)` → `SecurityReviewLogItem[]` — READ-ONLY: Vault's **full** review log (clean ones included), newest-first, for the [[../dashboard/security-tests]] surface. Derives a `verdict` per row (`clean｜false-positive｜real-vuln｜needs-human｜running｜failed` — a completed review's verdict is parsed off the `log_tail` prefix) and resolves each reviewed slug's [[../tables/specs|spec]] title. Bounded (≤500).
- `countOpenSecurityReviews(admin, workspaceId)` → `number` — the surfaced count (`needs_approval`/`needs_attention`) for the Security tests sidebar badge. Clean reviews never count.

## Trigger — event-driven (`diff` + `branch`) + a daily cron (`dep-watch`)

- **`diff`** rides the **merge hook**: [[agent-jobs]] `applyMergedBuildEffects` (the shared body of BOTH `reconcileMergedJobs` — a manual squash-merge — and `handleAutoMergedBuildBranch` — the auto-merge webhook) calls `enqueueSecurityReviewJob` with the merge SHA. So every merged `claude/*` build diff gets exactly one review, deduped by SHA. **Backstop ([[../specs/vault-post-merge-diff-backstop]] Phase 1 + [[../specs/fix-vault-post-merge-diff-backstop-7fbde0]] cheap if-due cron backstop):** the merge-hook send is fire-and-forget — if it's dropped (a Vercel deploy reaps the Inngest sync mid-flight, the box is down, a transient error) the merged commit never gets its post-merge security pass. Three nets close the gap: (1) the cheap **[[../inngest/security-diff-backstop-cron]]** runs the sweep every 15 min — the PRIMARY backstop, so an orphan window never exceeds ~15 min (the 7fbde0 pre-merge spec-test observed orphan_count=45 across ~14d because the pre-existing 24h daily leg was the shortest cadence — this cron shrinks that to 15 min); (2) the [[../../../scripts/builder-worker]] platform-director **standing pass** runs it every beat next to `backstopPreMergeChecks`; (3) the daily [[../inngest/security-dep-watch]] cron hangs it off as a final net so a full Inngest / box outage that takes both above out still can't leave merged commits unreviewed indefinitely. All three call `enqueueSecurityDiffIfDue`, which walks `spec_status_history` for `actor='merge:<sha>'` rows in the 14d window (audit-authoritative, survives fold) and (idempotently, via the existing 14d SHA dedup) re-fires the diff-mode enqueue for any SHA lacking a security-review row. This mirrors [[spec-test-runs]] `enqueueSpecTestIfDue` for the same dropped-event class of gap: the diff leg is now **reactive + backstopped**, matching the branch leg.
- **`branch`** (pre-merge) — **fused into the spec-test session (authoritative)** via `applyFusedSecurityAsBranchVerdict` in [[../../../scripts/builder-worker]] ([[../specs/fused-premerge-security-authoritative-drop-standalone]] Phase 2). The pre-merge `runSpecTestJob` emits BOTH the spec-test verdict AND a structured per-check security envelope in ONE Max session; on completion, the worker classifies the envelope with `classifyFusedSecurityEnvelope` ([[../src/lib/security-envelope]]), INSERTS a synthetic `security-review` job row for the branch (status='claimed' so the standalone RPC never picks it), and reuses `applySecurityVerdictToJob` to apply the terminal verdict — the row is INDISTINGUISHABLE to `isSecurityGreenForBranch` from a standalone-produced row. The standalone `maybeEnqueuePreMergeSecurityOnAccumulation` enqueue path (Phase 3 will retire it) is deprecated for branch mode; the fused row is authoritative. Because the Phase-1 validator downgrades a bare/self-declared `clean` to `needs_human`, a rubber-stamp CANNOT satisfy the M4 promote gate — only a clean-with-per-check-evidence envelope lands a `completed` row. **Dedup (branch mode, `enqueueSecurityReviewJob`), in order:** **(0) MERGED-BRANCH guard** (post-merge backstop loop fix) — if the latest `build` job for the branch is `status='merged'`, the `claude/build-*` branch is DELETED → refuse (a pre-merge review of a gone branch is meaningless and was the #837 loop: the post-merge state-advance bumps the build job's `updated_at`, which the step-(2) "branch changed" test mistook for a new push → re-enqueued every standing pass); **(1)** one OPEN review per branch (a live review still in flight → no-op); **(2)** one CLEAN review per UNCHANGED branch — a `completed` review blocks re-enqueue unless a **genuine new build push** landed since (compared against the latest NON-`merged` build job's `updated_at`, so a status-only merge flip never counts as a push). **Backstop (Gate-A class gap fix):** the preview-ready callback is a fire-and-forget poll; if it misses READY (worker restart, slow preview), the branch enqueue would never fire — so the [[../../../scripts/builder-worker]] platform-director **standing pass** also runs [[agent-jobs]] `backstopPreMergeChecks`, which re-evaluates every READY-preview `claude/build-*` branch each pass and (idempotently) re-enqueues BOTH pre-merge triggers (spec-test + security). Before this wiring landed, the branch-mode enqueue had **no caller at all** — `isSecurityGreenForBranch` was always false and the M4 tests gate could never pass. **In-flight-only filter (vault-security-review-loop-fix + post-merge backstop loop fix):** the backstop now SKIPS any branch whose spec is **MERGED** (latest `build` job `merged` OR a merged sibling via `findMergedSiblingBuild` — branch gone), **shipped/folded** (DB `specs.status`), or **archived** (in `listArchivedSlugs`). The merge-signal skip is required on top of the status skip because the post-merge-ships-only-one-phase bug left a merged spec reading `in_progress` (only one phase stamped) — so the status check never fired and security re-enqueued forever (#837). Combined with the branch-mode MERGED-BRANCH guard + CLEAN-review dedup above, an in-flight branch gets exactly ONE pre-merge security review per branch state, and a merged/shipped/folded spec gets NONE.
- **`dep-watch`** is the [[../inngest/security-dep-watch]] daily cron → `enqueueDepWatchJob`.

## The box loop — `runSecurityReviewJob`

1. **Disposer resume** — a routed `security_build` action approved (queue the build) / declined (dismiss).
2. **`dep-watch` mode** — run `npm audit --json` on the real tree; collect ≥ moderate advisories; on a finding author/refresh `security-dep-upgrades` to main + route; clean tree → complete silently with a healthy beat.
3. **`diff` mode** (post-merge) — `git fetch origin main` + `git show {sha}` read-only on Max → ONE verdict (see VERDICT TAIL below).
4. **`branch` mode** (pre-merge — [[../specs/archive.d/security-test-on-preview-pre-merge]] Phase 2) — `git fetch origin main` + `git fetch origin {branch}` + `git diff origin/main...origin/{branch}` read-only on Max; any runtime probe targets the per-build `preview_origin` from [[archive.d/per-build-vercel-preview-deploys]], **NOT prod**. Same review checks + verdict envelope as `diff` mode (the prompt shares one `securityReviewChecksAndVerdictBlock`), so Vault's contract is identical regardless of when she runs. On `real-vuln` the authored fix spec records `Fixes: [[parent-slug]]` + `**Security-of-branch:** \`{branch}\` · preview \`{previewOrigin}\`` (vs `**Security-of-merge:**` for `diff`) so a glance shows which mode caught it. The terminal state (`completed` clean / `needs_approval` routed-fix / `needs_attention` needs-human / `failed`) is what the **M4 promote gate** reads to refuse to auto-merge an unreviewed branch (Phase 3's `completedClean` signal).

**VERDICT TAIL** (shared between `diff` + `branch`):
   - `clean` / `false-positive` → complete, no action.
   - `needs-human` → `needs_attention` + an `escalated` activity row, no spec. The activity reason cites the mode-appropriate context (`merged {slug} (commit {sha})` for diff · `unmerged {slug} (branch {branch})` for branch).
   - `real-vuln` → **author the fix spec to main** (`authorSecurityFixSpec(..., source)` where `source = {kind:'diff', mergeSha}` or `{kind:'branch', branch, previewOrigin}`; `Fixes: [[parent-slug]]`), then `routeSecurityFix` via `resolveApproverLive('platform')` ([[approval-router]]): a live+autonomous director auto-queues the build within its leash; else surface a `security_build` Build card for the **CEO inbox** (the generic [[../specs/approval-routing-engine|approval-inbox]] reconciler emits the routed request — no per-agent route needed). Records `authored_fix`.
   - **Unparseable/unrecognized verdict** ([[../specs/needs-attention-triage-and-verdict-robustness]] Phase 2 + [[../specs/iteration-ingest-async-reports-fix-tooling-69594a]] + [[../specs/ada-director-spec-status-cards-fix-tooling-fa6848]]) → the shared `resolveReviewVerdict` helper (shared with [[repair-agent]] + [[regression-agent]]) first attempts a **same-session JSON parse-repair re-prompt** (`securityReviewRepairPrompt`) — the model already spent the budget doing real review work, so re-emitting ONLY the verdict envelope from the existing context recovers a verdict far cheaper + far more often than a fresh re-investigation. Two tooling guards underneath the repair (added when fa6848a7 parked despite the parse-repair already shipping): (a) `shAsync` closes stdin (`stdio: ["ignore", …]`) so the claude CLI doesn't burn 3s on its "no stdin data received" warning (the CLI's own recommended `< /dev/null` fix, in-process) — that warning was visible on every parked tooling_failure (fa6848a7, 69594acf); (b) `extractJson` now walks every fenced ```` ```json ```` block (last-wins) and scans the LAST balanced `{…}` (right-to-left close × left-to-right open), so a verdict envelope buried under prose-braces or an earlier example JSON parses correctly. If the parse-repair still fails (or no prior session exists), it fail-safes to an **actionable** `needs_attention` reason — `"security review produced no parseable verdict after 2 attempts — re-run or review manually: <excerpt>"` — **never** a bare "ended without a recognizable verdict", never an auto-pass. The director's `reconcileNeedsAttention` ([[platform-director]] Phase 1) then triages that parked item.

## Why npm audit runs on the box, not in the cron

`npm audit` needs the npm CLI + the committed lockfile + registry access — none reliably present in the Vercel/Inngest serverless runtime. So [[../inngest/security-dep-watch]] is the SCHEDULER (deduped, heartbeat) and the box (which has the full repo + npm) runs the actual scan. The observable behaviour matches the spec: a daily watch → an upgrade-fix spec + Build card on a finding, a healthy beat on a clean tree.
