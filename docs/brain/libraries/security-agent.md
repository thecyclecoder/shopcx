# libraries/security-agent

The queue plumbing + autonomy policy behind the **Security / Dependency Agent** box worker ([[../specs/security-dependency-agent]]) тАФ a worker the [[../specs/platform-director-agent|Platform/DevOps Director]] supervises. Persona: **Vault тАФ Security Guardian** (ЁЯФТ, [[agent-personas]]). It is **the supervisor on the auto-merge proxy**: auto-merge optimizes "ship the fix"; its degenerate state is shipping a fix that introduces an injection / secret-leak / authz hole. Vault gives **every merged diff an autonomous security pass** and **watches dependencies for CVEs** тАФ review read-only + author/surface a fix, **never auto-mutate**.

**File:** `src/lib/security-agent.ts` ┬╖ the box runner is `scripts/builder-worker.ts` `runSecurityReviewJob` (see [[../recipes/build-box-setup]]).

## North star тАФ review + escalate; the owner-gated build disposes

The agent **reviews + classifies** (a bounded proxy: "did this diff/dep introduce a vulnerability + here's the fix"); the **owner-gated build applies any fix** and is graded on whether it held ([[../operational-rules]] ┬з North star). It NEVER edits product code, opens its own PR, or bumps a dependency. Mirrors [[repair-agent]] / [[regression-agent]]: diagnose + author/surface, the disposer builds. Ambiguity escalates (`needs-human`). Every detect/escalate/author writes a [[../tables/director_activity|director_activity]] row (`director_function: 'platform'`). **Secret findings are referenced by location, never echoed in plaintext** into logs / `director_activity` / the surfaced spec.

## Three modes (one lane, `agent_jobs.kind='security-review'`)

- **`diff`** (post-merge) тАФ a **per-merged-diff** security pass. Reviews the merged `claude/*` commit for injection, secret/credential leak, authz / RLS-policy regressions, unsafe `createAdminClient()` exposure, and `_encrypted` handling (the [[../specs/security-dependency-agent]] / `/security-review` checklist + CLAUDE.md invariants).
- **`branch`** (pre-merge тАФ [[../specs/archive.d/security-test-on-preview-pre-merge]] Phase 1) тАФ the **per-branch** security pass that runs WHILE the build is still unmerged. Same checks as `diff`, but the box reads the UNMERGED diff (`git diff main...claude/<branch>`) and any runtime probe lands on the **per-build Vercel preview origin** (from [[archive.d/per-build-vercel-preview-deploys]]), not prod. The terminal verdict feeds the per-branch clean signal (`isSecurityGreenForBranch` / `getSecurityStateForBranch` тАФ Phase 3) the **M4 promote gate** reads to refuse to auto-merge an unreviewed branch. The clean definition is the SAME `completedClean` rule the post-ship fold gate uses (`getSecurityStateBySlug`), so the two gates can never disagree. **isolate-premerge-security-verdict Phase 1 тАФ pre-merge branch mode is ISOLATED as a standalone enqueue (reverts consolidate-premerge-checks-one-session Phase 1 fusion):** the security half of pre-merge is no longer fused into the spec-test session; `runSecurityReviewJob` (branch mode) runs as a separate, dedicated Max session per branch. The standalone [[../specs/archive.d/consolidate-premerge-checks-one-session|previous fusion]] ran both verdicts in one session and inserted a synthetic security row; the isolated mode re-establishes `maybeEnqueuePreMergeSecurityOnAccumulation` as the ONLY producer of pre-merge security verdicts. `isSecurityGreenForBranch` reads the agent_jobs rows directly (no synthetic rows from the spec-test envelope). The M4 promote gate and the post-ship fold gate remain in lockstep on the `completedClean` definition.
- **`dep-watch`** тАФ the daily **`npm audit`** CVE scan (enqueued by [[../inngest/security-dep-watch]]). On a vulnerable dep it authors a scoped upgrade-fix spec + surfaces a Build card тАФ never auto-bumps.

## Exports

- `type SecurityVerdict = "real-vuln" | "needs-human" | "false-positive"` тАФ the box's per-diff classification (`clean` is a no-action terminal alongside `false-positive`).
- `SECURITY_DIRECTOR_FUNCTION = "platform"` тАФ the function whose objective supervises this worker.
- `SECURITY_DEP_UPGRADE_SLUG = "security-dep-upgrades"` тАФ the stable slug of the dep-upgrade fix spec (find-or-update, never N of them).
- `SECURITY_DEP_WATCH_SLUG = "security-dep-watch"` тАФ the sentinel `spec_slug` a dep-watch scan job carries (so it dedups distinctly from per-diff jobs).
- `SECURITY_RECENT_FIX_WINDOW_MS` (24h) тАФ a fix authored within this window is "pending deploy"; don't re-scan/re-surface.
- `LIVE_SECURITY_STATUSES` тАФ the statuses that mean a security-review job is still live (working or surfaced).
- `securitySha12(mergeSha)` / `depFindingSignature(findings)` тАФ the per-diff dedup key (the merge SHA) and the advisory-set signature (sorted `pkg@severity`, sha1).
- `enqueueSecurityReviewJob(admin, input)` тЖТ enqueue ONE `security-review` [[agent_jobs]] job. **Best-effort, never throws.** Two input shapes (one function тАФ discriminated by the presence of `branch`):
  - **post-merge `diff`** `{ mergeSha, specSlug, prNumber?, workspaceId? }` тАФ **deduped by the merge SHA** so a re-run (the two merge-hook paths firing for one merge) never double-files.
  - **pre-merge `branch`** `{ branch, previewOrigin, specSlug, prNumber?, workspaceId? }` ([[../specs/security-test-on-preview-pre-merge]] Phase 1) тАФ **two-part dedup** (vault-security-review-loop-fix): (1) **ONE OPEN review per branch** (any live/surfaced security-review job with `spec_branch === branch`) AND (2) **ONE CLEAN review per UNCHANGED branch** тАФ a `completed` review for the branch blocks re-enqueue UNLESS a newer build-job push landed since (compare the latest `completed` review's `created_at` vs the latest `build` job's `updated_at` on the branch). A `failed` review never blocks (no verdict produced). This stopped the **Vault re-review loop**: the standing-pass backstop re-fired branch-mode security every pass and the old dedup (live-only) never saw the prior CLEAN review, so a fully-built branch got a fresh security-review every ~5 min forever. We re-review only when the branch's diff actually advanced. The `securitySha12`/`depFindingSignature` signatures still converge same-finding recurrences inside the box's review of the branch's diff.
- `enqueueDepWatchJob(admin, { workspaceId? })` тЖТ enqueue the daily dep-watch scan. Deduped to тЙд1 live scan + a recent-window guard. **Best-effort, never throws.**
- `enqueueSecurityDiffIfDue(admin, { sinceMs?, workspaceId? })` тЖТ `{enqueued: string[], scanned: number, resolved: number}` ([[../specs/vault-post-merge-diff-backstop]] Phase 1 + [[../specs/fix-vault-post-merge-diff-backstop-7fbde0]] cheap if-due cron backstop) тАФ the standing-pass/cron BACKSTOP for the post-merge `diff` review, sibling of `enqueueSpecTestIfDue` / `enqueueSpecReviewIfDue`. Enumerates recently-merged `claude/*` builds (agent_jobs kind='build', status='merged', updated_at within `sinceMs` тАФ default 14d), resolves each build's merge SHA(s) via spec provenance (per-phase `spec_phases.merge_sha` for multi-phase specs + card-level `specs.last_merge_sha` for one-shots), and (idempotently) calls `enqueueSecurityReviewJob` in diff mode for each SHA. Idempotency comes for free from the 14d SHA dedup inside `enqueueSecurityReviewDiff`. Driven every 15 min by [[../inngest/security-diff-backstop-cron]] (the cheap if-due backstop), and additionally by the platform-director standing pass + the daily [[../inngest/security-dep-watch]] cron. **Best-effort, never throws.**
- `getOpenSecurityReviews(admin, workspaceId)` тЖТ `SecuritySurfaceItem[]` тАФ READ-ONLY: open security items awaiting the owner (`needs_approval` = a routed fix with a `security_build` Build action, or `needs_attention` = needs-human). Clean reviews complete silently.
- `getSecurityStateBySlug(admin, workspaceId)` тЖТ `Record<slug, SecurityStateBySlug>` тАФ READ-ONLY per-spec rollup (`live`/`surfaced`/`completedClean`) for the build-card lifecycle timeline's Security node + the fold gate (they MUST agree). Excludes infra dep-watch jobs by **identity** (the `security-dep-watch` sentinel slug OR a `dep-watch`-MODE row), NOT by slug name тАФ the dep-watch agent's authored `security-dep-upgrades` fix spec is a REAL shippable spec whose `diff`/`branch` reviews DO roll up (excluding that slug blanked its Security node forever, so its fold gate could never clear).
- `getSecurityStateForBranch(admin, branch)` тЖТ `SecurityStateForBranch` ([[../specs/archive.d/security-test-on-preview-pre-merge]] Phase 3) тАФ READ-ONLY **per-branch** rollup keyed to `spec_branch === branch` (the pre-merge `branch`-mode jobs from Phase 1). Same shape + same `completedClean` definition as `getSecurityStateBySlug` (a `completed` job exists AND no live/surfaced sibling) so the **M4 pre-merge promote gate** and the post-ship fold gate ([[../specs/build-card-lifecycle-timeline]] Phase 3 / [[spec-test-runs]] `getAutoFoldEligibleSlugs`) can never disagree on what "security green" means. A branch with no security-review record yet returns all-false (absence тЙа clean, same as the post-ship gate).
- `isSecurityGreenForBranch(admin, branch)` тЖТ `boolean` ([[../specs/security-test-on-preview-pre-merge]] Phase 3) тАФ convenience boolean the **M4 pre-merge promote gate** reads. Green iff `getSecurityStateForBranch(...).completedClean === true` тАФ the SAME `completedClean` definition `getSecurityStateBySlug` uses, so the pre-merge gate and the post-ship fold gate stay in lockstep.
- `listSecurityReviews(admin, workspaceId, limit?)` тЖТ `SecurityReviewLogItem[]` тАФ READ-ONLY: Vault's **full** review log (clean ones included), newest-first, for the [[../dashboard/security-tests]] surface. Derives a `verdict` per row (`cleanя╜Ьfalse-positiveя╜Ьreal-vulnя╜Ьneeds-humanя╜Ьrunningя╜Ьfailed` тАФ a completed review's verdict is parsed off the `log_tail` prefix) and resolves each reviewed slug's [[../tables/specs|spec]] title. Bounded (тЙд500).
- `countOpenSecurityReviews(admin, workspaceId)` тЖТ `number` тАФ the surfaced count (`needs_approval`/`needs_attention`) for the Security tests sidebar badge. Clean reviews never count.

## Trigger тАФ event-driven (`diff` + `branch`) + a daily cron (`dep-watch`)

- **`diff`** rides the **merge hook**: [[agent-jobs]] `applyMergedBuildEffects` (the shared body of BOTH `reconcileMergedJobs` тАФ a manual squash-merge тАФ and `handleAutoMergedBuildBranch` тАФ the auto-merge webhook) calls `enqueueSecurityReviewJob` with the merge SHA. So every merged `claude/*` build diff gets exactly one review, deduped by SHA. **Backstop ([[../specs/vault-post-merge-diff-backstop]] Phase 1 + [[../specs/fix-vault-post-merge-diff-backstop-7fbde0]] cheap if-due cron backstop):** the merge-hook send is fire-and-forget тАФ if it's dropped (a Vercel deploy reaps the Inngest sync mid-flight, the box is down, a transient error) the merged commit never gets its post-merge security pass. Three nets close the gap: (1) the cheap **[[../inngest/security-diff-backstop-cron]]** runs the sweep every 15 min тАФ the PRIMARY backstop, so an orphan window never exceeds ~15 min (the 7fbde0 pre-merge spec-test observed orphan_count=45 across ~14d because the pre-existing 24h daily leg was the shortest cadence тАФ this cron shrinks that to 15 min); (2) the [[../../../scripts/builder-worker]] platform-director **standing pass** runs it every beat next to `backstopPreMergeChecks`; (3) the daily [[../inngest/security-dep-watch]] cron hangs it off as a final net so a full Inngest / box outage that takes both above out still can't leave merged commits unreviewed indefinitely. All three call `enqueueSecurityDiffIfDue`, which enumerates recently-merged `claude/*` builds, resolves each merge SHA from spec provenance, and (idempotently, via the existing 14d SHA dedup) re-fires the diff-mode enqueue for any SHA lacking a security-review row. This mirrors [[spec-test-runs]] `enqueueSpecTestIfDue` for the same dropped-event class of gap: the diff leg is now **reactive + backstopped**, matching the branch leg.
- **`branch`** (pre-merge) тАФ **standalone enqueue** via `maybeEnqueuePreMergeSecurityOnAccumulation` ([[../specs/isolate-premerge-security-verdict]] Phase 1, reverts consolidate-premerge-checks-one-session Phase 1). The pre-merge spec-test session (`runSpecTestJob`) reads the branch diff + preview and emits ONLY the spec-test verdict тАФ the security pass is no longer fused and runs as a **separate Max session** via `runSecurityReviewJob` in `branch` mode. The trigger `maybeEnqueuePreMergeSecurityOnAccumulation` is restored to active status (previously inert during the fusion). **Dedup (branch mode, `enqueueSecurityReviewJob`), in order:** **(0) MERGED-BRANCH guard** (post-merge backstop loop fix) тАФ if the latest `build` job for the branch is `status='merged'`, the `claude/build-*` branch is DELETED тЖТ refuse (a pre-merge review of a gone branch is meaningless and was the #837 loop: the post-merge state-advance bumps the build job's `updated_at`, which the step-(2) "branch changed" test mistook for a new push тЖТ re-enqueued every standing pass); **(1)** one OPEN review per branch (a live review still in flight тЖТ no-op); **(2)** one CLEAN review per UNCHANGED branch тАФ a `completed` review blocks re-enqueue unless a **genuine new build push** landed since (compared against the latest NON-`merged` build job's `updated_at`, so a status-only merge flip never counts as a push). **Backstop (Gate-A class gap fix):** the preview-ready callback is a fire-and-forget poll; if it misses READY (worker restart, slow preview), the branch enqueue would never fire тАФ so the [[../../../scripts/builder-worker]] platform-director **standing pass** also runs [[agent-jobs]] `backstopPreMergeChecks`, which re-evaluates every READY-preview `claude/build-*` branch each pass and (idempotently) re-enqueues BOTH pre-merge triggers (spec-test + security). Before this wiring landed, the branch-mode enqueue had **no caller at all** тАФ `isSecurityGreenForBranch` was always false and the M4 tests gate could never pass. **In-flight-only filter (vault-security-review-loop-fix + post-merge backstop loop fix):** the backstop now SKIPS any branch whose spec is **MERGED** (latest `build` job `merged` OR a merged sibling via `findMergedSiblingBuild` тАФ branch gone), **shipped/folded** (DB `specs.status`), or **archived** (in `listArchivedSlugs`). The merge-signal skip is required on top of the status skip because the post-merge-ships-only-one-phase bug left a merged spec reading `in_progress` (only one phase stamped) тАФ so the status check never fired and security re-enqueued forever (#837). Combined with the branch-mode MERGED-BRANCH guard + CLEAN-review dedup above, an in-flight branch gets exactly ONE pre-merge security review per branch state, and a merged/shipped/folded spec gets NONE.
- **`dep-watch`** is the [[../inngest/security-dep-watch]] daily cron тЖТ `enqueueDepWatchJob`.

## The box loop тАФ `runSecurityReviewJob`

1. **Disposer resume** тАФ a routed `security_build` action approved (queue the build) / declined (dismiss).
2. **`dep-watch` mode** тАФ run `npm audit --json` on the real tree; collect тЙе moderate advisories; on a finding author/refresh `security-dep-upgrades` to main + route; clean tree тЖТ complete silently with a healthy beat.
3. **`diff` mode** (post-merge) тАФ `git fetch origin main` + `git show {sha}` read-only on Max тЖТ ONE verdict (see VERDICT TAIL below).
4. **`branch` mode** (pre-merge тАФ [[../specs/archive.d/security-test-on-preview-pre-merge]] Phase 2) тАФ `git fetch origin main` + `git fetch origin {branch}` + `git diff origin/main...origin/{branch}` read-only on Max; any runtime probe targets the per-build `preview_origin` from [[archive.d/per-build-vercel-preview-deploys]], **NOT prod**. Same review checks + verdict envelope as `diff` mode (the prompt shares one `securityReviewChecksAndVerdictBlock`), so Vault's contract is identical regardless of when she runs. On `real-vuln` the authored fix spec records `Fixes: [[parent-slug]]` + `**Security-of-branch:** \`{branch}\` ┬╖ preview \`{previewOrigin}\`` (vs `**Security-of-merge:**` for `diff`) so a glance shows which mode caught it. The terminal state (`completed` clean / `needs_approval` routed-fix / `needs_attention` needs-human / `failed`) is what the **M4 promote gate** reads to refuse to auto-merge an unreviewed branch (Phase 3's `completedClean` signal).

**VERDICT TAIL** (shared between `diff` + `branch`):
   - `clean` / `false-positive` тЖТ complete, no action.
   - `needs-human` тЖТ `needs_attention` + an `escalated` activity row, no spec. The activity reason cites the mode-appropriate context (`merged {slug} (commit {sha})` for diff ┬╖ `unmerged {slug} (branch {branch})` for branch).
   - `real-vuln` тЖТ **author the fix spec to main** (`authorSecurityFixSpec(..., source)` where `source = {kind:'diff', mergeSha}` or `{kind:'branch', branch, previewOrigin}`; `Fixes: [[parent-slug]]`), then `routeSecurityFix` via `resolveApproverLive('platform')` ([[approval-router]]): a live+autonomous director auto-queues the build within its leash; else surface a `security_build` Build card for the **CEO inbox** (the generic [[../specs/approval-routing-engine|approval-inbox]] reconciler emits the routed request тАФ no per-agent route needed). Records `authored_fix`.
   - **Unparseable/unrecognized verdict** ([[../specs/needs-attention-triage-and-verdict-robustness]] Phase 2 + [[../specs/iteration-ingest-async-reports-fix-tooling-69594a]] + [[../specs/ada-director-spec-status-cards-fix-tooling-fa6848]]) тЖТ the shared `resolveReviewVerdict` helper (shared with [[repair-agent]] + [[regression-agent]]) first attempts a **same-session JSON parse-repair re-prompt** (`securityReviewRepairPrompt`) тАФ the model already spent the budget doing real review work, so re-emitting ONLY the verdict envelope from the existing context recovers a verdict far cheaper + far more often than a fresh re-investigation. Two tooling guards underneath the repair (added when fa6848a7 parked despite the parse-repair already shipping): (a) `shAsync` closes stdin (`stdio: ["ignore", тАж]`) so the claude CLI doesn't burn 3s on its "no stdin data received" warning (the CLI's own recommended `< /dev/null` fix, in-process) тАФ that warning was visible on every parked tooling_failure (fa6848a7, 69594acf); (b) `extractJson` now walks every fenced ```` ```json ```` block (last-wins) and scans the LAST balanced `{тАж}` (right-to-left close ├Ч left-to-right open), so a verdict envelope buried under prose-braces or an earlier example JSON parses correctly. If the parse-repair still fails (or no prior session exists), it fail-safes to an **actionable** `needs_attention` reason тАФ `"security review produced no parseable verdict after 2 attempts тАФ re-run or review manually: <excerpt>"` тАФ **never** a bare "ended without a recognizable verdict", never an auto-pass. The director's `reconcileNeedsAttention` ([[platform-director]] Phase 1) then triages that parked item.

## Why npm audit runs on the box, not in the cron

`npm audit` needs the npm CLI + the committed lockfile + registry access тАФ none reliably present in the Vercel/Inngest serverless runtime. So [[../inngest/security-dep-watch]] is the SCHEDULER (deduped, heartbeat) and the box (which has the full repo + npm) runs the actual scan. The observable behaviour matches the spec: a daily watch тЖТ an upgrade-fix spec + Build card on a finding, a healthy beat on a clean tree.
