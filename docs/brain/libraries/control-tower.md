# libraries/control-tower

The Control Tower module ([[../specs/control-tower]] Phase 1 + Phase 2; [[../specs/control-tower-agent-coverage]] adds the inline AI agents; [[../specs/control-tower-complete-coverage]] Phase 1 adds **full cron coverage + the reactive event-driven agents**) ‚Äî the registry, the heartbeat emit helper, and the monitor/snapshot logic (liveness + cron-freshness + stuck-jobs + inline/reactive-agent assertions + Phase 2 output assertions) that powers the [[../inngest/control-tower-monitor]] cron and the [[../dashboard/control-tower]] dashboard.

**Files:** `src/lib/control-tower/registry.ts` ¬∑ `src/lib/control-tower/heartbeat.ts` ¬∑ `src/lib/control-tower/monitor.ts` ¬∑ `src/lib/control-tower/error-feed.ts` ¬∑ `src/lib/control-tower/supabase-log-poll.ts` ¬∑ `src/lib/control-tower/migration-drift.ts` ¬∑ (Phase 2 detection layer: [[control-tower-self-audit]] ‚Äî `self-audit.ts` ¬∑ `src/lib/inngest/sync.ts` ¬∑ `src/lib/inngest/registered-functions.ts`)

## `registry.ts` ‚Äî the loop registry (code config)

The single source of truth for every loop the monitor watches. **Add a row here when you ship a new cron / worker / agent-kind** ([[../operational-rules]] "register-or-it's-incomplete").

- `type LoopKind = "worker" | "cron" | "agent-kind" | "inline-agent" | "reactive"` ‚Äî `reactive` = an **event-driven Inngest function** (the inbound ticket handler, dunning, returns, journey-outcomes, agent-todo-execute, chargeback); it shares the inline-agent evaluation model (idle = green, liveness-when-work-exists + error-rate) and is split out only for the dashboard's "Reactive agents" group. ([[../specs/control-tower-complete-coverage]] P1.)
- `type OutputAssertionId = "escalation-idle" | "spec-test-persisted" | "renewal-integrity" | "renewal-outcome-distribution" | "stuck-dunning" | "migration-drift"` ‚Äî Phase 2 output assertions (see monitor.ts). The renewal pair ([[../specs/control-tower-renewal-integrity-assertions]] P1) completes the renewal-integrity trio; `migration-drift` ([[../specs/control-tower-migration-drift-check]] P1) reads a created-table-vs-live-schema diff conveyed in a box-emitted beat (see `migration-drift.ts` below).
- `type InlineWorkSignalId = "tickets-awaiting-qc" | "journeys-awaiting-delivery" | "orders-awaiting-fraud-screen" | "tickets-awaiting-decision"` ‚Äî the read-only independent upstream-demand probes the inline-agent liveness-when-work-exists check evaluates.
- `type RenewalOutcome = "charged" | "skipped_no_payment_method" | "skipped_zero_total" | "declined_to_dunning" | "comp_shipped" | "comp_blocked" | "skipped_other"` + `RENEWAL_OUTCOME_LOOP_ID = "internal-subscription-renewal-outcome"` + `RENEWAL_BAD_OUTCOMES` (the anomalous subset: no-PM-skip, declined, comp-blocked) ‚Äî the per-sub renewal outcome taxonomy the outcome-distribution assertion aggregates ([[../specs/control-tower-renewal-integrity-assertions]] P1). Every terminal path of `internal-subscription-renewal-attempt` emits ONE beat (loop_id = `RENEWAL_OUTCOME_LOOP_ID`, `kind:'reactive'`, `produced.outcome`) ‚Äî the only uniform channel that captures SKIPS, which leave no transaction row. NOT a monitored tile (a data channel).
- `interface MonitoredLoop { id, kind, label, description, expectedCadence, livenessWindowMs?, shaGraceMs?, agentKind?, stuckThresholdMs?, outputAssertion?, outputAssertions?, inlineWorkSignal?, errorRateThreshold?, minRunsForErrorRate? }` ‚Äî `outputAssertions?: OutputAssertionId[]` is for a loop carrying MORE THAN ONE assertion (the renewal cron: `["renewal-integrity", "renewal-outcome-distribution"]`); the monitor runs each in order, first-to-fail wins. Use it OR the singular `outputAssertion`.
- `MONITORED_LOOPS: MonitoredLoop[]` ‚Äî the box worker (`box`), **the full Inngest cron set (50 crons** ‚Äî each cron's inngest fn id + a cadence-derived `livenessWindowMs`, from minute-crons like `deliver-pending-sends`/`meta-capi-dispatch-cron` through the daily/weekly/yearly ones; [[../specs/control-tower-complete-coverage]] P1 closed the gap from the original 8**)**, 11 agent kinds (`agent:<kind>` + a per-kind `stuckThresholdMs`), **4 inline AI agents** (`ai:ticket-analyzer`, `ai:journey-delivery`, `ai:fraud-detector`, `ai:orchestrator` ‚Äî each with a `livenessWindowMs`, optional `inlineWorkSignal`, and `errorRateThreshold`/`minRunsForErrorRate`), and **6 reactive event-driven agents** (`unified-ticket-handler` ‚Äî THE inbound ticket pipeline, with `inlineWorkSignal: 'tickets-awaiting-decision'` ‚Äî plus `dunning-payment-failed`, `returns-process-delivery`, `journey-session-completed`, `agent-todo-execute`, `chargeback-received`; `kind: 'reactive'`, error-rate by default), the **`auto-merge-gate`** (`AUTO_MERGE_GATE_LOOP_ID`) ‚Äî a `reactive` loop that runs inside the [[../integrations/github-webhook|GitHub webhook]] (the Auto-Ship Pipeline's Gate A, [[../specs/auto-ship-pipeline]] P1), error-rate only (a failed merge attempt beats ok:false; idle = green), and the **`auto-fold-gate`** (`AUTO_FOLD_GATE_LOOP_ID`) ‚Äî a `reactive` loop that auto-folds fully-verified shipped specs (the Auto-Ship Pipeline's Gate B, [[../specs/auto-ship-pipeline]] P2; `autoFoldVerifiedSpecs` in [[spec-test-runs]], fired by the spec-test worker + the human-queue POST + the daily spec-test cron), error-rate only (a failed enqueue beats ok:false; idle = green). Output assertions: `triage-escalations-cron` ‚Üí `escalation-idle`, `spec-test-cron` ‚Üí `spec-test-persisted`, `internal-subscription-renewal-cron` ‚Üí `["renewal-integrity", "renewal-outcome-distribution"]`, `dunning-payday-retry-cron` ‚Üí `stuck-dunning`, `migration-drift-check` (box-emitted cron) ‚Üí `migration-drift`.
- `WORKER_BOX_ID = "box"` (matches `scripts/builder-worker.ts`) ¬∑ `agentLoopId(kind)` ‚Üí `agent:<kind>` ¬∑ `INLINE_AGENT_IDS = { ticketAnalyzer, journeyDelivery, fraudDetector, orchestrator }` (the `ai:<agent>` loop ids, shared with the agent files) ¬∑ `AUTO_MERGE_GATE_LOOP_ID = "auto-merge-gate"` (the auto-merge gate's reactive loop_id, shared with [[github-pr-resolve]]) ¬∑ `AUTO_FOLD_GATE_LOOP_ID = "auto-fold-gate"` (the auto-fold gate's reactive loop_id, shared with [[spec-test-runs]]).
- `MIGRATION_DRIFT_LOOP_ID = "migration-drift-check"` ([[../specs/control-tower-migration-drift-check]] P1) ‚Äî the loop_id of the **box-emitted** migration-drift cron (a `kind:'cron'` MONITORED_LOOPS tile, owner platform, `outputAssertion:'migration-drift'`, ~30 min window 90m). NOT an Inngest fn: the deployed runtime can't read the `supabase/migrations/*.sql` files, so `scripts/builder-worker.ts` runs the parse+diff on the box and beats here, carrying the missing tables in `produced`. Freshness keeps a dead check visible; the assertion flips the tile red on drift. (Not in the in-code‚ÜîInngest self-audit diff ‚Äî `auditCronCoverage` only flags Inngest crons *without* a tile, never a tile without an Inngest fn.)
- `INTENTIONALLY_UNMONITORED_CRONS: Record<id, reason>` (Phase 2) ‚Äî cron `createFunction`s deliberately given no tile, so the [[control-tower-self-audit]] doesn't flag them as "unregistered loop: X". Silence is never the default: each needs a reason. Keep it short ‚Äî the right answer for a real loop is a `MONITORED_LOOPS` row, not an exemption. (Currently empty.)

## `heartbeat.ts` ‚Äî end-of-run emit

Best-effort writes of one [[../tables/loop_heartbeats]] row (never throws).

- `emitLoopHeartbeat(loopId, kind, { ok?, produced?, detail?, durationMs? })`
- `emitCronHeartbeat(functionId, ‚Ä¶)` ‚Äî `kind:'cron'`, `loop_id` = the inngest fn id. Called by each monitored cron inside a `step.run("emit-heartbeat", ‚Ä¶)` before its return.
- `emitAgentHeartbeat(agentKind, ‚Ä¶)` ‚Äî `kind:'agent-kind'`, `loop_id` = `agent:<kind>`. (The box worker writes its agent beats via its own inline `writeLoopHeartbeat` against its existing admin client, not this helper ‚Äî same shape.)
- `emitInlineAgentHeartbeat(agentId, ‚Ä¶)` ‚Äî `kind:'inline-agent'`, `loop_id` = an `INLINE_AGENT_IDS` value (e.g. `ai:ticket-analyzer`, `ai:orchestrator`). Called in a **try/finally at the END of each run** of the 4 inline AI agents ‚Äî `ok:true` on success/intentional-skip, `ok:false` on a thrown run or real failure (for `ai:orchestrator`, also a degraded/fallback decision).
- `emitReactiveHeartbeat(functionId, ‚Ä¶)` ‚Äî `kind:'reactive'`, `loop_id` = the inngest fn id (e.g. `unified-ticket-handler`). Called in a **top-level try/finally** inside each of the 6 reactive event-driven Inngest functions ‚Äî `ok:true` on success/skip-return, `ok:false` on a thrown run. ([[../specs/control-tower-complete-coverage]] P1.)
- `emitRenewalOutcomeHeartbeat(outcome: RenewalOutcome)` ‚Äî records ONE per-sub renewal outcome beat (`loop_id = RENEWAL_OUTCOME_LOOP_ID`, `kind:'reactive'`, `produced:{outcome}`, `ok:true`). Called from every terminal path of `internal-subscription-renewal-attempt` ([[../inngest/internal-subscription-renewals]]) ‚Äî the only uniform channel that captures SKIPS (a no-PM / zero-total skip writes no transaction row). Uncaught errors aren't beat (a sub that errored never advances ‚Üí caught by the renewal-integrity overdue assertion). ([[../specs/control-tower-renewal-integrity-assertions]] P1.)
- `aggregateRenewalOutcomes(admin, sinceIso, untilIso?)` ‚Üí `RenewalOutcomeCounts` ‚Äî READ-ONLY count breakdown of the outcome beats in a window (bounded `limit 5000`, best-effort ‚Üí zeros on read failure). Used by the renewal cron (bakes the just-completed cycle's breakdown into its heartbeat's `produced.last_cycle_outcomes`) AND by the outcome-distribution assertion (live current cycle + rolling baseline).

## `monitor.ts` ‚Äî snapshot + monitor

- `buildControlTowerSnapshot(admin?)` ‚Üí `ControlTowerSnapshot { generatedAt, counts:{green,amber,red}, loops: LoopStatus[], selfAudit }`. **READ-ONLY**: one batched read of [[../tables/worker_heartbeats]], [[../tables/loop_heartbeats]] (the **`control_tower_loop_beats(p_history_limit)` RPC** ‚Äî latest ‚â§10 beats **per loop_id** for cron + agent-kind kinds, **excluding `inline-agent`/`reactive`**; a **lateral join** over the distinct loop_ids rides the `(loop_id, ran_at desc)` index, reading ‚â§N rows per loop. **Presence is the ever-beaten signal** ‚Äî a loop with zero beats is absent from the result ‚áí never_fired candidate (no all-time count is returned). Replaced the old global "last 600 beats" window ‚Äî and then the per-row `count(*) OVER`/`row_number() OVER` body ([[../specs/control-tower-loop-beats-rpc-perf]]), both of which scanned + sorted the whole growing table and 500-ed on statement timeout), open [[../tables/loop_alerts]], active [[../tables/agent_jobs]], plus `fetchInlineAgentState` (per inline agent: its work probe + exact ok/errored window counts + latest + history) **and `buildCoverageAudit()`** ([[control-tower-self-audit]], Phase 2); evaluates each loop to a `LoopStatus { color, statusText, lastRanAt, lastProduced, detail, violation, history, openAlert }`. The `selfAudit` findings (unregistered crons + in-code‚ÜîInngest-registered gaps) are folded into the honest amber `counts`. Used **verbatim** by the dashboard API.
- `runControlTowerMonitor()` ‚Üí `MonitorResult`. Builds the snapshot, then **acts**: opens a de-duped [[../tables/loop_alerts]] incident on each newly-red loop (paging owners via [[../libraries/notify-ops-alert]]), bumps `last_seen_at` while still red (no re-page), and resolves on recovery. On a **newly-opened** alert it also enqueues a [[repair-agent]] `repair` job (signature `loop:<id>`) ‚Äî the second diagnose‚Üípropose-fix trigger ([[../specs/repair-agent]]). Called only by the cron.
- Evaluators: `evalWorker` (liveness + SHA-behind + **all-Max-accounts-capped**: reads the box heartbeat's `accounts.all_capped` ‚Üí **amber** "all Max accounts capped ‚Äî builds parked, auto-resume" [NOT red ‚Äî `blocked_on_usage` auto-resumes at reset, no manual rebuild], so a green box tile can't hide a throughput stall where every account hit its 5-hour wall; per-account load rides the tile's `lastProduced` ‚Äî [[../specs/box-multi-account-failover]] P2), `evalCron` (freshness + **never-fired**), `evalAgentKind` (stuck jobs), `evalInlineAgent` (inline AI agents **and** the `reactive` event-driven agents ‚Äî both routed through it, and both excluded from the `control_tower_loop_beats` RPC / sourced from `fetchInlineAgentState`). Genuinely-idle/healthy ‚Üí green; a freshly-shipped cron with no beat yet ‚Üí amber (never a false red).
- **`evalCron` never-fired detection** ([[../specs/control-tower-complete-coverage]] Phase 2, accuracy-fixed by [[../specs/control-tower-monitor-accuracy]] Phase 1): a cron with **0 heartbeats** is no longer amber-forever. `deployRefAgeMs(worker)` gives a trustworthy "how long has this deploy been live" lower bound ‚Äî the box worker self-updates to origin/main + restarts on a new SHA, so once `worker.running_sha` matches `VERCEL_GIT_COMMIT_SHA`, `worker.started_at` ‚âà deploy time (returns null = unknown ‚áí stay amber, never a false red). **never-fired keys off whether the loop has EVER beaten (`everBeatCount` ‚Äî now presence in the lateral-join read, not an all-time count), not 0-since-deploy** ‚Äî a cron with **any** historical beat is being invoked by Inngest, so a longer-than-window real cadence (daily `today-sync` w/ 99 beats, bursty `meta-capi-dispatch` w/ 494) is at most a freshness alert, **never** `never_fired`. Only a cron with **0 beats in all of history** whose deploy is **older than its `livenessWindowMs` (cadence + grace)** ‚Üí **red `never_fired`** ("registered but has never run ‚Äî Inngest is not invoking it"), pages like any red. A genuinely-fresh cron (deploy < one window) or an unknown deploy age stays **amber "awaiting first run"**. **registered-but-not-firing** ([[../specs/spec-drift-reconcile-not-firing]] P1): the deploy-anchored `never_fired` misses a cron that has been registered for **weeks but never fires once**, because the box self-updates + restarts on every ship so `deployRefAgeMs` keeps RESETTING under it ‚Äî with frequent deploys its current-deploy clock rarely exceeds the window, so the dead cron hides forever (the real `spec-drift-reconcile` failure: registered with Inngest, `diffInngestRegistered` says `missing:[]`, but its `20,50 * * * *` schedule never executed ‚Üí 0 `loop_heartbeats` beats ever). So a SECOND, **deploy-surviving** reference: `monitorUptimeMs` = the age of the **oldest `control-tower-monitor` beat** (how long the watchdog itself has been continuously running ‚Äî beat retention can only shorten it, never inflate it ‚áí conservative). A cron with **0 beats ever** while the watchdog has itself been beating for **longer than the cron's window** ‚Üí **red `registered_not_firing`** ("registered with Inngest but not firing ‚Äî its cron schedule isn't active in the prod env; re-sync the app"). Distinct from the [[control-tower-self-audit]] Inngest-registration diff (never-*registered*): this fn IS in the registered set, the SCHEDULE just isn't active. **Newly-added-cron grace** ([[../specs/control-tower-registered-not-firing-newcron-grace]]): `monitorUptimeMs` is the watchdog's OWN run-span, independent of when a given cron was ADDED ‚Äî so a long-cadence cron shipped AFTER the watchdog passed its window would false-trip on day one, hours before its first scheduled tick (signal `loop:storefront-experiments-refresh-cron`). The optional **`MonitoredLoop.registeredAt`** (an ISO code constant ‚Äî deploy-SURVIVING, unlike `deployRefAgeMs`) gives a per-loop "how long has THIS cron been registered" reference: `registered_not_firing` now requires `(now - registeredAt) > window` **in addition to** `monitorUptimeMs > window`. A cron registered for less than a full window stays **amber "awaiting first run ‚Äî registered Xh ago"**, graced just like the deploy-anchored `never_fired`. Each newly-added long-cadence cron must carry its own `registeredAt` to claim the grace ‚Äî the same false positive recurred on a sibling daily cron ([[../specs/control-tower-registered-not-firing-new-cron-grace]], signal `loop:storefront-lever-decay-cron`), fixed by stamping that loop's `registeredAt` too. Unset (legacy crons) ‚áí no extra gate (registered long ago). `null` monitor uptime (bootstrapping / pruned) ‚áí stay amber. Closes the `control-tower-monitor` "awaiting first run for days" gap; pairs with the [[control-tower-self-audit]] in-code‚ÜîInngest-registered diff (same gap, registration side). **Beats-read-failure guard** ([[../specs/control-tower-beats-read-failure-guard]]): `buildControlTowerSnapshot` now captures the `control_tower_loop_beats` RPC **error** (not just `data`) ‚Äî a statement-timeout/500 returns `beats=null`, which would otherwise leave every cron with `everBeatCount=0`/`latest=null` and false-fire `never_fired`. When `beatsReadFailed` is true `evalCron` treats the zero as **UNKNOWN, not a true zero**: it suppresses both `never_fired` and `cron_freshness` reds and returns **amber "beat read unavailable ‚Äî status unknown"** ‚Äî the same conservative posture as the `deployAgeMs==null` guard.
- **Inline-agent assertions** ([[../specs/control-tower-agent-coverage]], `evalInlineAgent` + `fetchInlineAgentState`): event-driven AI agents have no cron cadence and no `agent_jobs` queue, so silence only matters when work exists. **liveness-when-work-exists** ‚Äî the loop's `inlineWorkSignal` (an independent upstream-demand count: unanalyzed closed-AI [[../tables/tickets]]; recent [[../tables/journey_sessions]]; recent [[../tables/orders]]; inbound-customer [[../tables/ticket_messages]] for `ai:orchestrator`) is > 0 AND **0 successful** beats in the window ‚Üí red, `reason='idle_while_work'`. The `tickets-awaiting-qc` probe carries a **feeder-cadence grace** (`TICKET_ANALYSIS_FEEDER_GRACE_MS` = 40 min, [[../specs/ticket-analyzer-workprobe-cron-grace]]): since the analyzer is fed by `ticket-analysis-cron` every 30 min, a closed AI ticket only counts once its `updated_at` is ALSO older than the grace ‚Äî i.e. it has survived a full feeder cycle still unprocessed ‚Äî so a ticket closing between the 30-min ticks doesn't false-fire `idle_while_work` ~15 min before any cron cycle could service it (a genuinely-stuck analyzer still trips it a cycle later). The `tickets-awaiting-decision` probe (shared by `unified-ticket-handler` + `ai:orchestrator`) counts only inbound-customer [[../tables/ticket_messages]] that DRIVE the handler ([[../specs/control-tower-ticket-decision-workprobe-scope]]): it subtracts messages on `csat:reopened`-tagged tickets, since the CSAT-reopen path (`/api/csat/[ticketId]`) inserts an inbound customer message + reopens + routes to a human but emits NO `ticket/inbound-message` event, so the handler never beats on it ‚Äî a lone reopen note in a quiet window used to read as work=1 / 0 beats and false-fire `idle_while_work`. NULL-safe subtraction (not a negated array filter) so NULL/empty-tag tickets always count. **error-rate** ‚Äî `errCount/total ‚â• errorRateThreshold` (default 0.5) once `total ‚â• minRunsForErrorRate` (default 5) ‚Üí red, `reason='error_rate'`. Inline-agent beats are fetched separately (exact counts, true-latest-regardless-of-idle) so the agents' high beat volume can't crowd the cron read (and are excluded from the `control_tower_loop_beats` RPC).
- **Phase 2 output assertions** (`evalOutputAssertion(assertionId, loop, latest, inputs)`, layered on top of the P1 tile ‚Äî only escalates green/amber ‚Üí red, a P1 red stays; a loop with `outputAssertions` runs each in order, first-to-fail wins): `fetchAssertionInputs(admin)` adds cheap read-only queries to the snapshot batch ‚Äî open routine-escalated [[../tables/tickets]] (`escalated_at` set, `escalated_to` null, not closed/archived) ‚Äî both the count AND the OLDEST waiting ticket's `escalated_at` (how long real work has actually waited) ‚Äî the latest `triage-escalations` + `spec-test` [[../tables/agent_jobs]] `created_at`, active overdue internal [[../tables/subscriptions]] (`next_billing_date` before today UTC), the latest renewal-cron beat (marks the live cycle boundary), the live + rolling-baseline renewal outcome breakdowns (`aggregateRenewalOutcomes`), and stuck [[../tables/dunning_cycles]] (`status='retrying'` with `next_retry_at` > 48h in the past). The workspace-scoped integrity queries (overdue subscriptions, stuck dunning) exclude the permanent **spec-test sandbox** tenant (`is_test=true`, `SPEC_TEST_SANDBOX_WORKSPACE_ID` = `SPEC_TEST_FIXTURES.workspaceId`, [[spec-test-sandbox]]) via `.neq("workspace_id", ‚Ä¶)` ‚Äî its seeded fixtures are deliberately stuck (e.g. the comp sub whose customer has no `comp_role`, so the fail-closed comp gate never advances it) and would otherwise read as a real production anomaly that trips a loop tile RED ([[../specs/control-tower-exclude-spec-test-sandbox-from-integrity-asser]]). The five assertions: **escalation-idle** (the OLDEST waiting ticket has waited past the cadence grace [keyed off its `escalated_at`, not the last job's age] AND no triage job was created since it escalated ‚Üí `reason='idle_while_work'`; re-based by [[../specs/control-tower-escalation-idle-grace]] so a healthy hourly cron isn't flagged in the normal gap between an escalation and the next tick), **spec-test-persisted** (beat reports `enqueued>0` but no spec-test job landed ‚Üí `reason='false_success'`), **renewal-integrity** (N active internal subs overdue ‚Üí `reason='renewal_integrity'`), **renewal-outcome-distribution** (the cron ran + each decline routed correctly, but the per-cycle outcome MIX is anomalous: a systemic rate ‚â•50% [hard floor: bad creds declining everyone / mass no-PM] OR a spike ‚â•2.5√ó / +15pp vs the rolling baseline, gated by a ‚â•10-sample cycle + ‚â•50-sample baseline ‚Üí `reason='renewal_outcome_distribution'`; [[../specs/control-tower-renewal-integrity-assertions]] P1), **stuck-dunning** (N dunning_cycles still 'retrying' >48h past `next_retry_at` ‚Äî a sub correctly mid-dunning within schedule is NOT flagged ‚Üí `reason='stuck_dunning'`; same spec). A sixth, **migration-drift** ([[../specs/control-tower-migration-drift-check]] P1), is unlike the others ‚Äî it reads NO live DB state in the monitor: detection happens on the box (it parses the `.sql` files + diffs the live schema, which the deployed runtime can't) and the result rides the loop's heartbeat, so `evalOutputAssertion('migration-drift', ‚Ä¶)` just reads `latest.produced.missing` (via `readMissingTables`) and flips red on a non-empty list (`reason='migration_drift'`). See `migration-drift.ts`. A violation flows through `runControlTowerMonitor` exactly like a P1 red (de-duped alert + page). Thresholds are tunable constants in monitor.ts (`RENEWAL_MIN_CYCLE_SAMPLE`, `RENEWAL_HARD_FLOOR_RATE`, `RENEWAL_SPIKE_FACTOR`, `RENEWAL_SPIKE_MARGIN`, `RENEWAL_MIN_BASELINE_SAMPLE`, `STUCK_DUNNING_GRACE_MS`).

## `error-feed.ts` ‚Äî the error feed (error-feed-monitoring Phase 1 + 2)

The capture + page + snapshot layer for the "hidden surfaces" ([[../specs/error-feed-monitoring]]) ‚Äî Vercel runtime errors, Inngest failed runs, app-layer Supabase errors, and (Phase 2) DB-level Supabase logs ‚Äî into [[../tables/error_events]].

- `recordError({ source, keyParts, title, detail?, sample?, occurrences? }, admin?)` ‚Üí upserts a **grouped** [[../tables/error_events]] incident on `(source, signature)` (signature = a stable hash of the normalized `keyParts`), bumping `count`/`last_seen_at`. Pages owners ([[notify-ops-alert]]) on a **new signature or a re-firing spike**, rate-limited to one page per incident per 30 min. **Best-effort ‚Äî never throws**; a `23505` race falls back to the update path. On a **new signature** it also enqueues a [[repair-agent]] `repair` job ([[../specs/repair-agent]]) ‚Äî the diagnose‚Üípropose-fix trigger (deduped by signature, best-effort).
- `reportDbError(error, { op, table?, ‚Ä¶ }, admin?)` ‚Äî the **app-layer Supabase reporter**: a no-op on a null error, else `recordError({ source: "supabase", ‚Ä¶ })`. Call it anywhere code gets a non-null Supabase `{ error }` it would otherwise swallow (the scorecard-upsert class ‚Äî wired in `src/lib/meta/scorecards.ts`). See [[../operational-rules]] "don't swallow a Supabase error".
- `signatureFor(source, keyParts)` ‚Üí the grouping key (normalizes out uuids/hex/numbers/quoted strings).
- `recordFeedDelivery(source, admin?)` / `feedLoopId(source)` ‚Üí **liveness** ([[../specs/error-feed-honest-panels]]): write a lightweight "received a delivery" beat to [[../tables/loop_heartbeats]] under `loop_id='feed:<source>'`, `kind='feed'` ‚Äî proof the feed is wired + live, SEPARATE from error rows. Emitted on each clean Vercel drain POST (`/api/webhooks/vercel-logs`), each successful Supabase-logs poll (`pollSupabaseLogs`), and each `/api/client-errors` report **and** per-session heartbeat (`source='client'` ‚Äî [[../specs/client-error-capture]]). Best-effort, never throws.
- `buildErrorFeedSnapshot(admin?)` ‚Üí `ErrorFeedSnapshot { generatedAt, panels: ErrorFeedPanel[] }` ‚Äî **READ-ONLY**, **connection-aware** ([[../specs/error-feed-honest-panels]]): last 7 days of [[../tables/error_events]] per source (5 panels: `vercel`, `inngest`, `supabase`, `supabase-logs`, `client`). Each panel carries `{ color, connectionState, configured, lastReceivedAt, statusText, hint, ‚Ä¶ }`. State machine (errors win, then connection): **errors present** ‚Üí red ‚â§1 h / amber ‚â§24 h (unchanged); **not configured** (`VERCEL_LOG_DRAIN_SECRET` unset ¬∑ no stored Supabase token) ‚Üí **amber "not configured"** + a how-to-wire `hint`, never green; **configured but 0 deliveries ever** ‚Üí **amber "awaiting first event"**; **configured + received + clean** ‚Üí **green "connected ¬∑ 0 errors (last delivery Ns ago)"** ‚Äî the only true green. Per-source receipt: `vercel`/`supabase-logs`/`client` read their `feed:<source>` beat (`client` is always-configured ‚Äî the reporters are wired in code ‚Äî so it's amber "awaiting" until the first storefront/portal report or heartbeat lands); `inngest` is proxied by the freshness of Inngest itself (latest `cron` beat ‚Äî the capture fn is failure-only, no clean delivery to observe); `supabase` (app-layer) needs no receipt (`reportDbError` is wired unconditionally). Used by the dashboard API (`GET /api/developer/control-tower`, merged in as `errorFeed`); the route also **folds the panel colors into the header health count** so an unconfigured (amber) panel is never counted healthy.
- Feeders: [[../inngest/inngest-failure-capture]] (`source='inngest'`), `/api/webhooks/vercel-logs` ([[../integrations/vercel-log-drain]], `source='vercel'` + `recordFeedDelivery('vercel')`), `reportDbError` call sites (`source='supabase'`), the [[../inngest/supabase-log-poll]] cron (`source='supabase-logs'` + `recordFeedDelivery('supabase-logs')`), and `POST /api/client-errors` (`source='client'` + `recordFeedDelivery('client')` ‚Äî fed by the storefront + portal reporters, [[client-error-reporter]] ¬∑ [[../specs/client-error-capture]]).

## `supabase-log-poll.ts` ‚Äî Management Logs poller (error-feed-monitoring Phase 2)

Pulls DB-level Supabase errors our app never sees ‚Äî Postgres/auth/API ‚Äî from the [[../integrations/supabase-management-logs]] `logs.all` endpoint into [[../tables/error_events]] (`source='supabase-logs'`). Driven by the [[../inngest/supabase-log-poll]] cron.

- `pollSupabaseLogs(admin?)` ‚Üí `{ status: 'no-token'|'ok'|'error', incidents, rows, errors }` ‚Äî reads + decrypts the token, polls the `(last_polled_at, now]` window (‚â§24h) across 3 source queries, **groups by `(source, signature)`** client-side, `recordError`s each, advances the cursor on partial success. **No-op (no-token)** until the owner pastes the token. Best-effort ‚Äî per-source failures collected in `errors`, never thrown.
- `getSupabaseLogConfig(admin?)` ‚Üí `{ token, projectRef, lastPolledAt }` or null ¬∑ `isSupabaseLogPollConfigured(admin?)` ‚Üí boolean ¬∑ `setSupabaseAccessToken(token, { projectRef? }, admin?)` (encrypt + upsert) ¬∑ `clearSupabaseAccessToken(admin?)` ¬∑ `projectRefFromEnv()` (parses `NEXT_PUBLIC_SUPABASE_URL`). Config: [[../tables/error_feed_supabase_config]] (single row, service-role only). Owner endpoint: `/api/developer/control-tower/supabase-token`.

## `migration-drift.ts` ‚Äî schema-vs-migrations drift check (control-tower-migration-drift-check Phase 1)

Catches a **silently-skipped migration**: a migration that never applied is invisible ‚Äî the code references a table, the table isn't there, every upsert hits `PGRST205`, and (pre-error-feed) it degrades silently. (Found 2026-06-22: `20260618140000_meta_performance_tables.sql` was skipped in the apply pipeline ‚Üí the iteration engine ran on empty ROAS data for weeks.) The Control Tower's first **schema-shaped** check.

- **Pure parse + diff** (no fs/pg at module load ‚Äî the caller supplies the dir + a `fetchLiveTables` callback, so the LOGIC is testable and the box owns the raw-SQL read):
  - `parseExpectedTables(migrationsDir)` ‚Üí `{ expected: Map<table, firstMigration>, dropped, parsedFiles }` ‚Äî every `*.sql` (lexical sort = apply order) parsed for the tables it CREATEs, **net of any later DROP** (drop-aware: create-then-drop = net-absent, not expected; a re-create revives it). Strips `--` + `/* */` comments and skips `temp`/`unlogged` tables; handles `if not exists`, `public.`-qualified, and `"quoted"` names.
  - `extractCreatedTables(sql)` / `extractDroppedTables(sql)` ‚Äî the per-file regex primitives.
  - `computeDrift(expected, liveTables, allowlist?)` ‚Üí `{ missing, allowlistedMissing }` ‚Äî expected-but-absent tables, split by the allowlist.
  - `runMigrationDriftCheck({ migrationsDir, fetchLiveTables, allowlist? })` ‚Üí `DriftResult { status:'ok'|'drift'|'skipped', missing, allowlistedMissing, expectedCount, liveCount, parsedFiles, reason? }`. `fetchLiveTables` returning **null** ‚áí `status:'skipped'` (no DB password on this host) ‚Äî an HONEST missing read, never a false "no drift". `driftSummary(r)` ‚Üí a one-line log/detail string.
- `MIGRATION_DRIFT_ALLOWLIST = ["klaviyo_*"]` + `isAllowlisted(table, allowlist?)` ‚Äî **scope the noise** (spec): a sunset table (Klaviyo is being retired) that disappears lands in `allowlistedMissing` (surfaced for visibility) and NEVER alerts. Entries match exactly or as a `prefix*` wildcard. Keep it SHORT ‚Äî the right answer for a real missing table is to re-apply its migration.
- **Where it runs:** `scripts/builder-worker.ts` ‚Üí `runMigrationDriftJob` (a periodic box job, ~30 min, in-flight-guarded so it never overlaps/stalls the 5s poll) reads the live `public` BASE TABLE names off the pooler (`information_schema.tables` ‚Äî not exposed via PostgREST, so raw `pg` via `poolerConnectionString()`) and writes the `DriftResult` into a `loop_heartbeats` beat (`loop_id = MIGRATION_DRIFT_LOOP_ID`, `kind:'cron'`, `produced.missing`, `ok = status!=='skipped'`). The deployed Next runtime can't do this ‚Äî the `.sql` files aren't bundled.
- **Surfacing** = the standard monitor path: `monitor.ts` `evalOutputAssertion('migration-drift', ‚Ä¶)` reads `latest.produced.missing` (allowlisted ones already excluded box-side) and flips the tile **red** (`reason='migration_drift'`) ‚Üí de-duped [[../tables/loop_alerts]] + page + [[repair-agent]] enqueue, exactly like any P1 red. Cron-freshness/never-fired keeps a *dead* check visible too.

## Gotchas

- **SHA-behind needs `VERCEL_GIT_COMMIT_SHA`** (the deployed commit) as the origin/main proxy; unset locally ‚áí the check is skipped (no false positive). It only fires red after `shaGraceMs` (default 30m) so an in-progress deploy / self-update never pages.
- **Agent-kind alert is off [[../tables/agent_jobs]], not the heartbeat** ‚Äî idle = green. The heartbeat only feeds last-ran/history.
