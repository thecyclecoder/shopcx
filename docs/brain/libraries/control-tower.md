# libraries/control-tower

The Control Tower module ([[../specs/control-tower]] Phase 1 + Phase 2; [[../specs/control-tower-agent-coverage]] adds the inline AI agents; [[../specs/control-tower-complete-coverage]] Phase 1 adds **full cron coverage + the reactive event-driven agents**) ŌĆö the registry, the heartbeat emit helper, and the monitor/snapshot logic (liveness + cron-freshness + stuck-jobs + inline/reactive-agent assertions + Phase 2 output assertions) that powers the [[../inngest/control-tower-monitor]] cron and the [[../dashboard/control-tower]] dashboard.

**Files:** `src/lib/control-tower/registry.ts` ┬Ę `src/lib/control-tower/heartbeat.ts` ┬Ę `src/lib/control-tower/monitor.ts` ┬Ę `src/lib/control-tower/error-feed.ts` ┬Ę `src/lib/control-tower/supabase-log-poll.ts` ┬Ę `src/lib/control-tower/migration-drift.ts` ┬Ę (Phase 2 detection layer: [[control-tower-self-audit]] ŌĆö `self-audit.ts` ┬Ę `src/lib/inngest/sync.ts` ┬Ę `src/lib/inngest/registered-functions.ts`)

## `registry.ts` ŌĆö the loop registry (code config)

The single source of truth for every loop the monitor watches. **Add a row here when you ship a new cron / worker / agent-kind** ([[../operational-rules]] "register-or-it's-incomplete").

- `type LoopKind = "worker" | "cron" | "agent-kind" | "inline-agent" | "reactive"` ŌĆö `reactive` = an **event-driven Inngest function** (the inbound ticket handler, dunning, returns, journey-outcomes, agent-todo-execute, chargeback); it shares the inline-agent evaluation model (idle = green, liveness-when-work-exists + error-rate) and is split out only for the dashboard's "Reactive agents" group. ([[../specs/control-tower-complete-coverage]] P1.)
- `type OutputAssertionId = "escalation-idle" | "spec-test-persisted" | "renewal-integrity" | "renewal-outcome-distribution" | "stuck-dunning" | "migration-drift"` ŌĆö Phase 2 output assertions (see monitor.ts). The renewal pair ([[../specs/control-tower-renewal-integrity-assertions]] P1) completes the renewal-integrity trio; `migration-drift` ([[../specs/control-tower-migration-drift-check]] P1) reads a created-table-vs-live-schema diff conveyed in a box-emitted beat (see `migration-drift.ts` below).
- `type InlineWorkSignalId = "tickets-awaiting-qc" | "journeys-awaiting-delivery" | "orders-awaiting-fraud-screen" | "tickets-awaiting-decision"` ŌĆö the read-only independent upstream-demand probes the inline-agent liveness-when-work-exists check evaluates.
- `type RenewalOutcome = "charged" | "skipped_no_payment_method" | "skipped_zero_total" | "declined_to_dunning" | "comp_shipped" | "comp_blocked" | "skipped_other"` + `RENEWAL_OUTCOME_LOOP_ID = "internal-subscription-renewal-outcome"` + `RENEWAL_BAD_OUTCOMES` (the anomalous subset: no-PM-skip, declined, comp-blocked) ŌĆö the per-sub renewal outcome taxonomy the outcome-distribution assertion aggregates ([[../specs/control-tower-renewal-integrity-assertions]] P1). Every terminal path of `internal-subscription-renewal-attempt` emits ONE beat (loop_id = `RENEWAL_OUTCOME_LOOP_ID`, `kind:'reactive'`, `produced.outcome`) ŌĆö the only uniform channel that captures SKIPS, which leave no transaction row. NOT a monitored tile (a data channel).
- `interface MonitoredLoop { id, kind, label, description, expectedCadence, livenessWindowMs?, shaGraceMs?, agentKind?, stuckThresholdMs?, outputAssertion?, outputAssertions?, inlineWorkSignal?, errorRateThreshold?, minRunsForErrorRate? }` ŌĆö `outputAssertions?: OutputAssertionId[]` is for a loop carrying MORE THAN ONE assertion (the renewal cron: `["renewal-integrity", "renewal-outcome-distribution"]`); the monitor runs each in order, first-to-fail wins. Use it OR the singular `outputAssertion`.
- `MONITORED_LOOPS: MonitoredLoop[]` ŌĆö the box worker (`box`), **the full Inngest cron set (49 crons** ŌĆö each cron's inngest fn id + a cadence-derived `livenessWindowMs`, from minute-crons like `deliver-pending-sends`/`meta-capi-dispatch-cron` through the daily/weekly/yearly ones; [[../specs/control-tower-complete-coverage]] P1 closed the gap from the original 8**)**, 11 agent kinds (`agent:<kind>` + a per-kind `stuckThresholdMs`), **4 inline AI agents** (`ai:ticket-analyzer`, `ai:journey-delivery`, `ai:fraud-detector`, `ai:orchestrator` ŌĆö each with a `livenessWindowMs`, optional `inlineWorkSignal`, and `errorRateThreshold`/`minRunsForErrorRate`), and **6 reactive event-driven agents** (`unified-ticket-handler` ŌĆö THE inbound ticket pipeline, with `inlineWorkSignal: 'tickets-awaiting-decision'` ŌĆö plus `dunning-payment-failed`, `returns-process-delivery`, `journey-session-completed`, `agent-todo-execute`, `chargeback-received`; `kind: 'reactive'`, error-rate by default). Output assertions: `triage-escalations-cron` ŌåÆ `escalation-idle`, `spec-test-cron` ŌåÆ `spec-test-persisted`, `internal-subscription-renewal-cron` ŌåÆ `["renewal-integrity", "renewal-outcome-distribution"]`, `dunning-payday-retry-cron` ŌåÆ `stuck-dunning`, `migration-drift-check` (box-emitted cron) ŌåÆ `migration-drift`.
- `WORKER_BOX_ID = "box"` (matches `scripts/builder-worker.ts`) ┬Ę `agentLoopId(kind)` ŌåÆ `agent:<kind>` ┬Ę `INLINE_AGENT_IDS = { ticketAnalyzer, journeyDelivery, fraudDetector, orchestrator }` (the `ai:<agent>` loop ids, shared with the agent files).
- `MIGRATION_DRIFT_LOOP_ID = "migration-drift-check"` ([[../specs/control-tower-migration-drift-check]] P1) ŌĆö the loop_id of the **box-emitted** migration-drift cron (a `kind:'cron'` MONITORED_LOOPS tile, owner platform, `outputAssertion:'migration-drift'`, ~30 min window 90m). NOT an Inngest fn: the deployed runtime can't read the `supabase/migrations/*.sql` files, so `scripts/builder-worker.ts` runs the parse+diff on the box and beats here, carrying the missing tables in `produced`. Freshness keeps a dead check visible; the assertion flips the tile red on drift. (Not in the in-codeŌåöInngest self-audit diff ŌĆö `auditCronCoverage` only flags Inngest crons *without* a tile, never a tile without an Inngest fn.)
- `INTENTIONALLY_UNMONITORED_CRONS: Record<id, reason>` (Phase 2) ŌĆö cron `createFunction`s deliberately given no tile, so the [[control-tower-self-audit]] doesn't flag them as "unregistered loop: X". Silence is never the default: each needs a reason. Keep it short ŌĆö the right answer for a real loop is a `MONITORED_LOOPS` row, not an exemption. (Currently empty.)

## `heartbeat.ts` ŌĆö end-of-run emit

Best-effort writes of one [[../tables/loop_heartbeats]] row (never throws).

- `emitLoopHeartbeat(loopId, kind, { ok?, produced?, detail?, durationMs? })`
- `emitCronHeartbeat(functionId, ŌĆ”)` ŌĆö `kind:'cron'`, `loop_id` = the inngest fn id. Called by each monitored cron inside a `step.run("emit-heartbeat", ŌĆ”)` before its return.
- `emitAgentHeartbeat(agentKind, ŌĆ”)` ŌĆö `kind:'agent-kind'`, `loop_id` = `agent:<kind>`. (The box worker writes its agent beats via its own inline `writeLoopHeartbeat` against its existing admin client, not this helper ŌĆö same shape.)
- `emitInlineAgentHeartbeat(agentId, ŌĆ”)` ŌĆö `kind:'inline-agent'`, `loop_id` = an `INLINE_AGENT_IDS` value (e.g. `ai:ticket-analyzer`, `ai:orchestrator`). Called in a **try/finally at the END of each run** of the 4 inline AI agents ŌĆö `ok:true` on success/intentional-skip, `ok:false` on a thrown run or real failure (for `ai:orchestrator`, also a degraded/fallback decision).
- `emitReactiveHeartbeat(functionId, ŌĆ”)` ŌĆö `kind:'reactive'`, `loop_id` = the inngest fn id (e.g. `unified-ticket-handler`). Called in a **top-level try/finally** inside each of the 6 reactive event-driven Inngest functions ŌĆö `ok:true` on success/skip-return, `ok:false` on a thrown run. ([[../specs/control-tower-complete-coverage]] P1.)
- `emitRenewalOutcomeHeartbeat(outcome: RenewalOutcome)` ŌĆö records ONE per-sub renewal outcome beat (`loop_id = RENEWAL_OUTCOME_LOOP_ID`, `kind:'reactive'`, `produced:{outcome}`, `ok:true`). Called from every terminal path of `internal-subscription-renewal-attempt` ([[../inngest/internal-subscription-renewals]]) ŌĆö the only uniform channel that captures SKIPS (a no-PM / zero-total skip writes no transaction row). Uncaught errors aren't beat (a sub that errored never advances ŌåÆ caught by the renewal-integrity overdue assertion). ([[../specs/control-tower-renewal-integrity-assertions]] P1.)
- `aggregateRenewalOutcomes(admin, sinceIso, untilIso?)` ŌåÆ `RenewalOutcomeCounts` ŌĆö READ-ONLY count breakdown of the outcome beats in a window (bounded `limit 5000`, best-effort ŌåÆ zeros on read failure). Used by the renewal cron (bakes the just-completed cycle's breakdown into its heartbeat's `produced.last_cycle_outcomes`) AND by the outcome-distribution assertion (live current cycle + rolling baseline).

## `monitor.ts` ŌĆö snapshot + monitor

- `buildControlTowerSnapshot(admin?)` ŌåÆ `ControlTowerSnapshot { generatedAt, counts:{green,amber,red}, loops: LoopStatus[], selfAudit }`. **READ-ONLY**: one batched read of [[../tables/worker_heartbeats]], [[../tables/loop_heartbeats]] (the **`control_tower_loop_beats(p_history_limit)` RPC** ŌĆö latest Ōēż10 beats **per loop_id** for cron + agent-kind kinds, **excluding `inline-agent`/`reactive`**; a **lateral join** over the distinct loop_ids rides the `(loop_id, ran_at desc)` index, reading ŌēżN rows per loop. **Presence is the ever-beaten signal** ŌĆö a loop with zero beats is absent from the result ŌćÆ never_fired candidate (no all-time count is returned). Replaced the old global "last 600 beats" window ŌĆö and then the per-row `count(*) OVER`/`row_number() OVER` body ([[../specs/control-tower-loop-beats-rpc-perf]]), both of which scanned + sorted the whole growing table and 500-ed on statement timeout), open [[../tables/loop_alerts]], active [[../tables/agent_jobs]], plus `fetchInlineAgentState` (per inline agent: its work probe + exact ok/errored window counts + latest + history) **and `buildCoverageAudit()`** ([[control-tower-self-audit]], Phase 2); evaluates each loop to a `LoopStatus { color, statusText, lastRanAt, lastProduced, detail, violation, history, openAlert }`. The `selfAudit` findings (unregistered crons + in-codeŌåöInngest-registered gaps) are folded into the honest amber `counts`. Used **verbatim** by the dashboard API.
- `runControlTowerMonitor()` ŌåÆ `MonitorResult`. Builds the snapshot, then **acts**: opens a de-duped [[../tables/loop_alerts]] incident on each newly-red loop (paging owners via [[../libraries/notify-ops-alert]]), bumps `last_seen_at` while still red (no re-page), and resolves on recovery. On a **newly-opened** alert it also enqueues a [[repair-agent]] `repair` job (signature `loop:<id>`) ŌĆö the second diagnoseŌåÆpropose-fix trigger ([[../specs/repair-agent]]). Called only by the cron.
- Evaluators: `evalWorker` (liveness + SHA-behind), `evalCron` (freshness + **never-fired**), `evalAgentKind` (stuck jobs), `evalInlineAgent` (inline AI agents **and** the `reactive` event-driven agents ŌĆö both routed through it, and both excluded from the `control_tower_loop_beats` RPC / sourced from `fetchInlineAgentState`). Genuinely-idle/healthy ŌåÆ green; a freshly-shipped cron with no beat yet ŌåÆ amber (never a false red).
- **`evalCron` never-fired detection** ([[../specs/control-tower-complete-coverage]] Phase 2, accuracy-fixed by [[../specs/control-tower-monitor-accuracy]] Phase 1): a cron with **0 heartbeats** is no longer amber-forever. `deployRefAgeMs(worker)` gives a trustworthy "how long has this deploy been live" lower bound ŌĆö the box worker self-updates to origin/main + restarts on a new SHA, so once `worker.running_sha` matches `VERCEL_GIT_COMMIT_SHA`, `worker.started_at` Ōēł deploy time (returns null = unknown ŌćÆ stay amber, never a false red). **never-fired keys off whether the loop has EVER beaten (`everBeatCount` ŌĆö now presence in the lateral-join read, not an all-time count), not 0-since-deploy** ŌĆö a cron with **any** historical beat is being invoked by Inngest, so a longer-than-window real cadence (daily `today-sync` w/ 99 beats, bursty `meta-capi-dispatch` w/ 494) is at most a freshness alert, **never** `never_fired`. Only a cron with **0 beats in all of history** whose deploy is **older than its `livenessWindowMs` (cadence + grace)** ŌåÆ **red `never_fired`** ("registered but has never run ŌĆö Inngest is not invoking it"), pages like any red. A genuinely-fresh cron (deploy < one window) or an unknown deploy age stays **amber "awaiting first run"**. Closes the `control-tower-monitor` "awaiting first run for days" gap; pairs with the [[control-tower-self-audit]] in-codeŌåöInngest-registered diff (same gap, registration side). **Beats-read-failure guard** ([[../specs/control-tower-beats-read-failure-guard]]): `buildControlTowerSnapshot` now captures the `control_tower_loop_beats` RPC **error** (not just `data`) ŌĆö a statement-timeout/500 returns `beats=null`, which would otherwise leave every cron with `everBeatCount=0`/`latest=null` and false-fire `never_fired`. When `beatsReadFailed` is true `evalCron` treats the zero as **UNKNOWN, not a true zero**: it suppresses both `never_fired` and `cron_freshness` reds and returns **amber "beat read unavailable ŌĆö status unknown"** ŌĆö the same conservative posture as the `deployAgeMs==null` guard.
- **Inline-agent assertions** ([[../specs/control-tower-agent-coverage]], `evalInlineAgent` + `fetchInlineAgentState`): event-driven AI agents have no cron cadence and no `agent_jobs` queue, so silence only matters when work exists. **liveness-when-work-exists** ŌĆö the loop's `inlineWorkSignal` (an independent upstream-demand count: unanalyzed closed-AI [[../tables/tickets]]; recent [[../tables/journey_sessions]]; recent [[../tables/orders]]; inbound-customer [[../tables/ticket_messages]] for `ai:orchestrator`) is > 0 AND **0 successful** beats in the window ŌåÆ red, `reason='idle_while_work'`. **error-rate** ŌĆö `errCount/total Ōēź errorRateThreshold` (default 0.5) once `total Ōēź minRunsForErrorRate` (default 5) ŌåÆ red, `reason='error_rate'`. Inline-agent beats are fetched separately (exact counts, true-latest-regardless-of-idle) so the agents' high beat volume can't crowd the cron read (and are excluded from the `control_tower_loop_beats` RPC).
- **Phase 2 output assertions** (`evalOutputAssertion(assertionId, loop, latest, inputs)`, layered on top of the P1 tile ŌĆö only escalates green/amber ŌåÆ red, a P1 red stays; a loop with `outputAssertions` runs each in order, first-to-fail wins): `fetchAssertionInputs(admin)` adds cheap read-only queries to the snapshot batch ŌĆö open routine-escalated [[../tables/tickets]] (`escalated_at` set, `escalated_to` null, not closed/archived) ŌĆö both the count AND the OLDEST waiting ticket's `escalated_at` (how long real work has actually waited) ŌĆö the latest `triage-escalations` + `spec-test` [[../tables/agent_jobs]] `created_at`, active overdue internal [[../tables/subscriptions]] (`next_billing_date` before today UTC), the latest renewal-cron beat (marks the live cycle boundary), the live + rolling-baseline renewal outcome breakdowns (`aggregateRenewalOutcomes`), and stuck [[../tables/dunning_cycles]] (`status='retrying'` with `next_retry_at` > 48h in the past). The five assertions: **escalation-idle** (the OLDEST waiting ticket has waited past the cadence grace [keyed off its `escalated_at`, not the last job's age] AND no triage job was created since it escalated ŌåÆ `reason='idle_while_work'`; re-based by [[../specs/control-tower-escalation-idle-grace]] so a healthy hourly cron isn't flagged in the normal gap between an escalation and the next tick), **spec-test-persisted** (beat reports `enqueued>0` but no spec-test job landed ŌåÆ `reason='false_success'`), **renewal-integrity** (N active internal subs overdue ŌåÆ `reason='renewal_integrity'`), **renewal-outcome-distribution** (the cron ran + each decline routed correctly, but the per-cycle outcome MIX is anomalous: a systemic rate Ōēź50% [hard floor: bad creds declining everyone / mass no-PM] OR a spike Ōēź2.5├Ś / +15pp vs the rolling baseline, gated by a Ōēź10-sample cycle + Ōēź50-sample baseline ŌåÆ `reason='renewal_outcome_distribution'`; [[../specs/control-tower-renewal-integrity-assertions]] P1), **stuck-dunning** (N dunning_cycles still 'retrying' >48h past `next_retry_at` ŌĆö a sub correctly mid-dunning within schedule is NOT flagged ŌåÆ `reason='stuck_dunning'`; same spec). A sixth, **migration-drift** ([[../specs/control-tower-migration-drift-check]] P1), is unlike the others ŌĆö it reads NO live DB state in the monitor: detection happens on the box (it parses the `.sql` files + diffs the live schema, which the deployed runtime can't) and the result rides the loop's heartbeat, so `evalOutputAssertion('migration-drift', ŌĆ”)` just reads `latest.produced.missing` (via `readMissingTables`) and flips red on a non-empty list (`reason='migration_drift'`). See `migration-drift.ts`. A violation flows through `runControlTowerMonitor` exactly like a P1 red (de-duped alert + page). Thresholds are tunable constants in monitor.ts (`RENEWAL_MIN_CYCLE_SAMPLE`, `RENEWAL_HARD_FLOOR_RATE`, `RENEWAL_SPIKE_FACTOR`, `RENEWAL_SPIKE_MARGIN`, `RENEWAL_MIN_BASELINE_SAMPLE`, `STUCK_DUNNING_GRACE_MS`).

## `error-feed.ts` ŌĆö the error feed (error-feed-monitoring Phase 1 + 2)

The capture + page + snapshot layer for the "hidden surfaces" ([[../specs/error-feed-monitoring]]) ŌĆö Vercel runtime errors, Inngest failed runs, app-layer Supabase errors, and (Phase 2) DB-level Supabase logs ŌĆö into [[../tables/error_events]].

- `recordError({ source, keyParts, title, detail?, sample?, occurrences? }, admin?)` ŌåÆ upserts a **grouped** [[../tables/error_events]] incident on `(source, signature)` (signature = a stable hash of the normalized `keyParts`), bumping `count`/`last_seen_at`. Pages owners ([[notify-ops-alert]]) on a **new signature or a re-firing spike**, rate-limited to one page per incident per 30 min. **Best-effort ŌĆö never throws**; a `23505` race falls back to the update path. On a **new signature** it also enqueues a [[repair-agent]] `repair` job ([[../specs/repair-agent]]) ŌĆö the diagnoseŌåÆpropose-fix trigger (deduped by signature, best-effort).
- `reportDbError(error, { op, table?, ŌĆ” }, admin?)` ŌĆö the **app-layer Supabase reporter**: a no-op on a null error, else `recordError({ source: "supabase", ŌĆ” })`. Call it anywhere code gets a non-null Supabase `{ error }` it would otherwise swallow (the scorecard-upsert class ŌĆö wired in `src/lib/meta/scorecards.ts`). See [[../operational-rules]] "don't swallow a Supabase error".
- `signatureFor(source, keyParts)` ŌåÆ the grouping key (normalizes out uuids/hex/numbers/quoted strings).
- `recordFeedDelivery(source, admin?)` / `feedLoopId(source)` ŌåÆ **liveness** ([[../specs/error-feed-honest-panels]]): write a lightweight "received a delivery" beat to [[../tables/loop_heartbeats]] under `loop_id='feed:<source>'`, `kind='feed'` ŌĆö proof the feed is wired + live, SEPARATE from error rows. Emitted on each clean Vercel drain POST (`/api/webhooks/vercel-logs`), each successful Supabase-logs poll (`pollSupabaseLogs`), and each `/api/client-errors` report **and** per-session heartbeat (`source='client'` ŌĆö [[../specs/client-error-capture]]). Best-effort, never throws.
- `buildErrorFeedSnapshot(admin?)` ŌåÆ `ErrorFeedSnapshot { generatedAt, panels: ErrorFeedPanel[] }` ŌĆö **READ-ONLY**, **connection-aware** ([[../specs/error-feed-honest-panels]]): last 7 days of [[../tables/error_events]] per source (5 panels: `vercel`, `inngest`, `supabase`, `supabase-logs`, `client`). Each panel carries `{ color, connectionState, configured, lastReceivedAt, statusText, hint, ŌĆ” }`. State machine (errors win, then connection): **errors present** ŌåÆ red Ōēż1 h / amber Ōēż24 h (unchanged); **not configured** (`VERCEL_LOG_DRAIN_SECRET` unset ┬Ę no stored Supabase token) ŌåÆ **amber "not configured"** + a how-to-wire `hint`, never green; **configured but 0 deliveries ever** ŌåÆ **amber "awaiting first event"**; **configured + received + clean** ŌåÆ **green "connected ┬Ę 0 errors (last delivery Ns ago)"** ŌĆö the only true green. Per-source receipt: `vercel`/`supabase-logs`/`client` read their `feed:<source>` beat (`client` is always-configured ŌĆö the reporters are wired in code ŌĆö so it's amber "awaiting" until the first storefront/portal report or heartbeat lands); `inngest` is proxied by the freshness of Inngest itself (latest `cron` beat ŌĆö the capture fn is failure-only, no clean delivery to observe); `supabase` (app-layer) needs no receipt (`reportDbError` is wired unconditionally). Used by the dashboard API (`GET /api/developer/control-tower`, merged in as `errorFeed`); the route also **folds the panel colors into the header health count** so an unconfigured (amber) panel is never counted healthy.
- Feeders: [[../inngest/inngest-failure-capture]] (`source='inngest'`), `/api/webhooks/vercel-logs` ([[../integrations/vercel-log-drain]], `source='vercel'` + `recordFeedDelivery('vercel')`), `reportDbError` call sites (`source='supabase'`), the [[../inngest/supabase-log-poll]] cron (`source='supabase-logs'` + `recordFeedDelivery('supabase-logs')`), and `POST /api/client-errors` (`source='client'` + `recordFeedDelivery('client')` ŌĆö fed by the storefront + portal reporters, [[client-error-reporter]] ┬Ę [[../specs/client-error-capture]]).

## `supabase-log-poll.ts` ŌĆö Management Logs poller (error-feed-monitoring Phase 2)

Pulls DB-level Supabase errors our app never sees ŌĆö Postgres/auth/API ŌĆö from the [[../integrations/supabase-management-logs]] `logs.all` endpoint into [[../tables/error_events]] (`source='supabase-logs'`). Driven by the [[../inngest/supabase-log-poll]] cron.

- `pollSupabaseLogs(admin?)` ŌåÆ `{ status: 'no-token'|'ok'|'error', incidents, rows, errors }` ŌĆö reads + decrypts the token, polls the `(last_polled_at, now]` window (Ōēż24h) across 3 source queries, **groups by `(source, signature)`** client-side, `recordError`s each, advances the cursor on partial success. **No-op (no-token)** until the owner pastes the token. Best-effort ŌĆö per-source failures collected in `errors`, never thrown.
- `getSupabaseLogConfig(admin?)` ŌåÆ `{ token, projectRef, lastPolledAt }` or null ┬Ę `isSupabaseLogPollConfigured(admin?)` ŌåÆ boolean ┬Ę `setSupabaseAccessToken(token, { projectRef? }, admin?)` (encrypt + upsert) ┬Ę `clearSupabaseAccessToken(admin?)` ┬Ę `projectRefFromEnv()` (parses `NEXT_PUBLIC_SUPABASE_URL`). Config: [[../tables/error_feed_supabase_config]] (single row, service-role only). Owner endpoint: `/api/developer/control-tower/supabase-token`.

## `migration-drift.ts` ŌĆö schema-vs-migrations drift check (control-tower-migration-drift-check Phase 1)

Catches a **silently-skipped migration**: a migration that never applied is invisible ŌĆö the code references a table, the table isn't there, every upsert hits `PGRST205`, and (pre-error-feed) it degrades silently. (Found 2026-06-22: `20260618140000_meta_performance_tables.sql` was skipped in the apply pipeline ŌåÆ the iteration engine ran on empty ROAS data for weeks.) The Control Tower's first **schema-shaped** check.

- **Pure parse + diff** (no fs/pg at module load ŌĆö the caller supplies the dir + a `fetchLiveTables` callback, so the LOGIC is testable and the box owns the raw-SQL read):
  - `parseExpectedTables(migrationsDir)` ŌåÆ `{ expected: Map<table, firstMigration>, dropped, parsedFiles }` ŌĆö every `*.sql` (lexical sort = apply order) parsed for the tables it CREATEs, **net of any later DROP** (drop-aware: create-then-drop = net-absent, not expected; a re-create revives it). Strips `--` + `/* */` comments and skips `temp`/`unlogged` tables; handles `if not exists`, `public.`-qualified, and `"quoted"` names.
  - `extractCreatedTables(sql)` / `extractDroppedTables(sql)` ŌĆö the per-file regex primitives.
  - `computeDrift(expected, liveTables, allowlist?)` ŌåÆ `{ missing, allowlistedMissing }` ŌĆö expected-but-absent tables, split by the allowlist.
  - `runMigrationDriftCheck({ migrationsDir, fetchLiveTables, allowlist? })` ŌåÆ `DriftResult { status:'ok'|'drift'|'skipped', missing, allowlistedMissing, expectedCount, liveCount, parsedFiles, reason? }`. `fetchLiveTables` returning **null** ŌćÆ `status:'skipped'` (no DB password on this host) ŌĆö an HONEST missing read, never a false "no drift". `driftSummary(r)` ŌåÆ a one-line log/detail string.
- `MIGRATION_DRIFT_ALLOWLIST = ["klaviyo_*"]` + `isAllowlisted(table, allowlist?)` ŌĆö **scope the noise** (spec): a sunset table (Klaviyo is being retired) that disappears lands in `allowlistedMissing` (surfaced for visibility) and NEVER alerts. Entries match exactly or as a `prefix*` wildcard. Keep it SHORT ŌĆö the right answer for a real missing table is to re-apply its migration.
- **Where it runs:** `scripts/builder-worker.ts` ŌåÆ `runMigrationDriftJob` (a periodic box job, ~30 min, in-flight-guarded so it never overlaps/stalls the 5s poll) reads the live `public` BASE TABLE names off the pooler (`information_schema.tables` ŌĆö not exposed via PostgREST, so raw `pg` via `poolerConnectionString()`) and writes the `DriftResult` into a `loop_heartbeats` beat (`loop_id = MIGRATION_DRIFT_LOOP_ID`, `kind:'cron'`, `produced.missing`, `ok = status!=='skipped'`). The deployed Next runtime can't do this ŌĆö the `.sql` files aren't bundled.
- **Surfacing** = the standard monitor path: `monitor.ts` `evalOutputAssertion('migration-drift', ŌĆ”)` reads `latest.produced.missing` (allowlisted ones already excluded box-side) and flips the tile **red** (`reason='migration_drift'`) ŌåÆ de-duped [[../tables/loop_alerts]] + page + [[repair-agent]] enqueue, exactly like any P1 red. Cron-freshness/never-fired keeps a *dead* check visible too.

## Gotchas

- **SHA-behind needs `VERCEL_GIT_COMMIT_SHA`** (the deployed commit) as the origin/main proxy; unset locally ŌćÆ the check is skipped (no false positive). It only fires red after `shaGraceMs` (default 30m) so an in-progress deploy / self-update never pages.
- **Agent-kind alert is off [[../tables/agent_jobs]], not the heartbeat** ŌĆö idle = green. The heartbeat only feeds last-ran/history.
