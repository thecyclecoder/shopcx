# libraries/control-tower

The Control Tower module ([[../specs/control-tower]] Phase 1 + Phase 2; [[../specs/control-tower-agent-coverage]] adds the inline AI agents; [[../specs/control-tower-complete-coverage]] Phase 1 adds **full cron coverage + the reactive event-driven agents**) — the registry, the heartbeat emit helper, and the monitor/snapshot logic (liveness + cron-freshness + stuck-jobs + inline/reactive-agent assertions + Phase 2 output assertions) that powers the [[../inngest/control-tower-monitor]] cron and the [[../dashboard/control-tower]] dashboard.

**Files:** `src/lib/control-tower/registry.ts` · `src/lib/control-tower/heartbeat.ts` · `src/lib/control-tower/monitor.ts` · `src/lib/control-tower/error-feed.ts` · `src/lib/control-tower/supabase-log-poll.ts` · `src/lib/control-tower/migration-drift.ts` · (Phase 2 detection layer: [[control-tower-self-audit]] — `self-audit.ts` · `src/lib/inngest/sync.ts` · `src/lib/inngest/registered-functions.ts`)

## `registry.ts` — the loop registry (code config)

The single source of truth for every loop the monitor watches. **Add a row here when you ship a new cron / worker / agent-kind** ([[../operational-rules]] "register-or-it's-incomplete").

- `type LoopKind = "worker" | "cron" | "agent-kind" | "inline-agent" | "reactive"` — `reactive` = an **event-driven Inngest function** (the inbound ticket handler, dunning, returns, journey-outcomes, agent-todo-execute, chargeback); it shares the inline-agent evaluation model (idle = green, liveness-when-work-exists + error-rate) and is split out only for the dashboard's "Reactive agents" group. ([[../specs/control-tower-complete-coverage]] P1.)
- `type OutputAssertionId = "escalation-idle" | "spec-test-persisted" | "renewal-integrity" | "renewal-outcome-distribution" | "stuck-dunning" | "migration-drift" | "segment-coverage"` — Phase 2 output assertions (see monitor.ts). The renewal pair ([[../specs/control-tower-renewal-integrity-assertions]] P1) completes the renewal-integrity trio; `migration-drift` ([[../specs/control-tower-migration-drift-check]] P1) reads a created-table-vs-live-schema diff conveyed in a box-emitted beat (see `migration-drift.ts` below); `segment-coverage` ([[../specs/fix-segment-refresh-coverage]] P2) polls the LIVE [[../tables/customers]] table each tick and reds when <95% of SMS-subscribed rows are refreshed within 26h OR any subscribed row is >48h stale — the alarm that would have caught the 2026-07 1000/138K whole-book regression on [[../inngest/refresh-customer-segments]]. **Run-in-progress grace** ([[../specs/segment-coverage-assertion-run-in-progress-grace]], signal `loop:refresh-customer-segments-cron`, verdict monitor-false-positive): the fresh-cohort ratio check is suppressed while the loop's own most recent beat is within `SEGMENT_COVERAGE_RUN_GRACE_MS` (6h — comfortably longer than the observed worst-case ~4-5h fanout across 138K rows), since a below-floor ratio mid-run is the expected shape of the daily walk still in progress, not a break. The stale48h check stays fully active during the grace (a subscriber >48h old is a genuine break even mid-run), and once the grace elapses the 95% floor re-engages and would still catch a truly stalled/partial refresh.
- `type InlineWorkSignalId = "tickets-awaiting-qc" | "journeys-awaiting-delivery" | "orders-awaiting-fraud-screen" | "tickets-awaiting-decision" | "tickets-awaiting-handler-dispatch"` — the read-only independent upstream-demand probes the inline-agent liveness-when-work-exists check evaluates.
- `type RenewalOutcome = "charged" | "skipped_no_payment_method" | "skipped_zero_total" | "declined_to_dunning" | "comp_shipped" | "comp_blocked" | "skipped_other"` + `RENEWAL_OUTCOME_LOOP_ID = "internal-subscription-renewal-outcome"` + `RENEWAL_BAD_OUTCOMES` (the anomalous subset: no-PM-skip, declined, comp-blocked) — the per-sub renewal outcome taxonomy the outcome-distribution assertion aggregates ([[../specs/control-tower-renewal-integrity-assertions]] P1). Every terminal path of `internal-subscription-renewal-attempt` emits ONE beat (loop_id = `RENEWAL_OUTCOME_LOOP_ID`, `kind:'reactive'`, `produced.outcome`) — the only uniform channel that captures SKIPS, which leave no transaction row. NOT a monitored tile (a data channel).
- `interface MonitoredLoop { id, kind, label, description, expectedCadence, livenessWindowMs?, shaGraceMs?, agentKind?, stuckThresholdMs?, outputAssertion?, outputAssertions?, inlineWorkSignal?, errorRateThreshold?, minRunsForErrorRate?, runsOnBox? }` — `outputAssertions?: OutputAssertionId[]` is for a loop carrying MORE THAN ONE assertion (the renewal cron: `["renewal-integrity", "renewal-outcome-distribution"]`); the monitor runs each in order, first-to-fail wins. Use it OR the singular `outputAssertion`. `runsOnBox?: boolean` marks a `kind:'cron'` that is **box-emitted** — detection/heartbeat runs on the box worker, not Inngest (e.g., `migration-drift-check` parses SQL migrations on the box, the db-health crons run detection passes on the box); when the box is stale/down, `evalCron` suppresses their freshness/never-fired/registered-not-firing violations (returns amber "waiting on box worker outage") since the cron can't fire without the box. See [[../specs/control-tower-suppress-box-cron-freshness-during-worker-outa]] P1.
- `MONITORED_LOOPS: MonitoredLoop[]` — the box worker (`box`), **the full Inngest cron set (50 crons** — each cron's inngest fn id + a cadence-derived `livenessWindowMs`, from minute-crons like `deliver-pending-sends`/`meta-capi-dispatch-cron` through the daily/weekly/yearly ones; [[../specs/control-tower-complete-coverage]] P1 closed the gap from the original 8**)**, 11 agent kinds (`agent:<kind>` + a per-kind `stuckThresholdMs`), **4 inline AI agents** (`ai:ticket-analyzer`, `ai:journey-delivery`, `ai:fraud-detector`, `ai:orchestrator` — each with a `livenessWindowMs`, optional `inlineWorkSignal`, and `errorRateThreshold`/`minRunsForErrorRate`), and **6 reactive event-driven agents** (`unified-ticket-handler` — THE inbound ticket pipeline, with `inlineWorkSignal: 'tickets-awaiting-handler-dispatch'` ([[../specs/control-tower-unified-handler-dispatch-workprobe]]) — plus `dunning-payment-failed`, `returns-process-delivery`, `journey-session-completed`, `agent-todo-execute`, `chargeback-received`; `kind: 'reactive'`, error-rate by default), the **`auto-merge-gate`** (`AUTO_MERGE_GATE_LOOP_ID`) — a `reactive` loop that runs inside the [[../integrations/github-webhook|GitHub webhook]] (the Auto-Ship Pipeline's Gate A, [[../specs/auto-ship-pipeline]] P1), error-rate only (a failed merge attempt beats ok:false; idle = green), and the **`auto-fold-gate`** (`AUTO_FOLD_GATE_LOOP_ID`) — a `reactive` loop that auto-folds fully-verified shipped specs (the Auto-Ship Pipeline's Gate B, [[../specs/auto-ship-pipeline]] P2; `autoFoldVerifiedSpecs` in [[spec-test-runs]], fired by the spec-test worker + the human-queue POST + the daily spec-test cron), error-rate only (a failed enqueue beats ok:false; idle = green). Output assertions: `triage-escalations-cron` → `escalation-idle`, `spec-test-cron` → `spec-test-persisted`, `internal-subscription-renewal-cron` → `["renewal-integrity", "renewal-outcome-distribution"]`, `dunning-payday-retry-cron` → `stuck-dunning`, `migration-drift-check` (box-emitted cron) → `migration-drift`. **Cadence-accuracy invariant** (signal `loop:ticket-analysis-cron`, verdict monitor-false-positive): each cron row's `expectedCadence` + `livenessWindowMs` MUST mirror the real Inngest `cron:` schedule of the function with that `id` — a window shorter than the cadence makes `evalCron` false-fire `cron_freshness` every cycle (`ticket-analysis-cron` was registered `*/5` + 15-min while the fn fires `*/30` → red for 15 of every 30 min between healthy beats; corrected to `*/30` + 90-min, matching its `─ Every-30-min crons (window ~90 min) ─` group). When a cron's schedule changes, update its registry row in the same PR.
- `WORKER_BOX_ID = "box"` (matches `scripts/builder-worker.ts`) · `agentLoopId(kind)` → `agent:<kind>` · `INLINE_AGENT_IDS = { ticketAnalyzer, journeyDelivery, fraudDetector, orchestrator }` (the `ai:<agent>` loop ids, shared with the agent files) · `AUTO_MERGE_GATE_LOOP_ID = "auto-merge-gate"` (the auto-merge gate's reactive loop_id, shared with [[github-pr-resolve]]) · `AUTO_FOLD_GATE_LOOP_ID = "auto-fold-gate"` (the auto-fold gate's reactive loop_id, shared with [[spec-test-runs]]).
- `MIGRATION_DRIFT_LOOP_ID = "migration-drift-check"` ([[../specs/control-tower-migration-drift-check]] P1) — the loop_id of the **box-emitted** migration-drift cron (a `kind:'cron'` MONITORED_LOOPS tile, owner platform, `outputAssertion:'migration-drift'`, `runsOnBox: true`, ~30 min window 90m). NOT an Inngest fn: the deployed runtime can't read the `supabase/migrations/*.sql` files, so `scripts/builder-worker.ts` runs the parse+diff on the box and beats here, carrying the missing tables in `produced`. Freshness keeps a dead check visible; the assertion flips the tile red on drift. When the box is unavailable the cron freshness alert is suppressed (amber "waiting on box worker outage" — [[../specs/control-tower-suppress-box-cron-freshness-during-worker-outa]] P1). (Not in the in-code↔Inngest self-audit diff — `auditCronCoverage` only flags Inngest crons *without* a tile, never a tile without an Inngest fn.)
- `INTENTIONALLY_UNMONITORED_CRONS: Record<id, reason>` (Phase 2) — cron `createFunction`s deliberately given no tile, so the [[control-tower-self-audit]] doesn't flag them as "unregistered loop: X". Silence is never the default: each needs a reason. Keep it short — the right answer for a real loop is a `MONITORED_LOOPS` row, not an exemption. **Current entries:** `slack-roadmap-notify` ("intentionally unmonitored — owner-confirmed via the [[coverage-register-agent]]") — surfaced by [[control-tower-self-audit]], proposed via the [[coverage-register-agent]]'s `exempt` path, and owner-confirmed rather than registered as a tile.
- **`MONITOR_TICK_FLOOR_MS` + `REGISTRY_LIVENESS_JITTER_GRACE` + `assertRegistryInvariants(loops)`** ([[../specs/monitor-cadence-scaled-liveness-window]] Phase 1 + Phase 2, codifies the **CEO 2026-07-11 monitoring-cost guardrail**) — a **build-time invariant** over `MONITORED_LOOPS` that runs at module import and throws on registration bugs the recurring `ticket-analysis-cron` / `storefront-experiments-refresh-cron` / `storefront-lever-decay-cron` / `security-dep-watch` one-off fixes never structurally prevented. **`MONITOR_TICK_FLOOR_MS = 5 * 60 * 1000`** (5 min) is the smallest cron cadence the registry accepts — matches the pinned [[../inngest/control-tower-monitor]] cron (`*/5 * * * *`), since a cron finer than the monitor's own tick can't be reliably alerted on. **`REGISTRY_LIVENESS_JITTER_GRACE = 1.2`** — every `livenessWindowMs` must be at least `cadenceMs * 1.2` (20% jitter slack), mirroring the existing pattern (hourly → 2h, every-30-min → 90m, daily → **30h**, weekly → **9d**, monthly → **37d**). `assertRegistryInvariants` iterates `kind:'cron'` loops with a parseable `expectedCadence`, parses the cron via `extractCronExpr` + `parseCronExpr` (extracted into `cron-parse.ts` to break the registry↔monitor cycle), computes `meanCadenceMsFromSets`, and: (1) throws NAMING `MONITOR_TICK_FLOOR_MS` if `cadenceMs < MONITOR_TICK_FLOOR_MS` ("widen the cadence or make this event-driven"), (2) throws NAMING the loop id + required window if `livenessWindowMs < cadenceMs * REGISTRY_LIVENESS_JITTER_GRACE` ("Widen the window to at least N min so cron_freshness doesn't false-fire"). Non-cron kinds + unparseable cadences (`"box job"`, `"per event"`) are skipped — the invariant is about SCHEDULED crons. The bootstrap block calls it unconditionally (no try/catch) — a regression is caught at the import site (`next build`, `tsx` script, `npx tsc --noEmit`) with a clear line-numbered error. Phase 2's widening pass corrected every existing offender: all daily windows 26h → **30h**, weekly 8d → **9d**, monthly 32d → **37d**, and all sub-5-min crons (`claude-status-poll-cron`, `deploy-guardian-cron`, `mario-stall-cron`, `deliver-pending-sends`, `marketing-text-campaign-send-tick`, `meta-capi-dispatch-cron`) widened from `* * * * *` to `*/5 * * * *` on BOTH the Inngest schedule AND the registry `expectedCadence` (their batch-drain / poll patterns tolerate the up-to-4-min extra latency). Fixture-tested in `src/lib/control-tower/registry.test.ts` (`npx tsx --test`). **A future daily cron gets a ~day window, not 5 min** — the assertion prevents the class of false-positive `cron_freshness` red the brain has been fixing one-off. Cross-linked from [[../../../CLAUDE.md]] § Local conventions.

## `heartbeat.ts` — end-of-run emit

Best-effort writes of one [[../tables/loop_heartbeats]] row (never throws).

- `emitLoopHeartbeat(loopId, kind, { ok?, produced?, detail?, durationMs? })`
- `emitCronHeartbeat(functionId, …)` — `kind:'cron'`, `loop_id` = the inngest fn id. Called by each monitored cron inside a `step.run("emit-heartbeat", …)` before its return.
- `emitAgentHeartbeat(agentKind, …)` — `kind:'agent-kind'`, `loop_id` = `agent:<kind>`. (The box worker writes its agent beats via its own inline `writeLoopHeartbeat` against its existing admin client, not this helper — same shape.)
- `emitInlineAgentHeartbeat(agentId, …)` — `kind:'inline-agent'`, `loop_id` = an `INLINE_AGENT_IDS` value (e.g. `ai:ticket-analyzer`, `ai:orchestrator`). Called in a **try/finally at the END of each run** of the 4 inline AI agents — `ok:true` on success/intentional-skip, `ok:false` on a thrown run or real failure (for `ai:orchestrator`, also a degraded/fallback decision).
- `emitReactiveHeartbeat(functionId, …)` — `kind:'reactive'`, `loop_id` = the inngest fn id (e.g. `unified-ticket-handler`). Called in a **top-level try/finally** inside each of the 6 reactive event-driven Inngest functions — `ok:true` on success/skip-return, `ok:false` on a thrown run. ([[../specs/control-tower-complete-coverage]] P1.)
- `emitRenewalOutcomeHeartbeat(outcome: RenewalOutcome)` — records ONE per-sub renewal outcome beat (`loop_id = RENEWAL_OUTCOME_LOOP_ID`, `kind:'reactive'`, `produced:{outcome}`, `ok:true`). Called from every terminal path of `internal-subscription-renewal-attempt` ([[../inngest/internal-subscription-renewals]]) — the only uniform channel that captures SKIPS (a no-PM / zero-total skip writes no transaction row). Uncaught errors aren't beat (a sub that errored never advances → caught by the renewal-integrity overdue assertion). ([[../specs/control-tower-renewal-integrity-assertions]] P1.)
- `aggregateRenewalOutcomes(admin, sinceIso, untilIso?)` → `RenewalOutcomeCounts` — READ-ONLY count breakdown of the outcome beats in a window (bounded `limit 5000`, best-effort → zeros on read failure). Used by the renewal cron (bakes the just-completed cycle's breakdown into its heartbeat's `produced.last_cycle_outcomes`) AND by the outcome-distribution assertion (live current cycle + rolling baseline).

## `monitor.ts` — snapshot + monitor

- `buildControlTowerSnapshot(admin?)` → `ControlTowerSnapshot { generatedAt, counts:{green,amber,red}, loops: LoopStatus[], selfAudit }`. **READ-ONLY**: one batched read of [[../tables/worker_heartbeats]], [[../tables/loop_heartbeats]] (the **`control_tower_loop_beats(p_history_limit)` RPC** — latest ≤10 beats **per loop_id** for cron + agent-kind kinds, **excluding `inline-agent`/`reactive`**; a **lateral join** over the distinct loop_ids rides the `(loop_id, ran_at desc)` index, reading ≤N rows per loop. **Presence is the ever-beaten signal** — a loop with zero beats is absent from the result ⇒ never_fired candidate (no all-time count is returned). Replaced the old global "last 600 beats" window — and then the per-row `count(*) OVER`/`row_number() OVER` body ([[../specs/control-tower-loop-beats-rpc-perf]]), both of which scanned + sorted the whole growing table and 500-ed on statement timeout), open [[../tables/loop_alerts]], active [[../tables/agent_jobs]], plus `fetchInlineAgentState` (per inline agent: its work probe + exact ok/errored window counts + latest + history) **and `buildCoverageAudit()`** ([[control-tower-self-audit]], Phase 2); evaluates each loop to a `LoopStatus { color, statusText, lastRanAt, lastProduced, detail, violation, history, openAlert }`. The `selfAudit` findings (unregistered crons + in-code↔Inngest-registered gaps) are folded into the honest amber `counts`. Used **verbatim** by the dashboard API.
- `runControlTowerMonitor()` → `MonitorResult`. Builds the snapshot, then **acts**: opens a de-duped [[../tables/loop_alerts]] incident on each newly-red loop (paging owners via [[../libraries/notify-ops-alert]]), bumps `last_seen_at` while still red (no re-page), and resolves on recovery. On a **newly-opened** alert it also enqueues a [[repair-agent]] `repair` job (signature `loop:<id>`) — the second diagnose→propose-fix trigger ([[../specs/repair-agent]]). It also drives the [[coverage-register-agent]]: for each `selfAudit.unregistered` loop it calls `enqueueCoverageRegisterJob` (best-effort), which proposes the inferred `MONITORED_LOOPS` entry for one-tap owner Build ([[../specs/coverage-auto-register-agent]] P1). Called only by the cron. **Environment-scoped (`isCanonicalProductionDeploy` = `VERCEL_ENV==="production"`):** the act phase — all [[../tables/loop_alerts]] writes + paging + Repair/coverage enqueues — runs **only on the canonical production deploy**. A preview/branch deploy carries a branch-local `MONITORED_LOOPS` (e.g. a cron that exists only on an unmerged WIP branch), so it still evaluates the read-only snapshot for its own dashboard but returns before any write — otherwise it leaks phantom `registered_not_firing` alerts into the shared prod feed and wakes the Repair Agent for loops absent from HEAD ([[../specs/control-tower-loop-alerts-prod-scoped]], the `loop:claude-status-poll-cron` incident). **Snapshot-vs-write race guard on `cron_freshness` opens** ([[../specs/control-tower-cron-freshness-recheck-latest-beat-before-open]], signal `loop:portal-auto-resume-cron`, verdict monitor-false-positive): the snapshot's beats read happens ~ms before the fresh-alert insert, so on a cron whose `livenessWindowMs` is ~2× its cadence the next scheduled firing's heartbeat can land in the same second the alert row is being written (`portal-auto-resume-cron`: the 20:15 UTC beat wrote at 20:15:12.049, the snapshot-based alert would have written at 20:15:12.281 → false page). Right before the fresh-open `loop_alerts` insert, when `violation.reason === 'cron_freshness'`, the actLoop re-reads the single most recent `loop_heartbeats` row for that `loop_id` (`.select('ran_at').eq('loop_id', loop.id).order('ran_at', {ascending: false}).limit(1).maybeSingle()`); if `Date.now() - new Date(fresh.ran_at).getTime() <= livenessWindowMs` (from the loop's `MonitoredLoop` def, fallback 26h) the insert is skipped — the cron already recovered mid-tick. A truly stale cron still has no fresh beat so the assertion is unchanged for real failures, and the already-open branch (`last_seen_at` bump) is untouched. One extra single-row read per `cron_freshness` candidate — effectively free.

### Assertion helpers

- `countRenewalIntegrityOverdueSubs(admin: Admin, startOfTodayIso: string, latestRenewalCronBeatIso?: string | null): Promise<number>` — returns the count of active internal subscriptions with `next_billing_date` before today UTC, **excluding subscriptions whose `updated_at` is AFTER the latest renewal-cron beat** (post-cron-activation grace — subscriptions that changed state after the cron already ran were not eligible, so the cron cannot be blamed for missing them), **and excluding subscriptions currently in an active/retrying/open/paused/skipped dunning cycle** (joined via `subscription_id` per [[../../../CLAUDE.md]] UUID-join rule). Used by the `renewal-integrity` output assertion to distinguish a healthy payment-failed subscription waiting for its dunning retry from a genuinely-missed renewal. The spec-test sandbox is excluded from both sides of the join via `.neq("workspace_id", SPEC_TEST_SANDBOX_WORKSPACE_ID)`. ([[../specs/control-tower-renewal-integrity-exclude-active-dunning]] P1; [[../specs/control-tower-renewal-integrity-post-cron-activation-grace]] P1 — post-cron grace filtering.)

- `ticketAnalyzerEligibilityReadyAt(args: { closedAtMs, aiHandledAtMs, solHandledAtMs, latestCustomerMessageAtMs, coraSettleMs }): number | null` — exported pure helper used by the `tickets-awaiting-qc` work probe ([[../specs/ticket-analyzer-workprobe-eligibility-grace]]) to compute when a ticket becomes truly eligible for the ticket-analyzer to process. Returns the **MAX of three simultaneous anchors** the cron's `passesCoraSelectionGate` requires: (1) `closedAtMs` (ticket must be closed), (2) the later of `aiHandledAtMs` or `solHandledAtMs` (ticket must be handled), and (3) `latestCustomerMessageAtMs + coraSettleMs` (customer message must be settled). Returns `null` if any anchor is missing — the ticket is not eligible at all and the probe skips it. The probe uses this to compute the true moment a ticket becomes gradeable, then only counts it once `(now - readyAt) >= TICKET_ANALYSIS_FEEDER_GRACE_MS`, giving the 30-min feeder cron a full cycle to process it before flagging it as awaited work. Mirrors the cron's real eligibility requirements exactly (zero mismatches) — a freshly-closed ticket on an old customer message stays green through the between-tick gap, while a genuinely-stalled analyzer still alerts after a full cycle has passed.

- `fetchShaDirection(deployed: string, running: string)` → `{ direction: ShaDirection, firstDivergentAt: string | null }` ([[../specs/control-tower-box-behind-elapsed-anchored-to-drift-not-uptim]] Phase 1) — determines if the box worker's `running_sha` is behind origin/main. Returns the direction (`"same"` / `"worker-behind"` / `"worker-ahead"` / `"unknown"`), plus the ISO timestamp of the **first commit past the running SHA** when the GitHub compare API reports status=`"ahead"` (i.e. exactly when origin/main advanced past the worker's running SHA). This `firstDivergentAt` timestamp lets the box-liveness assertion anchor its shaGrace grace window to when drift began, not when the worker booted — so a long-lived worker that remains idle gives the full 30-min grace from first divergence, instead of false-firing on the very next commit just because the worker has been up longer than the grace period. When the compare API returns any other status or is unavailable, `firstDivergentAt` is `null` (fallback to uptime-based elapsed measurement).

- Evaluators: `evalWorker` (liveness + SHA-behind via **drift-elapsed-anchored shaGrace** + **all-Max-accounts-capped**: reads the box heartbeat's `accounts.all_capped` → **amber** "all Max accounts capped — builds parked, auto-resume" [NOT red — `blocked_on_usage` auto-resumes at reset, no manual rebuild], so a green box tile can't hide a throughput stall where every account hit its 5-hour wall; per-account load rides the tile's `lastProduced` — [[../specs/box-multi-account-failover]] P2). **Drift-elapsed-anchored shaGrace** ([[../specs/control-tower-box-behind-elapsed-anchored-to-drift-not-uptim]]): when the worker is idle and behind origin/main, the "behind too long" red is triggered by `Math.min(uptimeElapsedMs, driftElapsedMs) > shaGraceMs` (default 30 min), where `driftElapsedMs` = `firstDivergentAt ? ageMs(firstDivergentAt) : ∞` — i.e. the elapsed time since the first-divergent commit (GitHub's compare API result), or infinite (fallback to uptime) when unknown. So a fresh commit landing on a long-lived worker stays green/amber for the full shaGrace from first divergence, preventing false-reds on healthy workers whose uptime exceeds the grace. The red violation detail reports the accurate drift-age when known (e.g. "behind for 45m"), or worker uptime as fallback when drift timestamp is unavailable.

`evalCron` (freshness + **never-fired** + **box-emitted-cron-worker-outage suppression**: [[../specs/control-tower-suppress-box-cron-freshness-during-worker-outa]] P1 — when `isWorkerUnavailable()` is true, box-emitted crons (marked with `runsOnBox: true`) return amber "waiting on box worker outage" instead of red on `cron_freshness` / `never_fired` / `registered_not_firing` violations; non-box crons page normally; the `isBoxEmittedCronLoop(loop)` predicate checks `loop.kind === 'cron' && loop.runsOnBox === true`; four crons run on the box: `migration-drift-check` [[../specs/control-tower-migration-drift-check]], `db-health-slow-query` / `db-health-instance-saturation` / `db-health-size-sweep` [[../specs/db-health-detector]] — all feed by the box worker's detection pass, so when the worker is stale they're meaningless until it recovers), `evalAgentKind` (stuck jobs + **worker-unavailable suppression**: [[../specs/control-tower-suppress-agent-stuck-during-worker-outage]] P1 — when `isWorkerUnavailable()` is true, `queued` and `queued_resume` jobs are filtered out of the stuck check since they're waiting on the worker to recover, not a lane-specific failure; `building` and `claimed` jobs still trigger red since they're in-flight work the worker already claimed), `evalInlineAgent` (inline AI agents **and** the `reactive` event-driven agents — both routed through it, and both excluded from the `control_tower_loop_beats` RPC / sourced from `fetchInlineAgentState`). Genuinely-idle/healthy → green; a freshly-shipped cron with no beat yet → amber (never a false red).
- **`evalCron` never-fired detection** ([[../specs/control-tower-complete-coverage]] Phase 2, accuracy-fixed by [[../specs/control-tower-monitor-accuracy]] Phase 1): a cron with **0 heartbeats** is no longer amber-forever. `deployRefAgeMs(worker)` gives a trustworthy "how long has this deploy been live" lower bound — the box worker self-updates to origin/main + restarts on a new SHA, so once `worker.running_sha` matches `VERCEL_GIT_COMMIT_SHA`, `worker.started_at` ≈ deploy time (returns null = unknown ⇒ stay amber, never a false red). **never-fired keys off whether the loop has EVER beaten (`everBeatCount` — now presence in the lateral-join read, not an all-time count), not 0-since-deploy** — a cron with **any** historical beat is being invoked by Inngest, so a longer-than-window real cadence (daily `today-sync` w/ 99 beats, bursty `meta-capi-dispatch` w/ 494) is at most a freshness alert, **never** `never_fired`. Only a cron with **0 beats in all of history** whose deploy is **older than its `livenessWindowMs` (cadence + grace)** → **red `never_fired`** ("registered but has never run — Inngest is not invoking it"), pages like any red. A genuinely-fresh cron (deploy < one window) or an unknown deploy age stays **amber "awaiting first run"**. **registered-but-not-firing** ([[../specs/spec-drift-reconcile-not-firing]] P1): the deploy-anchored `never_fired` misses a cron that has been registered for **weeks but never fires once**, because the box self-updates + restarts on every ship so `deployRefAgeMs` keeps RESETTING under it — with frequent deploys its current-deploy clock rarely exceeds the window, so the dead cron hides forever (the real `spec-drift-reconcile` failure: registered with Inngest, `diffInngestRegistered` says `missing:[]`, but its `20,50 * * * *` schedule never executed → 0 `loop_heartbeats` beats ever). So a SECOND, **deploy-surviving** reference: `monitorUptimeMs` = the age of the **oldest `control-tower-monitor` beat** (how long the watchdog itself has been continuously running — beat retention can only shorten it, never inflate it ⇒ conservative). A cron with **0 beats ever** while the watchdog has itself been beating for **longer than the cron's window** → **red `registered_not_firing`** ("registered with Inngest but not firing — its cron schedule isn't active in the prod env; re-sync the app"). Distinct from the [[control-tower-self-audit]] Inngest-registration diff (never-*registered*): this fn IS in the registered set, the SCHEDULE just isn't active. **Newly-added-cron grace** ([[../specs/control-tower-registered-not-firing-newcron-grace]]): `monitorUptimeMs` is the watchdog's OWN run-span, independent of when a given cron was ADDED — so a long-cadence cron shipped AFTER the watchdog passed its window would false-trip on day one, hours before its first scheduled tick (signal `loop:storefront-experiments-refresh-cron`). The optional **`MonitoredLoop.registeredAt`** (an ISO code constant — deploy-SURVIVING, unlike `deployRefAgeMs`) gives a per-loop "how long has THIS cron been registered" reference: `registered_not_firing` now requires `(now - firstScheduledFiringAtOrAfter(registeredAt)) > window` **in addition to** `monitorUptimeMs > window`. A cron registered for less than a full window stays **amber "awaiting first run"**, graced just like the deploy-anchored `never_fired`. Each newly-added long-cadence cron must carry its own `registeredAt` to claim the grace — the same false positive recurred on a sibling daily cron ([[../specs/control-tower-registered-not-firing-new-cron-grace]], signal `loop:storefront-lever-decay-cron`), fixed by stamping that loop's `registeredAt` too. **First-firing anchor** ([[../specs/control-tower-cron-grace-uses-next-firing-after-registration]], signal `loop:security-dep-watch`, verdict monitor-false-positive): the grace clock can't be `registeredAt` itself — a daily cron whose `registeredAt` falls BEFORE its hour-of-day (e.g. `security-dep-watch` `0 4 * * *` with `registeredAt` 00:00 UTC + a deploy that landed at 04:08 missed that day's 04:00 tick) would false-page 26h after midnight, 2h before the cron's first real chance to fire (the next day's 04:00). `monitor.ts` parses the cron expression from `expectedCadence` (e.g. `"daily (0 4 * * *)"` → `0 4 * * *`) once at module load via `extractCronExpr` + `parseCronExpr` + `nextFiringAtOrAfter`, computes the FIRST scheduled firing at-or-after `registeredAt`, memoizes it per `loop.id`, and gates the grace on that. Box-job cadences (`"every ~30 min (box job)"`) have no Inngest schedule and fall back to `registeredAt` (preserves prior behavior). Unit-tested in `monitor.test.ts` (`npx tsx --test src/lib/control-tower/monitor.test.ts`). Unset (legacy crons) ⇒ no extra gate (registered long ago). `null` monitor uptime (bootstrapping / pruned) ⇒ stay amber. Closes the `control-tower-monitor` "awaiting first run for days" gap; pairs with the [[control-tower-self-audit]] in-code↔Inngest-registered diff (same gap, registration side). **Beats-read-failure guard** ([[../specs/control-tower-beats-read-failure-guard]]): `buildControlTowerSnapshot` now captures the `control_tower_loop_beats` RPC **error** (not just `data`) — a statement-timeout/500 returns `beats=null`, which would otherwise leave every cron with `everBeatCount=0`/`latest=null` and false-fire `never_fired`. When `beatsReadFailed` is true `evalCron` treats the zero as **UNKNOWN, not a true zero**: it suppresses both `never_fired` and `cron_freshness` reds and returns **amber "beat read unavailable — status unknown"** — the same conservative posture as the `deployAgeMs==null` guard.
- **Inline-agent assertions** ([[../specs/control-tower-agent-coverage]], `evalInlineAgent` + `fetchInlineAgentState`): event-driven AI agents have no cron cadence and no `agent_jobs` queue, so silence only matters when work exists. **liveness-when-work-exists** — the loop's `inlineWorkSignal` (an independent upstream-demand count: unanalyzed closed-AI [[../tables/tickets]]; recent [[../tables/journey_sessions]]; recent [[../tables/orders]]; inbound-customer [[../tables/ticket_messages]] for `ai:orchestrator`) is > 0 AND **0 successful** beats in the window → red, `reason='idle_while_work'`. The `tickets-awaiting-qc` probe carries a **feeder-cadence grace** (`TICKET_ANALYSIS_FEEDER_GRACE_MS` = 40 min, [[../specs/ticket-analyzer-workprobe-cron-grace]]): since the analyzer is fed by `ticket-analysis-cron` every 30 min, a closed AI ticket only counts once its true eligibility is older than the grace — i.e. it has survived a full feeder cycle still unprocessed — so a ticket closing between the 30-min ticks doesn't false-fire `idle_while_work` ~15 min before any cron cycle could service it (a genuinely-stuck analyzer still trips it a cycle later). **Eligibility helper** (`ticketAnalyzerEligibilityReadyAt`, [[../specs/ticket-analyzer-workprobe-eligibility-grace]]): the probe's true eligibility gate is NOT a single timestamp (e.g. `updated_at` alone or `last_customer_message_at` alone), but the **MAX of three simultaneous anchors the cron's `passesCoraSelectionGate` requires**:
  1. `closed_at` (the ticket must be closed)
  2. the later of `ai_handled_at` or `sol_handled_at` (the ticket must be handled)
  3. `last_customer_message_at + CORA_SETTLE` (the settle window on customer activity)
  A ticket with any missing anchor (no customer message, no handled stamp, no close stamp) returns `null` — not eligible at all. The prior probe used only the customer message's settle window as the eligibility clock, so a freshly-closed ticket on an old message (e.g. customer messaged an hour ago, ticket closed 2 minutes ago) was already "past cutoff" and counted as overdue work — even though the ticket literally became gradeable a moment earlier, before any 30-min cron tick could have picked it up, manufacturing a false `idle_while_work` on loop:ai:ticket-analyzer during the between-tick gap. `ticketAnalyzerEligibilityReadyAt` computes the real earliest moment the cron would even *consider* the ticket for analysis. It also **mirrors the cron's `.eq("analyzer_locked", false)` source filter** ([[../specs/ticket-analyzer-workprobe-exclude-analyzer-locked]], signal `loop:ai:ticket-analyzer`): a human-vetoed row ([[../tables/tickets]] `analyzer_locked=true`, Phase 2 of [[../specs/human-directives-hard-gates-over-ticket-ai]]) stays closed / `ai`-tagged / `last_analyzed_at=null` forever by design — the cron will never select it, so counting it here manufactures a false `idle_while_work` on the analyzer tile during otherwise-quiet windows. It also **mirrors the cron's `.not("closed_at", "is", null)` + `.not("sol_handled_at", "is", null)` source filters** ([[../specs/journey-completion-stamps-closed-at-so-cora-can-grade]] Fix 1, signal `loop:ai:ticket-analyzer`): a closed AI ticket whose close writer forgot to stamp `closed_at` (the origin bug — a journey completion route shipping status='closed'+resolved_at with `closed_at` null) is invisible to the cron even after the Phase-1 code fix stops the source, so counting the pre-existing malformed row as work manufactures a false `idle_while_work` red tile that never clears; same for `sol_handled_at`-null rows the cron will never select. The probe uses `ticketAnalyzerEligibilityReadyAt({ closedAtMs, aiHandledAtMs, solHandledAtMs, latestCustomerMessageAtMs, coraSettleMs })` to compute the ready time, then only counts a candidate once `(now - readyAt) >= TICKET_ANALYSIS_FEEDER_GRACE_MS` — giving the next feeder cycle a fair chance before flagging the ticket as awaited work. The alignment is pinned by `src/lib/control-tower/tickets-awaiting-qc-workprobe.regression.test.ts` on the probe AND the cron so a relaxation on either side red-lights loudly. Standing pattern for this class of monitor false-positive (siblings: [[../specs/ticket-decision-workprobe-exclude-positive-close]], [[../specs/ticket-decision-workprobe-exclude-active-playbook]]) — mirror the source filter at the probe, not a JS post-filter or a grace bump. The `tickets-awaiting-handler-dispatch` probe (for `unified-ticket-handler` only, [[../specs/control-tower-unified-handler-dispatch-workprobe]]) is the handler's OWN work signal based on **durable dispatch intents**. Every ingest chokepoint routes through [[../inngest/dispatch-inbound-message]] `dispatchInboundMessage`, which stamps `dispatch_pending_at = now` on the just-inserted inbound `ticket_messages` row BEFORE firing `ticket/inbound-message`. The counterpart in [[../inngest/unified-ticket-handler]] `clearDispatchIntent` clears the stamp at the TOP of every claimed run (regardless of disposition — real turn, ai_disabled skip, empty inbound, spam), so an un-cleared stamp survives ONLY when the handler never received the event. Aging that stamp past `HANDLER_DISPATCH_SETTLE_MS` (mirrors `INTENT_SETTLE_MS` in [[../inngest/unanswered-inbound-backstop-cron]], drift-pinned by a regression test `tickets-awaiting-handler-dispatch-workprobe.regression.test.ts`) turns it into an unambiguous LOST handler dispatch — the exact signal `loop:unified-ticket-handler` is supposed to alert on. Why the split from the old shared `tickets-awaiting-decision` (which counts inbound customer messages in-window and subtracts several bypass classes): that probe measures the AI orchestrator's upstream demand, which is a strict superset of the handler's. A raw inbound row that was NOT stamped (because its ingest path deliberately bypassed the handler — CSAT reopen inserts, sentinel merges) still counted as "handler work" under the old probe and could fire `idle_while_work` on a quiet-window inbound that never should have invoked the handler at all. Keying the probe on `dispatch_pending_at` eliminates that class at the source: no stamp ⇒ no work ⇒ no alert. The filter: `dispatch_pending_at IS NOT NULL` AND `dispatch_pending_at <= (now - HANDLER_DISPATCH_SETTLE_MS)` AND `direction='inbound'` AND `author_type='customer'`. Non-dispatched raw inbounds (rows created by paths that did NOT go through `dispatchInboundMessage`) carry NO stamp and are NOT counted, so the tile can't false-page on a customer message that never should have invoked the handler.

The `tickets-awaiting-decision` probe (now for `ai:orchestrator` only) counts only inbound-customer [[../tables/ticket_messages]] that DRIVE an `ai:orchestrator` beat, **gated by a settle window** (`TICKET_DECISION_SETTLE_MS` = 3 min, mirrors `HANDLER_DISPATCH_SETTLE_MS` [[../specs/ticket-decision-workprobe-settle-and-outreach-bypass]] Phase 1) — a fresh inbound's `created_at` must be older than `now - TICKET_DECISION_SETTLE_MS` before it counts as orchestrator demand. This waits through the deterministic outreach lane's async dispatch boundary (tag application + handler close + job enqueue) before counting a cold pitch / brand outreach inbound as work (the monitor-false-positive race that flipped `loop:ai:orchestrator` red on fresh Flippa-style outreach when a single such message landed in a quiet window; a genuinely-fresh customer inbound in an otherwise-quiet window has 3 min to form its own fresh-send `ok:true` beat, but the first check _after_ the settle window still alerts on a stalled orchestrator). The settle grace is applied to **`MAX(msg.created_at, ticket.created_at)`**, not `msg.created_at` alone ([[../specs/ticket-decision-workprobe-grace-merge-remapped-inbounds-by-t]], PR #2138, signal `loop:ai:orchestrator`, verdict monitor-false-positive): ticket auto-merge ([[../inngest/unified-ticket-handler]] § 1a `mergeTickets` picks the NEWEST ticket as the target) remaps inbound [[../tables/ticket_messages]] rows from a customer's older tickets onto it via `.update({ ticket_id: target.id })` while keeping their ORIGINAL `created_at`, so a 16-minute-old inbound can end up parented to a 13-second-old ticket. Without the ticket-side cutoff that remapped message reads as "past the settle boundary" on its own timestamp alone and counts as orchestrator demand before the classifier + first-touch dispatch have had a chance to run on the FRESH parent ticket — a healthy remapped inbound races the same pre-orchestrator gates as a raw inbound on a brand-new ticket. Every count query therefore filters on `tickets.created_at <= decisionSettleCutoffIso` (adding `tickets!inner(id)` to the queries that don't already join `tickets`, so PostgREST binds the nested-column predicate instead of silently dropping it): a message counts only when BOTH its own timestamp AND its current parent ticket's timestamp clear the cutoff. The filter is **symmetric** across `allRes` / `excludedRes` / both Sol first-touch subtractions — an asymmetric application would leave a message in the total-count residual after its counterpart was pruned upstream — and it never leaks into a false negative, since a genuine outage that dispatches a real inbound onto an older ticket still counts and still fires `idle_while_work`. Pinned by source-inspection tests in `src/lib/control-tower/monitor.test.ts` that assert the ticket-side cutoff count equals the msg-side cutoff count. It subtracts messages whose ticket is **closed OR `csat:reopened`-tagged OR `outreach`-tagged OR `cls:outreach`-tagged OR has an `active_playbook_id`** (a single positive-match `.or(...)` over the joined ticket) — plus a **parallel** subtraction for the Sol first-touch class, expressed as two channel-scoped signals: (a) messages whose ticket has a [[../tables/ticket_resolution_events]] row with `reasoning='sol_first_touch_ack'` (the chat ack ledger — only written on chat), and (b) messages whose ticket has an [[../tables/agent_jobs]] row with `kind='ticket-handle'` and `instructions.reason='first_touch'` (the channel-agnostic dispatch signal — written for chat AND every async channel). Five exclusions: (1) the CSAT-reopen path (`/api/csat/[ticketId]`, [[../specs/control-tower-ticket-decision-workprobe-scope]]) inserts an inbound customer message + reopens + routes to a human but emits NO `ticket/inbound-message` event, so the handler never beats on it; (2) the pre-orchestrator short-circuit gates ([[../specs/ticket-decision-workprobe-exclude-positive-close]]) — positive-close (`unified-ticket-handler.ts:1635`), fraud-gate, chargeback-gate — close the ticket BEFORE `callSonnetOrchestratorV2`, so those inbound messages never drive a beat either; (3) active-playbook continuations ([[../specs/ticket-decision-workprobe-exclude-active-playbook]]) — a customer mid-playbook (refund/cancel/return) routes through `executePlaybookStep` and never reaches `callSonnetOrchestratorV2`, so the orchestrator legitimately emits no beat (two such inbounds in a quiet 120m window were enough to flip the `ai:orchestrator` tile red, signal `loop:ai:orchestrator`); (4) deterministic outreach dispatch ([[../specs/ticket-decision-workprobe-settle-and-outreach-bypass]] Phase 1) — the `outreach` and `cls:outreach` tags (always stamped together by [[../inngest/unified-ticket-handler]] § 1a automated-sender and § 1c outreach-classifier routes) mark a customer message routed through the deterministic outreach lane **instead** of `callSonnetOrchestratorV2`, so no `ai:orchestrator` beat is emitted; (5) Sol first-touch dispatch ([[../specs/ticket-decision-workprobe-exclude-sol-first-touch]] + [[../specs/ticket-decision-workprobe-exclude-async-sol-first-touch]], sibling of [[../specs/sol-ticket-direction-artifact-and-first-touch-box-session]]) — a first-touch inbound on a channel with `sol_first_touch_enabled=true` is served by a Sol `ticket-handle` `agent_job`; `unified-ticket-handler.ts` § 2b acks the customer on chat + writes a [[../tables/ticket_resolution_events]] row with `reasoning='sol_first_touch_ack'` (chat only — the ack send + ledger row are SKIPPED on async channels like email/SMS/portal because Sol's real reply is the sole first-touch customer message), and in every case enqueues the box worker via a [[../tables/agent_jobs]] row (`kind='ticket-handle'`, `instructions.reason='first_touch'`) that authors the Direction + first reply, so `callSonnetOrchestratorV2` never runs and no `ai:orchestrator` beat is emitted (on Superfoods every channel has the flag on, so one quiet-window Sol first-touch inbound was enough to flip the tile red — signal `loop:ai:orchestrator`, verdict monitor-false-positive; the originating page was a plain inbound customer EMAIL, which is why the async dispatch signal (b) was added on top of the chat-only ack signal (a) — the ack ledger row wasn't written for that email, so the ack-only exclusion subtracted nothing). A lone reopen note, a 'Thank you' positive-close, a playbook step reply, a cold outreach pitch, or a Sol-served chat OR async first-touch in an otherwise-quiet window used to read as work=1 / 0 beats and false-fire `idle_while_work`. A still-OPEN, no-playbook, no-Sol-first-touch, no-outreach ticket with no beat keeps counting, so a genuine orchestrator outage (inbound piling up on tickets nothing closes or hands to a playbook / Sol / outreach) still alerts; a ticket the orchestrator served + closed has its own ok beat, so dropping it never manufactures a false negative. The first four run as a single NULL-safe positive-match `.or(...)` subtraction (not a negated array filter, and overlap-free — a csat:reopened ticket that later closes is counted once) so NULL/empty-tag, NULL-`active_playbook_id` tickets always count; the Sol-first-touch class lives on sibling tables so its (a) chat-ack signal runs as a **parallel** `tickets!inner(ticket_resolution_events!inner(id))` join with `eq('tickets.ticket_resolution_events.reasoning','sol_first_touch_ack')`, and its (b) dispatch signal runs as a **parallel** `.eq('kind','ticket-handle').like('instructions','%"reason":"first\_touch"%')` query on [[../tables/agent_jobs]] whose extracted ticket-id set (via the pure `extractSolFirstTouchDispatchTicketIds(rows)` helper — [[../tables/agent_jobs]] has no `ticket_id` column, so ticket_ids are parsed out of the JSON-encoded `instructions` text) drives a `.in('ticket_id', ids)` count over the message window; overlap across exclusions (2), (4), (5a), and (5b) can double-subtract into `Math.max(0, …)`, which only lowers the work count further — never inflates it — so the tile still cannot false-fire and a genuinely orchestrator-owned ticket sits in none of the sets. The `orders-awaiting-fraud-screen` probe (for `ai:fraud-detector`) mirrors the same feeder-surface rule ([[../specs/control-tower-fraud-detector-workprobe-exclude-internal-renewals]], signal `loop:ai:fraud-detector`, verdict monitor-false-positive): it counts recent [[../tables/orders]] but **excludes rows whose `source_name` matches one of the two internal-renewal markers** — `internal_subscription_renewal` (the regular subscription-renewal order path in [[../inngest/internal-subscription-renewals]]) and `internal_subscription_comp_renewal` (the $0 comp renewal marker). Those orders are created by the billing renewal cron and by design never emit `fraud/order.check`, so [[fraud-detector]] `checkOrderForFraud` never runs on them; counting one as awaited work made a healthy detector look silent (work=1 / 0 beats → false `idle_while_work` red tile for Platform on quiet renewal-only windows). Real Shopify webhooks pass upstream `source_name` through and DO fire the fraud gate ([[shopify-webhooks]]); the internal storefront route (`src/app/api/checkout/route.ts`) stamps `source_name='storefront'` and calls `checkOrderForFraud` directly — both stay in the count, plus NULL-`source_name` orders (defensive: unknown-source rows are still treated as real fraud-detector work). The exclusion set is exported as `INTERNAL_RENEWAL_ORDER_SOURCE_NAMES` + the pure `isOrderAwaitingFraudScreen(order)` predicate so the DB probe filter (`.or('source_name.is.null,source_name.not.in.(…)')`) and the unit test share one definition — same standing pattern as the ticket probes above (mirror the source filter at the probe, not a JS post-filter). **error-rate** — `errCount/total ≥ errorRateThreshold` (default 0.5) once `total ≥ minRunsForErrorRate` (default 5) → red, `reason='error_rate'`. Inline-agent beats are fetched separately (exact counts, true-latest-regardless-of-idle) so the agents' high beat volume can't crowd the cron read (and are excluded from the `control_tower_loop_beats` RPC).
- **Phase 2 output assertions** (`evalOutputAssertion(assertionId, loop, latest, inputs)`, layered on top of the P1 tile — only escalates green/amber → red, a P1 red stays; a loop with `outputAssertions` runs each in order, first-to-fail wins): `fetchAssertionInputs(admin)` adds cheap read-only queries to the snapshot batch — open routine-escalated [[../tables/tickets]] (`escalated_at` set, `escalated_to` null, not closed/archived) — both the count AND the OLDEST waiting ticket's `escalated_at` (how long real work has actually waited) — the latest `triage-escalations` + `spec-test` [[../tables/agent_jobs]] `created_at`, active overdue internal [[../tables/subscriptions]] (`next_billing_date` before today UTC), the latest renewal-cron beat (marks the live cycle boundary), the live + rolling-baseline renewal outcome breakdowns (`aggregateRenewalOutcomes`), and stuck [[../tables/dunning_cycles]] (`status='retrying'` with `next_retry_at` > 48h in the past). The five assertions: **escalation-idle** (the OLDEST waiting ticket has waited past the cadence grace [keyed off its `escalated_at`, not the last job's age] AND no triage job was created since it escalated → `reason='idle_while_work'`; re-based by [[../specs/control-tower-escalation-idle-grace]] so a healthy hourly cron isn't flagged in the normal gap between an escalation and the next tick), **spec-test-persisted** (beat reports `enqueued>0` but no spec-test job landed → `reason='false_success'`), **renewal-integrity** (N active internal subs overdue → `reason='renewal_integrity'`), **renewal-outcome-distribution** (the cron ran + each decline routed correctly, but the per-cycle outcome MIX is anomalous: a systemic rate ≥50% [hard floor: bad creds declining everyone / mass no-PM] OR a spike ≥2.5× / +15pp vs the rolling baseline, gated by a ≥10-sample cycle + ≥50-sample baseline → `reason='renewal_outcome_distribution'`; [[../specs/control-tower-renewal-integrity-assertions]] P1), **stuck-dunning** (N dunning_cycles still 'retrying' >48h past `next_retry_at` — a sub correctly mid-dunning within schedule is NOT flagged → `reason='stuck_dunning'`; same spec). A sixth, **migration-drift** ([[../specs/control-tower-migration-drift-check]] P1), is unlike the others — it reads NO live DB state in the monitor: detection happens on the box (it parses the `.sql` files + diffs the live schema, which the deployed runtime can't) and the result rides the loop's heartbeat, so `evalOutputAssertion('migration-drift', …)` just reads `latest.produced.missing` (via `readMissingTables`) and flips red on a non-empty list (`reason='migration_drift'`). See `migration-drift.ts`. A violation flows through `runControlTowerMonitor` exactly like a P1 red (de-duped alert + page). Thresholds are tunable constants in monitor.ts (`RENEWAL_MIN_CYCLE_SAMPLE`, `RENEWAL_HARD_FLOOR_RATE`, `RENEWAL_SPIKE_FACTOR`, `RENEWAL_SPIKE_MARGIN`, `RENEWAL_MIN_BASELINE_SAMPLE`, `STUCK_DUNNING_GRACE_MS`).

## `error-feed.ts` — the error feed (error-feed-monitoring Phase 1 + 2)

The capture + page + snapshot layer for the "hidden surfaces" ([[../specs/error-feed-monitoring]]) — Vercel runtime errors, Inngest failed runs, app-layer Supabase errors, and (Phase 2) DB-level Supabase logs — into [[../tables/error_events]].

- `recordError({ source, keyParts, title, detail?, sample?, occurrences?, transient? }, admin?)` → upserts a **grouped** [[../tables/error_events]] incident on `(source, signature)` (signature = a stable hash of the normalized `keyParts`), bumping `count`/`last_seen_at`. Pages owners ([[notify-ops-alert]]) on a **new signature or a re-firing spike**, rate-limited to one page per incident per 30 min. **Best-effort — never throws**; a `23505` race falls back to the update path. On a **new signature** it also enqueues a [[repair-agent]] `repair` job ([[../specs/repair-agent]]) — the diagnose→propose-fix trigger (deduped by signature, best-effort).
  - **`transient: true`** ([[../specs/error-feed-drop-inngest-transport-http-unreachable]]) — a noise CLASS that's non-actionable on a single sighting (e.g. an Inngest `http_unreachable` transport reset): a **first sighting is auto-resolved** (recorded + grouped for visibility, `status='resolved'`, `last_paged_at=null` — **no page, no repair fan-out**), and it escalates to a real **open+page** only if the SAME signature **recurs within `TRANSIENT_RECUR_WINDOW_MS` (1 h)** of the prior sighting (chronic → still broken). A prior sighting older than the window is treated as another isolated blip and re-resolved. Mirrors the outage-window auto-resolve, but keyed on recurrence instead of the Claude breaker.
- `reportDbError(error, { op, table?, … }, admin?)` — the **app-layer Supabase reporter**: a no-op on a null error, else `recordError({ source: "supabase", … })`. Call it anywhere code gets a non-null Supabase `{ error }` it would otherwise swallow (the scorecard-upsert class — wired in `src/lib/meta/scorecards.ts`). See [[../operational-rules]] "don't swallow a Supabase error".
- `signatureFor(source, keyParts)` → the grouping key (normalizes out uuids/hex/numbers/quoted strings).

## `migration-drift.ts` — schema-vs-migrations drift check (control-tower-migration-drift-check Phase 1; ci-guard-migrations-applied-not-just-merged Phase 1 added the applied-set reconcile axis, Phase 2 closes the loop with auto-apply + approval-gate; migration-drift-reconciler-idempotent-on-already-applied-objects Phase 1 makes auto-apply idempotent when the object already exists, Phase 2 adds the one-time backlog reconcile, Phase 3 makes reconciliation statement-level)

Catches a **silently-skipped migration** on either of two axes, both surfaced on the same `migration-drift-check` tile: **(a) table-presence** — a migration that never applied is invisible: the code references a table, the table isn't there, every upsert hits `PGRST205`, and (pre-error-feed) it degrades silently (2026-06-22: `20260618140000_meta_performance_tables.sql` was skipped in the apply pipeline → the iteration engine ran on empty ROAS data for weeks); **(b) applied-set reconcile** — a migration file that MERGED to main but never applied to the DB, so dependent code silently no-ops until someone notices (2026-07-06: `20260918120000_order_refunds_mirror.sql` merged with PR #1265 but never applied — `order_refunds` did not exist in prod until a manual apply). The Control Tower's first **schema-shaped** check. **Phase 2 closes the loop:** the box classifies each merged-but-unapplied file's SQL via `classifyMigrationSql` ([[../libraries/migration-safety]]) and auto-applies the additive ones on the pooler (sanctioned pattern from [[../recipes/write-a-migration-apply-script]]), recording the version in `supabase_migrations.schema_migrations` so the reconciler clears the tile on the next tick; destructive DDL is **gated for approval** (NEVER auto-applied — the tile alert stays until a human runs the sanctioned apply script). **Idempotent on ledger-only drift** ([[../specs/migration-drift-reconciler-idempotent-on-already-applied-objects]] Phase 1): when the additive-apply DDL throws a duplicate-object-class SQLSTATE (`42P07` duplicate_table, `42710` duplicate_object, `42723` duplicate_function, `42P06` duplicate_schema, `42701` duplicate_column, `42P13` cannot_change_return_type) the object is already there — an earlier apply the ledger never recorded — so the version is inserted into `supabase_migrations.schema_migrations` (`ON CONFLICT (version) DO NOTHING`) and the outcome is tagged `already-applied` (distinct from `applied`). Genuine failures (syntax, undefined_table, undefined_column, or any non-duplicate class) still stay `apply-failed` and stay red. The 2026-07 incident: the ledger was 103 versions behind so the reconciler kept re-running additive DDL and getting duplicate-object back → recorded as apply-failed, retried every tick, filling the log with `relation already exists` / `cannot change return type of existing function`. The classifier fix stops the loop; a one-time reconcile (Phase 2) clears the backlog.

Scans `supabase/migrations/*.sql` for every `COMMENT ON … IS` statement and fails the build when the payload isn't a legal literal. **Why it earns a rail:** `COMMENT ON … IS` takes a string LITERAL, not an expression, so a `||` there is a **parse** error (`42601`) — and per the statement-level contract above, a non-duplicate-object throw means the version is deliberately NEVER recorded. The migration therefore stays in `mergedButUnapplied` forever: every ~30-min box tick re-reads the file, re-runs the idempotent DDL as a no-op, re-throws `42601`, and never advances. Manual application cannot clear it (the failure is at parse time, before anything runs) — only fixing the file does. Live incident 2026-07-20: `20261119120000_creative_skeletons_do_not_use.sql` landed its columns + index on the first tick but its two `||`-concatenated comments never applied, re-firing `42601` every 30 min (~48/day in the Postgres log, 16 hits in a 7h window) with version `20261119120000` absent from `supabase_migrations.schema_migrations`. The guard encodes the three payload forms, **verified empirically against the pooler**: literals separated by whitespace containing a NEWLINE are legal SQL-standard implicit continuation (the house style across ~37 migrations, deliberately preserved); same-line adjacency (`'a ' 'b'`) is `42601`; `||` is `42601`. Quote-aware statement scan (a `;`, a `||`, or an `is` inside a literal or quoted identifier is never a boundary). Unit-pinned in `scripts/_check-sql-comment-literals.test.ts` (`npx tsx --test`), including a test that the live migrations tree is clean.
  - `applyMergedMigrations(migrations, { readSql, applyMigration, onApprovalNeeded? })` → the Phase 2 auto-apply / approval-gate control loop (ci-guard-migrations-applied-not-just-merged Phase 2). PURE apart from the injected callbacks. Genuine failures (`42601` syntax_error, `42P01` undefined_table, `42703` undefined_column, and any code outside the set) return false — they stay apply-failed. Splits a Postgres SQL script at TOP-LEVEL `;` boundaries — semicolons inside `'…'` single-quoted string literals (with `''` escape), `"…"` double-quoted identifiers, `$$…$$` / `$tag$…$tag$` dollar-quoted function bodies, `--` line comments, and `/* … */` block comments (nested per Postgres) stay inside their host token. Phase 3 makes the reconciliation **statement-level** — the security review's finding: the previous whole-file duplicate shortcut could stamp a version applied even though the migration's tail statements never ran (duplicate-object on statement #1 → savepoint rollback → recorded, but statements #2..N were never executed). The fix: for every merged-but-unrecorded version, split the file via `splitSqlStatements` and classify each statement in its OWN nested savepoint. Per-statement outcomes: statement succeeds → **applied-fresh**; throws duplicate-object SQLSTATE → **verified-already-present** (savepoint rolled back — the specific object is in the live schema; safe to skip THIS statement); throws anything else → the whole file is broken (roll back the outer container savepoint, record NOTHING, name the offending statement index in the summary). File-level outcomes when every statement is accounted for: all fresh → **recorded-genuinely-applied**; all verified-already-present → **recorded-already-applied**; mixed → **recorded-partially-applied** (some DDL work AND some duplicates — the file is now fully present). A **skipped-broken** file records nothing (the script must NEVER force-record a broken migration into the ledger — the ordinary reconciler keeps naming it until a human looks). Everything runs inside one outer transaction (all-or-nothing on `--apply`; the outer ROLLBACK cleans dry-run); each version's container savepoint, per-statement savepoint, AND each record-insert shape-fallback are isolated so a bad file never poisons the outer transaction. Prints a per-outcome summary + per-broken details naming the failing statement + per-partial details naming the fresh/already counts. Destructive files log an `APPROVAL NEEDED` warning naming the version + severity + classifier matches; the tile alert (below) is the durable surface, the log is oncall visibility.

## Gotchas

- **SHA-behind needs `VERCEL_GIT_COMMIT_SHA`** (the deployed commit) as the origin/main proxy; unset locally ⇒ the check is skipped (no false positive). It only fires red after `shaGraceMs` (default 30m) so an in-progress deploy / self-update never pages.
- **Agent-kind alert is off [[../tables/agent_jobs]], not the heartbeat** — idle = green. The heartbeat only feeds last-ran/history.
- **Worker-restart clamp on queued stuck-jobs** ([[../specs/control-tower-stuck-jobs-clamp-on-worker-restart]], signal `loop:agent:spec-test`, verdict monitor-false-positive): `jobStuckSince` clamps the `queued`/`queued_resume` floor to `worker_heartbeats.started_at` — a worker that wasn't alive earlier can't have claimed earlier than it started, so `floor = max(updated_at/created_at, worker.started_at)`. Without this, a backlog enqueued during a worker-down window is mis-attributed as a stuck lane the moment the worker restarts (the originating false page: 8 spec-test rows queued at 10:45 by the regression-backlog sweep, box offline until 11:45, monitor at 12:00 read 75-min ages and went red even though the queue had only existed for 15 min of worker uptime). `building`/`claimed` jobs are NOT clamped — `claimed_at` already reflects the worker that picked them up. Null/malformed `started_at` ⇒ no clamp (preserves prior behavior, never lifts a real stuck-since forward). Unit-tested in `monitor.test.ts` (`npx tsx --test src/lib/control-tower/monitor.test.ts`).
- **Queue-aware self-update deferral** ([[../specs/control-tower-self-update-tile-queue-aware]], signal `loop:box`, verdict monitor-false-positive): `evalWorker`'s `behind && idle` branch now mirrors the worker's own self-update logic (`scripts/builder-worker.ts:4290` self-restart-defers-to-idle). The worker INTENTIONALLY parks an idle self-update while `{queued, queued_resume} > 0` so a cascade of queued builds isn't restarted between specs — reading that as "self-update stuck" was the monitor false positive. `buildControlTowerSnapshot` now passes the queued count (derived from the existing active-jobs read) AND `worker_controls.drain_for_update` into `evalWorker`. When `behind && idle && queued > 0 && !manualDrain` ⇒ GREEN "idle — update deferred · ${n} queued" (never red); `behindTooLong` now requires `queued === 0` in addition to the SHA-grace. A MANUAL queue restart (`drain_for_update=true`) still reds at grace, because that flag's whole purpose is "restart at idle regardless of the queue."
- **SHA-direction gate before evalWorker reds** (signal `loop:box`, verdict monitor-false-positive): the box tile's behind-check now verifies the DIRECTION of the SHA delta before paging. The prior test `deployed.slice(0, running.length) !== running` reddened on ANY string mismatch — false-paging on the healthy worker-AHEAD case where the self-hosted box had already pulled a newer main commit while the deployed Vercel runtime was still reporting the previous ancestor (originating incident: worker on `6f43ec9e0`, deployed still on `b3934ff` which is an ANCESTOR of `6f43ec9e0`). `buildControlTowerSnapshot` now calls `fetchShaDirection(deployed, running)` — a small GitHub compare API wrapper — before `evalWorker`, and passes the classification through as a new `shaDirection` parameter. Semantics: `"same"` (identical or prefix-equal) ⇒ healthy · `"worker-behind"` (deployed is a descendant of running — real stuck code) ⇒ the previous queue-aware / behindTooLong / behind-busy paths still apply · `"worker-ahead"` (running is a descendant of deployed — deploy lag) ⇒ **GREEN** "healthy · ${sha} · deploy lag (Vercel on ${deployed7})", never red · `"unknown"` (missing token, unrelated SHAs, compare API blip, `diverged` status) ⇒ conservative GREEN — no red on an ambiguous compare (same posture as `deployAgeMs==null` in `evalCron`). `classifyShaDirectionLocal(deployed, running)` short-circuits the prefix-equal case without a round-trip; every other case defers to `/repos/thecyclecoder/shopcx/compare/${running}...${deployed}` (uses `GITHUB_TOKEN`/`AGENT_TODO_GITHUB_TOKEN`). Unit-tested in `monitor.test.ts` (`npx tsx --test src/lib/control-tower/monitor.test.ts`) covering worker-ahead (originating false page), unknown fallback, same, and the pre-existing worker-behind reds.

## Predicates

- `isWorkerUnavailable(row: WorkerRow | null, livenessWindowMs: number = 5 * 60_000): boolean` — checks whether the box worker is currently stale/unavailable. Used by `evalAgentKind` to suppress queued/queued_resume stuck alerts during worker outages ({{control-tower-suppress-agent-stuck-during-worker-outage}} P1). Returns `true` if: the worker row is null/missing, `last_poll_at` is missing, the worker's heartbeat is older than `livenessWindowMs`, or the worker's status is `'needs_attention'`. The same predicates `evalWorker` uses for liveness, so the two tiles never disagree on worker availability.
