# libraries/control-tower

The Control Tower module ([[../specs/control-tower]] Phase 1 + Phase 2; [[../specs/control-tower-agent-coverage]] adds the inline AI agents; [[../specs/control-tower-complete-coverage]] Phase 1 adds **full cron coverage + the reactive event-driven agents**) — the registry, the heartbeat emit helper, and the monitor/snapshot logic (liveness + cron-freshness + stuck-jobs + inline/reactive-agent assertions + Phase 2 output assertions) that powers the [[../inngest/control-tower-monitor]] cron and the [[../dashboard/control-tower]] dashboard.

**Files:** `src/lib/control-tower/registry.ts` · `src/lib/control-tower/heartbeat.ts` · `src/lib/control-tower/monitor.ts` · `src/lib/control-tower/error-feed.ts` · `src/lib/control-tower/supabase-log-poll.ts` · `src/lib/control-tower/migration-drift.ts` · (Phase 2 detection layer: [[control-tower-self-audit]] — `self-audit.ts` · `src/lib/inngest/sync.ts` · `src/lib/inngest/registered-functions.ts`)

## `registry.ts` — the loop registry (code config)

The single source of truth for every loop the monitor watches. **Add a row here when you ship a new cron / worker / agent-kind** ([[../operational-rules]] "register-or-it's-incomplete").

- `type LoopKind = "worker" | "cron" | "agent-kind" | "inline-agent" | "reactive"` — `reactive` = an **event-driven Inngest function** (the inbound ticket handler, dunning, returns, journey-outcomes, agent-todo-execute, chargeback); it shares the inline-agent evaluation model (idle = green, liveness-when-work-exists + error-rate) and is split out only for the dashboard's "Reactive agents" group. ([[../specs/control-tower-complete-coverage]] P1.)
- `type OutputAssertionId = "escalation-idle" | "spec-test-persisted" | "renewal-integrity" | "renewal-outcome-distribution" | "stuck-dunning" | "migration-drift" | "segment-coverage"` — Phase 2 output assertions (see monitor.ts). The renewal pair ([[../specs/control-tower-renewal-integrity-assertions]] P1) completes the renewal-integrity trio; `migration-drift` ([[../specs/control-tower-migration-drift-check]] P1) reads a created-table-vs-live-schema diff conveyed in a box-emitted beat (see `migration-drift.ts` below); `segment-coverage` ([[../specs/fix-segment-refresh-coverage]] P2) polls the LIVE [[../tables/customers]] table each tick and reds when <95% of SMS-subscribed rows are refreshed within 26h OR any subscribed row is >48h stale — the alarm that would have caught the 2026-07 1000/138K whole-book regression on [[../inngest/refresh-customer-segments]]. **Run-in-progress grace** ([[../specs/segment-coverage-assertion-run-in-progress-grace]], signal `loop:refresh-customer-segments-cron`, verdict monitor-false-positive): the fresh-cohort ratio check is suppressed while the loop's own most recent beat is within `SEGMENT_COVERAGE_RUN_GRACE_MS` (6h — comfortably longer than the observed worst-case ~4-5h fanout across 138K rows), since a below-floor ratio mid-run is the expected shape of the daily walk still in progress, not a break. The stale48h check stays fully active during the grace (a subscriber >48h old is a genuine break even mid-run), and once the grace elapses the 95% floor re-engages and would still catch a truly stalled/partial refresh.
- `type InlineWorkSignalId = "tickets-awaiting-qc" | "journeys-awaiting-delivery" | "orders-awaiting-fraud-screen" | "tickets-awaiting-decision"` — the read-only independent upstream-demand probes the inline-agent liveness-when-work-exists check evaluates.
- `type RenewalOutcome = "charged" | "skipped_no_payment_method" | "skipped_zero_total" | "declined_to_dunning" | "comp_shipped" | "comp_blocked" | "skipped_other"` + `RENEWAL_OUTCOME_LOOP_ID = "internal-subscription-renewal-outcome"` + `RENEWAL_BAD_OUTCOMES` (the anomalous subset: no-PM-skip, declined, comp-blocked) — the per-sub renewal outcome taxonomy the outcome-distribution assertion aggregates ([[../specs/control-tower-renewal-integrity-assertions]] P1). Every terminal path of `internal-subscription-renewal-attempt` emits ONE beat (loop_id = `RENEWAL_OUTCOME_LOOP_ID`, `kind:'reactive'`, `produced.outcome`) — the only uniform channel that captures SKIPS, which leave no transaction row. NOT a monitored tile (a data channel).
- `interface MonitoredLoop { id, kind, label, description, expectedCadence, livenessWindowMs?, shaGraceMs?, agentKind?, stuckThresholdMs?, outputAssertion?, outputAssertions?, inlineWorkSignal?, errorRateThreshold?, minRunsForErrorRate? }` — `outputAssertions?: OutputAssertionId[]` is for a loop carrying MORE THAN ONE assertion (the renewal cron: `["renewal-integrity", "renewal-outcome-distribution"]`); the monitor runs each in order, first-to-fail wins. Use it OR the singular `outputAssertion`.
- `MONITORED_LOOPS: MonitoredLoop[]` — the box worker (`box`), **the full Inngest cron set (50 crons** — each cron's inngest fn id + a cadence-derived `livenessWindowMs`, from minute-crons like `deliver-pending-sends`/`meta-capi-dispatch-cron` through the daily/weekly/yearly ones; [[../specs/control-tower-complete-coverage]] P1 closed the gap from the original 8**)**, 11 agent kinds (`agent:<kind>` + a per-kind `stuckThresholdMs`), **4 inline AI agents** (`ai:ticket-analyzer`, `ai:journey-delivery`, `ai:fraud-detector`, `ai:orchestrator` — each with a `livenessWindowMs`, optional `inlineWorkSignal`, and `errorRateThreshold`/`minRunsForErrorRate`), and **6 reactive event-driven agents** (`unified-ticket-handler` — THE inbound ticket pipeline, with `inlineWorkSignal: 'tickets-awaiting-decision'` — plus `dunning-payment-failed`, `returns-process-delivery`, `journey-session-completed`, `agent-todo-execute`, `chargeback-received`; `kind: 'reactive'`, error-rate by default), the **`auto-merge-gate`** (`AUTO_MERGE_GATE_LOOP_ID`) — a `reactive` loop that runs inside the [[../integrations/github-webhook|GitHub webhook]] (the Auto-Ship Pipeline's Gate A, [[../specs/auto-ship-pipeline]] P1), error-rate only (a failed merge attempt beats ok:false; idle = green), and the **`auto-fold-gate`** (`AUTO_FOLD_GATE_LOOP_ID`) — a `reactive` loop that auto-folds fully-verified shipped specs (the Auto-Ship Pipeline's Gate B, [[../specs/auto-ship-pipeline]] P2; `autoFoldVerifiedSpecs` in [[spec-test-runs]], fired by the spec-test worker + the human-queue POST + the daily spec-test cron), error-rate only (a failed enqueue beats ok:false; idle = green). Output assertions: `triage-escalations-cron` → `escalation-idle`, `spec-test-cron` → `spec-test-persisted`, `internal-subscription-renewal-cron` → `["renewal-integrity", "renewal-outcome-distribution"]`, `dunning-payday-retry-cron` → `stuck-dunning`, `migration-drift-check` (box-emitted cron) → `migration-drift`. **Cadence-accuracy invariant** (signal `loop:ticket-analysis-cron`, verdict monitor-false-positive): each cron row's `expectedCadence` + `livenessWindowMs` MUST mirror the real Inngest `cron:` schedule of the function with that `id` — a window shorter than the cadence makes `evalCron` false-fire `cron_freshness` every cycle (`ticket-analysis-cron` was registered `*/5` + 15-min while the fn fires `*/30` → red for 15 of every 30 min between healthy beats; corrected to `*/30` + 90-min, matching its `─ Every-30-min crons (window ~90 min) ─` group). When a cron's schedule changes, update its registry row in the same PR.
- `WORKER_BOX_ID = "box"` (matches `scripts/builder-worker.ts`) · `agentLoopId(kind)` → `agent:<kind>` · `INLINE_AGENT_IDS = { ticketAnalyzer, journeyDelivery, fraudDetector, orchestrator }` (the `ai:<agent>` loop ids, shared with the agent files) · `AUTO_MERGE_GATE_LOOP_ID = "auto-merge-gate"` (the auto-merge gate's reactive loop_id, shared with [[github-pr-resolve]]) · `AUTO_FOLD_GATE_LOOP_ID = "auto-fold-gate"` (the auto-fold gate's reactive loop_id, shared with [[spec-test-runs]]).
- `MIGRATION_DRIFT_LOOP_ID = "migration-drift-check"` ([[../specs/control-tower-migration-drift-check]] P1) — the loop_id of the **box-emitted** migration-drift cron (a `kind:'cron'` MONITORED_LOOPS tile, owner platform, `outputAssertion:'migration-drift'`, ~30 min window 90m). NOT an Inngest fn: the deployed runtime can't read the `supabase/migrations/*.sql` files, so `scripts/builder-worker.ts` runs the parse+diff on the box and beats here, carrying the missing tables in `produced`. Freshness keeps a dead check visible; the assertion flips the tile red on drift. (Not in the in-code↔Inngest self-audit diff — `auditCronCoverage` only flags Inngest crons *without* a tile, never a tile without an Inngest fn.)
- `INTENTIONALLY_UNMONITORED_CRONS: Record<id, reason>` (Phase 2) — cron `createFunction`s deliberately given no tile, so the [[control-tower-self-audit]] doesn't flag them as "unregistered loop: X". Silence is never the default: each needs a reason. Keep it short — the right answer for a real loop is a `MONITORED_LOOPS` row, not an exemption. **Current entries:** `slack-roadmap-notify` ("intentionally unmonitored — owner-confirmed via the [[coverage-register-agent]]") — surfaced by [[control-tower-self-audit]], proposed via the [[coverage-register-agent]]'s `exempt` path, and owner-confirmed rather than registered as a tile.

## `heartbeat.ts` — end-of-run emit

Best-effort writes of one [[../tables/loop_heartbeats]] row (never throws).

- `emitLoopHeartbeat(loopId, kind, { ok?, produced?, detail?, durationMs? })`
- `emitCronHeartbeat(functionId, …)` — `kind:'cron'`, `loop_id` = the inngest fn id. Called by each monitored cron inside a `step.run("emit-heartbeat", …)` before its return.
- `emitAgentHeartbeat(agentKind, …)` — `kind:'agent-kind'`, `loop_id` = `agent:<kind>`. (The box worker writes its agent beats via its own inline `writeLoopHeartbeat` against its existing admin client, not this helper — same shape.)
- `emitInlineAgentHeartbeat(agentId, …)` — `kind:'inline-agent'`, `loop_id` = an `INLINE_AGENT_IDS` value (e.g. `ai:ticket-analyzer`, `ai:orchestrator`). Called in a **try/finally at the END of each run** of the 4 inline AI agents — `ok:true` on success/intentional-skip, `ok:false` on a thrown run or real failure (for `ai:orchestrator`, also a degraded/fallback decision).
- `emitReactiveHeartbeat(functionId, …)` — `kind:'reactive'`, `loop_id` = the inngest fn id (e.g. `unified-ticket-handler`). Called in a **top-level try/finally** inside each of the 6 reactive event-driven Inngest functions — `ok:true` on success/skip-return, `ok:false` on a thrown run. ([[../specs/control-tower-complete-coverage]] P1.)
- `emitRenewalOutcomeHeartbeat(outcome: RenewalOutcome)` — records ONE per-sub renewal outcome beat (`loop_id = RENEWAL_OUTCOME_LOOP_ID`, `kind:'reactive'`, `produced:{outcome}`, `ok:true`). Called from every terminal path of `internal-subscription-renewal-attempt` ([[../inngest/internal-subscription-renewals]]) — the only uniform channel that captures SKIPS (a no-PM / zero-total skip writes no transaction row). Uncaught errors aren't beat (a sub that errored never advances → caught by the renewal-integrity overdue assertion). ([[../specs/control-tower-renewal-integrity-assertions]] P1.)
- `aggregateRenewalOutcomes(admin, sinceIso, untilIso?)` → `RenewalOutcomeCounts` — READ-ONLY count breakdown of the outcome beats in a window (bounded `limit 5000`, best-effort → zeros on read failure). Used by the renewal cron (bakes the just-completed cycle's breakdown into its heartbeat's `produced.last_cycle_outcomes`) AND by the outcome-distribution assertion (live current cycle + rolling baseline).

## `monitor.ts` — snapshot + monitor

- `buildControlTowerSnapshot(admin?)` → `ControlTowerSnapshot { generatedAt, counts:{green,amber,red}, loops: LoopStatus[], selfAudit }`. **READ-ONLY**: one batched read of [[../tables/worker_heartbeats]], [[../tables/loop_heartbeats]] (the **`control_tower_loop_beats(p_history_limit)` RPC** — latest ≤10 beats **per loop_id** for cron + agent-kind kinds, **excluding `inline-agent`/`reactive`**; a **lateral join** over the distinct loop_ids rides the `(loop_id, ran_at desc)` index, reading ≤N rows per loop. **Presence is the ever-beaten signal** — a loop with zero beats is absent from the result ⇒ never_fired candidate (no all-time count is returned). Replaced the old global "last 600 beats" window — and then the per-row `count(*) OVER`/`row_number() OVER` body ([[../specs/control-tower-loop-beats-rpc-perf]]), both of which scanned + sorted the whole growing table and 500-ed on statement timeout), open [[../tables/loop_alerts]], active [[../tables/agent_jobs]], plus `fetchInlineAgentState` (per inline agent: its work probe + exact ok/errored window counts + latest + history) **and `buildCoverageAudit()`** ([[control-tower-self-audit]], Phase 2); evaluates each loop to a `LoopStatus { color, statusText, lastRanAt, lastProduced, detail, violation, history, openAlert }`. The `selfAudit` findings (unregistered crons + in-code↔Inngest-registered gaps) are folded into the honest amber `counts`. Used **verbatim** by the dashboard API.
- `runControlTowerMonitor()` → `MonitorResult`. Builds the snapshot, then **acts**: opens a de-duped [[../tables/loop_alerts]] incident on each newly-red loop (paging owners via [[../libraries/notify-ops-alert]]), bumps `last_seen_at` while still red (no re-page), and resolves on recovery. On a **newly-opened** alert it also enqueues a [[repair-agent]] `repair` job (signature `loop:<id>`) — the second diagnose→propose-fix trigger ([[../specs/repair-agent]]). It also drives the [[coverage-register-agent]]: for each `selfAudit.unregistered` loop it calls `enqueueCoverageRegisterJob` (best-effort), which proposes the inferred `MONITORED_LOOPS` entry for one-tap owner Build ([[../specs/coverage-auto-register-agent]] P1). Called only by the cron. **Environment-scoped (`isCanonicalProductionDeploy` = `VERCEL_ENV==="production"`):** the act phase — all [[../tables/loop_alerts]] writes + paging + Repair/coverage enqueues — runs **only on the canonical production deploy**. A preview/branch deploy carries a branch-local `MONITORED_LOOPS` (e.g. a cron that exists only on an unmerged WIP branch), so it still evaluates the read-only snapshot for its own dashboard but returns before any write — otherwise it leaks phantom `registered_not_firing` alerts into the shared prod feed and wakes the Repair Agent for loops absent from HEAD ([[../specs/control-tower-loop-alerts-prod-scoped]], the `loop:claude-status-poll-cron` incident).
- Evaluators: `evalWorker` (liveness + SHA-behind + **all-Max-accounts-capped**: reads the box heartbeat's `accounts.all_capped` → **amber** "all Max accounts capped — builds parked, auto-resume" [NOT red — `blocked_on_usage` auto-resumes at reset, no manual rebuild], so a green box tile can't hide a throughput stall where every account hit its 5-hour wall; per-account load rides the tile's `lastProduced` — [[../specs/box-multi-account-failover]] P2), `evalCron` (freshness + **never-fired**), `evalAgentKind` (stuck jobs), `evalInlineAgent` (inline AI agents **and** the `reactive` event-driven agents — both routed through it, and both excluded from the `control_tower_loop_beats` RPC / sourced from `fetchInlineAgentState`). Genuinely-idle/healthy → green; a freshly-shipped cron with no beat yet → amber (never a false red).
- **`evalCron` never-fired detection** ([[../specs/control-tower-complete-coverage]] Phase 2, accuracy-fixed by [[../specs/control-tower-monitor-accuracy]] Phase 1): a cron with **0 heartbeats** is no longer amber-forever. `deployRefAgeMs(worker)` gives a trustworthy "how long has this deploy been live" lower bound — the box worker self-updates to origin/main + restarts on a new SHA, so once `worker.running_sha` matches `VERCEL_GIT_COMMIT_SHA`, `worker.started_at` ≈ deploy time (returns null = unknown ⇒ stay amber, never a false red). **never-fired keys off whether the loop has EVER beaten (`everBeatCount` — now presence in the lateral-join read, not an all-time count), not 0-since-deploy** — a cron with **any** historical beat is being invoked by Inngest, so a longer-than-window real cadence (daily `today-sync` w/ 99 beats, bursty `meta-capi-dispatch` w/ 494) is at most a freshness alert, **never** `never_fired`. Only a cron with **0 beats in all of history** whose deploy is **older than its `livenessWindowMs` (cadence + grace)** → **red `never_fired`** ("registered but has never run — Inngest is not invoking it"), pages like any red. A genuinely-fresh cron (deploy < one window) or an unknown deploy age stays **amber "awaiting first run"**. **registered-but-not-firing** ([[../specs/spec-drift-reconcile-not-firing]] P1): the deploy-anchored `never_fired` misses a cron that has been registered for **weeks but never fires once**, because the box self-updates + restarts on every ship so `deployRefAgeMs` keeps RESETTING under it — with frequent deploys its current-deploy clock rarely exceeds the window, so the dead cron hides forever (the real `spec-drift-reconcile` failure: registered with Inngest, `diffInngestRegistered` says `missing:[]`, but its `20,50 * * * *` schedule never executed → 0 `loop_heartbeats` beats ever). So a SECOND, **deploy-surviving** reference: `monitorUptimeMs` = the age of the **oldest `control-tower-monitor` beat** (how long the watchdog itself has been continuously running — beat retention can only shorten it, never inflate it ⇒ conservative). A cron with **0 beats ever** while the watchdog has itself been beating for **longer than the cron's window** → **red `registered_not_firing`** ("registered with Inngest but not firing — its cron schedule isn't active in the prod env; re-sync the app"). Distinct from the [[control-tower-self-audit]] Inngest-registration diff (never-*registered*): this fn IS in the registered set, the SCHEDULE just isn't active. **Newly-added-cron grace** ([[../specs/control-tower-registered-not-firing-newcron-grace]]): `monitorUptimeMs` is the watchdog's OWN run-span, independent of when a given cron was ADDED — so a long-cadence cron shipped AFTER the watchdog passed its window would false-trip on day one, hours before its first scheduled tick (signal `loop:storefront-experiments-refresh-cron`). The optional **`MonitoredLoop.registeredAt`** (an ISO code constant — deploy-SURVIVING, unlike `deployRefAgeMs`) gives a per-loop "how long has THIS cron been registered" reference: `registered_not_firing` now requires `(now - firstScheduledFiringAtOrAfter(registeredAt)) > window` **in addition to** `monitorUptimeMs > window`. A cron registered for less than a full window stays **amber "awaiting first run"**, graced just like the deploy-anchored `never_fired`. Each newly-added long-cadence cron must carry its own `registeredAt` to claim the grace — the same false positive recurred on a sibling daily cron ([[../specs/control-tower-registered-not-firing-new-cron-grace]], signal `loop:storefront-lever-decay-cron`), fixed by stamping that loop's `registeredAt` too. **First-firing anchor** ([[../specs/control-tower-cron-grace-uses-next-firing-after-registration]], signal `loop:security-dep-watch`, verdict monitor-false-positive): the grace clock can't be `registeredAt` itself — a daily cron whose `registeredAt` falls BEFORE its hour-of-day (e.g. `security-dep-watch` `0 4 * * *` with `registeredAt` 00:00 UTC + a deploy that landed at 04:08 missed that day's 04:00 tick) would false-page 26h after midnight, 2h before the cron's first real chance to fire (the next day's 04:00). `monitor.ts` parses the cron expression from `expectedCadence` (e.g. `"daily (0 4 * * *)"` → `0 4 * * *`) once at module load via `extractCronExpr` + `parseCronExpr` + `nextFiringAtOrAfter`, computes the FIRST scheduled firing at-or-after `registeredAt`, memoizes it per `loop.id`, and gates the grace on that. Box-job cadences (`"every ~30 min (box job)"`) have no Inngest schedule and fall back to `registeredAt` (preserves prior behavior). Unit-tested in `monitor.test.ts` (`npx tsx --test src/lib/control-tower/monitor.test.ts`). Unset (legacy crons) ⇒ no extra gate (registered long ago). `null` monitor uptime (bootstrapping / pruned) ⇒ stay amber. Closes the `control-tower-monitor` "awaiting first run for days" gap; pairs with the [[control-tower-self-audit]] in-code↔Inngest-registered diff (same gap, registration side). **Beats-read-failure guard** ([[../specs/control-tower-beats-read-failure-guard]]): `buildControlTowerSnapshot` now captures the `control_tower_loop_beats` RPC **error** (not just `data`) — a statement-timeout/500 returns `beats=null`, which would otherwise leave every cron with `everBeatCount=0`/`latest=null` and false-fire `never_fired`. When `beatsReadFailed` is true `evalCron` treats the zero as **UNKNOWN, not a true zero**: it suppresses both `never_fired` and `cron_freshness` reds and returns **amber "beat read unavailable — status unknown"** — the same conservative posture as the `deployAgeMs==null` guard.
- **Inline-agent assertions** ([[../specs/control-tower-agent-coverage]], `evalInlineAgent` + `fetchInlineAgentState`): event-driven AI agents have no cron cadence and no `agent_jobs` queue, so silence only matters when work exists. **liveness-when-work-exists** — the loop's `inlineWorkSignal` (an independent upstream-demand count: unanalyzed closed-AI [[../tables/tickets]]; recent [[../tables/journey_sessions]]; recent [[../tables/orders]]; inbound-customer [[../tables/ticket_messages]] for `ai:orchestrator`) is > 0 AND **0 successful** beats in the window → red, `reason='idle_while_work'`. The `tickets-awaiting-qc` probe carries a **feeder-cadence grace** (`TICKET_ANALYSIS_FEEDER_GRACE_MS` = 40 min, [[../specs/ticket-analyzer-workprobe-cron-grace]]): since the analyzer is fed by `ticket-analysis-cron` every 30 min, a closed AI ticket only counts once its `updated_at` is ALSO older than the grace — i.e. it has survived a full feeder cycle still unprocessed — so a ticket closing between the 30-min ticks doesn't false-fire `idle_while_work` ~15 min before any cron cycle could service it (a genuinely-stuck analyzer still trips it a cycle later). The `tickets-awaiting-decision` probe (shared by `unified-ticket-handler` + `ai:orchestrator`) counts only inbound-customer [[../tables/ticket_messages]] that DRIVE an `ai:orchestrator` beat. It subtracts messages whose ticket is **closed OR `csat:reopened`-tagged OR has an `active_playbook_id`** (a single positive-match `.or(...)` over the joined ticket). Three exclusions: (1) the CSAT-reopen path (`/api/csat/[ticketId]`, [[../specs/control-tower-ticket-decision-workprobe-scope]]) inserts an inbound customer message + reopens + routes to a human but emits NO `ticket/inbound-message` event, so the handler never beats on it; (2) the pre-orchestrator short-circuit gates ([[../specs/ticket-decision-workprobe-exclude-positive-close]]) — positive-close (`unified-ticket-handler.ts:1635`), fraud-gate, chargeback-gate — close the ticket BEFORE `callSonnetOrchestratorV2`, so those inbound messages never drive a beat either; (3) active-playbook continuations ([[../specs/ticket-decision-workprobe-exclude-active-playbook]]) — a customer mid-playbook (refund/cancel/return) routes through `executePlaybookStep` and never reaches `callSonnetOrchestratorV2`, so the orchestrator legitimately emits no beat (two such inbounds in a quiet 120m window were enough to flip the `ai:orchestrator` tile red, signal `loop:ai:orchestrator`). A lone reopen note, a 'Thank you' positive-close, or a playbook step reply in an otherwise-quiet window used to read as work=1 / 0 beats and false-fire `idle_while_work`. A still-OPEN, no-playbook ticket with no beat keeps counting, so a genuine orchestrator outage (inbound piling up on tickets nothing closes or hands to a playbook) still alerts; a ticket the orchestrator served + closed has its own ok beat, so dropping it never manufactures a false negative. Expressed as a single NULL-safe positive-match subtraction (not a negated array filter, and overlap-free — a csat:reopened ticket that later closes is counted once) so NULL/empty-tag, NULL-`active_playbook_id` tickets always count. **error-rate** — `errCount/total ≥ errorRateThreshold` (default 0.5) once `total ≥ minRunsForErrorRate` (default 5) → red, `reason='error_rate'`. Inline-agent beats are fetched separately (exact counts, true-latest-regardless-of-idle) so the agents' high beat volume can't crowd the cron read (and are excluded from the `control_tower_loop_beats` RPC).
- **Phase 2 output assertions** (`evalOutputAssertion(assertionId, loop, latest, inputs)`, layered on top of the P1 tile — only escalates green/amber → red, a P1 red stays; a loop with `outputAssertions` runs each in order, first-to-fail wins): `fetchAssertionInputs(admin)` adds cheap read-only queries to the snapshot batch — open routine-escalated [[../tables/tickets]] (`escalated_at` set, `escalated_to` null, not closed/archived) — both the count AND the OLDEST waiting ticket's `escalated_at` (how long real work has actually waited) — the latest `triage-escalations` + `spec-test` [[../tables/agent_jobs]] `created_at`, active overdue internal [[../tables/subscriptions]] (`next_billing_date` before today UTC), the latest renewal-cron beat (marks the live cycle boundary), the live + rolling-baseline renewal outcome breakdowns (`aggregateRenewalOutcomes`), and stuck [[../tables/dunning_cycles]] (`status='retrying'` with `next_retry_at` > 48h in the past). The workspace-scoped integrity queries (overdue subscriptions, stuck dunning) exclude the permanent **spec-test sandbox** tenant (`is_test=true`, `SPEC_TEST_SANDBOX_WORKSPACE_ID` = `SPEC_TEST_FIXTURES.workspaceId`, [[spec-test-sandbox]]) via `.neq("workspace_id", …)` — its seeded fixtures are deliberately stuck (e.g. the comp sub whose customer has no `comp_role`, so the fail-closed comp gate never advances it) and would otherwise read as a real production anomaly that trips a loop tile RED ([[../specs/control-tower-exclude-spec-test-sandbox-from-integrity-asser]]). The five assertions: **escalation-idle** (the OLDEST waiting ticket has waited past the cadence grace [keyed off its `escalated_at`, not the last job's age] AND no triage job was created since it escalated → `reason='idle_while_work'`; re-based by [[../specs/control-tower-escalation-idle-grace]] so a healthy hourly cron isn't flagged in the normal gap between an escalation and the next tick), **spec-test-persisted** (beat reports `enqueued>0` but no spec-test job landed → `reason='false_success'`), **renewal-integrity** (N active internal subs overdue → `reason='renewal_integrity'`), **renewal-outcome-distribution** (the cron ran + each decline routed correctly, but the per-cycle outcome MIX is anomalous: a systemic rate ≥50% [hard floor: bad creds declining everyone / mass no-PM] OR a spike ≥2.5× / +15pp vs the rolling baseline, gated by a ≥10-sample cycle + ≥50-sample baseline → `reason='renewal_outcome_distribution'`; [[../specs/control-tower-renewal-integrity-assertions]] P1), **stuck-dunning** (N dunning_cycles still 'retrying' >48h past `next_retry_at` — a sub correctly mid-dunning within schedule is NOT flagged → `reason='stuck_dunning'`; same spec). A sixth, **migration-drift** ([[../specs/control-tower-migration-drift-check]] P1), is unlike the others — it reads NO live DB state in the monitor: detection happens on the box (it parses the `.sql` files + diffs the live schema, which the deployed runtime can't) and the result rides the loop's heartbeat, so `evalOutputAssertion('migration-drift', …)` just reads `latest.produced.missing` (via `readMissingTables`) and flips red on a non-empty list (`reason='migration_drift'`). See `migration-drift.ts`. A violation flows through `runControlTowerMonitor` exactly like a P1 red (de-duped alert + page). Thresholds are tunable constants in monitor.ts (`RENEWAL_MIN_CYCLE_SAMPLE`, `RENEWAL_HARD_FLOOR_RATE`, `RENEWAL_SPIKE_FACTOR`, `RENEWAL_SPIKE_MARGIN`, `RENEWAL_MIN_BASELINE_SAMPLE`, `STUCK_DUNNING_GRACE_MS`).

## `error-feed.ts` — the error feed (error-feed-monitoring Phase 1 + 2)

The capture + page + snapshot layer for the "hidden surfaces" ([[../specs/error-feed-monitoring]]) — Vercel runtime errors, Inngest failed runs, app-layer Supabase errors, and (Phase 2) DB-level Supabase logs — into [[../tables/error_events]].

- `recordError({ source, keyParts, title, detail?, sample?, occurrences?, transient? }, admin?)` → upserts a **grouped** [[../tables/error_events]] incident on `(source, signature)` (signature = a stable hash of the normalized `keyParts`), bumping `count`/`last_seen_at`. Pages owners ([[notify-ops-alert]]) on a **new signature or a re-firing spike**, rate-limited to one page per incident per 30 min. **Best-effort — never throws**; a `23505` race falls back to the update path. On a **new signature** it also enqueues a [[repair-agent]] `repair` job ([[../specs/repair-agent]]) — the diagnose→propose-fix trigger (deduped by signature, best-effort).
  - **`transient: true`** ([[../specs/error-feed-drop-inngest-transport-http-unreachable]]) — a noise CLASS that's non-actionable on a single sighting (e.g. an Inngest `http_unreachable` transport reset): a **first sighting is auto-resolved** (recorded + grouped for visibility, `status='resolved'`, `last_paged_at=null` — **no page, no repair fan-out**), and it escalates to a real **open+page** only if the SAME signature **recurs within `TRANSIENT_RECUR_WINDOW_MS` (1 h)** of the prior sighting (chronic → still broken). A prior sighting older than the window is treated as another isolated blip and re-resolved. Mirrors the outage-window auto-resolve, but keyed on recurrence instead of the Claude breaker.
- `reportDbError(error, { op, table?, … }, admin?)` — the **app-layer Supabase reporter**: a no-op on a null error, else `recordError({ source: "supabase", … })`. Call it anywhere code gets a non-null Supabase `{ error }` it would otherwise swallow (the scorecard-upsert class — wired in `src/lib/meta/scorecards.ts`). See [[../operational-rules]] "don't swallow a Supabase error".
- `signatureFor(source, keyParts)` → the grouping key (normalizes out uuids/hex/numbers/quoted strings).
- `isAbortedStreamNoise(message, status)` → **capture filter** ([[../specs/error-feed-drop-aborted-stream-noise]]): `true` for the Node Web-Streams client-abort teardown family — `status === 0` (the SSR response never completed → visitor disconnected) **and** a message in the abort set (`transformAlgorithm is not a function`, `Invalid state: Controller is already closed`, `ERR_STREAM_PREMATURE_CLOSE`/`aborted`) **and** an `at ignore-listed frames`-only stack (zero frames in our code). The documented TransformStream race (nodejs/node#62040) surfaced by Next.js streaming — non-actionable framework noise, the same genre as the bare-Lambda-lifecycle drop. Factored here (not in the route) so any feed can reuse it; wired in `/api/webhooks/vercel-logs` `isError()` beside `isBareLifecycle`. A real error with a frame in our code is **kept**.
- `isTransientInngestTransportError(errName, errMessage)` → **classifier** ([[../specs/error-feed-drop-inngest-transport-http-unreachable]]): `true` when the error name OR message names a member of the Inngest TRANSPORT-failure family — `http_unreachable`, `performing request to SDK URL`, `reset the connection`, or `Unexpected ending response`. That's Inngest unable to get a clean reply from our Vercel SDK URL (a deploy-boundary Lambda reap / connection reset against an every-15-min cron), NOT an app throw — the function body never ran/threw and the next beat recovers. Unlike the vercel `isBareLifecycle`/`isAbortedStreamNoise` drop-before-record filters, this only CLASSIFIES: `inngest-failure-capture` passes the result as the `transient` flag to `recordError`, which auto-resolves a first sighting and escalates only on recurrence within the window — so a one-off blip that opened `inngest:06e8cf82e141fbaa` is dropped while a chronic always-failing function still surfaces. Unit-tested in `error-feed.test.ts`. A real application throw (a `TypeError`, an Avalara 422) is **kept**.
- `isTransientInngestStepRetryThrow(path, message)` → **classifier** ([[../archive.d/error-feed-drop-inngest-step-retry-throws]]): the **vercel-log-drain** sibling of `isTransientInngestTransportError`. `true` when the log is from `/api/inngest` AND the message carries an `(attempt N/M)` marker with `N < M` — i.e. an Inngest `step.run` intentionally threw to trigger its OWN retry (the same `socialPublish` family that surfaces a transient Meta Graph failure as `throw new Error('transient publish failure (attempt 1/5): …')` — `PUBLISH_RETRIES=4`, so attempt 1/5 means 4 attempts remain). The function body never finally-failed and the next attempt is the retry mechanism working, not a defect — minting a fresh OPEN incident for every transient Meta blip pages Platform owners on a healthy loop that self-heals (`vercel:0ffd0e07c0fe9336`). Wired in `/api/webhooks/vercel-logs` alongside `isBareLifecycle`/`isAbortedStreamNoise`, but unlike those drop-before-record filters it only CLASSIFIES: the route passes the result as the `transient` flag to `recordError`, which auto-resolves a first sighting and escalates only on recurrence within `TRANSIENT_RECUR_WINDOW_MS` — a one-off blip is dropped while a function that throws on every retry still surfaces. **Kept** (not transient): the FINAL attempt (`N==M` — terminal failure, retries exhausted), a non-`/api/inngest` path with the same marker, and a real `/api/inngest` error without the `(attempt N/M)` marker. Unit-tested in `error-feed.test.ts`.
- `isBareLifecycle(message)` → **capture filter** ([[../specs/error-feed-drop-bare-502-proxy-wrapper]]): `true` when EVERY line of a Vercel/Lambda log is request-lifecycle scaffolding — `START`/`END`/`REPORT RequestId` blocks (+ split `Duration`/`Memory`/`XRAY` metric lines) and the bare `[METHOD] path status=NNN` proxy summary — i.e. a 5xx wrapper carrying no error body around a failure the function already logged itself. The proxy-summary matcher is **not** `$`-anchored after `status=NNN` (it tolerates trailing duration/region/byte tokens like `… status=502 669ms`); the original `$`-anchored regex never matched the real proxy line, so the wrapper was captured anyway and reopened `vercel:ebdf493a37c60c34` on a healthy, ticketed Appstle-502 loop. A lifecycle block that *also* carries a real message/stack is **not** bare and is **kept**.

Wired in `/api/webhooks/vercel-logs` `isError()` beside `isBareLifecycle` / `isAbortedStreamNoise`.
- `isTransientShopifyWebhookHmacFailure(path, message)` → **classifier** ([[../specs/error-feed-shopify-webhook-hmac-transient]]): the shopify-webhook sibling of `isTransientInngestStepRetryThrow`. `true` when the log is from `/api/webhooks/shopify` OR `/api/webhooks/shopify-returns` AND the message begins with `"Shopify webhook HMAC failed"` / `"Shopify returns webhook HMAC failed"` — the route's 401-reject of an unverified delivery. - `isTransientClientNetworkAbort(message, stack)` → **classifier** ([[../specs/error-feed-drop-safari-load-failed-client-network-abort-noise]]): the **client-feed** sibling of `isTransientInngestTransportError` / `isTransientShopifyWebhookHmacFailure`. `true` when BOTH: the (trimmed, case-insensitive) message is EXACTLY one of the well-known browser network-abort TypeErrors — Safari `"Load failed"`, Chrome/Firefox `"Failed to fetch"` / `"NetworkError when attempting to fetch resource"` (± trailing `.`), iOS URLSession `"The network connection was lost"` (± trailing `.`) — AND the stack is empty/null. ## `migration-drift.ts` — schema-vs-migrations drift check (control-tower-migration-drift-check Phase 1)

Catches a **silently-skipped migration**: a migration that never applied is invisible — the code references a table, the table isn't there, every upsert hits `PGRST205`, and (pre-error-feed) it degrades silently. (Found 2026-06-22: `20260618140000_meta_performance_tables.sql` was skipped in the apply pipeline → the iteration engine ran on empty ROAS data for weeks.) The Control Tower's first **schema-shaped** check.

- **Pure parse + diff** (no fs/pg at module load — the caller supplies the dir + a `fetchLiveTables` callback, so the LOGIC is testable and the box owns the raw-SQL read):
  - `parseExpectedTables(migrationsDir)` → `{ expected: Map<table, firstMigration>, dropped, parsedFiles }` — every `*.sql` (lexical sort = apply order) parsed for the tables it CREATEs, **net of any later DROP or RENAME**. Drop-aware (create-then-drop = net-absent, not expected; a re-create revives it) AND **rename-aware** (`ALTER TABLE … RENAME TO …`: the old name is dropped from `expected`, the new name carried forward mapped to the migration that first created the *original* — fallback the rename migration). Strips `--` + `/* */` comments and skips `temp`/`unlogged` tables; handles `if not exists`, `public.`-qualified, and `"quoted"` names. Rename-awareness ([[../specs/migration-drift-track-table-renames]] P1, verdict monitor-false-positive) fixes the `loop:migration-drift-check` false positive where `20260705150000_worker_to_agent_rename.sql` renamed `worker_*`→`agent_*` with no DROP, leaving the four `worker_*` names stuck in `expected` and reddening the tile against a healthy DB.
  - `extractCreatedTables(sql)` / `extractDroppedTables(sql)` / `extractRenamedTables(sql)` (→ `{from, to}[]`) — the per-file regex primitives. `foldMigrations(files: {file, sql}[])` → `{ expected, dropped }` is the **pure, fs-free core** of `parseExpectedTables` (folds an ordered file list, applying create→rename→drop) — unit-tested in `migration-drift.test.ts` (`npm run test:migration-drift`), covering the create-then-rename net.
  - `computeDrift(expected, liveTables, allowlist?)` → `{ missing, allowlistedMissing }` — expected-but-absent tables, split by the allowlist.
  - `runMigrationDriftCheck({ migrationsDir, fetchLiveTables, allowlist? })` → `DriftResult { status:'ok'|'drift'|'skipped', missing, allowlistedMissing, expectedCount, liveCount, parsedFiles, reason? }`. `fetchLiveTables` returning **null** ⇒ `status:'skipped'` (no DB password on this host) — an HONEST missing read, never a false "no drift". `driftSummary(r)` → a one-line log/detail string.
- `MIGRATION_DRIFT_ALLOWLIST = ["klaviyo_*"]` + `isAllowlisted(table, allowlist?)` — **scope the noise** (spec): a sunset table (Klaviyo is being retired) that disappears lands in `allowlistedMissing` (surfaced for visibility) and NEVER alerts. Entries match exactly or as a `prefix*` wildcard. Keep it SHORT — the right answer for a real missing table is to re-apply its migration.
- **Where it runs:** `scripts/builder-worker.ts` → `runMigrationDriftJob` (a periodic box job, ~30 min, in-flight-guarded so it never overlaps/stalls the 5s poll) reads the live `public` BASE TABLE names off the pooler (`information_schema.tables` — not exposed via PostgREST, so raw `pg` via `poolerConnectionString()`) and writes the `DriftResult` into a `loop_heartbeats` beat (`loop_id = MIGRATION_DRIFT_LOOP_ID`, `kind:'cron'`, `produced.missing`, `ok = status!=='skipped'`). The deployed Next runtime can't do this — the `.sql` files aren't bundled.
- **Surfacing** = the standard monitor path: `monitor.ts` `evalOutputAssertion('migration-drift', …)` reads `latest.produced.missing` (allowlisted ones already excluded box-side) and flips the tile **red** (`reason='migration_drift'`) → de-duped [[../tables/loop_alerts]] + page + [[repair-agent]] enqueue, exactly like any P1 red. Cron-freshness/never-fired keeps a *dead* check visible too.

## Gotchas

- **SHA-behind needs `VERCEL_GIT_COMMIT_SHA`** (the deployed commit) as the origin/main proxy; unset locally ⇒ the check is skipped (no false positive). It only fires red after `shaGraceMs` (default 30m) so an in-progress deploy / self-update never pages.
- **Agent-kind alert is off [[../tables/agent_jobs]], not the heartbeat** — idle = green. The heartbeat only feeds last-ran/history.
- **Worker-restart clamp on queued stuck-jobs** ([[../specs/control-tower-stuck-jobs-clamp-on-worker-restart]], signal `loop:agent:spec-test`, verdict monitor-false-positive): `jobStuckSince` clamps the `queued`/`queued_resume` floor to `worker_heartbeats.started_at` — a worker that wasn't alive earlier can't have claimed earlier than it started, so `floor = max(updated_at/created_at, worker.started_at)`. Without this, a backlog enqueued during a worker-down window is mis-attributed as a stuck lane the moment the worker restarts (the originating false page: 8 spec-test rows queued at 10:45 by the regression-backlog sweep, box offline until 11:45, monitor at 12:00 read 75-min ages and went red even though the queue had only existed for 15 min of worker uptime). `building`/`claimed` jobs are NOT clamped — `claimed_at` already reflects the worker that picked them up. Null/malformed `started_at` ⇒ no clamp (preserves prior behavior, never lifts a real stuck-since forward). Unit-tested in `monitor.test.ts` (`npx tsx --test src/lib/control-tower/monitor.test.ts`).
- **Queue-aware self-update deferral** ([[../specs/control-tower-self-update-tile-queue-aware]], signal `loop:box`, verdict monitor-false-positive): `evalWorker`'s `behind && idle` branch now mirrors the worker's own self-update logic (`scripts/builder-worker.ts:4290` self-restart-defers-to-idle). The worker INTENTIONALLY parks an idle self-update while `{queued, queued_resume} > 0` so a cascade of queued builds isn't restarted between specs — reading that as "self-update stuck" was the monitor false positive. `buildControlTowerSnapshot` now passes the queued count (derived from the existing active-jobs read) AND `worker_controls.drain_for_update` into `evalWorker`. When `behind && idle && queued > 0 && !manualDrain` ⇒ GREEN "idle — update deferred · ${n} queued" (never red); `behindTooLong` now requires `queued === 0` in addition to the SHA-grace. A MANUAL queue restart (`drain_for_update=true`) still reds at grace, because that flag's whole purpose is "restart at idle regardless of the queue."
