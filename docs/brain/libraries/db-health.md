# libraries/db-health

The detection + classification + dedup + spec-templating brain behind the **DB Health Agent** ([[../specs/db-health-agent]] Phase 1) — a standing box agent that watches Postgres health and **proposes** fixes (retention / index / query-rewrite / index-drop / vacuum). It mirrors [[repair-agent]] (propose-don't-auto-apply) + [[control-tower-migration-drift-check]] (box-side DB introspection). Commissioned after the `loop_heartbeats` flood silently blinded the Control Tower — exactly the class of problem a DB agent catches *early*.

**File:** `src/lib/control-tower/db-health.ts` · box runners: `scripts/builder-worker.ts` `runDbHealthSlowQueryJob` / `runDbHealthSizeJob` / `runDbHealthJob` (see [[../recipes/build-box-setup]]).

## North star — surface-don't-auto-apply

DDL/deletes are higher-stakes autonomy ([[../operational-rules]] § North star): a bad index slows writes; a retention delete loses data. So the agent applies **zero** DDL/deletes — it detects read-only (reads `pg_stat_*`, runs `EXPLAIN`), classifies the root cause, **pre-authors** the matching fix spec, and **surfaces** it for one-tap owner Build (the [[agent_jobs]] `db_health` proposal + the Control Tower DB Health panel). A tool optimizing a bounded proxy (DB perf/size), supervised by Platform.

## Where it runs — the box, not the deployed runtime

Like the migration-drift check, detection runs on the box: `EXPLAIN` + `pg_stat_statements` + `pg_class`/`pg_stat_user_*` need a raw pooler connection the deployed Next runtime can't use. The box passes the read rows into this PURE module (no fs/pg/network at module load) so the LOGIC stays testable.

## Exports

- **Loop ids** `DB_HEALTH_SLOWQ_LOOP_ID` (`"db-health-slow-query"`) + `DB_HEALTH_SIZE_LOOP_ID` (`"db-health-size-sweep"`) + `DB_HEALTH_INSTANCE_LOOP_ID` (`"db-health-instance-saturation"`, [[../specs/db-health-instance-saturation-detector]] Phase 1) — re-exported from [[control-tower]] `registry.ts`; the loop_ids of the THREE box-emitted passes (`kind:'cron'` beats). Liveness tiles (green when beating) — findings surface as **proposals**, not by reddening the tile (Phase 2 lets an active instance finding also redden the tile since it's an ACTIVE-incident signal, unlike growth/bloat).
- **Taxonomy:** `DbHealthCause` (17: `seq_scan` · `no_index_match` · `sort_spill` · `full_aggregate` · `missing_limit` · `high_call_volume` · `bloat_stale_stats` · `unbounded_growth` · `missing_index` · `unused_index` · `bloat_vacuum_lag` + **instance-saturation** `statement_timeout_pressure` · `temp_spill_pressure` · `connection_saturation` · `cache_pressure` · `rollback_error_rate` + **request-volume/egress** `request_volume_pressure` (2026-07-08 re-tool)), `DbHealthFixKind` (`retention_cron` · `add_index` · `drop_index` · `query_rewrite` · `reduce_calls` · `vacuum_tuning` + **escalation-shaped** `raise_compute` · `raise_work_mem` · `investigate_timeouts` — advisory only, never auto-appliable), `DbHealthCategory` (`slow_query` · `growth` · `index` · `bloat` · `instance`). `FIX_KIND_BY_CAUSE` maps each cause → its fix (`high_call_volume` → `reduce_calls`, `statement_timeout_pressure`/`rollback_error_rate` → `investigate_timeouts`, `temp_spill_pressure` → `raise_work_mem`, `cache_pressure`/`connection_saturation` → `raise_compute`).
- **Input row shapes** the box feeds in: `SlowQueryRow` (a `pg_stat_statements` row), `TableSizeRow` (a [[../tables/db_table_size_history]] row), `IndexStatRow` (a `pg_stat_user_indexes` row + size + is_unique/is_primary), **`InstanceHealthInput`** (a periodic `pg_stat_database` + `pg_stat_activity` + `pg_roles.rolconfig` snapshot: `xactCommit`/`xactRollback`/`deadlocks`/`tempFiles`/`tempBytes`/`blksHit`/`blksRead` + `activeBackends`/`waitingBackends`/`maxConnections`/`statementsNearTimeout` + `nearTimeoutSamples?` (offender query text + how long each had been running — [[../specs/db-investigate-timeouts-instance]]) + `authenticatedStatementTimeoutMs` — the 8s ceiling from the 2026-07-02 incident), `NearTimeoutSample` (one near-timeout offender the box captured out of pg_stat_activity: `{ durationSec, query }` — enriches the `statement_timeout_pressure` evidence).
- **Thresholds** (the bounded proxy's guardrails, all tunable): `SLOW_QUERY_MIN_MEAN_MS` (100) · `SLOW_QUERY_MIN_TOTAL_MS` (30s) · `SLOW_PER_CALL_MEAN_MS` (50 — the slow-per-call vs high-call-volume boundary) · `SIZE_MIN_BYTES` (200 MB) · `GROWTH_FLAG_FRACTION` (0.25/day) · `SEQ_SCAN_SHARE_FLAG` (0.5) + `SEQ_SCAN_MIN_ABS` (10k) · `UNUSED_INDEX_MIN_BYTES` (50 MB) · `BLOAT_DEAD_RATIO_FLAG` (0.2) + `BLOAT_AUTOVACUUM_STALE_MS` (1d) · **Phase-2 trend:** `TREND_MIN_POINTS` (3) + `TREND_MIN_SPAN_DAYS` (2) · `GROWTH_TREND_HORIZON_DAYS` (30) + `GROWTH_TREND_CEILING_BYTES` (10 GB) · `BLOAT_TREND_MIN_RATIO` (0.1) + `BLOAT_TREND_RISE` (0.05) · **Instance-saturation:** `INSTANCE_ROLLBACK_RATIO_FLAG` (5% — incident was 7.43%) · `INSTANCE_TEMP_BYTES_WINDOW_FLAG` (100 GB cumulative — the fallback when no prior reading) + `INSTANCE_TEMP_BYTES_RATE_FLAG` (2 GB/hr — the preferred SELF-CLEARING signal, `db-health-temp-spill-rate`) · `INSTANCE_CACHE_HIT_FLOOR` (0.99 — incident was 0.9869) · `INSTANCE_CONN_UTIL_FLAG` (80%) · `INSTANCE_TIMEOUT_HEADROOM_FRACTION` (0.5 — a live query past half the `authenticated` `statement_timeout` is "near timeout") · `RETENTION_AWARE_TABLES` (don't re-propose retention — `loop_heartbeats` has its pruner) · `DB_HEALTH_SUNSET_ALLOWLIST` (sunset/retiring tables — `klaviyo_*`; skipped on every pass) · `DB_HEALTH_REPROPOSE_WINDOW_MS` (7d).
- **Accuracy filters (db-health-agent-accuracy spec):** `isForeignQuery(query)` — a query we don't own is **never** a proposal: matches the Supabase Realtime WAL decoder (`wal->>`), PostgREST internals (`pgrst_`), `pg_catalog` / `information_schema`, the `realtime`/`_realtime` schema, `supabase_admin`-role queries (mirrors the [[repair-agent]] `foreign-app-noise` class). `isSunsetQuery(query, allowlist?)` — a query touching a `DB_HEALTH_SUNSET_ALLOWLIST` table (`klaviyo_*`) gets no proposal (mirrors the [[../specs/control-tower-migration-drift-check]] allowlist via `isAllowlisted`). Both applied at the top of `analyzeSlowQuery`; the allowlist also short-circuits `analyzeGrowth`/`analyzeIndexUsage`/`analyzeBloat`.
- **The EXPLAIN root-cause brain:** `isSafeSelect(query)` (only EXPLAIN read-only SELECT/WITH, never a write — a `WITH` with any write keyword is rejected), `classifyExplainPlan(planText)` → `{ cause, seqScanTables, hint }` (order matters: disk-sort + full-aggregate beat a bare seq scan), `analyzeSlowQuery(row, planText|null)` → a `DbHealthFinding` over the impact floor (null below it, null for a foreign/sunset query). **Slow-vs-volume classifier:** keyed off `mean_exec_time` — `≥ SLOW_PER_CALL_MEAN_MS` (50ms) is a genuine per-call problem → EXPLAIN → index/rewrite (the orders-class win); below it the query only ranks by *volume* (e.g. 4ms × 1.27M calls = a hot poll) → cause `high_call_volume`, fix `reduce_calls` (cut call frequency / cache / a hot-predicate or, for an array `@>` predicate, a **GIN** index — never a vacuum), citing the call count + mean. **Shipped examples:** (1) `dbhealth:slowq:-7821780334453251234` — the top offender by volume was PostgREST's per-request RLS-context `set_config(...)` statement (38.9M calls, 0ms mean, no table touched ⇒ no index/vacuum lever) — the only fix is fewer authenticated REST round-trips. The dominant source was the always-on dashboard [[../dashboard/home|sidebar]] (`src/app/dashboard/sidebar.tsx`) polling badge counts; widening its poll 10s→30s cut the rate ~3×, the canonical landed `reduce_calls` win. (2) `dbhealth:slowq:-7726440967385220442` — the same RLS `set_config(...)` pattern re-emerged (high call volume, low per-call latency); followed-on widen 30s→60s to the sidebar's `fetchCounts()` poll, same lever. (3) `dbhealth:slowq:2430642232831032083` — the same set_config-class pattern re-emerged AGAIN; widening past 60s hurts badge freshness so the next lever was the RE-FIRING multiplier: the `fetchCounts` effect was deps'd on `[workspace.id, pathname]`, so every navigation tore down the interval and immediately re-issued all 13-17 requests on top of the 60s poll — dropping `pathname` from the deps keeps the interval alive across navigations (counts change on events, not routes). (4) `dbhealth:slowq:5250820927256751610` — the same set_config-class pattern surfaced a fourth time; with widening bottomed out and the navigation-tear gone, the next remaining lever was the BACKGROUND-TAB multiplier: any dashboard tab left open (an operator flipping between apps all day) kept firing the 13-17-request fan-out every 60s despite not being visible. Guarding the poll on `document.visibilityState === "visible"` skips the tick while hidden and refreshes immediately on `visibilitychange` back to visible, so badges are still fresh the moment the tab returns to the foreground (counts change on events, not tab focus, so deferring while hidden is safe). All four are reduce-calls wins targeting the same high-frequency authenticated round-trip source; the fourth stops the fan-out from firing at all for the (large) population of hidden dashboard tabs, not just its cadence or its re-firing. A null plan (a parameterized statement EXPLAIN couldn't plan) still falls back to a conservative text-based cause + a rewrite-review fix.
- **No-vacuum-from-slow-query-path guardrail ([[../specs/agent-mandate-hardening-dbhealth]] — rolled coaching baked into the classifier):** the slow-query path (`classifyExplainPlan` + `analyzeSlowQuery`) NEVER emits `bloat_stale_stats` / `vacuum_tuning`. Vacuum belongs to the size-sweep bloat path (`analyzeBloat` / `analyzeBloatTrend` — real dead-tuple + stale autovacuum evidence), never as the ambiguous "no Seq Scan isolated" fallback. Devi sat at 4.8/10 after 10 unstuck coaching attempts because the classifier fell through to `bloat_stale_stats` whenever the plan didn't isolate a Seq Scan; the baked-in guardrail replaces that fallback with: (a) any index-driven plan (Bitmap Index Scan / Index Scan / Index Only Scan / Bitmap Heap Scan present, no Seq Scan) → `no_index_match` → `add_index` (a covering / composite / GIN index — vacuum can't help an index-driven plan); (b) no scan node at all (pure Aggregate/Limit) → `high_call_volume` → `reduce_calls`; (c) `stddev_exec_time > mean_exec_time` (the "erratic plan" tell) → force `no_index_match` (plan instability = missing selective index, never bloat), with a GIN framing when the predicate is `@>`/`<@`; (d) no EXPLAIN plan available (parameterized statement) → `high_call_volume` with the "vacuum belongs to the size-sweep bloat path" explanation, never `seq_scan`/`bloat_stale_stats`; (e) belt-and-suspenders — if `fixKind` ever computes to `vacuum_tuning` inside `analyzeSlowQuery`, snap it back to `reduce_calls` before the finding leaves the function. Regression-locked by the `agent-mandate-hardening-dbhealth guardrails` block in `db-health.test.ts` (the exact dismissed shapes from the coaching — tickets `tags @> $3` at 4ms×1.27M, sms_campaign_recipients `message_sid = $1` at 31ms×157k — must resolve to a non-vacuum fix, plus the invariant grid over erratic/bitmap/no-plan/aggregate shapes).
- **The size/index/bloat detectors:** `analyzeGrowth(latest, prior)` (unbounded growth vs the prior daily snapshot — none on the first sweep), `analyzeIndexUsage(tables, indexes)` (missing-index = high seq-scan share on a big table; unused-index = `idx_scan=0` non-PK/non-unique index ≥50 MB → drop), `analyzeBloat(tables, now)` (high dead-tuple ratio + stale autovacuum).
- **The instance-saturation detector ([[../specs/db-health-instance-saturation-detector]] Phase 1):** `analyzeInstanceHealth(input, thresholds?)` — PURE classifier over the box's periodic `pg_stat_database` + `pg_stat_activity` + `pg_roles.rolconfig` snapshot. Fires ≥0 `category:'instance'` findings, one per tripped signal, in a stable order (statement_timeout_pressure → temp_spill_pressure → rollback_error_rate → cache_pressure → connection_saturation). Each finding QUOTES the offending numbers in `evidence` (rollback ratio × commit/rollback counts, temp_bytes × temp_files, cache-hit ratio × blks_hit/blks_read, connection util × active+waiting/max_connections, the `authenticated` `statement_timeout` ceiling) so the operator can verify without a separate DB probe — mirrors how `analyzeBloat`/`analyzeSlowQuery` quote stats. Signatures are `dbhealth:instance:<cause>` (one per cause). Fills the blind spot the per-query slow-query pass has for the 2026-07-02-class outage.
- **Advisory templating for instance findings ([[../specs/db-health-instance-saturation-detector]] Phase 2):** `buildFixSpecMarkdown` resolves guidance by CAUSE first (five per-cause paragraphs — statement_timeout_pressure / temp_spill_pressure / rollback_error_rate / cache_pressure / connection_saturation) then falls back to the coarser fixKind paragraph. Every instance paragraph opens with "**Owner-approval-only — instance-saturation signal, no auto-apply.**" and quotes the finding's `impact` verbatim so the operator sees the exact numbers that tripped the flag. Compute/timeout changes are HIGH-STAKES (infra resize / per-connection setting / query rewrite) — the agent NEVER auto-authors a migration or apply-script for a resize, mirroring the DDL/delete stance in [[../operational-rules]] § North star. Phase 2 also lets `runDbHealthInstanceJob` beat `ok:false` when a finding is live, so the DB Health instance tile reddens on an ACTIVE incident (unlike growth/bloat, which are slow-burn advisories).
- **The Phase-2 trend detectors ([[../specs/db-health-agent]] Phase 2):** read the whole [[../tables/db_table_size_history]] window (not one reading) and project it forward. `analyzeGrowthTrend(history, now)` — least-squares fit (`linearFit`, exported) of each big table's size series; flags the ones projected to cross `GROWTH_TREND_CEILING_BYTES` (10 GB) within `GROWTH_TREND_HORIZON_DAYS` (30d), catching a steady climb *under* `analyzeGrowth`'s +25%/day spike threshold. `analyzeBloatTrend(history, now)` — flags a big table whose dead-tuple ratio is RISING (≥`BLOAT_TREND_RISE`, 5 pts) across the window to ≥`BLOAT_TREND_MIN_RATIO` (10%, **below** the single-snapshot 20% flag) while autovacuum isn't keeping up (its `last_autovacuum` hasn't advanced across the window, or is stale); skipped if autovacuum advanced AND isn't stale (transient churn it's handling). Both require ≥`TREND_MIN_POINTS` (3) snapshots spanning ≥`TREND_MIN_SPAN_DAYS` (2d) — fewer ⇒ no projection (honest). **Both reuse the Phase-1 signatures** (`dbhealth:growth:<table>` / `dbhealth:bloat:<table>`) so a spike + a trend for one table collapse to ONE proposal under `dedupeFindings` — no new taxonomy / migration / panel. The spec evidence quotes the fitted slope, the projection, and sampled history points.
- **Ranking + dedup:** `rankFindings` (by impact score desc), `dedupeFindings` (collapse same-signature, keep the highest), `summarizeFindings` (one-line beat detail).
- **Spec templating:** `buildFixSpecMarkdown(finding)` → the single-phase fix spec (`⏳`, owner platform) with the cause-specific guidance + the EXPLAIN/stat **evidence quoted verbatim** + the `**DBHealth-signature:**` / `**DBHealth-fix:**` machine markers.
- **Parse-safe markdown shape (`dbhealth-spec-evidence-survives-parse`):** the template MUST follow the proven author-spec shape or [[brain-roadmap]] `parseSpec` mangles it — the **intent paragraph FIRST** (it, and only it, becomes `specs.summary`; a `**Owner:**`/meta line placed before it gets swallowed as the summary), then the Owner/Parent + `**DBHealth-*:**` meta lines, then a SINGLE `## Phase 1` whose BODY carries the guidance **plus a `### Evidence` subsection** with `finding.evidence`. A top-level `## Evidence` section (the old shape) survives in NEITHER the summary NOR a phase body, so Bo read an **empty Evidence block and stalled `needs_input`** while trying to author the `CREATE INDEX` (the `db-index-q-…` / `db-reduce-calls-…` incident — both stored with 0 phases + a metadata-blob summary). `### Verification` under the phase splits into the `verification` column. `finding.evidence` also gets a non-empty fallback so the section is never hollow. Regression-locked by the `dbhealth-spec-evidence-survives-parse` test in [[db-health]]'s test file (generates the markdown → runs the real `parseAuthoredSpecMarkdown` + `extractPhaseBodies` → asserts summary=intent, 1 phase, EXPLAIN evidence in the phase body, verification populated).
- **Surface (admin client):** `enqueueDbHealthProposal(admin, finding)` → enqueue ONE deduped `db_health` [[agent_jobs]] proposal (`needs_approval`; the pre-authored `spec_body` + structured `finding` ride the `db_health_build` `pending_actions` action — un-clobberable by an approval that overwrites `instructions`, which now carries panel metadata only). `getDbHealthPanel(admin, workspaceId)` → READ-ONLY `{ topTables, slowQueries, proposals, lastSizeSweepAt, lastSlowQueryAt }` for the dashboard.

## The three box passes

- **`runDbHealthSlowQueryJob`** (~hourly): top 25 `pg_stat_statements` offenders by `total_exec_time` → `EXPLAIN` each safe SELECT (plain `EXPLAIN`, no `ANALYZE` — non-executing, safe on prod; retries `EXPLAIN (GENERIC_PLAN)` for a parameterized statement on PG16+) → `analyzeSlowQuery` → surface top-N. Beats with `produced.slow_queries` for the panel.
- **`runDbHealthSizeJob`** (daily): snapshot every `public` table's size + stats into [[../tables/db_table_size_history]] (so growth is computable), read the prior batch, run `analyzeGrowth` + `analyzeIndexUsage` + `analyzeBloat`. **Phase 2:** also reads a `DB_HEALTH_TREND_WINDOW_DAYS` (21d) history window — bounded to the currently-big (≥`SIZE_MIN_BYTES`) tables so the row count stays under PostgREST's default cap — and runs `analyzeGrowthTrend` + `analyzeBloatTrend` alongside. Surface top-N. Beats with `produced.top` (biggest tables) + `produced.history_rows` (trend-window size, honest about a sparse window).
- **`runDbHealthInstanceJob`** (~15 min, [[../specs/db-health-instance-saturation-detector]] Phase 1+2): reads `pg_stat_database` (rollback ratio · deadlocks · temp spill · cache-hit) + `pg_stat_activity` (active+waiting backends vs `max_connections` · live queries past a fraction of the `authenticated` `statement_timeout`) + `pg_roles.rolconfig` (the ceiling itself). Feeds an `InstanceHealthInput` into `analyzeInstanceHealth`. Cadence is short (active-incident timescale) so a 2026-07-02-class outage doesn't wait for the daily sweep. Beats with `produced.top_findings` + `produced.signals` (rollback ratio, cache-hit ratio, temp bytes, connection utilization, statement_timeout ceiling, near-timeout offender samples) for the panel. **Phase 2:** surfaces each ranked finding as a deduped `db_health` proposal via `surfaceDbHealthFindings` → `enqueueDbHealthProposal` (`dbhealth:instance:<cause>` signature, `DB_HEALTH_REPROPOSE_WINDOW_MS` honored) so the DB Health panel renders active-incident signals alongside growth/bloat proposals, AND flips the tile's heartbeat to `ok:false` (`produced.status:'active_incident'`) when ≥1 finding is live — the cron-freshness lever that reddens the Control Tower tile. Ok=true / status="ok" when no findings. **[[../specs/db-investigate-timeouts-instance]]:** the near-timeout `pg_stat_activity` probe is FILTERED to `usename = 'authenticated'` (only queries under the role subject to the 8s ceiling can be killed by it — `supabase_admin` dashboard SELECTs + `postgres` maintenance queries have their own timeouts and produced false positives before this filter), and CAPTURES the top 3 offender rows (query text + how many seconds they'd been running); `analyzeInstanceHealth` splices them into the `statement_timeout_pressure` finding's evidence as `Near-timeout offender(s): Ns — <preview>` lines so the finding NAMES the query to route to the right slow-query fix, not just "1 live query near timeout" (un-actionable without it). Mirrors the `isForeignQuery` role filter in the slow-query path.
- All three are fire-and-forget in the poll loop with an in-flight guard (never overlap/stall the ~30s poll), cap proposals to `DB_HEALTH_MAX_PROPOSALS_PER_PASS` (5), and **never throw** (a pass failure can't break the box). No DB password on this host ⇒ the pass beats `status:'skipped'` (honest, never a false "healthy").

## 2026-07-08 re-tool — temp-spill attribution · infra filter · request-volume/egress

Triggered by a real incident: a single **unindexed disk-sort** (the storefront LTV proxy paging every subscription ordered by `created_at` with no supporting index — [[storefront-ltv-proxy]]) spilled **314 GB** to temp files (98% of all instance temp-spill) and dragged the instance for a week, while Devi's instance pass proposed only a generic `raise_work_mem` bump (7/2) and never named the offender. Post-mortem also found the 8M-call PostgREST `set_config` request preamble being re-proposed as a doomed `reduce_calls` spec every 7 days, and NO signal at all for the ~78K req/hr · ~182K rows/hr internal-polling **egress** firehose. Three durable fixes:

- **Temp-spill offender attribution (`db-health-temp-spill-attribution`).** `pg_stat_database.temp_bytes` is an instance aggregate with no query attribution, so the instance pass could only say "the instance is spilling". Now `runDbHealthInstanceJob` also reads the top `pg_stat_statements` rows by `temp_blks_written` and feeds them as `InstanceHealthInput.tempOffenders` (`TempSpillOffender[]` — `{ queryid, query, calls, tempBytes }`). `analyzeInstanceHealth`'s `temp_spill_pressure` branch NAMES the top spillers in the evidence and, when the dominant spiller is an APP query (not infrastructural), points the fix at an index on its ORDER BY/GROUP BY or a rewrite/aggregate RPC — "a work_mem bump is the fallback, not the fix." (Would have fingered the subscriptions scan in one pass.) The aggregate flag still fires with no offenders (back-compat).
- **Self-clearing temp detection via RATE (`db-health-temp-spill-rate`).** `pg_stat_database.temp_bytes` is CUMULATIVE since the last stats reset — and Supabase's `postgres` role is **not** superuser, so `pg_stat_reset()` is permission-denied and the counter can't be zeroed after a fix (`pg_stat_statements_reset()` IS grantable and works). A pure cumulative flag therefore stays lit forever once tripped (the 314 GB the subscriptions scan left behind never drops). So `runDbHealthInstanceJob` reads the PRIOR instance heartbeat's `produced.signals.temp_bytes` + `ran_at` and passes `InstanceHealthInput.tempBytesPrev` + `tempReadingAgeHours`; `analyzeInstanceHealth` prefers the **rate** (Δbytes / Δhours) against `INSTANCE_TEMP_BYTES_RATE_FLAG` (2 GB/hr) — an active spiller keeps the rate high, a fixed one drops to ~0 so the finding **self-clears**. Falls back to the cumulative `INSTANCE_TEMP_BYTES_WINDOW_FLAG` (100 GB) only when there's no prior reading (first pass / unreadable), preserving acute-incident detection. The evidence quotes the rate + Δ + age; `score`/`title`/`impact` switch to the rate when available.
- **Infrastructural-query filter (`isInfrastructuralQuery`).** New export matching the PostgREST per-request preamble (`select set_config('role'|'request.*'|'search_path'|…)` / the normalized `SELECT set_config($2,$1,$3)` at statement start) + GoTrue auth reads (`auth.` schema, `mfa_*`/`refresh_tokens`/`flow_state`/…, or GoTrue-distinctive columns). These are platform-issued per request — no endpoint/predicate/poll to fix, so a `reduce_calls` spec is un-buildable. `analyzeSlowQuery` now returns `null` for them (stops the 7-day re-propose churn of queryid `-7821780334453251234`). Distinct from `isForeignQuery` (which filters queries we don't own); an infra query is ours-by-platform but not spec-fixable. Anchored so a mutation CTE that merely references `set_config` for response headers is NOT swept in (would otherwise over-suppress real app writes like the `account_usage_snapshots` INSERT).
- **Request-volume / egress escalation (`analyzeRequestVolume`, cause `request_volume_pressure`).** New pure analyzer over an aggregate snapshot (`RequestVolumeInput`: `windowHours` from `pg_stat_statements_info.stats_reset`, `totalCalls`, `totalRows`, `topByCalls`, `topByRows`). Fires when req/hr ≥ `REQUEST_VOLUME_CALLS_PER_HR_FLAG` (100k) OR rows-shipped/hr ≥ `REQUEST_VOLUME_ROWS_PER_HR_FLAG` (100k) over a window ≥ `REQUEST_VOLUME_MIN_WINDOW_HOURS` (1h). It's an ESCALATION (fixKind `reduce_calls`, signature `dbhealth:request-volume:instance`, category `slow_query`) that NAMES the top internal callers (labelling `[infrastructural]` ones) — rows-shipped is the egress meter — and its spec guidance is upstream levers (widen poll intervals · cache/batch · reuse authenticated connections · replace row-shipping reads with aggregate RPCs), never a per-query index. Wired into `runDbHealthSlowQueryJob` (reuses its `pg_stat_statements` access; additive + best-effort). This is the signal that was missing when egress crept.
- **Related fix (the offender itself):** [[storefront-ltv-proxy]] `estimateSubLTV` was rewritten from a full-scan-then-aggregate-in-JS pattern into the `public.estimate_sub_ltv` RPC (migration `20260708120000`) + a GIN index on `subscriptions.items` — 29–65× faster AND correct (the old `.in()` order reads silently hit Supabase's 1000-row cap, undercounting renewal survival ~20% and zeroing the subscriber-LTV cross-check). See [[../tables/subscriptions]].
- **Self-clearing SLOW-QUERY detection via RATE ([[../specs/db-reduce-calls-q-1756037457588317045]] Fix 1).** Same shape as the temp-spill rate flag above, applied to per-query `pg_stat_statements` counters. `calls` / `total_exec_time` are CUMULATIVE since the last `pg_stat_reset()` — and Supabase's `postgres` role can't call it — so a cumulative `SLOW_QUERY_MIN_TOTAL_MS` flag (30s) stays lit forever once tripped: a cache/batch fix that reduces the RATE going forward can't lower the historical counter. This is the exact regression Phase 1's Fix 1 caught — after the 2s `getSpec` TTL cache landed, the DB Health Agent's next pass STILL flagged `dbhealth:slowq:-1756037457588317045` with impact `0ms mean × 220,990 calls = 32s total` (the cumulative floor tripped even though the post-cache call rate had dropped). `analyzeSlowQuery` now accepts optional `priorRow?: SlowQueryRow` + `readingAgeHours?: number`; when both are present + positive, it computes `Δcalls / Δhr` and `Δtotal_ms / Δhr` and BOTH must be under `SLOW_QUERY_MIN_CALLS_PER_HR` (3,600 — ~1 call/sec sustained) + `SLOW_QUERY_MIN_TOTAL_MS_PER_HR` (30s/hr — the cumulative floor projected over one hour) for the finding to self-clear. A counter reset (`row.calls < priorRow.calls`) yields a negative delta — the rate branch stays silent and the cumulative flag remains authoritative. Absent priorRow / non-positive age ⇒ fall back to cumulative (first pass / fresh box preserves acute-incident detection). `runDbHealthSlowQueryJob` in `scripts/builder-worker.ts` reads the PRIOR slow-query heartbeat's `produced.slow_query_rows` (a compact `{queryid, calls, total_exec_time}` snapshot the SAME pass writes for the NEXT pass to read) + `ran_at` and threads them into the analyzer for every current row. Regression-locked by three tests in `db-health.test.ts` — the exact 220,990-call self-clear, an active-spike-still-fires check, and a counter-reset falls-back check.
- **Post-deploy measurement — `scripts/_measure-egress-drop.ts`** ([[../specs/cut-internal-egress-pooler-and-spec-rpcs]] Phase 4). Read-only measurement tool that runs the SAME `pg_stat_statements` aggregate `analyzeRequestVolume` uses (`windowHours` from `pg_stat_statements_info.stats_reset`, `sum(calls)`/`sum(rows)`, top-8 by calls + rows) plus targeted slices for `set_config` share, `specs`/`spec_phases`/`agent_jobs`/`claude_health` per-hour rates, and the new `get_spec_with_phases` / `list_specs_with_phases` / `control_tower_snapshot` RPC call rates. Prints each rate against the recorded **pre-Phases-1-3 baseline** (~200K req/hr, ~182K rows/hr, ~50K/hr on specs, ~44K/hr on spec_phases, ~45% set_config share — the numbers from the spec header) with a ▼/▲ delta, and reports whether `request_volume_pressure` would fire on the current window. Use to confirm the Phase-4 verification claim: `req/hr + rows/hr are materially below the baseline; set_config preamble share + raw specs/spec_phases call rate are materially reduced; Devi's request_volume_pressure is under the flag (or clearly lower)`. Read-only — never mutates. Run: `npx tsx scripts/_measure-egress-drop.ts`.

## Dedup discipline

One open proposal per finding **signature** (`dbhealth:<category>:<key>`, e.g. `dbhealth:growth:loop_heartbeats`, `dbhealth:unused-index:<name>`, `dbhealth:slowq:<queryid>:<table>`). `enqueueDbHealthProposal` skips if a **live** proposal (`needs_approval`/queued/…/`needs_attention`) exists for the signature **or** a non-dismissed proposal for it was built within `DB_HEALTH_REPROPOSE_WINDOW_MS` (its fix is deploying — don't flap while the condition resolves). Never proposes retention for a `RETENTION_AWARE_TABLES` table, anything for a `DB_HEALTH_SUNSET_ALLOWLIST` (sunset) table or a foreign/not-ours query, an index already present, or a PK/unique index for drop.

## Owner action — the gate

`POST /api/developer/control-tower/db-health` (`{ jobId, action:'build'|'dismiss' }`, owner-gated). **Dismiss** resolves the proposal directly (`completed`). **Build** flips it to `queued_resume`; the box's `runDbHealthJob` re-claims it on the `db_health` lane, commits the pre-authored fix spec to `main` (`putFileMain`), and queues the actual `build` job (auto-build-deduped via `hasActiveBuildForSlug`) — the build runs through the normal pipeline + lands the migration/apply-script in a PR for owner review. NOTHING touches schema/data without that tap.

## Gotchas

- **No migration for the proposal queue** — `db_health` is a free-text [[agent_jobs]] `kind`; the surfaced fix parks in the existing `pending_actions` jsonb (`type:'db_health_build'`, `spec_slug` + `spec_title` + the un-clobberable `spec_body` + `finding`). Only [[../tables/db_table_size_history]] is new.
- **The spec is pre-authored at detection, committed on Build** — unlike [[repair-agent]] (which runs an LLM diagnosis), DB-health diagnosis is deterministic (EXPLAIN classification), so the full spec body is built at detection time and the box only writes it to `main` when the owner taps Build (no orphan spec files for dismissed findings).
- **The diagnostic is un-clobberable, and the agent NEVER authors an empty spec** (`db-health-spec-body-robust` + `spec-body-never-silently-empty`) — the build-critical `spec_body` **and** the structured `finding` it re-renders from live on the `db_health_build` `pending_actions` action (which approve mutates by **status only**), NOT on the free-text `instructions` a manual approval might overwrite with a plain string. The owner-Build resume (`resolveDbHealthSpecBody`) reads the body from the action first, falls back to the older `instructions.spec_body`, and if it's still empty **re-renders** `buildFixSpecMarkdown(finding)`; only a genuinely unreconstructable proposal goes `needs_input` (a clear reason) — the resume refuses to author a 0-byte/phaseless body. Belt-and-suspenders **at every author-write** (spec-body-never-silently-empty Phase 1): (a) the DB Health render/resume already surfaces `needs_input` / `needs_attention` on an unreconstructable body; (b) the [[author-spec]] chokepoint (`authorSpecRowStructured` + `authorSpecRowFromMarkdown`) throws `EmptyPhaseBodyError` **before** the DB write on ANY phase whose body is empty/whitespace — an un-buildable phase row never reaches `public.spec_phases`; (c) the generic builder gate (`specHasBuildableContent`/`unbuildableReason`) parks the job as **`needs_attention`** (visible + actionable, NOT `failed` which hides the reason from the operator queue) on a 0-content spec, so the proposal STAYS LIVE (the enqueue dedup counts `needs_attention` as live → no re-propose flap while the operator fixes it). This is why `db-index-orders` shipped a 0-byte spec + silently `completed` — the failure is now surfaced at every point it can occur.
- **`EXPLAIN` never `ANALYZE`s in Phase 1** — plain `EXPLAIN` doesn't execute the query (safe on prod, and a normalized `$1` statement can't be ANALYZEd anyway). The plan still shows Seq Scan / estimated rows — enough to classify. `EXPLAIN (ANALYZE, BUFFERS)` on confirmed-safe SELECTs is a Phase 2 option.
- **The agent is watched too** — all three passes are `kind:'cron'` MONITORED_LOOPS tiles ([[control-tower]] `registry.ts`); a dead pass shows red via cron freshness, with `registeredAt` gracing the first-run window.
- **Why an INSTANCE pass exists** ([[../specs/db-health-instance-saturation-detector]]) — the slow-query pass EXPLAINs each statement in isolation on an idle pooler connection so it can never see instance-level saturation (rollback ratio, temp spill, cache-hit floor, connection utilization, per-role `statement_timeout`). The 2026-07-02 outage sat there for ~30 min while the slow-query pass reported "healthy". The instance pass is that missing lens; it runs on its own short cadence so an active incident surfaces in ≤15 min, not on the daily sweep.
