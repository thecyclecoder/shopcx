# github-pr-resolve.ts

`src/lib/github-pr-resolve.ts` — the [[../integrations/github-webhook|GitHub webhook]]'s two **mirror gates** over open `claude/*` PRs. North star (supervisable autonomy): both are bounded proxies the owner supervises (kill-switch + every action surfaced), neither force-pushes through a guardrail.

1. **Dirty-PR Resolver detection** ([[../specs/dirty-pr-resolver-agent]]) — the CONFLICTING half: detect dirty `claude/*` PRs and enqueue ONE resolver job each; the box agent (`runPrResolveJob`) owns the actual merge + the resolve-vs-rebuild-vs-human decision.
2. **Auto-merge gate** ([[../specs/auto-ship-pipeline]] Phase 1 / Gate A) — the READY half: squash-merge + delete branch ONE ready (mergeable + all-checks-green) `claude/*` PR, directly via the GitHub REST API (no box worker — the merge is one API call). Automates the owner's rubber-stamp "merge" click; serialized, sync-aware, owner kill-switch.

Touches **`claude/*` build branches only**.

## Exports

- **`verifyGithubWebhook(rawBody, signatureHeader, secret) → boolean`** — verify the `X-Hub-Signature-256` header GitHub sends (`sha256=<hex>` = HMAC-SHA256(secret, raw body)). Constant-time (`timingSafeEqual`); rejects a missing/malformed header, an unconfigured secret, or a length/digest mismatch. Mirrors [[meta]] `verifyMetaWebhookSignature` — the **raw bytes as received** MUST be used (JSON re-encoding breaks the digest). Secret = `GITHUB_WEBHOOK_SECRET`.
- **`detectAndEnqueueDirtyPrs(admin?) → DirtyPrResult`** — the webhook's whole job: list open PRs, keep `claude/*` ones, check each `mergeable`. For a newly-`CONFLICTING` one it FIRST checks `findAlreadyMergedDuplicate` ([[../specs/dirty-pr-resolver-duplicate-detection]]): if the PR's work already merged via a sibling build → **close it + delete the branch with a comment** (`closeDuplicatePr`) and enqueue **nothing**; otherwise enqueue a deduped `pr-resolve` job. Best-effort + bounded — a transient GitHub error on one PR doesn't block the rest. Returns `{ checked, conflicting, enqueued, closedDuplicate, prs[] }`.
- **`enqueuePrResolveJob(admin, { workspaceId, prNumber, branch, reason? }) → { enqueued, reason? }`** — insert a `kind='pr-resolve'` [[../tables/agent_jobs]] row (`spec_slug = pr-{number}`, `spec_branch = branch`, `pr_number`, `instructions = {pr_number, branch, reason}`). **Idempotent:** no-op if an active pr-resolve job already exists for this PR (dedupe key = `pr-{number}`), so a burst of `push` + `synchronize` events for the same PR enqueues once. **Retry-capped ([[../specs/pr-resolve-retry-cap-and-fold-closes-orphan-pr]] Phase 1):** counts EVERY pr-resolve job ever enqueued for the PR (any status) **INCLUDING `needs_attention` rows** ([[../specs/pr-resolve-retry-cap-and-fold-closes-orphan-pr]]); excludes only the exhausted-cap SENTINEL marker row (`error` starts with "pr-resolve retry cap reached"), which surfaces the owner but is NOT an attempt — at/over `MAX_PR_RESOLVE_ATTEMPTS` (3) genuine attempts it stops enqueueing and surfaces the PR to the owner **once** (a single `needs_attention` pr-resolve sentinel row + a `notifyOpsAlert`) so no PR loops the resolver forever. **Why:** advisory-supersede parks a pr-resolve job to `needs_attention`, but the old counter EXCLUDED `needs_attention`, so the dirty-PR backstop would re-enqueue every pass → infinite storm (the 7h / 61-job run on #1893). The fix: every `needs_attention` pr-resolve row (advisory-supersede park from `builder-worker.ts`, etc.) DID run the resolver to a verdict and MUST count; the cap now reliably parks ONCE.
- **`findAlreadyMergedDuplicate(admin, branch) → { specSlug, mergedBranch, mergedPr } | null`** ([[../specs/dirty-pr-resolver-duplicate-detection]] Phase 1) — branch-keyed duplicate probe: map the PR branch → its originating `kind='build'` job (spec_slug + workspace + instructions) → `findMergedSiblingBuild` ([[agent-jobs]]) for a SIBLING build of the same spec + same phase scope that already **merged**. Such a PR is unresolvable by definition (its diff is already on `main`). Never guesses the spec from the branch name — reads the build job that created it; no build-job mapping ⇒ `null` (resolve normally).
- **`closeDuplicatePr(prNumber, branch, comment) → boolean`** — close a duplicate `claude/*` PR + delete its branch with an explanatory comment (best-effort comment/branch-delete). Used by both gates instead of resolving when the work already merged.
- **`autoMergeReadyPrs(admin?) → AutoMergeResult`** (Gate A) — list open `claude/*` PRs, find the READY ones (open · non-draft · `mergeable === true` · `mergeable_state === "clean"`), squash-merge **ONE** (serialized) + delete its branch via the GitHub REST API. No-op when the kill-switch is off or a sync is active. Beats the Control Tower (`loop_id = AUTO_MERGE_GATE_LOOP_ID`, `kind:'reactive'`) once per pass in a `finally`. Returns `{ enabled, syncActive, checked, ready, buildGateBlocked, accumulationBlocked, goalBoundBlocked, testsGateBlocked, foldGateBlocked, merged (0|1), mergedPr?, prs[] }`. **Fold gate (fold-pr-auto-merge + goal-fold):** a `claude/fold-*` OR `claude/goal-fold-*` PR is a brain-doc merge of ALREADY-shipped work the SYSTEM authored — the box worker's `runFoldJob` batches every owner-verified spec into ONE `claude/fold-…` branch; `runGoalFoldJob` folds ONE complete goal into a `claude/goal-fold-…` branch (the post-M5 `finalizePromotedGoal` lane). So the build / accumulation / pre-merge-tests / goal-bound gates DO NOT apply — they each guard a not-yet-shipped feature build reaching main. Inside the serialize block, a branch matching `claude/fold-` **or** `claude/goal-fold-` takes a dedicated path: require ONLY mergeable+clean (`isPrReady`) AND that its OWNING fold/goal-fold job is completed/succeeded — resolved by [[agent-jobs]] `getBranchBuildSuccess(branch)`, which matches the job to its PR via the `spec_branch` column (the worker stamps `spec_branch = claude/(goal-)fold-…`; `BRANCH_OWNING_KINDS` now includes BOTH `'fold'` and `'goal-fold'`). On pass it falls through to the SAME squash-merge action build PRs use; on fail it counts into `foldGateBlocked` and is left for the next pass. A (goal-)fold merge counts as the one merge for the serialized pass (the post-merge push webhook drives the next). **Safety rail intact:** a branch owned by NOTHING (a manual/untracked push with no `agent_jobs` row, e.g. a hand-pushed `claude/order-now-…`) reads `getBranchBuildSuccess → ok:false ("no build job owns this branch")` and is STILL refused + left for the owner — only a COMPLETED system build/fold/goal-fold job clears this gate. *(Goal-fold recognition fixed live 2026-06-29: `'goal-fold'` was absent from `JobKind`/`BRANCH_OWNING_KINDS` and the gate's prefix matched `claude/fold-` only, so goal-fold PR #866 (`claude/goal-fold-mqzkd5po`) fell into the build-gate path and sat unmergeable as "no build job owns this branch" until a human merged it.)* **Goal-bound guard (spec-goal-branch-pm-flow M4/M5):** Gate A merges a spec branch straight to `main`, which is correct ONLY for a ONE-OFF spec. A GOAL-BOUND spec (resolves via [[agent-jobs]] `resolveGoalSlugForSpec`) is HANDED OFF — never jumped to main here — because it promotes spec→`goal/{slug}` (Gate B / M4) and the whole goal lands on main atomically (Gate C / M5, the only shipped-writer for goal-bound specs). Jumping it to main via Gate A would (a) ship it outside the atomic promotion and (b) double-stamp (`applyMergedBuildEffects` then `applyGoalPromotionEffects`). The guard sits FIRST (before the accumulation/tests gates), counts into `goalBoundBlocked`, and fails OPEN (a resolve error ⇒ treat as one-off; the gates below still protect the merge). **Accumulation gate (spec-goal-branch-pm-flow M2):** between the build gate and the tests gate, a MULTI-PHASE spec branch is refused promotion until EVERY phase is built on the branch (`isSpecAccumulationComplete` — each phase carries a `build_sha` or is terminal; NO phase still `planned`). Under M1's branch-accumulation model the phases build one-by-one onto ONE persistent `claude/build-{slug}` PR (no per-phase main merge), so without this gate the existing auto-merge would promote a PARTIAL spec the moment its first phase went green. A one-shot/single-phase spec (0–1 phases) passes trivially; fails OPEN on a PM-read error (the tests/build gates still apply). This is the M5 SEAM — M5 ([[../specs/spec-goal-branch-pm-flow]]) owns the actual atomic promotion + the `build_sha`→`shipped` flip; M2 only keeps auto-merge from jumping a half-accumulated branch. (`isSpecAccumulationComplete` was promoted to the [[specs-table]] SDK in M3 so this gate, the M3 pre-merge-test trigger [[agent-jobs]] `maybeEnqueuePreMergeSpecTestOnAccumulation`, and [[agent-jobs]] `isSpecPromoteEligible` all read the IDENTICAL predicate; this gate imports it.) **Post-merge hook (chain-and-cardstate-under-automerge Phase 1):** right after a successful squash-merge it calls [[agent-jobs]] `handleAutoMergedBuildBranch(branch, mergeSha)` — the merge happened server-side, so this advances the build's post-merge state (flip the job `merged` → roll up the card status → queue the next ⏳ phase of a "Build all" chain) **without** waiting for a board-render reconcile. `squashMergeAndDelete` now also returns the squash commit's `mergeSha` (the card's `last_merge_sha`). Idempotent + best-effort. **Deploy-watch hook (deploy-health-rollback-guardian Phase 1):** the same squash-merge triggers a Vercel deploy, so it also calls [[deploy-guardian]] `openDeployWatch({ branch, prNumber, mergeSha })` — Reva snapshots the pre-deploy error/loop baseline + opens a `pending` [[../tables/deploy_watches]] over the canary window; [[../inngest/deploy-guardian-cron]] stamps the verdict once the window elapses. Best-effort + never throws.
- **`compareGoalBranchToSpecBranch(goalBranch, specBranch) → GoalBranchCompareResult | null`** ([[../specs/parallel-build-serialized-merge-and-deadlock-autobreak]] Phase 3) — compare goal branch vs spec branch to determine if a rebase is needed before merge. Returns `{ status, aheadBy, behindBy, baseSha, headSha }` or `null` on failure (fail closed — no rebase decided without a positive compare). `status ∈ { identical, ahead, behind, diverged }`: identical = spec branch head equals goal branch head (no-op merge); ahead = spec branch is ahead of goal (safe fast-forward); behind = spec branch is behind goal (already integrated, no-op); diverged = both are ahead (sibling has advanced goal beneath us, rebase needed before merge). Used by `mergeSpecBranchIntoGoalBranch` to gate the rebase logic.

- **`decideGoalBranchRebaseMerge(input: { compare, priorRebaseAttempts?, maxAttempts? }) → GoalBranchRebaseDecision`** (pure predicate — parallel-build-serialized-merge-and-deadlock-autobreak Phase 3) — decide whether to merge, rebase-then-merge, skip, or escalate. **Fails CLOSED on missing `compare.status`** (malformed payload → escalate). Returns `{ action: 'merge' | 'rebase-then-merge' | 'skip' | 'escalate', reason }`: `skip` on identical or behind (no-op); `merge` on ahead (fast-forwardable); `rebase-then-merge` on diverged with attempts remaining (goal has advanced, bring spec up to date first); `escalate` on diverged with attempts exhausted (irreducible conflict or too many attempts — two parallel goal-mates genuinely touched the same lines, never force-merge). Kept pure so the named-failing-state — "two parallel goal-mates touching same file merge cleanly via serialized rebase" — is unit-testable. See `src/lib/goal-branch-rebase-merge.test.ts`.

- **`withGoalBranchMergeLock(goalBranch, body) → Promise<T>`** (Phase 3 seam — internal serialization) — per-goal in-process promise chain that queues concurrent invocations of `mergeSpecBranchIntoGoalBranch` for the SAME goal one-at-a-time; different goals proceed independently. Wrapped inside `mergeSpecBranchIntoGoalBranch` so the standing pass and a GitHub webhook cannot race the compare→rebase→merge sequence for one goal. Two in-flight goal merges for the same goal chain; two for different goals run in parallel (cross-goal independence preserved). Exported test accessor `_withGoalBranchMergeLockForTests` allows pure-predicate tests to observe/drive the mutex without live GitHub.

- **`mergeSpecBranchIntoGoalBranch(specBranch, goalSlug) → GoalBranchMergeResult`** (spec-goal-branch-pm-flow M4 · parallel-build-serialized-merge-and-deadlock-autobreak Phase 3) — merge a spec's `claude/build-{slug}` branch INTO its goal branch `goal/{goalSlug}` via the GitHub REST API (POST `/merges` — a real merge commit, not a squash, so the goal branch carries each spec's history for M5's atomic goal→main promotion). **Seeds the goal branch from `origin/main`** (POST `/git/refs`) if it doesn't exist yet — the FIRST spec of a goal creates it (`created:true`). **Phase 3: Serialized rebase-merge.** Wraps the entire body in `withGoalBranchMergeLock(goalBranch, body)` so concurrent calls (standing pass + webhook) queue. Inside, BEFORE the merge (POST `/merges`), it calls `compareGoalBranchToSpecBranch` and invokes `decideGoalBranchRebaseMerge`. On `rebase-then-merge`, it merges `goal/{goalSlug}` → `spec/{slug}` first to bring the spec up to date, then re-compares and loops (up to 2 rebase attempts). On `escalate` (conflict or exhausted attempts), returns `{ merged:false, escalated:true, reason }` WITHOUT force-merging — the caller records this and leaves it for human resolution. API-only (no local checkout) so it runs from the box worker AND the webhook. Returns `{ merged, conflict, escalated?, created, mergeSha, reason? }`: a `201` = merged (`mergeSha` = merge commit); a `204` (nothing to merge — head already in base) = idempotent success with the goal-branch tip as `mergeSha`; **`escalated:true`** (rebase conflict or too many attempts — surfaced, NOT merged, caller records `director_activity` row); a `409` = `conflict:true` (direct merge conflict, rare now that rebase runs first); `null` compare or rebase result ⇒ fail closed (no merge without positive signal). The caller ([[agent-jobs]] `promoteEligibleSpecsToGoalBranch`) stamps `specs.goal_branch_sha = mergeSha` on success, records a `director_activity` row on escalation, and pushes to `conflicts[]` or `escalated[]` respectively.
- **`mergeGoalBranchIntoMain(goalSlug) → GoalToMainMergeResult`** (spec-goal-branch-pm-flow M5 — the ATOMIC promotion) — merge a goal's `goal/{goalSlug}` branch INTO `main` in ONE merge via the GitHub REST API (POST `/merges` — a real merge commit, so `main` carries the goal's full per-spec history). Mirrors `mergeSpecBranchIntoGoalBranch`'s API-only style (no checkout) so it runs from the box worker AND the webhook. Returns `{ merged, conflict, missingBranch, mergeSha, reason? }`: a `201` = merged (`mergeSha` = the main merge commit) + deletes the spent goal branch; a `204` (nothing to merge — goal branch already in main) = idempotent success with the `main` tip as `mergeSha`; a **`409` = `conflict:true`** (the integrated goal branch conflicts with `main` → promotion HELD, NOT stamped shipped, left for the owner); a missing goal branch = `missingBranch:true` (skip, not a failure). **The atomic merge IS the GREEN combination check** (option b): the caller ([[agent-jobs]] `promoteCompleteGoalsToMain`) only invokes this once every member spec is individually `isSpecPromoteEligible` AND on the goal branch, and because each dependent built OFF the goal branch the integrated whole was already compiled together — this clean atomic land confirms no `main` drift. The caller then runs `applyGoalPromotionEffects(goalSlug, mergeSha)` to flip every member phase shipped.
- **`listOpenClaudeBuildBranches() → Set<string> | null`** ([[../specs/build-accumulation-stamp-gap-and-rollback-guard]] Phase 2) — read-once helper the [[platform-director]] `backstopStuckAccumulation` uses to decide "is a build branch NOT already surfaced as an open PR?" (the P2 wedge signature). Lists open PRs (`GET /pulls?state=open&per_page=100`) and returns the set of `claude/build-*` head refs. Returns **`null`** on ANY failure (no token, list failed, non-array payload) so the caller can fail CLOSED (never treat a GitHub blip as "no open PRs" — that would over-stamp).
- **`readBranchPhaseTrailers(branch) → { headSha, positions } | null`** ([[../specs/build-accumulation-stamp-gap-and-rollback-guard]] Phase 2) — read-once helper the P2 backstop uses to prove the branch's committed content: the branch head SHA (from `body.head_commit.sha`, fallback the last compare commit) + the SET of `Phase: N` trailer positions on `compare(main...branch)` commits. Same line-anchored `Phase:\s*N` regex as the P1 finalize path (`scripts/builder-worker.ts`), so a stray "Phase 2" mention in a commit body never counts — only a bona-fide trailer. Returns **`null`** on ANY failure (no token, branch missing, compare API blip, bad payload) so the caller fails CLOSED (no stamp without a positive branch read).
- **`getPr(prNumber) → { ok:true, merged, state, closedAt, mergeableState, baseRef } | { ok:false }`** ([[../specs/pr-resolve-park-clears-on-pr-merged]] + serialize-goal-member-spec-builds Phase 2) — one-shot PR-state read for the [[approval-inbox]] stale-park reconciler AND the goal-member PR redrive detector. **Not a poll** (unlike `fetchMergeable`); the caller only asks "is this PR still open, and if so, is it dirty?" Fields: `merged` / `state` / `closedAt` (same as before); **`mergeableState`** (from GitHub's `mergeable_state`, one of `unknown|null|clean|dirty|behind|unstable|blocked|draft`) — the canonical signal a PR is DIRTY (behind its base) and needs rebase/rebuild; **`baseRef`** (from `pr.base.ref`, the PR's target branch like `main` or `goal/my-goal`) — used by `runPrResolveJob` to merge into `origin/{baseRef}` dynamically instead of hardcoded main. Returns `{ok:false}` on ANY failure (no token, non-2xx, malformed body, network throw) so the caller fails **CLOSED** — a null read never clears a live park card or re-drives a PR. Consumed by `reconcileStaleParkCards` (approval-inbox.ts) via the pure decision helper `prResolveParkOutcome(pr)` AND the [[agent-jobs]] `reconcileDirtyGoalMemberPrs` via `decideGoalMemberPrRedrive(pr)`.
- **`isAutoMergeEnabled(admin) → boolean`** — the Gate A kill-switch: reads `workspaces.auto_merge_enabled` for the build-console workspace via `select("*")`. **Default ENABLED** — only an explicit `=== false` pauses it; a missing column (pre-migration) or a read error ⇒ enabled.
- **`isInngestSyncActive(admin) → boolean`** — true when a [[../tables/sync_jobs]] row is `pending`/`running` AND created within the last 2 h (recency-guarded so a stale 'running' row can't block forever). Gate A defers while true (a deploy would reap the running sync — the standing rule).
- Types `DirtyPrResult`, `AutoMergeResult`, `GoalBranchMergeResult`, `GoalToMainMergeResult`.

## Mergeable / mergeable_state is lazy

GitHub computes `mergeable` + `mergeable_state` **asynchronously** — `null`/`"unknown"` on the `GET /pulls` list endpoint and right after a push to `main` or while checks run, then settles a moment later. Both gates poll the single PR up to 4× (~1.2s apart) until it settles; unsettled after the budget = skip (a later event re-checks).

- Dirty gate: `fetchMergeable` settles on `mergeable` — only `mergeable === false` (CONFLICTING) enqueues.
- Auto-merge gate: `fetchReadyPr` settles on **both** `mergeable` AND `mergeable_state` (a PR can be `mergeable===true` while its state is still `"unknown"` mid-check). `mergeable_state === "clean"` is the canonical **all-green** signal: mergeable AND every commit status / check passing AND not behind. `"unstable"` (a non-required check failing/pending), `"blocked"` (a required check failing/pending), `"dirty"` (conflicts → the resolver), `"behind"`, `"draft"` are all NOT ready → left alone. The squash-merge is pinned to the evaluated head `sha` (TOCTOU guard — GitHub 409s if the head moved).

## Workspace

A repo-level webhook isn't workspace-scoped, so `resolveBuildWorkspaceId(admin)` attaches the job to the **build-console workspace** — the most-recent `agent_jobs.workspace_id` (i.e. the workspace that actually runs builds), falling back to the first `workspaces` row. Mirrors the system-level enqueue pattern in [[../inngest/spec-test-cron]].

## Callers

- [[../integrations/github-webhook]] (`POST /api/webhooks/github`) — calls `verifyGithubWebhook`, then `detectAndEnqueueDirtyPrs` on `push` to `main` / `pull_request` opened·synchronize·reopened·ready_for_review, AND `autoMergeReadyPrs` on those PLUS `check_suite`/`check_run` completed / `status` success (the events that flip a PR to READY). On the SAME merge-relevant events it also runs **Gate B — [[agent-jobs]] `promoteEligibleSpecsToGoalBranch`** (spec-goal-branch-pm-flow M4), the goal-branch integration: merge every goal-bound, promote-eligible spec branch onto its `goal/{goal-slug}` branch (does NOT touch main); THEN **Gate C — [[agent-jobs]] `promoteCompleteGoalsToMain`** (spec-goal-branch-pm-flow M5), the ATOMIC goal→main promotion: for every COMPLETE + GREEN goal branch, merge `goal/{slug}` → main in ONE merge and stamp every member phase shipped (the only shipped-writer), then trigger the fold pipeline. Parent goals are SKIPPED. Each gate is independent. The box worker's platform-director standing pass runs the same B+C promotes, so they fire in both contexts.

## Gotchas

- **`claude/*` only.** Both gates filter `head.ref` to `claude/*` before doing anything — a human PR or `main` is never touched (guardrail).
- **Detection ≠ resolution (dirty gate).** This library only *enqueues* the resolve. The merge, the additive conflict resolution, the **tsc gate**, the push, and the rebuild-on-main / surface-to-owner decision all live in `scripts/builder-worker.ts` `runPrResolveJob` (the `pr-resolve` lane). See [[../recipes/build-box-setup]] + [[../tables/agent_jobs]].
- **Auto-merge IS the action (Gate A).** Unlike the dirty gate, `autoMergeReadyPrs` performs the squash-merge itself in the webhook — no box worker, no `tsc` gate (the build already tsc-passed before the PR opened; green CI is the gate). The post-merge [[../specs/spec-test-on-ship|spec-test-on-ship]] is the safety net for a bad-but-green build.
- **Serialized.** Gate A merges at most ONE PR per pass; the resulting push-to-main webhook re-enters the gate to merge the next — so N ready PRs never fan out to N simultaneous Vercel deploys.
- **Idempotent + deduped (dirty gate).** One active pr-resolve job per PR; a PR it can't fix lands `needs_attention` (surfaced) or is rebuilt, never retried in a loop.
- **Already-merged duplicate ≠ resolvable ([[../specs/dirty-pr-resolver-duplicate-detection]]).** A conflicting PR whose work merged via a *sibling* build is closed (not resolved) at BOTH the webhook (`detectAndEnqueueDirtyPrs`) and the box worker (`runPrResolveJob` re-checks before doing any work) — defense in depth. This is the fix for the loop where one such PR spawned 9 pr-resolve jobs that could never succeed. The phase-scope match (same `instructions`) keeps a multi-phase chain from self-closing.
- **Pax (pr-resolve) agent mandate ([[../specs/agent-mandate-hardening-pr-resolve]]):** the agent's resolve strategy is **durable** — baked into its prompt in `scripts/builder-worker.ts` `runPrResolveClaude` and enforced pre-flight in `findSupersededExportedSymbolsOnMain` (non-negotiable guard rails, not coaching). Four pillars:
  1. **SUPERSEDE CHECK:** origin/main has ALREADY exported a symbol this branch is adding (the "fix-of-fix" pattern pr-1004 hit)? → close the PR as superseded (with a comment citing the duplicates). No merge can succeed when the result is duplicate exports. Pre-flight reads the branch diff **once**, matches additions to main's exports (product code only; tests don't count), and closes immediately if duplicates exist.
  2. **ANCESTRY HARD GATE:** never conclude "resolved" while `git merge-base --is-ancestor origin/main HEAD` returns non-zero. A non-ancestor HEAD is a *failed* merge (rebase broken, wrong foundation), not a resolved one. If main can't become an ancestor, escalate — don't force a broken commit.
  3. **NO LEFTOVER CONFLICT MARKERS:** tsc must pass after merge. A "Duplicate identifier" / "Duplicate function implementation" error is SUPERSEDED (abort merge, escalate). Never retry blindly past 2 tsc attempts — diagnose the root (parallel rewrite? supersede? stale rebase?) and escalate with one-line reasoning.
  4. **ADDITIVE UNION PREFERRED:** the vast majority of conflicts here are two builds appending to the same list / enum / doc section. KEEP BOTH. Deleting the other build's code to "win" the merge is the recurring slip that dragged the pr-resolve rolling grade down — never do it.
