# github-pr-resolve.ts

`src/lib/github-pr-resolve.ts` — the [[../integrations/github-webhook|GitHub webhook]]'s two **mirror gates** over open `claude/*` PRs. North star (supervisable autonomy): both are bounded proxies the owner supervises (kill-switch + every action surfaced), neither force-pushes through a guardrail.

1. **Dirty-PR Resolver detection** ([[../specs/dirty-pr-resolver-agent]]) — the CONFLICTING half: detect dirty `claude/*` PRs and enqueue ONE resolver job each; the box agent (`runPrResolveJob`) owns the actual merge + the resolve-vs-rebuild-vs-human decision.
2. **Auto-merge gate** ([[../specs/auto-ship-pipeline]] Phase 1 / Gate A) — the READY half: squash-merge + delete branch ONE ready (mergeable + all-checks-green) `claude/*` PR, directly via the GitHub REST API (no box worker — the merge is one API call). Automates the owner's rubber-stamp "merge" click; serialized, sync-aware, owner kill-switch.

Touches **`claude/*` build branches only**.

## Exports

- **`verifyGithubWebhook(rawBody, signatureHeader, secret) → boolean`** — verify the `X-Hub-Signature-256` header GitHub sends (`sha256=<hex>` = HMAC-SHA256(secret, raw body)). Constant-time (`timingSafeEqual`); rejects a missing/malformed header, an unconfigured secret, or a length/digest mismatch. Mirrors [[meta]] `verifyMetaWebhookSignature` — the **raw bytes as received** MUST be used (JSON re-encoding breaks the digest). Secret = `GITHUB_WEBHOOK_SECRET`.
- **`detectAndEnqueueDirtyPrs(admin?) → DirtyPrResult`** — the webhook's whole job: list open PRs, keep `claude/*` ones, check each `mergeable`. For a newly-`CONFLICTING` one it FIRST checks `findAlreadyMergedDuplicate` ([[../specs/dirty-pr-resolver-duplicate-detection]]): if the PR's work already merged via a sibling build → **close it + delete the branch with a comment** (`closeDuplicatePr`) and enqueue **nothing**; otherwise enqueue a deduped `pr-resolve` job. Best-effort + bounded — a transient GitHub error on one PR doesn't block the rest. Returns `{ checked, conflicting, enqueued, closedDuplicate, prs[] }`.
- **`enqueuePrResolveJob(admin, { workspaceId, prNumber, branch, reason? }) → { enqueued, reason? }`** — insert a `kind='pr-resolve'` [[../tables/agent_jobs]] row (`spec_slug = pr-{number}`, `spec_branch = branch`, `pr_number`, `instructions = {pr_number, branch, reason}`). **Idempotent:** no-op if an active pr-resolve job already exists for this PR (dedupe key = `pr-{number}`), so a burst of `push` + `synchronize` events for the same PR enqueues once. **Retry-capped ([[../specs/dirty-pr-resolver-duplicate-detection]]):** counts EVERY pr-resolve job ever enqueued for the PR (any status); at/over `MAX_PR_RESOLVE_ATTEMPTS` (3) it stops enqueueing and surfaces the PR to the owner **once** (a single `needs_attention` pr-resolve sentinel row + a `notifyOpsAlert`) so no PR loops the resolver forever.
- **`findAlreadyMergedDuplicate(admin, branch) → { specSlug, mergedBranch, mergedPr } | null`** ([[../specs/dirty-pr-resolver-duplicate-detection]] Phase 1) — branch-keyed duplicate probe: map the PR branch → its originating `kind='build'` job (spec_slug + workspace + instructions) → `findMergedSiblingBuild` ([[agent-jobs]]) for a SIBLING build of the same spec + same phase scope that already **merged**. Such a PR is unresolvable by definition (its diff is already on `main`). Never guesses the spec from the branch name — reads the build job that created it; no build-job mapping ⇒ `null` (resolve normally).
- **`closeDuplicatePr(prNumber, branch, comment) → boolean`** — close a duplicate `claude/*` PR + delete its branch with an explanatory comment (best-effort comment/branch-delete). Used by both gates instead of resolving when the work already merged.
- **`autoMergeReadyPrs(admin?) → AutoMergeResult`** (Gate A) — list open `claude/*` PRs, find the READY ones (open · non-draft · `mergeable === true` · `mergeable_state === "clean"`), squash-merge **ONE** (serialized) + delete its branch via the GitHub REST API. No-op when the kill-switch is off or a sync is active. Beats the Control Tower (`loop_id = AUTO_MERGE_GATE_LOOP_ID`, `kind:'reactive'`) once per pass in a `finally`. Returns `{ enabled, syncActive, checked, ready, buildGateBlocked, accumulationBlocked, testsGateBlocked, merged (0|1), mergedPr?, prs[] }`. **Accumulation gate (spec-goal-branch-pm-flow M2):** between the build gate and the tests gate, a MULTI-PHASE spec branch is refused promotion until EVERY phase is built on the branch (`isSpecAccumulationComplete` — each phase carries a `build_sha` or is terminal; NO phase still `planned`). Under M1's branch-accumulation model the phases build one-by-one onto ONE persistent `claude/build-{slug}` PR (no per-phase main merge), so without this gate the existing auto-merge would promote a PARTIAL spec the moment its first phase went green. A one-shot/single-phase spec (0–1 phases) passes trivially; fails OPEN on a PM-read error (the tests/build gates still apply). This is the M5 SEAM — M5 ([[../specs/spec-goal-branch-pm-flow]]) owns the actual atomic promotion + the `build_sha`→`shipped` flip; M2 only keeps auto-merge from jumping a half-accumulated branch. (`isSpecAccumulationComplete` was promoted to the [[specs-table]] SDK in M3 so this gate, the M3 pre-merge-test trigger [[agent-jobs]] `maybeEnqueuePreMergeSpecTestOnAccumulation`, and [[agent-jobs]] `isSpecPromoteEligible` all read the IDENTICAL predicate; this gate imports it.) **Post-merge hook (chain-and-cardstate-under-automerge Phase 1):** right after a successful squash-merge it calls [[agent-jobs]] `handleAutoMergedBuildBranch(branch, mergeSha)` — the merge happened server-side, so this advances the build's post-merge state (flip the job `merged` → roll up the card status → queue the next ⏳ phase of a "Build all" chain) **without** waiting for a board-render reconcile. `squashMergeAndDelete` now also returns the squash commit's `mergeSha` (the card's `last_merge_sha`). Idempotent + best-effort. **Deploy-watch hook (deploy-health-rollback-guardian Phase 1):** the same squash-merge triggers a Vercel deploy, so it also calls [[deploy-guardian]] `openDeployWatch({ branch, prNumber, mergeSha })` — Reva snapshots the pre-deploy error/loop baseline + opens a `pending` [[../tables/deploy_watches]] over the canary window; [[../inngest/deploy-guardian-cron]] stamps the verdict once the window elapses. Best-effort + never throws.
- **`mergeSpecBranchIntoGoalBranch(specBranch, goalSlug) → GoalBranchMergeResult`** (spec-goal-branch-pm-flow M4) — merge a spec's `claude/build-{slug}` branch INTO its goal branch `goal/{goalSlug}` via the GitHub REST API (POST `/merges` — a real merge commit, not a squash, so the goal branch carries each spec's history for M5's atomic goal→main promotion). **Seeds the goal branch from `origin/main`** (POST `/git/refs`) if it doesn't exist yet — the FIRST spec of a goal creates it (`created:true`). API-only (no local checkout) so it runs from the box worker AND the webhook. Returns `{ merged, conflict, created, mergeSha, reason? }`: a `201` = merged (`mergeSha` = merge commit); a `204` (nothing to merge — head already in base) = idempotent success with the goal-branch tip as `mergeSha`; a **`409` = `conflict:true`** (surfaced, NOT merged — the caller escalates, never silently drops). The caller ([[agent-jobs]] `promoteEligibleSpecsToGoalBranch`) stamps `specs.goal_branch_sha = mergeSha` on success.
- **`isAutoMergeEnabled(admin) → boolean`** — the Gate A kill-switch: reads `workspaces.auto_merge_enabled` for the build-console workspace via `select("*")`. **Default ENABLED** — only an explicit `=== false` pauses it; a missing column (pre-migration) or a read error ⇒ enabled.
- **`isInngestSyncActive(admin) → boolean`** — true when a [[../tables/sync_jobs]] row is `pending`/`running` AND created within the last 2 h (recency-guarded so a stale 'running' row can't block forever). Gate A defers while true (a deploy would reap the running sync — the standing rule).
- Types `DirtyPrResult`, `AutoMergeResult`.

## Mergeable / mergeable_state is lazy

GitHub computes `mergeable` + `mergeable_state` **asynchronously** — `null`/`"unknown"` on the `GET /pulls` list endpoint and right after a push to `main` or while checks run, then settles a moment later. Both gates poll the single PR up to 4× (~1.2s apart) until it settles; unsettled after the budget = skip (a later event re-checks).

- Dirty gate: `fetchMergeable` settles on `mergeable` — only `mergeable === false` (CONFLICTING) enqueues.
- Auto-merge gate: `fetchReadyPr` settles on **both** `mergeable` AND `mergeable_state` (a PR can be `mergeable===true` while its state is still `"unknown"` mid-check). `mergeable_state === "clean"` is the canonical **all-green** signal: mergeable AND every commit status / check passing AND not behind. `"unstable"` (a non-required check failing/pending), `"blocked"` (a required check failing/pending), `"dirty"` (conflicts → the resolver), `"behind"`, `"draft"` are all NOT ready → left alone. The squash-merge is pinned to the evaluated head `sha` (TOCTOU guard — GitHub 409s if the head moved).

## Workspace

A repo-level webhook isn't workspace-scoped, so `resolveBuildWorkspaceId(admin)` attaches the job to the **build-console workspace** — the most-recent `agent_jobs.workspace_id` (i.e. the workspace that actually runs builds), falling back to the first `workspaces` row. Mirrors the system-level enqueue pattern in [[../inngest/spec-test-cron]].

## Callers

- [[../integrations/github-webhook]] (`POST /api/webhooks/github`) — calls `verifyGithubWebhook`, then `detectAndEnqueueDirtyPrs` on `push` to `main` / `pull_request` opened·synchronize·reopened·ready_for_review, AND `autoMergeReadyPrs` on those PLUS `check_suite`/`check_run` completed / `status` success (the events that flip a PR to READY). On the SAME merge-relevant events it also runs **Gate B — [[agent-jobs]] `promoteEligibleSpecsToGoalBranch`** (spec-goal-branch-pm-flow M4), the goal-branch integration: merge every goal-bound, promote-eligible spec branch onto its `goal/{goal-slug}` branch (does NOT touch main). Independent of Gate A. The box worker's platform-director standing pass runs the same promote, so it fires in both contexts.

## Gotchas

- **`claude/*` only.** Both gates filter `head.ref` to `claude/*` before doing anything — a human PR or `main` is never touched (guardrail).
- **Detection ≠ resolution (dirty gate).** This library only *enqueues* the resolve. The merge, the additive conflict resolution, the **tsc gate**, the push, and the rebuild-on-main / surface-to-owner decision all live in `scripts/builder-worker.ts` `runPrResolveJob` (the `pr-resolve` lane). See [[../recipes/build-box-setup]] + [[../tables/agent_jobs]].
- **Auto-merge IS the action (Gate A).** Unlike the dirty gate, `autoMergeReadyPrs` performs the squash-merge itself in the webhook — no box worker, no `tsc` gate (the build already tsc-passed before the PR opened; green CI is the gate). The post-merge [[../specs/spec-test-on-ship|spec-test-on-ship]] is the safety net for a bad-but-green build.
- **Serialized.** Gate A merges at most ONE PR per pass; the resulting push-to-main webhook re-enters the gate to merge the next — so N ready PRs never fan out to N simultaneous Vercel deploys.
- **Idempotent + deduped (dirty gate).** One active pr-resolve job per PR; a PR it can't fix lands `needs_attention` (surfaced) or is rebuilt, never retried in a loop.
- **Already-merged duplicate ≠ resolvable ([[../specs/dirty-pr-resolver-duplicate-detection]]).** A conflicting PR whose work merged via a *sibling* build is closed (not resolved) at BOTH the webhook (`detectAndEnqueueDirtyPrs`) and the box worker (`runPrResolveJob` re-checks before doing any work) — defense in depth. This is the fix for the loop where one such PR spawned 9 pr-resolve jobs that could never succeed. The phase-scope match (same `instructions`) keeps a multi-phase chain from self-closing.
