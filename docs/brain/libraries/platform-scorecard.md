# libraries/platform-scorecard

The **department-level KPI aggregation engine** behind the [[../goals/platform-department-scorecard|Platform Department Scorecard]] ([[../archive.d/platform-scorecard-engine]] Phases 1‚Äì2; milestone (a) Daily pulse). Where [[director-xp]] / [[director-recap]] compute **per-director** gamification/EOD counts and [[meta__scorecards]] is the **ad** domain, this engine rolls the platform's own truth up to a **department KPI that trends over time**: each KPI computed over a **trailing window** with a **prior equal-length window delta** (mirroring the [[meta__scorecards]] window model), upserted into [[../tables/platform_scorecard_snapshots]].

**File:** `src/lib/agents/platform-scorecard.ts` (server-only ‚Äî `createAdminClient` + [[brain-roadmap]] fs reads, like [[director-xp]] / [[director-recap]]).

**North-star invariant** ([[../operational-rules]] ¬ß supervisable autonomy): every KPI is **derived + read-only** ‚Äî computed from existing tables, persisted for trend, **never** written back as a target the directors/workers optimize. This engine is the **only writer** of [[../tables/platform_scorecard_snapshots]]; downstream readers read the snapshot table, never the raw tables.

## Exports

- **`type Cadence = 'daily' | 'weekly' | 'monthly'`** ¬∑ **`type MetricUnit = 'count' | 'ratio' | 'hours' | 'pct' | 'grade'`** (`'grade'` = the 1‚Äì10 scale shared by the worker + director rollups ‚Äî added in [[../specs/devops-kpi-review-sdk-and-data-fix]] Phase 2 after the pre-fix `'ratio'` stamp rendered an 8.5/10 grade as "850%"; [[platform-scorecard-display]] `formatMetricValue` renders it as `X.X / 10`)
- **`interface ScorecardSnapshotRow`** ‚Äî one persisted row (`workspace_id, metric_key, cadence, snapshot_date, window_days, value, prior_value, delta_pct, unit, detail, updated_at`).
- **`interface ComputeOptions { cadence; snapshotDate?; windowDays? }`** ‚Äî `snapshotDate` defaults to today (UTC) EXCEPT for the daily cadence, where it defaults to yesterday UTC so the daily window represents the **previous complete UTC day** (‚úÖ [[../specs/devops-kpi-daily-snapshot-date-lag-fix]] Phase 1); `windowDays` to the cadence default (daily=1).
- **`computePlatformScorecard(workspaceId, opts): Promise<ScorecardSnapshotRow[]>`** ‚Äî for each KPI in the cadence's registry, compute the trailing-window `value`, the prior equal-length `prior_value`, and `delta_pct`, then **upsert** every row on `(workspace_id, metric_key, cadence, snapshot_date)`. Idempotent (a same-day re-run upserts in place). A quiet workspace writes zeros, never errors; a single metric's read failing writes a zero row + logs (the rest still land). Thin wrapper around `computeScorecardValuesOnly` + the upsert.
- **`computeScorecardValuesOnly(workspaceId, opts): Promise<ScorecardSnapshotRow[]>`** ‚Äî the same compute pass WITHOUT the upsert, so [[kpi-review]] can re-derive ground truth without violating the "one writer" rule (the SDK is read-only). Same window math + same rounding ‚Üí byte-equivalent rows to what `computePlatformScorecard` would persist.
- **`getRegisteredMetrics(cadence): ReadonlyArray<{ key; unit; currentState?; liveSpecSetDependent? }>`** ‚Äî the `(metric_key, unit, currentState?, liveSpecSetDependent?)` tuples for every metric in the cadence's registry; lets [[kpi-review]] enumerate the advertised KPI set without re-importing the private `MetricDef` shape. `currentState?: true` flags a point-read metric (today: `lane_utilization` ¬∑ `loop_health` ¬∑ `needs_attention` ¬∑ `worker_grade_rollup` ¬∑ `director_call_grade`) ‚Äî [[kpi-review]] skips it because comparing a frozen point-read snapshot against a re-read moments later is moving-target noise, not drift. `liveSpecSetDependent?: true` flags a metric whose ground truth depends on the LIVE brain-roadmap spec set (e.g. `specs_per_week`, `regression_coverage_pct`) ‚Äî [[kpi-review]] skips it because specs fold/archive between snapshot write and audit re-read, so the membership delta surfaces as "drift" that isn't engine drift.
- **`getDefaultWindowDays(cadence): number`** ‚Äî the engine's default trailing-window length per cadence (daily=1 / weekly=7 / monthly=30).
- **`CADENCES: readonly Cadence[]`** ‚Äî `["daily","weekly","monthly"]`; the single source of truth for callers that enumerate the registry across cadences.
- **`isAuditSkippedKpiDriftLoop(loopId): boolean`** ‚Äî given a `loop_alerts.loop_id`, returns whether it's a `kpi_drift:<metric>:<cadence>` alert for an **audit-skipped** metric (a `currentState` point-read OR a `liveSpecSetDependent` meta-metric). Reuses the same registry flags `getRegisteredMetrics` exposes ‚Äî the **single predicate** behind both the [[kpi-review]] audit skip and the [[deploy-guardian]] non-attribution gate (`isExcludedFromDeployRegressionLoop` ‚Üí `verdictFor`), so a weekly-aggregate / live-spec-set KPI drift never auto-reverts a deploy. Non-`kpi_drift`, unparseable, or genuine-windowed-aggregate loop_ids return `false`.

## The daily KPI registry

A declarative registry keyed by `metric_key` keeps the per-cadence metric set extensible ([[../specs/platform-scorecard-weekly]] adds the **weekly** set below; [[../specs/platform-scorecard-monthly]] adds the **monthly** leading curve). The **daily** set:

| `metric_key` | unit | Derivation |
|---|---|---|
| `loop_health` | ratio | Share of [[../libraries/control-tower]] `MONITORED_LOOPS` green = **no open [[../tables/loop_alerts]]** AND (for a `cron`) a [[../tables/loop_heartbeats]] beat within its `livenessWindowMs` (latest beat per loop via the `control_tower_loop_beats` RPC). Worker/agent-kind/reactive/inline loops are idle-green (judged on the open-alert signal). `detail` lists every unhealthy loop. **Current-state metric** (`currentState: true` ‚Äî NOW-state read of `loop_alerts` + latest heartbeat per loop, not a windowed aggregate) ‚Üí prior comes from the prior stored snapshot; [[kpi-review]] SKIPS it (same moving-target-noise class as `lane_utilization`). |
| `error_backlog` | count | Recent [[../tables/error_events]] incidents (by `last_seen_at` in-window), **excluding `outage_correlated`** symptoms. `detail` carries the last-1h/last-24h/by-source recency colouring `buildErrorFeedSnapshot` uses. |
| `error_mttr_hours` | hours | **Derived, NOT from `error_events.status`** (reserved/unmaintained, no `resolved_at`). Median over the window of `(resolution_ts ‚àí first_seen_at)`, where `resolution_ts` = the **concluded** terminal time of the correlated [[../tables/agent_jobs]] repair (`kind='repair'`, `spec_slug = signature` ‚Äî the repair dedupe key; `kind='regression'` is the documented fallback). Errors with no concluded correlated repair are excluded + surfaced in `detail.still_open`. |
| `build_throughput` | count | [[../tables/agent_jobs]] `kind='build'` + `status='merged'` with `updated_at` (the merge flip) in-window ‚Äî the exact rule [[director-recap]] uses for `specsShipped`. |
| `lane_utilization` | ratio | **Build-pool saturation NOW** ([[../specs/director-initiation-throughput]] Phase 4): build/plan [[../tables/agent_jobs]] OCCUPYING a lane (`claimedÔΩúbuildingÔΩúneeds_inputÔΩúneeds_approvalÔΩúqueued_resume` ‚Äî a plain `queued` job is backlog WAITING for a lane, not occupying one) √∑ `BUILD_POOL_CAPACITY` (8, the [[platform-director]] lane count). Capped at 1. The headline saturation KPI ‚Äî the curve should trend toward full. Current-state metric ‚Üí prior from the prior snapshot. |
| `build_enqueue_rate` | count | The pool FEED-rate ([[../specs/director-initiation-throughput]] Phase 4): [[../tables/agent_jobs]] `kind='build'` with `created_at` (the enqueue) in-window ‚Äî paired with `lane_utilization` it shows whether the director keeps the pool topped up. |
| `autonomy_ratio` | ratio | [[../tables/approval_decisions]] `autonomous=true` √∑ all terminal decisions (`decision ‚àà approvedÔΩúdeclined`) in-window (escalated rows excluded from the denominator). Computed by [[director-kpis]] `autonomyRatio` ‚Äî the metric-def is a thin wrapper (Phase 2 SSOT). |
| `escalations` | count | [[../tables/approval_decisions]] `decision='escalated'` + [[../tables/director_activity]] `action_kind='escalated'`, **attributed to the platform function** (`raised_by_function` / `director_function = 'platform'`) in-window. Only a real `functions/*.md` slug counts (a stray slug is ignored ‚Äî [[director-xp]]'s guard). |
| `needs_attention` | count | **Open parked work the director triages** ([[../specs/needs-attention-triage-and-verdict-robustness]] Phase 3): [[../tables/agent_jobs]] `status='needs_attention'` EXCLUDING the kinds another lane owns (`build` ‚Üí the build loop-guard; `repair` ‚Üí the repair-dismissal lane; `platform-director` ‚Üí the director's own jobs). The count is the headline value; `detail` carries the **OLDEST** open item's age (`oldest_hours`) + a `by_kind` breakdown ‚Äî so a rotting parked item is a tracked, trending KPI, not just a transient board line. **Current-state metric** (`currentState: true` ‚Äî NOW-state read of `agent_jobs` in `needs_attention`, not a windowed aggregate) ‚Üí prior comes from the prior stored snapshot; [[kpi-review]] SKIPS it (same moving-target-noise class as `lane_utilization` / `loop_health` ‚Äî a parked item routes/resolves or a new park lands between snapshot write and audit re-read). Same scope `reconcileNeedsAttention` ([[platform-director]] Phase 1) triages + the daily board-watch reports ("N items need attention, oldest Xh"). |
| `regressions` | count | **The day's regression flow** ([[../specs/regression-backlog-reconciliation-scorecard]] Phase 1): platform-attributed [[../tables/director_activity]] rows in-window with `action_kind ‚àà detected_regressionÔΩúauthored_fixÔΩúreconciled_regressionÔΩúescalated`, with `escalated` filtered to a regression context (`metadata.signature` starts `regression:`, `metadata.kind='regression'`, or `metadata.escalation_kind='loop_guard'`). The headline `value` is the sum (`detected + fixed + reconciled + escalated`); `detail` carries each leg so the board-watch line + tile can render the breakdown without a second read. Only the real platform slug counts ([[director-xp]]'s guard). |

**Window model** (mirrors [[meta__scorecards]]): `curr = [snapshotDate ‚àí (windowDays‚àí1), snapshotDate]`, `prev = [snapshotDate ‚àí (2¬∑windowDays‚àí1), snapshotDate ‚àí windowDays]`. `loop_health` is current-state, so its prior is read from the snapshot `windowDays` ago rather than recomputed.

## The weekly KPI registry

The **weekly** lens ([[../specs/platform-scorecard-weekly]]) ‚Äî how much the build org shipped this week and how good it was, over a **trailing 7-day** window with a **prior-week delta**. Reuses the same engine machinery (window/delta/upsert); only the derivations are new:

| `metric_key` | unit | Derivation |
|---|---|---|
| `specs_per_week` | count | [[../tables/agent_jobs]] `kind='build'` + `status='merged'` with `updated_at` (the merge flip) in-window, **`spec_slug` mapped to the `platform` owner** via [[director-kpis]] `shippedSpecsByOwner` ‚Äî the map is built from the FULL [[specs-table]] `listSpecs` set (folded INCLUDED). Fixes the [[../specs/director-kpi-sdk]] Phase 1 bug: previously the map came from [[brain-roadmap]] `getRoadmap()` (live-only), so a merged build whose spec had folded that day dropped off. Only platform-owned merged builds count; `detail.slugs` lists them. |
| `build_success_rate` | ratio | `merged √∑ (merged + failed)` over the window from [[../tables/agent_jobs]] `kind='build'` (success = `status='merged'`, failure = `status ‚àà failedÔΩúneeds_attention`), terminal flip by `updated_at`. Computed by [[director-kpis]] `buildSuccessRate` ‚Äî the metric-def is a thin wrapper (Phase 2 SSOT). `detail` carries the raw `merged`/`failed`/`total` counts; prior is null when the prior week had no terminal builds. |
| `idea_to_merge_hours` | hours | Median over merged builds in-window of `(updated_at ‚àí created_at)` ‚Äî the queued‚Üímerged "idea‚Üímerged-PR" north-star cycle time. `value` = p50; `detail` carries `p50`/`p90` + `merged_count`. |
| `approvals_untouched_pct` | pct | [[../tables/approval_decisions]] `autonomous=true` √∑ all terminal decisions (`decision ‚àà approvedÔΩúdeclined`) in-window, **as a percentage** ‚Äî "% of platform approvals you never have to touch". A `decided_by ‚àà ceoÔΩúhuman` decision is a TOUCHED approval (surfaced in `detail`); escalated rows are excluded from the denominator. |
| `worker_grade_rollup` | grade | Per `agent_kind` average from [[../tables/agent_action_grades]] via [[agent-grader]] `computeAgentRollup` (the **same** last-`ROLLUP_WINDOW` rollup the coaching loop reads ‚Äî not a second source of truth). `value` = the fleet mean; `detail.by_worker` carries each graded kind's `average`/`prior`/`drop`/`count`, so a slipping worker is visible. Prior = the mean of the per-worker prior-window averages. **Current-state metric** (`currentState: true` ‚Äî a rolling last-`ROLLUP_WINDOW` point read of `agent_action_grades`, not a windowed aggregate bound to the snapshot date) ‚Üí prior comes from the prior stored snapshot; [[kpi-review]] SKIPS it (same moving-target-noise class as `lane_utilization` / `loop_health` / `needs_attention` ‚Äî a grading write between the snapshot and the audit re-read rotates the top-`ROLLUP_WINDOW` window and shifts the mean). Repair Agent verdict on signature `loop:kpi_drift:worker_grade_rollup:weekly`. |
| `regressions_caught` | count | [[../tables/agent_jobs]] `kind='regression'` **concluded** in-window (terminal status, `updated_at`) **+** [[../tables/director_activity]] rows the [[../specs/regression-agent|Regression Agent]] emits (`action_kind ‚àà detected_regressionÔΩúauthored_fix`). `detail` splits `detected` / `dismissed` / `fix_authored` / `regression_jobs_concluded` (a `dismissed_regression` is tracked but not counted toward the headline). |
| `regression_coverage_pct` | pct | **The coverage half of regression health** ([[../specs/regression-backlog-reconciliation-scorecard]] Phase 1): `(distinct shipped spec_slugs with a [[../tables/spec_test_runs]] row in-window) √∑ (live shipped specs from [[brain-roadmap]] `getRoadmap()`) √ó 100`. `detail = { verified, shipped, missing[] }` ‚Äî `missing` lists the live shipped slugs with no spec-test run this week (the queue [[../specs/regression-backlog-reconciliation]] Phase 1's sweep should pick up next pass). Archived/folded specs have left the live set and are intentionally excluded. |

## The monthly KPI registry

The **monthly** leading curve ([[../specs/platform-scorecard-monthly]]) ‚Äî the slow-moving indicators that prove autonomy is compounding, over a **trailing 30-day** window with a **prior-month delta**. Reuses the same engine machinery (window/delta/upsert); only the derivations are new:

| `metric_key` | unit | Derivation |
|---|---|---|
| `human_touch_per_build` | ratio | **The goal's headline.** `(approval_decisions where decided_by ‚àà ceoÔΩúhuman in-window) √∑ (agent_jobs kind='build' status='merged' in-window)`. Lower is better; the prior-month `delta_pct` is the "declining MoM" signal. Computed by [[director-kpis]] `humanTouchPerBuild` ‚Äî the metric-def is a thin wrapper (Phase 2 SSOT). `detail` carries `touched` (numerator) + `builds` (denominator). |
| `goals_escorted_unbabysat` | count | Goals whose milestones advanced WITHOUT CEO touch: [[../tables/director_activity]] `action_kind='escorted_goal'` (`director_function='platform'`) in-window ‚Üí distinct `goal_slug`, intersected with [[brain-roadmap]] `getGoals()` shipped milestones; counted only when NO non-autonomous [[../tables/approval_decisions]] (`decided_by ‚àà ceoÔΩúhuman`) in-window touched a spec belonging to that goal's milestones (the touch-check joins `agent_job_id` ‚Üí `agent_jobs.spec_slug`). Computed by [[director-kpis]] `goalsEscortedUnbabysat` ‚Äî the metric-def is a thin wrapper (Phase 2 SSOT). `detail.goals` lists the escorted-without-touch goals + their milestones. |
| `time_to_approve_hours` | hours | Median over terminal [[../tables/approval_decisions]] in-window of `(decision.created_at ‚àí raisedAt)`, where `raisedAt` is approximated by the raising [[../tables/agent_jobs]] row's `created_at` (the floor ‚Äî the job couldn't request approval before it existed; `pending_actions` carry no timestamp and `agent_jobs.updated_at` is post-approval by the time we read it). `value` = p50; `detail` carries `p50`/`p90` + `decided_count`. |
| `deploy_reliability` | ratio | The [[../specs/deploy-health-rollback-guardian|Deploy-Health & Auto-Rollback guardian]] half of the reliability KPI (CI-green is the weekly `build_success_rate`): [[../tables/director_activity]] `action_kind='deploy_healthy'` √∑ (`deploy_healthy` + `deploy_rolled_back`) in-window. When the guardian has written NO verdicts in the window we surface `detail.no_data=true` (value=0) so the UI shows "no data yet" rather than a fabricated 100%. |
| `director_call_grade` | grade | The CEO's grade of the Platform director's calls in-window: average [[../tables/director_decision_grades]] `grade` (1‚Äì10) split by `dimension ‚àà auto-approvalÔΩúgoal-escort` ‚Äî the same shape [[director-leash-recommendations]] `computeDirectorGradeReport` reads. `value` = the blended mean across both dimensions; `detail.by_dimension` carries each dimension's mean + count. Populated by [[../specs/director-loop-grading]] (‚úÖ). **Current-state metric** (`currentState: true` ‚Äî a point read of the current `director_decision_grades` rows in the cadence window, not a windowed aggregate bound to the snapshot date) ‚Üí prior comes from the prior stored snapshot; [[kpi-review]] SKIPS it (same moving-target-noise class as `worker_grade_rollup` ‚Äî a new director grade between the snapshot and the audit re-read shifts the blended mean). |

## Caller

The daily snapshot beat on [[../inngest/platform-director-cron]] (`snapshot-platform-scorecard` step) ‚Äî once per UTC day per build-console workspace, `computePlatformScorecard(ws, { cadence:'daily', windowDays:1 })` with the daily cadence's default snapshotDate (yesterday UTC, per [[../specs/devops-kpi-daily-snapshot-date-lag-fix]] Phase 1) so the persisted `snapshot_date` represents the **previous complete UTC day**, not today-so-far. The **weekly** beat is the `snapshot-platform-scorecard-weekly` step on the same cron ‚Äî **once per ISO week** per workspace (guarded on any weekly row already taken on/after the ISO-week Monday), `computePlatformScorecard(ws, { cadence:'weekly', windowDays:7 })` with no lag (snapshotDate defaults to today). The **monthly** beat is the `snapshot-platform-scorecard-monthly` step on the same cron ‚Äî **once per calendar month** per workspace (guarded on any monthly row already taken on/after the 1st of the UTC month), `computePlatformScorecard(ws, { cadence:'monthly', windowDays:30 })` with no lag (snapshotDate defaults to today). All three run in the deployed runtime (DB access), best-effort + idempotent. [[../specs/platform-scorecard-surface]] reads the snapshot table for the scorecard page.

## Recovery ‚Äî stranded pre-lag-fix daily rows

Daily rows written **before** [[../specs/devops-kpi-daily-snapshot-date-lag-fix]] landed (commit `232f6ad8`, 2026-06-28 12:11 UTC) were stamped `snapshot_date = today UTC` over an in-flight `[today, today]` window ‚Äî they captured a partial-day reading. The [[kpi-review]] audit pass re-derives those closed days over the same `[snapshot_date, snapshot_date]` window using the now-complete data and logs the resulting drift, opening `kpi_drift:<metric>:daily` [[../tables/loop_alerts]] (e.g. `loop:kpi_drift:autonomy_ratio:daily`) once a metric goes ‚â•2 consecutive snapshots over-tolerance. **`scripts/resnapshot-pre-lag-fix-daily-kpis.ts`** ([[../specs/devops-kpi-snapshot-guard-stale-row-fix]] Phase 1) sweeps the table for closed-day daily rows where `snapshot_date >= updated_at::date UTC` (the pre-lag-fix shape ‚Äî post-lag-fix lags by one day so `snapshot_date < updated_at::date`) and re-runs `computePlatformScorecard` per `(workspace_id, snapshot_date)` against the now-complete day ‚Äî the `(workspace_id, metric_key, cadence, snapshot_date)` upsert key overwrites every metric row in place. Dry-run by default; `--apply` writes; idempotent (a post-lag-fix row re-run by the script re-upserts byte-equivalent values). The next audit-platform-scorecard pass then finds the rows within tolerance and auto-resolves the `kpi_drift:*:daily` alerts.
