# `src/lib/agents/agent-grader.ts` — the worker-action grader

The DevOps Director's **worker-action grade** — one row per graded **concluded `agent_jobs` row**, 1–10 + reasoning ([[../specs/worker-grading-and-director-management]] Phase 1; the devops-director goal "the org learns + self-manages"). One level **down the org chart** from [[director-grader]]: there the CEO grades the Director's calls; here the Director grades each WORKER's actions, and a slip in a worker's rollup triggers a coaching pass. Persists to [[../tables/agent_action_grades]]; calibrated by approved [[../tables/agent_grader_prompts]] rules; coaches via [[worker-instructions]] `coachAgent`.

The defining invariant (inherited from [[director-grader]]): **grade the work, not outcome luck** — a sound, well-scoped action that hit a rare reversible bump still grades well if the reasoning was right; a careless action that happened to land grades low. The grader is a SUPERVISED TOOL ([[../operational-rules]] § North star): it scores a bounded proxy (action quality); the Director owns the objective and the CEO overrides it.

## The gradeable unit
One **concluded `agent_jobs` row** (terminal status `completed｜merged｜failed｜needs_attention`) — a build merged, an error fixed/dismissed, an index proposed, a spec verified. **`merged` IS terminal** — a build goes `building → completed → merged` (PR landed on `main`; `SUCCESSFUL_BUILD_STATUSES` in [[agent-jobs]]), and the `build` rubric literally scores "PR merged clean", so a `merged` build is the canonical thing to grade. (Omitting `merged` from `TERMINAL_JOB_STATUSES` silently no-op'd every merged build for days — the same STARVED-grading root cause as [[director-grader]]; a post-merge finalization step that leaves the job at `merged` must not starve worker grading.) The job IS the worker's atomic action, so the director grader's polymorphic key collapses to a single FK (`agent_job_id`) + one rubric per worker `kind`.

## Per-worker rubrics
`AGENT_RUBRICS` (exported) maps each `agent_jobs.kind` → `{name, criteria}` (the spec's locked config): `build`→Bo (spec phases · `tsc` · clean merge), `repair`→Rafa (root-cause · fix held), `regression`→Remi, `db_health`→Devi, `spec-test`→Vera, `migration-fix`→Mira, `pr-resolve`→Pax, `fold`→Fenn, `coverage-register`→Cole, `monitor`→Tao, `plan`→Pia, `product-seed`→Piper, `spec-chat`→Sage, `dev-ask`→Dex. `GRADEABLE_KINDS` = its keys — only rubric-backed kinds are graded (the Director's own + non-worker kinds are excluded).

The rubric must **name the worker's actual artifact**, not the transport buffer / scratchpad it happens to write on the way there ([[../specs/spec-chat-db-authoring-clarity]] Phase 2 — the rubric is embedded in both the grader system prompt and the box coach synthesis prompt, so its wording propagates directly into future coaching guidance). E.g. `spec-chat`'s "correct DB spec authoring on finalize" clause explicitly names `public.specs` + `public.spec_phases` via `upsertSpec` as the artifact and frames the `docs/brain/specs/{slug}.md` the box writes as a throwaway scratch buffer the worker discards. Rubric wording that names the transport buffer instead of the DB row lets a coaching learning inherit MD-based-spec framing.

## A director grades only its own charge
[[../specs/director-grades-only-own-charge]] (2026-07-01): the worker-grader is parameterized by a DIRECTOR function (`fn`, default `'platform'`) and grades ONLY the kinds whose owning function in the Control Tower registry equals `fn` — mirroring the same owner-scoping [[approval-inbox]] `ownerFunctionForKind` already enforces. So Ada (`fn='platform'`) grades the platform-owned workers (build/fold/spec-test/repair/pr-resolve/security-review/spec-review/plan/dev-ask/spec-chat/coverage-register/db_health) but NEVER the CS/CMO/Retention/Growth workers (`ticket-improve`/`triage-escalations` = cs, `product-seed` = cmo, `migration-fix` = retention, `storefront-optimizer` = growth). Grading is a director-tier supervisory function; no director → no worker grading. A cross-function worker stays UNGRADED until its own director's sweep goes live — never a CEO reach-in fail-safe (ungraded-until-their-director-is-live is the intended behavior). Same rule the [[../operational-rules]] § North star mandates: a supervisor owns the layer BELOW it, not adjacent departments.

- **`gradeableKindsForFunction(fn)`** (exported): `GRADEABLE_KINDS.filter(k => ownerFunctionForKind(k) === fn)` — the source of truth for "which kinds this director grades". Unmapped kinds (no `ownerFunctionForKind` entry) are NOT owned by anyone — a legitimate NO-OP, never a reach-in.
- **`ungradedConcludedJobs` / `pickAgentGradeBatch` / `gradeConcludedAgentActions` / `agentGradingBatchReady`** all take `fn` (default `'platform'`) and pull the pool via `gradeableKindsForFunction(fn)`.
- **`detectGradeDropCoaching` / `applyBoxCoaching`** also take `fn` and REFUSE to coach when `ownerFunctionForKind(agentKind) !== fn` (defense-in-depth against a stale caller or a legacy queued agent-coach row) — `reason='not_owned_by_director'`.
- **[[../inngest/platform-director-cron]]** iterates `gradeableKindsForFunction(PLATFORM)` for the coaching cascade, threads `fn: PLATFORM` into `agentGradingBatchReady` / `pickAgentGradeBatch` / `detectGradeDropCoaching`, and stamps `fn` on the enqueued `agent-grade` payload so the box lane's post-batch fan-out stays scoped.
- **Box lanes** (`runAgentGradeJob` / `runAgentCoachJob` in [[../../scripts/builder-worker.ts]]) read `instructions.fn` (default `'platform'` for legacy rows) and thread it into the coach + box-coaching calls.
- **Follow-up** (deliberately deferred — CEO ask only): the already-written cross-function `agent_action_grades` rows from before the fix (Ada grading CS/CMO/Retention/Growth kinds) are now-invalid data; a cleanup script could delete them, but leave them unless asked.

## Per-workspace cadence gate (Phase 1)

[[../specs/ada-grading-sampled-adaptive-cadence]] (2026-07-02): a grading pass runs at most **once per `GRADE_CADENCE_MS` (~2h)** per workspace. The `*/5`-min platform-director cron still calls `agentGradingBatchReady` every beat, but the ready check reads `MAX(agent_action_grades.created_at)` for the workspace's owned kinds — if that's within the cadence window, the pass is a hard no-op (`reason:'within_cadence'`), regardless of how many ungraded jobs have piled up (they ride the next window).

Why: the deployed grader was running on every beat with no per-workspace floor, so a large backlog could produce a runaway grading pass — 32 grades in minutes was observed live, burning a Max session bank in a single sweep. Bounding to one pass per ~2h decouples grading spend from the cron beat while keeping the batch cap (`GRADE_BATCH_CAP=12`) as a per-pass ceiling — steady-state grading spend is now bounded to `12 grades × 12 windows/day ≈ 144/day per workspace`, tunable via env.

The cadence is **strict**: even a genuine failure landing mid-window waits for the next window. A pending failure is high-signal but not more valuable than the Max session budget the runaway would burn — the coaching signal is a statistical read (see below), not a per-event alarm.

## Adaptive-by-rollup sampling (Phase 1)

Reinforced by the same spec: each active worker's contribution to the ≤`GRADE_BATCH_CAP` sample is sized INVERSELY to its rolling average via `gradeSampleRateForAvg` (learning=1.0 → excellent=0.1). Combined with the coverage pass in `selectGradingBatch`, a proven worker contributes strictly fewer sample slots than a struggling one at equal pool sizes — the Max session budget flows to the workers that need supervision, not the ones that don't. The 0.1 floor for excellent workers keeps regression catchable (a slip pulls the sampled average into a heavier tier on the very next window — self-re-arm).

## Infra-cancellation exclusion (Phase 1)

A concluded job with `status IN ('failed','needs_attention')` whose `error` matches `INFRA_CANCEL_ERR_RE` (`runaway｜zombie｜cancelled｜stale-session｜session died｜reaper`) is treated as a NON-failure by both `selectGradingBatch`'s failure-priority set AND `agentGradingBatchReady`'s `hasFailure` gate. These are box-side infra events (reaper-killed session, Max-budget cap trip, auto-cancelled build) — not worker mistakes to learn from. Grading them as real failures dragged Vault (security-review) to 2.0/10 on 2026-07-02 from a string of reaper kills that had nothing to do with the security-review agent's judgment. Genuine `tsc failed` / `build failed` / conflict errors don't match the pattern and stay in the priority set.

**Bounded, prioritized cadence (CEO directive — don't jam the box).** A grading session grades **at most `GRADE_BATCH_CAP`** jobs. Selection (`selectGradingBatch`): **every failure** (`failed`/`needs_attention` — a worker mistake worth coaching on, newest first) up to the cap, then the remaining slots filled by a **round-robin-by-worker** random sample of the successes — so noisy routine workers (fold, pr-resolve) get spot-checked without crowding out quieter ones, and the rest stay ungraded for a later beat. A failure also makes `agentGradingBatchReady` true immediately (graded promptly, not on the ~3h fallback).

**Graduated per-worker success sample rate ([[../specs/grading-graduated-sample-rate]]).** Grading is a statistical read — a worker whose rolling average is high enough only needs a SAMPLE of its good work, not every action. `gradeSampleRateForAvg(avg, count)` maps the last-`ROLLUP_WINDOW` rolling {average, count} → the share of that worker's successful jobs to grade this beat. Replaces the old binary proven/not throttle so that a slip pulls the sampled average into a heavier tier on the very next beat (self-re-arm), while a consistently-excellent worker still gets a light-touch spot-check floor (never 0 — a regression would otherwise be invisible until failures land). FAILURES are always graded and short-circuit the sample (see `selectGradingBatch` `isFail`) — the coaching signal isn't sampled.

| Rolling avg | Sample rate | Meaning |
|---|---|---|
| `count < 5` OR `avg == null` | **1.0** | too little data — grade all until we have a read |
| `avg < 7` | **1.0** | learning zone — full grading; where the coaching signal lives (`COACH_LOW_ROLLUP=7`) |
| `7 ≤ avg < 8` | **0.5** | "sample, not all" begins |
| `8 ≤ avg < 9` | **0.2** (`GRADE_PROVEN_SAMPLE`) | proven — spot-check |
| `avg ≥ 9` | **0.1** | excellent — light-touch floor; never 0 so a regression is still catchable |

`PROVEN_AVG` was lowered `9→8` in the same pass — the old 9.0 bar was so high it almost never engaged, so nearly every action got graded. `8.0` matches the 6-7 acceptable / 8-9 strong break in the SCORING rubric. Coaching thresholds (`COACH_LOW_ROLLUP=7`, `DROP_THRESHOLD=1.5`, `COACH_MIN_SAMPLE=3`) are unchanged — with fewer grades landing per beat, `detectGradeDropCoaching` fires less on its own, no separate throttle needed. Effective daily `agent_action_grades` volume falls materially while failure coverage stays at 100%.

## How it grades
- **Inputs:** the concluded job's row — `kind` / `spec_slug` / `status` / `error` / `log_tail` / `pr_url` / the approved `pending_action`. (Phase 1 grades from the job-row context the runtime has; the Phase-2 Max box session that reads the real PR diff is the cadence layer.)
- **Model:** Sonnet ([[ai-models]]) for the fallback `gradeConcludedAgentActions` API path. Cost via [[ai-usage]] (`purpose='agent_action_grading'`). Both the box-hosted primary grader (Phase 1) and the box-hosted coach (Phase 2) run on Max — no per-token API bill; `purpose='agent_coaching_synthesis'` rows are retired.
- **Output (strict JSON):** `{ grade, reasoning }`, 1–10. The grade weights judgment-within-the-rubric over outcome luck.

## The coaching cascade

> **consolidate-grade-coach-one-session (2026-07-01): coaching is a FOLLOW-ON of the `agent-grade` session, not a separate lane.** The `agent-grade` box session already read every diff to grade the batch, so it also emits `coachings[]` (one durable learning per slipped batch worker) in the SAME JSON — near-zero extra tokens. `runAgentGradeJob` applies each via `applyBoxCoaching` (the full decision core), then calls `detectGradeDropCoaching` per graded kind with `alreadyCoachedKinds` so the enqueue is SKIPPED for a kind coached inline (`reason='coached_inline'`) — that call is now only the recovery-announce + a fallback enqueue for a kind the agent graded low but proposed no coaching. **[[../inngest/platform-director-cron]] no longer fans out coaching** (it dropped the per-kind `detectGradeDropCoaching` loop that spawned one `agent-coach` box job — each re-hydrating ~550K context + re-reading diffs — per slipped kind; the live "4 agent-coach queued" pileup, 2026-07-01). Net: ONE grade+coach lane per director pass; the standalone `agent-coach` lane remains only as a fallback + to drain any legacy in-flight rows.

`detectGradeDropCoaching` computes the rollup; on a slip (and when NOT already coached inline) it pulls the worker's recent low-graded actions and **enqueues an `agent-coach` box job** ([[../specs/grading-cascade-to-box-sessions]] Phase 2 — the LLM synthesis moved off the deployed API to a Max session that reads the REAL merged diffs). The box lane (`runAgentCoachJob` in [[../../scripts/builder-worker.ts]]) distills ONE durable learning (`CoachingLearning = {errorClass, triggeringPattern, guidance, reasoning}`) grounded in `git show <merge_sha>` output, then calls `applyBoxCoaching` — which re-checks rollup + loop-guard and delegates the actual write to the unchanged [[worker-instructions]] `coachAgent` (`coachedBy=PLATFORM` — the Director is the gate, a worker can't coach itself). That amends [[../tables/agent_instructions]] and logs the director→worker message in [[../tables/agent_coaching_log]]. Loop-guard: ≥`PLATFORM_DIRECTOR_LOOP_GUARD_MAX` coaching attempts that never `stuck` → `rollCoachingIntoFixSpec` (deterministic; unchanged from Phase 1) — never a CEO escalation.

**Blameless box-outage bypass** (before the Max session dispatches). If every low grade in the batch matches a box-level infra signature (Claude CLI auth eviction · breaker trip · identical box error) AND no grade carries a worker-attributable marker, `runAgentCoachJob` short-circuits: it writes a `kind='blameless_outage'` row to [[../tables/agent_coaching_log]] (deduped for 24h per (workspace, agent_kind)), marks the coach `agent_jobs` row `completed` — **never `needs_attention`** — and skips the Max session entirely. So an all-outage batch produces no [[../tables/dashboard_notifications]] card even while the outage grades age out. Boundary: one genuine worker-attributable low grade in the batch demotes the whole batch back to the normal coach → route → escalate path. See [[agent-coaching#blameless-box-outage-auto-resolve-skip-coaching-no-ceo-escalation]] · [[../specs/agent-coach-auto-resolves-blameless-box-outage-grade-batches-instead-of-escalating]].

## Where it's wired
- **Phase 1 ✅:** the store + grader library + the rollup/coaching trigger.
- **Phase 2 ✅:** the batched cadence — [[../inngest/platform-director-cron]]'s `grade-and-coach-workers` step runs `agentGradingBatchReady` (gate: ≥5 ungraded OR oldest >~3h) → **enqueues an `agent-grade` box job** (grading moved box-side, see below) → `detectGradeDropCoaching` per rubric-backed worker (idempotent + loop-guarded; catches slips landed by the box job within one beat), on the same `*/5` beat as the director grade sweep. Best-effort + idempotent. The step's `{considered,graded}` feeds the same cron's grading-liveness beat (`loop_id='director-decision-grading'`) so a starved worker-grading sweep is observable, not silent — `graded` here counts enqueued box jobs (a real backlog with a broken box lane still trips starvation after ≥2 consecutive sweeps).
- **Phase 3 ✅:** each worker profile (`/dashboard/agents/[role]`) gets a **rollup-grade card** (last-10 avg + trend) + a **graded-actions feed** (recent concluded jobs each with its grade) via `AgentGradePanel` ← `GET /api/developer/agents/agent-grades?kind=…` (owner-gated: `computeAgentRollup` + the recent concluded `agent_jobs` joined to their grades).

## The box-hosted coaching lane (Phase 2)

[[../specs/grading-cascade-to-box-sessions]] Phase 2 (same CEO directive 2026-06-30): the coaching **synthesis** LLM call also runs box-side on Max, so it reads the low-graded jobs' REAL merged diffs — not the paraphrased `agent_action_grades.reasoning` the deployed `synthesizeCoaching` used (which was itself paraphrased from the `log_tail`, coaching a self-report-of-a-self-report three layers removed from the actual code).

**Wiring:**
- New `agent_jobs.kind='agent-coach'` box lane in [[../../scripts/builder-worker.ts]] (`runAgentCoachJob` + `runAgentCoachClaude`, concurrency-1, `sandbox:'max'`, `ANTHROPIC_API_KEY` stripped). Skill: `.claude/skills/agent-coach/SKILL.md`.
- `detectGradeDropCoaching` on slip → ENQUEUES ONE `agent-coach` row with `instructions.agent_kind + low_grade_ids + low_agent_job_ids + rollup_average + rollup_drop + open_coaching_count` (dedup: skip re-enqueue while an `agent-coach` job for this `(workspace, agent_kind)` is already `queued｜queued_resume｜claimed｜building｜needs_input`). The recovery-announce + loop-guard (`rollCoachingIntoFixSpec`) paths remain deterministic and inline.
- The box lane fetches the low grades + their `agent_jobs` rows and asks the session to `git fetch origin main` + `git show <merge_sha>` (or `git diff origin/main...origin/<branch>`) each diff, spot the recurring code mistake, and emit a single-object verdict `{status, errorClass, triggeringPattern, guidance, reasoning}`.
- The learning lands via `applyBoxCoaching` — which re-verifies the rollup (`recovered_before_coach` no-op) + the loop-guard (`rolled_into_fix_spec` handoff), then calls the unchanged `coachAgent` + announces `🛠️ Ada coached <Worker>: <guidance>` on `#directors`. `agent_coaching_log` + `agent_instructions` versioning + `attempt` counting are byte-for-byte the same as the Phase-1 deployed path — only the LLM source moved.
- `synthesizeCoaching` (the deployed API call in `agent-grader.ts`) is fully **removed**. `ai_token_usage` rows with `purpose='agent_coaching_synthesis'` stop accruing.

## The box-hosted grading lane (Phase 1)

[[../specs/grading-cascade-to-box-sessions]] Phase 1 (CEO directive 2026-06-30): every grader that emits a grade runs box-side on Max. The deployed Sonnet sweep only saw the job-row metadata + `log_tail` — so it graded the worker's SELF-REPORT and capped itself at 7-9 explicitly "because I can't see the diff" (visible in `agent_action_grades.reasoning`). Grading compute moved to a Max `claude -p` session that reads the REAL diff (`git show <merge_sha>` / `git diff origin/main...origin/<branch>`), Reads the touched files, runs `npx tsc --noEmit` if in doubt, and checks `gh` CI status.

**Wiring:**
- New `agent_jobs.kind='agent-grade'` box lane in [[../../scripts/builder-worker.ts]] (`runAgentGradeJob` + `runAgentGradeClaude`, concurrency-1, `sandbox:'max'`, `ANTHROPIC_API_KEY` stripped). Skill: `.claude/skills/agent-grade/SKILL.md`.
- [[../inngest/platform-director-cron]] `grade-and-coach-workers` step: picks the batch via `pickAgentGradeBatch` (same selection as the deployed sweep — failures first, round-robin sample, proven workers throttled), enqueues one `agent-grade` row per batch-ready workspace with `instructions.agent_job_ids=[…]`. Dedup: skip re-enqueueing while an `agent-grade` job for this workspace is already `queued｜queued_resume｜building｜claimed` — one batched box session at a time.
- The box lane fetches each id, builds a prompt with the per-kind rubrics + approved [[../tables/agent_grader_prompts]] calibration rules + a compact pointer block per job (kind / spec / status / PR / branch / gated action / log_tail), and asks the session to emit `{decisions:[{agent_job_id, grade, reasoning}]}`. Grades land via `applyBoxGrade` — `model='box-max-session'`, `cost_cents=0`, same UNIQUE(agent_job_id) upsert + human-override invariant.
- After the batch writes, `runAgentGradeJob` fires `detectGradeDropCoaching` per newly graded worker kind — the rollup + `coachAgent` + eventual `rollCoachingIntoFixSpec` chain is unchanged; only the grade source moved.
- `gradeConcludedAgentActions` (the deployed API sweep) stays in the codebase as a **fallback** — an operator can invoke it manually if the box lane is offline. Retire once the box lane is proven green for a full rollup window.

## Gotchas
- **The ungraded-pool window must track the RECENT tail, not the OLDEST head** (fix-starved-grading round 2 — the recurring "she doesn't grade her agents at all"). `ungradedConcludedJobs` fetches gradeable+terminal jobs and subtracts the already-graded set. It MUST `order('created_at', { ascending: false }).limit(N)` (newest N), then re-sort ascending for the batch-ready gate. Ordering **ascending** clipped the window to the OLDEST N jobs — which are 100% already-graded once grading has ever run — so the `considered`/`ungraded` count was permanently ~0 while hundreds of fresh concluded jobs sat ungraded (the heartbeat read `considered:0` = "nothing to grade", not starvation, so it never even paged). Symptom in prod: 2972 gradeable-kind+terminal jobs all-time, the oldest 1000 all graded → `ungraded:1` despite a ~1000-job real backlog. After the fix: `ungraded:1000`, and every `*/5` beat drains 12 (`GRADE_BATCH_CAP`).
- **The already-graded set must be PAGINATED** — PostgREST caps a single response at 1000 rows regardless of `.limit()`, so the graded-job set (one row per ever-graded job, thousands over time) is fetched with a `range(from, from+999)` loop. A truncated graded set would falsely re-surface old graded jobs as ungraded and re-spend the LLM.
- **A failed grade attempt is logged LOUD — but in-flight skips stay silent.** In `gradeConcludedAgentActions`, a `gradeAgentAction` result with `!r.ok` is `console.error`'d ONLY when `r.reason` is NOT in `AGENT_INFLIGHT_SKIP_REASONS` (`not_concluded` · `job_not_found`). Those two are benign TOCTOU races with director triage (the director re-queues a parked job between the sweep snapshot and the grader's re-fetch — `needs_attention → queued`) and would otherwise light up the Control Tower / Vercel error feed as if the grader were broken. True grader errors (`grader_http_429` · `parse_failed` · `no_api_key` · `not_a_gradeable_worker`) still page — mirrors [[director-grader]]'s `INFLIGHT_SKIP_REASONS` / `isInflightSkip` pattern.
- **Supervised tool** ([[../operational-rules]] § North star) — scores a bounded proxy (action quality); the Director owns the objective, the CEO overrides. The grade only recommends a coaching pass, never a leash change.
- **Concluded-only + rubric-gated** — an in-flight job returns `not_concluded`; a non-worker / Director kind returns `not_a_gradeable_worker`.
- **Idempotent + human-safe** — keyed on the `agent_job_id` unique; a re-run UPDATEs in place, and `graded_by='human'` is never re-written.
- **Best-effort, never blocking** — the sweep is wrapped so a grader failure never breaks the caller; a missing `ANTHROPIC_API_KEY` is a clean no-op.
