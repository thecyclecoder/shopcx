# `src/lib/agents/agent-grader.ts` ŌĆö the worker-action grader

The DevOps Director's **worker-action grade** ŌĆö one row per graded **concluded `agent_jobs` row**, 1ŌĆō10 + reasoning ([[../specs/worker-grading-and-director-management]] Phase 1; the devops-director goal "the org learns + self-manages"). One level **down the org chart** from [[director-grader]]: there the CEO grades the Director's calls; here the Director grades each WORKER's actions, and a slip in a worker's rollup triggers a coaching pass. Persists to [[../tables/agent_action_grades]]; calibrated by approved [[../tables/agent_grader_prompts]] rules; coaches via [[worker-instructions]] `coachAgent`.

The defining invariant (inherited from [[director-grader]]): **grade the work, not outcome luck** ŌĆö a sound, well-scoped action that hit a rare reversible bump still grades well if the reasoning was right; a careless action that happened to land grades low. The grader is a SUPERVISED TOOL ([[../operational-rules]] ┬¦ North star): it scores a bounded proxy (action quality); the Director owns the objective and the CEO overrides it.

## The gradeable unit
One **concluded `agent_jobs` row** (terminal status `completed’Į£merged’Į£failed’Į£needs_attention`) ŌĆö a build merged, an error fixed/dismissed, an index proposed, a spec verified. **`merged` IS terminal** ŌĆö a build goes `building ŌåÆ completed ŌåÆ merged` (PR landed on `main`; `SUCCESSFUL_BUILD_STATUSES` in [[agent-jobs]]), and the `build` rubric literally scores "PR merged clean", so a `merged` build is the canonical thing to grade. (Omitting `merged` from `TERMINAL_JOB_STATUSES` silently no-op'd every merged build for days ŌĆö the same STARVED-grading root cause as [[director-grader]]; a post-merge finalization step that leaves the job at `merged` must not starve worker grading.) The job IS the worker's atomic action, so the director grader's polymorphic key collapses to a single FK (`agent_job_id`) + one rubric per worker `kind`.

## Per-worker rubrics
`AGENT_RUBRICS` (exported) maps each `agent_jobs.kind` ŌåÆ `{name, criteria}` (the spec's locked config): `build`ŌåÆBo (spec phases ┬Ę `tsc` ┬Ę clean merge), `repair`ŌåÆRafa (root-cause ┬Ę fix held), `regression`ŌåÆRemi, `db_health`ŌåÆDevi, `spec-test`ŌåÆVera, `migration-fix`ŌåÆMira, `pr-resolve`ŌåÆPax, `fold`ŌåÆFenn, `coverage-register`ŌåÆCole, `monitor`ŌåÆTao, `plan`ŌåÆPia, `product-seed`ŌåÆSol, `spec-chat`ŌåÆSage, `dev-ask`ŌåÆDex. `GRADEABLE_KINDS` = its keys ŌĆö only rubric-backed kinds are graded (the Director's own + non-worker kinds are excluded).

**Constants:** `ROLLUP_WINDOW=10` ┬Ę `COACH_LOW_ROLLUP=7` ┬Ę `DROP_THRESHOLD=1.5` ┬Ę `BATCH_MIN=5` ┬Ę `BATCH_FALLBACK_MS=~3h` ┬Ę `GRADE_BATCH_CAP=12`.

**Bounded, prioritized cadence (CEO directive ŌĆö don't jam the box).** A grading session grades **at most `GRADE_BATCH_CAP`** jobs. Selection (`selectGradingBatch`): **every failure** (`failed`/`needs_attention` ŌĆö a worker mistake worth coaching on, newest first) up to the cap, then the remaining slots filled by a **round-robin-by-worker** random sample of the successes ŌĆö so noisy routine workers (fold, pr-resolve) get spot-checked without crowding out quieter ones, and the rest stay ungraded for a later beat. A failure also makes `agentGradingBatchReady` true immediately (graded promptly, not on the ~3h fallback).

## How it grades
- **Inputs:** the concluded job's row ŌĆö `kind` / `spec_slug` / `status` / `error` / `log_tail` / `pr_url` / the approved `pending_action`. (Phase 1 grades from the job-row context the runtime has; the Phase-2 Max box session that reads the real PR diff is the cadence layer.)
- **Model:** Sonnet ([[ai-models]]) for the fallback `gradeConcludedAgentActions` API path. Cost via [[ai-usage]] (`purpose='agent_action_grading'`). Both the box-hosted primary grader (Phase 1) and the box-hosted coach (Phase 2) run on Max ŌĆö no per-token API bill; `purpose='agent_coaching_synthesis'` rows are retired.
- **Output (strict JSON):** `{ grade, reasoning }`, 1ŌĆō10. The grade weights judgment-within-the-rubric over outcome luck.

## The coaching cascade
`detectAgentCoachingSlips` computes each worker's rollup and returns the slip payload for any worker that needs coaching this pass. Recovery announces + the loop-guard `rollCoachingIntoFixSpec` fire INLINE from the detector (deterministic DB writes; no session needed). Slips that still need coaching are folded into the SAME `agent-grade` box session that grades the batch ([[../specs/box-grading-session-and-account-count-fixes]] Phase 1 ŌĆö retires the `agent-coach` fan-out cadence; the box's `agent-grade` session already read the low-graded diffs to grade adjacent jobs, so distilling the coaching off the same context is free). The box lane (`runAgentGradeJob` in [[../../scripts/builder-worker.ts]]) distills ONE durable learning per slip (`CoachingLearning = {errorClass, triggeringPattern, guidance, reasoning}`) grounded in `git show <merge_sha>` output, then the worker (deterministic) calls `applyBoxCoaching` per slip ŌĆö which re-checks rollup + loop-guard and delegates the actual write to the unchanged [[worker-instructions]] `coachAgent` (`coachedBy=PLATFORM` ŌĆö the Director is the gate, a worker can't coach itself). That amends [[../tables/agent_instructions]] and logs the directorŌåÆworker message in [[../tables/agent_coaching_log]]. Loop-guard: Ōēź`PLATFORM_DIRECTOR_LOOP_GUARD_MAX` coaching attempts that never `stuck` ŌåÆ `rollCoachingIntoFixSpec` (deterministic; unchanged) ŌĆö never a CEO escalation.

## Where it's wired
- **Phase 1 Ō£ģ:** the store + grader library + the rollup/coaching trigger.
- **Phase 2 Ō£ģ:** the batched cadence ŌĆö [[../inngest/platform-director-cron]]'s `grade-and-coach-workers` step runs `agentGradingBatchReady` (gate: Ōēź5 ungraded OR oldest >~3h) + `detectAgentCoachingSlips` (per-beat slip check across every rubric-backed worker), then **enqueues at most ONE combined `agent-grade` box job per (workspace, platform-director)** carrying both the grading batch (`instructions.agent_job_ids=[ŌĆ”]`) AND every slipped worker's coaching material (`instructions.slips=[ŌĆ”]`) ŌĆö [[../specs/box-grading-session-and-account-count-fixes]] Phase 1 retired the separate `agent-coach` fan-out. Best-effort + idempotent. The step's `{considered,graded,coached}` feeds the same cron's grading-liveness beat (`loop_id='director-decision-grading'`) so a starved worker-grading sweep is observable, not silent ŌĆö `graded` here counts enqueued combined sessions and `coached` counts slipped workers riding that session (a real backlog with a broken box lane still trips starvation after Ōēź2 consecutive sweeps).
- **Phase 3 Ō£ģ:** each worker profile (`/dashboard/agents/[role]`) gets a **rollup-grade card** (last-10 avg + trend) + a **graded-actions feed** (recent concluded jobs each with its grade) via `AgentGradePanel` ŌåÉ `GET /api/developer/agents/agent-grades?kind=ŌĆ”` (owner-gated: `computeAgentRollup` + the recent concluded `agent_jobs` joined to their grades).

## The box-hosted grading lane (Phase 1)

[[../specs/grading-cascade-to-box-sessions]] Phase 1 (CEO directive 2026-06-30): every grader that emits a grade runs box-side on Max. The deployed Sonnet sweep only saw the job-row metadata + `log_tail` ŌĆö so it graded the worker's SELF-REPORT and capped itself at 7-9 explicitly "because I can't see the diff" (visible in `agent_action_grades.reasoning`). Grading compute moved to a Max `claude -p` session that reads the REAL diff (`git show <merge_sha>` / `git diff origin/main...origin/<branch>`), Reads the touched files, runs `npx tsc --noEmit` if in doubt, and checks `gh` CI status.

**Wiring:**
- New `agent_jobs.kind='agent-grade'` box lane in [[../../scripts/builder-worker.ts]] (`runAgentGradeJob` + `runAgentGradeClaude`, concurrency-1, `sandbox:'max'`, `ANTHROPIC_API_KEY` stripped). Skill: `.claude/skills/agent-grade/SKILL.md`.
- [[../inngest/platform-director-cron]] `grade-and-coach-workers` step: picks the batch via `pickAgentGradeBatch` (same selection as the deployed sweep ŌĆö failures first, round-robin sample, proven workers throttled), enqueues one `agent-grade` row per batch-ready workspace with `instructions.agent_job_ids=[ŌĆ”]`. Dedup: skip re-enqueueing while an `agent-grade` job for this workspace is already `queued’Į£queued_resume’Į£building’Į£claimed` ŌĆö one batched box session at a time.
- The box lane fetches each id + each slip's low-graded action rows, builds a combined prompt (per-kind rubrics + approved [[../tables/agent_grader_prompts]] calibration rules + a per-job pointer block for the grading batch + a per-slip block listing low-graded actions with the prior grader's reasoning + additional-context blocks for any coaching-only jobs), and asks the session to emit `{decisions:[{agent_job_id, grade, reasoning}], coachings:[{agent_kind, errorClass, triggeringPattern, guidance, reasoning}]}`. Grades land via `applyBoxGrade` ŌĆö `model='box-max-session'`, `cost_cents=0`, same UNIQUE(agent_job_id) upsert + human-override invariant. Coachings land via `applyBoxCoaching` (per slip; `sourceGradeId` = the slip's first `low_grade_ids` entry).
- `gradeConcludedAgentActions` (the deployed API sweep) stays in the codebase as a **fallback** ŌĆö an operator can invoke it manually if the box lane is offline. Retire once the box lane is proven green for a full rollup window.

## Gotchas
- **The ungraded-pool window must track the RECENT tail, not the OLDEST head** (fix-starved-grading round 2 ŌĆö the recurring "she doesn't grade her agents at all"). `ungradedConcludedJobs` fetches gradeable+terminal jobs and subtracts the already-graded set. It MUST `order('created_at', { ascending: false }).limit(N)` (newest N), then re-sort ascending for the batch-ready gate. Ordering **ascending** clipped the window to the OLDEST N jobs ŌĆö which are 100% already-graded once grading has ever run ŌĆö so the `considered`/`ungraded` count was permanently ~0 while hundreds of fresh concluded jobs sat ungraded (the heartbeat read `considered:0` = "nothing to grade", not starvation, so it never even paged). Symptom in prod: 2972 gradeable-kind+terminal jobs all-time, the oldest 1000 all graded ŌåÆ `ungraded:1` despite a ~1000-job real backlog. After the fix: `ungraded:1000`, and every `*/5` beat drains 12 (`GRADE_BATCH_CAP`).
- **The already-graded set must be PAGINATED** ŌĆö PostgREST caps a single response at 1000 rows regardless of `.limit()`, so the graded-job set (one row per ever-graded job, thousands over time) is fetched with a `range(from, from+999)` loop. A truncated graded set would falsely re-surface old graded jobs as ungraded and re-spend the LLM.
- **A failed grade attempt is logged LOUD ŌĆö but in-flight skips stay silent.** In `gradeConcludedAgentActions`, a `gradeAgentAction` result with `!r.ok` is `console.error`'d ONLY when `r.reason` is NOT in `AGENT_INFLIGHT_SKIP_REASONS` (`not_concluded` ┬Ę `job_not_found`). Those two are benign TOCTOU races with director triage (the director re-queues a parked job between the sweep snapshot and the grader's re-fetch ŌĆö `needs_attention ŌåÆ queued`) and would otherwise light up the Control Tower / Vercel error feed as if the grader were broken. True grader errors (`grader_http_429` ┬Ę `parse_failed` ┬Ę `no_api_key` ┬Ę `not_a_gradeable_worker`) still page ŌĆö mirrors [[director-grader]]'s `INFLIGHT_SKIP_REASONS` / `isInflightSkip` pattern.
- **Supervised tool** ([[../operational-rules]] ┬¦ North star) ŌĆö scores a bounded proxy (action quality); the Director owns the objective, the CEO overrides. The grade only recommends a coaching pass, never a leash change.
- **Concluded-only + rubric-gated** ŌĆö an in-flight job returns `not_concluded`; a non-worker / Director kind returns `not_a_gradeable_worker`.
- **Idempotent + human-safe** ŌĆö keyed on the `agent_job_id` unique; a re-run UPDATEs in place, and `graded_by='human'` is never re-written.
- **Best-effort, never blocking** ŌĆö the sweep is wrapped so a grader failure never breaks the caller; a missing `ANTHROPIC_API_KEY` is a clean no-op.
