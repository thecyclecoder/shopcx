# libraries/agent-personas

The **reusable director-persona + SVG-mascot design system** ([[../specs/agents-hub-role-inboxes]] Phase 2) and the org-chart/inbox readers behind the [[../dashboard/agents|Agents hub]]. ONE reskinnable source of truth for the org-chart cast that every later milestone reuses — the gamified board (M3) and the live director's board posts (M4) render the same characters from here, so names/mascots/colors are never hardcoded across components ([[../operational-rules]]: reskinnable personas).

**Files:** `src/lib/agents/personas.ts` · `src/lib/agents/org-chart.ts` · `src/lib/agents/inbox.ts` · `src/components/agents/mascots.tsx` · `src/components/agents/persona-chip.tsx`

## `personas.ts` — the cast (reskin here)

- **`PERSONAS: Record<string, AgentPersona>`** — keyed by `functions/*.md` slug + the special `ceo` seat: 🛠️ **Ada**/platform · 🚀 **Max**/growth · 🎨 **Iris**/cmo · 💬 **June**/cs · 🧲 **Theo**/retention · 👑 **You**/ceo. The Platform **director** + **worker** cast: directors from `functions/` (Ada/platform · Max/growth · Iris/cmo · June/cs · Theo/retention); workers keyed by `agent_jobs` kind (Rafa/repair · Remi/regression · Devi/db_health · Cole/coverage-register · Vera/spec-test · Bo/build · Mira/migration-fix · Pax/pr-resolve · Fenn/fold · Tao/monitor · Pia/plan). **CEO-owned workers** (Phase 2 of [[../lifecycles/god-mode#autonomous-executive-assistant--god-mode-becomes-ceos-agent-phase-8]]): 🌙 **Eve**/god-mode — the founder's executive assistant from the god-mode cockpit, rendered UNDER the CEO seat (not under Platform) with liveness derived from cockpit activity + loop heartbeats. Each `AgentPersona` = `{ key, name, role, emoji, personality, voice?, pronouns?, responsibilities?, mascotId, avatarUrl?, chip, dot, ring, accent }`. Colors are **explicit Tailwind class strings** (not interpolated) so the compiler never purges them. **Persona upsert semantics** (builder): the builder's persona-add step checks for an existing entry with the same key and REPLACES it (upsert) rather than appending — preventing TS1117 duplicate-key errors when a build adds a persona whose key already exists. New personas are appended once; re-runs with the same key idempotently update the entry and regenerate its avatar.
- **`pronouns?: { subject, object, possessive }`** — gendered pronouns for prose written *about* a persona, so generated copy reads correctly per person instead of hardcoding one gender. Set on the **director cast** (Henry he/him · Ada she/her · Max he/him · Iris she/her · June she/her · Theo he/him); workers/unknowns are omitted and the **call site falls back to they/them**. Consumed by the [[../dashboard/agents|Director Guide]] (`director-guide.tsx`) — it resolves `persona.pronouns ?? {they/them/their}` and capitalizes at sentence start, so the guide says "His auto-approve envelope" for Max but "Her auto-approve envelope" for Ada. The guide does **not** repeat the identity header (avatar/name/role/personality) — the profile-page header above it already renders that; the guide opens with the plain-status callout.
- **Avatar style + assets** — `avatarUrl` is the real photoreal close-crop headshot in the public `agent-avatars` Supabase bucket (one per persona, `{name}-{role}.jpg?v=N`, bump `?v=` on replace; style rule lives at the top of `personas.ts`); `mascotId` is the inline-SVG fallback `PersonaAvatar` renders when no `avatarUrl` OR the headshot fails to load (`onError`). **Avatar generation is automated via the builder's persona-add step** — when the builder creates a new agent persona, it calls [[gemini]] `generateNanoBananaProCombine` with the personas.ts AVATAR STYLE + a role-appropriate prompt (e.g. "executive assistant doing X"), uploads to the `agent-avatars` bucket, and wires `avatarUrl` with a `?v=1` query stamp. Backfill is idempotent (re-runs don't regenerate existing avatars). Re-running the builder is safe — personas are **upserted by key** (never duplicate keys); a rebuild with the same key replaces the entry, re-generates the avatar, and bumps the version query-stamp.
- **`RESPONSIBILITIES: Record<string, string[]>`** (Phase 5) — the **precise per-worker responsibility list** keyed by `agent_jobs` kind, merged onto the persona by `getPersona`. Workers carry the most precise list (their exact mandate); directors + the CEO derive theirs from the brain (mandates / goals), so this map is the worker cast only. Reskin a worker's duties by editing one entry — rendered on its `/dashboard/agents/[role]` profile.
- **`getPersona(slug, label?)`** — resolves a persona by slug and folds in its `RESPONSIBILITIES` if any; **falls back to a neutral persona** for an unknown director/worker, so adding a new `functions/*.md` (or `agent-kind` lane) reskins the hub with **no code change** (brain-driven). Reskin a director by editing one entry — that is the template every other director inherits.
- Pure config (no server imports) → **safe to import from client components**.

## `mascots.tsx` — inline SVG avatars

- **`<Mascot id={MascotId} />`** — one friendly rounded-face SVG component per persona (`ada|max|iris|june|theo|ceo|default`), each with a role-hinting accessory (hard hat, rocket fins, paint palette, speech bubble, magnet horns, crown). **No asset pipeline** — inline SVG so M3's board can render them directly. Color inherits via `currentColor` from the persona's `accent`.

## `persona-chip.tsx` — the building blocks

- **`<PersonaAvatar persona />`** — the colored mascot tile. **`<PersonaChip persona />`** — a name/role pill. **`<StatusBadge status />`** — the live/autonomous badge; M1 default is `offline` ("routes to CEO"), flipped per-function by M2.

## `org-chart.ts` — the CEO → Directors → Workers reader (server-only)

- **`getOrgChart(): Promise<OrgChart>`** — builds the tree entirely from the brain: **directors** = `functions/*.md` via [[brain-roadmap]] `getFunctions()` (title, summary, mandates with spec counts, owned goals); the **CEO** seat carries the finite `goals/*.md` via `getGoals()`; **workers** = the **reconciled roster** (`buildRoster`, below) grouped under their owning director, each with a live status. **No hand-maintained second copy of the org chart** — one read, no drift. Server-only (brain-roadmap reads the bundled fs copy at request time + this now reads `agent_jobs`/`loop_heartbeats` via the service role). Used by `GET /api/developer/agents`.
- **`buildRoster(directorSlugs, liveKinds): RosterEntry[]`** ([[../dashboard/agents|reconciled roster]], folded — [[../archive]]) — the pure 3-source reconciler that makes the org view reflect **100% of running agents**: (1) `agent-kind` MONITORED_LOOPS (queue lanes), (2) persona-backed `cron` loops — a [[control-tower|registry]] `personaKind` (e.g. `control-tower-monitor`→`monitor`/Tao; the two `db-health-*` crons merge into one `db_health`/Devi), (3) any live `agent_jobs.kind` (rows in the last 7 days) with no registry row → unioned under Platform, `flagged:true` (e.g. `coverage-register`/Cole). De-duped by persona key. Shared with `scripts/audit-agent-roster.ts` so the drift guard measures the SAME roster.
- **`computeWorkerStatus(entry, admin, sinceIso, nowMs)`** (Phase 2) — derives `active` (recent job/beat) · `idle-healthy` (fired before / beating cron, idle now) · `inactive` (zero `agent_jobs` rows **ever** — e.g. Remi/regression) from `agent_jobs` recency + `loop_heartbeats` beats. The inactive flag reuses [[control-tower]] `evalCron`'s never-fired grace (a `registeredAt`-within-grace lane and any cron-backed loop are never death-flagged) so an idle-but-healthy / freshly-shipped lane is never falsely called dead.

## `inbox.ts` — the three-tab shell config (client-safe)

- **`INBOX_TABS`** — Messages · Approval Requests · Daily Summaries, each declaring the **reserved [[../tables/dashboard_notifications]] `type`** that fills it: `agent_message` (M3 board) · `agent_approval_request` (M2 routed queue) · `agent_daily_summary` (M3/M4 EOD recap). **`AGENT_INBOX_TYPES`** scopes the CEO inbox query so the generic bell's notifications stay out; **`tabForType(type)`** buckets a row. The backing store is `dashboard_notifications` (it already has type/title/body/link/read/dismissed) — **no new table** for the shell. M1 ships the shell + filters + empty states; M2/M3/M4 emit the rows.

## Why this exists

The [[../goals/devops-director]] goal stands up the Agent Org (CEO → Directors → Workers). M1 is the foundation surface; the persona/mascot system is the **design-system piece every milestone reuses**, and the org-chart + inbox readers are the **brain-driven data layer** under the hub — so the org chart is always a projection of `functions/` + `goals/`, never a copy.
