# libraries/repair-agent

The queue plumbing + autonomy policy behind the **Repair Agent** box agent ([[../specs/repair-agent]] Phase 1) ‚Äî "escalation-triage, but for the Control Tower." The Control Tower *detects* problems ([[error_events]] via `recordError`, [[loop_alerts]] via the monitor); this is the objective-owner loop above that proxy ‚Äî it *diagnoses* the root cause read-only and *proposes* the fix, the owner *approves* the build.

**File:** `src/lib/repair-agent.ts` ¬∑ the box runner is `scripts/builder-worker.ts` `runRepairJob` (see [[../recipes/build-box-setup]]).

## North star ‚Äî surface-don't-auto-build

The agent optimizes the bounded proxy "clear the error." Auto-spawning code builds from a noisy/flapping feed (many PRs + merge churn + cost) is the exact Goodhart failure ([[../operational-rules]] ¬ß North star). So the diagnosis + fix-spec is the high-value low-risk half the agent does autonomously; **building** (writes code / opens PRs / applies migrations) stays owner-gated ([[../specs/build-approval-gates]]). The DEFAULT authors the fix spec to main and **surfaces** it for one-tap owner Build; only a narrow mechanical allow-list auto-queues.

## Exports

- `type RepairVerdict = "real-bug" | "monitor-false-positive" | "foreign-app-noise" | "transient" | "needs-human"` ‚Äî the box's per-error verdict (it cites the root cause for each).
- `REPAIR_AUTOBUILD_KINDS: Partial<Record<RepairVerdict, string>>` ‚Äî the **narrow auto-queue allow-list**: only known-safe, mechanical, *monitor-only* classes may auto-queue their build, each with its justification. Default = `foreign-app-noise` ("scope the capture") + `monitor-false-positive` ("add a grace / tighten the assertion"). `real-bug` (touches product code) is NOT on it ‚Äî it surfaces for approval. Silence/auto is never the default.
- `isRepairAutobuildKind(verdict)` ‚Üí is this verdict on the allow-list (so its build may auto-queue)?
- `enqueueRepairJob(admin, { source, signature, title, errorEventId?, loopAlertId?, members? })` ‚Üí enqueue ONE `repair` [[agent_jobs]] job for a NEW Control Tower problem. Deduped GLOBALLY by signature (errors are global infra) ‚Äî skipped if a repair job for that signature is already **live** (active OR surfaced: `queued`‚Ä¶`queued_resume` ‚à™ `needs_attention`). `spec_slug` = the signature; `instructions` = the JSON brief the box loads from. **Best-effort, never throws** (it rides the error path). Workspace resolved from the latest [[agent_jobs]] row (the build queue is single-tenant), falling back to the first workspace. **Per-cycle cluster cap:** if more than `REPAIR_CLUSTER_CAP` (5) live repair jobs were queued inside the burst window (`REPAIR_CLUSTER_WINDOW_MS`, 10 min), further signatures fold into ONE `cluster:repair` job (find-or-append to its `members`) instead of N independent diagnoses. **`members` (cluster re-enqueue):** when `source==='cluster'` (e.g. an owner **Re-open** of a director-dismissed `cluster:repair`), the batched `members: ClusterMember[]` are persisted onto the inserted job's `instructions` so the re-triage investigates the SAME cluster ‚Äî `cluster:repair` is excluded from the burst counter, so a re-enqueue takes the standard insert path, which previously dropped `members` and emitted an empty `'CLUSTER ‚Äî 0 signatures'` brief ([[../specs/cluster-repair-reopen-preserves-members]]).
- **Dedup helpers (repair-agent-dedup):** `rootCauseKey(target, verdict)` ‚Üí a stable `implicated-file::failure-mode` key that collapses sibling signatures onto one spec; `normalizeImplicatedFile(target)` (lowercase, drop `./`/leading slashes, strip a `:line[:col]` suffix); `parseRepairSpecMeta(markdown)` ‚Üí `{ rootCause, signatures[] }` parsed from a spec's `**Repair-root-cause:**` / `**Repair-signature:**` marker lines; constants `REPAIR_CLUSTER_CAP`, `REPAIR_CLUSTER_WINDOW_MS`, `REPAIR_CLUSTER_SLUG` (`"cluster:repair"`), `REPAIR_RECENT_FIX_WINDOW_MS` (24h, the "pending deploy" window).
- `getOpenRepairs(admin, workspaceId)` ‚Üí `RepairSurfaceItem[]` ‚Äî READ-ONLY: the open repair items awaiting the owner (jobs in `needs_approval` = a proposed fix spec with a Build button, or `needs_attention` = a needs-human verdict, Dismiss only). Auto-queued + transient-resolved jobs complete silently and never appear here. Drives the Control Tower **Repair feed**. The [[platform-director|Platform/DevOps Director]] also reads the **`needs-human`** bucket here to **supervise** Rafa's no-fix calls ([[../specs/director-supervised-repair-dismissal]]) ‚Äî she adversarially re-checks each and dismisses ONLY what she can independently confirm benign (via the same Dismiss path), escalating a suspected masked real bug instead.
- `getDirectorDismissedRepairs(admin, workspaceId)` ‚Üí `DirectorDismissedRepairItem[]` ‚Äî READ-ONLY ([[../specs/director-supervised-repair-dismissal]] Phase 2): the Director's recent (14-day) repair dismissals STILL STANDING ‚Äî the `dismissed_repair` [[../tables/director_activity]] rows minus any whose `repair_job_id` carries a later `reopened_repair` row (the owner tapped Re-open). Title comes from the dismissal metadata. Surfaced under the Control Tower **Repair feed** as `üõÝÔ∏è Dismissed by Ada ‚Äî <reasoning>` with a one-tap **Re-open** (the CEO's override over the supervisor) that re-opens the `error_events` row + re-enqueues Rafa.

## Trigger ‚Äî event-driven (NOT a cron)

The error appearing IS the trigger; `enqueueRepairJob` is called inline at the two places the Control Tower records a NEW problem:
- [[control-tower]] `error-feed.ts` `recordError` ‚Äî the `!existing` (new signature) branch, right after it pages owners.
- [[control-tower]] `monitor.ts` `runControlTowerMonitor` ‚Äî right after it inserts a newly-opened [[loop_alerts]] incident (signature `loop:<loop_id>`).

## The box runner (`runRepairJob`)

A `kind='repair'` job runs on its own concurrency-limited lane (`MAX_REPAIR`, default 2) as a top-level `claude -p` on Max (web search on, no `ANTHROPIC_API_KEY`, KEEPS read-only DB/crypto secrets). Per error/alert:
0. **Already-fixed skip (before diagnosing)** ‚Äî `findAlreadyAddressing(signature)` reads the dedup ledger (recent `repair` jobs + the `root_cause` / `authored_slug` each persisted onto its own `instructions` JSON). If a prior repair already authored a fix spec for THIS signature within `REPAIR_RECENT_FIX_WINDOW_MS` (24h), the problem is addressed (its build is in-flight or merged-but-not-yet-deployed ‚Äî the stale-error trap) ‚Üí resolve the [[error_events]] row "fixed by [[spec]], pending deploy" + `completed`, **no LLM diagnosis**. (The enqueue dedup only blocks *live* same-signature jobs, so a re-fire after the prior COMPLETED lands here.)
1. **Investigate (READ-ONLY)** ‚Äî `loadRepairBrief` bakes in the signature + sample (the [[error_events]] row) or the [[loop_alerts]] row ‚Äî or, for a `cluster:repair` job, the batched list of `members`; the box traces the implicated route/cron/fn in the working tree. No prod writes.
2. **Classify** ‚Äî one `RepairVerdict`.
3. **Act:**
   - `real-bug` / `monitor-false-positive` / `foreign-app-noise` ‚Üí `groupOrAuthorRepairSpec` commits/groups a single-phase fix spec on main: **root-cause grouping** first ‚Äî if a sibling repair spec authored in the window covers the SAME `rootCauseKey` (implicated file + failure mode), this signature is *added to it* (one root cause ‚Üí one spec, N `**Repair-signature:**` lines) rather than authoring a near-duplicate; else author new (stamping `**Repair-root-cause:**`). Same-slug stays idempotent. The job persists `{ root_cause, authored_slug }` to `instructions` (the ledger the next repair reads). Then: on the allow-list ‚Üí **auto-queue** a `build` [[agent_jobs]] ‚Äî but only after `hasActiveBuildForSlug` confirms no build for that slug is already live / has an open `claude/<slug>-*` PR (**auto-build dedup: ‚â§1 build per slug** ‚Äî the 4-identical-PRs guard); else ‚Üí **surface** (`needs_approval` + a `repair_build` `pending_actions` entry carrying the slug).
   - `transient` ‚Üí no spec; resolve the [[error_events]] row + `completed` with a logged reason.
   - `needs-human` ‚Üí `needs_attention` with a plain one-line note (no spec, no loop).
   - **Terminal error-row close ‚Äî every disposition** ([[../specs/fix-error-reconcile-endless-loop]] Phase 1): `resolveRepairErrorRow(instr, reason)` drives the originating [[error_events]] row to a terminal `resolved` state **with a recorded reason** on EVERY disposition ‚Äî `real-bug`/etc (`fix [[slug]] authored, pending deploy`, whether auto-queued or surfaced), `transient`, already-fixed, `needs-human` (`needs-human: ‚Ä¶ (parked for human review on the repair feed)`), no-valid-spec, and an unparseable verdict. The human-review item for a `needs_attention`/`needs_approval` disposition lives on the **repair `agent_jobs` row** (which `getOpenRepairs` + the director's supervision read independently of `error_events.status`), so closing the row is safe and **reversible** ‚Äî a genuine re-fire re-opens it (recordError) and the owner Re-open path restores it. Only a hard run-error (`failed` job) leaves the row `open` so the reconcile re-drives a fresh diagnosis. This is what makes the row leave the `open` feed **exactly once** ‚Äî the [[platform-director]] reconcile no longer re-churns it. (`error_events` has no `updated_at` ‚Äî only `resolved_at`/`resolution_reason`; the old code wrote the missing `updated_at`, so the resolve silently failed and the row stayed stuck `open`.)
   - **Unparseable/unrecognized verdict** ([[../specs/needs-attention-triage-and-verdict-robustness]] Phase 2) ‚Üí the shared `resolveReviewVerdict` helper (shared with [[regression-agent]] + [[security-agent]]) RE-RUNS the diagnosis **once**, then fail-safes to an **actionable** `needs_attention` reason ‚Äî `"repair produced no parseable verdict after 2 attempts ‚Äî re-run or review manually: <excerpt>"` ‚Äî never a bare "ended without a recognizable verdict", never assume-resolved. The director's `reconcileNeedsAttention` ([[platform-director]] Phase 1) leaves repair items to `superviseRepairDismissals`, which reviews this parked item.
4. **Owner action resume** ‚Äî `POST /api/developer/control-tower/repair` flips the job to `queued_resume` with the `repair_build` action `approved` (Build) or `declined` (Dismiss); the box re-claims it and either queues the feature `build` job (the owner-gate ‚Äî also `hasActiveBuildForSlug`-deduped) or resolves the error row.

## Dedup + persistence discipline (repair-agent-dedup, repair-verify-spec-persisted-before-build)

The first live run (2026-06-22) **over-produced** ‚Äî 8 specs + 6 PRs for ~3 root causes + 1 real bug. Five guards harden it, all keyed off [[agent_jobs]] (no migration; the `instructions` JSON is the ledger) and the authored spec's DB persistence:
- **Phantom-fix guard ‚Äî verify spec persisted before enqueue** ([[../specs/repair-verify-spec-persisted-before-build]] Phase 1, 2) ‚Äî `groupOrAuthorRepairSpec` returns `{ authored_slug, ‚Ä¶}`, but a DB insert can silently fail (e.g., author-write exception not surfaced, or a race where the row vanishes before read). In `runRepairJob` (scripts/builder-worker.ts ~13868), AFTER authoring returns, call `getSpec(workspace_id, authored_slug)` to confirm the row exists (require the card, not phases ‚â•1 ‚Äî a one-shot spec authors 0 phases). On a missing row: (1) **do NOT enqueue** `repair_build` (Phase 1) ‚Äî the fix never lands, so queuing a build is phantom work; (2) **do NOT mark** `completed` ‚Äî leave the `repair` job in `needs_attention` with the error `"authored fix spec [[slug]] did not persist to public.specs ‚Äî silent author-write fallout"` so `getOpenRepairs` surfaces it as state='needs-human', and `error_events` row stays OPEN for reconcile re-scans (not resolved). (3) **Stamp director_activity** (action='repair_author_write_failed') for Ada's feed. (4) **Phase 2: escalate spec_row_missing build parks** ‚Äî when a `build` job later parks `spec_row_missing` (claim-gate ~ line 18022, or parked-router) and no spec row exists, flip the originating `repair` job (matched by `instructions->>authored_slug`) to `needs_attention` OR stamp a fallback `director_activity` row (action='spec_row_missing_escalated') so the silent dismissal surfaces ‚Äî never auto-delete a phantom without owner review.
- **Root-cause grouping** ‚Äî `rootCauseKey(target, verdict)` = `implicated-file::failure-mode`. Sibling signatures that share a key collapse onto ONE spec (N `**Repair-signature:**` lines), never N specs.
- **Already-fixed skip** ‚Äî a re-fire of a signature a prior repair already addressed (spec authored within 24h, build in-flight/merged-but-undeployed) resolves "pending deploy" with no re-diagnosis. Closes the stale-open-error trap (an error recorded before its fix deployed).
- **Auto-build dedup** ‚Äî `hasActiveBuildForSlug(slug)` (active `build` job OR open `claude/<slug>-*` PR) gates BOTH the auto-queue and owner-Build paths: ‚â§1 build per slug.
- **Per-cycle cluster cap** ‚Äî a burst beyond `REPAIR_CLUSTER_CAP` live jobs folds into one `cluster:repair` "investigate this cluster" job (it likely shares a cause), authoring one spec for the shared root cause.

## Gotchas

- **`instructions` is a TEXT column, not JSONB** ‚Äî filter predicates that assume JSON extraction (PostgREST `.filter('instructions->>field', ‚Ä¶)`) silently match 0 rows on a TEXT column. Use `.filter('instructions', 'ilike', '%"field":"value"%')` and parse client-side (precedent: scripts/builder-worker.ts:11958). This is why the phantom-completed redrive script (scripts/_redrive-phantom-completed-repairs.ts) filters on text pattern, not JSON operators; the client-side `JSON.parse` loop is authoritative and drops false-positive text matches. ([[../specs/repair-verify-spec-persisted-before-build]] Fix 1)
- **Dedupe is on the repair job, not a separate table** ‚Äî `recordError` only enqueues on a NEW signature, so a transient-resolved row that later bumps its `count` never re-triggers; a `needs_attention` (needs-human) item blocks re-enqueue (no loop) until the owner Dismisses it.
- **The dedup ledger is the `instructions` JSON** ‚Äî an authored/grouped repair job persists `{ root_cause, authored_slug }` back onto its row; `groupOrAuthorRepairSpec` + `findAlreadyAddressing` read recent rows in JS (one query, 24h window). Within a single burst, sibling jobs that finish near-simultaneously can race the ledger; same-slug convergence + the cluster cap are the backstops.
- **Authored specs are HUMAN-READABLE ‚Äî the founder reviews them before any build** (repair-spec-human-readable, 2026-06-29). The Repair Agent runs `claude -p` on Max, traces the implicated code read-only, and emits structured JSON; `repairSpecMarkdown` (in `runRepairJob`) materializes it. The diagnosis is LLM-authored (NOT templated from the signature) ‚Äî the prompt now requires prose `problem` (what's failing + the user-facing/system impact), `proposedChange` (concretely what changes), `why` (why it's the right fix), and a SPECIFIC `phase` step (not the old generic "Scope from the problem above; land the fix"). The spec renders these as `### Problem` / `### Proposed change` / `### Why` / `### Build step` subsections **inside the Phase 1 body** ‚Äî deliberately, because [[brain-roadmap]] `serializeSpecRowToMarkdown` (which rebuilds `getSpec().raw` the detail page renders) only preserves the H1, column-backed header lines, the summary, the per-phase body (verbatim), and `## Verification` ‚Äî arbitrary top-level `## Problem` H2s would be DROPPED on the DB round-trip. The summary is the LLM's one-paragraph `intent`.
- **Spec marker lines are machine-parsed** ‚Äî `**Repair-root-cause:**` + `**Repair-signature:**` (one per grouped signature) live in the **Phase 1 body** (so they survive the verbatim round-trip ‚Äî the summary path would mangle them: `firstParagraph` + `cleanInline` strip the `**`); `parseRepairSpecMeta` reads them anywhere in the body. Don't reformat them away.
- **Two rendering-bug fixes** (repair-spec-human-readable): (a) `serializeSpecRowToMarkdown` now strips a leading `Phase N ‚Äî/-/:` off the stored phase title before re-emitting `## Phase N ‚Äî {title}` ‚Äî `parseSpec` keeps the whole `Phase 1 ‚Äî close it` as the title, so without the strip it double-prefixed (`## Phase 1 ‚Äî Phase 1 ‚Äî close it`); this fix is global, not repair-only. (b) The template puts the `intent` summary FIRST under the H1 and the `**Owner:** ¬∑ **Parent:**` line BELOW it (no trailing `¬∑ **Verdict:**`, which `parseSpec`'s greedy Parent regex would have absorbed into the `parent` column) ‚Äî so the meta line is no longer swallowed into the summary and then DUPLICATED against the header the serializer regenerates from the Owner/Parent columns. Verdict is carried in the footer's `(verdict: ‚Ä¶)`.
- **No migration** ‚Äî `repair` is a free-text [[agent_jobs]] `kind`; the surfaced fix parks in the existing `pending_actions` jsonb (`type:'repair_build'`, `spec_slug`).
- **The repairer is watched too** ‚Äî registered as an `agent:repair` agent-kind tile in [[control-tower]] `registry.ts` (idle = green; a stuck repair job ‚Üí red).
- **Outage-aware ‚Äî don't triage into a dead API** ([[../specs/agent-outage-resilience]] Phase 2, [[claude-health]]): while the Claude-down breaker is tripped, `recordError` does NOT enqueue a repair job (it would just 529 ‚Äî the repair agent needs Claude to triage) and tags the error `outage_correlated` (a NEW signature is auto-resolved as transient ‚Äî outage-window 5xx are symptoms, not bugs). Already-queued `repair` jobs park `blocked_on_dependency` on the box and drain on recovery. So a Claude outage no longer spawns N redundant "retry the 5xx" proposals; the genuine gap (a call with no retry) is Phase 1's comprehensive fix, not a per-error churn.
