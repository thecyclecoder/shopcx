# mario

Mario's M3 SDK — the deterministic "is THIS spec stalled?" decision that fires the once-per-minute detector cron. Two exports: `evaluateStalledSpecs` turns the M1 [[../tables/spec_timecard_events]] ledger + the M2 wait-span vocabulary + every spec's uncleared blockedBy into candidates; `enqueueMarioJob` files one dedupe-guarded `kind='mario'` [[../tables/agent_jobs]] row per candidate. Called from [[../inngest/mario-stall-cron]] every minute; called nowhere else.

## Why this exists

The stall-vs-legit-wait distinction is the whole point of Mario, and it must be a DETERMINISTIC decision — not a Sonnet judgment call. A stall means the pipeline is silent for a reason nobody named (a wedged worker, an unqueued next phase, a missing status transition). A legit wait means someone or something explicitly paused this spec (an uncleared blocker, a needs_input question, an approval gate, a usage wall, a fold-cooldown). Mario's supervisability comes from that determinism: every drop below cites a concrete signal a human can inspect.

The [[../specs/spec-mario-stall-detector-cron-and-thresholds]] spec is explicit about which drops count as legit waits. This SDK implements that discriminator once so the M4 reasoning agent + any future Mario-adjacent code can share it.

## Exports

### `evaluateStalledSpecs(admin, workspace_id?): Promise<StalledCandidate[]>`

Returns EXACTLY the specs whose next lifecycle step is genuinely overdue.

Steps (mirroring the spec's a-e):

1. **Read every `mario_thresholds` row for the workspace.** Each row carries an `sla_ms` for a `(from_event, to_event)` pair — the M4 self-tuner is the only writer.
2. **Per row, call [[./spec-timecards]] `listStalledCandidates(admin, { older_than_ms: sla_ms })`.** Every returned candidate whose `last_event_kind` matches the threshold's `from_event` is a stall against THIS pair (the last event was `from_event`, and `to_event` has not landed within `sla_ms`). A candidate that hits under multiple thresholds surfaces once — first-match wins, threshold-scan order breaks the tie.
   - **Second candidate source — failed/orphaned builds (`readFailedBuildStalls`).** The timecard thresholds only watch happy-path `from → to` transitions, so a build that dies AFTER claiming (orphaned by a worker restart, crashed, or errored) emits no `build_done` event and trips no threshold — and a build that died before the M2 instrumentation went live has NO timecard events at all. This second scan reads the failure straight from `agent_jobs`: a spec whose LATEST `kind='build'` job is `failed` (not superseded by a newer active/completed build) and older than `MARIO_FAILED_BUILD_GRACE_MS` (20 min — past the worker's own orphan-reaper/re-drive window) surfaces as a `build_started → build_done` candidate with `brief.current_job_status='failed'`. Deduped against the timecard candidates, then run through the SAME drops below (a folded / blocked / phantom spec with a stale failed build is still dropped).
   - **Third candidate source — promote-gate held / loop-guard escalated (`readPromoteGateHeldStalls`).** A spec can pass every step yet be HELD unmerged because its pre-merge spec-test verdicted `issues` and the fix loop-guard fired (`director_activity` `escalated`, `metadata.signature='fixes-as-phases-loop-guard'`). Its LAST timecard event is a terminal verdict (no open gap) and its build COMPLETED (not failed), so neither the thresholds nor the failed-build source sees it. This reads the loop-guard escalation from `director_activity` (past `MARIO_PROMOTE_GATE_GRACE_MS`), then confirms the spec is STILL held (latest `spec_test_runs` verdict is still `issues`, so a since-resolved spec never re-fires), surfacing a `spec_test_verdict → promoted` candidate. The usual root cause is a **malformed verification** (a runtime-only bullet the preview can't satisfy, or a self-referential Fix-phase check) → Mario repairs it (see below).
   - **Fourth candidate source — review-failed / MISSING verification (`readReviewFailedVerificationStalls`).** A spec Vale bounced to `vale_pass=false` where at least one NON-FIX phase has an EMPTY `verification` column (NULL — never authored, distinct from the third source's *malformed* verification), aged past a 60-min grace (`MARIO_REVIEW_VERIFICATION_GRACE_MS`). This is the failure mode of a RAW `upsertSpec` bypass authored BEFORE the writer self-gate landed — the 4 stuck 2026-07-10 specs (plus any legacy stragglers) that Vale needs-fix'd and nobody ever re-authored. Surfaces a candidate with `brief.current_job_status='review_failed_missing_verification'` and a `review_started → review_passed` transition. Deduped + run through the SAME drops. Routes to the EXISTING `verification_repair` verb (below): Mario re-authors REAL per-phase verification through the author-spec gate and re-opens the spec to review.
   - **Fifth candidate source — review-failed / MISSING blocked_by (`readReviewFailedBlockerStalls`).** A spec Vale bounced to `vale_pass=false` whose body's `**Blocked-by:** [[slug]]` metadata line names a prerequisite that is ABSENT from the `specs.blocked_by` column — the mechanical Vale-bounce class where the declared blocker never made it onto the row (an authoring bug or a partial `upsertSpec` write). Scoped to the COMPLEMENT of the fourth source: every real (kind='phase') phase HAS verification, so no candidate is double-routed. Aged past the shared 60-min grace (`MARIO_REVIEW_VERIFICATION_GRACE_MS`). The raw markdown body is read from [[./brain-roadmap]] `getSpec(slug).raw` (via `serializeSpecRowToMarkdown`) — the authoritative reconstruction from `spec_phases` that [[../tables/specs]] `parseSpec` used to populate `specs.blocked_by` in the first place. `specs.body` has never existed as a column: [[../tables/specs]] carries card metadata; the body lives in [[../tables/spec_phases]]. Both `readReviewFailedBlockerStalls` (M1 detector) and `repairSpecBlockedBy` (M5 applier, Phase 2) source the body identically from spec_phases via [[./brain-roadmap]] to keep the missing-blocker detection and repair synchronized. Surfaces a `review_started → review_passed` candidate with `brief.current_job_status='review_failed_missing_blocker'` and pre-seeds `brief.review_failed_context = { blocked_by, body, vale_needs_fix_reason }` (the reason is stamped from [[./spec-investigation]] `whyDidSpecReviewFail` once the candidate survives (b)/(c)/(d)). A spec with no `**Blocked-by:**` line, or whose every named prerequisite is already on the row, is NOT surfaced — precisely the missing-blocker class, not every Vale needs_fix. Routes to the `blocked_by_repair` verb (Phase 2), which unions the missing slug into `specs.blocked_by` (additive-only, no removals) and re-opens the spec to review.
   - **Sixth candidate source — eligible-never-enqueued keystones (`readEligibleNeverEnqueuedStalls`).** The highest-value detector (Phase 1 of [[../specs/mario-detects-job-and-pr-wedges-not-just-spec-lifecycle]]): a spec with `auto_build=true`, every declared blocker shipped, but NO active build job strands every downstream dependent yet emits zero lifecycle signals (no timecard gap, no failed build, no Vale bounce). Reads specs table for `auto_build=true` candidates, probes in a single IN query for any `kind='build'` row across all candidates (scalable), and surfaces: status IN (`planned`, `null`) AND no build job AND aged past `MARIO_ELIGIBLE_NEVER_ENQUEUED_GRACE_MS` grace. Survivors reuse the existing `reclaim_and_redrive` verb to enqueue the missing build.
   - **Seventh candidate source — stuck-queued-build lane wedge (`readStuckQueuedBuildStalls`).** Phase 2 detector: a build sitting `status='queued'` for hours with `claimed_at IS NULL` — no worker ever picked it up — emits no failed-build signal and no timecard gap. Reads `agent_jobs` for `kind='build' AND status='queued' AND claimed_at IS NULL AND created_at < now - grace` (45 min grace, past the failed-build grace so a wedge that later fails is caught by source 2 first). Pre-seeds `brief.job_pr_context.job_id` for the M4 agent's `requeue_unclaimed_job` verb to target the exact starved row without re-scanning. Candidate carries a `from_event='build_queued'` which gates the job/PR-scoped drop relaxation (see (d0) below).
   - **Eighth candidate source — pr-resolve storm (`readPrResolveStorms`).** Phase 2 detector: a single PR's dirty-resolver keeps landing in `needs_attention`/`failed` and piling up rows — 2026-07-15: 61 parked rows for one PR over ~8h. Reads `agent_jobs` for `kind='pr-resolve' AND status IN ('needs_attention','failed') AND created_at > now - 24h`, groups by `pr_number`, and surfaces a `pr_number` with `≥ MARIO_PR_RESOLVE_STORM_MIN_PARKED` (3) parked rows with pseudo-slug `pr-<number>` (no matching `public.specs` row by design). Pre-seeds `brief.job_pr_context = { pr_number, parked_count }` for the M4 agent's new `cancel_pr_resolve_storm` verb. Candidate's `from_event='pr_resolve_storm'` also gates the job/PR-scoped drop relaxation (see (d0) below).
   - **Ninth candidate source — orphaned folded/shipped-spec open PR (`readOrphanedFoldedPrs`).** Phase 3 detector: PR #1893 sat open 7h because its spec had folded and no detector caught it. Reads `agent_jobs` for `kind='build' AND pr_number IS NOT NULL AND status NOT IN ('merged')`, cross-references with `specs.status IN ('folded','shipped')`, keeps newest build per spec slug (so a terminal spec with N historical builds fans out once). **Then DROPS any pr_number that has a SIBLING `merged` build row** — a PR merged via a different build row is not orphaned; because the primary scan excludes `merged` rows, a PR with an earlier `completed`/`needs_attention` build row (same pr_number) would otherwise survive as a FALSE orphan and re-enqueue a no-op mario job every cron tick (911 fix `mario-orphaned-pr-skip-merged` 2026-07-16: ~24 long-merged PRs like #868 mass-enqueued).  **Bounded to RECENT builds** (`MARIO_ORPHANED_PR_MAX_AGE_MS`, 7d): a genuinely orphaned PR is days-old at most, so months-old settled `completed` build rows (pre-`merged`-status convention, no `merged` sibling for the merged-dedupe to catch) are never re-litigated — the `mario-orphaned-pr-age-ceiling` 911 fix after ~99 no-op jobs re-enqueued each tick. Pre-seeds `brief.job_pr_context = { pr_number }` for the M4 agent's new `close_orphaned_pr` verb. The `from_event='orphaned_folded_pr'` gates the terminal-spec drop relaxation (see (d) below).
3. **The legit-wait discriminator** — the drops:
   - **(b) Uncleared blocker.** [[./brain-roadmap]] `getSpecBlockers(slug)` — any entry with `cleared:false` means the pipeline is intentionally holding this spec behind an upstream one. Drop.
   - **(c) Wait-status on the live job.** The candidate's current active [[../tables/agent_jobs]] row (any status in `ACTIVE_STATUSES`) with a status in `{ blocked_on_dependency, blocked_on_usage, needs_input, needs_approval }` is a paused build. Drop.
   - **(d) Phantom spec guard — no `public.specs` row.** A candidate with no matching `public.specs` row is a PHANTOM: the triggering timecard or failed-build event was backfilled from a stale `spec_status_history` row whose spec authorship FAILED (e.g., InvalidParentError) and never became a real spec. There is no pipeline to plumb — drop it. RELAXED for job/PR-scoped sources (Phase 2, [[../specs/mario-detects-job-and-pr-wedges-not-just-spec-lifecycle]]): a pr-resolve storm carries pseudo-slug `pr-<number>` that has NO matching `public.specs` row by design (the resolver runs against a PR, not a spec); the unconditional drop discards every genuine job/PR wedge. `JOB_PR_SCOPED_FROM_EVENTS` (sources 7–8) marks candidates whose `from_event` guarantees a job/PR class — those survive the no-specRow case and skip the specRow-dependent (d)/(d2) filters below.
   - **(d) Fold-cooldown / deferred / shipped override.** The candidate's `specs.status = 'folded'` or `'deferred'` (read straight through [[./specs-table]] `getSpec`) means the spec stopped emitting events on purpose. Drop. **Also drops a fully-SHIPPED spec, derived from the phase rollup** (`specRow.phases` non-empty AND every phase `shipped`) — a shipped spec presents with a NULL raw `specs.status` (that override column is only stamped for folded/deferred/in_review), so the folded/deferred check alone does NOT catch it, and Mario would otherwise fire on long-shipped specs that merely had a rough build history (911 fix `mario-skip-shipped-specs` 2026-07-16: a pipeline visibility increase surfaced ~50 such specs and mass-enqueued mario jobs against them). RELAXED for terminal-targeted sources (Phase 3, [[../specs/mario-detects-job-and-pr-wedges-not-just-spec-lifecycle]]): source 9 (`orphaned_folded_pr`) EXPECTS a folded/shipped spec — that's the exact class it chases (a terminal spec with a still-open PR). `TERMINAL_OK_FROM_EVENTS` marks that source so the terminal (folded/deferred/shipped) spec survives to the M4 agent's `close_orphaned_pr` verdict.
   - **(d2) Goal member awaiting atomic promotion.** A goal member whose build has already merged onto its `goal/<goalSlug>` branch (`specs.goal_branch_sha` stamped by M4 [[./specs-table]] `stampSpecGoalBranchSha`) and whose goal has NOT yet atomically promoted to `main` (`goals.main_merge_sha` still `null`) is a legit wait — its integration target is the goal branch, not `main`, and the promotion gate owns driving it to `main` via M5 `promoteCompleteGoalsToMain`. The pure predicate `isGoalMemberAwaitingPromotion({ milestoneId, goalBranchSha, goalMainMergeSha })` is the seam; its row-aware wrapper `isSpecRowAwaitingGoalPromotion(admin, spec)` reads the milestone → `goals.main_merge_sha` right after step (d) so the same `specRow` is reused (no second `getSpecFromDb`). Source-level drop is stronger than the Phase-1 applier-level catch: no `mario_fired` audit row, no wasted Max session, no ≥3-in-24h loop-guard pressure that would otherwise fire `mario_loop_guard` from repeated firings on the same awaiting slug. Belt-and-suspenders: Phase-1's applier gates (the `reclaim_and_redrive` vocabulary case + `surfaceMarioEscalationToAda` path) stay in place so a "surprise" mario job (a race with a mid-flight promotion, say) is still refused a reclaim build. A goal member that is NOT yet on its goal branch (`goal_branch_sha` null) is a genuine stall and still surfaces; a member of an already-promoted goal (`main_merge_sha` non-null) reaches the applier for normal handling; a standalone spec (no `milestone_id`) is unaffected. Pinned by `src/lib/mario.goal-member.test.ts` (Phase-2 tests).
4. **Attach a `MarioBrief` to every survivor.** The brief carries the last 10 timecard events, the `blockedBy` state, and the current job status — the exact payload the M4 reasoning agent needs, JSON-encoded onto `agent_jobs.instructions` by `enqueueMarioJob` so M4 picks it up without another read.

Idempotent (read-only). Safe to call every minute.

### `enqueueMarioJob(admin, candidate): Promise<{ enqueued, job_id?, reason? }>`

Files a `kind='mario'` [[../tables/agent_jobs]] row for a candidate, gated on "no active mario row for this spec_slug already exists".

Contract (from the spec):
- **SELECT first.** `.eq('workspace_id', ...).eq('kind', 'mario').eq('spec_slug', ...).in('status', ACTIVE_STATUSES)` — the workspace + spec + active-status filter proves the row belongs to THIS spec (never a cross-workspace slug collision).
- **If a row exists** → return `{ enqueued: false, reason: 'active_mario_exists' }` (no insert).
- **Re-fire COOLDOWN** (`MARIO_REFIRE_COOLDOWN_MS`, 60 min) → return `{ enqueued: false, reason: 'refire_cooldown' }`. The active-mario dedupe only blocks a CONCURRENT job; the moment a mario job completes with an escalate (or a fix that didn't clear the stall), the stall persists and the next ~1-min sweep re-enqueues — a per-minute Max-session burn on the same spec that the `mario_fixed` loop-guard never catches (escalations are `mario_fired`, not `mario_fixed`). The cooldown suppresses a re-fire when a `mario_fired` row for this spec landed within the window, so Mario looks at a still-stalled spec at most once per hour.
- **Else INSERT.** `agent_jobs { workspace_id, kind: 'mario', status: 'queued', spec_slug, instructions: JSON.stringify(candidate.brief) }`. Returns `{ enqueued: true, job_id }`.

This is app-layer dedupe (SELECT-then-INSERT). Safe under the once-per-minute cron because at most one tick evaluates a given spec at a time. A cross-cron race would insert a second row; M4's own claim step is designed to no-op on that (first-claim-wins).

## Self-service + escalation (applyBoxMario)

- **`reclaim_and_redrive` live-fix** — the built-but-unmerged class (a spec spec-test-approved + security-clean but whose latest build is `failed`/orphaned and never merged). Enqueues a FRESH build via owner-gated `queueRoadmapBuild` (rebases onto current `main` via [[builder-worker]] § Rebase-onto-main before repo-wide checks → clean branch → clean merge). Lets Mario self-service reviewing+merging a green PR instead of escalating — routine platform work, not a CEO decision. The worker's `ensureWorktreeSlotFree` frees a `BUILDS_DIR`-pinned branch first; the ephemeral `/tmp`-pinned edge is the `builder-worktree-self-heal` fix-spec. The `kind='build'` filter in `queueRoadmapBuild`'s existing-job guard (fixed in [[../specs/fix-queue-roadmap-build-kind-filter]]) is what lets this action enqueue at all — the live Mario job that INVOKES `reclaim_and_redrive` would otherwise be treated as the existing active build and coalesce the reclaim into a silent `{alreadyActive:true}` no-op.
  - **Goal-member exception — `isGoalMemberAwaitingPromotion` gate.** A goal member's correct integration target is its GOAL BRANCH (M4 `stampSpecGoalBranchSha` stamps `specs.goal_branch_sha`), NOT `main` — the member only reaches `main` via M5's atomic `promoteCompleteGoalsToMain`. So the "green spec-test + security but never merged" signal FALSELY reads as stuck for an already-integrated goal member. Reclaiming that member rebuilds it on a fresh branch head → resets `isSpecTestGreenForBranch` → flips `isSpecPromoteEligible` false → BLOCKS the whole goal's atomic promotion (the 2026-07-12 director-chats stall: generalize-director-coach-backend, an already-integrated member, was reclaimed and stalled the 6/6 promotion). A pure predicate `isGoalMemberAwaitingPromotion({ milestoneId, goalBranchSha, goalMainMergeSha })` returns true when ALL THREE hold — `milestoneId !== null` (goal member), `goalBranchSha !== null` (integrated on the goal branch), and `goalMainMergeSha === null` (goal not yet promoted). Its I/O wrapper `isSpecAwaitingGoalPromotion(admin, workspaceId, specSlug)` resolves the fields from `specs` + `goal_milestones` + `goals`, fails CLOSED to `false` on lookup error (unknown → prior reclaim behavior). `applyBoxMario` gates BOTH paths that would create a reclaim build for the goal member: (1) the `reclaim_and_redrive` vocabulary case — skips the fix with `fixReason = 'goal_member_awaiting_promotion'` (no `mario_fixed` row); (2) the escalate → `surfaceMarioEscalationToAda` path (below) — records `escalate_skipped_reason = 'goal_member_awaiting_promotion'` on the `mario_fired` metadata and never creates the `reclaim_stuck_build` `needs_approval` target for Ada. A genuinely orphaned goal member NOT yet on its goal branch (`goalBranchSha === null`), a member of an already-promoted goal (`goalMainMergeSha !== null`), and a standalone spec (`milestoneId === null`) all still reclaim as today. Pinned by `src/lib/mario.goal-member.test.ts` (4 verification bullets from [[../specs/mario-never-reclaims-a-goal-member-already-integrated-on-its-goal-branch]] Phase 1).
- **`verification_repair` (`MarioVerificationRepair`)** — repairs TWO classes into one verb: (a) the promote-gate-held / loop-guard class (3rd source), where the root cause is a MALFORMED verification (a runtime-only bullet the pre-merge preview can't satisfy so the spec-test agent auto-FAILs it, or a self-referential Fix-phase check); and (b) the review-failed / MISSING-verification class (4th source, `readReviewFailedVerificationStalls`), a Vale-bounced spec with a NULL `verification` column on a non-fix phase (a raw-`upsertSpec`-bypass authoring failure). Mario proposes the corrected — or, for (b), the first real — locally-checkable verification per REAL phase; `applyBoxMario` (`repairSpecVerification`) re-authors the spec with it via `authorSpecRowStructured` — DROPPING the auto-generated Fix phases that caused the loop — which re-opens the spec → re-review → rebuild → the pre-merge spec-test now has a passable check → it promotes. A phase Mario doesn't correct keeps its current verification; a bare-function parent is never passed through (falls back to the owner's `infra-devops-reliability` mandate). Recorded as `mario_fired.metadata.verification_repaired`.
- **`blocked_by_repair` (`MarioBlockedByRepair`)** — the fifth candidate source (Vale-review-failed with missing-blocker class): a spec whose body declares a `**Blocked-by:**` prerequisite line and that blocker slug is NOT yet in `specs.blocked_by` (the blocker's PR isn't on `main` yet). Mario verifies the spec is in the missing-blocker class per `shouldSurfaceMissingBlocker` (not folded/deferred, within grace window, real phases have verified, body's `**Blocked-by:**` line is non-empty, not every named prerequisite is already on specs.blocked_by), then proposes adding the named missing blockers. `repairSpecBlockedBy` (the Phase 2 applier) reads the spec body via `getSpec` from [[./brain-roadmap]] (dynamic-imported to reconstruct markdown from spec_phases via `serializeSpecRowToMarkdown`) — the same authoritative body read that `readReviewFailedBlockerStalls` uses, kept consistent by sourcing from spec_phases not from a non-existent `specs.body` column. `applyBoxMario` re-authors the spec with the merged `blocked_by` (UNION of existing + verdict.add_blocked_by) via `authorSpecRowStructured`, re-opens the spec to Vale. TWO pure predicates gate the application for security (scope-mario-blocked-by-repair-target Phase 1): `checkRepairBlockedByScope(input: { jobSpecSlug, repair, spec, graceMs, now })` rejects any verdict whose spec_slug differs from the job row (LLM can't retarget) OR whose add_blocked_by contains a slug outside the derived missing set (current body `**Blocked-by:**` prerequisites MINUS current specs.blocked_by); `mergeBlockedByForRepair(input: { existing, add })` rejects empty add-lists or any payload that would drop an existing blocker (ADDITIVE-ONLY). Returns `{ ok: false, reason }` if rejected, else `{ ok: true, missingSet/merged }`. The deterministic service-role worker only applies the exact missing-blocker repair the stall detector surfaced, regardless of what the LLM verdict or spec body says. Recorded as `mario_fired.metadata.blocked_by_repair_applied` or `blocked_by_repair_error`.
- **`cancel_pr_resolve_storm` (Phase 2, [[../specs/mario-detects-job-and-pr-wedges-not-just-spec-lifecycle]])** — the eighth candidate source (pr-resolve storm). Flips every parked resolver row for the target PR (`agent_jobs.kind='pr-resolve' AND pr_number=N AND status IN ('needs_attention','failed')`) to `status='completed'` with a cancellation reason. The deduper's active-row guard then lets a fresh resolve enqueue on the next webhook without a phantom pile (the remedy for the 2026-07-15 overnight 61-row storm). Deterministic compare-and-set: throws if zero rows matched (the storm dissipated in the meantime), and the applier records the reason on the `mario_fired` row → the escalate path surfaces to Ada if the cancel is a no-op. Recorded as `mario_fired.metadata.pr_storm_cancelled` or `pr_storm_cancel_error`.
- **`close_orphaned_pr` (Phase 3, [[../specs/mario-detects-job-and-pr-wedges-not-just-spec-lifecycle]])** — the ninth candidate source (orphaned folded/shipped-spec open PR). Three guards before mutation: (1) re-read spec status via [[./specs-table]] `getSpec` and refuse if not folded/shipped anymore (stale observation), (2) re-scan `agent_jobs` for the most-recent open-PR build row + assert branch starts with `claude/` (never touches a human PR), (3) fetch head sha via new `getPrHead` helper in [[./github-pr-resolve]] (fail-closed: returns head ref + sha only for open non-fork PRs). Then calls existing `closeDuplicatePr` with the authorized `expectedHeadSha` + a mario-branded comment + branch delete. Recorded as `mario_fired.metadata.orphaned_pr_closed` or `orphaned_pr_close_error`.
- **Escalate → Ada (actionable, not a dead row).** When Mario escalates AND applied no live fix, `applyBoxMario` creates an ACTIONABLE target Ada can fix: a fresh `kind='build'` job for the stuck spec parked `needs_approval` with a `reclaim_stuck_build` pending action (deduped — no target if a build is already in-flight for the spec). Because `build` routes to platform, Ada's `enqueuePlatformDirectorJobs` sweep ([[agents-platform-director]]) picks it up; `reclaim_stuck_build` is in-leash (`error_fix` in `LEASH_ACTION_TYPES`) so she AUTO-APPROVES after her read-only investigation, and on approval the job resumes and **rebuilds the spec on current `main` (clean branch → clean merge) — the build IS the reclaim**. Ada is the reviewer+merger; the CEO is never in the loop for routine platform work. Recorded as `mario_fired.metadata.escalated_to_ada`. This closes the loop: Mario self-services the common built-but-unmerged case (`reclaim_and_redrive`); the residual he can't/shouldn't fix himself he hands to Ada, who actually acts.

### Fix-phases carry a typed machine check (mario-fix-authoring-emits-machine-checks-not-needs-human Phase 1)

Every phase Mario's fix-authoring hands `authorSpecRowStructured` MUST carry a `checks[]` with ≥1 machine-runnable check. Without it, the chokepoint falls back to `parseVerificationBlobToChecks(verification)` in [[./spec-phase-checks-table]] — every prose bullet stamped `exec_kind='needs_human'` — and [[./author-spec]] `assertEveryPhaseHasMachineCheck` throws `MissingMachineCheckError` because no check is auto-testable. The fix-spec never lands → the origin stays red → the stall detector re-fires → structural oscillation (observed live on the media-buyer neq-mock-fix spec: code green, spec-test 63/63, held ~2.5h and escalated to CEO). Mario cannot self-author the fix because its fix-authoring has the SAME defect (chicken-and-egg), so the invariant is enforced INSIDE mario.ts:

- `authorMarioFixSpec` (the `MarioDurableFixSpec` applier) attaches `defaultMarioFixPhaseChecks()` to every phase.
- `repairSpecVerification` (the `MarioVerificationRepair` re-author path — the fourth-source class + malformed-verification class) attaches the same default to every real (`kind='phase'`) phase it re-authors.
- `repairSpecBlockedBy` (the `MarioBlockedByRepair` re-author path — fifth-source missing-blocker class) attaches the same default, since it re-authors real phases too and would otherwise trip the same gate.

`defaultMarioFixPhaseChecks()` returns a single `{ position: 1, kind: 'auto', exec_kind: 'tsc', params: null }` check — the same safe default `buildStructuredSpecInputFromMarkdown` in [[./author-spec]] uses on the markdown author path, and [[./repair-agent]]'s `derivedDefaultRepairChecks` uses on the coverage-repair path. A bare `tsc` gate is safe across every Mario fix class: a pipeline-reliability fix landing on `main` MUST typecheck. The LLM-proposed prose rides verbatim on the phase's `verification` column (human-facing); the tsc check is what the deterministic runner in [[./spec-check-runner]] actually executes.

### Types

- **`MarioBrief`** — `{ last_events: [{ event_kind, phase_index, actor, at, wait_kind, waiting_on }; ≤10], blocked_by_state: [{ slug, cleared }], current_job_status: string | null, review_failed_context?: { blocked_by, body, vale_needs_fix_reason }, job_pr_context?: { job_id?, pr_number?, parked_count? } }`. The bounded payload M4 picks up. `review_failed_context` is set ONLY on the fifth-source (missing-blocker) candidates. `job_pr_context` is set ONLY on Phase-2 sources (7–8: stuck-queued, pr-resolve-storm) and Phase-3 source (9: orphaned-folded-pr) so the M4 agent can target the exact job_id / pr_number without re-scanning.
- **`StalledCandidate`** — `{ workspace_id, spec_slug, from_event, to_event, gap_ms, sla_ms, brief }`. One per surviving candidate.
- **`MarioThreshold`** — `{ workspace_id, from_event, to_event, sla_ms, min_count }`. Mirrors a [[../tables/mario_thresholds]] row.
- **`MarioBlockedByRepair`** — `{ spec_slug: string, add_blocked_by: string[], reasoning: string }`. Mario's verdict to add one or more missing blocker slugs (the fifth candidate source); each slug in add_blocked_by must be a bare slug (no `../specs/` prefix or `.md` suffix). Gated by `checkRepairBlockedByScope` (rejects slug-mismatched verdicts and out-of-class specs) and `mergeBlockedByForRepair` (rejects empty adds and drop-existing payloads).
- **`ACTIVE_MARIO_STATUSES`** — the SDK's dedupe status set. Currently identical to [[./agent-jobs]] `ACTIVE_STATUSES`; inlined so a future widening of `ACTIVE_STATUSES` doesn't silently change Mario's dedupe surface without an intentional edit here.

## Legit-wait discriminator (explicit)

Every drop below is a legit wait, NOT a stall. Mario surfaces only the pipeline silences nobody named.

| Signal | Where read | Why it's a legit wait |
|---|---|---|
| `blockedBy[i].cleared === false` | [[./brain-roadmap]] `getSpecBlockers` | An upstream spec's code isn't on `main` yet; the build pipeline refuses to dispatch until the blocker clears (spec-blockers). The gap is by design. |
| `agent_jobs.status === 'blocked_on_dependency'` | live `agent_jobs` row | The worker parked the build waiting for a Claude-breaker to unblock a dependency (Claude-breaker path). The status transition itself is the "waiting" marker. |
| `agent_jobs.status === 'blocked_on_usage'` | live `agent_jobs` row | Every Max account hit its usage wall ([[./box-multi-account-failover]]); the worker parks the build until reset. Auto-resumes; not a stall. |
| `agent_jobs.status === 'needs_input'` | live `agent_jobs` row | The build asked the CEO a question and is waiting for the answer. |
| `agent_jobs.status === 'needs_approval'` | live `agent_jobs` row | The build proposed a gated action (apply_migration / run_prod_script / …) and is waiting for the CEO's approve. |
| `specs.status === 'folded'` | [[./specs-table]] `getSpec` | A folded spec is archived — every downstream reader knows it stopped emitting events on purpose. |
| `specs.status === 'deferred'` | [[./specs-table]] `getSpec` | The CEO explicitly deferred this spec. The pipeline holds no work here. |

Every OTHER silence (last event was `build_done` and no `phase_shipped` in 30min; a `job_queued` that no worker has claimed in 10min; a `review_started` that never emitted a verdict) is a stall — Mario enqueues a job, M4 investigates.

## Pure predicates for repair validation (scope-mario-blocked-by-repair-target Phase 1)

Two pure decision functions gate the blocked_by_repair application so the deterministic service-role applier never silently deviates from the exact scope the stall detector surfaced:

### `checkRepairBlockedByScope(input: { jobSpecSlug, repair, spec, graceMs, now })`

Pure security predicate — recomputes the fifth-source (missing-blocker) class as-of NOW and rejects any verdict that either (a) names a spec_slug different from the Mario job row's spec_slug (LLM can't retarget), (b) points at a spec whose current state no longer fits the missing-blocker class per `shouldSurfaceMissingBlocker` (folded/deferred, still within grace window, a real phase lost its verification, the body's `**Blocked-by:**` line was cleared, or every named prerequisite is already on specs.blocked_by), or (c) requests any add_blocked_by entry that isn't in the derived missing set (current body `**Blocked-by:**` prerequisites MINUS current specs.blocked_by). Returns `{ ok: true, missingSet: string[] }` on acceptance, or `{ ok: false, reason: string }` with the rejection reason (e.g., `spec_slug_mismatch`, `not_missing_blocker_class`, `add_not_in_missing_set`). Split out as pure so the security contract is unit-testable without a Supabase stub.

### `mergeBlockedByForRepair(input: { existing: string[], add: string[] })`

Pure decision predicate — computes the merged blocked_by (UNION of existing + add) or rejects the verdict. ADDITIVE-ONLY: an empty add-list is rejected; any payload that would DROP an existing blocker is rejected (verification bullet: "drop existing blocker → rejected"). Returns `{ ok: true, merged: string[] }` on success, or `{ ok: false, reason: "empty_add" | "would_drop_existing" }` on rejection. Split out as pure so the contract is unit-testable.
