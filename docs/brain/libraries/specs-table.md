# libraries/specs-table

The read/write surface for the DB-resident spec body ‚Äî [[../tables/specs]] (the card) + [[../tables/spec_phases]] (the per-phase rows). Authored by [[../specs/spec-body-table-and-backfill]] Phase 2; parallel to [[spec-card-state]] (status-only mirror) until [[../specs/spec-readers-from-db-retire-parser]] (M3) retires the markdown parser.

**File:** `src/lib/specs-table.ts`

## Why this exists

[[../specs/spec-body-table-and-backfill]] adds the relations [[../tables/specs]] + [[../tables/spec_phases]] so the spec body is queryable + indexable in the DB (and so a phase can move between specs preserving its stable id + PR/SHA provenance ‚Äî see below). This module is the canonical writer + read surface those rows are managed through. NO reader has been retargeted yet ‚Äî `getRoadmap` / `getSpec` / the board still read markdown via [[brain-roadmap]] in this milestone; the [[../recipes/backfill-specs-from-markdown]] one-time backfill seeds the rows so M2/M3 can lean on them.

## Types

- **`SpecStatus`** = `'in_review' ÔΩú 'planned' ÔΩú 'in_progress' ÔΩú 'shipped' ÔΩú 'deferred' ÔΩú 'folded'` ‚Äî the caller-facing enum (the type callers may PASS). `SpecRow.status` is typed `SpecStatus | null`. **`specs-status-overrides-only`** (migration `20260907130000`): the stored `specs.status` column carries ONLY `deferred` / `folded` / `NULL` ‚Äî the CHECK REJECTS everything else. `in_review` is no longer stored; it is DERIVED at read time from the phase rollup + `vale_review_passed_at` ([[brain-roadmap]] `deriveSpecCardStatus`), alongside the planned/in_progress/shipped axis (no trigger ‚Äî dropped in `20260725160000`). Every writer clears a non-`{deferred,folded}` destination to `NULL` (the [[spec-card-state]] `isOverrideStatus` gate) ‚Äî so a caller may still PASS `in_review`/derived values, but they land as `NULL`. A zero-phase one-shot spec derives `shipped` from its `merged_pr`/`last_merge_sha` provenance, never from the stored column.
- **`Phase`** ‚Äî re-exported from [[brain-roadmap]] so callers don't churn imports. `'planned' ÔΩú 'in_progress' ÔΩú 'shipped' ÔΩú 'rejected'`.
- **`SpecRow`** ‚Äî `{ id, workspace_id, slug, title, summary, owner, parent, blocked_by, priority, deferred, intended_status, status, intended_status_set_by, repair_signature, regression_of_slug, regression_signature, auto_build, vale_pass, vale_review_passed_at, ada_disposition, vale_disposition, vale_disposition_reason, milestone_id, merged_pr, last_merge_sha, created_at, updated_at, phases: SpecPhaseRow[] }` ‚Äî the parent + joined ordered phases. `vale_review_passed_at` (timestamptz) is the DURABLE "passed Vale review" stamp the claim-time build gate reads (build-gate-durable-review-signal) ‚Äî set on a Vale PASS alongside the transient `vale_pass`, but NOT consumed by Ada's disposition. **`vale_disposition`** (`'planned'`/`'deferred'`/null) + **`vale_disposition_reason`** (text/null) carry Vale's reasoned recommendation on a PASS ([[../specs/vale-reasons-the-disposition]] Phase 1); Ada's Phase-2 sweep consumes them (retiring the trust-the-author stub). `regression_of_slug` / `regression_signature` mirror the regression-agent header lines (`**Regression-of:** [[<slug>]]` / `**Regression-signature:** `<sig>``) ‚Äî typed columns added in [[../specs/spec-authoring-writes-db-and-worker-materialize]] Phase 1, sibling to `repair_signature`. `merged_pr` (int) / `last_merge_sha` (text) carry a **one-shot spec's** card-level shipping PR + SHA ([[../specs/spec-status-phase-pr-provenance]]) ‚Äî a zero-phase spec has no `spec_phases` slot to hold provenance, so it lands on the parent row; multi-phase specs use `spec_phases.pr` instead and leave these null.
- **`SpecPhaseRow`** ‚Äî `{ id, spec_id, position, title, body, status, pr, merge_sha, verification, created_at, updated_at }`. `position` is 1-indexed.
- **`SpecRowInput`** / **`SpecPhaseInput`** ‚Äî the writable field sets `upsertSpec` accepts.

## Exports

- **`getSpec(workspaceId, slug)`** ‚Üí `SpecRow | null` ‚Äî the parent row + its [[../tables/spec_phases]] rows ordered by `position`. Read by authenticated users (RLS) ‚Äî service role bypasses for backfill / writers. Sourced from **`public.get_spec_with_phases(uuid, text)`** (migration `20261004120000_get_spec_with_phases_rpc.sql`, [[../specs/cut-internal-egress-pooler-and-spec-rpcs]] Phase 2) ‚Äî the sibling of `list_specs_with_phases` for a single-spec read. On the box the Phase-1 [[pg-pool]] path is preferred (one pooled query, no PostgREST `set_config` preamble); anywhere the pool is unavailable falls back to the supabase-js RPC call. Retires the two `.from()` round trips (`.from('specs')` then `.from('spec_phases')` keyed on the returned id) that used to charge every getSpec call ‚Äî same set_config preamble + auth churn `list_specs_with_phases` already retired for [[listSpecs]]. Returned SpecRow is byte-identical to the pre-Phase-2 path.
- **`listSpecs(workspaceId, filter?)`** ‚Üí `SpecRow[]` ‚Äî every spec in a workspace, optionally filtered by `{ status, owner, milestone_id, scope }`. Sourced from the **`public.list_specs_with_phases(uuid, text)`** Postgres RPC (migration `20261001120000_list_specs_with_phases_rpc.sql`, [[../specs/list-specs-with-phases-rpc-retire-in-array-client-join]] Phase 1): the RPC does the `specs` + `spec_phases` join **server-side** and streams `(spec jsonb, phases jsonb)` rows back, so no id array crosses the wire. Retired the interim `.in("spec_id", [ids])` slug-batching ‚Äî the DURABLE fix for the `UND_ERR_HEADERS_OVERFLOW` (~16KB undici header cap) that wedged getSpec + the build claim-gate + the spec-review enqueue reaper once the workspace held a few hundred specs. `filter.scope ‚àà 'active' | 'archived' | 'all'` picks the server-side row set (`'active'` = `status IS NULL OR status <> 'folded'`, `'archived'` = `folded`, `'all'` = every row); defaults to `'all'` so folded-inclusive callers ([[../libraries/director-kpis]] owner attribution, [[spec-drift]], [[../libraries/agents/spec-dispose]] audits) keep pre-RPC semantics. `status` / `owner` / `milestone_id` are applied client-side on the bounded RPC result set.
- **`getActiveSpecs(workspaceId)`** / **`getAllSpecs(workspaceId)`** ‚Üí `SpecRow[]` ([[../specs/list-specs-with-phases-rpc-retire-in-array-client-join]] Phase 2) ‚Äî thin scope wrappers over `listSpecs`. `getActiveSpecs` calls the RPC with `scope='active'` (folded rows filtered SERVER-SIDE) ‚Äî prefer this on board / pipeline / claim-gate readers that never want folded rows across the wire. `getAllSpecs` calls with `scope='all'` ‚Äî the folded-inclusive set the pre-RPC `listSpecs()` returned; use it on readers that need the full set (owner attribution, drift audits, dispose sweeps).
- **`getFixSpecForOrigin(workspaceId, originSlug)`** ‚Üí `SpecRow | null` ‚Äî resolve the regression FIX spec for an origin by the typed linkage `specs.regression_of_slug = originSlug` (NOT a hand-typed/deterministic slug), most-recently-created first, phases joined like `getSpec`. The inline spec-test card ([[../specs/spec-test-request-fix-inline-author-and-approve]] Phase 2) reads it so a renamed fix slug still surfaces ‚Äî the `regression_of_slug` column (set by the request-fix authoring path + the regression-agent's `**Regression-of:**` header) is the source of truth. Read-only, service-role.
- **`upsertSpec(workspaceId, row, phases)`** ‚Üí `{ spec_id, phase_ids }` ‚Äî the canonical write. UPSERT by `(workspace_id, slug)` + REPLACE phases under the same `spec_id`:
  - matching `(spec_id, position)` rows are UPDATED in place ‚Äî preserving `id` (and `pr` / `merge_sha` unless explicitly overridden)
  - new positions INSERT (with the supplied `pr` / `merge_sha` if any)
  - vanished positions DELETE
  - status is DERIVED at read time (no trigger ‚Äî dropped in `20260725160000`); `specs-status-overrides-only`: an explicit `row.status` is persisted ONLY if it's a true override (`deferred`/`folded`) ‚Äî a derived value OR `in_review` is normalized to `null`. Omitting `row.status` keeps the DB default (`NULL` since migration `20260907130000`); a fresh spec then DERIVES `in_review` (no phases + `vale_review_passed_at IS NULL`), so the build pipeline still holds it.
  - **`folded-spec-must-stay-folded`: `folded` is TERMINAL ‚Äî a re-author NEVER un-folds it.** Before the upsert, `upsertSpec` reads the existing `specs.status`; if it's already `folded`, the override is PRESERVED unless the caller is EXPLICITLY re-folding (`row.status === 'folded'`). This closes the **db-reduce-calls split**: a folded spec's markdown is in `docs/brain/archive.d/`, but a later re-author that passed a derived/absent status would normalize `folded` ‚Üí `null`, the rollup would re-derive `planned`/`in_progress`, and the archived spec would re-appear on the active board while `cancelJobsForArchivedSpecs` auto-cancelled its builds as "spec archived". The ONLY sanctioned un-fold is an explicit `setSpecStatus(slug, null)` by the CEO ‚Äî a re-author is never an implicit un-fold. (Symmetric runtime backstop: [[spec-drift]] `reconcileArchivedNotFolded` re-folds any row that still drifted.)
  - per-`SpecPhaseInput`: PASS `undefined` for `pr` / `merge_sha` / `verification` to PRESERVE the existing value, PASS `null` to CLEAR
  - **`upsertSpec` is SELF-GATING ‚Äî the runtime authoring chokepoint (harden-spec-submission hotfix).** BEFORE any DB write it calls `assertUpsertFullyAuthored` ‚Üí the pure `computeUpsertAuthoringProblems(existing, row, phases)`, and THROWS the exported `UngatedSpecAuthorError` if: any phase's EFFECTIVE `verification` is empty, OR the spec's EFFECTIVE `why`/`what` is empty, OR there are zero phases. This makes "every spec is testable + human-intelligible" a runtime INVARIANT no write path can skip ‚Äî closing the hole where a raw `upsertSpec` call (a Claude session or any of ~17 box-worker lanes) bypassed the [[author-spec]] gates and landed 4 verification-NULL specs on 2026-07-10 that Vale bounced and nobody re-authored. It is REDUNDANT with the [[author-spec]] gates for sanctioned paths (they assert the same fields FIRST) and only bites a raw bypass.
    - **"Effective" = preserve-on-undefined.** A field's gated value is the one the caller PASSES, or (when OMITTED/`undefined` on a preserve-update) the value already STORED on the existing row. So a narrow status-only re-author of an already-gated spec still passes (its stored verification/intent satisfies the gate); only a fresh or content-changing write with empty verification/intent throws.
    - **Requires per-phase `verification` + SPEC-LEVEL `why`/`what` ONLY ‚Äî NOT per-phase intent.** The markdown author path ([[author-spec]] `authorSpecRowFromMarkdown`, used by ~11 box-worker lanes) legitimately leaves phase `why`/`what` NULL; requiring them would halt those lanes. `verification` is what agents TEST; spec `why`/`what` are the human-readable intent on the detail page. (`### Verification` markdown is a RENDER-TIME artifact `renderSpecRow` synthesizes FROM the `verification` column ‚Äî the DB stores the column, not markdown-as-data; that pattern is unchanged.)
    - **`appendFixPhases` does not route through this guard** ‚Äî fix-as-phase is a separate raw INSERT that does NOT go through `upsertSpec`, so `assertUpsertFullyAuthored` never fires on it. (It does its own thing: appends the phase rows AND their typed `spec_phase_checks` rows ‚Äî see its export entry below.)
  - **`computeUpsertAuthoringProblems(existing, row, phases)`** ‚Üí `string[]` ‚Äî the exported PURE predicate `assertUpsertFullyAuthored` throws on (empty = fully authored). Effective-value semantics as above; import it to check authoring completeness without triggering a write.
  - **`UngatedSpecAuthorError`** ‚Äî the exported error class the self-gate throws. Its message lists every empty-verification phase + missing spec-intent field so a caller (or a raw-bypass author) sees exactly what to fill.
- **`movePhase(phaseId, newSpecId, newPosition)`** ‚Äî the **lift-a-phase-between-specs primitive**: a SINGLE `UPDATE spec_phases` that preserves the phase's `id`, `pr`, `merge_sha`, `created_at` ([[../specs/spec-status-phase-pr-provenance]] provenance chain). The trigger fires twice (old + new `spec_id`) and rolls both rollups in one transaction. The unique `(spec_id, position)` index may reject the move if the destination slot is occupied ‚Äî the caller is responsible for shifting positions first when so.
- **`stampPhaseShipped(workspaceId, slug, position, { merge_sha, pr? })`** ‚Äî stamp ONE phase `shipped` with its MAIN-promotion PR provenance (the canonical leaf write that advances the now-DERIVED `specs.status`). The only status-write needed to ship a phase. Under the branch-flow ([[../specs/spec-goal-branch-pm-flow]]) this is the M5 PROMOTION stamp ‚Äî fired when the spec/goal branch lands on main, never from a spec-branch build.
- **`stampPhaseBuilt(workspaceId, slug, position, { build_sha })`** ‚Äî the BRANCH-BUILD counterpart ([[../specs/spec-goal-branch-pm-flow]] M2). Records the `spec_phases.build_sha` (the `claude/build-{slug}` spec-branch commit where the phase built) and moves the phase to `in_progress` (built ‚Äî NOT shipped). NEVER writes `status='shipped'` or `merge_sha` (those stay for the M5 promotion stamp); no-ops on a `shipped`/`rejected` phase (never regresses a promoted one). Called by the box worker the moment a phase commits+pushes on the spec branch (`scripts/builder-worker.ts`); the only writer of `build_sha`. **The `finalizeBuiltPhase` seam invokes this from BRANCH GROUND-TRUTH** ([[../specs/build-accumulation-stamp-gap-and-rollback-guard]] P1): the finalize path scans `origin/main..HEAD` for `Phase: N` trailers (the M1 commit-trailer convention) and stamps EVERY position present on the branch ‚Äî not just the current session's phase. Closes the accumulation-stamp-gap wedge where a phase committed during a `needs_input`/`needs_approval` PAUSE (or whose position couldn't be derived from a resume) never got stamped, so `isSpecAccumulationComplete` permanently reported it "not built" and the PR never opened. Idempotent (terminal phases no-op; re-stamping the same `build_sha` is harmless).
- **`isSpecAccumulationComplete(workspaceId, slug)`** ‚Üí `{ complete, reason }` ([[../specs/spec-goal-branch-pm-flow]] M2/M3) ‚Äî "is this spec FULLY accumulated on its `claude/build-{slug}` branch?" Complete iff every phase carries a `build_sha` (`stampPhaseBuilt`) OR is terminal (shipped/rejected); a 0‚Äì1-phase spec is trivially complete (one-shot ‚Äî nothing to accumulate). Fails OPEN on a read error / missing row (a PM blip mustn't wedge a complete one-off; downstream gates still apply). The SHARED predicate read by THREE seams so they never disagree on "fully built on branch": the [[github-pr-resolve]] auto-merge accumulation gate, the [[agent-jobs]] M3 pre-merge-test trigger (`maybeEnqueuePreMergeSpecTestOnAccumulation`), and [[agent-jobs]] `isSpecPromoteEligible`. (Promoted from a private helper in `github-pr-resolve.ts` to the SDK in M3.)
- **`markPhaseInProgress(workspaceId, slug)`** / **`markRemainingPhasesShipped(workspaceId, slug)`** / **`restampPhases(...)`** ‚Äî the derive-rollup-status phase-advance writers (build-start in_progress flip ¬∑ fold-now "close the rest" ¬∑ audit re-stamp). All move `spec_phases` rows; the readers derive `specs.status` from them.
- **`appendFixPhases(workspaceId, slug, fixPhases[])`** ‚Äî the fix-as-phase writer (a spec-test/regression fix appended as new `spec_phases` rows, NOT routed through `upsertSpec`). It now ALSO writes typed [[../tables/spec_phase_checks]] rows for each appended fix phase ‚Äî derived from the fix's verification blob via [[spec-phase-checks-table]] `parseVerificationBlobToChecks` (best-effort) ‚Äî so a fix phase is a first-class STRUCTURED object with its own check rows, not just a `verification` column string. Keeps a fix phase renderable from the same checks-source-of-truth path [[build-spec-materializer]] uses for authored phases. Invalidates the (ws, slug) cache on success.

PM-write SDK surfaces (the explicit `specs.status` / `blocked_by` / one-shot-provenance overrides ‚Äî added by [[../specs/pm-db-agent-toolkit]] to retire raw `.from("specs").update(...)` from agent code):

- **`setSpecStatus(workspaceId, slug, status, actor)`** ‚Äî write the EXPLICIT lifecycle override on `specs.status` (`deferred` / `folded`), or **`null` to CLEAR the override** so status is PURELY DERIVED from the phase rollup. `specs-status-overrides-only`: a DERIVED value (`planned`/`in_progress`/`shipped`) OR `in_review` (now derived) passed here is normalized to `null` with a warning, so the override-only invariant can't be broken by a stray caller. **To SEND a spec back to review, NULL `vale_review_passed_at` via [[spec-card-state]] `markSpecCardBackToReview`** ‚Äî a status write alone no longer expresses "in review". Slug-resolved single UPDATE, mirrors goals-table `setGoalStatus`. The fold worker calls this with `'folded'`; the disposition lane calls it with `null` (‚Üí derived) / `'deferred'`. The **only** sanctioned `specs.status` writer outside `upsertSpec`. **Parks the `deferred` boolean in lockstep (2026-07-03 fix):** the board's `deriveSpecCardStatus` reads `row.deferred` (the BOOLEAN column), never `status === 'deferred'` ‚Äî so `setSpecStatus` writes `deferred = (status === 'deferred')` in the same UPDATE: set when parking, cleared for every other destination (`folded`/`null` un-park), matching the canonical [[spec-card-state]] `upsertCardState` ‚Üí `dualWriteSpecRow` ‚Üí `specs.deferred` flag path.
- **`setSpecBlockers(workspaceId, slug, blockedBy: string[])`** ‚Äî write `specs.blocked_by` (the spec-blockers gate the build pipeline reads). The milestone-sequence reconciler computes the order-preserving union and persists it here.
- **`setSpecAutoBuild(workspaceId, slug, autoBuild: boolean)`** ‚Äî set the `specs.auto_build` toggle (the init/groom lanes SKIP a spec with `auto_build=false`; the CEO commissioning a build flips it on). The only narrow SDK writer for the column outside a full `upsertSpec` re-author ‚Äî a one-off (re-opening a stuck spec) can flip it without round-tripping the whole body. Slug-resolved single UPDATE, sibling to `setSpecBlockers`/`setSpecStatus`.
- **`setSpecParent(workspaceId, slug, { parent?, parentKind?, parentRef?, milestoneId?, relatedSpec? })`** ‚Äî re-parent a spec in ONE UPDATE: the free-text `parent` prose + the typed `parent_kind`/`parent_ref` pair + the `milestone_id` FK + the `related_spec` LINK (no-spec-parent), without round-tripping the body through `upsertSpec` (which replaces phases). **one-off-spec-parent:** a one-off spec's home is a function MANDATE (`parentKind:"mandate"`, `parentRef:"{owner}#{mandate-slug}"`, `milestoneId:null`); a goal-bound spec's is a specific MILESTONE (`parentKind:"milestone"`, `milestoneId:"<goal_milestones.id>"`). Pass the `parent` prose so it names the same mandate/milestone the typed pair points at (Vale reads the prose). Undefined fields untouched. Slug-resolved single UPDATE, sibling to `setSpecBlockers`/`setSpecStatus`/`setSpecAutoBuild`.
- **`stampSpecMergeProvenance(workspaceId, slug, { pr, merge_sha })`** ‚Äî write a **one-shot spec's** card-level `merged_pr` + `last_merge_sha` (the zero-phase merge-webhook path in [[agent-jobs]]). It does NOT touch `specs.status`; this provenance IS what the deriver reads to surface a zero-phase spec as `shipped` (`deriveSpecCardStatus`: `merged_pr`/`last_merge_sha` set ‚áí `shipped`, else `planned`) ‚Äî so the merge stamp is the only write needed to ship a one-shot spec.
- **`stampSpecGoalBranchSha(workspaceId, slug, goalBranchSha | null)`** ‚Äî the **"on the goal branch" marker writer** ([[../specs/spec-goal-branch-pm-flow]] M4). Records `specs.goal_branch_sha` (the `goal/{goal-slug}` merge commit where this spec's `claude/build-{slug}` branch merged onto its goal branch), set by [[agent-jobs]] `promoteEligibleSpecsToGoalBranch` on a successful merge. The only writer of `goal_branch_sha`. Writes NEITHER status NOR merge_sha ‚Äî on-the-goal-branch is NOT shipped; M5 flips the build_sha'd phases to shipped on the atomic goal‚Üímain promotion. Pass `null` to clear.
- **`isSpecOnGoalBranch(workspaceId, slug)`** ‚Üí `boolean` ([[../specs/spec-goal-branch-pm-flow]] M4) ‚Äî "is this spec's branch merged onto its goal branch?" (i.e. `goal_branch_sha` set). The claim-time blocked_by gate reads it to clear a GOAL-MATE blocker (a goal-mate never ships to main until M5's atomic promotion, so "on the goal branch" ‚Äî not "shipped" ‚Äî is its clearance signal). Fails CLOSED on a read error (unknown goal-branch state must not release a dependent).
- **`goalBranchState(workspaceId, goalSlug)`** ‚Üí `{ goalSlug, specs[], allOnGoalBranch }` ([[../specs/spec-goal-branch-pm-flow]] M4) ‚Äî the whole-goal goal-branch state (every spec linked to the goal's milestones, each tagged `onGoalBranch`). `allOnGoalBranch` (‚â•1 spec AND every spec stamped) is the **seam M5 consumes** to detect goal-complete: [[agent-jobs]] `promoteCompleteGoalsToMain` reads it (gate 2), then on the atomic goal‚Üímain merge `applyGoalPromotionEffects` iterates `state.specs` to flip every member phase shipped (build_sha‚Üíshipped). Read-only; resolves goal‚Üímilestones‚Üíspecs through [[goals-table]] `getGoal` + `specsForMilestone`.
- **`listSpecPhaseAnomalies(workspaceId)`** ‚Üí `{ orphans, provenanceGaps }` ‚Äî the integrity-scan reader: (a) ORPHAN `spec_phases` rows whose parent `specs` row is missing (global ‚Äî no parent to read a workspace from), and (b) PROVENANCE-GAP shipped phases with both `pr` + `merge_sha` null (workspace-scoped, folded-excluded). Resolves `spec_id ‚Üí {slug, workspace}` internally so callers (the [[spec-drift]] reconciler's anomaly sweep) never touch raw PM tables.

All writers route through `createAdminClient()` (service-role; the RLS policy `specs_service` / `spec_phases_service` grants full access). No client-side writes.

**Raw PM-table writes are CI-forbidden outside this SDK.** `scripts/_check-pm-sdk-compliance.ts` (chained into `predeploy`) fails the build on any `.from('specs'|'spec_phases'|'goals'|'goal_milestones').update()/.insert()/.upsert()/.delete()` in `scripts/builder-worker.ts` or `src/lib/**` that isn't in this SDK (or `goals-table.ts`) and isn't on its `SANCTIONED_RAW_WRITES` allow-list. One sanctioned exception: [[spec-card-state]] `dualWriteSpecRow` (the legacy mirror dual-write, broad column set).

## Not atomic across parent + children

supabase-js has no transaction surface, so `upsertSpec` is a sequence of writes (UPSERT specs, then DELETE / UPDATE / INSERT phases). The trigger keeps `specs.status` consistent after each write, and re-running the same call is idempotent (position-keyed REPLACE is deterministic). Callers requiring true atomicity must compose at the SQL layer.

## Callers

- **[[../recipes/backfill-specs-from-markdown]]** (`scripts/backfill-specs-from-markdown.ts`) ‚Äî runs [[brain-roadmap]] `parseSpec` ONE LAST TIME over `docs/brain/specs/*.md` and upserts the rows.
- **[[author-spec]]** (`src/lib/author-spec.ts`) ‚Äî the dual-write chokepoint every spec-author surface routes through ([[../specs/spec-authoring-writes-db-and-worker-materialize]] Phase 1). Parses the just-committed markdown body and calls `upsertSpec`, so the DB row stays in step with the `.md` commit until [[../specs/spec-readers-from-db-retire-parser]] (M3) cuts readers over.
- **[[build-spec-materializer]]** ([[../specs/spec-authoring-writes-db-and-worker-materialize]] Phase 2; adopted by [[../specs/spec-fold-from-db-row]] Phase 1) reads `getSpec` to render a temp `.box/spec-{slug}.md` for the [[../skills/build-spec]] skill (and [[../skills/fold-to-brain]] during folding) ‚Äî Bo never reads the on-disk spec body once a `public.specs` row exists, and fold-to-brain never reads `docs/brain/specs/{slug}.md`.
- **[[spec-card-state]] `upsertCardState`** ([[../specs/spec-authoring-writes-db-and-worker-materialize]] Phase 3) dual-writes the corresponding typed columns on this table (status, deferred, priority, intended_status) on every mirror flip, so the future-canonical row stays in sync with the mirror.
- **[[agent-jobs]] `applyMergedBuildEffects`** ([[../specs/spec-authoring-writes-db-and-worker-materialize]] Phase 2) double-writes per-phase PR provenance to [[../tables/spec_phases]] (`UPDATE spec_phases SET pr=‚Ä¶, merge_sha=‚Ä¶ WHERE spec_id=‚Ä¶ AND position=‚Ä¶`) alongside `spec_card_state.phase_states[i]` ‚Äî the typed phase row is the canonical phase-PR provenance surface.
- **Build queue gates** ([[../specs/planner-gates-build-queue-on-authored-specs]]) ‚Äî `runPlanJob` and the foreman dispatch gates both call `getSpec(workspaceId, slug)` to **verify a spec row actually exists** in `public.specs` before queuing a build. A missing row ‚Üí the job is parked `needs_attention` with `needs_attention_class='spec_row_missing'` (the author lane silently failed upstream). This prevents phantom builds when an authored spec never reached the DB.
- **Fold process** ([[../specs/spec-fold-from-db-row]] Phase 1) ‚Äî the box worker's `runFoldJob` calls `getSpec` with a `status='shipped'` guard to fetch shipped specs, then materializes each into `.box/spec-{slug}.md` for the fold-agent to read. After fold commits succeed, the worker updates the row to `status='folded'`. The spec row is PRESERVED (not deleted) so archive views + audit history can render it unchanged.

## Invariant ‚Äî no client-side id/slug array batching ([[../specs/retire-residual-in-array-batching-to-server-side-rpcs]] Phase 3)

**All specs/phase reads on this file go through server-side RPCs ‚Äî NEVER a client-side id/slug array over the wire.** The 2026-07-08 DB-overload incident and its precedent ([[../specs/list-specs-with-phases-rpc-retire-in-array-client-join]]) traced to `.in("id", [large-array])` calls whose URL overflowed the ~16KB undici header cap (`UND_ERR_HEADERS_OVERFLOW`) once the workspace held a few hundred specs, wedging `getSpec` / the roadmap / the build claim-gate / the spec-review enqueue reaper. The three sanctioned reads on this file:

- `listSpecs` / `getActiveSpecs` / `getAllSpecs` ‚Üí **`list_specs_with_phases(uuid, text)`** RPC (server-side specs+phases join).
- `listSpecPhaseAnomalies` ‚Üí **`list_spec_phase_anomalies(uuid)`** RPC (server-side orphan + provenance-gap sweep via a single `spec_phases LEFT JOIN specs`).
- `getSpec` ‚Üí **`get_spec_with_phases(uuid, text)`** RPC (server-side single-spec join; [[../specs/cut-internal-egress-pooler-and-spec-rpcs]] Phase 2) ‚Äî collapses the pre-Phase-2 two-round-trip `.from()` reads into one call, no id array by construction. Pooled through [[pg-pool]] on the box.

**CI-enforced** by `scripts/_check-specs-phases-no-client-in-batching.ts` (chained into `predeploy`): a reintroduced `.in("id", <array>)` or `.in("spec_slug", <array>)` in `src/lib/specs-table.ts` or `src/lib/brain-roadmap.ts` fails the build. Genuinely-bounded LITERAL arrays (`.in("id", ["a","b"])`) still pass ‚Äî a variable, `.slice(...)` batch, or spread does not.

## `getSpec` ‚Äî 2-second in-process cache ([[../specs/db-reduce-calls-q-1756037457588317045]])

`getSpec(workspaceId, slug)` reads through a module-level TTL cache before hitting the RPC. The `public.get_spec_with_phases(uuid, text)` query topped the box's `pg_stat_statements` sample at 215,257 calls (0ms mean, 31s total DB time) ‚Äî per-call cost was already fine (index scan on `specs_ws_slug`); the fix is FEWER calls, not a cheaper one. Common code paths that were duplicate-reading the same (ws, slug) inside one request:

- `queueRoadmapBuild` (`src/lib/roadmap-actions.ts`) ‚Äî a `getSpecRow` at the top plus two `brain-roadmap.getSpec` reads further down, each internally re-calling this SDK `getSpec` on the same slug.
- `brain-roadmap.getSpec(slug, wsId)` ‚Äî every wrapper call bounces to this SDK `getSpec` under the hood.
- The box worker's tick handlers ‚Äî nearby jobs on the same spec re-read it inside a burst.

Design invariants ([[../specs/db-reduce-calls-q-1756037457588317045]]):

- **TTL is 2 seconds.** Bounds staleness to under one poll tick even if a mutation happens through a path this SDK cannot see (a raw SQL migration, an admin script outside this module).
- **Every writer in this file invalidates the cached (ws, slug) on success** ‚Äî `upsertSpec` / `stampPhaseShipped` / `stampPhaseBuilt` / `markRemainingPhasesShipped` / `restampPhases` / `appendFixPhases` / `setSpecStatus` / `setSpecBlockers` / `stampSpecValeReviewPassed` / `setSpecParent` / `setSpecAutoBuild` / `stampSpecMergeProvenance` / `stampSpecGoalBranchSha` all call `invalidateSpecCache(workspaceId, slug)` after their DB write. The read-after-write pattern in [[author-spec]] (`getSpec` ‚Üí `upsertSpec` ‚Üí `getSpec` to verify persistence) stays correct because `upsertSpec`'s invalidation forces the second read to hit the RPC. `movePhase` alone relies on the TTL (its signature is keyed by phase id, not slug, so it can't invalidate cheaply ‚Äî the 2s bound is enough for a low-volume primitive).
- **Null results are cached** so a nonexistent slug in a tight retry loop doesn't hammer the RPC either. Same invalidation rules apply.
- **`clearSpecCacheForTests()`** is exported for tests that share process state.

## Gotchas

- **The trigger is the rail.** Don't write `specs.status='shipped'` directly while a phase is still `planned` ‚Äî the next phase write will roll it back. That's the DB-enforced rule that kills the [[../specs/spec-review-agent]] "shipped with 1 phase" class.
- **`in_review` / `folded` are not auto-cleared.** The rollup early-returns on those; an explicit status flip ([[spec-card-state]] `applyAdaDisposition`, the fold worker) is required.
- **`upsertSpec` preserves provenance.** Existing `(spec_id, position)` rows keep their `id` + `pr` + `merge_sha` unless the caller passes new values. The backfill relies on this so it never destroys a per-phase PR tag while resyncing markdown.
- **`movePhase` requires a free destination slot.** The `(spec_id, position)` unique constraint will reject a move onto an occupied position; callers either renumber siblings first or pop the destination's existing phase elsewhere.
- **`Phase`'s `rejected` is NOT a `specs.status` value.** It's a phase-level state. A whole-spec rolled-up status never goes to `rejected`.
- **`SpecRowInput.human_review` ‚Äî OPTIONAL, non-blocking ([[../specs/every-spec-writer-authors-machine-runnable-verifications]] Phase 2).** Additive `alter table public.specs add column if not exists human_review text` (migration 20261014120000). `upsertSpec` persists it via the preserve-on-undefined rule (PASS `null` to CLEAR, OMIT to PRESERVE). NEVER read by the fold gate / promote gate / deterministic spec-check runner ‚Äî machine-runnable `spec_phase_checks` remain the sole ship gate. `SPEC_COLUMNS` DELIBERATELY OMITS this column (the RPC readers use `to_jsonb(s)` which returns it automatically; only the explicit-column `getFixSpecForOrigin` uses the list and never reads `human_review`) so the SDK stays resilient to an unapplied migration ‚Äî a legacy DB missing the column reads the field as `null` via `specRowFromDb`'s `db.human_review ?? null` coalesce. Pinned by a unit test in `src/lib/author-spec.test.ts` ‚Äî "SPEC_COLUMNS explicit selector deliberately OMITS human_review". Rendered on the spec card by [[brain-roadmap]] `serializeSpecRowToMarkdown` as a `**Human-review:**` header.
