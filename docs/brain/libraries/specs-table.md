# libraries/specs-table

The read/write surface for the DB-resident spec body тАФ [[../tables/specs]] (the card) + [[../tables/spec_phases]] (the per-phase rows). Authored by [[../specs/spec-body-table-and-backfill]] Phase 2; parallel to [[spec-card-state]] (status-only mirror) until [[../specs/spec-readers-from-db-retire-parser]] (M3) retires the markdown parser.

**File:** `src/lib/specs-table.ts`

## Why this exists

[[../specs/spec-body-table-and-backfill]] adds the relations [[../tables/specs]] + [[../tables/spec_phases]] so the spec body is queryable + indexable in the DB (and so a phase can move between specs preserving its stable id + PR/SHA provenance тАФ see below). This module is the canonical writer + read surface those rows are managed through. NO reader has been retargeted yet тАФ `getRoadmap` / `getSpec` / the board still read markdown via [[brain-roadmap]] in this milestone; the [[../recipes/backfill-specs-from-markdown]] one-time backfill seeds the rows so M2/M3 can lean on them.

## Types

- **`SpecStatus`** = `'in_review' я╜Ь 'planned' я╜Ь 'in_progress' я╜Ь 'shipped' я╜Ь 'deferred' я╜Ь 'folded'` тАФ the `specs.status` enum (CHECK-constrained in the migration). Includes `in_review` ([[../specs/spec-review-agent]]) and `folded` ([[../goals/db-driven-specs]] M4). The DB trigger keeps it consistent with the child [[../tables/spec_phases]] rows (terminal-ish `in_review` / `folded` excepted).
- **`Phase`** тАФ re-exported from [[brain-roadmap]] so callers don't churn imports. `'planned' я╜Ь 'in_progress' я╜Ь 'shipped' я╜Ь 'rejected'`.
- **`SpecRow`** тАФ `{ id, workspace_id, slug, title, summary, owner, parent, blocked_by, priority, deferred, intended_status, status, intended_status_set_by, repair_signature, regression_of_slug, regression_signature, auto_build, vale_pass, vale_review_passed_at, ada_disposition, milestone_id, merged_pr, last_merge_sha, created_at, updated_at, phases: SpecPhaseRow[] }` тАФ the parent + joined ordered phases. `vale_review_passed_at` (timestamptz) is the DURABLE "passed Vale review" stamp the claim-time build gate reads (build-gate-durable-review-signal) тАФ set on a Vale PASS alongside the transient `vale_pass`, but NOT consumed by Ada's disposition. `regression_of_slug` / `regression_signature` mirror the regression-agent header lines (`**Regression-of:** [[<slug>]]` / `**Regression-signature:** `<sig>``) тАФ typed columns added in [[../specs/spec-authoring-writes-db-and-worker-materialize]] Phase 1, sibling to `repair_signature`. `merged_pr` (int) / `last_merge_sha` (text) carry a **one-shot spec's** card-level shipping PR + SHA ([[../specs/spec-status-phase-pr-provenance]]) тАФ a zero-phase spec has no `spec_phases` slot to hold provenance, so it lands on the parent row; multi-phase specs use `spec_phases.pr` instead and leave these null.
- **`SpecPhaseRow`** тАФ `{ id, spec_id, position, title, body, status, pr, merge_sha, verification, created_at, updated_at }`. `position` is 1-indexed.
- **`SpecRowInput`** / **`SpecPhaseInput`** тАФ the writable field sets `upsertSpec` accepts.

## Exports

- **`getSpec(workspaceId, slug)`** тЖТ `SpecRow | null` тАФ the parent row + its [[../tables/spec_phases]] rows ordered by `position`. Read by authenticated users (RLS) тАФ service role bypasses for backfill / writers.
- **`listSpecs(workspaceId, filter?)`** тЖТ `SpecRow[]` тАФ every spec in a workspace, optionally filtered by `{ status, owner, milestone_id }`. Phases joined in one extra round-trip.
- **`getFixSpecForOrigin(workspaceId, originSlug)`** тЖТ `SpecRow | null` тАФ resolve the regression FIX spec for an origin by the typed linkage `specs.regression_of_slug = originSlug` (NOT a hand-typed/deterministic slug), most-recently-created first, phases joined like `getSpec`. The inline spec-test card ([[../specs/spec-test-request-fix-inline-author-and-approve]] Phase 2) reads it so a renamed fix slug still surfaces тАФ the `regression_of_slug` column (set by the request-fix authoring path + the regression-agent's `**Regression-of:**` header) is the source of truth. Read-only, service-role.
- **`upsertSpec(workspaceId, row, phases)`** тЖТ `{ spec_id, phase_ids }` тАФ the canonical write. UPSERT by `(workspace_id, slug)` + REPLACE phases under the same `spec_id`:
  - matching `(spec_id, position)` rows are UPDATED in place тАФ preserving `id` (and `pr` / `merge_sha` unless explicitly overridden)
  - new positions INSERT (with the supplied `pr` / `merge_sha` if any)
  - vanished positions DELETE
  - the trigger rolls `specs.status` after each phase write (terminal-ish `in_review` / `folded` are left alone)
  - per-`SpecPhaseInput`: PASS `undefined` for `pr` / `merge_sha` / `verification` to PRESERVE the existing value, PASS `null` to CLEAR
- **`movePhase(phaseId, newSpecId, newPosition)`** тАФ the **lift-a-phase-between-specs primitive**: a SINGLE `UPDATE spec_phases` that preserves the phase's `id`, `pr`, `merge_sha`, `created_at` ([[../specs/spec-status-phase-pr-provenance]] provenance chain). The trigger fires twice (old + new `spec_id`) and rolls both rollups in one transaction. The unique `(spec_id, position)` index may reject the move if the destination slot is occupied тАФ the caller is responsible for shifting positions first when so.
- **`stampPhaseShipped(workspaceId, slug, position, { merge_sha, pr? })`** тАФ stamp ONE phase `shipped` with its MAIN-promotion PR provenance (the canonical leaf write that advances the now-DERIVED `specs.status`). The only status-write needed to ship a phase. Under the branch-flow ([[../specs/spec-goal-branch-pm-flow]]) this is the M5 PROMOTION stamp тАФ fired when the spec/goal branch lands on main, never from a spec-branch build.
- **`stampPhaseBuilt(workspaceId, slug, position, { build_sha })`** тАФ the BRANCH-BUILD counterpart ([[../specs/spec-goal-branch-pm-flow]] M2). Records the `spec_phases.build_sha` (the `claude/build-{slug}` spec-branch commit where the phase built) and moves the phase to `in_progress` (built тАФ NOT shipped). NEVER writes `status='shipped'` or `merge_sha` (those stay for the M5 promotion stamp); no-ops on a `shipped`/`rejected` phase (never regresses a promoted one). Called by the box worker the moment a phase commits+pushes on the spec branch (`scripts/builder-worker.ts`); the only writer of `build_sha`.
- **`isSpecAccumulationComplete(workspaceId, slug)`** тЖТ `{ complete, reason }` ([[../specs/spec-goal-branch-pm-flow]] M2/M3) тАФ "is this spec FULLY accumulated on its `claude/build-{slug}` branch?" Complete iff every phase carries a `build_sha` (`stampPhaseBuilt`) OR is terminal (shipped/rejected); a 0тАУ1-phase spec is trivially complete (one-shot тАФ nothing to accumulate). Fails OPEN on a read error / missing row (a PM blip mustn't wedge a complete one-off; downstream gates still apply). The SHARED predicate read by THREE seams so they never disagree on "fully built on branch": the [[github-pr-resolve]] auto-merge accumulation gate, the [[agent-jobs]] M3 pre-merge-test trigger (`maybeEnqueuePreMergeSpecTestOnAccumulation`), and [[agent-jobs]] `isSpecPromoteEligible`. (Promoted from a private helper in `github-pr-resolve.ts` to the SDK in M3.)
- **`markPhaseInProgress(workspaceId, slug)`** / **`markRemainingPhasesShipped(workspaceId, slug)`** / **`restampPhases(...)`** тАФ the derive-rollup-status phase-advance writers (build-start in_progress flip ┬╖ fold-now "close the rest" ┬╖ audit re-stamp). All move `spec_phases` rows; the readers derive `specs.status` from them.

PM-write SDK surfaces (the explicit `specs.status` / `blocked_by` / one-shot-provenance overrides тАФ added by [[../specs/pm-db-agent-toolkit]] to retire raw `.from("specs").update(...)` from agent code):

- **`setSpecStatus(workspaceId, slug, status, actor)`** тАФ write the EXPLICIT lifecycle override on `specs.status` (`shipped` / `folded` / `in_review` / `deferred`) тАФ the states NOT derivable from the phase rollup. Slug-resolved single UPDATE, mirrors goals-table `setGoalStatus`. The fold worker calls this with `'folded'`; the **only** sanctioned `specs.status` writer outside `upsertSpec`.
- **`setSpecBlockers(workspaceId, slug, blockedBy: string[])`** тАФ write `specs.blocked_by` (the spec-blockers gate the build pipeline reads). The milestone-sequence reconciler computes the order-preserving union and persists it here.
- **`stampSpecMergeProvenance(workspaceId, slug, { pr, merge_sha })`** тАФ write a **one-shot spec's** card-level `merged_pr` + `last_merge_sha` (the zero-phase merge-webhook path in [[agent-jobs]]).
- **`stampSpecGoalBranchSha(workspaceId, slug, goalBranchSha | null)`** тАФ the **"on the goal branch" marker writer** ([[../specs/spec-goal-branch-pm-flow]] M4). Records `specs.goal_branch_sha` (the `goal/{goal-slug}` merge commit where this spec's `claude/build-{slug}` branch merged onto its goal branch), set by [[agent-jobs]] `promoteEligibleSpecsToGoalBranch` on a successful merge. The only writer of `goal_branch_sha`. Writes NEITHER status NOR merge_sha тАФ on-the-goal-branch is NOT shipped; M5 flips the build_sha'd phases to shipped on the atomic goalтЖТmain promotion. Pass `null` to clear.
- **`isSpecOnGoalBranch(workspaceId, slug)`** тЖТ `boolean` ([[../specs/spec-goal-branch-pm-flow]] M4) тАФ "is this spec's branch merged onto its goal branch?" (i.e. `goal_branch_sha` set). The claim-time blocked_by gate reads it to clear a GOAL-MATE blocker (a goal-mate never ships to main until M5's atomic promotion, so "on the goal branch" тАФ not "shipped" тАФ is its clearance signal). Fails CLOSED on a read error (unknown goal-branch state must not release a dependent).
- **`goalBranchState(workspaceId, goalSlug)`** тЖТ `{ goalSlug, specs[], allOnGoalBranch }` ([[../specs/spec-goal-branch-pm-flow]] M4) тАФ the whole-goal goal-branch state (every spec linked to the goal's milestones, each tagged `onGoalBranch`). `allOnGoalBranch` (тЙе1 spec AND every spec stamped) is the **seam M5 consumes** to detect goal-complete: [[agent-jobs]] `promoteCompleteGoalsToMain` reads it (gate 2), then on the atomic goalтЖТmain merge `applyGoalPromotionEffects` iterates `state.specs` to flip every member phase shipped (build_shaтЖТshipped). Read-only; resolves goalтЖТmilestonesтЖТspecs through [[goals-table]] `getGoal` + `specsForMilestone`.
- **`listSpecPhaseAnomalies(workspaceId)`** тЖТ `{ orphans, provenanceGaps }` тАФ the integrity-scan reader: (a) ORPHAN `spec_phases` rows whose parent `specs` row is missing (global тАФ no parent to read a workspace from), and (b) PROVENANCE-GAP shipped phases with both `pr` + `merge_sha` null (workspace-scoped, folded-excluded). Resolves `spec_id тЖТ {slug, workspace}` internally so callers (the [[spec-drift]] reconciler's anomaly sweep) never touch raw PM tables.

All writers route through `createAdminClient()` (service-role; the RLS policy `specs_service` / `spec_phases_service` grants full access). No client-side writes.

**Raw PM-table writes are CI-forbidden outside this SDK.** `scripts/_check-pm-sdk-compliance.ts` (chained into `predeploy`) fails the build on any `.from('specs'|'spec_phases'|'goals'|'goal_milestones').update()/.insert()/.upsert()/.delete()` in `scripts/builder-worker.ts` or `src/lib/**` that isn't in this SDK (or `goals-table.ts`) and isn't on its `SANCTIONED_RAW_WRITES` allow-list. One sanctioned exception: [[spec-card-state]] `dualWriteSpecRow` (the legacy mirror dual-write, broad column set).

## Not atomic across parent + children

supabase-js has no transaction surface, so `upsertSpec` is a sequence of writes (UPSERT specs, then DELETE / UPDATE / INSERT phases). The trigger keeps `specs.status` consistent after each write, and re-running the same call is idempotent (position-keyed REPLACE is deterministic). Callers requiring true atomicity must compose at the SQL layer.

## Callers

- **[[../recipes/backfill-specs-from-markdown]]** (`scripts/backfill-specs-from-markdown.ts`) тАФ runs [[brain-roadmap]] `parseSpec` ONE LAST TIME over `docs/brain/specs/*.md` and upserts the rows.
- **[[author-spec]]** (`src/lib/author-spec.ts`) тАФ the dual-write chokepoint every spec-author surface routes through ([[../specs/spec-authoring-writes-db-and-worker-materialize]] Phase 1). Parses the just-committed markdown body and calls `upsertSpec`, so the DB row stays in step with the `.md` commit until [[../specs/spec-readers-from-db-retire-parser]] (M3) cuts readers over.
- **[[build-spec-materializer]]** ([[../specs/spec-authoring-writes-db-and-worker-materialize]] Phase 2; adopted by [[../specs/spec-fold-from-db-row]] Phase 1) reads `getSpec` to render a temp `.box/spec-{slug}.md` for the [[../skills/build-spec]] skill (and [[../skills/fold-to-brain]] during folding) тАФ Bo never reads the on-disk spec body once a `public.specs` row exists, and fold-to-brain never reads `docs/brain/specs/{slug}.md`.
- **[[spec-card-state]] `upsertCardState`** ([[../specs/spec-authoring-writes-db-and-worker-materialize]] Phase 3) dual-writes the corresponding typed columns on this table (status, deferred, priority, intended_status) on every mirror flip, so the future-canonical row stays in sync with the mirror.
- **[[agent-jobs]] `applyMergedBuildEffects`** ([[../specs/spec-authoring-writes-db-and-worker-materialize]] Phase 2) double-writes per-phase PR provenance to [[../tables/spec_phases]] (`UPDATE spec_phases SET pr=тАж, merge_sha=тАж WHERE spec_id=тАж AND position=тАж`) alongside `spec_card_state.phase_states[i]` тАФ the typed phase row is the canonical phase-PR provenance surface.
- **Build queue gates** ([[../specs/planner-gates-build-queue-on-authored-specs]]) тАФ `runPlanJob` and the foreman dispatch gates both call `getSpec(workspaceId, slug)` to **verify a spec row actually exists** in `public.specs` before queuing a build. A missing row тЖТ the job is parked `needs_attention` with `needs_attention_class='spec_row_missing'` (the author lane silently failed upstream). This prevents phantom builds when an authored spec never reached the DB.
- **Fold process** ([[../specs/spec-fold-from-db-row]] Phase 1) тАФ the box worker's `runFoldJob` calls `getSpec` with a `status='shipped'` guard to fetch shipped specs, then materializes each into `.box/spec-{slug}.md` for the fold-agent to read. After fold commits succeed, the worker updates the row to `status='folded'`. The spec row is PRESERVED (not deleted) so archive views + audit history can render it unchanged.

## Gotchas

- **The trigger is the rail.** Don't write `specs.status='shipped'` directly while a phase is still `planned` тАФ the next phase write will roll it back. That's the DB-enforced rule that kills the [[../specs/spec-review-agent]] "shipped with 1 phase" class.
- **`in_review` / `folded` are not auto-cleared.** The rollup early-returns on those; an explicit status flip ([[spec-card-state]] `applyAdaDisposition`, the fold worker) is required.
- **`upsertSpec` preserves provenance.** Existing `(spec_id, position)` rows keep their `id` + `pr` + `merge_sha` unless the caller passes new values. The backfill relies on this so it never destroys a per-phase PR tag while resyncing markdown.
- **`movePhase` requires a free destination slot.** The `(spec_id, position)` unique constraint will reject a move onto an occupied position; callers either renumber siblings first or pop the destination's existing phase elsewhere.
- **`Phase`'s `rejected` is NOT a `specs.status` value.** It's a phase-level state. A whole-spec rolled-up status never goes to `rejected`.
