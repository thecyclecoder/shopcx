# libraries/specs-table

The read/write surface for the DB-resident spec body â€” [[../tables/specs]] (the card) + [[../tables/spec_phases]] (the per-phase rows). Authored by [[../specs/spec-body-table-and-backfill]] Phase 2; parallel to [[spec-card-state]] (status-only mirror) until [[../specs/spec-readers-from-db-retire-parser]] (M3) retires the markdown parser.

**File:** `src/lib/specs-table.ts`

## Why this exists

[[../specs/spec-body-table-and-backfill]] adds the relations [[../tables/specs]] + [[../tables/spec_phases]] so the spec body is queryable + indexable in the DB (and so a phase can move between specs preserving its stable id + PR/SHA provenance â€” see below). This module is the canonical writer + read surface those rows are managed through. NO reader has been retargeted yet â€” `getRoadmap` / `getSpec` / the board still read markdown via [[brain-roadmap]] in this milestone; the [[../recipes/backfill-specs-from-markdown]] one-time backfill seeds the rows so M2/M3 can lean on them.

## Types

- **`SpecStatus`** = `'in_review' ï½œ 'planned' ï½œ 'in_progress' ï½œ 'shipped' ï½œ 'deferred' ï½œ 'folded'` â€” the caller-facing enum (the type callers may PASS). `SpecRow.status` is typed `SpecStatus | null`. **`specs-status-overrides-only`** (migration `20260907130000`): the stored `specs.status` column carries ONLY `deferred` / `folded` / `NULL` â€” the CHECK REJECTS everything else. `in_review` is no longer stored; it is DERIVED at read time from the phase rollup + `vale_review_passed_at` ([[brain-roadmap]] `deriveSpecCardStatus`), alongside the planned/in_progress/shipped axis (no trigger â€” dropped in `20260725160000`). Every writer clears a non-`{deferred,folded}` destination to `NULL` (the [[spec-card-state]] `isOverrideStatus` gate) â€” so a caller may still PASS `in_review`/derived values, but they land as `NULL`. A zero-phase one-shot spec derives `shipped` from its `merged_pr`/`last_merge_sha` provenance, never from the stored column.
- **`Phase`** â€” re-exported from [[brain-roadmap]] so callers don't churn imports. `'planned' ï½œ 'in_progress' ï½œ 'shipped' ï½œ 'rejected'`.
- **`SpecRow`** â€” `{ id, workspace_id, slug, title, summary, owner, parent, blocked_by, priority, deferred, intended_status, status, intended_status_set_by, repair_signature, regression_of_slug, regression_signature, auto_build, vale_pass, vale_review_passed_at, ada_disposition, vale_disposition, vale_disposition_reason, milestone_id, merged_pr, last_merge_sha, created_at, updated_at, phases: SpecPhaseRow[] }` â€” the parent + joined ordered phases. `vale_review_passed_at` (timestamptz) is the DURABLE "passed Vale review" stamp the claim-time build gate reads (build-gate-durable-review-signal) â€” set on a Vale PASS alongside the transient `vale_pass`, but NOT consumed by Ada's disposition. **`vale_disposition`** (`'planned'`/`'deferred'`/null) + **`vale_disposition_reason`** (text/null) carry Vale's reasoned recommendation on a PASS ([[../specs/vale-reasons-the-disposition]] Phase 1); Ada's Phase-2 sweep consumes them (retiring the trust-the-author stub). `regression_of_slug` / `regression_signature` mirror the regression-agent header lines (`**Regression-of:** [[<slug>]]` / `**Regression-signature:** `<sig>``) â€” typed columns added in [[../specs/spec-authoring-writes-db-and-worker-materialize]] Phase 1, sibling to `repair_signature`. `merged_pr` (int) / `last_merge_sha` (text) carry a **one-shot spec's** card-level shipping PR + SHA ([[../specs/spec-status-phase-pr-provenance]]) â€” a zero-phase spec has no `spec_phases` slot to hold provenance, so it lands on the parent row; multi-phase specs use `spec_phases.pr` instead and leave these null.
- **`SpecPhaseRow`** â€” `{ id, spec_id, position, title, body, status, pr, merge_sha, verification, created_at, updated_at }`. `position` is 1-indexed.
- **`SpecRowInput`** / **`SpecPhaseInput`** â€” the writable field sets `upsertSpec` accepts.

## Exports

- **`getSpec(workspaceId, slug)`** â†’ `SpecRow | null` â€” the parent row + its [[../tables/spec_phases]] rows ordered by `position`. Read by authenticated users (RLS) â€” service role bypasses for backfill / writers.
- **`listSpecs(workspaceId, filter?)`** â†’ `SpecRow[]` â€” every spec in a workspace, optionally filtered by `{ status, owner, milestone_id, scope }`. Sourced from the **`public.list_specs_with_phases(uuid, text)`** Postgres RPC (migration `20261001120000_list_specs_with_phases_rpc.sql`, [[../specs/list-specs-with-phases-rpc-retire-in-array-client-join]] Phase 1): the RPC does the `specs` + `spec_phases` join **server-side** and streams `(spec jsonb, phases jsonb)` rows back, so no id array crosses the wire. Retired the interim `.in("spec_id", [ids])` slug-batching â€” the DURABLE fix for the `UND_ERR_HEADERS_OVERFLOW` (~16KB undici header cap) that wedged getSpec + the build claim-gate + the spec-review enqueue reaper once the workspace held a few hundred specs. `filter.scope âˆˆ 'active' | 'archived' | 'all'` picks the server-side row set (`'active'` = `status IS NULL OR status <> 'folded'`, `'archived'` = `folded`, `'all'` = every row); defaults to `'all'` so folded-inclusive callers ([[../libraries/director-kpis]] owner attribution, [[spec-drift]], [[../libraries/agents/spec-dispose]] audits) keep pre-RPC semantics. `status` / `owner` / `milestone_id` are applied client-side on the bounded RPC result set.
- **`getActiveSpecs(workspaceId)`** / **`getAllSpecs(workspaceId)`** â†’ `SpecRow[]` ([[../specs/list-specs-with-phases-rpc-retire-in-array-client-join]] Phase 2) â€” thin scope wrappers over `listSpecs`. `getActiveSpecs` calls the RPC with `scope='active'` (folded rows filtered SERVER-SIDE) â€” prefer this on board / pipeline / claim-gate readers that never want folded rows across the wire. `getAllSpecs` calls with `scope='all'` â€” the folded-inclusive set the pre-RPC `listSpecs()` returned; use it on readers that need the full set (owner attribution, drift audits, dispose sweeps).
- **`getFixSpecForOrigin(workspaceId, originSlug)`** â†’ `SpecRow | null` â€” resolve the regression FIX spec for an origin by the typed linkage `specs.regression_of_slug = originSlug` (NOT a hand-typed/deterministic slug), most-recently-created first, phases joined like `getSpec`. The inline spec-test card ([[../specs/spec-test-request-fix-inline-author-and-approve]] Phase 2) reads it so a renamed fix slug still surfaces â€” the `regression_of_slug` column (set by the request-fix authoring path + the regression-agent's `**Regression-of:**` header) is the source of truth. Read-only, service-role.
- **`upsertSpec(workspaceId, row, phases)`** â†’ `{ spec_id, phase_ids }` â€” the canonical write. UPSERT by `(workspace_id, slug)` + REPLACE phases under the same `spec_id`:
  - matching `(spec_id, position)` rows are UPDATED in place â€” preserving `id` (and `pr` / `merge_sha` unless explicitly overridden)
  - new positions INSERT (with the supplied `pr` / `merge_sha` if any)
  - vanished positions DELETE
  - status is DERIVED at read time (no trigger â€” dropped in `20260725160000`); `specs-status-overrides-only`: an explicit `row.status` is persisted ONLY if it's a true override (`deferred`/`folded`) â€” a derived value OR `in_review` is normalized to `null`. Omitting `row.status` keeps the DB default (`NULL` since migration `20260907130000`); a fresh spec then DERIVES `in_review` (no phases + `vale_review_passed_at IS NULL`), so the build pipeline still holds it.
  - **`folded-spec-must-stay-folded`: `folded` is TERMINAL â€” a re-author NEVER un-folds it.** Before the upsert, `upsertSpec` reads the existing `specs.status`; if it's already `folded`, the override is PRESERVED unless the caller is EXPLICITLY re-folding (`row.status === 'folded'`). This closes the **db-reduce-calls split**: a folded spec's markdown is in `docs/brain/archive.d/`, but a later re-author that passed a derived/absent status would normalize `folded` â†’ `null`, the rollup would re-derive `planned`/`in_progress`, and the archived spec would re-appear on the active board while `cancelJobsForArchivedSpecs` auto-cancelled its builds as "spec archived". The ONLY sanctioned un-fold is an explicit `setSpecStatus(slug, null)` by the CEO â€” a re-author is never an implicit un-fold. (Symmetric runtime backstop: [[spec-drift]] `reconcileArchivedNotFolded` re-folds any row that still drifted.)
  - per-`SpecPhaseInput`: PASS `undefined` for `pr` / `merge_sha` / `verification` to PRESERVE the existing value, PASS `null` to CLEAR
- **`movePhase(phaseId, newSpecId, newPosition)`** â€” the **lift-a-phase-between-specs primitive**: a SINGLE `UPDATE spec_phases` that preserves the phase's `id`, `pr`, `merge_sha`, `created_at` ([[../specs/spec-status-phase-pr-provenance]] provenance chain). The trigger fires twice (old + new `spec_id`) and rolls both rollups in one transaction. The unique `(spec_id, position)` index may reject the move if the destination slot is occupied â€” the caller is responsible for shifting positions first when so.
- **`stampPhaseShipped(workspaceId, slug, position, { merge_sha, pr? })`** â€” stamp ONE phase `shipped` with its MAIN-promotion PR provenance (the canonical leaf write that advances the now-DERIVED `specs.status`). The only status-write needed to ship a phase. Under the branch-flow ([[../specs/spec-goal-branch-pm-flow]]) this is the M5 PROMOTION stamp â€” fired when the spec/goal branch lands on main, never from a spec-branch build.
- **`stampPhaseBuilt(workspaceId, slug, position, { build_sha })`** â€” the BRANCH-BUILD counterpart ([[../specs/spec-goal-branch-pm-flow]] M2). Records the `spec_phases.build_sha` (the `claude/build-{slug}` spec-branch commit where the phase built) and moves the phase to `in_progress` (built â€” NOT shipped). NEVER writes `status='shipped'` or `merge_sha` (those stay for the M5 promotion stamp); no-ops on a `shipped`/`rejected` phase (never regresses a promoted one). Called by the box worker the moment a phase commits+pushes on the spec branch (`scripts/builder-worker.ts`); the only writer of `build_sha`. **The `finalizeBuiltPhase` seam invokes this from BRANCH GROUND-TRUTH** ([[../specs/build-accumulation-stamp-gap-and-rollback-guard]] P1): the finalize path scans `origin/main..HEAD` for `Phase: N` trailers (the M1 commit-trailer convention) and stamps EVERY position present on the branch â€” not just the current session's phase. Closes the accumulation-stamp-gap wedge where a phase committed during a `needs_input`/`needs_approval` PAUSE (or whose position couldn't be derived from a resume) never got stamped, so `isSpecAccumulationComplete` permanently reported it "not built" and the PR never opened. Idempotent (terminal phases no-op; re-stamping the same `build_sha` is harmless).
- **`isSpecAccumulationComplete(workspaceId, slug)`** â†’ `{ complete, reason }` ([[../specs/spec-goal-branch-pm-flow]] M2/M3) â€” "is this spec FULLY accumulated on its `claude/build-{slug}` branch?" Complete iff every phase carries a `build_sha` (`stampPhaseBuilt`) OR is terminal (shipped/rejected); a 0â€“1-phase spec is trivially complete (one-shot â€” nothing to accumulate). Fails OPEN on a read error / missing row (a PM blip mustn't wedge a complete one-off; downstream gates still apply). The SHARED predicate read by THREE seams so they never disagree on "fully built on branch": the [[github-pr-resolve]] auto-merge accumulation gate, the [[agent-jobs]] M3 pre-merge-test trigger (`maybeEnqueuePreMergeSpecTestOnAccumulation`), and [[agent-jobs]] `isSpecPromoteEligible`. (Promoted from a private helper in `github-pr-resolve.ts` to the SDK in M3.)
- **`markPhaseInProgress(workspaceId, slug)`** / **`markRemainingPhasesShipped(workspaceId, slug)`** / **`restampPhases(...)`** â€” the derive-rollup-status phase-advance writers (build-start in_progress flip Â· fold-now "close the rest" Â· audit re-stamp). All move `spec_phases` rows; the readers derive `specs.status` from them.

PM-write SDK surfaces (the explicit `specs.status` / `blocked_by` / one-shot-provenance overrides â€” added by [[../specs/pm-db-agent-toolkit]] to retire raw `.from("specs").update(...)` from agent code):

- **`setSpecStatus(workspaceId, slug, status, actor)`** â€” write the EXPLICIT lifecycle override on `specs.status` (`deferred` / `folded`), or **`null` to CLEAR the override** so status is PURELY DERIVED from the phase rollup. `specs-status-overrides-only`: a DERIVED value (`planned`/`in_progress`/`shipped`) OR `in_review` (now derived) passed here is normalized to `null` with a warning, so the override-only invariant can't be broken by a stray caller. **To SEND a spec back to review, NULL `vale_review_passed_at` via [[spec-card-state]] `markSpecCardBackToReview`** â€” a status write alone no longer expresses "in review". Slug-resolved single UPDATE, mirrors goals-table `setGoalStatus`. The fold worker calls this with `'folded'`; the disposition lane calls it with `null` (â†’ derived) / `'deferred'`. The **only** sanctioned `specs.status` writer outside `upsertSpec`. **Parks the `deferred` boolean in lockstep (2026-07-03 fix):** the board's `deriveSpecCardStatus` reads `row.deferred` (the BOOLEAN column), never `status === 'deferred'` â€” so `setSpecStatus` writes `deferred = (status === 'deferred')` in the same UPDATE: set when parking, cleared for every other destination (`folded`/`null` un-park), matching the canonical [[spec-card-state]] `upsertCardState` â†’ `dualWriteSpecRow` â†’ `specs.deferred` flag path.
- **`setSpecBlockers(workspaceId, slug, blockedBy: string[])`** â€” write `specs.blocked_by` (the spec-blockers gate the build pipeline reads). The milestone-sequence reconciler computes the order-preserving union and persists it here.
- **`setSpecAutoBuild(workspaceId, slug, autoBuild: boolean)`** â€” set the `specs.auto_build` toggle (the init/groom lanes SKIP a spec with `auto_build=false`; the CEO commissioning a build flips it on). The only narrow SDK writer for the column outside a full `upsertSpec` re-author â€” a one-off (re-opening a stuck spec) can flip it without round-tripping the whole body. Slug-resolved single UPDATE, sibling to `setSpecBlockers`/`setSpecStatus`.
- **`setSpecParent(workspaceId, slug, { parent?, parentKind?, parentRef?, milestoneId?, relatedSpec? })`** â€” re-parent a spec in ONE UPDATE: the free-text `parent` prose + the typed `parent_kind`/`parent_ref` pair + the `milestone_id` FK + the `related_spec` LINK (no-spec-parent), without round-tripping the body through `upsertSpec` (which replaces phases). **one-off-spec-parent:** a one-off spec's home is a function MANDATE (`parentKind:"mandate"`, `parentRef:"{owner}#{mandate-slug}"`, `milestoneId:null`); a goal-bound spec's is a specific MILESTONE (`parentKind:"milestone"`, `milestoneId:"<goal_milestones.id>"`). Pass the `parent` prose so it names the same mandate/milestone the typed pair points at (Vale reads the prose). Undefined fields untouched. Slug-resolved single UPDATE, sibling to `setSpecBlockers`/`setSpecStatus`/`setSpecAutoBuild`.
- **`stampSpecMergeProvenance(workspaceId, slug, { pr, merge_sha })`** â€” write a **one-shot spec's** card-level `merged_pr` + `last_merge_sha` (the zero-phase merge-webhook path in [[agent-jobs]]). It does NOT touch `specs.status`; this provenance IS what the deriver reads to surface a zero-phase spec as `shipped` (`deriveSpecCardStatus`: `merged_pr`/`last_merge_sha` set â‡’ `shipped`, else `planned`) â€” so the merge stamp is the only write needed to ship a one-shot spec.
- **`stampSpecGoalBranchSha(workspaceId, slug, goalBranchSha | null)`** â€” the **"on the goal branch" marker writer** ([[../specs/spec-goal-branch-pm-flow]] M4). Records `specs.goal_branch_sha` (the `goal/{goal-slug}` merge commit where this spec's `claude/build-{slug}` branch merged onto its goal branch), set by [[agent-jobs]] `promoteEligibleSpecsToGoalBranch` on a successful merge. The only writer of `goal_branch_sha`. Writes NEITHER status NOR merge_sha â€” on-the-goal-branch is NOT shipped; M5 flips the build_sha'd phases to shipped on the atomic goalâ†’main promotion. Pass `null` to clear.
- **`isSpecOnGoalBranch(workspaceId, slug)`** â†’ `boolean` ([[../specs/spec-goal-branch-pm-flow]] M4) â€” "is this spec's branch merged onto its goal branch?" (i.e. `goal_branch_sha` set). The claim-time blocked_by gate reads it to clear a GOAL-MATE blocker (a goal-mate never ships to main until M5's atomic promotion, so "on the goal branch" â€” not "shipped" â€” is its clearance signal). Fails CLOSED on a read error (unknown goal-branch state must not release a dependent).
- **`goalBranchState(workspaceId, goalSlug)`** â†’ `{ goalSlug, specs[], allOnGoalBranch }` ([[../specs/spec-goal-branch-pm-flow]] M4) â€” the whole-goal goal-branch state (every spec linked to the goal's milestones, each tagged `onGoalBranch`). `allOnGoalBranch` (â‰¥1 spec AND every spec stamped) is the **seam M5 consumes** to detect goal-complete: [[agent-jobs]] `promoteCompleteGoalsToMain` reads it (gate 2), then on the atomic goalâ†’main merge `applyGoalPromotionEffects` iterates `state.specs` to flip every member phase shipped (build_shaâ†’shipped). Read-only; resolves goalâ†’milestonesâ†’specs through [[goals-table]] `getGoal` + `specsForMilestone`.
- **`listSpecPhaseAnomalies(workspaceId)`** â†’ `{ orphans, provenanceGaps }` â€” the integrity-scan reader: (a) ORPHAN `spec_phases` rows whose parent `specs` row is missing (global â€” no parent to read a workspace from), and (b) PROVENANCE-GAP shipped phases with both `pr` + `merge_sha` null (workspace-scoped, folded-excluded). Resolves `spec_id â†’ {slug, workspace}` internally so callers (the [[spec-drift]] reconciler's anomaly sweep) never touch raw PM tables.

All writers route through `createAdminClient()` (service-role; the RLS policy `specs_service` / `spec_phases_service` grants full access). No client-side writes.

**Raw PM-table writes are CI-forbidden outside this SDK.** `scripts/_check-pm-sdk-compliance.ts` (chained into `predeploy`) fails the build on any `.from('specs'|'spec_phases'|'goals'|'goal_milestones').update()/.insert()/.upsert()/.delete()` in `scripts/builder-worker.ts` or `src/lib/**` that isn't in this SDK (or `goals-table.ts`) and isn't on its `SANCTIONED_RAW_WRITES` allow-list. One sanctioned exception: [[spec-card-state]] `dualWriteSpecRow` (the legacy mirror dual-write, broad column set).

## Not atomic across parent + children

supabase-js has no transaction surface, so `upsertSpec` is a sequence of writes (UPSERT specs, then DELETE / UPDATE / INSERT phases). The trigger keeps `specs.status` consistent after each write, and re-running the same call is idempotent (position-keyed REPLACE is deterministic). Callers requiring true atomicity must compose at the SQL layer.

## Invariant â€” no client-side id/slug array batching ([[../specs/retire-residual-in-array-batching-to-server-side-rpcs]] Phase 3)

**All specs/phase reads on this file go through server-side RPCs â€” NEVER a client-side id/slug array over the wire.** The 2026-07-08 DB-overload incident and its precedent ([[../specs/list-specs-with-phases-rpc-retire-in-array-client-join]]) traced to `.in("id", [large-array])` calls whose URL overflowed the ~16KB undici header cap (`UND_ERR_HEADERS_OVERFLOW`) once the workspace held a few hundred specs, wedging `getSpec` / the roadmap / the build claim-gate / the spec-review enqueue reaper. The three sanctioned reads on this file:

- `listSpecs` / `getActiveSpecs` / `getAllSpecs` â†’ **`list_specs_with_phases(uuid, text)`** RPC (server-side specs+phases join).
- `listSpecPhaseAnomalies` â†’ **`list_spec_phase_anomalies(uuid)`** RPC (server-side orphan + provenance-gap sweep via a single `spec_phases LEFT JOIN specs`).
- `getSpec` â€” one row by `(workspace_id, slug)` PK, then one phase-lookup by `(spec_id)` â€” no id array by construction.

**CI-enforced** by `scripts/_check-specs-phases-no-client-in-batching.ts` (chained into `predeploy`): a reintroduced `.in("id", <array>)` or `.in("spec_slug", <array>)` in `src/lib/specs-table.ts` or `src/lib/brain-roadmap.ts` fails the build. Genuinely-bounded LITERAL arrays (`.in("id", ["a","b"])`) still pass â€” a variable, `.slice(...)` batch, or spread does not.

## Gotchas

- **The trigger is the rail.** Don't write `specs.status='shipped'` directly while a phase is still `planned` â€” the next phase write will roll it back. That's the DB-enforced rule that kills the [[../specs/spec-review-agent]] "shipped with 1 phase" class.
- **`in_review` / `folded` are not auto-cleared.** The rollup early-returns on those; an explicit status flip ([[spec-card-state]] `applyAdaDisposition`, the fold worker) is required.
- **`upsertSpec` preserves provenance.** Existing `(spec_id, position)` rows keep their `id` + `pr` + `merge_sha` unless the caller passes new values. The backfill relies on this so it never destroys a per-phase PR tag while resyncing markdown.
- **`movePhase` requires a free destination slot.** The `(spec_id, position)` unique constraint will reject a move onto an occupied position; callers either renumber siblings first or pop the destination's existing phase elsewhere.
- **`Phase`'s `rejected` is NOT a `specs.status` value.** It's a phase-level state. A whole-spec rolled-up status never goes to `rejected`.
