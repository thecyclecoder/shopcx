# libraries/ticket-analyzer

Per-ticket AI analysis: sentiment, intent, summary, suggested action. Runs as a supervised box-session agent (`kind='ticket-analyze'`, Cora, a persona in [[../functions/cs]]'s team) dispatched by the builder worker, not a raw-API cron. Writes [[../tables/ticket_analyses]] through a typed SDK (not raw `.from('ticket_analyses')`). Each verdict is recorded to [[../tables/director_activity]] under CS for June (CS Director) to supervise and grade.

**File:** `src/lib/ticket-analyzer.ts`

**Supervision:** [[../functions/cs]] owns escalation triage quality. The analyzer runs on the box lane ([[../recipes/build-box-setup]]), with every verdict recorded to [[../tables/director_activity]] (`director_function='cs'`, `action_kind='analyzed_ticket'`), and the kind graded by the CS director's sweep ([[../libraries/agent-grader]] — June's gradeable kinds include `'ticket-analyze'`). Reasoning from each session is surfaced alongside the grade.

## Trigger contract — Sol-handled, 30-min settle, once per cycle

Cora is a **post-handling QC grader**, not an every-30-min re-scanner. She analyzes a ticket **exactly once per Sol handling cycle**, and never re-investigates a settled or June-decided ticket without new customer activity. This contract is enforced entirely inside [[../inngest/ticket-analysis-cron]]'s `find-tickets` step (the pure predicate `passesCoraSelectionGate`, pinned by `src/lib/inngest/ticket-analysis-cron.gate.test.ts`) so the box lane sees only tickets that pass every gate. Implements [[../specs/cora-only-investigates-after-sol-handles-and-ticket-closed-30min-no-reinvestigation]].

A ticket is selected only when **all four** hold:

1. **Sol has handled it.** There is a **LIVE** [[../tables/ticket_directions]] row (`superseded_at IS NULL`) written by Sol's first-touch box session ([[../specs/sol-ticket-direction-artifact-and-first-touch-box-session]]). No Direction → dropped. Cora never grades a ticket Sol hasn't taken.
2. **The ticket has been closed for ≥ 30 minutes.** `closed_at IS NOT NULL AND closed_at <= (now − 30 min)` (constant `CORA_CLOSE_SETTLE_MS` in the cron). This SQL-floor is applied at the source, so tickets still in the settle window are never even loaded — Cora never grades an in-flight window a customer might still say "thanks!" to.
3. **Cora hasn't already graded THIS handling cycle.** Dedup on the live Direction's `authored_at`: a `tickets.last_analyzed_at >= direction.authored_at` means we already ran once for this handling → skip. A stale `last_analyzed_at` from a **prior** cycle is fine — Cora re-grades the new cycle. (One grade per handling; N handlings on a ticket → up to N grades.)
4. **June (CS Director) hasn't already decided THIS handling cycle.** Any [[../tables/director_activity]] row with `action_kind='cs_director_call'` + `metadata.ticket_id=<t>` + `created_at >= direction.authored_at` → skip. June's verdict is the last word on the current handling; Cora doesn't re-grade a ticket the director already ruled on.

**No re-investigation of a settled ticket.** A June-decided closed ticket stays closed to Cora **forever on its own** — no updated_at bump, no dashboard nudge, no downstream write can re-open it to her. Re-eligibility only comes back through the natural Sol re-handling loop:

```
customer replies (new inbound)
  → Sol re-handles the ticket (superseDirection + writeDirection, new authored_at)
  → orchestrator closes it again (new closed_at)
  → 30-minute settle
  → Cora grades the NEW cycle once
```

Once Sol re-authors, `direction.authored_at` advances **past** every prior June decision and every prior `last_analyzed_at`, so the Phase-1 and Phase-2 checks naturally re-permit exactly one fresh grade per new handling. The stamp-on-slip guard in the cron (`tickets.last_analyzed_at = now()` on every skip except `already_in_flight`) ensures the cron never re-selects the same skipped ticket on the next tick.

This contract retires the loose pre-2026-07 rule ("re-analyze whenever `last_analyzed_at < updated_at`"): a Sol-uncovered updated_at bump (a new tag, an audit note, a merged tickets row) can no longer pull Cora back into a settled ticket, and a `cs-director-call` verdict closes the loop cleanly instead of re-tripping the close → analyze → reopen → close cycle June was hired to end.

## Cheap triage tier — Haiku pre-filter in front of the box session ([[cora-triage-pass]])

Before a ticket that passes the trigger contract spawns the expensive Max box session, `enqueueTicketAnalyzeJob` runs a **single inline Haiku call** ([[cora-triage-pass]] `runCheapTriagePass`) as a cheap first tier. Most closed, Sol-handled tickets were handled fine and a full grading session on them tells us nothing new — so the cheap pass grades those itself and only spawns the deep Cora session on the ones that show a **terminal-state problem**.

**Where it sits in the enqueue path** (all guards reused): the gate runs AFTER every skip gate (`merged_into` / `ai_disabled` / `analyzer_locked` / `do_not_reply` / `skip_tag`) AND the one-in-flight dedup, so it inherits the exact candidate set as the deep analyzer and never double-fires against an in-flight ticket. It is gated on `trigger === "auto_close"` **only** — a `manual` rescore, `manual_close`, or `reopen_close` always forces the deep grade (a human explicitly asked for the real thing).

- **Clean verdict** (`needsReview === false`) → `recordCheapPassClean` writes a lightweight `[cheap-pass]` [[../tables/ticket_analyses]] row (coarse Haiku score, `model=claude-haiku-4-5…`, `apiBilled=true` — the cheap pass runs on the paid API, never the Max box lane) and stamps `tickets.last_analyzed_at`. `enqueueTicketAnalyzeJob` returns `{ ok: false, reason: "cheap_pass_clean" }` and the deep session is **never spawned**.
- **needs_review verdict OR any hard failure** (`runCheapTriagePass` returns `null` — no API key, transport error, empty/unreadable transcript) → **falls through** to the normal deep `agent_jobs` enqueue. The gate is **recall-biased**: it never silently suppresses a ticket it couldn't confidently clear.

**The cheap pass NEVER escalates to June.** Its ONLY escalation is spawning the deep Cora session (by falling through). June (CS Director) is reached solely from inside that deep session's `decideEscalationAction` → [[../inngest/triage-escalations]] → `cs-director-call`. Two hops, never one — the cheap tier cannot reopen a ticket, notify a human, or write a director-decision row.

**Terminal-state principle** (shared with `hasResolvedActionClose` below): the pass flags only END-state failures — a customer left unresolved/frustrated in their LAST message, a promise made and not kept, an out-of-policy remedy offered, a false claim that an action happened, a wrong outcome shipped. A messy-but-recovered middle (a turn-1 stumble the agent later fixed) is deliberately NOT a trip — we escalate on getting it wrong at the END, not mid-turn. The allowed signal set is the exported `TRIAGE_SIGNALS` constant; the pure `buildTriagePrompt` / `parseTriageResult` helpers are unit-pinned in `src/lib/cora-triage-pass.test.ts` (unparseable / contradictory → fail open).

## Box-agent supervision

The analyzer is now a supervised box-session agent. `scripts/builder-worker.ts → runTicketAnalyzeJob` claims each `kind='ticket-analyze'` job, loads the ticket + guidance context, runs ONE Max session with `buildSystemPrompt` + `buildUserPrompt`, parses the JSON verdict via `parseAnalysis`, then calls `applySeverityActions` to write the outcome. Records to [[../tables/director_activity]] on completion (`actor='ticket-analyze:box-session'`, `action_kind='analyzed_ticket'`, `spec_slug=ticket_id`, `reason`, `metadata: { score, severity, issues, escalated?, ai_messages }`). Existin gates (do_not_reply skip, ai_disabled / analyzer_locked respect, force-override rules) are preserved in the dispatch path.

## Research CLI (bounded read-only, verify before flagging)

Cora may **verify a claim** she cannot confirm from the transcript with a bounded set of read-only lookups **before grading it**, so a fact missing from the prepared bundle is no longer a guess. Introduced in [[../specs/cora-gets-readonly-research-power-to-verify-claims-before-grading]] · Phases 1-3.

**Ordering (Phase 2):**

1. **PRIMARY PATH — research, then grade the truth.**
   - Claim **verified correct** by research → NOT an `inaccuracy` issue; the hard cap does not fire. Example: AI said "Berry"; `get_product_nutrition` confirms Berry is a real Superfoods variant → cleared.
   - Claim **verified contradicted** by research → real `inaccuracy` issue, `description` cites what was looked up. This is the case the inaccuracy hard cap is written for. Example: AI said "$47.99 per unit"; `get_customer_account`'s orders block shows the line at `$50.99` → kept.
2. **FALLBACK — grading-confidence guard.** Only when research still cannot settle the claim (tool returned nothing conclusive; fact is outside the read-only surface): do NOT emit `inaccuracy` on it; do NOT score-cap or force-escalate on an unverified detail. Prefer `kb_gap` (documented surface, missing fact) or omit — silence beats a fabrication flag. This fallback is subordinate to the primary research path, not a substitute for it.

**Bound (Phase 3):**

- **Read-only** at the allowlist level — the CLI (`scripts/analyzer-research-tools.ts`) refuses anything outside the fixed tool list.
- **No mutation** at the executor level — delegates to `src/lib/improve-tools.ts` → `sonnet-orchestrator-v2`'s `executeToolCall`, the same shared read-only executor `ticket-improve` / `ticket-handle` use. The analyzer's ONLY write remains its verdict, applied by `applyAnalyzerVerdict` on the deterministic worker.
- **Targeted, not open-ended** — a hard per-grade lookup counter (env `ANALYZER_RESEARCH_CAP`, default `8`) refuses further calls once the cap is hit within a single grade. The counter lives at `/tmp/analyzer-research-<ticketId>.count` and resets after 30 min of staleness (well past any single grade's runtime). Running count is echoed to stderr on each call so Cora can see how close she is; hitting the cap forces the FALLBACK path.

**Tools (all delegate to the shared read-only executor):**

| Tool | Surface | Verifies |
|---|---|---|
| `get_customer_account` | Subscriptions (with per-line `variant_id` + realized price + MSRP/floor context), last 180d of orders (with `line_items` per-unit price + discount codes + financial_status + subscription linkage), loyalty balance, marketing consent | Per-unit price claims (real charged amounts), subscription state, entitlement claims, linked-account facts |
| `get_product_knowledge` | Product info (title/description/positioning) | Product-level positioning claims |
| `get_product_nutrition` | Per-variant Supplement Facts (variant title / flavor / servings / key nutrients) — `json_input: {"query":"..."}` | Variant / flavor claims (e.g. "Berry"), per-variant nutrition claims |
| `get_returns` | Returns / exchanges on file for the customer | Return-status claims |
| `get_ticket_analysis` | Latest prior analysis for this ticket | Re-grade context (score / issues / summary) |

Brain / policy read is native (Claude Code `Read` / `Grep` against `docs/brain/`), so a policy claim can be verified against the current documented policy without going through the CLI.

Called from the `ticket-analyze` skill (`.claude/skills/ticket-analyze/SKILL.md`) — the box worker's `runTicketAnalyzeJob` prompt threads the ticket id in and points Cora at this CLI. See also [[../recipes/pm-flow-data-sources]] for the invariant that the analyzer never reads from `docs/brain/specs/*.md` (the DB row is the source of truth).

## Escalation routing

`analyzeTicket`'s severity actions — now decided by the pure predicate `decideEscalationAction` on severity / actionability, not on the raw score alone (see § Gotchas: a resolved ticket with only a minor quality note stays closed regardless of score; an actionable customer at ≤5 → escalate + notify; at 6 → escalate silently) — escalate to the **AI Routine**, not a human: the re-open sets `escalated_to = null` with `escalated_at` + `escalation_reason` set (it no longer round-robins to a workspace member or pre-assigns `assigned_to`). That `escalated_at`-set + `escalated_to`-null state is exactly what [[../inngest/triage-escalations]]'s cron picks up — the idle-triage routine then runs solver→skeptic→quorum and produces approval-gated todos, handing **up** to a human only on no-quorum. The orchestrator (`action-executor.ts`), workflow executor (`workflow-executor.ts`), and portal remediation (`portal/remediation.ts`) escalations default the same way. See [[../specs/escalate-to-routine-by-default]].

## Gotchas

- **The `customerThreat` scan must use the CLEANED body, not raw `body`.** Inbound email replies quote the message they're replying to. Our own order-confirmation emails carry a "Join our Facebook group!" footer, so every reply to one quotes it — and `CUSTOMER_ESCALATION_KEYWORDS` contains the bare substring `"facebook"`. Scanning raw `body` substring-matched that quoted footer → `customerThreat` true → silent force-escalate of a positively-closed ticket. Seen on ticket `246163b4` (Melissa Sachs, 2026-06-19, score **9** — "handled correctly, customer closed positively"). **Fixed:** the scan now reads `m.body_clean || cleanEmailBody(m.body)` ([[email-cleaner]] strips quoted history + signatures), so only the customer's actual new text is checked. This was systemic — any email reply quoting our Facebook footer would have tripped it.
- **`CUSTOMER_ESCALATION_KEYWORDS` substring-matches `"fraud"` against the entire inbound body**, which false-positives on benign bank phrasing — e.g. "my bank put a Fraud Alert on my card", "the fraud team called", "flagged for fraud". The customer is cooperating, not threatening, but `customerThreat` flips true and the ticket force-escalates silently regardless of score. Seen on ticket `a613e06e` (Elizabeth Fraser, 2026-06-05). **Mitigated** by `ESCALATION_KEYWORD_DENYLIST` + `matchesEscalationKeyword()` — a benign-phrase denylist that excludes the matching keyword when a cooperating-context phrase is present (added via the agent-todo `code_change`, PR #2), and now also by scanning cleaned bodies (above). The same substring risk still applies to other keywords (`"scam"`, `"report you"`, the social-platform names) on the customer's *own* text — a benign mention ("I saw your post on Facebook") could still match; tightening those to require threatening context is open work.
- **The grader mis-read the refund playbook's designed stand-firm round as an `inaccuracy`.** The refund playbook intentionally states the policy (a tier-0 "pre-exception stand firm") before offering the Tier 1 store-credit / Tier 2 cash exception — that's the retention arc: stand firm → customer pushes back → save with an exception → positive close. The grader has the policy in context, saw the stand-firm denial, concluded "Tier 1 was available, so this was a wrong denial," labeled it `inaccuracy` (a `SEVERE_ISSUE_TYPE`) → hard-cap 5 + force-escalate — on a ticket that ended in a clean store-credit save and a positive close. The in-flight-playbook guard (line ~761) didn't save it because the playbook had **completed** (cleared `active_playbook_id` on the positive close), so the guard's `active_playbook_id` check was null. Seen on ticket `cc3d6b9b` (2026-06-19). **Fixed** via an approved `grader_prompts` calibration rule (`900a4fa0`): the stand-firm-then-save arc is correct execution, not an inaccuracy/rule_violation; tone is at most a minor note; the arc is not an escalation trigger. Re-grade went 5 → 8 (no escalation). Note: the calibration deliberately stops at "solid 8, not escalated" rather than forcing 9-10 — the opening tone was a genuine empathy miss, fixed at its source in the **refund playbook** copy, not by blinding the grader.
- **Grade the AI against the data the AI actually had — a detail absent from the ANALYZER'S own surface is `unverified_from_surface`, not a fabrication.** The grader sees the conversation window + attached context blocks (agent guidance, playbook context, policies, calibration rules); it does NOT see the full product catalog, every SKU/variant/flavor, the full order line, or the customer's full history. When the AI makes a specific claim, the prompt (`buildGraderSystemPrompt`, SURFACE-BOUNDED GRADING section) triages it into three buckets: (A) **CONTRADICTS the surface** → real `inaccuracy`, HARD CAPS apply (max score 5); (B) **CONFIRMED by the surface** → not an issue; (C) **ABSENT from the surface** — specific, non-vague, but the surface neither confirms nor contradicts → tag `unverified_from_surface` (the exported string constant `UNVERIFIED_FROM_SURFACE_ISSUE_TYPE`). Bucket C is a limit of the ANALYZER'S context, not evidence the AI lied — it does NOT count as a fabrication/inaccuracy, does NOT cap the score, and is deliberately NOT in `SEVERE_ISSUE_TYPES` (so it can never force-escalate). Only surface-contradicting statements count as inaccuracies. `applySeverityActions` additionally short-circuits the reopen/escalate path when the analyzer's ISSUES ARRAY IS SOLELY `unverified_from_surface` AND the ticket is cleanly positively closed AND there is no customer-threat keyword: the pure predicate `analysisIssuesAreSoleUnverifiedFromSurface(issues)` flips true only in that narrow case, and a non-actionable audit note is logged instead of updating `tickets.escalated_at` — which is exactly what the [[../inngest/triage-escalations]] cron enumerates when enqueueing `cs-director-call` (June review) jobs, so no unverified-only resolved ticket ever reaches June. A genuine surface-contradicting `inaccuracy`, `false_promise`, `broken_action`, or unresolved customer situation (mixed with the unverified note, or alone) still escalates as before — the predicate flips false the moment any non-unverified issue appears in the set. Seen on ticket `cd2e4a9a-9be8-4457-94ef-7570405d2eff` (Cora capped the score and force-escalated on a "fabricated" flavor that was actually correct — she just couldn't see the product variant in her own order-line data). See [[../specs/cora-grades-against-ai-data-surface-no-false-fabrication-on-unseen-facts]].
- **Auto-escalation keys on severity / actionability, not on a raw middling score — a resolved minor-issue ticket stays closed.** Pre-Phase-2 `applySeverityActions` decided the tier straight from the score number: `if (score <= 5) escalate + notify; else if (score === 6 || forceEscalate) silent`. That fired an escalation + reopen on ANY score ≤5, regardless of whether the ticket was resolved and the customer happy — a `score 5` grade on a coaching-only minor note re-opened + escalated an otherwise-perfect ticket ([[../specs/escalation-keys-on-real-severity-not-a-middling-score-minor-issue-on-resolved-ticket-stays-closed]], regression source ticket `cd2e4a9a-9be8-4457-94ef-7570405d2eff`). **Fixed** by extracting the tier decision into a pure exported predicate `decideEscalationAction(input: EscalationDecisionInput): { action, reason }` that gates on severity / actionability, not the raw score: (1) `forceEscalate` (a severe issue class after the inaccuracy-only-on-positive-close override, OR a customer-threat keyword) → `escalate_silent` unconditionally; (2) `positivelyClosed` (`hasCleanPositiveClose(admin, ticketId)`) AND no severe issue class AND no customer-threat → **`"none"`** — the ticket stays closed regardless of the score; an internal audit note is logged (worded to never match `REOPEN_NOTE_MARKERS` or `POSITIVE_CLOSE_NOTE` so a later `hasCleanPositiveClose` scan is untouched); (3) otherwise (actionable customer — unresolved / mishandled / still needs something) tier by score exactly as before (`≤5` escalate_with_message, `6` escalate_silent, `≥7` none). The `positivelyClosed` state is computed ONCE per run and reused by the sole-unverified override, the inaccuracy-only override, and this Phase 2 gate (one DB scan, three decisions). Companion phase-1 rubric changes: `GRADER_RUBRIC_BODY`'s severity-aware HARD CAPS distinguish a customer-failing inaccuracy (`max score 5`) from a minor-note-on-otherwise-correct-resolution (`max score 7`, good-enough band with a coaching note). The predicate is exported so tests can pin it without a DB (see `src/lib/ticket-analyzer.escalation-decision.test.ts`); a severe issue class or an unresolved customer still auto-escalates as before — the change is narrow (a resolved, non-severe, non-threat ticket no longer reopens on the score number alone). See [[../specs/escalation-keys-on-real-severity-not-a-middling-score-minor-issue-on-resolved-ticket-stays-closed]] and [[../lifecycles/ai-analysis]].
- **An `inaccuracy`-only flag force-escalated a happy, 7/10, positively-closed ticket.** `applySeverityActions` set `forceEscalate = hasSevereIssue || customerThreat`, where `hasSevereIssue` is true for ANY `inaccuracy`/`false_promise`/`broken_action` issue — at any score, regardless of a positive close. The grader tagged a harmless closing phrase ("your loyalty points stay intact") as an `inaccuracy` on a 7/10 ticket the customer closed happily after a clean resolution (ground truth: loyalty fully intact). That alone re-opened + escalated the ticket to the routine/human — the same churn the `customerThreat` path already avoids. Seen on ticket `9a6e53d9` (Laurie Predmore). **Fixed:** when the **only** severe trigger is `inaccuracy` (no `false_promise`/`broken_action`), there's no `customerThreat`, the score is **≥7**, and the ticket is **cleanly positively closed**, the force-escalate is suppressed and logged as a non-actionable internal note. Positive close is detected DETERMINISTICALLY via `hasCleanPositiveClose()` — it scans the FULL `ticket_messages` history for the `[System] Positive close. Ticket closed.` note (emitted at [[../inngest/unified-ticket-handler]]:1593) and confirms it's the most recent lifecycle event: no re-open/escalation note after it (so a re-opened-then-reclosed ticket like `9a6e53d9`, which had a prior score-3 reopen, is classified by its LATEST close) and no unanswered inbound customer message after it. It is NOT keyed on `tickets.status/closed_at/resolved_at` — the analyzer only runs on closed tickets, so `status='closed'` is near-universal and would suppress the override globally. `false_promise` and `broken_action` still escalate exactly as before (no heal-verification gating — `verify_refund_issued` is an unimplemented future recipe, so gating those would be a silent-drop hole); the `customerThreat` path, the score ≤6 paths, and the `selectResearchRecipes` hook are unchanged. See [[../specs/analyzer-no-force-escalate-inaccuracy-on-positive-close]].
- **No idempotency check on repeat mutations within a ticket.** The orchestrator can fire `bill_now` (or any mutation) twice in consecutive turns without verifying the first attempt's outcome — the analyzer only surfaces it after the fact as `missed_opportunity`, but the prevention belongs in orchestrator rules, not grading. Seen on the same ticket: `bill_now` fired in turn 2 and again in turn 3 without checking turn 2's result, with real duplicate-charge risk. Addressed by a sonnet_prompt rule (todo `943de409`) instructing the orchestrator to check the prior action result before re-firing a mutation.
- **The grader fetch THROWS on a Claude failure (no `grader_http_*` swallow).** `analyzeTicketInner` used to `return { ok:false, reason: grader_http_${status} }` on a non-2xx — a silent drop: the cron then marked `last_analyzed_at` and the grade was lost for good during a Claude outage. Now it throws via [[anthropic-retry]] (retryable status / network → `AnthropicDependencyError`; terminal → `NonRetriableError`), and [[../inngest/ticket-analysis-cron]] catches the dependency case and **defers** the ticket (re-graded next */30 tick — park-and-drain). See [[../specs/agent-outage-resilience]] Phase 1. (The `grader_http_*` reason no longer occurs; a thrown grader error surfaces as `ok:false` via the `threw` branch of the heartbeat below.)
- **`ai_disabled` — human directive hard gate.** When a person clicks "Turn off AI on this ticket" in the dashboard, the row's `ai_disabled` flips true. `analyzeTicketInner` returns `{ok:false, reason:'ai_disabled'}` early (right next to the `do_not_reply` skip) so we don't grade an intentional silence and we don't force-escalate a ticket a human just told us to leave alone. `applySeverityActions` also re-reads `ai_disabled` on its `tBefore` fetch and short-circuits before the `agent_intervened`/`active_playbook_id` respect-closure guards — same reason, plus the human may have toggled it AFTER the initial row-read. Non-propagating on merge; the surviving ticket keeps its own value. Phase 1 of `docs/brain/specs/human-directives-hard-gates-over-ticket-ai.md`.
- **`analyzer_locked` — human veto over the cron.** Distinct from `ai_disabled` above: the inbound handler still runs (a customer reply still gets an AI response) but the analyzer is silenced. `analyzeTicketInner` bails early with `reason:'analyzer_locked'` if the lock landed between the cron's SELECT and the grade, so the caller's skip branch stamps `last_analyzed_at` and a later `updated_at` bump can't re-select. **The critical guard is at the TOP of `applySeverityActions`** — a fresh row read + compare-and-set BEFORE the `forceEscalate` math, so a severe-issue type (`false_promise`/`broken_action`/`inaccuracy`) or a customer-threat keyword can NOT punch through the human's "I reviewed this, do not re-open it" veto. Set automatically when a human closes+unescalates a previously-escalated ticket in one `PATCH /api/tickets/[id]`; also toggleable explicitly via the dashboard button. Non-propagating on merge. Phase 2 of the same spec.
- **Pinned-guidance "do not escalate" beats the force-escalate overrides.** `analyzeTicketInner` already loads a `guidanceBlock` from `ticket_messages` rows flagged `is_ai_guidance=true` (~:519). Phase 3 threads that block into `applySeverityActions` and runs `guidanceHasNoEscalateDirective(guidanceBlock)` — a strict regex (`/\b(?:don'?t|do\s+not|never|no)\s+(?:re[-\s]?)?(?:escalat\w*|open\w*|reopen\w*)\b/i`). When it matches AND a severe-issue type or customer-threat keyword would otherwise force-escalate, we HARD-SUPPRESS the force and audit-note the suppression citing which trigger(s) were beaten. The regex is deliberately strict (must be an imperative negation next to `escalate`/`open`/`reopen`, no window between them) so it doesn't false-positive on a customer's quoted threat text or a passing note like "customer asked us not to escalate". Directives from a real human are the ground truth; the overrides exist for tickets nobody has weighed in on yet. Phase 3 of `docs/brain/specs/human-directives-hard-gates-over-ticket-ai.md`.
- **Max-vs-API ticket cost — `$0` marginal for a box lane.** A `runTicketAnalyzeJob` run is a box lane on the Max subscription — the same "no per-token bill" honesty rail [[fleet-cost]] uses (`apiBilled === true` is the only gate that attaches `$`). `applyAnalyzerVerdict` routes `cost_cents` through `analyzerCostCentsForRun(usage, GRADER_MODEL)`, which mirrors that predicate exactly (`apiBilled && model ? usageCostCents(...) : 0`): a Max run persists `ticket_analyses.cost_cents = 0` and (via the `billing_source` tag, 'max' | 'api' | null) also SKIPS the [[ai-usage]] `logAiUsage` write — so the customer-facing sums (`/api/tickets/[id]/analysis` + `stampTicketAiCost` → `tickets.ai_cost_cents` via `add_ticket_ai_cost`) do not double-count Max tokens as a fabricated dollar figure. The deployed-analyzer fallback path (box down, running against the paid API) is the ONLY caller that stamps `apiBilled: true`; that path lands real cents in every surface. Fleet-level tokens for the Max run are still recorded via `meterAgentJob` → [[fleet-cost]] `recordAgentJobCost` as a subscription proxy on [[../tables/agent_job_costs]]. The ticket panel UI ([[TicketAnalysisPanel]]) shows a subtle "on Max (subscription)" indicator next to the `$0` analysis line so the reader sees why it's `$0` rather than assuming metering broke. See `docs/brain/specs/ticket-cost-distinguishes-max-subscription-from-real-api-spend.md`.
- **Control Tower coverage (`ai:ticket-analyzer`).** `analyzeTicket` is a public wrapper around `analyzeTicketInner` that emits ONE [[../tables/loop_heartbeats]] beat in a try/finally at the end of every run ([[../specs/control-tower-agent-coverage]] · [[control-tower]]). `ok:false` only on a thrown run or a grader HTTP / parse error — an intentional skip (no AI messages, spam tag, merged, `do_not_reply`) beats `ok:true` (the agent correctly chose not to grade), so a normal batch of skips never trips the error-rate alert; `produced` carries `{analysis_id, score, ai_messages}`. The monitor's liveness-when-work-exists alert fires when closed AI-handled [[../tables/tickets]] with `last_analyzed_at IS NULL` exist in the window but there are 0 successful beats.
