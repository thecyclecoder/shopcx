## Why this exists

The [[../dashboard/roadmap|Roadmap board]], goal/function maps, and detail pages all project a single unified view of the planning state: specs (from `public.specs` + `public.spec_phases`), goals (from `docs/brain/goals/*.md`), and functions (from `docs/brain/functions/*.md`). This module is the single loader for all three. Read-only at request time.

## Core types

- **`SpecCard`** ŌĆö one spec: `slug`, `title`, `status` (derived `SpecStatus`), `summary`, `phases[]`, `counts`, `owner?` (function slug from `**Owner:** [[../functions/x]]`), `parent?` (mandate/goal-milestone from `**Parent:**`), **`blockedBy[]`** (prerequisite specs ŌĆö see below), and **`autoBuild?`** (`false` ŌćÆ opted out of spec-blockers auto-queue via `**Auto-build:** off`).
- **`Phase`** = `"planned" | "in_progress" | "shipped" | "rejected"` ŌĆö a **phase-level** status (`SpecPhase.status`, `counts`).
- **`SpecStatus`** = `Phase | "in_testing" | "deferred" | "in_review"` ŌĆö the **whole-spec** board status ([[../specs/director-drives-all-specs-and-deferred-status]] Phase 1; `in_testing` added by [[../specs/archive.d/in-testing-derived-status]]). `deferred` is orthogonal to phase progress: the board reads it from the `public.specs.deferred` column (`dbRowToSpecCard` ŌåÆ `deriveSpecCardStatus` returns `"deferred"` over any phase rollup). `in_testing` is a derived status slot between `in_progress` and `shipped` when a build lands on a preview but pre-merge tests (spec-test + security-test green signals from [[spec-test-runs]] + [[security-agent]]) are not yet both satisfied. A deferred spec gets its **own board column** and is **excluded by every auto-build lane** ([[platform-director]]) until the CEO removes the marker (un-defers ŌåÆ Planned). `in_review` is also DERIVED, never stored (`specs-status-overrides-only`, migration `20260907130000`): `deriveSpecCardStatus` returns `"in_review"` iff the phase rollup is still `planned` (no build started) AND `vale_review_passed_at IS NULL` (Vale hasn't durably passed it ŌĆö the SAME signal the claim-time build gate reads). Because it can only appear while the rollup is `planned`, a built spec can NEVER read `in_review` ŌĆö the fix for the drift where a stale stored `in_review` pinned an all-phases-shipped spec in the In Review column. The only STORED `specs.status` overrides are now `deferred` / `folded` / `NULL`. Phases are never `deferred` or `in_testing` (no emoji maps to them). _(The legacy `parseSpec`/`deriveStatus` markdown deferred-flagging is now AUTHORING-only ŌĆö see the parser note below ŌĆö not a reader path.)_
- `ProjectTrack`, `RoadmapData`, `FunctionMap`/`FunctionGroup`/`ParentGroup`, `FunctionCard`/`Mandate`, `GoalCard`/`Milestone`, `ArchiveEntry`.

## Key exports

- **`getRoadmap(workspaceId)`** ŌåÆ `{ specs: SpecCard[], tracks }`. Reads `public.specs` + `public.spec_phases` ordered by position, **resolves each spec's `blockedBy`** against the live set, overlays `[[../tables/spec_card_state]]` for live status, sorts (in-progress ŌåÆ planned ŌåÆ shipped ŌåÆ deferred).
  - **Wrapper-level cache** ([[../specs/spec-read-efficiency-for-scaling-fleet]] Phase 3, tag `spec-read-eff-roadmap-cache`) ŌĆö a short-TTL per-workspace wrapper cache (`getRoadmapWrapperCache`, same `GET_SPEC_WRAPPER_TTL_MS` as the `getSpec` wrapper) skips the whole-board fan-out on a cache hit. Bounded by the SAME `onSpecCacheInvalidate` listener as `getSpec`: any spec write in the workspace evicts the entry immediately, so a call inside the TTL always sees fresh state on the next miss. Amortizes callers that walk blockers across many specs in one tick (Mario's stall detector, any tick handler that resolves blockedBy) ŌĆö a full sweep costs ONE board read per workspace, not one per candidate.
  - **Change-probe extension** ([[../specs/spec-read-efficiency-for-scaling-fleet]] Phase 5, tag `p_since`) ŌĆö the wrapper cache entry now carries a `maxUpdatedAt` high-water mark captured alongside the fetched snapshot. On a stale entry, `getRoadmap` calls [[pg-pool]] `specsMaxUpdatedAt` (an index-only scan on `specs_ws_updated_at_idx`) and ŌĆö when the DB's current max hasn't advanced past `maxUpdatedAt` ŌĆö extends the TTL and returns the stale value instead of re-running the whole-board fan-out. So an idle workspace amortizes to ZERO whole-board scans between real writes. Probe unavailability (pool blip) falls back to a full re-fetch ŌĆö over-fetching is always safer than serving stale data.

## Spec status & content ŌĆö DB-authoritative ([[../tables/spec_card_state]], [[../specs/spec-status-db-driven]])

Spec **content** (title, phases, owner, parent, blockedBy, repairSignature, summary) and **status** (planned/in-progress/shipped/rejected/deferred) are both **DB-authoritative**. `getRoadmap(workspaceId)` reads `public.specs` + `public.spec_phases` + `[[../tables/spec_card_state]]` (status, `flags.critical`, `flags.deferred`); the board renders DB state directly. A spec with no `spec_card_state` row defaults to `planned` (the merge-write auto-creates a row; the backfill seeded existing specs).

- **The DB is written instantly** on every status event: a build merge ([[agent-jobs]] `reconcileMergedJobs`), an owner status flip (`/api/roadmap/status`), an owner priority/defer (`/api/roadmap/priority`), Ada's drift-supervise ([[../specs/ada-director-spec-status-cards]]). Each writes `spec_card_state` + an audit row to [[../tables/spec_status_history]] ŌĆö zero markdown commits, zero deploys, zero GitHub API calls ([[../specs/spec-status-db-driven]] Phase 2).
- **`flags.deferred` wins over phase progress.** A `flags.deferred=true` card renders in the Deferred column regardless of `status` rollup. Un-defer restores phase progress.
- **Code-deploy chip.** `last_merge_sha` is compared to `VERCEL_GIT_COMMIT_SHA` to show **`shipped ┬Ę deploying`** until live, then **`shipped ┬Ę live`** ŌĆö this tracks code, not status (status writes no longer trigger deploys).

## Spec content ŌĆö `[[../tables/specs]]` schema

Spec metadata is **DB-stored** ([[spec-body-table-and-backfill]]). The `getSpec(slug, workspaceId)` query reads:

- **`specs` table columns:** `slug`, `title`, `status`, `summary`, `owner_id` (FK ŌåÆ function), `parent_id` (FK ŌåÆ goal/milestone), `blocked_by` (JSON array of spec slugs), `auto_build`, `repair_signature`
- **`spec_phases` table:** per-phase content `(index, title, status, pr, merge_sha, body)` ordered by position. `status` is `planned | in_progress | shipped | rejected`.
- **`spec_card_state` table:** live flags (`critical`, `deferred`, `blocked`, `deploy_pending`) and merged build provenance (`last_merge_sha`, `phase_states`).
- **`goals` + `goal_milestones` tables:** goal content (title, body, outcome, success_metric, owner, proposer_function, status) + milestone content (title, body) ([[goal-readers-from-db-retire-parsegoal]]). Milestone completion + goal `complete` are DERIVED from child specs ŌĆö there is no `goal_milestones.status` column.

Function metadata **still lives in markdown** (see [[#Goal metadata ŌĆö DB columns|Goal metadata below]]).

## `blockedBy` ŌĆö spec build prerequisites ([[../specs/spec-blockers]])

`SpecCard.blockedBy: { slug, title, status, cleared, kind?, memberSpecSlug? }[]` ŌĆö the specs (or goals) that must ship before this one can be built. `specs.blocked_by` (from the DB) provides the raw slugs; **`resolveBlockedBy` fills `title`/`status`/`cleared` against the live spec set + goal-membership map**:

- **`cleared`** is `true` when the blocking spec's derived `status` is `shipped`, **or** the slug is no longer a live spec at all (archived/folded or dangling). A prerequisite already on `main` never permanently blocks.
- Uncleared (`planned`/`in_progress`) = still blocking.
- **`getSpecBlockers(slug)`** ŌåÆ the resolved `blockedBy[]` for one spec. What the enqueue gate ([[roadmap-actions]] `queueRoadmapBuild`) checks before inserting a build row. Internally fires `getRoadmap()` ŌåÆ the wrapper cache above (Phase 3 `spec-read-eff-roadmap-cache`) amortizes the full-board scan across every caller inside the TTL window. A caller resolving blockers for a fleet of candidates (e.g. Mario's per-tick sweep ŌĆö [[mario]] `evaluateStalledCandidates`, tag `spec-read-eff-mario`) reads the board ONCE per unique workspace and looks up blockers from that snapshot, falling back to `getSpecBlockers` only when the slug isn't on the boardable set (e.g. a `pr-<n>` pseudo-slug) ŌĆö so the pre-Phase-3 M ├Ś 9-scan cost collapses to N ├Ś 1.
- **`SpecCard.autoBuild?: boolean`** (spec-blockers Phase 2 auto-queue) ŌĆö `specs.auto_build` (DB column, boolean). When `false` the spec is **never** auto-queued as its last blocker clears; **manual Build is unaffected**.

### Goal-blocker rule ([[../specs/one-off-spec-depending-on-goal-work-blocks-on-the-goal-not-the-member-spec]])

A standalone (outside) spec that depends on work inside a goal declares (or normalizes to) a blocker on the **goal**, not on the goal's member spec. `resolveBlockedBy` rewrites the outside dependent's raw `blocked_by` entry to a `kind:"goal"` blocker whose `slug` is the goal's slug (`title` = goal title; `memberSpecSlug` preserves the original raw spec slug for round-tripping through the author-spec re-author path). See [[blocker-goal-normalize]] `deriveEffectiveBlocker` for the pure rule.

**Clear predicate.** A `kind:"goal"` blocker is `cleared:true` iff `goals.main_merge_sha` is set ŌĆö the atomic goalŌåÆmain promotion happened. A goal whose members are all accumulated on `goal/{slug}` but has not yet atomic-promoted to main leaves the outside dependent BLOCKED (its member specs aren't on main yet). The instant `stampGoalPromotedToMain` writes `main_merge_sha`, the goal blocker flips cleared and the reactive auto-queue [[agent-jobs]] `autoQueueUnblockedByGoal` (fired from `promoteCompleteGoalsToMain` right after `applyGoalPromotionEffects`) enqueues every outside dependent whose LAST uncleared blocker was that goal.

**Goal-mates unchanged.** When the dependent AND the blocker are members of the SAME goal, the blocker stays `kind:"spec"` ŌĆö the intra-goal serializer ([[agent-jobs]] `sequencePromoteCandidates` + `areSpecsGoalMates`) still orders it against `specs.goal_branch_sha`. Goal-mate dependencies do NOT flow through the goal-blocker rule.

**Invariant.** EVERY enqueue path applies the SAME `.some((b) => !b.cleared)` predicate (queueRoadmapBuild's blocker gate, `enqueueBuildIfDue`, `autoQueueUnblockedBy` / `autoQueueUnblockedByGoal`, the platform-director autobuild sweeps). A named point-of-truth `isSpecEnqueueBlocked(card)` lives in [[blocker-goal-normalize]] with the Phase 3 tests that pin the invariant ŌĆö so an outside dependent is **never** claimed while its goal is off `main`.

## Invariant ŌĆö no client-side id/slug array batching ([[../specs/retire-residual-in-array-batching-to-server-side-rpcs]] Phase 3)

**All per-slug reads on this file go through server-side RPCs ŌĆö NEVER a client-side id/slug array over the wire.** The 2026-07-08 DB-overload incident (with [[../specs/list-specs-with-phases-rpc-retire-in-array-client-join]] as precedent) traced to `.in("spec_slug", [large-array])` calls whose URL overflowed the ~16KB undici header cap (`UND_ERR_HEADERS_OVERFLOW`) once the workspace held a few hundred specs, wedging the roadmap page load. Every per-slug reader on this file now calls one of the four server-side RPCs above (`list_specs_with_phases`, `roadmap_latest_build_signals`, `roadmap_latest_status_transitions`, `roadmap_latest_needs_fix_reasons`) and intersects with the requested slug set in-memory (bounded by the current board).

**CI-enforced** by `scripts/_check-specs-phases-no-client-in-batching.ts` (chained into `predeploy`): a reintroduced `.in("id", <array>)` or `.in("spec_slug", <array>)` in `src/lib/specs-table.ts` or `src/lib/brain-roadmap.ts` fails the build. Genuinely-bounded LITERAL arrays (`.in("spec_slug", ["a","b"])`) still pass ŌĆö a variable, `.slice(...)` batch, or spread does not.

## Callers

`src/app/dashboard/roadmap/**` (board, `[slug]`, map, goals, functions) ┬Ę `src/lib/roadmap-actions.ts` (the build gate) ┬Ę the spec-test cron ([[../specs/spec-test-agent]]) ┬Ę `src/lib/brain-links.ts`.

## Gotcha

- **`blockedBy` needs the full set** ŌĆö `getSpec(slug)` alone can't resolve whether a blocker has cleared, so a card's `blockedBy` is only meaningful *after* `getRoadmap` (which loads all specs). A single `getSpec` call leaves `cleared:false` for all; the board uses `getRoadmap` so all specs resolve together.
