# libraries/platform-director

The **Platform/DevOps Director agent** — the **first live director** ([[../specs/platform-director-agent]] Phase 1, [[../goals/devops-director]] M4). Investigate every Approval Request **routed to Platform** and either **auto-approve within the leash** (with the reasoning logged) or **leave it for the CEO**. It **supervises** the existing platform tools ([[../specs/repair-agent|repair]], [[../specs/db-health-agent|db-health]], [[../specs/coverage-auto-register-agent|coverage-register]]) — it does **not** rebuild them.

**File:** `src/lib/agents/platform-director.ts`

## Why this exists

North star ([[../operational-rules]] § supervisable autonomy): **CEO → Director → tool**. Platform's tools already work, but nothing supervised them *as a director* — every platform approval the CEO rubber-stamps landed in the CEO inbox. This module is the supervisor: it reuses the [[../specs/approval-routing-engine|approval-routing]] plumbing rather than adding a parallel approval path, and records every call to the supervisable-autonomy ledger so the CEO can audit **what** it decided and **why** — in history, never in the queue.

**Activated 2026-06-23 20:35** (by `ceo`). Platform's [[../tables/function_autonomy]] row is `live + autonomous`, so [[approval-router]] `resolveApprover` now routes platform-owned approvals here and **every surface below is live** — the runtime guards (`platformIsAutoApprover` etc.) currently **pass**. The per-surface "no-op unless live+autonomous" notes below describe the **conditional guard** (what each surface does when the flag is *off*): the code still no-ops if the flag is ever toggled back off ([[../dashboard/agents|Agents hub]]) — it is not a statement that Platform is dormant today.

## The leash (what it MAY auto-approve)

The `goals/devops-director` § leash + the standing autonomy rule ([[../operational-rules]]). A **structural** gate (which action class) **and** a **soundness** gate (the read-only investigation — *never rubber-stamps*):

| `LeashCategory` | Pending-action `type` | What |
|---|---|---|
| `error_fix` | `repair_build` | a repair-agent fix for a real bug |
| `db_health` | `db_health_build` | a DB index / health fix (no destructive DDL) |
| `additive_migration` | `apply_migration` | an **additive, reversible** migration (no DROP/DELETE/data loss) |
| `additive_backfill` | `run_prod_script` **only when bundled with an `apply_migration`** | an **idempotent, re-runnable** backfill that accompanies an additive migration in the SAME request (a standalone prod script is never in-leash) |
| `monitoring_fix` | *(reserved)* | a platform-monitoring registry fix |

**Multi-action (worker-grading P8).** The gate is `directorLeashCandidates` (not the old single-action `directorLeashCandidate`): it auto-approves a request with **one** in-leash action OR a **bundle** where **every** action is in-leash — most often an **additive migration + its dependent idempotent backfill** (the `a2edeca0` / `experiment-session-stamped-attribution` case). A bundle is **all-or-nothing**: one out-of-leash action escalates the whole request, and `applyDirectorApproval` approves all actions atomically (the target flips `queued_resume` only when none stay pending). The soundness gate is unchanged — the investigation must confirm EVERY action (and the bundle as a whole) is sound + reversible.

**Always escalates** (never auto-approves): destructive/irreversible actions, modifying/abandoning a goal, starting a new goal, a non-binary **multi-CHOICE** decision (coverage-register register-vs-exempt, storefront campaign), a standalone prod script, or anything it cannot confirm sound. **CEO escalation routing** is live (Phase 3): an escalation now **re-routes** the Approval Request to the CEO inbox carrying Ada's diagnosis (`escalateApprovalRequestToCeo`), instead of leaving it untouched. See *Loop-guard + CEO escalation* below.

**Founder-prompted out-of-leash actions** ([[../specs/ceo-authorized-out-of-leash-actions]]). The leash blocks AUTONOMOUS out-of-leash action, and Ada can re-route an Approval Request already in the pipeline UP to the CEO. The gap this closes: when the founder asks Ada AD-HOC in the [[director-coach-threads|Ask-Ada chat]] to do something outside her leash (e.g. to unstick a wedged pipeline), there is no pending request to escalate — the leash used to just read as "no". Phase 1 adds the missing **proactive origination**: the founder asks, Ada investigates read-only, and IF she independently AGREES it's sound + the right call she emits ONE `out-of-leash-request` action from her coach reply. That action is **auto-applied inline** (`applyOutOfLeashRequestActionInline` in `scripts/builder-worker.ts` — the same pattern as `spec-status` / `dismiss-park` / `request-audit`): it inserts an `agent_jobs` row (`kind='ceo-authorized-out-of-leash'`, `status='needs_approval'`) carrying (a) her REASONING (why it's right + why it's out of leash + reversibility note) as the pending action's `summary`+`preview`, and (b) a CONCRETE executable action (`type: 'run_prod_script' | 'apply_migration'`, with the exact `cmd`), plus `out_of_leash: true` + `authorized_by: 'ceo-pending'` markers on the pending action. The `preview` ALWAYS includes the exact command line (`$ ${cmd}`) — if the assembled preview would exceed 4000 chars, the trailing user-supplied detail is truncated but the cmd line is always re-appended, so the byte-identical command is never lost. `ceo-authorized-out-of-leash` is deliberately UNMAPPED in `KIND_TO_FUNCTION`, so [[approval-inbox]] `reconcileApprovalInbox` routes it to the CEO (the fail-safe root); the request surfaces in the CEO's normal Approvals inbox with Ada's reasoning + the exact action + the literal command, so the CEO approves with eyes open. Also records a `raised_out_of_leash_request` [[../tables/director_activity]] row. Idempotent (dedupe key `{thread_id}::{actionType}::{cmd}` on `instructions`). If Ada DISAGREES with the founder's ask (unsound / risky / wrong) she emits NO action — the reply is her reasoning + what she'd do instead. That independent-agreement gate is the safeguard that keeps her a supervisor, not a rubber-stamp — the founder's ask is necessary but NOT sufficient. **Never a leash widen** — one-time authorization for THIS action instance; the next out-of-leash ask needs its own CEO approval.

**Phase 2 — CEO approval EXECUTES the out-of-leash action** ([[../specs/ceo-authorized-out-of-leash-actions]] Phase 2). The standard `/api/roadmap/approve` gate ([[roadmap-actions]] `approveRoadmapAction`) flips each `pending_action` to `approved`/`declined` and — once no action is still pending — the job to `queued_resume`. The dispatcher lands `runCeoAuthorizedOutOfLeashJob` (`scripts/builder-worker.ts`): it reads the `out_of_leash: true` pending action off the job and, on `approved`, upgrades the marker `authorized_by:'ceo-pending'` → `'ceo'` (stamps `ceo_authorized_at`) and shell-executes the concrete `cmd` via `shAsync("bash", ["-lc", cmd], { timeout: 10min, cwd: REPO_DIR })` — the SAME executor shape [[../lifecycles/build-lifecycle]] uses for gated `run_prod_script`/`apply_migration` on a build resume. On `declined`, no shell runs — the row is closed cleanly. Job flips terminal: `completed` on approve+clean or decline; `needs_attention` on a shell-failure so the CEO sees the failure (a reversible action is one `git revert` / rollback away). **SCOPED + ONE-TIME + NEVER a leash widen:** the runner NEVER touches [[../tables/function_autonomy]] / the leash categories / any standing rule — the authorization applies to THIS action instance only; the next out-of-leash ask needs its own CEO approval. **AUDITED both sides:** the routed decision is already in [[../tables/approval_decisions]] via `approveRoadmapAction` (`decided_by='ceo'`, `routed_to_function='ceo'`); this runner additionally writes ONE `executed_ceo_authorized_out_of_leash` [[../tables/director_activity]] row on execute (carrying action type/id/cmd, reversibility, irreversibility, outcome ok|failed, result_tail) OR ONE `ceo_declined_out_of_leash_request` row on decline — so every out-of-leash execution is attributable after the fact. **Safety on irreversibility:** an `irreversible: true` action still executes on CEO approval (the CEO is the top authority), but the raise already flagged it so the CEO approves knowingly; prefer a reversible formulation when one exists.

## Escorting goals (Phase 2 — milestone progression within the leash)

`escortApprovedGoals` drives each **approved** goal the director owns forward, **leaning on the existing machinery** ([[../specs/spec-blockers|blocked_by auto-queue]] `autoQueueUnblockedBy` + the builder chain + auto-ship + fold) rather than reimplementing it. The reactive auto-queue fires on a *blocker's merge*; the escort is the **proactive** complement — a throttled box sweep (~5 min) that kicks off the unblocked specs the reactive path never caught and **logs the advance**.

- **Escortable goal** (`isEscortableGoal`) = `GoalCard.status === "greenlit"`, `pct < 100`, AND **either** in progress (`0 < pct < 100`, the original `isApprovedInProgress` leash) **or greenlit-at-0% with ≥1 BUILDABLE spec** (`isBuildableSpec` — reviewed [`specReviewDone`], not deferred, not shipped, `auto_build ≠ false`, unblocked). **Owner-agnostic** ([[../specs/escort-drives-all-greenlit-goals-not-just-platform-owned]]): the escort drives **every department's** greenlit goals, not only Platform-owned ones — Platform is the universal builder (CEO directive; every department's specs land through the same claude/build-* → agent_jobs → merge chain Platform owns). Was: `goals.filter((g) => g.owner === PLATFORM)` — the exact reason the 2026-07-08 autonomous-media-buyer-supervision stall persisted even after the per-spec readiness fixes (blocker-cleared, durable-vale-stamp) landed: the growth-owned goal never entered the escort loop at all. Applies to both `escortApprovedGoals` (primary lane) AND `reconcileReadyGoalMembers` (backstop sweep) — the two decision paths read the same guards. The CEO's greenlight is an **explicit goal state** ([[../specs/director-proposed-goals]] Phase 1), no longer inferred from `pct > 0`. A **`proposed`** goal is skipped — it awaits the CEO via its OWN Approval Request (the `proposed-goal` job), so the escort never re-escalates it. A **`greenlit` 0% goal with NO buildable spec** is *ready for decomposition* (Pia, Phase 2) and is NOT auto-started — the escort never starts a goal that has nothing to build. But a **`greenlit` 0% goal that ALREADY HAS reviewed + planned specs** (authored directly, or after Pia decomposed) is genuinely ready: the escort now ENTERS it and queues those specs (`goal-escort-ready-at-0pct`). WHY the gap mattered: the old `pct > 0` gate excluded an at-0% goal whose specs were already buildable, so it sat forever and its specs never built (`noop-goal-test`). This also **replaced** the old "every 0% owned goal escalates as a `new_goal` greenlight request" — a proposed goal is now a self-surfacing artifact, a greenlit-0% goal is either awaiting-Pia (no specs) or ready-to-build (decomposed).
  - **Goal→spec resolution fix (same change):** the escort's `goalSpecs` reads `GoalCard.milestones[].specSlugs`. That field came **only** from `**[[../specs/x]]**` wikilinks in the milestone body — but the canonical PM-flow links specs **relationally** via `specs.milestone_id` (Pia's decomposition + the structured author bind it; milestone bodies are prose like "Holds spec X." with NO wikilink). [[brain-roadmap]] `milestoneRowToCard` now **unions the relationally-`linked` spec slugs** into `specSlugs` (not just wikilinks), so the escort — and the goals dashboard's per-milestone spec list — see the relationally-linked specs. Without this, `goalSpecs` returned empty even for a fully-decomposed goal.
- **What it queues:** every spec linked across the goal's milestones that is **unshipped + unblocked** (all `blockedBy` cleared) + has **no build job yet**, as a `kind='build'` `agent_jobs` row (`created_by=null` — the same agent-enqueue shape as `autoQueueUnblockedBy` / `queueNextChainedPhase`, so the chain/auto-ship/fold pick it up unchanged). Skips blocked specs (the reactive auto-queue handles those on their blocker's merge) and `**Auto-build:** off` opt-outs.
- **Blocker-cleared predicate is grounded in [[spec-phase-provenance]] — NOT `target.status === "shipped"`** ([[../specs/escort-reliably-dispatches-ready-goal-members]] Phase 1). Every escort readiness scan (this lane, the reactive `autoQueueUnblockedBy`, and the Phase-3 platform-driver sweeps) reads `card.blockedBy[].cleared` from [[brain-roadmap]] `resolveBlockedBy`. The predicate now combines two spec-phase-provenance helpers so a **goal-mate blocker that accumulated on the goal branch** counts as cleared, matching what the intra-goal serializer sees:
  1. **`isCardShippedByPhaseProvenance(target)`** — every non-rejected phase carries `pr` OR `merge_sha` (or the one-shot card-level `shippedPr`). The blocker actually landed on `main` — cleared for anyone.
  2. **`isCardAccumulatedOnGoalBranch(target)`** (goal-mates only) — `specs.goal_branch_sha` is stamped, i.e. the blocker's spec branch merged into `goal/{goal-slug}` (M4). The goal branch now carries this diff, so a goal-mate dependent can build off it — even though the derived card status stays `in_progress` until the whole goal ships to main.
  **The stall this closes:** shadow-mode's spec branch merged into `goal/autonomous-media-buyer-supervision` (goal_branch_sha stamped) on 2026-07-08 with every phase `build_sha`'d, but its goal-mate dependents (`daily-cadence-cron` / `director-slack-digest`) never dispatched — the old `target.status === "shipped"` predicate demanded main-merge, which for a branch-flow spec is the atomic goal→main promotion. The dependents stalled indefinitely until manually escorted. Outside-of-goal dependents are still gated correctly: their blocker is normalized to `kind:"goal"` (by [[blocker-goal-normalize]] `deriveEffectiveBlockers`) with `cleared` keyed on `goals.main_merge_sha` — untouched by this change.
- **Audited + idempotent:** one `escorted_goal` [[../tables/director_activity]] row per goal it advances (`autonomous=true`, naming the goal + queued slugs); a spec that already has a build job is **confirmed in-flight**, never re-queued. **Dormant** (a no-op) until Platform is live+autonomous (Phase 4), exactly like the approval enqueuer.
- **Backstop reconcile sweep** — `reconcileReadyGoalMembers` ([[../specs/escort-reliably-dispatches-ready-goal-members]] Phase 2). Runs on the SAME standing tick RIGHT AFTER `escortApprovedGoals`, so the primary lane's just-queued rows are visible via `specBuildState` and skipped by the sweep's `inFlight` guard — meaning anything this sweep still catches is a MISS by the primary path (a race, a partial failure mid-loop, a fresh drift the Phase-1 blocker-cleared fix didn't cover). Per approved-goal member: apply the SAME static readiness set as the primary lane via `isReadyForBackstopQueue` (isBuildableSpec ∧ `state.inFlight === false` ∧ `state.failedCount < LOOP_GUARD` — the pure predicate is exported so a unit test drives the exact decision without a Supabase seam); on a hit, queue ONE `kind='build'` `agent_jobs` row (`created_by=null`, distinctive `instructions` labelling the sweep as "Backstop-reconciled") + `markLeafPhaseInProgress` (mirror the primary escort so the board flips) + write ONE `reconciled_ready_goal_member` [[../tables/director_activity]] row per queue. **Idempotent by construction** — the `inFlight` guard reads `agent_jobs` fresh each call, so a spec the primary just queued (or the sweep itself queued last tick) is immediately visible and skipped; running the sweep twice in a row queues at most ONE build. **Distinct audit trail** — the `reconciled_ready_goal_member` action_kind (vs `escorted_goal`) means the CEO can measure how often the backstop is needed; a persistent uptick is a signal the primary lane has a bug OR a new class of drift crept in. Same self-healing philosophy as [[../specs/builder-worktree-prune-before-add]] (catch state that MIGHT be already handled, apply the corrective action idempotently, log it). **Dormant** (a no-op) until Platform is live+autonomous, mirroring every other escort lane. **The stall this closes** — the 2026-07-08 shadow-mode incident (Phase 1 fixed the blocker-cleared predicate; Phase 2 ensures a similar class of miss self-heals within one standing tick even if a future predicate drift or race were to re-open the gap).

## Loop-guard + CEO escalation (Phase 3 — the high-stakes calls always route UP)

The leash is hard, so a high-stakes call **always escalates to the CEO** — it is never rubber-stamped, and a failing build is never resubmitted forever. Every escalation reuses the **existing M2 inbox** (a routed Approval Request notification — the inbox API shows an item to a role iff `metadata.routed_to_function === role`), never a parallel inbox.

- **Loop-guard.** The escort tracks each escorted spec's build state (`specBuildState` — recent `agent_jobs` `kind='build'` rows for the slug: is one **active/landed** → leave it; how many **`failed`/`needs_attention`** → the failure count + latest error). On a sweep, per unblocked spec: an in-flight build ⇒ confirm, don't re-queue; **≥ `PLATFORM_DIRECTOR_LOOP_GUARD_MAX` (2) failures** with nothing in-flight ⇒ **stop + escalate** ("likely a deeper issue"), never re-queue; a single prior failure ⇒ a bounded **retry** (`instructions` note `re-attempt #2`); no builds yet ⇒ the Phase-2 gap-fill queue.
- **BOUNCE an unsound repair fix → the Repair agent (NOT the CEO)** (director-supervises-repair). When investigating a `repair` target, if the bug is real but the AUTHORED FIX is unsound (broken mechanism / mis-scoped / the code contradicts its premise), the verdict is **`bounce`**: `bounceRepairToAgent` ([[repair-agent]]) resolves the current repair job and re-enqueues a fresh repair for the same signature carrying Ada's `directorFeedback` (the re-authoring pass reads it + fixes its work). It never reaches the CEO — `escalate` is reserved for a genuine out-of-leash/irreversible call, not a fix-quality problem. Logs `repair_bounced` [[../tables/director_activity]] + a 🔁 board post. The north-star supervisory loop (Director → tool): Ada doesn't do Repair's job, she hands the work back with feedback.
- **Re-route a declined request → CEO** (`escalateApprovalRequestToCeo`). The box lane's two escalate branches (out-of-leash / multi-choice, and the investigation `escalate`/ambiguous verdict — e.g. a **destructive** migration the runner won't auto-approve) now flip the target's routed Approval Request to `routed_to_function='ceo'` and prepend Ada's diagnosis to the body, so the **CEO** inbox shows it and Platform's no longer does. The target job stays `needs_approval` (never auto-approved); the CEO approves/declines via the normal `/api/roadmap/approve` gate. Creates a CEO-routed request if the reconciler hadn't emitted one yet (idempotent on `agent_job_id`).
- **Standalone diagnosis → CEO** (`escalateDiagnosisToCeo`). A high-stakes call with **no approvable target** — a loop-guard "deeper issue," or a **zero-progress owned goal** (only the CEO greenlights a NEW goal). Emits a CEO-routed Approval Request (no inline approve — it deep-links the CEO to the spec/goal) **and** an `escalated` [[../tables/director_activity]] row. Carries **no `agent_job_id`** so [[approval-inbox]] `reconcileApprovalInbox` — which dismisses any request whose job left `needs_approval` — never reaps this standalone escalation.
  - **Reliable surface (Phase 1).** Order is **notification-first, activity-second, both checked**: it inserts the [[../tables/dashboard_notifications]] row, **checks that insert's `error`**, and only on success records the `escalated` ledger row + returns `{ emitted:true }`. If the notification insert fails it returns `{ emitted:false, error }` (the caller logs a hard `console.error`) and writes **no** phantom `escalated` activity row — so the audit ledger never claims an escalation that the inbox never showed. (Before this, the insert was unguarded: a failed insert still logged `escalated` and returned `emitted:true` — logged-but-invisible.)
  - **Dedupe keys on a notification that ACTUALLY EXISTS.** The one-per-`dedupe_key` dedup now queries [[../tables/dashboard_notifications]] (`type='agent_approval_request'`, `metadata->>dedupe_key`), **not** the `director_activity` ledger. A logged-but-unsurfaced escalation (activity row, no notification) therefore does **not** suppress the retry — the next pass re-emits the missing notification. Once the notification exists (even dismissed/read) it pings once. Keys: `loopguard:{slug}` / `groom-unsure:{slug}` / `groom-loopguard:{slug}` / `init-unsure:{slug}` / `initguard:{slug}`. (The `newgoal:{goalSlug}` key is retired — director-proposed-goals replaced the zero-progress escalation with the explicit `proposed`→`greenlit` goal state.)
  - **needs_attention park escalation — per-PR for pr-resolve (2026-07-03).** The triage of parked jobs escalates each with `needsattn:{job_id}` — EXCEPT `pr-resolve`, which keys on `needsattn:{spec_slug}` (a pr-resolve job's slug is `pr-<n>`, so this is **per-PR**). WHY: a superseded/un-mergeable PR gets its pr-resolve job re-queued repeatedly, and a per-job key surfaced ONE CEO card per retry — two superseded PRs (#1070/#1071) had produced 11+ near-identical "Parked pr-resolve" cards. Per-PR keying collapses every retry against one PR to a single inbox card.

## Watch the platform + report to the board (Phase 4 — the human-legible top layer)

The director's top layer is **reporting up in human terms** — and it **supervises, doesn't rebuild**: it reads the *existing* [[../dashboard/control-tower]] snapshot rather than standing up new monitoring, and reuses the *existing* board/recap/dev-ask plumbing rather than a parallel path.

- **Daily board watch** (`postPlatformWatchUpdate`). Reads [[control-tower]] `buildControlTowerSnapshot` → the **platform department rollup** (worst-of color, healthy/total, open alerts) + its red loops, plus today's [[../tables/director_activity]] (squashed = `approved_approval`, escorting = `escorted_goal`, escalated = `escalated`; **+ the repair-review tallies** — `dismissed_repair` / `kept_repair` / repair-suspect `escalated`), and posts ONE conversational `update` as 🛠️ Ada to the [[../tables/director_messages]] board (`composePlatformWatchBody` — *"🛠️ Platform watch — all 9 platform loops green. Today: squashed 2 fixes · escorted 1 goal. Reviewed 3 of Rafa's calls — dismissed 2, escalated 1 back to you. Drove 3 specs for other departments (cs: 2, growth: 1). Errors tonight: 2 fixed & shipped, 1 dismissed-benign, 1 reconciled from backlog, 0 escalated to you as external."* — the trailing repair-review sentence appears only on a day she reviewed ≥1 of Rafa's calls; **the keystone-cover sentence** "Drove N specs for other departments (…)" appears only on a day she escorted/initiated/groomed ≥1 cross-dept spec, [[../specs/director-drives-all-specs-and-deferred-status-board-reflects-cross-dept-drive]] Phase 1; **the error-autonomy F/D/R/E sentence** ([[../specs/director-zero-backlog-error-autonomy-visible-reversible]] Phase 1) appears only on a day the error pipeline did ANY work — the supervisable-autonomy visibility layer over [[repair-agent]] + `reconcileErrorBacklog` + `superviseRepairDismissals`. Each F is a `claude/<slug>` PR with the full verification trail (tsc + CI + spec-test), so any one is one `git revert` away — the CEO gets after-the-fact visibility AND an instant undo without being in the loop up front). **Idempotent** per (workspace, UTC day) on `metadata.watch_date`; skips a fully-quiet all-green day (no empty-board spam); **dormant** until live+autonomous like the escort.
- **"Answers why?"** — no new code: the [[../specs/directors-board-gamified]] Phase-2 board→dev-ask wiring ([[director-board]] `routeBoardReply`) already defaults the answer brain to Platform, so a "why?" reply under Ada's post routes to a `dev-ask` box turn that posts the answer back in-thread.

## Board grooming (Phase 5 — the director MOVES the project board, [[../specs/board-grooming]])

The director doesn't just build queued specs — it **grooms the board** so nothing rots half-built. On its standing cadence it assesses every **partially-built** spec (≥1 phase **branch-built**, remaining ⏳, **no active build**) and decides what to do with the leftover phases — *moving the card* accordingly. This formalizes the need-now-vs-future call the operator made by hand (and replaces the escort's blind "re-queue every phase").

> **spec-goal-branch-pm-flow M2/M4/M5 alignment.** Under branch-accumulation a phase builds onto `claude/build-{slug}` with `build_sha` set + status `in_progress` (NOT shipped) — it earns a `pr` tag only at M5 promotion. So "started / partially-built" is keyed on **`branchBuiltCount` (build_sha or shipped), NOT `provenanceShippedCount`/`counts.shipped` (the pr-tag read, which is 0 for a whole branch-flow spec until promotion)**. `findGroomCandidates`, `findInitCandidates`, AND `escortFixSpecs` all gate on `branchBuiltCount` (the grooming/init lanes rewired in M2; the fix-escort aligned in the goal-branch audit). The grooming "continue" lane queues the next phase off the spec branch (**branch-build, never main-merge**). The shipped stamp + the fold trigger are owned by M5's `applyGoalPromotionEffects` (goal-bound) / `applyMergedBuildEffects` (one-off) — Gate-B (`promoteEligibleSpecsToGoalBranch`) + Gate-C (`promoteCompleteGoalsToMain`) run in Ada's standing pass.

- **The decision (the Max `claude -p` investigation, `groomInvestigationPrompt`):** per partially-shipped spec — **continue** (the next ⏳ phase is needed now: the spec's current promise / a dependent / a goal needs it → queue its build to completion, the chain carries it) · **split** (the leftover phase(s) are future enhancement/polish → author EACH as its own planned card `{slug}-{phase}.md` with a `**Deferred:**` note, then CLOSE OUT the parent — remove the split phases so its remaining phases are all-✅ → the parent **folds**) · **escalate** (genuinely unsure / possibly load-bearing → route the diagnosis to the CEO, move nothing — north-star: hit a rail → escalate).
- **Net effect:** every partially-shipped spec either **completes** (needed-now phases built) or **cleanly splits** (future phases become their own planned cards) — cards flow *out* of "In progress," the future work is captured (not lost) in "Planning." Future work is **never dropped** — a split always preserves the phase as a planned card; the director never deletes a phase.
- **Supervisable + audited:** splitting + queueing a next-phase build is low-risk/reversible (within the leash); every decision writes a [[../tables/director_activity]] row (`groomed_continue` / `groomed_split` / `escalated`) with the reasoning. **Dormant** until live+autonomous, like the escort. **Loop-guarded** (a continue whose build already failed ≥ `PLATFORM_DIRECTOR_LOOP_GUARD_MAX` with nothing in-flight escalates instead of resubmitting). **A malformed split never lands** (`validateGroomSplit` gates the commit — a broken board is worse than an un-groomed card; it escalates instead).
- **Idempotent:** a `continue`'s queued build flips the spec in-flight (the candidate filter excludes it next pass); a `split`/`escalate` writes a `groom_key` (`groom:{slug}`) the next pass dedups on via `alreadyGroomed` — this is what survives the box's bundled `fs` copy lagging `main` until its next self-update (without it the same card would re-split every pass).
- **Mechanical-advancer skip ([[../specs/ada-standing-pass-reasoning-gate]] Phase 2 — the reasoning gate on the `continue` verdict).** `backstopPhaseProgression` runs **earlier** in the same standing pass and DETERMINISTICALLY queues the next planned phase for every partially-built spec whose gates pass — which makes a groom `continue` on that same set a redundant Max session (same advance, ~600K tokens/session, 92% of it context re-hydration). The standing pass now captures the backstop's `advanced` slug set and hands it into `groomBoard` as an explicit pre-filter: any candidate whose slug is in that set is **skipped BEFORE the Max session fires** — logged as `skipped-by-backstop N` on the pass recap so the 0-token skip is visible (not a silent drop). This is additional to `findGroomCandidates`'s existing `state.activeBuild` guard (which would ordinarily also exclude a just-queued spec) — the explicit hand-off closes the race between backstop's DB insert and grooming's read. Preserves the `alreadyGroomed` ledger + the `min(PLATFORM_DIRECTOR_GROOM_CAP, idleBuildCapacity)` capacity cap unchanged; **only trims the `continue` overlap** — split / fold_now / author_followup_spec / dismiss_candidate / escalate all still run. **Phantom/future leftover:** the backstop queues them too (it can't tell phantom from real), so the same-pass skip elides an over-eager grooming decision; the build lands as a no-op / fail, and **the next pass re-surfaces** the spec (backstop's scoped-instruction dedup blocks a re-queue, `alreadyGroomed` is still false, and grooming's judgment lane then produces the `fold_now` / `split` verdict). Fail-open — if `backstopPhaseProgression` throws, the skip set is empty and grooming runs unchanged.

## Shepherding a multi-phase spec to completion (the phase-progression heartbeat)

**Ada is the shepherd of a spec to completion.** A multi-phase spec builds **one phase per commit** on `claude/build-{slug}`: phase N builds → its `build_sha` is stamped (`in_progress`, not shipped) → phase N+1 must be queued onto the same branch tip, and so on until every phase is built and the spec promotes. Ada owns that P1→P2→…→Pn progression — she **starts P1, advances P2 when P1 is built, P3 when P2 is built**, etc. There are **two triggers**, primary + backstop, both leaning on the SAME `agent-jobs` chain helper `queueNextChainedPhase` (which reads the spec's derived phases via `getSpec(slug, workspaceId)`, picks the **first `planned`** phase = accumulation order, dedupes on the scoped-instruction + any in-flight build, and inserts a `chain_phases:true` build onto the branch):

- **Primary — the REACTIVE chain.** `runBuildJob` (`scripts/builder-worker.ts`) calls `queueNextChainedPhase(job.workspace_id, slug)` the instant phase N's branch-build **completes** (un-gated from the legacy `chain_phases` "Build all" flag in spec-goal-branch-pm-flow M2; workspace-correct). This is the fast path — the next phase queues within seconds of the prior one landing.
- **Backstop — the standing-pass `backstopPhaseProgression`.** The reactive chain is **event-only**, so it MISSES on a worker crash between build-complete and the chain insert, a director-initiated single build with the chain off, or a transient DB hiccup — leaving the spec **stalled mid-accumulation**. This is the SAME event-only fragility Gate-A, the pre-merge legs, and the spec-review backstop each had, and it gets the SAME reliable-heartbeat fix: a cheap, un-gated, **always-runs** mechanical advance in `runPlatformDirectorStandingPass`. Each pass it scans every partially-built spec Ada drives (`branchBuiltCount ≥ 1` AND `counts.planned ≥ 1`) whose next planned phase has **no live build** and queues that phase via `queueNextChainedPhase`.
  - **Why a backstop is needed even though grooming exists:** the GROOM lane's *continue* verdict also advances the next phase — but grooming is a Max `claude -p` **judgment** lane (capacity-gated by `idleBuildCapacity`, deduped by `alreadyGroomed` so a prior split/escalate suppresses re-firing, and free to decline). So a spec whose reactive chain missed AND whose groom verdict wasn't *continue* would sit half-accumulated forever. The backstop is the mechanical floor under the judgment lane — no Max investigation, pure advance.
  - **Respects the rails:** the **spec-review gate** (`specReviewDone(card)` — the `vale_review_passed_at`-backed durable signal — so an `in_review` / un-Vale-passed spec is NEVER advanced); **one-phase-per-session** (exactly one `phaseScopedInstructions` phase per spec per pass; the worker chains the next); the **build-driver keystone** (`platformDrivesSpec` — Ada drives every spec while Platform is live+autonomous; owner is attribution, not the build driver); **accumulation order** (the helper picks the lowest planned phase); and **dedupe** (a precise `specBuildState.activeBuild` pre-filter PLUS the helper's own in-flight + scoped-instruction check — a phase already queued/building is never double-queued; a *landed* completed/merged build does NOT block, because that's exactly the case the reactive chain may have missed). Skips blocked / `**Auto-build:** off` / deferred / fully-shipped specs (mirrors the escort + groom filters).
  - **Audited + idempotent:** one `escorted` [[../tables/director_activity]] row (`metadata.lane='phase_progression_backstop'`, naming the advanced phase) per advance; a no-op when there's nothing to advance; best-effort per spec; never throws. **Dormant** until live+autonomous, like every other lane.

## Supervise + dismiss Rafa's no-fix items ([[../specs/director-supervised-repair-dismissal]] Phase 1 + 2)

The CEO used to manually **Dismiss** every Control Tower warning where the [[repair-agent|Repair Agent]] (Rafa) declined to propose a fix — a `needs-human` verdict parked as a `repair` [[../tables/agent_jobs]] in `needs_attention`, surfaced **Dismiss-only** by `getOpenRepairs`. This lane is the director **supervising** that no-fix call. It is **not** a janitor: Rafa optimizes the bounded proxy *'clear the error'*, whose degenerate state is clearing a warning by declaring a **real bug** benign ([[../operational-rules]] § North star). So Ada **adversarially re-checks** his verdict and dismisses ONLY what she can **independently confirm** is benign.

- **The decision (the Max `claude -p` investigation, `repairDismissalInvestigationPrompt`):** per `needs-human` item — Ada loads the originating [[../tables/error_events]] / [[../tables/loop_alerts]] sample + Rafa's logged no-fix reasoning, **independently re-derives the root cause**, and emits one JSON verdict. The stance is **hard — DEFAULT to not-dismissing**: **dismiss** ONLY if she can independently confirm the error is genuinely transient / foreign-app-noise / benign and NOT a masked real bug · **escalate** if she suspects a real bug **in OUR code** Rafa mislabeled (her contrary diagnosis routes to the CEO via `escalateDiagnosisToCeo`) · **external** if the verified root cause is **OUTSIDE our system** ([[../specs/director-zero-backlog-error-autonomy]] Phase 2, below) · **keep** a genuine needs-human call (leave it on the Control Tower untouched for the human).
- **The external-blocker escalation class (Phase 2) — the ONLY routine error→CEO touch.** When Ada's verified diagnosis is an **external dependency break** — a third-party API contract change, a vendor outage beyond our retry/breaker, or a credential/permission change on their side — she does **NOT** author a code fix (we can't code-fix someone else's system). She `escalateDiagnosisToCeo` with the diagnosis **+ 2–3 concrete ALTERNATIVE options** (wait/retry the vendor, swap provider, degrade that path gracefully — the box lane formats `parsed.alternatives` into a numbered "Your options:" block), `escalationKind='external_blocker'`, deep-linked to the Control Tower. **Deduped per signature** on `externalBlockerKey(signature)` = `external:{signature}` (distinct from the suspected-real-bug `dismiss:{signature}` key) so it pings the CEO **once**. Everything internally-fixable — a real bug in our code, a supervised-dismiss, an authored+built fix — is handled **without** the CEO. If the notification is already surfaced (or fails to insert), the box lane records a `kept_repair` review row so the same job isn't re-investigated each pass.
- **Dismiss reuses the EXISTING owner Dismiss plumbing** — `applyDirectorDismissal` mirrors `POST /api/developer/control-tower/repair` `dismiss` (decline the `repair_build` action, complete the job, resolve the `error_events` row) **minus the owner auth gate** — no new dismiss machinery, **no migration**. It then writes a `dismissed_repair` [[../tables/director_activity]] row carrying **Ada's OWN** independent reasoning (not a copy of Rafa's).
- **Leash:** dismissing a confirmed-benign monitoring warning is the **`monitoring_fix`** class — low-risk + reversible (a dismissed `needs_attention` item UN-blocks re-enqueue, so a wrongly-dismissed real problem re-fires and Rafa re-triages it). **Unsure ⇒ escalate, never dismiss.** NEVER dismisses a `real-bug` / fix-proposed item: `findRepairDismissalCandidates` only takes `state === "needs-human"` items, and `applyDirectorDismissal` **re-asserts** the job is still `needs_attention` before clearing it.
- **Idempotent + loop-guarded:** every review writes a ledger row carrying `metadata.repair_job_id` + `dismiss_key` (`dismiss:{signature}`); `alreadyReviewedDismissal` dedups on the **job id** so each item is reviewed once — a `keep`/`escalate`/`external` leaves the job open but is not re-investigated, while a **re-fire (a NEW repair job, same signature)** is a fresh review. An `external` verdict additionally dedups its CEO notification on `external:{signature}` (so even across re-fires the CEO pings once per live external blocker). **Dormant** until live+autonomous, like every other lane. Bounded by `PLATFORM_DIRECTOR_DISMISS_CAP` (6) per pass.
- **Phase 2 — surface the dismissal + a CEO re-open override.** A dismiss completes the job, so it leaves `getOpenRepairs` and would silently vanish — Phase 2 keeps it **visible**. `getDirectorDismissedRepairs` ([[repair-agent]]) reads the recent `dismissed_repair` ledger (title now carried in its metadata) minus any re-opened, and the [[../dashboard/control-tower]] **Repair feed** renders each as `🛠️ Dismissed by Ada — <reasoning>` with a one-tap **Re-open** (`POST /api/developer/control-tower/repair {action:'reopen'}`, owner-gated — re-opens the `error_events` row, re-enqueues Rafa, logs `reopened_repair`). The **daily board watch** appends a rollup sentence — *"Reviewed N of Rafa's calls — dismissed K, escalated J back to you."* — on any day she reviewed ≥1 (counting `dismissed_repair` + `kept_repair` + the repair-suspect `escalated` rows).

## `needs_attention` is a transient classification gate, not a state ([[../specs/no-parked-specs-auto-route-needs-attention]])

The CEO's rule: a spec is either building or being fixed via a new spec / new phase. **Parking is a bug.** `needs_attention` is therefore the **gate** every build flows through on the way to a routed action — never a resting state. Every park is classified ([[needs-attention-classify]]) into one of `already_shipped｜real_blocker｜tooling_failure｜design_change｜unknown` and the routing sweep ([[needs-attention-route]] `routeNeedsAttention`, wired into the standing pass next to `reconcileNeedsAttention`) dispatches:

| Class | Route | What |
|---|---|---|
| `already_shipped` | **AUTO-FOLD** | flip the spec to shipped (`markSpecCardStatus` + spec_status_history actor=director:platform) + `enqueue_fold` — the spec rides the next batch fold-build. |
| `real_blocker` / `tooling_failure` | **AUTO-SPEC** | author a child spec `{slug}-fix-{class}-{shortid}` owned by Platform, parented at the origin; origin's card flips `blocked` + carries `blocked_by_child`. The standing build escort picks up the child next pass. |
| `design_change` | **INVITE TO CHAT** | `escalateDiagnosisToCeo` with `metadata.chat_mode_request=true` so the [[../specs/ada-slack-routed-approvals]] Phase 3 dispatcher posts a `#cto-ada` chat-mode invitation instead of an Approve/Reject card — the only park class that legitimately surfaces to the CEO. |
| `unknown` (or null) | **BACKSTOP** (Phase 4) | a parked row >60 min still unclassified is re-classified once; a still-`unknown` verdict surfaces to the CEO with a clear diagnosis. A row >70 min in `needs_attention` (regardless of class) posts a `dashboard_notifications` `type='system'` invariant alarm. Goal: that alarm fires zero times. |

**Non-spec kinds short-circuit to dismiss** ([[../specs/ticket-improve-park-auto-route]]). The four class routers + the backstop all key off a real spec (fold flips the card, child-spec parents at the origin, chat-mode talks about the spec, the 70-min alarm protects "a spec sitting in needs_attention"). A `ticket-improve` job — a one-shot ticket co-pilot turn whose `spec_slug` is actually the chat-session id, not a spec — has no spec route to take, so a park would otherwise fall to `unknown`, sit 60 min, surface to the CEO for nothing the CEO can do about it, and trip the 70-min alarm against a non-existent spec. `NON_SPEC_KINDS` (currently `{ticket-improve}`) is dispatched BEFORE the class routers and the backstop: the row is **dismissed** via the same shape as `applyDismissParkActionInline` (status='dismissed' + needs_attention_class='dismissed_by_director' + a `dismissed_park` [[../tables/director_activity]] row + a `routed_needs_attention` ledger row), so the activity feed renders consistently with manual dismisses and the existing `POST /api/developer/director-activity/reopen-park` reopen flow works unchanged. Extend the set when a new non-spec job kind goes live.

**Dedupe + idempotency.** Every routing action writes a `routed_needs_attention` / `routed_park_alarm` [[../tables/director_activity]] row (`metadata.job_id`) AND stamps the job's `needs_attention_class` to a `routed_*` marker. **The next sweep CLEARS a `routed_*` row to terminal** (`clearRoutedZombie` → `status='completed'`, the same terminal a superseded/held build uses) instead of leaving it parked — routing IS the disposition (the fold / child-spec / chat followup is the live surface now), so the park has no further role. Leaving it parked made it a **zombie** that self-watch ("build stuck >90m") and `reconcileNeedsAttention` ("N parked, all triaged") re-reported on every standing pass forever (the `appstle-switch-payment-method-edit-guardrail` board-spam loop, 2026-06-26). (Non-spec dismissals stamp `dismissed_by_director` — the dismissed status filter already excludes the row, and the ledger row is belt-and-suspenders.) Bounded by `PLATFORM_DIRECTOR_ROUTE_CAP` (8) per pass; **dormant until live+autonomous** (same guard as every other standing lane). The classifier itself runs at park time (the single `update()` chokepoint in `scripts/builder-worker.ts`) so every parker — build, repair, regression, security, spec-test — stamps with no per-callsite plumbing.

**Sibling, not replacement, of `reconcileNeedsAttention`.** The existing triage lane handles the narrow "inconclusive QC verdict — re-run once" case (security-review/spec-test/regression). The class router handles every OTHER park class so no `needs_attention` row sits as ghost work after a re-run that parked again.

## Director coach cards — the chat surface ([[../specs/ada-director-spec-status-cards]])

Ada's coach chat (`/api/director/coach` + the `runDirectorCoachJob` box lane) emits one of a fixed set of **`pending_action`** card types — the CEO approves each, the worker executes. Every card stops at the leash; Ada never mutates anything herself. **One exception:** `spec-status` is **auto-applied** — see below.

| `type` | Resolution | What it does |
|---|---|---|
| `coaching` | CEO approval | Writes a durable coaching rule injected into Ada's future decisions ([[director-instructions]]). |
| `spec` | CEO approval | Commits a new `docs/brain/specs/{slug}.md` (the worker `putFileMain`s it). |
| `spec-edit` | CEO approval | Re-commits an EXISTING `docs/brain/specs/{slug}.md` with revised markdown. **Never** carries status (status is DB-only). |
| `spec-status` | **Auto-applied** — owner-only leash | Flips a spec on the BOARD (status / per-phase / `flags.critical` / `flags.deferred`) via the existing `markSpecCard*` writers + a `spec_status_history` row per field (`actor=director:{fn}`). No markdown commit, no inbox card, no Slack approval thread. ([[../specs/ada-director-spec-status-cards]]) |
| `goal` | CEO approval | Commits a proposed goal artifact + surfaces it for the CEO's greenlight ([[../specs/director-proposed-goals]]). |
| `directive` | CEO approval | Becomes the active directive (gates routine builds, marks critical specs, queues priority builds, cancels held). |
| `model_tier` | CEO approval (auto-applied within rail) | Routes a model-tier change to the target agent's supervisor ([[../specs/box-agent-model-tiers]]). |

### `spec-status` is auto-applied — why, and the rails

Status tracking is bookkeeping over already-shipped reality. Every flip is reversible, and the daily drift reconciler is the safety net. Accountability is the audit trail (`spec_status_history` row stamped `actor=director:{fn}` with a `reason`) **plus** a `director_activity` row (`action_kind="spec_status_flipped"` or `"invalid_spec_status_action"`) — not a per-flip gate. The leash holds because the underlying code/build approvals already happened — the row is just catching up to reality.

The rails (enforced in `applySpecStatusActionInline` inside `scripts/builder-worker.ts`):

- **Owner-only.** The spec's `**Owner:** [[../functions/{fn}]]` line MUST equal the emitting director's function. A flip on a spec another function owns is rejected as out-of-leash and logged with `invalid_spec_status_action` — cross-function flips still go through the spec's actual owner or the CEO owner UI.
- **Existence.** The slug must resolve to a `spec_card_state` row OR a `docs/brain/specs/{slug}.md` file. Phantom slugs are rejected.
- **Shape.** At least one of `status` / `phases` / `critical` / `deferred` must be set. `phases[].index` is 0-based and validated against the markdown's parsed phase count.
- **Reason required.** Stored verbatim on every `spec_status_history` row the flip writes.
- **Reversibility backstop.** [[spec-drift]] auto-reverts a director-stamped `shipped` row whose phase code isn't on `main` within ~24h and re-logs the revert to `director_activity` so a recurring mis-flip pattern surfaces on the daily watch.

**Worked `spec-status` example** — Ada notices `chat-fallback-absorbed-anthropic-overload-noise` is fully merged (PR #442) but its `spec_card_state` row still reads `planned`, and she wants to mark it shipped and clear the `**Priority:** critical` flag set by a prior directive:

```json
{
  "type": "spec-status",
  "summary": "chat-fallback-absorbed-anthropic-overload-noise merged — flip to shipped + clear priority",
  "slug": "chat-fallback-absorbed-anthropic-overload-noise",
  "status": "shipped",
  "critical": false,
  "reason": "PR #442 merged 2026-06-21; phases 1+2 backed by code on main. Clearing the critical flag set by the prior directive."
}
```

The moment the chat reply is persisted, the worker calls `markSpecCardStatus(workspaceId, slug, "shipped", existingPhaseStates, { actor:"director:platform", reason })` + `markSpecCardCritical(workspaceId, slug, false, { actor:"director:platform", reason })` — two `spec_status_history` rows, one `director_activity` row (`spec_status_flipped`), no markdown commit, no deploy, no CEO inbox card. Ada mentions the flip in her reply text so the CEO sees it in chat. If she got it wrong, the daily drift reconciler reverts the row within 24h and stamps a `reverted_director_flip` activity entry.

A `phases[]` entry flips just one phase by index (0-based, validated against the markdown's actual phase count); the rollup is recomputed via `rollupPhaseStatus` unless `status` is also supplied. `deferred:true` flips the parked flag (wins over phase progress for display via `effectiveStatusFromState`).

### `shortCircuit` — close a spec cleanly that we no longer need ([[../specs/director-dismiss-park-and-short-circuit-spec]] Phase 2)

`spec-status` carries a `shortCircuit:true` flag (paired with a required `reason`) for the "we changed our mind, this isn't needed anymore" case — closing a spec cleanly **without all phases shipping**. `rejected` reads as "the spec was wrong" and `shipped` implies the phases actually built; neither fits. Schema validation rejects `shortCircuit:true` without a `reason` and rejects `shortCircuit:true` with any `status` other than `'shipped'` (a bare `{shortCircuit:true,reason:…}` payload auto-forces `status:'shipped'`). `applySpecStatusActionInline` then flips status via `markSpecCardStatus` and stamps `flags.short_circuit=true` + `flags.short_circuit_reason` via `markSpecCardShortCircuit` — the **roadmap card** renders the spec as **shipped + short-circuited** with the reason in a sub-line, so the next reader doesn't think we built it. The fold-build is **skipped** (there's nothing to fold — the brain pages weren't updated by this; the spec + skill stay intact as reference). Reversible: an owner status flip back to `planned` (or a director `shortCircuit:false` action) clears the marker.

### `dismiss-park` — close a stale `agent_jobs` park ([[../specs/director-dismiss-park-and-short-circuit-spec]] Phase 1)

When a parked `agent_jobs` row is genuinely not worth pursuing (the underlying work is being short-circuited, a prereq won't be supplied, the park is stale and not auto-routable by [[../specs/no-parked-specs-auto-route-needs-attention]]), the director emits a `dismiss-park` action: `{type:'dismiss-park', jobId, reason}`. `applyDismissParkActionInline` (in `scripts/builder-worker.ts`) gates owner-only (the parked job's underlying spec slug must resolve to a spec whose `Owner:` matches the emitting director), flips `agent_jobs.status='dismissed'` + stamps `needs_attention_class='dismissed_by_director'`, and writes a `dismissed_park` [[../tables/director_activity]] row carrying the reason + the original park's class + the underlying spec slug. The auto-router filters `dismissed_by_director` (belt-and-suspenders — the status filter alone already excludes a dismissed row). The daily watch line counts the day's dismissals: *"dismissed N stale parks"*. **Reversible** — the CEO clicks **Re-open** on the dismissed activity row (`POST /api/developer/director-activity/reopen-park`, owner-gated): the job flips back to `needs_attention`, the dismissal marker clears, and a `reopened_park` row is logged so the button hides on the next render. A wrongly-dismissed park is one tap from re-routing.

## Safety invariants

- **Never rubber-stamps** — a structural leash candidate must *also* pass the investigation verdict; an ambiguous/errored result escalates, never approves.
- **The leash is hard** — destructive/irreversible, goal-touching, new-goal, and multi-choice requests always escalate (to the CEO inbox, with the diagnosis).
- **No infinite loops** — a build that fails ≥ `PLATFORM_DIRECTOR_LOOP_GUARD_MAX` (2) times is never resubmitted; the loop-guard stops and escalates a "deeper issue" diagnosis to the CEO.
- **Execution path unchanged** — the autonomous approve flips the same `queued_resume` the human path does; no action runs by a path that skips the gate.
- **Auditable** — every call (approve or escalate) logs reasoning to [[../tables/approval_decisions]] / [[../tables/director_activity]].
