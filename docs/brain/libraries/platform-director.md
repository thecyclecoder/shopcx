# libraries/platform-director

The **Platform/DevOps Director agent** ŌĆö the **first live director** ([[../specs/platform-director-agent]] Phase 1, [[../goals/devops-director]] M4). Investigate every Approval Request **routed to Platform** and either **auto-approve within the leash** (with the reasoning logged) or **leave it for the CEO**. It **supervises** the existing platform tools ([[../specs/repair-agent|repair]], [[../specs/db-health-agent|db-health]], [[../specs/coverage-auto-register-agent|coverage-register]]) ŌĆö it does **not** rebuild them.

**File:** `src/lib/agents/platform-director.ts`

## Why this exists

North star ([[../operational-rules]] ┬¦ supervisable autonomy): **CEO ŌåÆ Director ŌåÆ tool**. Platform's tools already work, but nothing supervised them *as a director* ŌĆö every platform approval the CEO rubber-stamps landed in the CEO inbox. This module is the supervisor: it reuses the [[../specs/approval-routing-engine|approval-routing]] plumbing rather than adding a parallel approval path, and records every call to the supervisable-autonomy ledger so the CEO can audit **what** it decided and **why** ŌĆö in history, never in the queue.

**Dormant until activation.** Until Platform's [[../tables/function_autonomy]] flag is flipped `live + autonomous` (owner-confirmed, [[../specs/platform-director-agent]] Phase 4), [[approval-router]] `resolveApprover` routes nothing here, so the enqueuer is a no-op ŌĆö the machinery is built but inert.

## The leash (what it MAY auto-approve)

The `goals/devops-director` ┬¦ leash + the standing autonomy rule ([[../operational-rules]]). A **structural** gate (which action class) **and** a **soundness** gate (the read-only investigation ŌĆö *never rubber-stamps*):

| `LeashCategory` | Pending-action `type` | What |
|---|---|---|
| `error_fix` | `repair_build` | a repair-agent fix for a real bug |
| `db_health` | `db_health_build` | a DB index / health fix (no destructive DDL) |
| `additive_migration` | `apply_migration` | an **additive, reversible** migration (no DROP/DELETE/data loss) |
| `additive_backfill` | `run_prod_script` **only when bundled with an `apply_migration`** | an **idempotent, re-runnable** backfill that accompanies an additive migration in the SAME request (a standalone prod script is never in-leash) |
| `monitoring_fix` | *(reserved)* | a platform-monitoring registry fix |

**Multi-action (worker-grading P8).** The gate is `directorLeashCandidates` (not the old single-action `directorLeashCandidate`): it auto-approves a request with **one** in-leash action OR a **bundle** where **every** action is in-leash ŌĆö most often an **additive migration + its dependent idempotent backfill** (the `a2edeca0` / `experiment-session-stamped-attribution` case). A bundle is **all-or-nothing**: one out-of-leash action escalates the whole request, and `applyDirectorApproval` approves all actions atomically (the target flips `queued_resume` only when none stay pending). The soundness gate is unchanged ŌĆö the investigation must confirm EVERY action (and the bundle as a whole) is sound + reversible.

**Always escalates** (never auto-approves): destructive/irreversible actions, modifying/abandoning a goal, starting a new goal, a non-binary **multi-CHOICE** decision (coverage-register register-vs-exempt, storefront campaign), a standalone prod script, or anything it cannot confirm sound. **CEO escalation routing** is live (Phase 3): an escalation now **re-routes** the Approval Request to the CEO inbox carrying Ada's diagnosis (`escalateApprovalRequestToCeo`), instead of leaving it untouched. See *Loop-guard + CEO escalation* below.

## Escorting goals (Phase 2 ŌĆö milestone progression within the leash)

`escortApprovedGoals` drives each **approved** goal the director owns forward, **leaning on the existing machinery** ([[../specs/spec-blockers|blocked_by auto-queue]] `autoQueueUnblockedBy` + the builder chain + auto-ship + fold) rather than reimplementing it. The reactive auto-queue fires on a *blocker's merge*; the escort is the **proactive** complement ŌĆö a throttled box sweep (~5 min) that kicks off the unblocked specs the reactive path never caught and **logs the advance**.

- **Escortable goal** = `GoalCard.owner === platform` (the `Owner: [[../functions/x]]` parse in [[brain-roadmap]] `parseGoal`) that is **greenlit AND in progress** ŌĆö `GoalCard.status === "greenlit"` with `0 < pct < 100`. The CEO's greenlight is now an **explicit goal state** ([[../specs/director-proposed-goals]] Phase 1), no longer inferred from `pct > 0`. A **`proposed`** goal is skipped ŌĆö it awaits the CEO via its OWN Approval Request (the `proposed-goal` job), so the escort never re-escalates it. A **`greenlit` 0%** goal is *ready for decomposition* (Pia, Phase 2) and is NOT auto-started here. This **replaced** the old "every 0% owned goal escalates as a `new_goal` greenlight request" ŌĆö a proposed goal is now a self-surfacing artifact and a greenlit-0% goal is genuinely approved-and-awaiting-Pia.
- **What it queues:** every spec linked across the goal's milestones that is **unshipped + unblocked** (all `blockedBy` cleared) + has **no build job yet**, as a `kind='build'` `agent_jobs` row (`created_by=null` ŌĆö the same agent-enqueue shape as `autoQueueUnblockedBy` / `queueNextChainedPhase`, so the chain/auto-ship/fold pick it up unchanged). Skips blocked specs (the reactive auto-queue handles those on their blocker's merge) and `**Auto-build:** off` opt-outs.
- **Audited + idempotent:** one `escorted_goal` [[../tables/director_activity]] row per goal it advances (`autonomous=true`, naming the goal + queued slugs); a spec that already has a build job is **confirmed in-flight**, never re-queued. **Dormant** (a no-op) until Platform is live+autonomous (Phase 4), exactly like the approval enqueuer.

## Loop-guard + CEO escalation (Phase 3 ŌĆö the high-stakes calls always route UP)

The leash is hard, so a high-stakes call **always escalates to the CEO** ŌĆö it is never rubber-stamped, and a failing build is never resubmitted forever. Every escalation reuses the **existing M2 inbox** (a routed Approval Request notification ŌĆö the inbox API shows an item to a role iff `metadata.routed_to_function === role`), never a parallel inbox.

- **Loop-guard.** The escort tracks each escorted spec's build state (`specBuildState` ŌĆö recent `agent_jobs` `kind='build'` rows for the slug: is one **active/landed** ŌåÆ leave it; how many **`failed`/`needs_attention`** ŌåÆ the failure count + latest error). On a sweep, per unblocked spec: an in-flight build ŌćÆ confirm, don't re-queue; **Ōēź `PLATFORM_DIRECTOR_LOOP_GUARD_MAX` (2) failures** with nothing in-flight ŌćÆ **stop + escalate** ("likely a deeper issue"), never re-queue; a single prior failure ŌćÆ a bounded **retry** (`instructions` note `re-attempt #2`); no builds yet ŌćÆ the Phase-2 gap-fill queue.
- **Re-route a declined request ŌåÆ CEO** (`escalateApprovalRequestToCeo`). The box lane's two escalate branches (out-of-leash / multi-choice, and the investigation `escalate`/ambiguous verdict ŌĆö e.g. a **destructive** migration the runner won't auto-approve) now flip the target's routed Approval Request to `routed_to_function='ceo'` and prepend Ada's diagnosis to the body, so the **CEO** inbox shows it and Platform's no longer does. The target job stays `needs_approval` (never auto-approved); the CEO approves/declines via the normal `/api/roadmap/approve` gate. Creates a CEO-routed request if the reconciler hadn't emitted one yet (idempotent on `agent_job_id`).
- **Standalone diagnosis ŌåÆ CEO** (`escalateDiagnosisToCeo`). A high-stakes call with **no approvable target** ŌĆö a loop-guard "deeper issue," or a **zero-progress owned goal** (only the CEO greenlights a NEW goal). Emits a CEO-routed Approval Request (no inline approve ŌĆö it deep-links the CEO to the spec/goal) **and** an `escalated` [[../tables/director_activity]] row. Carries **no `agent_job_id`** so [[approval-inbox]] `reconcileApprovalInbox` ŌĆö which dismisses any request whose job left `needs_approval` ŌĆö never reaps this standalone escalation.
  - **Reliable surface (Phase 1).** Order is **notification-first, activity-second, both checked**: it inserts the [[../tables/dashboard_notifications]] row, **checks that insert's `error`**, and only on success records the `escalated` ledger row + returns `{ emitted:true }`. If the notification insert fails it returns `{ emitted:false, error }` (the caller logs a hard `console.error`) and writes **no** phantom `escalated` activity row ŌĆö so the audit ledger never claims an escalation that the inbox never showed. (Before this, the insert was unguarded: a failed insert still logged `escalated` and returned `emitted:true` ŌĆö logged-but-invisible.)
  - **Dedupe keys on a notification that ACTUALLY EXISTS.** The one-per-`dedupe_key` dedup now queries [[../tables/dashboard_notifications]] (`type='agent_approval_request'`, `metadata->>dedupe_key`), **not** the `director_activity` ledger. A logged-but-unsurfaced escalation (activity row, no notification) therefore does **not** suppress the retry ŌĆö the next pass re-emits the missing notification. Once the notification exists (even dismissed/read) it pings once. Keys: `loopguard:{slug}` / `groom-unsure:{slug}` / `groom-loopguard:{slug}` / `init-unsure:{slug}` / `initguard:{slug}`. (The `newgoal:{goalSlug}` key is retired ŌĆö director-proposed-goals replaced the zero-progress escalation with the explicit `proposed`ŌåÆ`greenlit` goal state.)

## Watch the platform + report to the board (Phase 4 ŌĆö the human-legible top layer)

The director's top layer is **reporting up in human terms** ŌĆö and it **supervises, doesn't rebuild**: it reads the *existing* [[../dashboard/control-tower]] snapshot rather than standing up new monitoring, and reuses the *existing* board/recap/dev-ask plumbing rather than a parallel path.

- **Daily board watch** (`postPlatformWatchUpdate`). Reads [[control-tower]] `buildControlTowerSnapshot` ŌåÆ the **platform department rollup** (worst-of color, healthy/total, open alerts) + its red loops, plus today's [[../tables/director_activity]] (squashed = `approved_approval`, escorting = `escorted_goal`, escalated = `escalated`), and posts ONE conversational `update` as ­¤øĀ’ĖÅ Ada to the [[../tables/director_messages]] board (`composePlatformWatchBody` ŌĆö *"­¤øĀ’ĖÅ Platform watch ŌĆö all 9 platform loops green. Today: squashed 2 fixes ┬Ę escorted 1 goal."*). **Idempotent** per (workspace, UTC day) on `metadata.watch_date`; skips a fully-quiet all-green day (no empty-board spam); **dormant** until live+autonomous like the escort.
- **"Answers why?"** ŌĆö no new code: the [[../specs/directors-board-gamified]] Phase-2 boardŌåÆdev-ask wiring ([[director-board]] `routeBoardReply`) already defaults the answer brain to Platform, so a "why?" reply under Ada's post routes to a `dev-ask` box turn that posts the answer back in-thread.
- **EOD-recap slice** ŌĆö no new code: Platform is a director, so [[director-recap]] `generateDirectorRecap` already rolls its day's activity into the standup (`goalsAdvanced` ŌåÉ `escorted_goal`; `bugsFixed`/`approvalsHandled` ŌåÉ the director's [[../tables/approval_decisions]]) + the CEO roll-up.

**Activation (owner-confirmed).** `scripts/apply-platform-live-autonomous.ts` upserts the [[../tables/function_autonomy]] `platform` row to `live + autonomous` ŌĆö the Phase-4 switch that takes every dormant surface above live (the approval router then routes platform-owned approvals to the director instead of the CEO). Idempotent + reversible (toggle off from the [[../dashboard/agents|Agents hub]]).

## Board grooming (Phase 5 ŌĆö the director MOVES the project board, [[../specs/board-grooming]])

The director doesn't just build queued specs ŌĆö it **grooms the board** so nothing rots half-built. On its standing cadence it assesses every **partially-shipped** spec (Ōēź1 phase Ō£ģ, remaining ŌÅ│, **no active build**) and decides what to do with the leftover phases ŌĆö *moving the card* accordingly. This formalizes the need-now-vs-future call the operator made by hand (and replaces the escort's blind "re-queue every phase").

- **The decision (the Max `claude -p` investigation, `groomInvestigationPrompt`):** per partially-shipped spec ŌĆö **continue** (the next ŌÅ│ phase is needed now: the spec's current promise / a dependent / a goal needs it ŌåÆ queue its build to completion, the chain carries it) ┬Ę **split** (the leftover phase(s) are future enhancement/polish ŌåÆ author EACH as its own planned card `{slug}-{phase}.md` with a `**Deferred:**` note, then CLOSE OUT the parent ŌĆö remove the split phases so its remaining phases are all-Ō£ģ ŌåÆ the parent **folds**) ┬Ę **escalate** (genuinely unsure / possibly load-bearing ŌåÆ route the diagnosis to the CEO, move nothing ŌĆö north-star: hit a rail ŌåÆ escalate).
- **Net effect:** every partially-shipped spec either **completes** (needed-now phases built) or **cleanly splits** (future phases become their own planned cards) ŌĆö cards flow *out* of "In progress," the future work is captured (not lost) in "Planning." Future work is **never dropped** ŌĆö a split always preserves the phase as a planned card; the director never deletes a phase.
- **Supervisable + audited:** splitting + queueing a next-phase build is low-risk/reversible (within the leash); every decision writes a [[../tables/director_activity]] row (`groomed_continue` / `groomed_split` / `escalated`) with the reasoning. **Dormant** until live+autonomous, like the escort. **Loop-guarded** (a continue whose build already failed Ōēź `PLATFORM_DIRECTOR_LOOP_GUARD_MAX` with nothing in-flight escalates instead of resubmitting). **A malformed split never lands** (`validateGroomSplit` gates the commit ŌĆö a broken board is worse than an un-groomed card; it escalates instead).
- **Idempotent:** a `continue`'s queued build flips the spec in-flight (the candidate filter excludes it next pass); a `split`/`escalate` writes a `groom_key` (`groom:{slug}`) the next pass dedups on via `alreadyGroomed` ŌĆö this is what survives the box's bundled `fs` copy lagging `main` until its next self-update (without it the same card would re-split every pass).

## The box lane

`scripts/builder-worker.ts` `runPlatformDirectorJob` (concurrency-1 `platform-director` lane, claimed via `claim_agent_job`; `runDirectorClaude` is the Max session ŌĆö read-only DB creds kept, API key stripped). It re-loads the target, **re-confirms** it still `needs_approval` and still routes to Platform (fail-safe), runs the leash gate, investigates read-only, then:
- **`auto-approve`** ŌåÆ `applyDirectorApproval` + a `director_activity` `approved_approval` row ([[../tables/director_activity]]).
- **`escalate`** (or any ambiguous/unparseable verdict ŌĆö fail safe) ŌåÆ a `director_activity` `escalated` row **and** `escalateApprovalRequestToCeo` re-routes the request to the CEO inbox with the diagnosis (Phase 3). The target stays `needs_approval` (never auto-approved).

It **never** edits product code, opens a PR, or runs a migration ŌĆö it only decides on the existing gated action.

**Standing pass (Phase 4 + Phase 5 + init).** When the job carries **no `target_job_id`** ŌĆö the [[../inngest/platform-director-cron]] enqueue ŌĆö `runPlatformDirectorJob` instead runs `runPlatformDirectorStandingPass`: `escortApprovedGoals` (Phase 2), **`reconcileSwallowedEscalations`** ([[../specs/director-escalations-must-surface-to-ceo-backfill-swallowed]] ŌĆö the backstop that re-emits any CEO notification swallowed before the Phase-1 fix), **`reconcileErrorBacklog`** ([[../specs/director-zero-backlog-error-autonomy]] Phase 1 ŌĆö the zero-backlog backstop that re-drives every OPEN [[../tables/error_events]] / [[../tables/loop_alerts]] item Rafa's event trigger missed: enqueue a diagnosis where none exists, confirm a covered one, escalate a stuck fix), **`escortFixSpecs`** (worker-grading P4 ŌĆö the 0-phase authored-fix-spec lane), **`initiatePlatformSpecs`** ([[../specs/director-initialize-platform-specs-no-wait]] Phase 2 ŌĆö the unstarted **non-fix, non-goal** platform-spec initiation lane: per-candidate `claude -p` **soundness** investigation ŌåÆ `initiate` queues a build / `escalate` routes the diagnosis to the CEO; loop-guarded; bounded by `PLATFORM_DIRECTOR_INIT_CAP`), `groomBoard` (Phase 5 board-grooming ŌĆö the per-candidate `claude -p` investigation + dispatch: queue a build, `putFileMain` a split + closed-out parent, or escalate), **and** `postPlatformWatchUpdate` (the board watch) on the reliable cron beat. Best-effort (a failure in one part never blocks the others); all no-op unless live+autonomous. `groomBoard` is the only standing surface that **commits to `main`** (the split cards + the folded parent); both `initiatePlatformSpecs` and `groomBoard` run `claude -p` per candidate ŌĆö each bounded by its cap.

## Safety invariants

- **Never rubber-stamps** ŌĆö a structural leash candidate must *also* pass the investigation verdict; an ambiguous/errored result escalates, never approves.
- **The leash is hard** ŌĆö destructive/irreversible, goal-touching, new-goal, and multi-choice requests always escalate (to the CEO inbox, with the diagnosis).
- **No infinite loops** ŌĆö a build that fails Ōēź `PLATFORM_DIRECTOR_LOOP_GUARD_MAX` (2) times is never resubmitted; the loop-guard stops and escalates a "deeper issue" diagnosis to the CEO.
- **Execution path unchanged** ŌĆö the autonomous approve flips the same `queued_resume` the human path does; no action runs by a path that skips the gate.
- **Auditable** ŌĆö every call (approve or escalate) logs reasoning to [[../tables/approval_decisions]] / [[../tables/director_activity]].
