# libraries/platform-director

The **Platform/DevOps Director agent** — the **first live director** ([[../specs/platform-director-agent]] Phase 1, [[../goals/devops-director]] M4). Investigate every Approval Request **routed to Platform** and either **auto-approve within the leash** (with the reasoning logged) or **leave it for the CEO**. It **supervises** the existing platform tools ([[../specs/repair-agent|repair]], [[../specs/db-health-agent|db-health]], [[../specs/coverage-auto-register-agent|coverage-register]]) — it does **not** rebuild them.

**File:** `src/lib/agents/platform-director.ts`

## Why this exists

North star ([[../operational-rules]] § supervisable autonomy): **CEO → Director → tool**. Platform's tools already work, but nothing supervised them *as a director* — every platform approval the CEO rubber-stamps landed in the CEO inbox. This module is the supervisor: it reuses the [[../specs/approval-routing-engine|approval-routing]] plumbing rather than adding a parallel approval path, and records every call to the supervisable-autonomy ledger so the CEO can audit **what** it decided and **why** — in history, never in the queue.

**Activated 2026-06-23 20:35** (by `ceo`). Platform's [[../tables/function_autonomy]] row is `live + autonomous`, so [[approval-router]] `resolveApprover` now routes platform-owned approvals here and **every surface below is live** — the runtime guards (`platformIsAutoApprover` etc.) currently **pass**. The per-surface "no-op unless live+autonomous" notes below describe the **conditional guard** (what each surface does when the flag is *off*): the code still no-ops if the flag is ever toggled back off ([[../dashboard/agents|Agents hub]]) — it is not a statement that Platform is dormant today.

## The leash (what it MAY auto-approve)

The `goals/devops-director` § leash + the standing autonomy rule ([[../operational-rules]]). A **structural** gate (which action class) **and** a **soundness** gate (the read-only investigation — *never rubber-stamps*):

| `LeashCategory` | Pending-action `type` | What |
|---|---|---|
| `error_fix` | `repair_build` | a repair-agent fix for a real bug |
| `db_health` | `db_health_build` | a DB index / health fix (no destructive DDL) |
| `additive_migration` | `apply_migration` | an **additive, reversible** migration (no DROP/DELETE/data loss) |
| `additive_backfill` | `run_prod_script` **only when bundled with an `apply_migration`** | an **idempotent, re-runnable** backfill that accompanies an additive migration in the SAME request (a standalone prod script is never in-leash) |
| `monitoring_fix` | *(reserved)* | a platform-monitoring registry fix |

**Multi-action (worker-grading P8).** The gate is `directorLeashCandidates` (not the old single-action `directorLeashCandidate`): it auto-approves a request with **one** in-leash action OR a **bundle** where **every** action is in-leash — most often an **additive migration + its dependent idempotent backfill** (the `a2edeca0` / `experiment-session-stamped-attribution` case). A bundle is **all-or-nothing**: one out-of-leash action escalates the whole request, and `applyDirectorApproval` approves all actions atomically (the target flips `queued_resume` only when none stay pending). The soundness gate is unchanged — the investigation must confirm EVERY action (and the bundle as a whole) is sound + reversible.

**Always escalates** (never auto-approves): destructive/irreversible actions, modifying/abandoning a goal, starting a new goal, a non-binary **multi-CHOICE** decision (coverage-register register-vs-exempt, storefront campaign), a standalone prod script, or anything it cannot confirm sound. **CEO escalation routing** is live (Phase 3): an escalation now **re-routes** the Approval Request to the CEO inbox carrying Ada's diagnosis (`escalateApprovalRequestToCeo`), instead of leaving it untouched. See *Loop-guard + CEO escalation* below.

## Escorting goals (Phase 2 — milestone progression within the leash)

`escortApprovedGoals` drives each **approved** goal the director owns forward, **leaning on the existing machinery** ([[../specs/spec-blockers|blocked_by auto-queue]] `autoQueueUnblockedBy` + the builder chain + auto-ship + fold) rather than reimplementing it. The reactive auto-queue fires on a *blocker's merge*; the escort is the **proactive** complement — a throttled box sweep (~5 min) that kicks off the unblocked specs the reactive path never caught and **logs the advance**.

- **Escortable goal** = `GoalCard.owner === platform` (the `Owner: [[../functions/x]]` parse in [[brain-roadmap]] `parseGoal`) that is **greenlit AND in progress** — `GoalCard.status === "greenlit"` with `0 < pct < 100`. The CEO's greenlight is now an **explicit goal state** ([[../specs/director-proposed-goals]] Phase 1), no longer inferred from `pct > 0`. A **`proposed`** goal is skipped — it awaits the CEO via its OWN Approval Request (the `proposed-goal` job), so the escort never re-escalates it. A **`greenlit` 0%** goal is *ready for decomposition* (Pia, Phase 2) and is NOT auto-started here. This **replaced** the old "every 0% owned goal escalates as a `new_goal` greenlight request" — a proposed goal is now a self-surfacing artifact and a greenlit-0% goal is genuinely approved-and-awaiting-Pia.
- **What it queues:** every spec linked across the goal's milestones that is **unshipped + unblocked** (all `blockedBy` cleared) + has **no build job yet**, as a `kind='build'` `agent_jobs` row (`created_by=null` — the same agent-enqueue shape as `autoQueueUnblockedBy` / `queueNextChainedPhase`, so the chain/auto-ship/fold pick it up unchanged). Skips blocked specs (the reactive auto-queue handles those on their blocker's merge) and `**Auto-build:** off` opt-outs.
- **Audited + idempotent:** one `escorted_goal` [[../tables/director_activity]] row per goal it advances (`autonomous=true`, naming the goal + queued slugs); a spec that already has a build job is **confirmed in-flight**, never re-queued. **Dormant** (a no-op) until Platform is live+autonomous (Phase 4), exactly like the approval enqueuer.

## Loop-guard + CEO escalation (Phase 3 — the high-stakes calls always route UP)

The leash is hard, so a high-stakes call **always escalates to the CEO** — it is never rubber-stamped, and a failing build is never resubmitted forever. Every escalation reuses the **existing M2 inbox** (a routed Approval Request notification — the inbox API shows an item to a role iff `metadata.routed_to_function === role`), never a parallel inbox.

- **Loop-guard.** The escort tracks each escorted spec's build state (`specBuildState` — recent `agent_jobs` `kind='build'` rows for the slug: is one **active/landed** → leave it; how many **`failed`/`needs_attention`** → the failure count + latest error). On a sweep, per unblocked spec: an in-flight build ⇒ confirm, don't re-queue; **≥ `PLATFORM_DIRECTOR_LOOP_GUARD_MAX` (2) failures** with nothing in-flight ⇒ **stop + escalate** ("likely a deeper issue"), never re-queue; a single prior failure ⇒ a bounded **retry** (`instructions` note `re-attempt #2`); no builds yet ⇒ the Phase-2 gap-fill queue.
- **BOUNCE an unsound repair fix → the Repair agent (NOT the CEO)** (director-supervises-repair). When investigating a `repair` target, if the bug is real but the AUTHORED FIX is unsound (broken mechanism / mis-scoped / the code contradicts its premise), the verdict is **`bounce`**: `bounceRepairToAgent` ([[repair-agent]]) resolves the current repair job and re-enqueues a fresh repair for the same signature carrying Ada's `directorFeedback` (the re-authoring pass reads it + fixes its work). It never reaches the CEO — `escalate` is reserved for a genuine out-of-leash/irreversible call, not a fix-quality problem. Logs `repair_bounced` [[../tables/director_activity]] + a 🔁 board post. The north-star supervisory loop (Director → tool): Ada doesn't do Repair's job, she hands the work back with feedback.
- **Re-route a declined request → CEO** (`escalateApprovalRequestToCeo`). The box lane's two escalate branches (out-of-leash / multi-choice, and the investigation `escalate`/ambiguous verdict — e.g. a **destructive** migration the runner won't auto-approve) now flip the target's routed Approval Request to `routed_to_function='ceo'` and prepend Ada's diagnosis to the body, so the **CEO** inbox shows it and Platform's no longer does. The target job stays `needs_approval` (never auto-approved); the CEO approves/declines via the normal `/api/roadmap/approve` gate. Creates a CEO-routed request if the reconciler hadn't emitted one yet (idempotent on `agent_job_id`).
- **Standalone diagnosis → CEO** (`escalateDiagnosisToCeo`). A high-stakes call with **no approvable target** — a loop-guard "deeper issue," or a **zero-progress owned goal** (only the CEO greenlights a NEW goal). Emits a CEO-routed Approval Request (no inline approve — it deep-links the CEO to the spec/goal) **and** an `escalated` [[../tables/director_activity]] row. Carries **no `agent_job_id`** so [[approval-inbox]] `reconcileApprovalInbox` — which dismisses any request whose job left `needs_approval` — never reaps this standalone escalation.
  - **Reliable surface (Phase 1).** Order is **notification-first, activity-second, both checked**: it inserts the [[../tables/dashboard_notifications]] row, **checks that insert's `error`**, and only on success records the `escalated` ledger row + returns `{ emitted:true }`. If the notification insert fails it returns `{ emitted:false, error }` (the caller logs a hard `console.error`) and writes **no** phantom `escalated` activity row — so the audit ledger never claims an escalation that the inbox never showed. (Before this, the insert was unguarded: a failed insert still logged `escalated` and returned `emitted:true` — logged-but-invisible.)
  - **Dedupe keys on a notification that ACTUALLY EXISTS.** The one-per-`dedupe_key` dedup now queries [[../tables/dashboard_notifications]] (`type='agent_approval_request'`, `metadata->>dedupe_key`), **not** the `director_activity` ledger. A logged-but-unsurfaced escalation (activity row, no notification) therefore does **not** suppress the retry — the next pass re-emits the missing notification. Once the notification exists (even dismissed/read) it pings once. Keys: `loopguard:{slug}` / `groom-unsure:{slug}` / `groom-loopguard:{slug}` / `init-unsure:{slug}` / `initguard:{slug}`. (The `newgoal:{goalSlug}` key is retired — director-proposed-goals replaced the zero-progress escalation with the explicit `proposed`→`greenlit` goal state.)

## Watch the platform + report to the board (Phase 4 — the human-legible top layer)

The director's top layer is **reporting up in human terms** — and it **supervises, doesn't rebuild**: it reads the *existing* [[../dashboard/control-tower]] snapshot rather than standing up new monitoring, and reuses the *existing* board/recap/dev-ask plumbing rather than a parallel path.

- **Daily board watch** (`postPlatformWatchUpdate`). Reads [[control-tower]] `buildControlTowerSnapshot` → the **platform department rollup** (worst-of color, healthy/total, open alerts) + its red loops, plus today's [[../tables/director_activity]] (squashed = `approved_approval`, escorting = `escorted_goal`, escalated = `escalated`; **+ the repair-review tallies** — `dismissed_repair` / `kept_repair` / repair-suspect `escalated`), and posts ONE conversational `update` as 🛠️ Ada to the [[../tables/director_messages]] board (`composePlatformWatchBody` — *"🛠️ Platform watch — all 9 platform loops green. Today: squashed 2 fixes · escorted 1 goal. Reviewed 3 of Rafa's calls — dismissed 2, escalated 1 back to you."* — the trailing repair-review sentence appears only on a day she reviewed ≥1 of Rafa's calls). **Idempotent** per (workspace, UTC day) on `metadata.watch_date`; skips a fully-quiet all-green day (no empty-board spam); **dormant** until live+autonomous like the escort.
- **"Answers why?"** — no new code: the [[../specs/directors-board-gamified]] Phase-2 board→dev-ask wiring ([[director-board]] `routeBoardReply`) already defaults the answer brain to Platform, so a "why?" reply under Ada's post routes to a `dev-ask` box turn that posts the answer back in-thread.
- **EOD-recap slice** — no new code: Platform is a director, so [[director-recap]] `generateDirectorRecap` already rolls its day's activity into the standup (`goalsAdvanced` ← `escorted_goal`; `bugsFixed`/`approvalsHandled` ← the director's [[../tables/approval_decisions]]) + the CEO roll-up.

Idempotent + reversible (toggle off from the [[../dashboard/agents|Agents hub]]), which would re-arm the conditional no-op guards.

## Board grooming (Phase 5 — the director MOVES the project board, [[../specs/board-grooming]])

The director doesn't just build queued specs — it **grooms the board** so nothing rots half-built. On its standing cadence it assesses every **partially-shipped** spec (≥1 phase ✅, remaining ⏳, **no active build**) and decides what to do with the leftover phases — *moving the card* accordingly. This formalizes the need-now-vs-future call the operator made by hand (and replaces the escort's blind "re-queue every phase").

- **The decision (the Max `claude -p` investigation, `groomInvestigationPrompt`):** per partially-shipped spec — **continue** (the next ⏳ phase is needed now: the spec's current promise / a dependent / a goal needs it → queue its build to completion, the chain carries it) · **split** (the leftover phase(s) are future enhancement/polish → author EACH as its own planned card `{slug}-{phase}.md` with a `**Deferred:**` note, then CLOSE OUT the parent — remove the split phases so its remaining phases are all-✅ → the parent **folds**) · **escalate** (genuinely unsure / possibly load-bearing → route the diagnosis to the CEO, move nothing — north-star: hit a rail → escalate).
- **Net effect:** every partially-shipped spec either **completes** (needed-now phases built) or **cleanly splits** (future phases become their own planned cards) — cards flow *out* of "In progress," the future work is captured (not lost) in "Planning." Future work is **never dropped** — a split always preserves the phase as a planned card; the director never deletes a phase.
- **Supervisable + audited:** splitting + queueing a next-phase build is low-risk/reversible (within the leash); every decision writes a [[../tables/director_activity]] row (`groomed_continue` / `groomed_split` / `escalated`) with the reasoning. **Dormant** until live+autonomous, like the escort. **Loop-guarded** (a continue whose build already failed ≥ `PLATFORM_DIRECTOR_LOOP_GUARD_MAX` with nothing in-flight escalates instead of resubmitting). **A malformed split never lands** (`validateGroomSplit` gates the commit — a broken board is worse than an un-groomed card; it escalates instead).
- **Idempotent:** a `continue`'s queued build flips the spec in-flight (the candidate filter excludes it next pass); a `split`/`escalate` writes a `groom_key` (`groom:{slug}`) the next pass dedups on via `alreadyGroomed` — this is what survives the box's bundled `fs` copy lagging `main` until its next self-update (without it the same card would re-split every pass).

## Supervise + dismiss Rafa's no-fix items ([[../specs/director-supervised-repair-dismissal]] Phase 1 + 2)

The CEO used to manually **Dismiss** every Control Tower warning where the [[repair-agent|Repair Agent]] (Rafa) declined to propose a fix — a `needs-human` verdict parked as a `repair` [[../tables/agent_jobs]] in `needs_attention`, surfaced **Dismiss-only** by `getOpenRepairs`. This lane is the director **supervising** that no-fix call. It is **not** a janitor: Rafa optimizes the bounded proxy *'clear the error'*, whose degenerate state is clearing a warning by declaring a **real bug** benign ([[../operational-rules]] § North star). So Ada **adversarially re-checks** his verdict and dismisses ONLY what she can **independently confirm** is benign.

- **The decision (the Max `claude -p` investigation, `repairDismissalInvestigationPrompt`):** per `needs-human` item — Ada loads the originating [[../tables/error_events]] / [[../tables/loop_alerts]] sample + Rafa's logged no-fix reasoning, **independently re-derives the root cause**, and emits one JSON verdict. The stance is **hard — DEFAULT to not-dismissing**: **dismiss** ONLY if she can independently confirm the error is genuinely transient / foreign-app-noise / benign and NOT a masked real bug · **escalate** if she suspects a real bug **in OUR code** Rafa mislabeled (her contrary diagnosis routes to the CEO via `escalateDiagnosisToCeo`) · **external** if the verified root cause is **OUTSIDE our system** ([[../specs/director-zero-backlog-error-autonomy]] Phase 2, below) · **keep** a genuine needs-human call (leave it on the Control Tower untouched for the human).
- **The external-blocker escalation class (Phase 2) — the ONLY routine error→CEO touch.** When Ada's verified diagnosis is an **external dependency break** — a third-party API contract change, a vendor outage beyond our retry/breaker, or a credential/permission change on their side — she does **NOT** author a code fix (we can't code-fix someone else's system). She `escalateDiagnosisToCeo` with the diagnosis **+ 2–3 concrete ALTERNATIVE options** (wait/retry the vendor, swap provider, degrade that path gracefully — the box lane formats `parsed.alternatives` into a numbered "Your options:" block), `escalationKind='external_blocker'`, deep-linked to the Control Tower. **Deduped per signature** on `externalBlockerKey(signature)` = `external:{signature}` (distinct from the suspected-real-bug `dismiss:{signature}` key) so it pings the CEO **once**. Everything internally-fixable — a real bug in our code, a supervised-dismiss, an authored+built fix — is handled **without** the CEO. If the notification is already surfaced (or fails to insert), the box lane records a `kept_repair` review row so the same job isn't re-investigated each pass.
- **Dismiss reuses the EXISTING owner Dismiss plumbing** — `applyDirectorDismissal` mirrors `POST /api/developer/control-tower/repair` `dismiss` (decline the `repair_build` action, complete the job, resolve the `error_events` row) **minus the owner auth gate** — no new dismiss machinery, **no migration**. It then writes a `dismissed_repair` [[../tables/director_activity]] row carrying **Ada's OWN** independent reasoning (not a copy of Rafa's).
- **Leash:** dismissing a confirmed-benign monitoring warning is the **`monitoring_fix`** class — low-risk + reversible (a dismissed `needs_attention` item UN-blocks re-enqueue, so a wrongly-dismissed real problem re-fires and Rafa re-triages it). **Unsure ⇒ escalate, never dismiss.** NEVER dismisses a `real-bug` / fix-proposed item: `findRepairDismissalCandidates` only takes `state === "needs-human"` items, and `applyDirectorDismissal` **re-asserts** the job is still `needs_attention` before clearing it.
- **Idempotent + loop-guarded:** every review writes a ledger row carrying `metadata.repair_job_id` + `dismiss_key` (`dismiss:{signature}`); `alreadyReviewedDismissal` dedups on the **job id** so each item is reviewed once — a `keep`/`escalate`/`external` leaves the job open but is not re-investigated, while a **re-fire (a NEW repair job, same signature)** is a fresh review. An `external` verdict additionally dedups its CEO notification on `external:{signature}` (so even across re-fires the CEO pings once per live external blocker). **Dormant** until live+autonomous, like every other lane. Bounded by `PLATFORM_DIRECTOR_DISMISS_CAP` (6) per pass.
- **Phase 2 — surface the dismissal + a CEO re-open override.** A dismiss completes the job, so it leaves `getOpenRepairs` and would silently vanish — Phase 2 keeps it **visible**. `getDirectorDismissedRepairs` ([[repair-agent]]) reads the recent `dismissed_repair` ledger (title now carried in its metadata) minus any re-opened, and the [[../dashboard/control-tower]] **Repair feed** renders each as `🛠️ Dismissed by Ada — <reasoning>` with a one-tap **Re-open** (`POST /api/developer/control-tower/repair {action:'reopen'}`, owner-gated — re-opens the `error_events` row, re-enqueues Rafa, logs `reopened_repair`). The **daily board watch** appends a rollup sentence — *"Reviewed N of Rafa's calls — dismissed K, escalated J back to you."* — on any day she reviewed ≥1 (counting `dismissed_repair` + `kept_repair` + the repair-suspect `escalated` rows).

## Exports

> **Deferred specs are excluded from every auto-build lane** ([[../specs/director-drives-all-specs-and-deferred-status]] Phase 1). `escortApprovedGoals`, `escortFixSpecs`, `findGroomCandidates`, and `findInitCandidates` each skip a spec whose `SpecStatus` is `deferred` (a leading `**Deferred:**` / `**Status:** deferred` marker — [[brain-roadmap]] `deriveSpecStatus`). Only the CEO un-defers it (removes the marker → Planned) to make it startable; this lands first so widening the drive (Phase 2) can never accidentally start parked work.

- **`enqueuePlatformDirectorJobs(admin)`** → `{ enqueued, slugs }` — the poll-loop background sweep. Finds every open `needs_approval` [[../tables/agent_jobs]] routed to Platform and queues one `kind='platform-director'` job per target (instructions `{ target_job_id, target_kind }`). **Idempotent** (one director job per target, ever — the dedup that stops an infinite re-enqueue of a deferred target). No-op unless `platformIsAutoApprover`.
- **`directorLeashCandidates(job)`** → `{ actions: LeashAction[], verdict: 'none'｜'single'｜'multi' }` (worker-grading P8) — the structural gate. Returns every pending action with its leash class; `verdict='none'` (escalate) when the request is empty OR **any** pending action is out of leash (multi-choice / non-leash / destructive — all-or-nothing); `single`/`multi` carry the in-leash action(s). `directorLeashCandidate(job)` → `LeashAction | null` is kept as the back-compat single-action helper.
- **`buildDirectorBrief(job, candidates)`** / **`directorInvestigationPrompt(brief)`** — the read-only Max `claude -p` investigation prompt over the action(s): `DirectorBrief` carries `actions[]` + `categories[]` + `multi`. For a bundle the prompt is **all-or-nothing** (every action sound + the bundle reversible) → one JSON verdict `auto-approve｜escalate`.
- **`applyDirectorApproval(admin, target, actionIds, reasoning)`** → `{ ok, error? }` — the **autonomous approve** path; `actionIds` is one id or an array (the bundle). Mirrors [[roadmap-actions]] `approveRoadmapAction` **without** the owner gate: marks **every** listed action `approved`, flips the job to `queued_resume` when none stay pending (**execution path unchanged** — the worker runs each in order on resume), then writes the [[../tables/approval_decisions]] row via [[approval-decisions]] `recordApprovalDecision` (`decided_by='director'`, `autonomous=true`; `pending_action_id` null for a bundle).
- **`platformIsAutoApprover(autonomy)`** / **`routesToPlatform(kind, chart, autonomy)`** — routing predicates over [[approval-router]].
- **`directorLiveStateFact(admin, directorFunction?)`** → `Promise<string>` ([[../specs/brain-platform-live-autonomous-status]] Phase 2 — the recurrence guard) — a prompt block stating the director's **authoritative live-state**, read straight from [[../tables/function_autonomy]] (`live`, `autonomous`, dated `updated_at`/`updated_by`) — the SAME DB row the lanes' runtime guards gate on, **not** brain prose. Every read-only `claude -p` investigation (approval / groom / init / repair-dismissal) is wrapped with this (box-lane `directorDecisionPrompt`), so a decision is premised on the LIVE flag and can never again be premised on a stale 'not yet live / dormant / inert' line some brain page or spec still narrates. Best-effort + fail-safe: a missing row / read error renders 'UNKNOWN — treat as NOT live+autonomous'.
- **`specDriver(owner, chart, autonomy)`** → `string` / **`platformDrivesSpec(owner, chart, autonomy)`** → `boolean` ([[../specs/director-drives-all-specs-and-deferred-status]] Phase 2 — the **owner-agnostic keystone routing**, "first live boss else up"). `specDriver` reuses [[approval-router]] `resolveApprover` to walk UP from a spec's owning function to the first **live+autonomous** ancestor (the owner itself if its director is live), falling back to **platform** on a CEO-fallthrough (the keystone covers a department whose director isn't live yet); an owner-less spec defaults to platform. `platformDrivesSpec` is the `=== platform` predicate the **initiation lane** (`findInitCandidates`) and **fix-escort** (`escortFixSpecs`) now gate on instead of `owner === platform`. Today only platform is live, so it drives every non-deferred spec; as a Growth/CS/CMO director flips live+autonomous, `resolveApprover` returns that owner and its specs **auto-rebalance off platform** — no re-spec.
- **`escortApprovedGoals(admin)`** → `{ goals, queued, escalated }` (Phase 2 + Phase 3; goal-state gate added by [[../specs/director-proposed-goals]] Phase 1) — the proactive goal-escort sweep (see *Escorting goals* above). No-op until live+autonomous; escorts only **`greenlit` in-progress** goals the director owns (queues their unblocked specs + logs `escorted_goal`, **applies the loop-guard** — escalates a repeatedly-failing build). A `proposed` goal (awaits its own CEO Approval Request) and a `greenlit` 0% goal (awaits Pia decomposition) are **skipped**, never escalated — the old `new_goal` zero-progress escalation is **retired**. Driven by a throttled `scripts/builder-worker.ts` poll-loop sweep alongside `enqueuePlatformDirectorJobs`.
- **`escortFixSpecs(admin)`** → `{ fixQueued, escalated }` ([[../specs/worker-grading-and-director-management]] Phase 4; absorbed the removed `director-escort-inflight-specs` gap) — escorts the gap the goal-walk + board-grooming both miss: **unstarted authored fix specs** (a spec carrying a **Repair-signature**, authored by the box Repair/Regression agent for a real bug, with **no shipped phase** — `counts.shipped === 0`). **Owner-agnostic** ([[../specs/director-drives-all-specs-and-deferred-status]] Phase 2): the gate is the Repair-signature **+ `platformDrivesSpec(owner, chart, autonomy)`** (no longer `owner === platform`) — a repair-signed fix builds **straight through** (the already-greenlit `error_fix` mandate) regardless of owner, as long as this director drives it (its owning department-director isn't live yet); once that director flips live+autonomous, its fix specs route to IT, not here. The candidate test gates on `counts.shipped === 0` (not `phases.length === 0`) so a Repair-authored fix is escorted whether it has 0 ⏳ phases or the `## Phase 1 — close it ⏳` section the Repair agent now writes; the build chain then carries its phases to completion ([[../specs/director-initialize-platform-specs-no-wait]] Phase 1). Grooming needs ≥1 ✅ phase so it can't see an unstarted fix; the goal-walk only sees goal-linked specs. An unstarted **feature** spec (no Repair-signature) is the **init lane's** job (with a soundness check), never auto-built here. No waiting period — initiation has no prior build, so no cooldown applies. Same guards as the goal escort (dormant until live+autonomous, skips blocked/opted-out/in-flight, loop-guarded). Queues the build + writes the [[../tables/spec_card_state]] mirror (P6) + an `escorted_fix` [[../tables/director_activity]] row. (In-flight specs — ≥1 ✅ + ≥1 ⏳ — are already driven by `groomBoard`/`findGroomCandidates`, so they're intentionally NOT re-queued here.)

## The box lane

`scripts/builder-worker.ts` `runPlatformDirectorJob` (concurrency-1 `platform-director` lane, claimed via `claim_agent_job`; `runDirectorClaude` is the Max session — read-only DB creds kept, API key stripped). **Every** read-only investigation prompt (approval / groom / init / repair-dismissal) is assembled through the one `directorDecisionPrompt` seam ([[../specs/brain-platform-live-autonomous-status]] Phase 2): it appends `directorLiveStateFact` (the authoritative [[../tables/function_autonomy]] live-state) **and** the CEO's active coaching ([[director-instructions]] `appendDirectorInstructions`, P7) — so a decision reads the LIVE flag, never stale brain prose. It re-loads the target, **re-confirms** it still `needs_approval` and still routes to Platform (fail-safe), runs the leash gate, investigates read-only, then:
- **`auto-approve`** → `applyDirectorApproval` + a `director_activity` `approved_approval` row ([[../tables/director_activity]]).
- **`escalate`** (or any ambiguous/unparseable verdict — fail safe) → a `director_activity` `escalated` row **and** `escalateApprovalRequestToCeo` re-routes the request to the CEO inbox with the diagnosis (Phase 3). The target stays `needs_approval` (never auto-approved).

It **never** edits product code, opens a PR, or runs a migration — it only decides on the existing gated action.

**Standing pass (Phase 4 + Phase 5 + init).** When the job carries **no `target_job_id`** — the [[../inngest/platform-director-cron]] enqueue — `runPlatformDirectorJob` instead runs `runPlatformDirectorStandingPass`: **`reconcileMilestoneSequence`** ([[../specs/goal-milestone-build-sequencing]] Phase 3 — the FIRST step, before the escort: detect any milestone build that fanned out before its prerequisites via `findMilestoneSequenceViolations`, then per violation `ensureBlockedByLine`→`putFileMain` the authored `**Blocked-by:**` order onto the spec file, `holdPrematureBuild` to cancel the jammed fan-out, and write a `reconciled_sequence` activity row — the reactive auto-queue + escort re-release each held spec in sequence; this is the SECOND standing surface that commits to `main`, alongside `groomBoard`), `escortApprovedGoals` (Phase 2), **`reconcileSwallowedEscalations`** ([[../specs/director-escalations-must-surface-to-ceo-backfill-swallowed]] — the backstop that re-emits any CEO notification swallowed before the Phase-1 fix), **`reconcileErrorBacklog`** ([[../specs/director-zero-backlog-error-autonomy]] Phase 1 — the zero-backlog backstop that re-drives every OPEN [[../tables/error_events]] / [[../tables/loop_alerts]] item Rafa's event trigger missed: enqueue a diagnosis where none exists, confirm a covered one, escalate a stuck fix), **`reconcileRegressionCoverage`** ([[../specs/regression-backlog-reconciliation]] Phase 1 — the regression-side sibling: enqueue a Vera spec-test re-run for the shipped, unarchived specs least-recently verified so a silent regression is caught even when nothing event-triggered a re-test; an `issues` result flows to Remi via the existing `enqueueRegressionJob`; skips specs fresh within the 7d window, capped per pass), **`escortFixSpecs`** (worker-grading P4 — the 0-phase authored-fix-spec lane), **`initiatePlatformSpecs`** ([[../specs/director-initialize-platform-specs-no-wait]] Phase 2 — the unstarted **non-fix, non-goal** platform-spec initiation lane: per-candidate `claude -p` **soundness** investigation → `initiate` queues a build / `escalate` routes the diagnosis to the CEO; loop-guarded; bounded by `PLATFORM_DIRECTOR_INIT_CAP`), **`superviseRepairDismissals`** ([[../specs/director-supervised-repair-dismissal]] Phase 1 — supervise + clear Rafa's no-fix Control Tower items: per `needs-human` repair item a `claude -p` adversarial re-check → `dismiss` clears it via the existing owner Dismiss path / `escalate` routes a contrary diagnosis to the CEO / `keep` leaves it for the human; loop-guarded on `repair_job_id`, bounded by `PLATFORM_DIRECTOR_DISMISS_CAP`), `groomBoard` (Phase 5 board-grooming — the per-candidate `claude -p` investigation + dispatch: queue a build, `putFileMain` a split + closed-out parent, or escalate), **and** `postPlatformWatchUpdate` (the board watch) on the reliable cron beat. Best-effort (a failure in one part never blocks the others); all no-op unless live+autonomous. `groomBoard` is the only standing surface that **commits to `main`** (the split cards + the folded parent); both `initiatePlatformSpecs` and `groomBoard` run `claude -p` per candidate — each bounded by its saturation cap. **Parallel investigations ([[../specs/director-initiation-throughput]] Phase 2):** `initiatePlatformSpecs` runs its per-candidate soundness investigations CONCURRENTLY via a small bounded worker-pool (`runWithConcurrency`, `DIRECTOR_INVEST_CONCURRENCY`=4) instead of the old one-at-a-time loop, so a single pass produces a full batch — and **backs off on Max 529/overloaded** (`isMaxOverloaded`): a throttle mid-batch sets an `overloaded` flag that skips the remaining candidates (escalating nothing — it's a transient API throttle, not a spec problem) and resumes next beat (Phase 4 — the real ceiling is Max rate limits).

## Safety invariants

- **Never rubber-stamps** — a structural leash candidate must *also* pass the investigation verdict; an ambiguous/errored result escalates, never approves.
- **The leash is hard** — destructive/irreversible, goal-touching, new-goal, and multi-choice requests always escalate (to the CEO inbox, with the diagnosis).
- **No infinite loops** — a build that fails ≥ `PLATFORM_DIRECTOR_LOOP_GUARD_MAX` (2) times is never resubmitted; the loop-guard stops and escalates a "deeper issue" diagnosis to the CEO.
- **Execution path unchanged** — the autonomous approve flips the same `queued_resume` the human path does; no action runs by a path that skips the gate.
- **Auditable** — every call (approve or escalate) logs reasoning to [[../tables/approval_decisions]] / [[../tables/director_activity]].
