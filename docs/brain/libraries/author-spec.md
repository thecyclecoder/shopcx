# libraries/author-spec

The DB-only chokepoint every spec-AUTHOR surface (planner, director-coach, triage, regression, spec-chat, db-health, coverage-register, repair, security, migration-fix, storefront-optimizer, developer-message-center, director split/bounce-back) routes through to write the spec BODY to [[../tables/specs]] + [[../tables/spec_phases]]. There is NO `docs/brain/specs/{slug}.md` commit — the per-spec markdown was retired ([[../specs/spec-readers-from-db-retire-parser]] · spec-pm-markdown-purge); the readers read the DB rows.

## Structured is the ONLY sanctioned author path (retire-md-spec-writers-db-is-sole-spec Phase 4)

Every autonomous spec-writer lane authors via `authorSpecRowStructured` / `submitSpec` — the STRUCTURED chokepoint. Each phase carries typed `spec_phase_checks` (`exec_kind` ∈ tsc | grep | ci_status | http_get | db_probe_readonly | unit_test | build) so the deterministic spec-check runner ([[spec-check-runner]]) verifies the fix landed after merge. The retired MARKDOWN chokepoint (`authorSpecRowFromMarkdown` + its `scripts/builder-worker.ts` wrapper `markNewSpecInReview`) is closed to NEW autonomous callers: prose Verification bullets parse to `exec_kind='needs_human'`, which `assertEveryPhaseHasMachineCheck` rejects, parking the fix-spec at the CEO inbox — the exact failing state this whole spec exists to kill.

**CI guard.** `scripts/_check-no-markdown-spec-authoring.ts` (wired into `npm run check:no-markdown-spec-authoring` + `predeploy`) fails CI on any new caller of `markNewSpecInReview` / `authorSpecRowFromMarkdown` outside the pre-Phase-4 allow-list. Each allow-listed entry (db_health · spec-chat · migration-fix · security-agent · storefront-optimizer · director-coach · developer-message-center · groom/bounce-back split · regression-agent · repair-agent signature-append) is Phase-4 debt awaiting its own per-lane conversion spec — a NEW autonomous lane must author structured from day one.

**Regression test.** `src/lib/every-deterministic-body-producer.test.ts` asserts every deterministic spec-body producer (`buildRegisterSpecBody` / `buildExemptSpecBody` in [[coverage-register-agent]] Phase 1 · `buildRepairSpecInput` in [[repair-agent]] Phase 2 · `buildStructuredSpecInputFromMarkdown` in this module Phase 3) yields a `StructuredSpecInput` whose every phase carries ≥1 machine-runnable check that passes `assertEveryPhaseHasMachineCheck`. A NEW deterministic producer added by a follow-up spec MUST be appended there — the test is the single locked-in check across every producer.

**Follow-up per-lane conversions** — the allow-list in `scripts/_check-no-markdown-spec-authoring.ts` names each pre-Phase-4 lane with its `tag` (actor label) so a subsequent per-lane conversion spec (e.g. "retire-md-spec-writers-db-is-sole-spec Phase 5 — spec-chat lane") can (a) route the lane's box output through `authorSpecRowStructured` and (b) delete the entry from `SANCTIONED_MARKDOWN_CALLERS`. When the allow-list is empty and `authorSpecRowFromMarkdown` has no callers, the function itself can be removed and the wrapper `markNewSpecInReview` retired.

**File:** `src/lib/author-spec.ts`

## Why this exists

[[../specs/spec-body-table-and-backfill]] M1 stood up [[../tables/specs]] + [[../tables/spec_phases]] and the [[specs-table]] writer. This module is the single authoring chokepoint that EVERY author surface routes through to write the spec row + its phases — so the Verification-enforcement gate (no untestable spec) and the `upsertSpec` write discipline live in ONE place. Post-cutover the DB row is the sole source of truth; nothing commits a `.md`.

## Exports

- **`authorSpecRowFromMarkdown(workspaceId, slug, markdown, intendedStatus, opts?)`** → `Promise<boolean>` — author from a MARKDOWN body: parses it via [[brain-roadmap]] `parseSpec` + the local `extractPhaseBodies`, extracts the typed regression headers + repair signature, and calls [[specs-table]] `upsertSpec` to UPSERT [[../tables/specs]] + REPLACE [[../tables/spec_phases]] by `(spec_id, position)`. The Verification gate (`assertEveryPhaseHasVerification`) runs FIRST and THROWS `MissingVerificationError` on an untestable phase; a genuine DB error is best-effort (`false`). `opts.milestoneId` binds `specs.milestone_id` at author time. **pm-structured-intent-and-refs Phase 1** — `extractIntentHeaders` pulls `**Why:** …` / `**What:** …` from the markdown when present; missing headers land as NULL with a soft warn (surfaces migrate incrementally to emit them).
- **`submitSpec`** — the canonical, ergonomic ALIAS for `authorSpecRowStructured` (harden-spec-submission hotfix; same signature, same behavior). "Submit a spec" is the operation's plain name — **prefer `submitSpec` in new code**; `authorSpecRowStructured` stays exported for existing callers.
- **`authorSpecRowStructured(workspaceId, slug, spec, intendedStatus, opts?)`** → `Promise<boolean>` — author from ALREADY-TYPED fields + phases (NO markdown parse). Same Verification gate + same `upsertSpec` write path. This is the DB-driven entry point the **goal planner** uses: it holds the proposed spec as structured data (`StructuredSpecInput` — title/summary/owner/parent/blocked_by + **REQUIRED `why` + `what`** + `StructuredPhaseInput[]` each carrying a REQUIRED non-empty `verification` + `why` + `what` + optional `checks:[{position,description,kind:'auto'|'human'}]`) and never needs a `.md` scratch buffer on disk. `opts.milestoneId` binds the goal→milestone→spec link. Removes the disk-`.md`-round-trip that authored 0 specs (planner-authors-specs-to-db). **pm-structured-intent-and-refs Phase 1–3 gates run before the DB write:** `assertEveryNodeHasIntent` (plain-language why/what non-empty on spec + every phase, with a lint rejecting code fences / `file:line` refs / `**Header:**` lines) and `assertEveryPhaseHasChecks` (≥1 structured `spec_phase_check` row per phase — parsed from `verification` when the caller doesn't pass an explicit `checks` array). Both throw the same fail-loud shape as `assertEveryPhaseHasVerification`. AFTER a successful `upsertSpec`, the writer persists structured brain refs via [[spec-brain-refs-table]] `replaceSpecBrainRefs` (from `opts.brainRefs` or the derived suggester) and per-phase check rows via [[spec-phase-checks-table]] `upsertPhaseChecks`.
- **`auto_build` default = ON (`!== false`).** Both entry points write `auto_build: <input>.autoBuild !== false`, so an autonomously-authored spec **auto-builds by default** — only an EXPLICIT `autoBuild: false` parks it. This HONORS the [[brain-roadmap]] `parseSpec` contract for the markdown path (*"`**Auto-build:**` absent = on; only off/no/false/manual/disabled flips it false"*): `card.autoBuild` is `undefined` when no line is present, which means "on". **Prior bug (`=== true`):** a spec with no `**Auto-build:**` line (the common case) or a structured caller that omitted `autoBuild` landed `auto_build=false`, which silently PARKED **every** autonomously-authored spec — Pia's goal decomposition, spec-chat, repair/security/director-authored — so a greenlit goal's decomposed specs never built (`noop-goal-test` had to have it flipped by hand). The only deliberate opt-outs (`request-fix` route + `pre-merge-fix`) set `autoBuild: false` **explicitly** and are unaffected by the default flip — a fix spec still needs commission.
- **`extractPhaseBodies(raw)`** — per-phase body extractor (sibling to `parseSpec` which only captures title + status). Position is 1-indexed and lines up with `parseSpec().phases[i]`. Shared with [[../recipes/backfill-specs-from-markdown]] to keep one impl.
- **`extractRepairSignature(raw)`** — the SIGNATURE TEXT (not just presence) for a Repair-Agent-authored spec — `**Repair-signature:** ``<sig>```. Pulls into `specs.repair_signature`.
- **`extractRegressionHeaders(raw)`** → `{ ofSlug, signature }` — the regression-agent header lines `**Regression-of:** [[<slug>]]` + `**Regression-signature:** ``<sig>```. Pull into `specs.regression_of_slug` / `specs.regression_signature` (typed columns added in the [[../specs/spec-authoring-writes-db-and-worker-materialize]] Phase 1 additive migration).
- **`toDbStatus(s)`** — map a [[brain-roadmap]] `SpecStatus` to a [[specs-table]] `SpecStatus` DB enum. `'rejected'` (a phase-only state) is rewritten to `'planned'`.
- **(internal, both entry points) `**Brain refs:**` suggester** — [[../specs/spec-brain-refs]] Phase 2. After the Verification gate + before the `upsertSpec` write, [[brain-ref-suggest]] scans the body for the src/ files, tables, and existing brain wikilinks it names, resolves each to a `docs/brain/{libraries|inngest|tables|…}/{name}.md` verified on disk, and injects the top ≤4 as a `**Brain refs:**` line just below the last metadata header. Best-effort: nothing mappable ⇒ no line (Phase 1's grep-the-brain fallback covers it); a body already carrying a `**Brain refs:**` line (the author picked) is passed through unchanged. The line lives IN the summary text ([[build-spec-materializer]] Rendered shape) — no new column, no SDK change. The [[../.claude/skills/spec-chat|spec-chat]] finalize instructions also ask the box to propose refs; this suggester is the deterministic safety net. **Skippable ([[../specs/fix-spec-brain-refs]]):** both entry points also short-circuit the suggester when the body carries either durable skip signal — an empty `**Brain refs:**` header (no wikilinks after the colon) or an `<!-- brain-refs: skip -->` HTML comment marker anywhere on the scan surface. Both persist "author explicitly picked NONE" through re-authoring so a refine can skip refs without them being re-injected. See [[brain-ref-suggest]] `hasBrainRefsSkip`.
- **(internal) `reopenIfReauthoredAndChanged(workspaceId, slug, existing, next)`** — the **re-author-re-opens-dismissed** root patch. Both entry points snapshot the PRE-upsert row (`getSpec`) and, AFTER the upsert, call this: if the spec EXISTED and its CONTENT CHANGED (title / summary / any phase title·body·verification differs, whitespace-normalized) it RE-OPENS the spec — `[[spec-card-state]] markSpecCardBackToReview` resets the review signals (`vale_pass` / `vale_review_passed_at` / `ada_disposition` / `intended_status`) and sets `specs.status='in_review'`, and `[[director-activity]] clearDirectorSpecDismissals` clears the standing init/groom dismissal ledger. No-op for a brand-new spec, an identical/metadata-only re-author, or a folded spec. Best-effort (never fails the authoring write that already landed).

## Caller patterns

**The ONE sanctioned author path — structured, with typed machine checks:**

```ts
await authorSpecRowStructured(workspaceId, slug, {
  title, summary, owner, parent, blocked_by,
  why,  // plain-language WHY this spec exists (REQUIRED, non-empty)
  what, // plain-language WHAT changes when it ships (REQUIRED, non-empty)
  phases: [{
    title, body, verification,
    why, what,
    checks: [
      // At minimum a tsc gate — every autonomous fix-spec class already gates on tsc-clean before merge.
      { position: 1, description: "Repo typechecks clean after the fix lands.", kind: "auto", exec_kind: "tsc" },
      // When the fix touches a known file, add a grep check on that file for a fingerprint of the guard/fix.
      { position: 2, description: "guardBar landed in src/lib/foo.ts", kind: "auto",
        exec_kind: "grep", params: { path: "src/lib/foo.ts", pattern: "guardBar", expect: "present" } },
    ],
  }],
}, "planned", { intendedStatusSetBy: "planner", milestoneId });
```

**Deterministic producers (converted lanes)** — a lane authored by an LLM whose output is a structured proposal (repair-agent's Rafa, coverage-register's inferred entry, director-followup's markdown → structured coercion) uses a pure helper to build the `StructuredSpecInput` and hands it straight to `authorSpecRowStructured`. Each helper's tests assert every phase carries a machine-runnable `exec_kind`:

- [[coverage-register-agent]] `buildRegisterSpecBody` / `buildExemptSpecBody` (Phase 1) — typed `grep` on `src/lib/control-tower/registry.ts` for the loop id.
- [[repair-agent]] `buildRepairSpecInput` (Phase 2) — typed `tsc` default + optional `grep` on the implicated file for the fix fingerprint.
- `buildStructuredSpecInputFromMarkdown(slug, markdown)` in this module (Phase 3) — coerces a validated director-followup markdown body into the structured shape with a default `tsc` check per phase; prose Verification bullets ride verbatim on the phase's `verification` column for founder review.

**Retired markdown authoring** (pre-Phase-4, allow-listed):

```ts
// ❌ NEW autonomous callers of markNewSpecInReview / authorSpecRowFromMarkdown are rejected by
// `scripts/_check-no-markdown-spec-authoring.ts`. The pre-Phase-4 lanes below are Phase-4 debt
// awaiting per-lane conversion; each has an allow-list entry with a written reason and a target
// conversion helper (mirror buildRepairSpecInput / buildRegisterSpecBody / buildStructuredSpecInputFromMarkdown).
await markNewSpecInReview(workspaceId, slug, "planned", actor, reason, markdown);
```

`runPlanJob`'s RESUME pass asks Pia for the spec bodies as STRUCTURED JSON (phases + a non-empty `verification` per phase), then authors each via `authorSpecRowStructured` BEFORE queuing any build — so a build is never queued for a spec that wasn't authored. There is no `docs/brain/specs/*.md` write and no `git status` glob anywhere on this path.

**Bounded per-turn (planner-authoring-survives-large-multi-spec-output Phase 2).** The RESUME pass no longer asks Pia to author every approved spec in one giant final JSON — a 62KB, 6-spec envelope silently dropped to zero specs during ingestion (job `d5999907`). The worker now splits the approved list via `chunkForAuthoring` (`scripts/planner-chunk-authoring.ts`, default K=2) and runs one `claude -p` turn per batch, session-chained. Each authored spec commits to `public.specs` immediately, so a failure on batch k+1 leaves batches 1..k persisted in review — never a rollback to zero. The Phase 1 transcript-fallback (`scripts/planner-transcript-recover.ts`) still applies per batch. Downstream gates (getSpec re-check, `queueRoadmapBuild`) run once against the union of landed slugs.

**spec-chat authors DB-only (no `.md` commit).** The `runSpecChatJob` finalize/verify lanes (Sage's authoring chat) DON'T commit `docs/brain/specs/{slug}.md` to `main` — under spec-pm-markdown-purge / retire-md-reads the spec lives ONLY in `public.specs` + `public.spec_phases`. Sage WRITES the spec markdown into the worktree as a **scratch buffer**; the worker reads it and authors the DB row via this chokepoint (`markNewSpecInReview` for a fresh spec, `authorSpecRowFromMarkdown` for a refine), then **validates by reading the row back** (`getSpec` → a spec row + ≥1 phase must exist) — the old "box did not write a spec file" markdown-existence gate is gone. The finalize slug is derived from the buffer filename / the model's JSON hint and **guarded to a kebab slug** (a new-feature chat is keyed by a UUID; a UUID is never allowed to become the spec slug). For a refine/verify the worker first MATERIALIZES the existing row back to `docs/brain/specs/{slug}.md` (via [[build-spec-materializer]] `renderSpecRow`) so Sage can Read it as grounding — read-then-author, never read-from-disk-canonical.

## Idempotency

Re-running `authorSpecRowFromMarkdown` with the same body is a no-op against the rolled-up `specs.status` and the phase set: `upsertSpec` UPSERTs by `(workspace_id, slug)` and REPLACES phases by `(spec_id, position)` preserving each phase's stable `id + pr + merge_sha + created_at`. A same-slug convergence (e.g. a recurring repair-agent signature) keeps the original card row's provenance intact.

## Gotchas

- **`owner` is normalized to a bare function slug.** Both entry points call `normalizeOwnerSlug` to strip any `[[../functions/…]]` wikilink wrapper before writing to `specs.owner` — so an author can pass `"[[../functions/platform]]"` and the DB stores `"platform"`. This defensive normalizer prevents a regression where pre-merge-red authoring wrote mangled wikilink-wrapped owners ([[../specs/fix-pre-merge-red-owner-shape]]) and was added to every authoring surface.
- **`intendedStatus` is the author's SUGGESTION.** Vale's disposition lane reads the field; it's not binding. Default to `'planned'` (a bug fix you don't want built is a contradiction) unless the surface knows the author meant `'deferred'`.
- **Verification is a HARD gate, the DB write is best-effort.** A phase with no non-empty `## Verification` THROWS `MissingVerificationError` (caught by the planner loop → fails the job loudly). A genuine DB/upsert failure logs + returns `false`. Both entry points run the SAME gate.
- **Machine-runnable verification is a HARD gate too ([[../specs/every-spec-writer-authors-machine-runnable-verifications]] Phase 1).** After the presence gate above, `assertEveryPhaseHasMachineCheck(slug, phases, ctx?)` runs on the phase-check array (structured path: caller-provided `checks`, else derived from the verification blob; markdown path: derived from `parseVerificationBlobToChecks`). Every phase MUST yield ≥1 check whose `exec_kind` is auto-testable (tsc | grep | ci_status | http_get | db_probe_readonly | unit_test | build) AND passes `validateExecutableCheck`. A phase with only prose / only `needs_human` rows throws `MissingMachineCheckError` naming the offending phase + the first failing `validateExecutableCheck` reason (so a mistyped `unit_test.script` fails at author time, not runtime — closes the cs-director `npm test` class). `needs_human` rows are LEGAL as EXTRA advisory checks; they just can't be the SOLE verification. Applies to BOTH author entry points so every writer inherits it — no writer can land a prose-only spec. CEO decision 2026-07-11: machine-runnable is mandatory; human tests are advisory / non-blocking (see `human_review` below).
- **`human_review` — OPTIONAL, non-blocking founder-facing advisory ([[../specs/every-spec-writer-authors-machine-runnable-verifications]] Phase 2).** `StructuredSpecInput.human_review` (or `AuthorSpecOpts.humanReview` — opts wins over the input) + the markdown `**Human-review:**` header (via `extractHumanReviewHeader`) carry an eyeball prompt ("after ship, open /dashboard/x and confirm the layout reads right"). Persists to `public.specs.human_review` (additive migration). Rendered on the spec card via [[brain-roadmap]] `serializeSpecRowToMarkdown`. NEVER read by the fold gate / promote gate / deterministic spec-check runner — machine-runnable `spec_phase_checks` remain the sole ship gate. Absence is the norm.
- **`upsertSpec` now self-gates too — a FLOOR beneath these gates (harden-spec-submission hotfix).** [[specs-table]] `upsertSpec` throws `UngatedSpecAuthorError` before any write if a phase's effective `verification` is empty, the spec's effective `why`/`what` is empty, or there are zero phases. For the sanctioned author-spec paths this is REDUNDANT (the gates above assert the same fields FIRST, so the floor never fires here); it exists to catch a RAW `upsertSpec` bypass that skipped this chokepoint entirely (the 2026-07-10 verification-NULL specs). Note the floor requires per-phase verification + SPEC-level `why`/`what` only — it does NOT require per-phase intent, so the markdown path's legitimately-NULL phase intent still passes it.
- **Bare-function parent auto-anchors to a real mandate.** BEFORE `assertValidParent` fires, both entry points call `autoAnchorBareFunctionParent(parent, {title, why, what})` — a deterministic self-correction that catches the exact shape Vale would otherwise bounce forever. Matches: a parent that names `functions/{slug}` but has NO `#anchor`, NO `mandate` keyword, and NO goal reference (`detectBareFunctionParent`). On a match, [[function-mandates]] `resolveFunctionMandates({slug})` reads `docs/brain/functions/{slug}.md` for its `## Mandates` section — each `### heading` (with optional `{#slug}` Markdown-Extra anchor) becomes a mandate. If the function has **exactly one** mandate the parent anchors there; if **several**, `bestFitMandate` picks the mandate whose heading+body has the strongest DISTINCT-term overlap with the spec's `title + why + what` (ties → declaration order in the charter). The choicepoint REWRITES `parent` to the canonical `[[../functions/{slug}]] — "{Mandate heading}" mandate: {short reason}.` form + persists `parent_kind='mandate'` + `parent_ref='{slug}#{mandate-slug}'` — so `assertValidParent` no longer sees a bare-function parent, the row lands typed, and Vale reads a real mandate reference on the first pass. **Zero-mandate fallback (the throw survives):** when a function has NO `### mandate` headings under `## Mandates` (nothing to anchor to), the helper returns `null` and the caller falls through to the existing `assertValidParent` → `InvalidParentError` — the invariant "a parent must resolve to a real mandate or milestone" is preserved. Skipped when `opts.milestoneId` is bound (the goal path owns the parent) OR when the caller declared a typed `parentKind:"mandate"|"milestone"` (trust the caller). Optional `opts.onAutoAnchor(result)` callback hands the chosen `FunctionMandate` back so an author surface (the Improve tab's ticket-improve executor — [[../functions/cs]]) can show the human "auto-anchored to X." See [[function-mandates]] for the mandate resolver's design. Pure fs + no LLM.
- **Parent gate — `assertValidParent` (one-off-spec-parent + no-spec-parent).** Both entry points call `assertValidParent(parent, { milestoneId, parentKind })` before the write. It enforces the SAME rule Vale does — CLAUDE.md's `Function → (Mandate | Goal→Milestone) → Spec` hierarchy — so a bad parent fails FAST here instead of looping forever in review. A parent is valid iff it resolves to a function **MANDATE** (`[[../functions/{fn}]]` + a `#anchor` or the word "mandate") OR a goal **MILESTONE** (`[[../goals/{slug}]]` + a `#anchor`/`(M#)`/"milestone" token), OR the caller declared a typed `parentKind:"mandate"|"milestone"` / bound a `milestoneId`. Everything else THROWS `InvalidParentError`, with a message tuned to the shape: a bare goal (no milestone), a **sibling-`../specs/` parent** (a spec is NEVER a parent — use `relatedSpec`), a bare function (no mandate), or free-text provenance. This closes the loop where the self-healing agents (repair / db-health / security / regression / coverage) authored fix-specs with `extends [[../specs/…]]` parents that Vale bounced every pass. CONSERVATIVE only on the escape hatches (typed `parentKind`/`milestoneId` are trusted, so the blueprint→build path is untouched). On the structured path the throw propagates (fail-loud, like the Verification gate); on the softer markdown path it's caught → `false`. Re-parent an existing stuck spec via [[specs-table]] `setSpecParent`.
- **`opts.relatedSpec` + `extractRelatedSpecHeader` (no-spec-parent).** A fix-spec that relates to an origin spec sets the pointer on `specs.related_spec` (a LINK), NEVER the parent. `authorSpecRowStructured` takes `opts.relatedSpec`; the markdown path parses a `**Related-spec:** <slug>` header via `extractRelatedSpecHeader` (how the repair agent emits it). Both persist through `upsertSpec`; `undefined` preserves on re-author.
- **DB-only — no `.md`.** Nothing here commits `docs/brain/specs/{slug}.md`. The readers ([[brain-roadmap]] `getSpec`/`listSpecs`, the board, the build pipeline) read the DB rows.
- **Re-authoring a DISMISSED or already-reviewed spec RE-OPENS it (when content changed).** Ada's `init_dismissed` / a groom `groomed_dismissed` is a STANDING `director_activity` dedup row (`metadata.init_key`/`groom_key`) the init/groom lanes scan to SKIP a spec forever — and a prior Vale pass leaves `vale_review_passed_at` stamped. If a spec is corrected-after-rejection but those stale signals stay, the corrected content sits DEAD under the old verdict (the `migration-pricing-preserved-base-above-msrp` gap). `reopenIfReauthoredAndChanged` (above) closes it: a content-changing re-author resets the review signals + flips to `in_review` (Vale re-reviews the NEW content, Ada re-disposes) AND clears the dismissal ledger. Same class as the orphan-park fixes. A NO-OP re-author (identical content) does NOT churn Vale.

- **Goal-bound Parent prose is DERIVED, not hand-authored (`formatMilestoneParentProse` / `deriveMilestoneParentProse`).** When a caller passes `opts.milestoneId`, the chokepoint OVERRIDES `spec.parent` with a canonical `[[../goals/{slug}#{milestone}]] — M{n} "{title}" milestone.` rendered from the bound milestone. `milestone_id` is the single source of truth; the `**Parent:**` line is its projection — so a goal-bound spec can never drift to a bare-goal parent (the #1 Vale bounce). Closes the systematic 2026-07 planner regression: the plan-author set `milestone_id` correctly on all 14 specs but wrote the prose as the bare goal, and Vale (which reads the rendered prose, not the FK) bounced every one. Pure formatter unit-tested in `author-spec-parent-derive.test.ts`.
