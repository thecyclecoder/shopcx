# inngest/platform-director-cron

The **standing cadence** for the box-hosted **Platform/DevOps Director** ([[../specs/platform-director-agent]], M5 [[../specs/director-loop-grading]] Phase 1 + 3). The director runs **event-driven via [[approval-enqueue-director]]** ŌĆö a `platform-director` [[../tables/agent_jobs]] row is enqueued **sub-minute** when a Platform approval is routed to it ([[../specs/ada-reacts-to-approvals-immediately-never-sits]] Phase 1); this cron is the **gated backstop** (dropped events, cold workspaces). What this adds: a **reliable beat** so escorting approved goals through their milestones + watching the platform happen even when no approval happens to arrive. The box has no internal ticker, so (exactly like [[triage-escalations]] / [[spec-test-cron]]) an Inngest cron is the trigger. Mirrors [[daily-analysis-report-cron]]'s daily cron shape.

Eight halves: (1) the **enqueue** ŌĆö purely the box-job insert, no reasoning. (2) the **director grading loop** ([[../specs/director-loop-grading]] Phase 3 + [[../specs/grading-cascade-to-box-sessions]] Phase 3, `grade-concluded-director-calls` step) ŌĆö on the same beat it grades every recently-CONCLUDED director call (each autonomous auto-approval + each escorted milestone that landed) 1ŌĆō10 into [[../tables/director_decision_grades]] via a **box-hosted** grader. **Phase 3 moved director grading BOX-SIDE** (CEO directive 2026-06-30 ŌĆö every grader box-side): instead of calling `gradeConcludedDirectorCalls` inline (the deployed runtime can't see the approved diff and has to trust the director's own `reasoning` string + a repeat-failure count), the cron picks candidates via [[../libraries/director-grader]] `pickDirectorGradeBatch` and inserts ONE [[../tables/agent_jobs]] row `kind='director-grade'` per batch-ready workspace with `instructions.candidates=[ŌĆ”]`. The box lane (`runDirectorGradeJob` in `scripts/builder-worker.ts`, concurrency-1 Max session) git-shows the target build's merged commit for each auto-approval + the member specs' merged commits for each goal-escort, then writes `director_decision_grades` via `applyBoxDirectorGrade` (same partial-unique upsert + `graded_by='human'` override invariant as the deployed sweep). No more `ai_token_usage` rows with `purpose='director_decision_grading'`. A call is CONCLUDED once its target build reaches a terminal status ŌĆö `completed’Į£merged’Į£failed’Į£needs_attention`; **`merged` is terminal-success** (`building ŌåÆ completed ŌåÆ merged` = PR landed on `main`), so a build sitting at `merged` is fully gradeable (omitting it was the STARVED-grading root cause ŌĆö concluded builds that landed sat un-gradeable for days while the cron stayed green). (3) the **worker grading + coaching loop** ([[../specs/worker-grading-and-director-management]] Phase 2 + [[../specs/grading-cascade-to-box-sessions]] Phases 1+2, `grade-and-coach-workers` step) ŌĆö one level DOWN the cascade: it enqueues a box-hosted grading pass over every recently-CONCLUDED worker action (grading compute moved BOX-SIDE per the CEO directive 2026-06-30 ŌĆö the deployed Sonnet sweep only saw job-row metadata + `log_tail` and capped grades at 7-9 "because I can't see the diff"). Picks the batch via [[../libraries/agent-grader]] `pickAgentGradeBatch` (same selection as the deprecated sweep ŌĆö failures first, round-robin sample of successes, proven workers throttled), then inserts ONE [[../tables/agent_jobs]] row `kind='agent-grade'` per batch-ready workspace with `instructions.agent_job_ids=[ŌĆ”]`. The box lane (`runAgentGradeJob` in `scripts/builder-worker.ts`, concurrency-1 Max session) reads the REAL merged diff and writes [[../tables/agent_action_grades]] via `applyBoxGrade`. **consolidate-grade-coach-one-session (2026-07-01): coaching is now a FOLLOW-ON inside that SAME `agent-grade` session, and the cron no longer fans it out.** The cron dropped its per-kind `detectGradeDropCoaching` loop (which enqueued one `agent-coach` box job per slipped kind ŌĆö each re-hydrating ~550K context + re-reading diffs the grade session already read; the live "4 agent-coach queued" pileup). Now the grade session emits `coachings[]` off the diffs it already read and applies them via `applyBoxCoaching`, so a grading beat produces ONE grade+coach lane per director, not a stack. `detectGradeDropCoaching` (with `alreadyCoachedKinds`) still runs per graded kind INSIDE the box session as the recovery-announce + fallback path (a kind coached inline is skipped, `reason='coached_inline'`). **Phase 2 background:** the coaching SYNTHESIS runs box-side on Max (reads REAL diffs) instead of the deployed `synthesizeCoaching` API call. The box lane (`runAgentCoachJob`, concurrency-1 Max session) reads the low-graded jobs' REAL merged diffs, distills ONE durable coaching learning, then calls `applyBoxCoaching` ŌĆö which re-checks rollup + loop-guard and delegates the write to the unchanged `coachAgent` (or hands off to `rollCoachingIntoFixSpec` after too many un-stuck coachings). No more `ai_token_usage` rows with `purpose='agent_coaching_synthesis'`. **Batched** ŌĆö a workspace is enqueued only when `agentGradingBatchReady` (Ōēź5 ungraded concluded jobs OR the oldest >~3h) so the box lane runs at most one grading session per batch. Same-workspace dedup: skip re-enqueueing while an `agent-grade` job is already queued/building. **(8) the grading-liveness step** (`emit-grading-liveness`, fix-starved-grading) ŌĆö writes a dedicated [[../tables/loop_heartbeats]] beat `loop_id='director-decision-grading'` each sweep carrying `{director:{considered,graded}, worker:{considered,graded,coached}, starved}`, and opens a warn-level [[../tables/loop_alerts]] row (`grading_starved:director-decision-grading`, `owner='platform'`) when grading is STARVED for Ōēź2 consecutive sweeps (`considered>0` but `graded==0` across both layers), auto-resolving the moment grading flows again ŌĆö so a future un-gradeable status can't silently starve grading unseen. (4) the **Platform Scorecard daily pulse** ([[../archive.d/platform-scorecard-engine]] Phase 3, `snapshot-platform-scorecard` step) ŌĆö on the same beat it snapshots the daily KPI set (loop health ┬Ę error backlog + derived MTTR ┬Ę build throughput ┬Ę **lane utilization + enqueue rate** (the build-pool saturation KPIs, [[../specs/director-initiation-throughput]] Phase 4) ┬Ę autonomy ratio ┬Ę escalations) into [[../tables/platform_scorecard_snapshots]] via [[../libraries/platform-scorecard]] `computePlatformScorecard(ws, { cadence:'daily', windowDays:1 })` with the daily cadence's lagged snapshotDate (yesterday UTC, per [[../specs/devops-kpi-daily-snapshot-date-lag-fix]] Phase 1), **guarded to once per UTC day per workspace** ŌĆö but the done set is built only from rows whose `updated_at >= endIso(snapshotDate)` ([[../specs/kpi-daily-snapshot-heal-stale-inflight-rows]] Phase 1), so any stale pre-lagged-fix row (written DURING its own snapshot_date, before the day's UTC cutoff) falls OUT of the done set and is re-snapshotted on the next beat, healing in place via the idempotent upsert; the guard is otherwise spend-saving ŌĆö the upsert already makes a same-day re-run a no-op. (5) the **Platform Scorecard weekly rollup** ([[../specs/platform-scorecard-weekly]] Phase 2, `snapshot-platform-scorecard-weekly` step) ŌĆö on the same beat it snapshots the weekly throughput + quality KPI set (specs/week ┬Ę build success rate ┬Ę ideaŌåÆmerge cycle time ┬Ę % approvals untouched ┬Ę per-worker grade rollup ┬Ę regressions caught) into [[../tables/platform_scorecard_snapshots]] via [[../libraries/platform-scorecard]] `computePlatformScorecard(ws, { cadence:'weekly', windowDays:7, snapshotDate })` with the weekly cadence's lagged snapshotDate (the Sunday immediately before the current ISO Monday, per [[../specs/devops-kpi-weekly-snapshot-date-lag-fix]] Phase 1), so the trailing 7-day window is [prev Mon 00:00Z, prev Sun 23:59:59Z] ŌĆö a fully closed ISO week ŌĆö and the persisted weekly snapshot byte-matches the audit's ground-truth re-derivation (clears the `loop:kpi_drift:approvals_untouched_pct:weekly` false-positive at the writer instead of masking it at the audit). **Guarded to once per ISO week per workspace** by matching the same lagged `snapshot_date` PLUS a two-condition `updated_at` floor `.gte('updated_at', max(endIso, '2026-07-02T21:05:00Z'))` ([[../specs/kpi-weekly-snapshot-done-guard-heal-pre-sdk-stale-rows]] Phase 1) ŌĆö the `endIso` half mirrors the daily's heal-in-place pattern (a row written DURING its own snapshot_date, `updated_at < endIso`, falls OUT of the done set and is re-snapshotted, healing in place via the idempotent upsert), and the hotfix-cutoff half forces one-time re-write of pre-[[../specs/director-kpi-sdk]] weekly rows whose slugŌåÆowner map was built from the OLD `getRoadmap` live-only compute (so ISO weeks written days late ŌĆö after most `spec_slugs` had folded out of the live set ŌĆö came out as 0 and disagreed with the audit's folded-inclusive `listSpecs` re-run; this healed the 2026-06-28 stale specs_per_week row and cleared the `loop:kpi_drift:specs_per_week:weekly` false-positive at the writer instead of masking it at the audit); the upsert on (workspace_id, metric_key, cadence, snapshot_date) makes a same-week re-run a no-op anyway. (6) the **Platform Scorecard monthly leading curve** ([[../specs/platform-scorecard-monthly]] Phase 2, `snapshot-platform-scorecard-monthly` step) ŌĆö on the same beat it snapshots the monthly KPI set (human-touch-per-build ┬Ę goals escorted unbabysat ┬Ę time-to-approve ┬Ę deploy reliability ┬Ę director-call grade) into [[../tables/platform_scorecard_snapshots]] via [[../libraries/platform-scorecard]] `computePlatformScorecard(ws, { cadence:'monthly', windowDays:30, snapshotDate })` with the monthly cadence's lagged snapshotDate (yesterday UTC, per [[../specs/devops-kpi-monthly-snapshot-date-lag-fix]] Phase 1), so the trailing 30-day window is [snapshotDate ŌłÆ 29, snapshotDate] ŌĆö a fully closed window whose merged-build count and human-touch approval-decision count cannot change after the snapshot is written ŌĆö and the persisted monthly snapshot byte-matches the audit's ground-truth re-derivation (clears the `loop:kpi_drift:human_touch_per_build:monthly` false-positive at the writer instead of masking it at the audit). **Guarded to once per lagged UTC day per workspace** by matching the same lagged `snapshot_date` (a second beat within the same UTC day sees the yesterday-dated row and skips); the upsert makes a same-date re-run a no-op anyway. (7) the **Platform Scorecard audit pass** ([[../specs/devops-kpi-review-sdk-and-data-fix]] Phase 5, `audit-platform-scorecard` step) ŌĆö on the same beat it re-derives every advertised KPI from the raw tables via [[../libraries/kpi-review]] `auditAllKpis(workspaceId, cadence)` for daily / weekly / monthly, **upserts one [[../tables/kpi_audit_log]] row per metric** (idempotent on `snapshot_date`) with the persisted snapshot value, the ground-truth value, the drift, and the within-tolerance verdict, then opens a [[../tables/loop_alerts]] row (`loop_id`/`signature` = `kpi_drift:<metric>:<cadence>`, `owner='platform'`, `kind='kpi-drift'`, `reason='kpi_drift'`) when a metric has been over-tolerance for **Ōēź2 consecutive snapshots** ŌĆö a single-snapshot drift is logged but NOT alerted (self-healing on transient timing noise). Auto-resolves the open alert the moment the metric recovers to within-tolerance. The grade sweeps + all three scorecard snapshots + the audit pass run HERE (the deployed runtime has DB access + `ANTHROPIC_API_KEY`), not on the box; the grade sweeps mirror [[acquisition-research-cadence]]'s. Best-effort + idempotent; a no-op / zeros while nothing is ungraded or the workspace is quiet.

**File:** `src/lib/inngest/platform-director-cron.ts` (registered in `src/lib/inngest/registered-functions.ts` ŌåÆ served by `src/app/api/inngest/route.ts`)

## Functions

### `platform-director-cron`
- **Trigger:** cron `*/5 * * * *` (every 5 min ŌĆö a TIGHT standing beat so the build pool stays saturated, [[../specs/director-initiation-throughput]] Phase 3 tightened it 15ŌåÆ5; the in-flight dedupe below prevents pileup). This cron is now the **BACKSTOP** heartbeat ŌĆö the **event-driven top-up** ([[../libraries/agent-jobs]] `enqueueDirectorTopUp`, fired from `applyMergedBuildEffects` on a build merge) refills a freed lane within seconds, so the pool re-saturates between beats too. **Build-lane reactive companion** ([[../specs/bo-reactive-gated-build-enqueue]] Phase 2): a `build/spec-build-eligible` event fires on the Vale-pass / Ada-planned transition and [[build-on-eligible]] runs the same gated `enqueueBuildIfDue` chokepoint on receipt ŌĆö so this cron catches only dropped events / cold workspaces (Ada's lanes already gate on `specReviewDone`, so a cron/event race no-ops on the one-in-flight guard).
- **Retries:** 1
- **Concurrency:** `concurrency: [{ limit: 1 }]`

## What it enqueues

For each **build-console workspace** ŌĆö any workspace that uses the agent-jobs queue (has at least one [[../tables/agent_jobs]] row, mirrors [[spec-test-cron]]) ŌĆö it inserts one `queued` `agent_jobs` row `kind='platform-director'` with **no `target_job_id`**. The box claims it on its platform-director lane (`scripts/builder-worker.ts` ŌåÆ `runPlatformDirectorJob`); a target-less job runs the **standing pass** (`runPlatformDirectorStandingPass`): `escortApprovedGoals` (escort each approved goal's unblocked specs through their milestones, Phase 2), `groomBoard` (board-grooming, [[../specs/board-grooming]] ŌĆö assess each partially-shipped spec and **continue / split / escalate** its leftover phases via a per-candidate Max investigation), **and** `postPlatformWatchUpdate` (read [[../dashboard/control-tower]] health via its snapshot library + post the daily human-readable watch update as ­¤øĀ’ĖÅ Ada to the [[../tables/director_messages]] board, Phase 4). All are no-ops unless Platform is `live + autonomous` ([[../tables/function_autonomy]]).

**Idle work-gate + adaptive cadence backoff** ([[../specs/ada-standing-pass-reasoning-gate]] Phase 4). Before enqueueing a workspace's standing-pass job the cron calls [[../libraries/platform-director]] `platformHasPendingWork(workspaceId)` ŌĆö a cheap `.select("id").limit(1)` sweep across every input the pass consumes (active directive ┬Ę open `spec_drift` ┬Ę `agent_jobs` `needs_attention` or building > 90m ┬Ę open `error_events` ┬Ę open `loop_alerts` ┬Ę non-terminal `specs` `in_review’Į£planned’Į£in_progress` ┬Ę complete non-parent `goals` ┬Ę coverage past `PLATFORM_DIRECTOR_REVERIFY_WINDOW_MS` ┬Ę unresolved evidence-backed regression fails). PENDING ŌåÆ enqueue as usual. NOT PENDING ŌåÆ back off from `*/5` to hourly across a solid idle span: the cron reads its own per-workspace gate heartbeats (`loop_id='platform-standing-pass-gate:<workspaceId>'`) ŌĆö any beat with `produced.pending=true` inside the last `PLATFORM_STANDING_PASS_IDLE_MS` (30 min) is `recentlyActive` and the cron stays on `*/5` (skips only enqueue on a hourly-floor tick); a solid idle span (no pending signal in 30 min) actually flips the effective cadence to hourly (enqueue only when the UTC minute is 0). ANY pending signal on the very next tick snaps it back to `*/5`. Every tick per workspace writes ONE gate beat regardless ŌĆö the record next tick reads. The **worker** (`runPlatformDirectorStandingPass`) ALSO calls `platformHasPendingWork` at the top: pending=false ŌćÆ record a quiet beat + skip every lane this pass (a hourly-floor "confirming quiet" enqueue costs one gate call, not a full pass). Fail-open ŌĆö any gate read error runs the full pass. Scope: gates ONLY the box standing pass ŌĆö the grading/coaching cascade keeps its own [[../libraries/agent-grader]] `agentGradingBatchReady` gate (out of scope, see spec `grading-cascade-to-box-sessions`).

## Dedupe

It does **not** enqueue a second job for a workspace that already has an in-flight `platform-director` job (`status` Ōłł `queued’Į£queued_resume’Į£building’Į£claimed`) ŌĆö one standing pass per workspace at a time, never a daily pileup. On top of the in-flight dedupe, the [[../specs/ada-standing-pass-reasoning-gate]] Phase 4 **idle work-gate** additionally skips the enqueue when the workspace has NO pending work AND has been idle for Ōēź30 min ŌĆö enqueue only fires on the hourly-floor tick in that state. See *What it enqueues* above.

## Monitored

Registered in `MONITORED_LOOPS` (`src/lib/control-tower/registry.ts`, `owner: platform`, `livenessWindowMs: 26h`, `registeredAt` for the first-tick grace) so a dead cadence is visible on `/dashboard/developer/control-tower` and can't silently die ŌĆö the [[../specs/coverage-auto-register-agent]] contract. Emits a `loop_heartbeats` beat (`loop_id='platform-director-cron'`) at end-of-run via `emitCronHeartbeat`.

## Downstream events sent

_None._ The box polls [[../tables/agent_jobs]] and claims the row; there is no HTTP call into the box.
