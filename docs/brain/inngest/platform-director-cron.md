# inngest/platform-director-cron

The **standing cadence** for the box-hosted **Platform/DevOps Director** ([[../specs/platform-director-agent]], M5 [[../specs/director-loop-grading]] Phase 1 + 3). The director already runs **event-driven** — a `platform-director` [[../tables/agent_jobs]] row is enqueued when a Platform approval is routed to it ([[../specs/approval-routing-engine]]). What this adds: a **reliable beat** so escorting approved goals through their milestones + watching the platform happen even when no approval happens to arrive. The box has no internal ticker, so (exactly like [[triage-escalations]] / [[spec-test-cron]]) an Inngest cron is the trigger. Mirrors [[daily-analysis-report-cron]]'s daily cron shape.

Eight halves: (1) the **enqueue** — purely the box-job insert, no reasoning. (2) the **director grading loop** ([[../specs/director-loop-grading]] Phase 3 + [[../specs/grading-cascade-to-box-sessions]] Phase 3, `grade-concluded-director-calls` step) — on the same beat it grades every recently-CONCLUDED director call (each autonomous auto-approval + each escorted milestone that landed) 1–10 into [[../tables/director_decision_grades]] via a **box-hosted** grader. **Phase 3 moved director grading BOX-SIDE** (CEO directive 2026-06-30 — every grader box-side): instead of calling `gradeConcludedDirectorCalls` inline (the deployed runtime can't see the approved diff and has to trust the director's own `reasoning` string + a repeat-failure count), the cron picks candidates via [[../libraries/director-grader]] `pickDirectorGradeBatch` and inserts ONE [[../tables/agent_jobs]] row `kind='director-grade'` per batch-ready workspace with `instructions.candidates=[…]`. The box lane (`runDirectorGradeJob` in `scripts/builder-worker.ts`, concurrency-1 Max session) git-shows the target build's merged commit for each auto-approval + the member specs' merged commits for each goal-escort, then writes `director_decision_grades` via `applyBoxDirectorGrade` (same partial-unique upsert + `graded_by='human'` override invariant as the deployed sweep). No more `ai_token_usage` rows with `purpose='director_decision_grading'`. A call is CONCLUDED once its target build reaches a terminal status — `completed｜merged｜failed｜needs_attention`; **`merged` is terminal-success** (`building → completed → merged` = PR landed on `main`), so a build sitting at `merged` is fully gradeable (omitting it was the STARVED-grading root cause — concluded builds that landed sat un-gradeable for days while the cron stayed green). (3) the **worker grading + coaching loop** ([[../specs/worker-grading-and-director-management]] Phase 2 + [[../specs/grading-cascade-to-box-sessions]] Phases 1+2, `grade-and-coach-workers` step) — one level DOWN the cascade: it enqueues a box-hosted grading pass over every recently-CONCLUDED worker action (grading compute moved BOX-SIDE per the CEO directive 2026-06-30 — the deployed Sonnet sweep only saw job-row metadata + `log_tail` and capped grades at 7-9 "because I can't see the diff"). Picks the batch via [[../libraries/agent-grader]] `pickAgentGradeBatch` (same selection as the deprecated sweep — failures first, round-robin sample of successes, proven workers throttled), then inserts ONE [[../tables/agent_jobs]] row `kind='agent-grade'` per batch-ready workspace with `instructions.agent_job_ids=[…]`. The box lane (`runAgentGradeJob` in `scripts/builder-worker.ts`, concurrency-1 Max session) reads the REAL merged diff and writes [[../tables/agent_action_grades]] via `applyBoxGrade`. On the SAME beat, the cron fires [[../libraries/agent-grader]] `detectGradeDropCoaching` over ALL rubric-backed kinds so a slip triggered by newly box-written grades is caught within one beat (idempotent, loop-guarded, dedup-gated). **Phase 2 moved the coaching SYNTHESIS box-side too:** `detectGradeDropCoaching` now ENQUEUES an `agent-coach` `agent_jobs` row (dedup: skip re-enqueue while an `agent-coach` job for this `(workspace, agent_kind)` is already queued/building) instead of the deployed `synthesizeCoaching` API call. The box lane (`runAgentCoachJob`, concurrency-1 Max session) reads the low-graded jobs' REAL merged diffs, distills ONE durable coaching learning, then calls `applyBoxCoaching` — which re-checks rollup + loop-guard and delegates the write to the unchanged `coachAgent` (or hands off to `rollCoachingIntoFixSpec` after too many un-stuck coachings). No more `ai_token_usage` rows with `purpose='agent_coaching_synthesis'`. **Batched** — a workspace is enqueued only when `agentGradingBatchReady` (≥5 ungraded concluded jobs OR the oldest >~3h) so the box lane runs at most one grading session per batch. Same-workspace dedup: skip re-enqueueing while an `agent-grade` job is already queued/building. **(8) the grading-liveness step** (`emit-grading-liveness`, fix-starved-grading) — writes a dedicated [[../tables/loop_heartbeats]] beat `loop_id='director-decision-grading'` each sweep carrying `{director:{considered,graded}, worker:{considered,graded,coached}, starved}`, and opens a warn-level [[../tables/loop_alerts]] row (`grading_starved:director-decision-grading`, `owner='platform'`) when grading is STARVED for ≥2 consecutive sweeps (`considered>0` but `graded==0` across both layers), auto-resolving the moment grading flows again — so a future un-gradeable status can't silently starve grading unseen. (4) the **Platform Scorecard daily pulse** ([[../archive.d/platform-scorecard-engine]] Phase 3, `snapshot-platform-scorecard` step) — on the same beat it snapshots the daily KPI set (loop health · error backlog + derived MTTR · build throughput · **lane utilization + enqueue rate** (the build-pool saturation KPIs, [[../specs/director-initiation-throughput]] Phase 4) · autonomy ratio · escalations) into [[../tables/platform_scorecard_snapshots]] via [[../libraries/platform-scorecard]] `computePlatformScorecard(ws, { cadence:'daily', windowDays:1 })` with the daily cadence's lagged snapshotDate (yesterday UTC, per [[../specs/devops-kpi-daily-snapshot-date-lag-fix]] Phase 1), **guarded to once per UTC day per workspace** — but the done set is built only from rows whose `updated_at >= endIso(snapshotDate)` ([[../specs/kpi-daily-snapshot-heal-stale-inflight-rows]] Phase 1), so any stale pre-lagged-fix row (written DURING its own snapshot_date, before the day's UTC cutoff) falls OUT of the done set and is re-snapshotted on the next beat, healing in place via the idempotent upsert; the guard is otherwise spend-saving — the upsert already makes a same-day re-run a no-op. (5) the **Platform Scorecard weekly rollup** ([[../specs/platform-scorecard-weekly]] Phase 2, `snapshot-platform-scorecard-weekly` step) — on the same beat it snapshots the weekly throughput + quality KPI set (specs/week · build success rate · idea→merge cycle time · % approvals untouched · per-worker grade rollup · regressions caught) into [[../tables/platform_scorecard_snapshots]] via [[../libraries/platform-scorecard]] `computePlatformScorecard(ws, { cadence:'weekly', windowDays:7, snapshotDate })` with the weekly cadence's lagged snapshotDate (the Sunday immediately before the current ISO Monday, per [[../specs/devops-kpi-weekly-snapshot-date-lag-fix]] Phase 1), so the trailing 7-day window is [prev Mon 00:00Z, prev Sun 23:59:59Z] — a fully closed ISO week — and the persisted weekly snapshot byte-matches the audit's ground-truth re-derivation (clears the `loop:kpi_drift:approvals_untouched_pct:weekly` false-positive at the writer instead of masking it at the audit). **Guarded to once per ISO week per workspace** by matching the same lagged `snapshot_date` (a second beat in the same week sees the prev-Sunday row and skips); the upsert makes a same-week re-run a no-op anyway. (6) the **Platform Scorecard monthly leading curve** ([[../specs/platform-scorecard-monthly]] Phase 2, `snapshot-platform-scorecard-monthly` step) — on the same beat it snapshots the monthly KPI set (human-touch-per-build · goals escorted unbabysat · time-to-approve · deploy reliability · director-call grade) into [[../tables/platform_scorecard_snapshots]] via [[../libraries/platform-scorecard]] `computePlatformScorecard(ws, { cadence:'monthly', windowDays:30, snapshotDate })` with the monthly cadence's lagged snapshotDate (yesterday UTC, per [[../specs/devops-kpi-monthly-snapshot-date-lag-fix]] Phase 1), so the trailing 30-day window is [snapshotDate − 29, snapshotDate] — a fully closed window whose merged-build count and human-touch approval-decision count cannot change after the snapshot is written — and the persisted monthly snapshot byte-matches the audit's ground-truth re-derivation (clears the `loop:kpi_drift:human_touch_per_build:monthly` false-positive at the writer instead of masking it at the audit). **Guarded to once per lagged UTC day per workspace** by matching the same lagged `snapshot_date` (a second beat within the same UTC day sees the yesterday-dated row and skips); the upsert makes a same-date re-run a no-op anyway. (7) the **Platform Scorecard audit pass** ([[../specs/devops-kpi-review-sdk-and-data-fix]] Phase 5, `audit-platform-scorecard` step) — on the same beat it re-derives every advertised KPI from the raw tables via [[../libraries/kpi-review]] `auditAllKpis(workspaceId, cadence)` for daily / weekly / monthly, **upserts one [[../tables/kpi_audit_log]] row per metric** (idempotent on `snapshot_date`) with the persisted snapshot value, the ground-truth value, the drift, and the within-tolerance verdict, then opens a [[../tables/loop_alerts]] row (`loop_id`/`signature` = `kpi_drift:<metric>:<cadence>`, `owner='platform'`, `kind='kpi-drift'`, `reason='kpi_drift'`) when a metric has been over-tolerance for **≥2 consecutive snapshots** — a single-snapshot drift is logged but NOT alerted (self-healing on transient timing noise). Auto-resolves the open alert the moment the metric recovers to within-tolerance. The grade sweeps + all three scorecard snapshots + the audit pass run HERE (the deployed runtime has DB access + `ANTHROPIC_API_KEY`), not on the box; the grade sweeps mirror [[acquisition-research-cadence]]'s. Best-effort + idempotent; a no-op / zeros while nothing is ungraded or the workspace is quiet.

**File:** `src/lib/inngest/platform-director-cron.ts` (registered in `src/lib/inngest/registered-functions.ts` → served by `src/app/api/inngest/route.ts`)

## Functions

### `platform-director-cron`
- **Trigger:** cron `*/5 * * * *` (every 5 min — a TIGHT standing beat so the build pool stays saturated, [[../specs/director-initiation-throughput]] Phase 3 tightened it 15→5; the in-flight dedupe below prevents pileup). This cron is now the **BACKSTOP** heartbeat — the **event-driven top-up** ([[../libraries/agent-jobs]] `enqueueDirectorTopUp`, fired from `applyMergedBuildEffects` on a build merge) refills a freed lane within seconds, so the pool re-saturates between beats too.
- **Retries:** 1
- **Concurrency:** `concurrency: [{ limit: 1 }]`

## What it enqueues

For each **build-console workspace** — any workspace that uses the agent-jobs queue (has at least one [[../tables/agent_jobs]] row, mirrors [[spec-test-cron]]) — it inserts one `queued` `agent_jobs` row `kind='platform-director'` with **no `target_job_id`**. The box claims it on its platform-director lane (`scripts/builder-worker.ts` → `runPlatformDirectorJob`); a target-less job runs the **standing pass** (`runPlatformDirectorStandingPass`): `escortApprovedGoals` (escort each approved goal's unblocked specs through their milestones, Phase 2), `groomBoard` (board-grooming, [[../specs/board-grooming]] — assess each partially-shipped spec and **continue / split / escalate** its leftover phases via a per-candidate Max investigation), **and** `postPlatformWatchUpdate` (read [[../dashboard/control-tower]] health via its snapshot library + post the daily human-readable watch update as 🛠️ Ada to the [[../tables/director_messages]] board, Phase 4). All are no-ops unless Platform is `live + autonomous` ([[../tables/function_autonomy]]).

**Idle work-gate + adaptive cadence backoff** ([[../specs/ada-standing-pass-reasoning-gate]] Phase 4). Before enqueueing a workspace's standing-pass job the cron calls [[../libraries/platform-director]] `platformHasPendingWork(workspaceId)` — a cheap `.select("id").limit(1)` sweep across every input the pass consumes (active directive · open `spec_drift` · `agent_jobs` `needs_attention` or building > 90m · open `error_events` · open `loop_alerts` · non-terminal `specs` `in_review｜planned｜in_progress` · complete non-parent `goals` · coverage past `PLATFORM_DIRECTOR_REVERIFY_WINDOW_MS` · unresolved evidence-backed regression fails). PENDING → enqueue as usual. NOT PENDING → back off from `*/5` to hourly across a solid idle span: the cron reads its own per-workspace gate heartbeats (`loop_id='platform-standing-pass-gate:<workspaceId>'`) — any beat with `produced.pending=true` inside the last `PLATFORM_STANDING_PASS_IDLE_MS` (30 min) is `recentlyActive` and the cron stays on `*/5` (skips only enqueue on a hourly-floor tick); a solid idle span (no pending signal in 30 min) actually flips the effective cadence to hourly (enqueue only when the UTC minute is 0). ANY pending signal on the very next tick snaps it back to `*/5`. Every tick per workspace writes ONE gate beat regardless — the record next tick reads. The **worker** (`runPlatformDirectorStandingPass`) ALSO calls `platformHasPendingWork` at the top: pending=false ⇒ record a quiet beat + skip every lane this pass (a hourly-floor "confirming quiet" enqueue costs one gate call, not a full pass). Fail-open — any gate read error runs the full pass. Scope: gates ONLY the box standing pass — the grading/coaching cascade keeps its own [[../libraries/agent-grader]] `agentGradingBatchReady` gate (out of scope, see spec `grading-cascade-to-box-sessions`).

## Dedupe

It does **not** enqueue a second job for a workspace that already has an in-flight `platform-director` job (`status` ∈ `queued｜queued_resume｜building｜claimed`) — one standing pass per workspace at a time, never a daily pileup. On top of the in-flight dedupe, the [[../specs/ada-standing-pass-reasoning-gate]] Phase 4 **idle work-gate** additionally skips the enqueue when the workspace has NO pending work AND has been idle for ≥30 min — enqueue only fires on the hourly-floor tick in that state. See *What it enqueues* above.

## Monitored

Registered in `MONITORED_LOOPS` (`src/lib/control-tower/registry.ts`, `owner: platform`, `livenessWindowMs: 26h`, `registeredAt` for the first-tick grace) so a dead cadence is visible on `/dashboard/developer/control-tower` and can't silently die — the [[../specs/coverage-auto-register-agent]] contract. Emits a `loop_heartbeats` beat (`loop_id='platform-director-cron'`) at end-of-run via `emitCronHeartbeat`.

## Downstream events sent

_None._ The box polls [[../tables/agent_jobs]] and claims the row; there is no HTTP call into the box.

## Tables written

- [[../tables/agent_jobs]] (inserts the `platform-director` job)
- [[../tables/director_decision_grades]] (the director grading loop — one grade per concluded director call, via [[../libraries/director-grader]])
- [[../tables/agent_action_grades]] (the worker grading loop — one grade per concluded worker action, via [[../libraries/agent-grader]])
- [[../tables/agent_instructions]] · [[../tables/agent_coaching_log]] (a grade-drop coaching amendment via `coachAgent`)
- [[../tables/platform_scorecard_snapshots]] (the daily scorecard pulse + the **weekly** throughput/quality rollup + the **monthly** leading curve — one upserted row per daily/weekly/monthly KPI per workspace, via [[../libraries/platform-scorecard]])
- [[../tables/kpi_audit_log]] (the audit pass — one row per advertised KPI per cadence per workspace, drift verdict vs the persisted snapshot, via [[../libraries/kpi-review]] `auditAllKpis`)
- [[../tables/loop_alerts]] (the audit pass opens / refreshes / resolves a `kpi_drift:<metric>:<cadence>` incident — `owner='platform'`, persistent-drift gated to ≥2 consecutive over-tolerance snapshots; **the grading-liveness step** opens / resolves `grading_starved:director-decision-grading` when grading is starved for ≥2 consecutive sweeps)
- [[../tables/loop_heartbeats]] (end-of-run heartbeat `loop_id='platform-director-cron'`; **the grading-liveness beat** `loop_id='director-decision-grading'` carrying the per-layer `{considered,graded}` so a starved sweep is observable; **the per-workspace idle-gate beat** `loop_id='platform-standing-pass-gate:<workspaceId>'` written on every tick carrying `produced.pending` — the record the cron reads next tick to decide the enqueue cadence, [[../specs/ada-standing-pass-reasoning-gate]] Phase 4)

- [[../tables/error_events]] · [[../tables/loop_alerts]] · [[../tables/loop_heartbeats]] (**scorecard daily**: error backlog + MTTR + loop health)
