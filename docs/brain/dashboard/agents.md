# dashboard/agents

The owner-only **Agents hub** ([[../specs/agents-hub-role-inboxes]], M1 of [[../goals/devops-director]]) — the single surface that renders the live org chart (**CEO · Directors · Workers**) read from the brain, and gives every role the same **three-tab inbox** (Messages · Approval Requests · Daily Summaries). The foundation the routing engine (M2), the gamified board (M3), and the live Platform director (M4) all write *into*. Consolidates every approval into one routed surface (replacing the scattered Control Tower feeds, spec cards, the box `approvalHref`): platform-owned approvals route to the **live Platform/DevOps Director (Ada)** (`live + autonomous` since 2026-06-23, [[../tables/function_autonomy]]); the not-yet-automated directors' approvals still land in the **CEO inbox**.

**Route:** `/dashboard/agents` (client poller, owner-only)
**Sidebar:** the **Developer** portal (owner-only) → **Org** group → **Message Board** (`/dashboard/agents`), alongside Org Chart · Directors · Agents — the Org surfaces moved here off the main tree ([[developer]] portal).
**Sibling surface:** [[agents__scorecard|Platform scorecard]] — `/dashboard/agents/scorecard` ([[../specs/platform-scorecard-surface]] Phase 2, milestone (d) of [[../goals/platform-department-scorecard]]) — the daily/weekly/monthly KPI instrument panel, linked from the hub header.

## Surfaces

- **Active directive card** (M1 Phase 5 — [[../specs/director-executable-plans-and-priority-hub-pip]] ✅) — rendered at the **top** of `/dashboard/agents` and on `/dashboard/agents/[director]` (e.g. Platform's profile) when a `director_function='platform'` row exists in [[../tables/director_directives]] with `status='active'`. Displays the plan summary, its ordered steps, and a "builds GATED until {spec_slug} ships" badge if `gate_builds_until` is set. Backed by `GET /api/director/directive?function=platform` (owner-gated, 403 for non-owner) + a **Clear** button (`POST /api/director/directive { id, action:"clear" }`, owner-gated) that flips the row to `cleared` and logs a `directive_cleared` row in [[../tables/director_activity]]. Silent-first-paint when no active directive exists — no empty-state box or flicker. Mirrors the standing-pass + daily-board headlines that already surface the directive, just with a dedicated, always-visible hub surface. Implemented in `src/components/agents/directive-card.tsx` via the [[../libraries/director-directives]] `clearDirective()` export. The director card is mounted on the hub main view + on every role profile ([[../libraries/director-directives]] `clearDirective` is owner-gated; the table and library already existed; this spec added only the UI rendering + the clear affordance).
- **Org chart / Inbox view toggle** (M1 Phase 4) — top-right of the hub, defaulting to the **Org chart** (the company team page). The org-chart view renders `<OrgTree>` (`src/components/agents/org-tree.tsx`) over the **same** `getOrgChart()` payload (no second copy of the org chart): a real tree — **CEO** at top → a Directors rail (each director card carries its **SVG mascot + live/autonomous badge**) → each director's **Workers** (its box [[../tables/agent_jobs]] lanes) as smaller persona cards beneath it. Every node is a **clickable** persona card → that role's **profile detail page** `/dashboard/agents/[role]` (Phase 5 — the nodes are `<Link>`s; no node click is a dead end). The **Inbox** view is the left role nav + three-tab inbox below, and a deep-link `?view=inbox&role=…` opens straight to a role's inbox (how a profile's **Open inbox →** lands).
- **Per-agent KPI pages** ([[../specs/agent-kpi-pages-cleo-first]] — the 2nd supervision layer above [[../libraries/agent-grader|grades]] + [[../libraries/agent-coaching|coaching]]) — `/dashboard/agents/[role]/kpi` renders a per-agent-kind KPI dashboard that answers *"is this agent ON IT?"* (liveness/posture) + *"is it WINNING?"* (outcomes). Cleo (`storefront-optimizer`) is the featured bespoke definition — four tiers (On it / Cadence / Outcome / Quality) sourced from her real tables — with a `<PostureCard>` + predicted-LTV/visitor as the headline row. Every other agent kind (build/repair/ticket-improve/…) renders from the SDK's **generic fallback** (activity + quality tiers over [[../tables/agent_jobs]] · [[../tables/agent_action_grades]] · [[../tables/agent_coaching_log]]) — so a new kind gets a page from day one with no bespoke code. Backed by [[../libraries/agent-kpis|computeAgentKpis]] + `GET /api/workspaces/[id]/agent-kpis?kind=…` (owner-gated). Page: `src/app/dashboard/agents/[role]/kpi/page.tsx` (`"use client"`, polls every 30s, wrapped in `<Suspense>` via `layout.tsx` for `cacheComponents`).
  - **KPIs is a first-class sidebar section** (agents-sidebar-kpis-and-profile-redesign Phase 1) — every WORKER profile now registers a Director-style [[../libraries/section-nav-context|SectionNav]] takeover (Overview + KPIs), closing the discoverability gap where Cleo's KPIs were only reachable via a faint corner link on a profile the hub didn't surface. Directors' section navs gained a matching **KPIs** tab. Both tiers render the same view the standalone `/kpi` route renders via a shared `<AgentKpiView>` (`src/components/agents/agent-kpi-view.tsx`) — the extracted body of the KPI page — so the two surfaces never drift. The CEO profile keeps the main app sidebar (no takeover).
  - **Workers surfaced from the hub + direct KPIs affordance on every worker entry** (agents-sidebar-kpis-and-profile-redesign Phase 3) — closes the Cleo-discoverability gap where workers were only reachable via the Org-chart view or the standalone workers roster. The **Board view's `RoleNav`** now lists every worker beneath its owning director (persona chip + a small **KPIs** link that jumps straight to `/dashboard/agents/[kind]/kpi`), so Cleo (`storefront-optimizer`) is one click from the default hub — no view toggle needed. The **`OrgTree` worker nodes** carry the same small `KPIs →` link as a sibling under the node card (nested `<a>` isn't valid HTML, so it's a second `<Link>` below the persona card). Owner grouping mirrors [[../libraries/approval-inbox]] `ownerFunctionForKind` (a worker lists under the director that owns its kind) — the hub reads as the real org.
  - **Worker profile slimmed to persona-only Overview + one section per heavy element** (agents-sidebar-kpis-and-profile-redesign Phase 2) — the worker profile now mirrors the Director section-switch shape (one component per `?s=`, default = first key). Worker sidebar sections = **Overview** (default — persona avatar + name/role + reports-to + scope + Responsibilities + a small box-lane routing footer, and NOTHING else) · **KPIs** (`<AgentKpiView>`) · **Grades** (`<AgentGradePanel>`) · **Coaching** (`<AgentCoachingHistory>`) · **Activity** (`<DirectorActivity fn=<director> actor=<kind>>`) · **Model tier** (`<ModelTierCard>`). Each section renders in place of the previously-stacked block on the Overview — no data/SDK changes, just a layout redistribution of the existing components (they still fetch their own data). The faint zinc-400 `KPIs →` corner link in the profile header is **retired** — KPIs is a real sidebar tab now. The director profile is unchanged in behavior.
- **Profile detail pages** (M1 Phase 5) — `/dashboard/agents/[role]` renders one page for **every seat** (`ceo` · a director slug · a worker kind), resolved off the same `getOrgChart()` payload: avatar + name + persona + a **tiered responsibilities list**. The **CEO**'s list = the `goals/*.md` company objectives (a **director-proposed** goal — [[../specs/director-proposed-goals]] Phase 2 — reads "⏳ Proposed by {Director} · awaiting your greenlight"; a greenlit one reads "Greenlit · N% complete", so the CEO sees what a director is proposing vs what's activated — `getOrgChart` carries `status`/`proposedBy`/`proposedByLabel` on each `ceo.goals` entry); a **director**'s = its `functions/*.md` mandates (metric + spec count) plus a clickable **Team** strip of its workers; a **worker**'s = the **precise `RESPONSIBILITIES` list** from [[../libraries/agent-personas|personas.ts]] (its exact mandate) + a "routes to the CEO until {director} goes live" note. Below the responsibilities, a **Model tier card** (M1 Phase 5 — [[../specs/box-agent-model-tiers]] Phase 4) shows the agent's current **Model: {tier}** (the resolved tier the box passes as `--model` when running the agent; `null` = the Max-plan default) + the **change history** (recent governed proposals). A pending proposal shows a "Change awaiting approval" alert with a **deep-link to the routed inbox** where the supervisor decides it with one tap (the link mirrors [[../libraries/approval-router]] / [[approval-inbox]]: a worker's change routes to its director, a director's to the CEO). The card itself is read-only; changes route through the proposal path ([[../libraries/model-tier-proposals]]). An unknown role renders a graceful "No such role." card. Reachable from **any** surface — the org-tree nodes, the rosters below, and the inbox `RoleHeader`'s **View profile →** link — same profile page everywhere. Each profile's **Open inbox →** / **reports to …** links thread back to the inbox + the parent's profile.
- **Guide tab** (director-guide-tab spec) — a **second section tab** (right after Overview) on every **director** profile that renders a SUPER human-readable, plain-English intro for a non-technical founder: **who this is** (persona avatar + name + personality + the `functions/*.md` summary + a plain-words status pill — *"Live & autonomous — acts on its own within its limits and escalates the big calls to you" / "Live (review mode) — proposes everything for your approval" / "Offline — all decisions route to you"*), **what she handles vs. brings to you** (two columns derived from the director's leash), **her team** (the agents she supervises, each with avatar + one-line job + a live status dot), and **what she's working on** (her goals + owned-spec count + a link to her live activity). **Everything is runtime-derived, so it self-updates** — register a new agent or add a new `functions/*.md` director and it appears here on the next 45s poll, with **zero manual maintenance**. Rendered by `src/components/agents/director-guide.tsx` (`"use client"`, polls every 45s) over `GET /api/developer/agents/guide?slug=…`. The tab only renders for directors (a worker/CEO profile has no Guide tab). Avatars/names/team blurbs are resolved client-side from [[../libraries/agent-personas|personas.ts]] (`getPersona` + its `RESPONSIBILITIES`), so the route ships only the org-derived data.
  - **Leash generalized to every director** (the key win — the old `director-autonomy.tsx` was Platform-only + "she"-hardcoded) — `src/lib/agents/director-leash-guide.ts` `getLeashGuide(slug)` derives the "I handle these myself ✓ / I bring these to you ↑" lists from **that director's own `LEASH_CATEGORIES`** export (`platform-director.ts` · `growth-director.ts`), registered in a `DIRECTOR_LEASH` map and rendered via a per-category friendly-sentence map. So **Growth (Max)** shows Growth's real envelope (*reallocate spend within the approved budget on her own; escalate raising the total budget or entering a new ad platform*) and **Platform (Ada)** shows hers (*approve a confirmed bug fix / a safe additive migration*) — no hardcoding, no drift from the code gate. Escalations = the generic rails (destructive/irreversible · new feature/goal · anything unverifiable) plus per-director extras (Growth: raise-budget / new-platform / non-binary choice). A director with **no `<name>-director.ts` leash module** returns `{ defined: false }` → the Guide shows a graceful *"leash not yet defined."* **Adding a new leashed director** = add its `LEASH_CATEGORIES` import to `DIRECTOR_LEASH` + a copy line per new category.
- **Directors + Workers rosters** (M1 Phase 5 — the top-level Agents nav items, flipped live) — `/dashboard/agents/directors` lists the CEO + every `functions/*.md` director as clickable rows (mascot + status badge + mandate/worker counts) → their profiles; `/dashboard/agents/workers` lists every box `agent_jobs` lane grouped under its owning director, each chip → its profile. Both brain-driven over `GET /api/developer/agents` — add a director / lane and it appears with no code change.
- **Left role nav** — **CEO** (the `ceo` persona seat, with its active-goal count from `goals/*.md`), then one row per `functions/*.md` **director** ([[../functions/platform|Platform]] · [[../functions/growth|Growth]] · [[../functions/cmo|CMO]] · [[../functions/cs|CS]] · [[../functions/retention|Retention]]) showing its **persona chip + SVG mascot** + a **live/autonomous badge** (M1: every director is `offline` → "routes to CEO"; the flag itself lands in M2). Under each director, its **Workers** — the **reconciled roster** it owns ([[../archive]]): the `agent-kind` queue lanes (build/plan/fold/spec-chat/spec-test/dev-ask/pr-resolve/repair/regression → Platform; ticket-improve/triage-escalations → CS; migration-fix → Retention; storefront-optimizer → Growth) **plus** the persona-backed platform crons (Tao/monitor, Devi/db_health) **plus** any live unregistered lane (Cole/coverage-register), each with a live `active`/`idle-healthy`/`inactive` status badge.
- **Right pane** — the selected role's header (mascot + name + role + personality + mandates/goals). For a **director**, a gamified **XP card** (M3 Phase 3, shipped) atop its board channel — **specs shipped · bugs fixed · goals escorted · streak** (`<XpCard>` over `GET /api/developer/agents/xp`, [[../libraries/director-xp]]); the same card renders on the director's profile page. Counts are **derived + display-only** (a gamified proxy, never an objective — [[../operational-rules]] § North star). For a **director**, an owner-only **Autonomy toggle** (approval-routing-engine M2 / Phase 1) — two checkboxes, **Live** (the director-agent is running) + **Autonomous** (trusted to auto-decide), backed by [[../tables/function_autonomy]] via `POST /api/developer/agents/autonomy`. Autonomous is disabled until Live is on (an offline director can't auto-approve); the row reads "Approvals route here + log to history" when on, else "Approvals route to the CEO". The director's **live/autonomous badge** is now derived from these flags (seeded all-off ⇒ every director shows "routes to CEO" until the owner flips it). Below the header, its **three-tab inbox shell**:
  - **Messages** — the Slack-style **#directors board** (M3 Phase 1, shipped): a workspace-wide team channel backed by [[../tables/director_messages]] (NOT the notification store), rendered by `<BoardChannel>` ([[../libraries/director-board]]) — persona avatar + name/role per post, conversational body with @-mentions highlighted, threaded replies, kind badges. The live Platform director (M4) is the first real author; a system seed proves the surface. Two-way reply is M3 Phase 2.
  - **Approval Requests** (`agent_approval_request`) — the routed approval queue, and (Phase 4) the **single source** where approvals are surfaced *and* decided. **Live (M2 Phase 2):** every open [[../tables/agent_jobs]] `needs_approval` is surfaced here by the box reconciler ([[../libraries/approval-inbox]]), routed to the resolved approver function ([[../libraries/approval-router]] — first live+autonomous ancestor, else the CEO). Each request carries the agent's **investigation + proposed fix inline** (the `pending_actions[].summary`/`preview`/`cmd`) so the decision is one read. **Phase 4 — multi-action/multi-branch inline:** `<ApprovalRow>` renders **every** still-pending plain action as its own sub-card with **Approve / Decline** (→ `POST /api/roadmap/approve` per action, the unchanged execution path → worker flips `queued_resume` once all are decided) — so a multi-action `build` (migration + prod-script) and a **multi-branch `plan`** (each proposed spec branch, with its owner + parent) are resolved entirely here. The still-pending list is read **live** off the job (`inlineApproveActions`), not the emit-time snapshot, so a half-decided plan shows only the branches left. Only a genuinely **multi-CHOICE** job (coverage register-vs-exempt, hero reject-with-notes) carries no inline actions and **deep-links out** to its canonical surface ([[control-tower]] coverage section / the storefront optimizer). The request is auto-dismissed the moment its job leaves `needs_approval`. **The retired scattered surfaces** (Control Tower repair/db-health feeds, the spec-card [[roadmap|BuildButton/PlanButton]] cards, the box [[roadmap|`approvalHref`]]) no longer raise their own approval card — they `routedInboxHref()` **deep-link here** (one inbox, no orphans).
  - **Daily Summaries** (`agent_daily_summary`) — the EOD recaps (populated by M3/M4 — [[../libraries/director-recap]]). Each row **deep-links to the human-readable day narrative** ([[../specs/director-loop-grading]] Phase 5): `/dashboard/agents/recap/[date]` (`?function={slug}` for one director, omitted for the CEO roll-up) reads that day's [[../tables/director_activity]] rows via `GET /api/developer/agents/recap` and renders *what it fixed + why, which goal it moved, what it escalated* — a query over the log, never hand-maintained.
  - Each tab has **filters** (text + unread-only) + per-tab counts and an **empty/loading state**. The inbox queries the reserved `agent_*` types out of [[../tables/dashboard_notifications]] and buckets them by tab. **Approval routing:** a role sees only the approvals whose `metadata.routed_to_function` matches it (legacy/unrouted ⇒ the CEO); a director that is **not** an auto-approver captures nothing → `routesToCeo:true` (empty + a "routes to the CEO inbox" notice). The CEO additionally carries the (not-yet-routed) Messages & Daily-Summaries.

## Data source

- `GET /api/developer/agents` (`src/app/api/developer/agents/route.ts`) → `getOrgChart()` ([[../libraries/agent-personas|org-chart.ts]]) — the CEO → Directors → Workers tree from [[../libraries/brain-roadmap]] `getFunctions()` + `getGoals()` + the **reconciled worker roster** ([[../archive]]), with each director's `status`/`live`/`autonomous` derived from [[../tables/function_autonomy]] ([[../libraries/approval-router]] `loadAutonomyMap` + `isAutoApprover`). **Brain is the source of truth — never a hand-maintained second copy of the org chart.**
  - **Reconciled roster** (`buildRoster`) surfaces **100% of running agents** by unioning three sources, de-duped by persona key: (1) the `agent-kind` MONITORED_LOOPS (queue lanes), (2) the persona-backed `cron` loops — a [[../libraries/control-tower|registry]] `personaKind` (so `control-tower-monitor`→Tao + the two `db-health-*` crons→one Devi show as workers, **invisible before**), and (3) any live `agent_jobs.kind` with rows in the last 7 days but no registry row (e.g. `coverage-register`→Cole) — `flagged:true`, surfaced under Platform, never silently dropped.
  - **Live status** (`computeWorkerStatus`) — each worker carries `active` (recent job/beat) · `idle-healthy` (fired before / a beating cron, idle now) · `inactive` (zero `agent_jobs` rows **ever** — e.g. Remi/regression). The inactive flag mirrors [[../libraries/control-tower]] `evalCron`'s never-fired grace (a `registeredAt`-within-grace lane and any cron-backed loop are never death-flagged), so an idle-but-healthy / freshly-shipped lane is never falsely called dead. Rendered via `WorkerStatusBadge` (`persona-chip.tsx`) on the org tree + workers roster, with an `unregistered` chip on flagged lanes.
- `POST /api/developer/agents/autonomy` (`src/app/api/developer/agents/autonomy/route.ts`) → owner-gated upsert of a director's `live`/`autonomous` flag into [[../tables/function_autonomy]] (the progressive-offload switch behind [[../libraries/approval-router]] `resolveApprover`). Validates the slug against `functions/*.md`; `autonomous` is forced off when `live` is off.
- `GET /api/developer/agents/inbox?role={ceo|slug}` (`src/app/api/developer/agents/inbox/route.ts`) → the role's Approval Requests + Daily Summaries payload. Reads [[../tables/dashboard_notifications]] (`type IN agent_*`, `dismissed=false`); **routes by the live flags (M2):** `routesToCeo` is true (empty) for a director that isn't an auto-approver, and each `agent_approval_request` is filtered to the role its `metadata.routed_to_function` names. Read-only — the routing is done at emit time by the reconciler, not here.
- The routed Approval Requests are emitted by the **box reconciler** ([[../libraries/approval-inbox]] `reconcileApprovalInbox`, run each poll tick by `scripts/builder-worker.ts`) — the single "one inbox, no orphans" sweep over every `needs_approval` job. The **decision** rides the unchanged `POST /api/roadmap/approve` execution path ([[../libraries/roadmap-actions|roadmap-actions.ts]] `approveRoadmapAction` → `queued_resume`).
- `GET /api/developer/agents/recap?date=&function=` (`src/app/api/developer/agents/recap/route.ts`) → the **human-readable EOD day narrative** ([[../libraries/director-recap]] `buildDirectorDayNarrative`) the Daily Summaries rows deep-link to. Owner-gated, read-only; a pure query over [[../tables/director_activity]], recomputed each view.
- `GET /api/developer/agents/xp` (`src/app/api/developer/agents/xp/route.ts`) → `{ xp: { [functionSlug]: DirectorXp } }` ([[../libraries/director-xp]] `getDirectorXp`) — the four derived XP counts per director (merged builds owned · approved repair/regression fixes · shipped milestones in owned goals · consecutive active days). Owner-gated, read-only, display-only.
- `GET /api/developer/agents/model-tier` (`src/app/api/developer/agents/model-tier/route.ts`) → `{ kind, current: { model_tier, proposed_by, approved_by, updated_at }, history: […] }` ([[../libraries/agent-model-tiers]] `getModelTier`) — the agent's current model tier + the recent governed proposals (P4 of [[../specs/box-agent-model-tiers]]). The **model_tier** is the resolved tier the box passes as `--model` when dispatching jobs for this kind; `null` = the Max-plan default (no flag). The **history** lists the recent `proposed-model-tier` [[../tables/agent_jobs]] rows + their `apply_model_tier` action status, so the profile renders change history + any pending approval awaiting a supervisor's decision. Owner-gated, read-only.
- `GET /api/developer/agents/guide?slug={director}` (`src/app/api/developer/agents/guide/route.ts`) → the **director Guide** payload (director-guide-tab). Owner-gated, read-only; composes `getOrgChart()` (the brain functions + reconciled roster + `function_autonomy` flags) with `getLeashGuide(slug)` ([[../libraries/director-leash-guide]]) into `{ title, summary, status, statusPlain, live, autonomous, leash:{ defined, autonomous[], escalates[] }, team:[{kind,label,status,statusReason,flagged}], goals:[{slug,title,pct,status}], specCount }`. **100% runtime-derived** — a new agent/director appears on the next poll. 404 for a non-director slug.
- `GET /api/developer/agents/board` (`src/app/api/developer/agents/board/route.ts`) → the Messages tab's #directors channel — `getDirectorBoard` + `threadMessages` ([[../libraries/director-board]]) over [[../tables/director_messages]]. Owner-gated, read-only (the two-way reply that writes back is M3 Phase 2). The Messages tab renders this for **every** role (one shared team channel), so the per-role `routesToCeo` notice never covers it.
