# dashboard/roadmap

The project-manager board + build console for the brain. Reads `docs/brain/specs/` to show **planned / in progress / shipped**, and (owner-only) lets you edit status, author/refine specs with Opus, dispatch autonomous builds on the box, answer build questions, approve prod actions, and squash-merge вЂ” all phone-friendly. Full end-to-end: [[../lifecycles/roadmap-build-console]].

**Routes:** `/dashboard/roadmap` (board) + `/dashboard/roadmap/[slug]` (spec detail) + `/dashboard/roadmap/box` (live build-box view) + `/dashboard/roadmap/map` (taxonomy: Function в†’ Mandate/Goal в†’ Spec) + `/dashboard/roadmap/goals` + `/dashboard/roadmap/goals/[slug]` (goal board + detail) + `/dashboard/roadmap/functions/[slug]` (function home). Server components, `dynamic = "force-dynamic"` (the box view is a client poller).
**Sidebar:** **Developer** section (owner-only) в†’ **Roadmap** + **Build box** + **Goals** + **Taxonomy map** + **Spec Tests** + **Human-test queue** (amber count badge) + **Brain** + [[branches]].

## Surfaces

- **Board** вЂ” `src/lib/brain-roadmap.ts` `getRoadmap()` parses specs (+ `README.md` track chips). **Four columns** from each spec's `SpecStatus` (`вЏі planned В· рџљ§ in progress В· вњ… shipped В· вЏёпёЏ deferred`; phases can also be `вќЊ cut`). Cards show summary, phase count pills, and live build status. **Deferred** ([[../specs/director-drives-all-specs-and-deferred-status]] Phase 1) is its own column for parked work вЂ” a spec carrying a leading `**Deferred:**` marker (authored by [[../specs/board-grooming]] split cards) or `**Status:** deferred`; `deriveStatus` returns `deferred` over its вЏі phases, every auto-build lane skips it ([[../libraries/platform-director]]), and only the CEO un-defers it (remove the marker в†’ Planned). `resolveBoardStatus` keeps a deferral markdown-owned (the phase-progress DB mirror never overrides it).
- **Board filters** ([[../specs/roadmap-goal-and-source-filters]]) вЂ” `RoadmapFilters.tsx` (a client component, replaced `SpecSearch.tsx`) renders three filters that AND together as one pure-DOM visibility pass (no server round-trip): the **search box** (substring over `data-spec-search`), a **goal dropdown** (every `docs/brain/goals/*.md`; pick one в†’ only that goal's specs + a **progress header** `Title вЂ” X/Y shipped В· N building В· M planned` counting the visible cards; sticky in the `?goal=` URL param so a goal view is shareable), and **source chips** (All В· рџЋЇ Goal В· рџ”§ Repair В· вњ‹ Manual). The loader (`getRoadmapFilters()`) resolves goalв†’spec membership (goal-doc `[[spec]]` wikilinks в€Є a `**Parent:**` that names the goal) + per-spec source once; `page.tsx` stamps each card with `data-goal` / `data-source` / `data-status`. **Status accuracy** ([[../specs/roadmap-status-accuracy]]): (1) **phase consensus beats a stale title** вЂ” `deriveStatus` treats an all-`вњ…` phase set as **shipped** regardless of the H1 title emoji (a forgotten `рџљ§`/`вЏі` title no longer reads as "Doing"); the title still wins for an explicit `вќЊ` cut or when there are no phases. (2) **live-job overlay** (`page.tsx` `effectiveStatus`) вЂ” a **Planned** spec with an **active `agent_jobs`** row renders in **In progress**, so tapping Build moves the card immediately and it stays there while the build runs, reverting to its markdown status once the job is terminal. The overlay only *promotes* plannedв†’in_progress; it never demotes a Shipped spec. (3) **DB-first instant status + deploy flag** ([[../specs/spec-card-db-companion]]) вЂ” the board reads the [[../tables/spec_card_state]] live mirror ([[../libraries/spec-card-state]] `getSpecCardStates`) and overlays it via `resolveBoardStatus` (DB-first for the deploy-lag; a markdown that's ahead wins, so markdown stays canonical), so a merge / drift flip / owner mark flips the card **before** the markdown redeploys вЂ” **zero GitHub API calls for status** (this retired the quota-burning `roadmap-reads-specs-from-git` git read). A merged card shows a **`shipped В· deploying`** chip (its `last_merge_sha` в‰  the live `VERCEL_GIT_COMMIT_SHA`, `deploymentState`) until a deploy carrying that SHA is up, then **`shipped В· live`** вЂ” clean signal, no webhook.
- **Detail page** вЂ” `marked` в†’ `prose`; `[[wikilinks]]` to other specs become links. **Refine with Opus** button.
- **Editable status** (owner) вЂ” `StatusControl.tsx` (card: Planned/Doing/Shipped) and `PhaseList.tsx` (per-phase dots incl. **Cut**, + a per-phase **build**). Each click commits the emoji to the brain markdown on `main` via `POST /api/roadmap/status` (`phaseIndex` targets the Nth `## Phase`). The markdown stays the source of truth вЂ” no DB overrides.
- **Critical priority** (owner) вЂ” `PriorityControl.tsx` toggles a spec's `**Priority:** critical` marker via `POST /api/roadmap/priority` ([[../specs/director-executable-plans-and-priority]] Phase 1), committed to the brain markdown on `main` (`setCriticalMarker` is the shared writer; a directive's `criticalSpecs[]` is the chat lever). A critical spec shows a рџ”ґ pip on its card and is queued **ahead of normal Planned specs** by the director's build picker. Orthogonal to status вЂ” a critical spec stays in its phase column.
- **Authoring chat** вЂ” `AuthoringChat.tsx` + `POST /api/roadmap/chat` (Opus `claude-opus-4-8`, Anthropic API). **вњЁ New feature** (board header) writes a new `specs/{slug}.md`; **Refine with Opus** (detail page) edits an existing one. Finalize commits the spec to `main` (+ optional **Save & build** queues a job).
- **Build dispatch** вЂ” `BuildButton.tsx`: **Build/Rebuild** (hidden on shipped specs), **Build all** (chains every вЏі phase вЂ” see below), per-phase **build**, and **Report issue** (queues a scoped *fix-build* via `instructions` вЂ” works on shipped specs, no spec edit). All hit `POST /api/roadmap/build` в†’ inserts an [[../tables/agent_jobs]] row (one active per spec).
- **Build all вЂ” phase chain** ([[../specs/build-all-phases-chain]]) вЂ” **Build all** (`POST /api/roadmap/build { chainPhases:true }` в†’ `queueRoadmapBuild(..., { chainPhases:true })`, [[../libraries/roadmap-actions]]) queues only the **first вЏі phase** scoped, tagged `chain_phases:true` on the [[../tables/agent_jobs]] row. When that phase's PR merges (auto, via the auto-ship pipeline), `reconcileMergedJobs` в†’ `queueNextChainedPhase` ([[../libraries/agent-jobs]]) reads `main` and queues the **next вЏі phase** (also `chain_phases`, on fresh main atop the prior phase's code), repeating until all phases вњ… вЂ” so a multi-phase spec ships with **no owner clicks** between phases. A **failed / needs_approval** phase never reaches `merged`, so the chain stops/pauses there (resuming + merging it resumes the chain); single-phase / per-phase **Build N** builds stay non-chained (no `chain_phases` в†’ no auto-queue). The same **рџ› пёЏ Build all** action exists on the Slack App Home ([[../libraries/slack-home]] `HOME.build`). The chip polls `GET /api/roadmap/build?slug=` until terminal. After a queuing POST (build / report-issue / verify) the button calls `router.refresh()` so the server board re-runs and the live overlay re-buckets the card right away (instant, real-DB вЂ” not a client-only guess).
- **Create PR recovery** ([[../specs/build-recover-pr-create]]) вЂ” a build that **finished + pushed its `claude/*` branch** but failed the final `gh pr create` lands `needs_attention` with `error="branch pushed but PR creation failed"`. The card detects this recoverable sub-case (status + that exact error + a `spec_branch`) and shows **Create PR** as the **primary** action вЂ” `POST /api/roadmap/build { jobId, recoverPr:true }` в†’ `createPrForJob` ([[../libraries/roadmap-actions]]) opens the PR for the pushed branch (idempotently adopting one that already exists) and flips the job в†’ `completed` with the PR attached вЂ” instead of discarding the completed build. **Rebuild** stays as a demoted, clearly-labeled *discard-and-redo* fallback. A genuinely-stuck `needs_attention` (no pushed branch) keeps its plain treatment.
- **Build prerequisites вЂ” spec blockers** ([[../specs/spec-blockers]]) вЂ” a spec can declare `**Blocked-by:** [[a]], [[b]]` under its H1 ([[../libraries/brain-roadmap]] `parseSpec`); the board resolves each to `blockedBy: { slug, title, status, cleared }` (cleared when the blocking spec is shipped, archived/folded, or dangling). A spec with any **uncleared** blocker shows a **"рџ”’ Blocked by вЂ¦"** chip (each prerequisite + its вњ…/вЏі/рџљ§) and a **disabled Build** button ("рџ”’ Blocked", tooltip names what must ship first); cleared blockers render вњ…. The real enforcement is server-side: `queueRoadmapBuild` ([[../libraries/roadmap-actions]]) checks `getSpecBlockers` **before inserting** the build row and refuses an uncleared spec with `409 { error:"Blocked by: <slug> (<emoji>)" }` вЂ” so the dashboard **and** the Slack `/build` (the shared chokepoint) both refuse a blocked spec. A spec with no Blocked-by (or all cleared) builds normally вЂ” no regression. *(P2 вЂ” auto-queue the dependent build when the last blocker ships, + routing the box worker's planner enqueue through the same gate вЂ” is not yet shipped.)*
- **Live build-box view** (`/box`, [[../specs/build-box-status-view]]) вЂ” `BoxChip.tsx` (board header) + `box/page.tsx` poll `GET /api/roadmap/box`, which returns the singleton [[../tables/worker_heartbeats]] row (enriched with `build_lanes`/`fold_lanes` totals + a `lanes` array of what each in-flight lane is building) plus open [[../tables/agent_jobs]] split into `queue` (waiting) / `paused` (needs_input|needs_approval), and a `failed` list (build/plan jobs whose **latest** attempt is `status='failed'` вЂ” a since-rebuilt spec is excluded, so it never shows a stale failure). The page shows a **health header** (healthy/stale by `last_poll_at` age, `running_sha` в†’ commit, uptime since `started_at`), a **lane grid** (build/plan + fold cells; in-use cells show spec_slug В· kind chip В· elapsed вЂ” a chained/per-phase build also appends `В· Phase N` after the slug, derived by the worker from the job's instructions [[../specs/box-lane-show-phase]], slug-only for a whole-spec build; free cells "open"), **queue depth**, a **failed-builds callout**, and a **paused callout**. The **failed-builds callout** (red) is the "I don't have to dig into a spec card to know a build died" surface: each failed build shows its slug (в†’ detail page), how long ago, the real **failure reason parsed from `log_tail`** (the `claude -p` result JSON's `api_error_status` + `result` вЂ” so a transient **529 Overloaded** / a Max-limit error reads as such vs a real bug; a `tsc` failure shows its output tail), and a **в†» Retry build** button (`POST /api/roadmap/build {slug}` вЂ” re-queues; optimistically clears the row). The board-header **chip** ("Build box В· N/8 lanes В· healthy") goes stale/red when the heartbeat is old. Owner-only, read-only вЂ” pausing/killing lanes stays a CLI job ([[../recipes/manage-the-build-queue]]). **Dead-link defense ([[../specs/auto-ship-pipeline]]/fold-guard-live-build):** `GET /api/roadmap/box` stamps each paused/failed build with `spec_missing` (computed by `specMissing(kind, spec_slug)` against the live, non-archived spec slugs); a job whose spec was folded/archived/deleted carries `spec_missing:true`, and `box/page.tsx` routes its callout link to **`/dashboard/roadmap`** (the board, always 200) with a `(spec archived)` tag instead of the 404 `/dashboard/roadmap/{slug}` вЂ” so a fold that races a build never leaves a dead link. A job whose spec still exists links to the spec detail page as before. The preventive + cleanup guards live in [[../libraries/agent-jobs]] (`getLiveJobForSlug` / `cancelJobsForArchivedSpecs`).
- **Build feedback** вЂ” when a build pauses: `needs_input` shows an **answer form** (`POST /api/roadmap/answer`); `needs_approval` shows **Approve & apply** cards with the command preview (`POST /api/roadmap/approve`). Both flip the job to `queued_resume` so the box worker resumes it. Completed builds show **Squash & merge** (reuses `POST /api/branches/[number]/merge`).
- **Verify + archive** (`BuildButton.tsx`, owner) вЂ” shipped cards show **Mark verified & archive**: the owner-only, human "I tested it in prod" gate distinct from Shipped (built + deployed, automated). It marks the spec [[../tables/pending_folds|pending-fold]] (`POST /api/roadmap/build` `{ verify: true }` в†’ `enqueue_fold`) and **coalesces** into ONE batch fold-build ([[../specs/fold-build-batching]]): a single `kind='fold'` job folds **all** pending-fold specs into their brain homes, writes a per-spec `archive.d/{slug}.md`, regenerates [[../archive]] + README counts, `git rm`s the specs, and opens **one** PR. The card flips to **"FoldingвЂ¦"** (`getPendingFolds`) until merge в†’ the spec leaves the board into the **Archived** section. The Shipped column is relabeled **"Shipped вЂ” awaiting verification"** so it stays a short, real to-do list.
- **Verification card** (`VerificationCard.tsx`, spec detail) вЂ” the "how to test this" checklist ([[../specs/verification-guides]]), rendered as a teal **"вњ… How to verify in prod"** card in the detail sidebar **right under the build/verify buttons**, so the test steps sit where **Mark verified & archive** is. `extractSpecSection`/`stripSpecSection` (`src/lib/brain-roadmap.ts`) lift the spec's `## Verification` block out of the markdown (and out of the article body so it's not shown twice). A spec with **no** section shows a "No test plan yet" card with an owner-only **Generate test plan** button в†’ `POST /api/roadmap/chat` `{ action:"generate_verification" }` в†’ Opus drafts a concrete section, brain-grounded, and commits it to `specs/{slug}.md` (appears after the next deploy). Builds + the authoring chat write the section at ship time, so new specs arrive test-ready. **Box QA results** ([[../specs/spec-test-agent]]): when the spec has a latest [[../tables/spec_test_runs]] run, the card also renders the **Agent-tested** stamp + chip, the **per-bullet verdicts** (вњ…/вњ—/рџ‘¤/? with expandable evidence), and a distinct **рџ‘¤ Needs human testing** list вЂ” so the owner does only those, then verifies. An owner-only **Test now** queues a fresh box run.
- **Spec Tests page** (`/dashboard/developer/spec-tests`, [[../specs/spec-test-agent]]) вЂ” the box spec-test QA agent's report home: every shipped-but-unverified spec with its latest run (the **Agent-tested вњ… / вљ пёЏ issues** stamp, the `вњ…В·вњ—В·рџ‘¤В·?` chip, each check's verdict + expandable evidence) plus an aggregated **рџ‘¤ Needs human testing** list across all specs and a per-spec **Test now** button. Reads via `src/lib/spec-test-runs.ts` (`getLatestSpecTestRuns`). Owner-only. **Browser-check screenshot evidence** ([[../specs/spec-test-deep-verification]] Phase 1): a check carrying a `screenshot` (the headless-browser tool's full-page capture, stored in the **private** `spec-test-evidence` bucket) renders the image inline under its expandable evidence вЂ” the page signs the path per-render server-side (`signSpecTestScreenshot`, short-TTL signed URL; never a public URL, since a dashboard screenshot can contain real customer data). The roadmap board's shipped cards also carry the compact chip + stamp (linking here). The agent **stamps**, it never marks a spec verified вЂ” that owner-only gate stays untouched. A regressed spec (latest run `agent_verdict='issues'`) also gets a **Propose fix spec** button here + on the VerificationCard (see Human-test queue below).
- **Human-test queue** (`/dashboard/developer/spec-tests/human-queue`, [[../specs/spec-test-agent]] Phase 2) вЂ” a focused queue (modeled on [[../specs/improve-queue|improve-queue]]) aggregating every `needs_human` check across the latest run of every shipped-unverified spec вЂ” the parts only the owner can run (visual/UX, prod-mutating, or **fault-injection / forced-failure** checks the read-only agent can't exercise). Each waiting check has **вњ“ Tested** / **Dismiss**; a collapsed **Done** group lets the owner **Re-open** one. Owner resolutions persist in [[../tables/spec_test_human_checks]] (keyed by a stable `check_key` hash of the bullet, so a resolution survives re-runs). **Regressions** (a shipped spec with an **evidence-backed `fail`** in its latest run вЂ” the agent observed breakage; a `needs_human`/`inconclusive` check never counts as a regression, see [[../specs/spec-test-classification]]) surface loudly at the top with a one-click **Propose fix spec** в†’ `POST /api/roadmap/chat` `{ action:"propose_fix" }`, which seeds a box authoring chat (`Fix: {slug}`) with the failing checks as the brief + kicks off the first box turn ([[../specs/box-spec-chat]]); the owner reviews + finalizes it into a fix spec (the agent proposes, the owner finalizes). The brief now tags each failing check with its `[check {key}]` hash and instructs the box to stamp a machine-readable `**Fixes:** {origin} (check {key}вЂ¦)` line in the fix spec вЂ” so when the fix build **merges**, [[../libraries/agent-jobs]] `retestOriginIfFixMerged` auto-re-tests the **origin** and its stale "issues" badge clears with no manual re-queue ([[../specs/fix-ship-retests-origin]]). Reads/writes via `GET`/`POST /api/developer/spec-test/human-queue` (`getHumanTestQueue`, owner-gated); the sidebar shows an amber count badge (waiting + regressions). Polls every ~8s.
- **Archived section** вЂ” a collapsed `<details>` below the columns lists verified features (link в†’ their lifecycle/brain page). `getArchive()` reads the per-spec `docs/brain/archive.d/*.md` entries (falling back to the generated [[../archive]]). Each row also offers **New spec from brain** (re-hydration) seeded with that page.
- **New spec from brain** (re-hydration) вЂ” `AuthoringChat seed` (board header **рџ§  New spec from brain**, or per archived entry with a fixed `seedSlug`). Pick a brain page (lifecycle/dashboard/table) or archived entry в†’ `POST /api/roadmap/chat` seeds Opus with the **current** brain page content в†’ drafts a *fresh* spec to extend/fix it (never reactivates a stale snapshot) в†’ normal Save / Save & build.
- **Taxonomy map** (`/map`, `getFunctionMap()`) вЂ” every spec grouped Function в†’ Mandate/Goal в†’ Spec, built from each spec's owner + parent (so it never drifts). Function headers link to the function home; an **Orphan specs** panel flags any spec with no owner (the no-orphan lint).
- **Goals board + detail** (`/goals`, `/goals/[slug]`, `getGoals()`/`getGoal()`) вЂ” the [[../specs/goal-decomposition-engine|goal-decomposition engine]] surface. Cards show success metric, a **rollup % bar** (mean of milestone completion вЂ” `parseGoal` averages each milestone's linked-spec phase completion; advances automatically as leaf specs ship), and milestone/spec counts. Detail renders the goal doc + a **milestone tree** with each leaf spec's live status, and an owner-only **Plan / Re-plan** control (`PlanButton.tsx`).
- **Function home** (`/functions/[slug]`, `getFunction()`/`parseFunction()`) вЂ” the director's charter rendered + a sidebar of mandates (each perpetual: metric + owned specs, **no %**) and owned/contributed goals. A spec card's owner chip links here.
- **Plan a goal** (`PlanButton.tsx`, owner) вЂ” **Plan goal** / **Re-plan** в†’ `POST /api/roadmap/plan` inserts a `kind='plan'` [[../tables/agent_jobs]] row (one active plan per goal). The box worker runs the `plan-goal` skill в†’ proposes a milestoneв†’spec tree as `pending_actions` (`type:'spec'`) в†’ `needs_approval`. The button shows each proposed branch (owner В· parent В· intent) with **Approve / Decline** (reuses `POST /api/roadmap/approve`); once every branch is decided the job в†’ `queued_resume` and the worker auto-authors the approved specs (committed to main), wikilinks them into the goal doc, records declines (вќЊ), and queues their builds. Polls `GET /api/roadmap/plan?goalSlug=`.

## Data sources

- **Brain markdown** (board + detail) вЂ” `docs/brain/specs/*.md` + `docs/brain/goals/*.md` + `docs/brain/functions/*.md`, read at request time via `src/lib/brain-roadmap.ts` (`getRoadmap`/`getSpec`/`getGoals`/`getGoal`/`getFunctions`/`getFunction`/`getFunctionMap`; `parseGoal`/`parseFunction` extract milestones/mandates + linked specs + rollup). `[[wikilinks]]` resolve to dashboard routes via `src/lib/brain-links.ts` (`preprocessBrainWikilinks` вЂ” specsв†’`/roadmap/{slug}`, goalsв†’`/roadmap/goals/{slug}`, functionsв†’`/roadmap/functions/{slug}`). The static, canonical layer.
- **`agent_jobs`** (live build + plan state) вЂ” read via `getLatestJobsBySlug(workspaceId)` (build cards) + `getLatestPlanJob(workspaceId, goalSlug)` (goal Plan control); `BuildButton`/`PlanButton` poll the API for updates. The live, actionable layer.

## Vercel gotcha

The board/detail read files under `docs/brain/`, which Vercel's tracer would prune. `next.config.ts` в†’ `outputFileTracingIncludes` ships the spec/lifecycle/**goals/functions** markdown into the `/dashboard/roadmap`, `/[slug]`, `/map`, `/goals`, `/goals/[slug]`, and `/functions/[slug]` function bundles. Without it those routes render empty in prod. (The chat/status/approve/**plan** API routes read the brain from **GitHub** at request time, so they don't need tracing.) **`/api/roadmap/build` is the exception:** its POST runs the spec-blockers gate (`queueRoadmapBuild в†’ getSpecBlockers в†’ getRoadmap`) which reads `docs/brain/specs` from the **local bundle** at request time, so it's traced too вЂ” otherwise the gate would see zero blockers in prod and silently pass.
