# dashboard/roadmap

The project-manager board + build console for the brain. Reads `docs/brain/specs/` to show **planned / in progress / shipped**, and (owner-only) lets you edit status, author/refine specs with Opus, dispatch autonomous builds on the box, answer build questions, approve prod actions, and squash-merge вҖ” all phone-friendly. Full end-to-end: [[../lifecycles/roadmap-build-console]].

**Routes:** `/dashboard/roadmap` (board) + `/dashboard/roadmap/[slug]` (spec detail) + `/dashboard/roadmap/box` (live build-box view) + `/dashboard/roadmap/map` (taxonomy: Function вҶ’ Mandate/Goal вҶ’ Spec) + `/dashboard/roadmap/goals` + `/dashboard/roadmap/goals/[slug]` (goal board + detail) + `/dashboard/roadmap/functions/[slug]` (function home). Server components, `dynamic = "force-dynamic"` (the box view is a client poller).
**Sidebar:** **Developer** section (owner-only) вҶ’ **Roadmap** + **Build box** + **Goals** + **Taxonomy map** + **Spec Tests** + **Human-test queue** (amber count badge) + **Brain** + [[branches]].

## Surfaces

- **Board** вҖ” `src/lib/brain-roadmap.ts` `getRoadmap()` parses specs (+ `README.md` track chips). Three columns from each spec's status (`вҸі planned В· рҹҡ§ in progress В· вң… shipped`; phases can also be `вқҢ cut`). Cards show summary, phase count pills, and live build status. **Status accuracy** ([[../specs/roadmap-status-accuracy]]): (1) **phase consensus beats a stale title** вҖ” `deriveStatus` treats an all-`вң…` phase set as **shipped** regardless of the H1 title emoji (a forgotten `рҹҡ§`/`вҸі` title no longer reads as "Doing"); the title still wins for an explicit `вқҢ` cut or when there are no phases. (2) **live-job overlay** (`page.tsx` `effectiveStatus`) вҖ” a **Planned** spec with an **active `agent_jobs`** row renders in **In progress**, so tapping Build moves the card immediately and it stays there while the build runs, reverting to its markdown status once the job is terminal. The overlay only *promotes* plannedвҶ’in_progress; it never demotes a Shipped spec.
- **Detail page** вҖ” `marked` вҶ’ `prose`; `[[wikilinks]]` to other specs become links. **Refine with Opus** button.
- **Editable status** (owner) вҖ” `StatusControl.tsx` (card: Planned/Doing/Shipped) and `PhaseList.tsx` (per-phase dots incl. **Cut**, + a per-phase **build**). Each click commits the emoji to the brain markdown on `main` via `POST /api/roadmap/status` (`phaseIndex` targets the Nth `## Phase`). The markdown stays the source of truth вҖ” no DB overrides.
- **Authoring chat** вҖ” `AuthoringChat.tsx` + `POST /api/roadmap/chat` (Opus `claude-opus-4-8`, Anthropic API). **вңЁ New feature** (board header) writes a new `specs/{slug}.md`; **Refine with Opus** (detail page) edits an existing one. Finalize commits the spec to `main` (+ optional **Save & build** queues a job).
- **Build dispatch** вҖ” `BuildButton.tsx`: **Build/Rebuild** (hidden on shipped specs), per-phase **build**, and **Report issue** (queues a scoped *fix-build* via `instructions` вҖ” works on shipped specs, no spec edit). All hit `POST /api/roadmap/build` вҶ’ inserts an [[../tables/agent_jobs]] row (one active per spec). The chip polls `GET /api/roadmap/build?slug=` until terminal. After a queuing POST (build / report-issue / verify) the button calls `router.refresh()` so the server board re-runs and the live overlay re-buckets the card right away (instant, real-DB вҖ” not a client-only guess).
- **Build prerequisites вҖ” spec blockers** ([[../specs/spec-blockers]]) вҖ” a spec can declare `**Blocked-by:** [[a]], [[b]]` under its H1 ([[../libraries/brain-roadmap]] `parseSpec`); the board resolves each to `blockedBy: { slug, title, status, cleared }` (cleared when the blocking spec is shipped, archived/folded, or dangling). A spec with any **uncleared** blocker shows a **"рҹ”’ Blocked by вҖҰ"** chip (each prerequisite + its вң…/вҸі/рҹҡ§) and a **disabled Build** button ("рҹ”’ Blocked", tooltip names what must ship first); cleared blockers render вң…. The real enforcement is server-side: `queueRoadmapBuild` ([[../libraries/roadmap-actions]]) checks `getSpecBlockers` **before inserting** the build row and refuses an uncleared spec with `409 { error:"Blocked by: <slug> (<emoji>)" }` вҖ” so the dashboard **and** the Slack `/build` (the shared chokepoint) both refuse a blocked spec. A spec with no Blocked-by (or all cleared) builds normally вҖ” no regression. *(P2 вҖ” auto-queue the dependent build when the last blocker ships, + routing the box worker's planner enqueue through the same gate вҖ” is not yet shipped.)*
- **Live build-box view** (`/box`, [[../specs/build-box-status-view]]) вҖ” `BoxChip.tsx` (board header) + `box/page.tsx` poll `GET /api/roadmap/box`, which returns the singleton [[../tables/worker_heartbeats]] row (enriched with `build_lanes`/`fold_lanes` totals + a `lanes` array of what each in-flight lane is building) plus open [[../tables/agent_jobs]] split into `queue` (waiting) / `paused` (needs_input|needs_approval), and a `failed` list (build/plan jobs whose **latest** attempt is `status='failed'` вҖ” a since-rebuilt spec is excluded, so it never shows a stale failure). The page shows a **health header** (healthy/stale by `last_poll_at` age, `running_sha` вҶ’ commit, uptime since `started_at`), a **lane grid** (build/plan + fold cells; in-use cells show spec_slug В· kind chip В· elapsed, free cells "open"), **queue depth**, a **failed-builds callout**, and a **paused callout**. The **failed-builds callout** (red) is the "I don't have to dig into a spec card to know a build died" surface: each failed build shows its slug (вҶ’ detail page), how long ago, the real **failure reason parsed from `log_tail`** (the `claude -p` result JSON's `api_error_status` + `result` вҖ” so a transient **529 Overloaded** / a Max-limit error reads as such vs a real bug; a `tsc` failure shows its output tail), and a **вҶ» Retry build** button (`POST /api/roadmap/build {slug}` вҖ” re-queues; optimistically clears the row). The board-header **chip** ("Build box В· N/8 lanes В· healthy") goes stale/red when the heartbeat is old. Owner-only, read-only вҖ” pausing/killing lanes stays a CLI job ([[../recipes/manage-the-build-queue]]).
- **Build feedback** вҖ” when a build pauses: `needs_input` shows an **answer form** (`POST /api/roadmap/answer`); `needs_approval` shows **Approve & apply** cards with the command preview (`POST /api/roadmap/approve`). Both flip the job to `queued_resume` so the box worker resumes it. Completed builds show **Squash & merge** (reuses `POST /api/branches/[number]/merge`).
- **Verify + archive** (`BuildButton.tsx`, owner) вҖ” shipped cards show **Mark verified & archive**: the owner-only, human "I tested it in prod" gate distinct from Shipped (built + deployed, automated). It marks the spec [[../tables/pending_folds|pending-fold]] (`POST /api/roadmap/build` `{ verify: true }` вҶ’ `enqueue_fold`) and **coalesces** into ONE batch fold-build ([[../specs/fold-build-batching]]): a single `kind='fold'` job folds **all** pending-fold specs into their brain homes, writes a per-spec `archive.d/{slug}.md`, regenerates [[../archive]] + README counts, `git rm`s the specs, and opens **one** PR. The card flips to **"FoldingвҖҰ"** (`getPendingFolds`) until merge вҶ’ the spec leaves the board into the **Archived** section. The Shipped column is relabeled **"Shipped вҖ” awaiting verification"** so it stays a short, real to-do list.
- **Verification card** (`VerificationCard.tsx`, spec detail) вҖ” the "how to test this" checklist ([[../specs/verification-guides]]), rendered as a teal **"вң… How to verify in prod"** card in the detail sidebar **right under the build/verify buttons**, so the test steps sit where **Mark verified & archive** is. `extractSpecSection`/`stripSpecSection` (`src/lib/brain-roadmap.ts`) lift the spec's `## Verification` block out of the markdown (and out of the article body so it's not shown twice). A spec with **no** section shows a "No test plan yet" card with an owner-only **Generate test plan** button вҶ’ `POST /api/roadmap/chat` `{ action:"generate_verification" }` вҶ’ Opus drafts a concrete section, brain-grounded, and commits it to `specs/{slug}.md` (appears after the next deploy). Builds + the authoring chat write the section at ship time, so new specs arrive test-ready. **Box QA results** ([[../specs/spec-test-agent]]): when the spec has a latest [[../tables/spec_test_runs]] run, the card also renders the **Agent-tested** stamp + chip, the **per-bullet verdicts** (вң…/вң—/рҹ‘Ө/? with expandable evidence), and a distinct **рҹ‘Ө Needs human testing** list вҖ” so the owner does only those, then verifies. An owner-only **Test now** queues a fresh box run.
- **Spec Tests page** (`/dashboard/developer/spec-tests`, [[../specs/spec-test-agent]]) вҖ” the box spec-test QA agent's report home: every shipped-but-unverified spec with its latest run (the **Agent-tested вң… / вҡ пёҸ issues** stamp, the `вң…В·вң—В·рҹ‘ӨВ·?` chip, each check's verdict + expandable evidence) plus an aggregated **рҹ‘Ө Needs human testing** list across all specs and a per-spec **Test now** button. Reads via `src/lib/spec-test-runs.ts` (`getLatestSpecTestRuns`). Owner-only. **Browser-check screenshot evidence** ([[../specs/spec-test-deep-verification]] Phase 1): a check carrying a `screenshot` (the headless-browser tool's full-page capture, stored in the **private** `spec-test-evidence` bucket) renders the image inline under its expandable evidence вҖ” the page signs the path per-render server-side (`signSpecTestScreenshot`, short-TTL signed URL; never a public URL, since a dashboard screenshot can contain real customer data). The roadmap board's shipped cards also carry the compact chip + stamp (linking here). The agent **stamps**, it never marks a spec verified вҖ” that owner-only gate stays untouched. A regressed spec (latest run `agent_verdict='issues'`) also gets a **Propose fix spec** button here + on the VerificationCard (see Human-test queue below).
- **Human-test queue** (`/dashboard/developer/spec-tests/human-queue`, [[../specs/spec-test-agent]] Phase 2) вҖ” a focused queue (modeled on [[../specs/improve-queue|improve-queue]]) aggregating every `needs_human` check across the latest run of every shipped-unverified spec вҖ” the parts only the owner can run (visual/UX, prod-mutating, or **fault-injection / forced-failure** checks the read-only agent can't exercise). Each waiting check has **вң“ Tested** / **Dismiss**; a collapsed **Done** group lets the owner **Re-open** one. Owner resolutions persist in [[../tables/spec_test_human_checks]] (keyed by a stable `check_key` hash of the bullet, so a resolution survives re-runs). **Regressions** (a shipped spec with an **evidence-backed `fail`** in its latest run вҖ” the agent observed breakage; a `needs_human`/`inconclusive` check never counts as a regression, see [[../specs/spec-test-classification]]) surface loudly at the top with a one-click **Propose fix spec** вҶ’ `POST /api/roadmap/chat` `{ action:"propose_fix" }`, which seeds a box authoring chat (`Fix: {slug}`) with the failing checks as the brief + kicks off the first box turn ([[../specs/box-spec-chat]]); the owner reviews + finalizes it into a fix spec (the agent proposes, the owner finalizes). The brief now tags each failing check with its `[check {key}]` hash and instructs the box to stamp a machine-readable `**Fixes:** {origin} (check {key}вҖҰ)` line in the fix spec вҖ” so when the fix build **merges**, [[../libraries/agent-jobs]] `retestOriginIfFixMerged` auto-re-tests the **origin** and its stale "issues" badge clears with no manual re-queue ([[../specs/fix-ship-retests-origin]]). Reads/writes via `GET`/`POST /api/developer/spec-test/human-queue` (`getHumanTestQueue`, owner-gated); the sidebar shows an amber count badge (waiting + regressions). Polls every ~8s.
- **Archived section** вҖ” a collapsed `<details>` below the columns lists verified features (link вҶ’ their lifecycle/brain page). `getArchive()` reads the per-spec `docs/brain/archive.d/*.md` entries (falling back to the generated [[../archive]]). Each row also offers **New spec from brain** (re-hydration) seeded with that page.
- **New spec from brain** (re-hydration) вҖ” `AuthoringChat seed` (board header **рҹ§  New spec from brain**, or per archived entry with a fixed `seedSlug`). Pick a brain page (lifecycle/dashboard/table) or archived entry вҶ’ `POST /api/roadmap/chat` seeds Opus with the **current** brain page content вҶ’ drafts a *fresh* spec to extend/fix it (never reactivates a stale snapshot) вҶ’ normal Save / Save & build.
- **Taxonomy map** (`/map`, `getFunctionMap()`) вҖ” every spec grouped Function вҶ’ Mandate/Goal вҶ’ Spec, built from each spec's owner + parent (so it never drifts). Function headers link to the function home; an **Orphan specs** panel flags any spec with no owner (the no-orphan lint).
- **Goals board + detail** (`/goals`, `/goals/[slug]`, `getGoals()`/`getGoal()`) вҖ” the [[../specs/goal-decomposition-engine|goal-decomposition engine]] surface. Cards show success metric, a **rollup % bar** (mean of milestone completion вҖ” `parseGoal` averages each milestone's linked-spec phase completion; advances automatically as leaf specs ship), and milestone/spec counts. Detail renders the goal doc + a **milestone tree** with each leaf spec's live status, and an owner-only **Plan / Re-plan** control (`PlanButton.tsx`).
- **Function home** (`/functions/[slug]`, `getFunction()`/`parseFunction()`) вҖ” the director's charter rendered + a sidebar of mandates (each perpetual: metric + owned specs, **no %**) and owned/contributed goals. A spec card's owner chip links here.
- **Plan a goal** (`PlanButton.tsx`, owner) вҖ” **Plan goal** / **Re-plan** вҶ’ `POST /api/roadmap/plan` inserts a `kind='plan'` [[../tables/agent_jobs]] row (one active plan per goal). The box worker runs the `plan-goal` skill вҶ’ proposes a milestoneвҶ’spec tree as `pending_actions` (`type:'spec'`) вҶ’ `needs_approval`. The button shows each proposed branch (owner В· parent В· intent) with **Approve / Decline** (reuses `POST /api/roadmap/approve`); once every branch is decided the job вҶ’ `queued_resume` and the worker auto-authors the approved specs (committed to main), wikilinks them into the goal doc, records declines (вқҢ), and queues their builds. Polls `GET /api/roadmap/plan?goalSlug=`.

## Data sources

- **Brain markdown** (board + detail) вҖ” `docs/brain/specs/*.md` + `docs/brain/goals/*.md` + `docs/brain/functions/*.md`, read at request time via `src/lib/brain-roadmap.ts` (`getRoadmap`/`getSpec`/`getGoals`/`getGoal`/`getFunctions`/`getFunction`/`getFunctionMap`; `parseGoal`/`parseFunction` extract milestones/mandates + linked specs + rollup). `[[wikilinks]]` resolve to dashboard routes via `src/lib/brain-links.ts` (`preprocessBrainWikilinks` вҖ” specsвҶ’`/roadmap/{slug}`, goalsвҶ’`/roadmap/goals/{slug}`, functionsвҶ’`/roadmap/functions/{slug}`). The static, canonical layer.
- **`agent_jobs`** (live build + plan state) вҖ” read via `getLatestJobsBySlug(workspaceId)` (build cards) + `getLatestPlanJob(workspaceId, goalSlug)` (goal Plan control); `BuildButton`/`PlanButton` poll the API for updates. The live, actionable layer.

## Billing

Authoring chat вҶ’ Anthropic API (cheap). Builds вҶ’ **Max** (box `claude -p`, no API key). See [[../lifecycles/roadmap-build-console]] В§ Billing.

## Vercel gotcha

The board/detail read files under `docs/brain/`, which Vercel's tracer would prune. `next.config.ts` вҶ’ `outputFileTracingIncludes` ships the spec/lifecycle/**goals/functions** markdown into the `/dashboard/roadmap`, `/[slug]`, `/map`, `/goals`, `/goals/[slug]`, and `/functions/[slug]` function bundles. Without it those routes render empty in prod. (The chat/status/approve/**plan** API routes read the brain from **GitHub** at request time, so they don't need tracing.) **`/api/roadmap/build` is the exception:** its POST runs the spec-blockers gate (`queueRoadmapBuild вҶ’ getSpecBlockers вҶ’ getRoadmap`) which reads `docs/brain/specs` from the **local bundle** at request time, so it's traced too вҖ” otherwise the gate would see zero blockers in prod and silently pass.

## Status / open work

**Shipped:** board, detail pages, editable card + per-phase status (incl. Cut), authoring chat (new + refine + **seed/re-hydrate**), build dispatch + per-phase build + report-issue fix-builds, answer loop, approval gates, squash-merge, **verify вҶ’ fold-build вҶ’ archive** (Mark verified & archive), the **Archived** section, **New spec from brain**, the **taxonomy map**, and the **goals + functions layer** (goal board/detail with rollup, function homes, **Plan/Re-plan** вҶ’ planner вҶ’ approve tree вҶ’ auto-author + queue builds вҖ” the [[../specs/goal-decomposition-engine|goal-decomposition engine]]). The box worker runs builds + plans on Max ([[../recipes/build-box-setup]]). **Status accuracy** ([[../specs/roadmap-status-accuracy]]): phase-consensus beats a stale H1 title + a live-`agent_jobs` overlay that re-buckets a Planned card into In progress the instant a build is queued.

**Spec blockers** ([[../specs/spec-blockers]], P1): a spec declares `**Blocked-by:** [[other]]`; the board shows a "рҹ”’ Blocked by вҖҰ" chip + disabled Build, and `queueRoadmapBuild` refuses to enqueue an uncleared spec (the dashboard + Slack chokepoint). P2 (auto-queue on unblock + routing the worker's planner enqueue through the gate) is still вҸі.

**Open:** instant card re-bucket on **manual status-control** change (the build/verify paths now re-bucket via `router.refresh()` + the live overlay вҖ” see [[../specs/roadmap-status-accuracy]]; `StatusControl`'s direct emoji edits still reflect on reload); README track-emoji auto-sync; the `needs_input`/`needs_approval` round-trips + the **planвҶ’approveвҶ’author** loop await their first real-run exercise; the **New spec from brain** picker is a typed brain-slug (no autocomplete over the ~600 pages yet). The planner commits authored specs **straight to main** (no separate planning PR) so queued builds find them on `origin/main`; the build PRs remain the code merge-gate.
