# dashboard/roadmap

The project-manager board + build console for the brain. Reads `docs/brain/specs/` to show **planned / in progress / shipped**, and (owner-only) lets you edit status, author/refine specs with Opus, dispatch autonomous builds on the box, answer build questions, approve prod actions, and squash-merge ‚Äî all phone-friendly. Full end-to-end: [[../lifecycles/roadmap-build-console]].

**Routes:** `/dashboard/roadmap` (board) + `/dashboard/roadmap/[slug]` (spec detail) + `/dashboard/roadmap/box` (live build-box view) + `/dashboard/roadmap/map` (taxonomy: Function ‚Üí Mandate/Goal ‚Üí Spec) + `/dashboard/roadmap/goals` + `/dashboard/roadmap/goals/[slug]` (goal board + detail) + `/dashboard/roadmap/functions/[slug]` (function home). Server components, `dynamic = "force-dynamic"` (the box view is a client poller).
**Sidebar:** **Developer** section (owner-only) ‚Üí **Roadmap** + **Build box** + **Goals** + **Taxonomy map** + **Brain** + [[branches]].

## Surfaces

- **Board** ‚Äî `src/lib/brain-roadmap.ts` `getRoadmap()` parses specs (+ `README.md` track chips). Three columns from each spec's status (`‚è≥ planned ¬∑ üöß in progress ¬∑ ‚úÖ shipped`; phases can also be `‚ùå cut`). Cards show summary, phase count pills, and live build status.
- **Detail page** ‚Äî `marked` ‚Üí `prose`; `[[wikilinks]]` to other specs become links. **Refine with Opus** button.
- **Editable status** (owner) ‚Äî `StatusControl.tsx` (card: Planned/Doing/Shipped) and `PhaseList.tsx` (per-phase dots incl. **Cut**, + a per-phase **build**). Each click commits the emoji to the brain markdown on `main` via `POST /api/roadmap/status` (`phaseIndex` targets the Nth `## Phase`). The markdown stays the source of truth ‚Äî no DB overrides.
- **Authoring chat** ‚Äî `AuthoringChat.tsx` + `POST /api/roadmap/chat` (Opus `claude-opus-4-8`, Anthropic API). **‚ú® New feature** (board header) writes a new `specs/{slug}.md`; **Refine with Opus** (detail page) edits an existing one. Finalize commits the spec to `main` (+ optional **Save & build** queues a job).
- **Build dispatch** ‚Äî `BuildButton.tsx`: **Build/Rebuild** (hidden on shipped specs), per-phase **build**, and **Report issue** (queues a scoped *fix-build* via `instructions` ‚Äî works on shipped specs, no spec edit). All hit `POST /api/roadmap/build` ‚Üí inserts an [[../tables/agent_jobs]] row (one active per spec). The chip polls `GET /api/roadmap/build?slug=` until terminal.
- **Live build-box view** (`/box`, [[../specs/build-box-status-view]]) ‚Äî `BoxChip.tsx` (board header) + `box/page.tsx` poll `GET /api/roadmap/box`, which returns the singleton [[../tables/worker_heartbeats]] row (enriched with `build_lanes`/`fold_lanes` totals + a `lanes` array of what each in-flight lane is building) plus open [[../tables/agent_jobs]] split into `queue` (waiting) / `paused` (needs_input|needs_approval). The page shows a **health header** (healthy/stale by `last_poll_at` age, `running_sha` ‚Üí commit, uptime since `started_at`), a **lane grid** (build/plan + fold cells; in-use cells show spec_slug ¬∑ kind chip ¬∑ elapsed, free cells "open"), **queue depth**, and a **paused callout**. The board-header **chip** ("Build box ¬∑ N/5 lanes ¬∑ healthy") goes stale/red when the heartbeat is old. Owner-only, read-only ‚Äî pausing/killing lanes stays a CLI job ([[../recipes/manage-the-build-queue]]).
- **Build feedback** ‚Äî when a build pauses: `needs_input` shows an **answer form** (`POST /api/roadmap/answer`); `needs_approval` shows **Approve & apply** cards with the command preview (`POST /api/roadmap/approve`). Both flip the job to `queued_resume` so the box worker resumes it. Completed builds show **Squash & merge** (reuses `POST /api/branches/[number]/merge`).
- **Verify + archive** (`BuildButton.tsx`, owner) ‚Äî shipped cards show **Mark verified & archive**: the owner-only, human "I tested it in prod" gate distinct from Shipped (built + deployed, automated). It marks the spec [[../tables/pending_folds|pending-fold]] (`POST /api/roadmap/build` `{ verify: true }` ‚Üí `enqueue_fold`) and **coalesces** into ONE batch fold-build ([[../specs/fold-build-batching]]): a single `kind='fold'` job folds **all** pending-fold specs into their brain homes, writes a per-spec `archive.d/{slug}.md`, regenerates [[../archive]] + README counts, `git rm`s the specs, and opens **one** PR. The card flips to **"Folding‚Ä¶"** (`getPendingFolds`) until merge ‚Üí the spec leaves the board into the **Archived** section. The Shipped column is relabeled **"Shipped ‚Äî awaiting verification"** so it stays a short, real to-do list.
- **Archived section** ‚Äî a collapsed `<details>` below the columns lists verified features (link ‚Üí their lifecycle/brain page). `getArchive()` reads the per-spec `docs/brain/archive.d/*.md` entries (falling back to the generated [[../archive]]). Each row also offers **New spec from brain** (re-hydration) seeded with that page.
- **New spec from brain** (re-hydration) ‚Äî `AuthoringChat seed` (board header **üßÝ New spec from brain**, or per archived entry with a fixed `seedSlug`). Pick a brain page (lifecycle/dashboard/table) or archived entry ‚Üí `POST /api/roadmap/chat` seeds Opus with the **current** brain page content ‚Üí drafts a *fresh* spec to extend/fix it (never reactivates a stale snapshot) ‚Üí normal Save / Save & build.
- **Taxonomy map** (`/map`, `getFunctionMap()`) ‚Äî every spec grouped Function ‚Üí Mandate/Goal ‚Üí Spec, built from each spec's owner + parent (so it never drifts). Function headers link to the function home; an **Orphan specs** panel flags any spec with no owner (the no-orphan lint).
- **Goals board + detail** (`/goals`, `/goals/[slug]`, `getGoals()`/`getGoal()`) ‚Äî the [[../specs/goal-decomposition-engine|goal-decomposition engine]] surface. Cards show success metric, a **rollup % bar** (mean of milestone completion ‚Äî `parseGoal` averages each milestone's linked-spec phase completion; advances automatically as leaf specs ship), and milestone/spec counts. Detail renders the goal doc + a **milestone tree** with each leaf spec's live status, and an owner-only **Plan / Re-plan** control (`PlanButton.tsx`).
- **Function home** (`/functions/[slug]`, `getFunction()`/`parseFunction()`) ‚Äî the director's charter rendered + a sidebar of mandates (each perpetual: metric + owned specs, **no %**) and owned/contributed goals. A spec card's owner chip links here.
- **Plan a goal** (`PlanButton.tsx`, owner) ‚Äî **Plan goal** / **Re-plan** ‚Üí `POST /api/roadmap/plan` inserts a `kind='plan'` [[../tables/agent_jobs]] row (one active plan per goal). The box worker runs the `plan-goal` skill ‚Üí proposes a milestone‚Üíspec tree as `pending_actions` (`type:'spec'`) ‚Üí `needs_approval`. The button shows each proposed branch (owner ¬∑ parent ¬∑ intent) with **Approve / Decline** (reuses `POST /api/roadmap/approve`); once every branch is decided the job ‚Üí `queued_resume` and the worker auto-authors the approved specs (committed to main), wikilinks them into the goal doc, records declines (‚ùå), and queues their builds. Polls `GET /api/roadmap/plan?goalSlug=`.

## Data sources

- **Brain markdown** (board + detail) ‚Äî `docs/brain/specs/*.md` + `docs/brain/goals/*.md` + `docs/brain/functions/*.md`, read at request time via `src/lib/brain-roadmap.ts` (`getRoadmap`/`getSpec`/`getGoals`/`getGoal`/`getFunctions`/`getFunction`/`getFunctionMap`; `parseGoal`/`parseFunction` extract milestones/mandates + linked specs + rollup). `[[wikilinks]]` resolve to dashboard routes via `src/lib/brain-links.ts` (`preprocessBrainWikilinks` ‚Äî specs‚Üí`/roadmap/{slug}`, goals‚Üí`/roadmap/goals/{slug}`, functions‚Üí`/roadmap/functions/{slug}`). The static, canonical layer.
- **`agent_jobs`** (live build + plan state) ‚Äî read via `getLatestJobsBySlug(workspaceId)` (build cards) + `getLatestPlanJob(workspaceId, goalSlug)` (goal Plan control); `BuildButton`/`PlanButton` poll the API for updates. The live, actionable layer.

## Billing

Authoring chat ‚Üí Anthropic API (cheap). Builds ‚Üí **Max** (box `claude -p`, no API key). See [[../lifecycles/roadmap-build-console]] ¬ß Billing.

## Vercel gotcha

The board/detail read files under `docs/brain/`, which Vercel's tracer would prune. `next.config.ts` ‚Üí `outputFileTracingIncludes` ships the spec/lifecycle/**goals/functions** markdown into the `/dashboard/roadmap`, `/[slug]`, `/map`, `/goals`, `/goals/[slug]`, and `/functions/[slug]` function bundles. Without it those routes render empty in prod. (The chat/build/status/approve/**plan** API routes read the brain from **GitHub** at request time, so they don't need tracing.)
