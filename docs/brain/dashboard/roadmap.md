# dashboard/roadmap

The project-manager board + build console for the brain. Reads `docs/brain/specs/` to show **planned / in progress / shipped**, and (owner-only) lets you edit status, author/refine specs with Opus, dispatch autonomous builds on the box, answer build questions, approve prod actions, and squash-merge — all phone-friendly. Full end-to-end: [[../lifecycles/roadmap-build-console]].

**Routes:** `/dashboard/roadmap` (board) + `/dashboard/roadmap/[slug]` (spec detail) + `/dashboard/roadmap/box` (live build-box view) + `/dashboard/roadmap/map` (taxonomy: Function → Mandate/Goal → Spec) + `/dashboard/roadmap/goals` + `/dashboard/roadmap/goals/[slug]` (goal board + detail) + `/dashboard/roadmap/functions/[slug]` (function home). Server components, `dynamic = "force-dynamic"` (the box view is a client poller).
**Sidebar:** **Developer** section (owner-only) → **Roadmap** + **Build box** + **Goals** + **Taxonomy map** + **Spec Tests** + **Human-test queue** (amber count badge) + **Brain** + [[branches]].

## Surfaces

- **Board** — `src/lib/brain-roadmap.ts` `getRoadmap()` parses specs (+ `README.md` track chips). **Four columns** from each spec's `SpecStatus` (`⏳ planned · 🚧 in progress · ✅ shipped · ⏸️ deferred`; phases can also be `❌ cut`). Cards show summary, phase count pills, and live build status. **Deferred** ([[../specs/director-drives-all-specs-and-deferred-status]] Phase 1) is its own column for parked work — a spec carrying a leading `**Deferred:**` marker (authored by [[../specs/board-grooming]] split cards) or `**Status:** deferred`; `deriveStatus` returns `deferred` over its ⏳ phases, every auto-build lane skips it ([[../libraries/platform-director]]), and only the CEO un-defers it (remove the marker → Planned). `resolveBoardStatus` keeps a deferral markdown-owned (the phase-progress DB mirror never overrides it).
- **Board filters** ([[../specs/roadmap-goal-and-source-filters]]) — `RoadmapFilters.tsx` (a client component, replaced `SpecSearch.tsx`) renders three filters that AND together as one pure-DOM visibility pass (no server round-trip): the **search box** (substring over `data-spec-search`), a **goal dropdown** (every `docs/brain/goals/*.md`; pick one → only that goal's specs + a **progress header** `Title — X/Y shipped · N building · M planned` counting the visible cards; sticky in the `?goal=` URL param so a goal view is shareable), and **source chips** (All · 🎯 Goal · 🔧 Repair · ✋ Manual). The loader (`getRoadmapFilters()`) resolves goal→spec membership (goal-doc `[[spec]]` wikilinks ∪ a `**Parent:**` that names the goal) + per-spec source once; `page.tsx` stamps each card with `data-goal` / `data-source` / `data-status`. **Status accuracy** ([[../specs/roadmap-status-accuracy]]): (1) **phase consensus beats a stale title** — `deriveStatus` treats an all-`✅` phase set as **shipped** regardless of the H1 title emoji (a forgotten `🚧`/`⏳` title no longer reads as "Doing"); the title still wins for an explicit `❌` cut or when there are no phases. (2) **live-job overlay** (`page.tsx` `effectiveStatus`) — a **Planned** spec with an **active `agent_jobs`** row renders in **In progress**, so tapping Build moves the card immediately and it stays there while the build runs, reverting to its markdown status once the job is terminal. The overlay only *promotes* planned→in_progress; it never demotes a Shipped spec. (3) **DB-first instant status + deploy flag** ([[../specs/spec-card-db-companion]]) — the board reads the [[../tables/spec_card_state]] live mirror ([[../libraries/spec-card-state]] `getSpecCardStates`) and overlays it via `resolveBoardStatus` (DB-first for the deploy-lag; a markdown that's ahead wins, so markdown stays canonical), so a merge / drift flip / owner mark flips the card **before** the markdown redeploys — **zero GitHub API calls for status** (this retired the quota-burning `roadmap-reads-specs-from-git` git read). A merged card shows a **`shipped · deploying`** chip (its `last_merge_sha` ≠ the live `VERCEL_GIT_COMMIT_SHA`, `deploymentState`) until a deploy carrying that SHA is up, then **`shipped · live`** — clean signal, no webhook.
- **Detail page** — `marked` → `prose`; `[[wikilinks]]` to other specs become links. **Refine with Opus** button.
- **Editable status** (owner) — `StatusControl.tsx` (card: Planned/Doing/Shipped) and `PhaseList.tsx` (per-phase dots incl. **Cut**, + a per-phase **build**). Each click commits the emoji to the brain markdown on `main` via `POST /api/roadmap/status` (`phaseIndex` targets the Nth `## Phase`). The markdown stays the source of truth — no DB overrides. **Built-on-branch phase render** (`chain-accumulation-built-phase-render`): under M1's branch-accumulation flow a phase that has BUILT on the spec branch carries a `build_sha` and stays `spec_phases.status='in_progress'` (M2 reserves `shipped` for M5's main promotion) — so `PhaseList` renders such a phase (`build_sha` set, not yet shipped/rejected) as **Built** (teal dot + "Built" badge, derived) rather than the amber "Building" of a phase that's merely queued/in-flight (`in_progress` with NO `build_sha`). Rendering overlay only — the stored `status` is untouched; the spec rolls up `in_testing` once ALL phases are built (handled in `rollupPhaseStatus` / the in-testing overlay, not here).
- **Critical priority pip + toggle** ([[../specs/director-executable-plans-and-priority-board-pip]] Phase 1) — a card carrying the line-anchored `**Priority:** critical` marker (derived as `SpecCard.critical` in [[../libraries/brain-roadmap]], `brain-roadmap.ts:255`) renders a **🔴 Critical** pip next to its title — human-visible at-a-glance signal for queue-first / gating specs (the build lanes already order critical-first, #455). Owner-only `CriticalToggle.tsx` (next to `StatusControl`) flips it via `POST /api/roadmap/priority` `{ slug, critical }`, which adds/removes the line-anchored marker in `docs/brain/specs/{slug}.md` on `main` (same convention the derive reads) — so the marker is settable from the board, not only via Ada's `spec-edit` card or a `directive`'s `criticalSpecs`. Optimistic, markdown stays canonical.
- **Authoring chat** — `AuthoringChat.tsx` + `POST /api/roadmap/chat` (Opus `claude-opus-4-8`, Anthropic API). **✨ New feature** (board header) writes a new `specs/{slug}.md`; **Refine with Opus** (detail page) edits an existing one. Finalize commits the spec to `main` (+ optional **Save & build** queues a job).
- **Build dispatch** — `BuildButton.tsx`: **Build/Rebuild** (hidden on shipped specs), **Build all** (chains every ⏳ phase — see below), per-phase **build**, and **Report issue** (queues a scoped *fix-build* via `instructions` — works on shipped specs, no spec edit). All hit `POST /api/roadmap/build` → inserts an [[../tables/agent_jobs]] row (one active per spec).
- **Build all — phase chain** ([[../specs/build-all-phases-chain]]) — **Build all** (`POST /api/roadmap/build { chainPhases:true }` → `queueRoadmapBuild(..., { chainPhases:true })`, [[../libraries/roadmap-actions]]) queues only the **first ⏳ phase** scoped, tagged `chain_phases:true` on the [[../tables/agent_jobs]] row. When that phase's PR merges (auto, via the auto-ship pipeline), `reconcileMergedJobs` → `queueNextChainedPhase` ([[../libraries/agent-jobs]]) reads `main` and queues the **next ⏳ phase** (also `chain_phases`, on fresh main atop the prior phase's code), repeating until all phases ✅ — so a multi-phase spec ships with **no owner clicks** between phases. A **failed / needs_approval** phase never reaches `merged`, so the chain stops/pauses there (resuming + merging it resumes the chain); single-phase / per-phase **Build N** builds stay non-chained (no `chain_phases` → no auto-queue). The same **🛠️ Build all** action exists on the Slack App Home ([[../libraries/slack-home]] `HOME.build`). The chip polls `GET /api/roadmap/build?slug=` until terminal. After a queuing POST (build / report-issue / verify) the button calls `router.refresh()` so the server board re-runs and the live overlay re-buckets the card right away (instant, real-DB — not a client-only guess).
- **Create PR recovery** ([[../specs/build-recover-pr-create]]) — a build that **finished + pushed its `claude/*` branch** but failed the final `gh pr create` lands `needs_attention` with `error="branch pushed but PR creation failed"`. The card detects this recoverable sub-case (status + that exact error + a `spec_branch`) and shows **Create PR** as the **primary** action — `POST /api/roadmap/build { jobId, recoverPr:true }` → `createPrForJob` ([[../libraries/roadmap-actions]]) opens the PR for the pushed branch (idempotently adopting one that already exists) and flips the job → `completed` with the PR attached — instead of discarding the completed build. **Rebuild** stays as a demoted, clearly-labeled *discard-and-redo* fallback. A genuinely-stuck `needs_attention` (no pushed branch) keeps its plain treatment.
- **Build prerequisites — spec blockers** ([[../specs/spec-blockers]]) — a spec can declare `**Blocked-by:** [[a]], [[b]]` under its H1 ([[../libraries/brain-roadmap]] `parseSpec`); the board resolves each to `blockedBy: { slug, title, status, cleared }` (cleared when the blocking spec is shipped, archived/folded, or dangling). A spec with any **uncleared** blocker shows a **"🔒 Blocked by …"** chip (each prerequisite + its ✅/⏳/🚧) and a **disabled Build** button ("🔒 Blocked", tooltip names what must ship first); cleared blockers render ✅. The real enforcement is server-side: `queueRoadmapBuild` ([[../libraries/roadmap-actions]]) checks `getSpecBlockers` **before inserting** the build row and refuses an uncleared spec with `409 { error:"Blocked by: <slug> (<emoji>)" }` — so the dashboard **and** the Slack `/build` (the shared chokepoint) both refuse a blocked spec. A spec with no Blocked-by (or all cleared) builds normally — no regression. *(P2 — auto-queue the dependent build when the last blocker ships, + routing the box worker's planner enqueue through the same gate — is not yet shipped.)*
- **Live build-box view** (`/box`, [[../specs/build-box-status-view]]) — `BoxChip.tsx` (board header) + `box/page.tsx` poll `GET /api/roadmap/box`, which returns the singleton [[../tables/worker_heartbeats]] row (enriched with `build_lanes`/`fold_lanes` totals + a `lanes` array of what each in-flight lane is building) plus open [[../tables/agent_jobs]] split into `queue` (waiting) / `paused` (needs_input|needs_approval), and a `failed` list (build/plan jobs whose **latest** attempt is `status='failed'` — a since-rebuilt spec is excluded, so it never shows a stale failure). The page shows a **health header** (healthy/stale by `last_poll_at` age, `running_sha` → commit, uptime since `started_at`), a **lane grid** (build/plan + fold cells; in-use cells show spec_slug · kind chip · elapsed — a chained/per-phase build also appends `· Phase N` after the slug, derived by the worker from the job's instructions [[../specs/box-lane-show-phase]], slug-only for a whole-spec build; free cells "open"), **queue depth**, a **failed-builds callout**, and a **paused callout**. **Chained phase cache status** ([[../specs/chained-phase-session-resume]] Phase 2 — `claude_session_id` resume observability): a chained/per-phase build's phase-display also appends a resume indicator — `· Phase N [cache-warm]` when the phase job carries a `claude_session_id` (the prior phase's session is being resumed via prompt-cache), or `· Phase N [fresh]` when enqueued as a fresh `queued` build (no session resumption, e.g., owning account was capped or the cache window expired). This surface makes session-resume savings **observable and auditable** — the builder knows which phases are reusing cached context vs. re-hydrating from scratch. **Cost attribution** — resumed phases show high `cache_read_input_tokens` vs low `input_tokens` in [[../tables/agent_job_costs]], whereas fresh phases show full `input_tokens` with no reads; the ratio directly proves the cache-warm optimization's efficiency. The **failed-builds callout** (red) is the "I don't have to dig into a spec card to know a build died" surface: each failed build shows its slug (→ detail page), how long ago, the real **failure reason parsed from `log_tail`** (the `claude -p` result JSON's `api_error_status` + `result` — so a transient **529 Overloaded** / a Max-limit error reads as such vs a real bug; a `tsc` failure shows its output tail), and a **↻ Retry build** button (`POST /api/roadmap/build {slug}` — re-queues; optimistically clears the row). The board-header **chip** ("Build box · N/8 lanes · healthy") goes stale/red when the heartbeat is old. A **grade/coach lane shows the ACTING DIRECTOR** ([[../specs/box-grading-session-and-account-count-fixes]] Phase 3), not a generic "Agent Grade" + default mascot: `agent-grade`/`agent-coach`/`director-grade`/`campaign-grade`/`gap-grade` render the supervising director's avatar + name + verb — **"Ada Grading"** (platform workers), **"Henry Grading"** (director-grade = the CEO grading directors), **"Cleo Grading"** (campaign/gap = Growth). Dynamic: `GET /api/roadmap/box` resolves `director_function` from the job's `instructions.fn` (else a per-kind default) onto each lane AND queued job; `box/page.tsx` `personaForKind(kind, director_function)` + `gradeCoachInfo` render it consistently across the in-flight lane grid and the "Jobs in queue" feed. Owner-only, read-only — pausing/killing lanes stays a CLI job ([[../recipes/manage-the-build-queue]]). **Dead-link defense ([[../specs/auto-ship-pipeline]]/fold-guard-live-build):** `GET /api/roadmap/box` stamps each paused/failed build with `spec_missing` (computed by `specMissing(kind, spec_slug)` against the live, non-archived spec slugs); a job whose spec was folded/archived/deleted carries `spec_missing:true`, and `box/page.tsx` routes its callout link to **`/dashboard/roadmap`** (the board, always 200) with a `(spec archived)` tag instead of the 404 `/dashboard/roadmap/{slug}` — so a fold that races a build never leaves a dead link. A job whose spec still exists links to the spec detail page as before. The preventive + cleanup guards live in [[../libraries/agent-jobs]] (`getLiveJobForSlug` / `cancelJobsForArchivedSpecs`). **consolidate-premerge-checks-one-session Phase 1 — fused pre-merge check.** A pre-merge `spec-test` lane (`kind='spec-test'` on a `claude/*` branch with a preview URL) now emits BOTH the spec-test verdict AND a security review off the same loaded diff (one session; halves pre-merge Codex sessions and skips a full cold re-hydration). The runtime call site is unchanged (`runSpecTestJob` in `scripts/builder-worker.ts`); the lane is Codex-primary via `runBoxSession` with Claude fallback. **Phase 2 — multi-persona lane card.** A fused pre-merge lane renders as BOTH agents on the box view — the in-flight lane cell AND the queued-jobs feed show TWO `PersonaAvatar`s (Vera the spec-tester + Vault the security reviewer, `-space-x-2` overlap), a **"Vera → Vault · pre-merge check"** title, and a static indigo **"spec-test · security"** sub-task chip. `GET /api/roadmap/box` enriches each `spec-test` lane (and queued spec-test job) by looking up its `agent_jobs.spec_branch`; a value starting with `claude/` stamps `fused_pre_merge:true`. `box/page.tsx` `fusedPreMergeInfo(kind, fused_pre_merge)` resolves that flag to `{ personas:[Vera,Vault], title, label }`; `LaneCell` + `QueuedJobsLog` render both avatars + the dual title when it fires, and fall through to the existing single-persona path otherwise. The sub-task label is **static** because the fused session is Codex-primary and Codex has no `TodoWrite` — the card must read correctly with an empty `session_checklist`. A non-fused `spec-test` lane (post-ship, no branch) still renders single-persona (Vera). The runtime indicator (Codex/Claude) is unchanged for a fused lane. **box-failed-build-supersede-and-dismiss Phase 1 — supersede by OUTCOME, not `created_at`.** The per-slug "latest attempt" selector in `GET /api/roadmap/box` now ranks build/plan agent_jobs by outcome precedence (`merged`/`completed` > in-flight `building`/`queued`/`queued_resume`/`needs_input`/`needs_approval` > `failed`), using `created_at` only as a same-tier tiebreaker (`selectLatestBuildBySlug` in `src/lib/box-failed-supersede.ts`, tested in `src/lib/box-failed-supersede.test.ts`). Before the fix (2026-07-02, `box-self-update-persist-skip-reason`) a long build that STARTED earlier and MERGED later (06:09 → 08:50) was masked on the card by a spurious quick failed attempt created in between (08:41) — raw recency preferred the failure over the actual success. Now a terminal success sibling always wins, so a since-rebuilt spec never resurfaces as failed. **Phase 2 — one-tap owner Dismiss.** Each failed-build card also carries a **Dismiss** button next to `↻ Retry build` (`POST /api/roadmap/box/dismiss-failed { jobId }`, owner-gated) that flips the target failed row to `status='completed'`, `error=null`, `log_tail='dismissed by owner (…) at <ts>'` and optimistically removes the card — the exact effect of the manual clear done on 2026-07-02, now one-tap instead of a psql edit. The row is UPDATED, not deleted, so grades/costs keep their `agent_job_id`. A non-owner gets 403; a non-`failed` job gets 400 (the button can't force-complete an in-flight or already-resolved job).
- **Build feedback** — when a build pauses: `needs_input` shows an **answer form** (`POST /api/roadmap/answer`); `needs_approval` shows **Approve & apply** cards with the command preview (`POST /api/roadmap/approve`). Both flip the job to `queued_resume` so the box worker resumes it. Completed builds show **Squash & merge** (reuses `POST /api/branches/[number]/merge`).
- **Verify + archive** (`BuildButton.tsx`, owner) — shipped cards show **Mark verified & archive**: the owner-only, human "I tested it in prod" gate distinct from Shipped (built + deployed, automated). It marks the spec [[../tables/pending_folds|pending-fold]] (`POST /api/roadmap/build` `{ verify: true }` → `enqueue_fold`) and **coalesces** into ONE batch fold-build ([[../specs/fold-build-batching]]): a single `kind='fold'` job folds **all** pending-fold specs into their brain homes, writes a per-spec `archive.d/{slug}.md`, regenerates [[../archive]] + README counts, `git rm`s the specs, and opens **one** PR. The card flips to **"Folding…"** (`getPendingFolds`) until merge → the spec leaves the board into the **Archived** section. The Shipped column is relabeled **"Shipped — awaiting verification"** so it stays a short, real to-do list.
- **Verification card** (`VerificationCard.tsx`, spec detail) — the "how to test this" checklist ([[../specs/verification-guides]]), rendered as a teal **"✅ How to verify in prod"** card in the detail sidebar **right under the build/verify buttons**, so the test steps sit where **Mark verified & archive** is. `extractSpecSection`/`stripSpecSection` (`src/lib/brain-roadmap.ts`) lift the spec's `## Verification` block out of the markdown (and out of the article body so it's not shown twice). A spec with **no** section shows a "No test plan yet" card with an owner-only **Generate test plan** button → `POST /api/roadmap/chat` `{ action:"generate_verification" }` → Opus drafts a concrete section, brain-grounded, and commits it to `specs/{slug}.md` (appears after the next deploy). Builds + the authoring chat write the section at ship time, so new specs arrive test-ready. **Box QA results** ([[../specs/spec-test-agent]]): when the spec has a latest [[../tables/spec_test_runs]] run, the card also renders the **Agent-tested** stamp + chip, the **per-bullet verdicts** (✅/✗/👤/? with expandable evidence), and a distinct **👤 Needs human testing** list — so the owner does only those, then verifies. An owner-only **Test now** queues a fresh box run.
- **Spec Tests page** (`/dashboard/developer/spec-tests`, [[../specs/spec-test-agent]]) — the box spec-test QA agent's report home: every shipped-but-unverified spec with its latest run (the **Agent-tested ✅ / ⚠️ issues** stamp, the `✅·✗·👤·?` chip, each check's verdict + expandable evidence) plus an aggregated **👤 Needs human testing** list across all specs and a per-spec **Test now** button. Reads via `src/lib/spec-test-runs.ts` (`getLatestSpecTestRuns`). Owner-only. **Browser-check screenshot evidence** ([[../specs/spec-test-deep-verification]] Phase 1): a check carrying a `screenshot` (the headless-browser tool's full-page capture, stored in the **private** `spec-test-evidence` bucket) renders the image inline under its expandable evidence — the page signs the path per-render server-side (`signSpecTestScreenshot`, short-TTL signed URL; never a public URL, since a dashboard screenshot can contain real customer data). The roadmap board's shipped cards also carry the compact chip + stamp (linking here). The agent **stamps**, it never marks a spec verified — that owner-only gate stays untouched. A regressed spec (latest run `agent_verdict='issues'`) also gets a **Propose fix spec** button here + on the VerificationCard (see Human-test queue below). **Pre-merge section** ([[../specs/premerge-spectest-rerun-and-visibility]] Phase 3): a `claude/*` PRE-MERGE (in_progress) spec's branch-scoped runs are invisible on the shipped list (which filters to `status='shipped'`), so they render in their own **Pre-merge** section below, showing the latest run per `(slug, branch)` regardless of verdict (`approved` / `needs_human` / `issues` / `error`). Each pre-merge row displays: spec title, **Agent-tested** verdict stamp, `✅·✗·👤·?` chip (for non-error runs), branch chip, preview link, and a **Test now** button. The section's intro explains the re-run button forces a fresh Vercel preview capture + re-test even after a stuck terminal verdict (`issues`/`error`), with the constraint that an open spec-test job for the same `(workspace, slug, branch)` still blocks stacking (case a from [[../libraries/agent-jobs]] `enqueuePreMergeSpecTest` dedup). Error runs highlight with amber styling + the error message + `raw output tail`; non-error runs with passes/fails are styled normally and expand to show the full check list. The section reads via `getPreMergeRuns(workspaceId)` (latest per-branch; null spec_branch excluded by the SQL join), and filters out any run whose spec is already listed above (shipped + archived set). A spec already listed under shipped is excluded from the pre-merge section. **Refresh-on-change** ([[../specs/premerge-spectest-rerun-and-visibility]] Phase 2): when re-running a pre-merge after a branch code change, [[../libraries/agent-jobs]] `backstopPreMergeChecks` standing pass ensures the `preview_url` reflects the branch's CURRENT HEAD (queries Vercel for the newest deployment of the branch, gated on newer `build`/`pr-resolve` activity vs the latest run timestamp) — never re-tests against a stale preview. **Owner-forced re-run** ([[../libraries/agent-jobs]] `enqueuePreMergeSpecTest` `opts.force`): the **Test now** button passes `{ force: true }` to skip the terminal-verdict dedup block (case b) so an owner can kick a stuck `approved`/`needs_human`/`issues` verdict from the dashboard without waiting for a fresh `build`/`pr-resolve` row or the standing-pass backstop. The API route `POST /api/roadmap/spec-test` also fresh-captures the branch preview before enqueueing, so the forced re-test always probes current code. Retries hit the per-build `*.vercel.app` preview, never prod.
- **Human-test queue** (`/dashboard/developer/spec-tests/human-queue`, [[../specs/spec-test-agent]] Phase 2) — a focused queue (modeled on [[../specs/improve-queue|improve-queue]]) aggregating every `needs_human` check across the latest run of every shipped-unverified spec — the parts only the owner can run (visual/UX, prod-mutating, or **fault-injection / forced-failure** checks the read-only agent can't exercise). Each waiting check has **✓ Tested** / **Dismiss**; a collapsed **Done** group lets the owner **Re-open** one. Owner resolutions persist in [[../tables/spec_test_human_checks]] (keyed by a stable `check_key` hash of the bullet, so a resolution survives re-runs). **Regressions** (a shipped spec with an **evidence-backed `fail`** in its latest run — the agent observed breakage; a `needs_human`/`inconclusive` check never counts as a regression, see [[../specs/spec-test-classification]]) surface loudly at the top with a one-click **Propose fix spec** → `POST /api/roadmap/chat` `{ action:"propose_fix" }`, which seeds a box authoring chat (`Fix: {slug}`) with the failing checks as the brief + kicks off the first box turn ([[../specs/box-spec-chat]]); the owner reviews + finalizes it into a fix spec (the agent proposes, the owner finalizes). The brief now tags each failing check with its `[check {key}]` hash and instructs the box to stamp a machine-readable `**Fixes:** {origin} (check {key}…)` line in the fix spec — so when the fix build **merges**, [[../libraries/agent-jobs]] `retestOriginIfFixMerged` auto-re-tests the **origin** and its stale "issues" badge clears with no manual re-queue ([[../specs/fix-ship-retests-origin]]). Reads/writes via `GET`/`POST /api/developer/spec-test/human-queue` (`getHumanTestQueue`, owner-gated); the sidebar shows an amber count badge (waiting + regressions). Polls every ~8s.
- **Archived section** — a collapsed `<details>` below the columns lists verified features (link → their lifecycle/brain page). `getArchive()` reads the per-spec `docs/brain/archive.d/*.md` entries (falling back to the generated [[../archive]]). Each row also offers **New spec from brain** (re-hydration) seeded with that page.
- **New spec from brain** (re-hydration) — `AuthoringChat seed` (board header **🧠 New spec from brain**, or per archived entry with a fixed `seedSlug`). Pick a brain page (lifecycle/dashboard/table) or archived entry → `POST /api/roadmap/chat` seeds Opus with the **current** brain page content → drafts a *fresh* spec to extend/fix it (never reactivates a stale snapshot) → normal Save / Save & build.
- **Taxonomy map** (`/map`, `getFunctionMap()`) — every spec grouped Function → Mandate/Goal → Spec, built from each spec's owner + parent (so it never drifts). Function headers link to the function home; an **Orphan specs** panel flags any spec with no owner (the no-orphan lint).
- **Goals board + detail** (`/goals`, `/goals/[slug]`, `getGoals()`/`getGoal()`) — the [[../specs/goal-decomposition-engine|goal-decomposition engine]] surface. Cards show success metric, a **rollup % bar** (mean of milestone completion — `parseGoal` averages each milestone's linked-spec phase completion; advances automatically as leaf specs ship), and milestone/spec counts. Detail renders the goal doc + a **milestone tree** with each leaf spec's live status, and an owner-only **Plan / Re-plan** control (`PlanButton.tsx`).
- **Function home** (`/functions/[slug]`, `getFunction()`/`parseFunction()`) — the director's charter rendered + a sidebar of mandates (each perpetual: metric + owned specs, **no %**) and owned/contributed goals. A spec card's owner chip links here.
- **Plan a goal** (`PlanButton.tsx`, owner) — **Plan goal** / **Re-plan** → `POST /api/roadmap/plan` inserts a `kind='plan'` [[../tables/agent_jobs]] row (one active plan per goal). The box worker runs the `plan-goal` skill → proposes a milestone→spec tree as `pending_actions` (`type:'spec'`) → `needs_approval`. The button shows each proposed branch (owner · parent · intent) with **Approve / Decline** (reuses `POST /api/roadmap/approve`); once every branch is decided the job → `queued_resume` and the worker auto-authors the approved specs (committed to main), wikilinks them into the goal doc, records declines (❌), and queues their builds. Polls `GET /api/roadmap/plan?goalSlug=`.

## Data sources

- **Brain markdown** (board + detail) — `docs/brain/specs/*.md` + `docs/brain/goals/*.md` + `docs/brain/functions/*.md`, read at request time via `src/lib/brain-roadmap.ts` (`getRoadmap`/`getSpec`/`getGoals`/`getGoal`/`getFunctions`/`getFunction`/`getFunctionMap`; `parseGoal`/`parseFunction` extract milestones/mandates + linked specs + rollup). `[[wikilinks]]` resolve to dashboard routes via `src/lib/brain-links.ts` (`preprocessBrainWikilinks` — specs→`/roadmap/{slug}`, goals→`/roadmap/goals/{slug}`, functions→`/roadmap/functions/{slug}`). The static, canonical layer.
- **`agent_jobs`** (live build + plan state) — read via `getLatestJobsBySlug(workspaceId)` (build cards) + `getLatestPlanJob(workspaceId, goalSlug)` (goal Plan control); `BuildButton`/`PlanButton` poll the API for updates. The live, actionable layer.

## Vercel gotcha

The board/detail read files under `docs/brain/`, which Vercel's tracer would prune. `next.config.ts` → `outputFileTracingIncludes` ships the spec/lifecycle/**goals/functions** markdown into the `/dashboard/roadmap`, `/[slug]`, `/map`, `/goals`, `/goals/[slug]`, and `/functions/[slug]` function bundles. Without it those routes render empty in prod. (The chat/status/approve/**plan** API routes read the brain from **GitHub** at request time, so they don't need tracing.) **`/api/roadmap/build` is the exception:** its POST runs the spec-blockers gate (`queueRoadmapBuild → getSpecBlockers → getRoadmap`) which reads `docs/brain/specs` from the **local bundle** at request time, so it's traced too — otherwise the gate would see zero blockers in prod and silently pass.
