# dashboard/control-tower

The single "is the machine healthy?" screen ([[../specs/control-tower]] Phase 1 + Phase 2; [[../specs/control-tower-agent-coverage]] adds the inline AI agents; [[../specs/control-tower-complete-coverage]] P1 adds full cron coverage + the reactive event-driven agents, P3 adds the department rollups) ‚Äî a green/amber/red tile per autonomous loop, rolled up by org function. The objective-owner's window from the north star: where the CEO sees whether every proxy-optimizing loop is still alive and doing its job.

**Route:** `/dashboard/developer/control-tower` (client poller, owner-only)
**Sidebar:** **Developer** section (owner-only) ‚Üí **Control Tower** (right under [[roadmap|Build box]]).

## Surfaces

- **Summary bar** ‚Äî counts of healthy / warning / alerting tiles + "updated Ns ago". Includes the **error panels** (the API folds their colors into the count, [[../specs/error-feed-honest-panels]]) so an unconfigured/amber panel is never counted as healthy.
- **Dependency health ‚Äî "is Claude up?"** ([[../specs/agent-outage-resilience]] Phase 2, [[../libraries/claude-health]]) ‚Äî a tile right under the summary bar showing the Claude-down circuit-breaker: the live **Claude API** + **Claude Code** component status (from the 1-min `status.claude.com` poll, [[../inngest/claude-status-poll-cron]]) plus the local consecutive-failure signal. Green = both operational + breaker closed; amber = degraded / local signal firing; red = a partial/major outage or the breaker tripped. When tripped it shows how long it's been down + "agents parked, repair fan-out suppressed" ‚Äî the box parks its autonomous agent jobs `blocked_on_dependency` and `recordError` suppresses the repair fan-out until Claude recovers (park-and-drain); the customer-facing ticket path retries across the outage (Phase 1). Fed by the `claudeHealth` field on the snapshot ([[../tables/claude_health]]).
- **Department health (CEO-glance rollups)** ([[../specs/control-tower-complete-coverage]] P3) ‚Äî the view **leads** with one rollup health tile per org function (**Platform / Growth / Retention / CS / CMO Health**, [[../functions]] ¬∑ [[../goals/ceo-mode]]). Every loop carries an `owner` function in [[../libraries/control-tower|the registry]]; each tile is the **worst-of** color across that function's loops (a single red loop turns its department amber/red) with a **healthy/total** count + **open-alert** count ‚Äî the CEO glance "which org function is healthy?". Below the rollups, each department is a **collapsible drill-in** (`<details>`, auto-open when not green) listing its loops sub-grouped by kind. A loop's owner = the function that owns the system it monitors (renewal/dunning/portal ‚Üí retention; the meta + social + creative loops ‚Üí growth/cmo; ticket-handler/triage ‚Üí cs; the build/spec-test/migration-fix agents + the data-infra/monitoring crons ‚Üí platform). `buildControlTowerSnapshot` returns `departments` (worst-of rollups from `OWNER_FUNCTIONS`) alongside `loops`.
- **Tiles, grouped by kind** (Worker ¬∑ Crons ¬∑ **Reactive agents** ¬∑ Agent lanes ¬∑ **Inline AI agents**), shown inside each department drill-in ‚Äî each tile shows the loop label + description, its expected cadence, a colored status dot + `statusText`, **last ran**, **last produced** (compacted from the heartbeat's `produced` jsonb), any **open alert** (with how long it's been open + the violation detail), and a **history strip** (last ~10 runs, green/red ticks). The Crons group now covers the **full Inngest cron set** (every `inngest.createFunction` cron), and **Reactive agents** are the event-driven Inngest functions ‚Äî `unified-ticket-handler` (THE inbound ticket pipeline), `dunning-payment-failed`, `returns-process-delivery`, `journey-session-completed`, `agent-todo-execute`, `chargeback-received` ‚Äî evaluated like inline agents (idle = green; red on liveness-while-work / error-rate). ([[../specs/control-tower-complete-coverage]] P1.)
  - **green** ‚Äî healthy, or genuinely idle (no work to do). **amber** ‚Äî warning (cron genuinely awaiting its first run, worker mid self-update, a not-ok cron beat). **red** ‚Äî an active violation that has paged the owners (a P1 liveness/freshness/stuck silence, the [[../specs/control-tower-complete-coverage]] **never-fired** check ‚Äî a registered cron with 0 beats whose deploy is older than its cadence+grace, "registered but has never run", distinct from a genuinely-fresh cron's amber "awaiting first run" ‚Äî an **inline-agent** liveness-while-work / error-rate failure ([[../specs/control-tower-agent-coverage]]), **or** a Phase 2 output-assertion failure ‚Äî escalation idle-while-work, spec-test false-success, renewal integrity ‚Äî where the loop ran but silently did nothing/wrong).
  - **Inline AI agents** (`ai:ticket-analyzer`, `ai:journey-delivery`, `ai:fraud-detector`, `ai:orchestrator`) ‚Äî event-driven, no cadence: green when healthy or genuinely idle; red on **liveness-when-work-exists** (upstream work waited but 0 successful runs in the window) or **error-rate** (>50% of ‚â•5 runs errored). `statusText` carries the window counts ("healthy ¬∑ N ok", "silent while N awaits", "failing: E/T errored"). `ai:orchestrator` ([[../specs/control-tower-agent-coverage]] Phase 2) is the per-ticket decision agent (`callSonnetOrchestratorV2`): its work-exists probe is inbound customer messages in the 2h window, and a run beats ok:false when it threw or returned a degraded/fallback decision ‚Äî so a model that errors / parse-fails on every ticket trips error-rate even though it "ran".
- **Errors section** ([[../specs/error-feed-monitoring]] Phase 1 + Phase 2; [[../specs/client-error-capture]]) ‚Äî five panels (**Vercel errors ¬∑ Inngest failures ¬∑ Supabase errors (app-layer) ¬∑ Supabase errors (DB logs) ¬∑ Client errors**) of grouped error incidents from the hidden surfaces. The fourth (Phase 2) is DB-level Supabase errors our app never saw ‚Äî Postgres/auth/API ‚Äî polled from the [[../integrations/supabase-management-logs]] API by [[../inngest/supabase-log-poll]] (`source='supabase-logs'`; green until the owner pastes the access token). The fifth is **client-side JS errors** on the storefront (PDP/customize/checkout/thank-you) + both portals ‚Äî browser crashes our server logs can't see, POSTed to `/api/client-errors` ([[../libraries/client-error-reporter]], `source='client'`; amber "awaiting first event" until the first report/heartbeat lands). Each panel is **connection-aware** ([[../specs/error-feed-honest-panels]]): green only when the feed is **configured + receiving deliveries + 0 errors** ("connected ¬∑ 0 errors, last delivery Ns ago"); **amber "not configured"** (+ a how-to-wire hint) when its secret/token is unset, or **amber "awaiting first event"** when configured but no delivery seen yet ‚Äî never a misleading green "0 errors" while disconnected. With errors it shows the recency color (red ‚â§1 h, amber ‚â§24 h), the active-signature + total-occurrence counts, and the recent incidents (title, `√ócount`, last-seen). Liveness comes from `feed:<source>` beats in [[../tables/loop_heartbeats]] (Vercel drain POST / Supabase-logs poll); Inngest is proxied by the freshness of Inngest itself (latest cron beat). A new signature or a re-firing spike pages the owners (rate-limited: a burst of the same error = one page).
- **Coverage self-audit section** ([[../specs/control-tower-complete-coverage]] Phase 2, [[../libraries/control-tower-self-audit]]) ‚Äî the watchdog auditing its own coverage. Amber **"Unregistered loop: X"** tiles for every cron `createFunction` in the serve route with no monitored-loop tile (the CODE‚ÜîREGISTRY diff ‚Äî surfaces a coverage gap the moment a new cron lands without a registry row, instead of waiting for someone to notice), plus **"Not registered with Inngest: X"** tiles for serve-route fns Inngest Cloud has no record of (the CODE‚ÜîINNGEST diff ‚Äî the deploy-didn't-re-sync gap). Empty = full coverage (green "every cron in code has a tile"). The in-code‚ÜîInngest diff is best-effort and shows an "unverified" note without an `INNGEST_SIGNING_KEY`; the never-fired cron check covers the same gap from the heartbeat side regardless. These are amber (folded into the summary count), **not** a page ‚Äî the paging signal for the same gap is the cron tile's **red `never_fired`** ("registered but has never run").
- **Coverage registration section** ([[../specs/coverage-auto-register-agent]], [[../libraries/coverage-register-agent]]) ‚Äî the [[../libraries/coverage-register-agent|coverage-register agent]]'s surfaced proposals: each unregistered loop becomes an **unregistered loop ‚Üí proposed `MONITORED_LOOPS` entry** card showing the inferred owner-function + cadence/window, with **[Register]** (lands the entry ‚Üí a monitored tile), **[Intentionally-unmonitored]** (adds the `INTENTIONALLY_UNMONITORED_CRONS` exemption), and **[Dismiss]** (`POST /api/developer/control-tower/coverage-register` ‚Üí `queued_resume` ‚Üí the box materializes the chosen registry fix spec to main + queues its `build` ‚Äî the owner-gate; the registry is never silently edited). Rendered just below the Coverage self-audit; hidden when there's no proposal waiting (the clean state lives in the audit section). A `coverage-register` [[../tables/agent_jobs]] in `needs_approval` drives it. **The register-vs-exempt choice is multi-way (not a binary approve/decline), so it stays the canonical decision surface** ([[../specs/approval-routing-engine]] Phase 4): the routed [[agents|Agents inbox]] surfaces the request and **deep-links here** (the inbox never guesses a register/exempt decision).
- **Repair feed section** ([[../specs/repair-agent]], [[../libraries/repair-agent]]) ‚Äî the [[../libraries/repair-agent|Repair Agent]]'s surfaced triage items: each is **error ‚Üí proposed fix [[spec]]** (mirrors escalation-triage ‚Äî the box proposes, the owner finalizes). **Since [[../specs/approval-routing-engine]] Phase 4 this is a read-only VIEW onto the routed inbox:** a `needs_approval` repair [[../tables/agent_jobs]] shows the authored fix spec with a **Review in the Agents inbox ‚Üí** deep-link ([[agents]]) ‚Äî the **Approve / Decline** decision (and its `POST /api/roadmap/approve` ‚Üí `queued_resume` ‚Üí the box queues the feature `build` + resolves the originating [[../tables/error_events]] row) is made there, the single source. A `needs_attention` repair job (no proposed fix ‚Äî not an approval) keeps its **needs human** note + a **Dismiss** here (`POST /api/developer/control-tower/repair` `dismiss`, resolves the error row). Auto-queued (allow-listed mechanical) + transient-resolved repairs complete silently and never appear here. Empty = nothing waiting on you. **Director-dismissed items stay visible** ([[../specs/director-supervised-repair-dismissal]] Phase 2): when the [[../libraries/platform-director|Platform/DevOps Director]] (Ada) clears one of Rafa's no-fix calls (after adversarially re-checking it), the completed job is surfaced below the open items as `üõÝÔ∏è Dismissed by Ada ‚Äî <her reasoning>` (`getDirectorDismissedRepairs`, the recent 14-day dismissals minus any already re-opened) with a one-tap **Re-open** (`POST /api/developer/control-tower/repair` `reopen`) ‚Äî the CEO's override over the supervisor: it re-opens the [[../tables/error_events]] row + re-enqueues Rafa for a fresh triage, then drops off the list.
- **DB Health section** ([[../specs/db-health-agent]], [[../libraries/db-health]]) ‚Äî the [[../libraries/db-health|DB Health Agent]]'s read-only Postgres watch: **top tables by size/growth** (latest [[../tables/db_table_size_history]] snapshot) + **slowest queries** root-caused via `EXPLAIN` (cause + impact, from the slow-query pass beat) + **proposed fixes** ‚Äî each a **finding ‚Üí proposed fix [[spec]]** card (mirrors the Repair feed). **Since [[../specs/approval-routing-engine]] Phase 4 a read-only VIEW onto the routed inbox:** a `needs_approval` `db_health` [[../tables/agent_jobs]] with a pre-authored fix spec shows a **Review in the Agents inbox ‚Üí** deep-link ([[agents]]) ‚Äî the Approve decision (`POST /api/roadmap/approve` ‚Üí `queued_resume` ‚Üí the box commits the spec to `main` + queues the `build`; **zero** DDL/deletes without it) is made there. A finding with no proposed spec keeps a **Dismiss** here. The two passes (`db-health-slow-query` ~hourly ¬∑ `db-health-size-sweep` daily) are their own green/red monitored-loop tiles above; a dead agent shows red. Empty proposals = nothing over threshold or already actioned.
- **Spec drift section** ([[../specs/spec-drift-agent]]) ‚Äî phases whose code is verifiably on `main` but whose emoji is still ‚è≥/üöß with **no merged build on record**: the residue the [[../libraries/spec-drift|Spec-Drift Agent]] won't auto-flip. Each open [[../tables/spec_drift]] row shows the spec + `P{n}` + the drift detail, with a **Mark P{n} ‚úÖ** one-tap flip (rewrites the phase emoji on `main` via `POST /api/roadmap/spec-drift` ‚Üí the shared `flipPhaseToShipped` writer, then resolves the row + enqueues a spec-test if now shipped) and a **Dismiss** (resolve the row, leave the markdown). Empty = no unresolved drift. (The agent auto-flips the confident cases ‚Äî merged build + code on main ‚Äî without surfacing them; this section is only the ambiguous residue.)
- Polls `GET /api/developer/control-tower` every ~15s. Owner-gated (re-checks `workspace_members.role='owner'`; non-owners see an owner-only notice).

## Data source

- `GET /api/developer/control-tower` (`src/app/api/developer/control-tower/route.ts`) ‚Üí `buildControlTowerSnapshot()` + `buildErrorFeedSnapshot()` (merged as `errorFeed`) + `getControlTowerDbPanels()` (the six raw-SELECT panels ‚Äî `dbHealth` / `repairs` / `directorDismissed` / `coverageRegister` / `specDrift` / `claudeHealth`, [[../libraries/control-tower]]). **Read-only** ‚Äî the dashboard never opens/resolves alerts or pages; that's the [[../inngest/control-tower-monitor]] cron's job (loops) + the feeders' `recordError` (errors). Same snapshot the monitor evaluates, so the screen and the alerting agree.
  - **cut-internal-egress-pooler-and-spec-rpcs Phase 3:** `getControlTowerDbPanels` (src/lib/control-tower/snapshot.ts) calls the new **`public.control_tower_snapshot(uuid)`** RPC (migration `20261005120000_control_tower_snapshot_rpc.sql`) that consolidates the six per-panel SELECTs (top-tables + two loop_heartbeats beats + open db_health / repair / director-dismissed / coverage-register / spec_drift rows + the claude_health singleton) into ONE round trip returning a jsonb payload. Retires ~10-15 PostgREST calls per ~15s tick (each carrying a set_config preamble). The heavier `buildControlTowerSnapshot` (per-loop iteration + rollups) + `buildErrorFeedSnapshot` (per-source dedup) stay as their own concurrent reads ‚Äî their fan-outs need their own derivation. The six original helpers (`getOpenRepairs` / `getDirectorDismissedRepairs` / `getOpenCoverageRegistrations` / `getOpenSpecDrift` / `getDbHealthPanel` / `getClaudeHealth`) stay unchanged for their other callers (platform-director, the Claude poll cron, etc.). **Poll effects on the Control Tower + Approvals + Regressions + spec-tests human-queue + roadmap/box pages are visibility-gated:** an effect skips its tick while `document.visibilityState !== 'visible'` and refreshes on `visibilitychange` ‚Üí visible, mirroring the shipped sidebar reduce-calls pattern in `src/app/dashboard/sidebar.tsx:347` ‚Äî a backgrounded tab issues no poll requests until the operator returns.
- Reads [[../tables/worker_heartbeats]] (box), [[../tables/loop_heartbeats]] (crons + agent kinds + inline agents), [[../tables/loop_alerts]] (open incidents), [[../tables/agent_jobs]] (stuck detection + Phase 2 enqueue checks), plus [[../tables/tickets]] + [[../tables/subscriptions]] for the Phase 2 output assertions, [[../tables/tickets]] + [[../tables/journey_sessions]] + [[../tables/orders]] + [[../tables/ticket_messages]] for the inline-agent work-exists probes, [[../tables/error_events]] for the Errors section, [[../tables/agent_jobs]] (`kind='repair'`, `needs_approval`/`needs_attention`) for the Repair feed, and [[../tables/spec_drift]] for the Spec drift section, plus [[../tables/db_table_size_history]] (latest size snapshot) + [[../tables/agent_jobs]] (`kind='db_health'`, `needs_approval`/`needs_attention`) for the DB Health section.

> **Stuck-jobs false-positive eliminated at the source.** The "N jobs stuck in {building|queued} past 60m" check used to fire false alarms on jobs affected by worker restarts, mis-attributing backlog to worker fault. Two fixes: (1) **Orphaned building jobs:** in-flight rows the dead instance left in `building` that the new worker never re-claimed. The box worker's [[../recipes/build-box-setup|startup orphan-reaper]] (`reapOrphans()`) now clears those on boot ‚Äî re-runnable kinds back to `queued`, work-producers to `failed` ‚Äî so a restart no longer leaves stuck-in-`building` rows for this check to flag. (2) **Queued-job clock reset:** jobs enqueued during a worker-down window are counted as "stuck" if the check uses their `created_at`‚Äîbut the worker couldn't claim them until it came back online. `jobStuckSince()` ([[../libraries/control-tower]]) now clamps the queued-job floor to `max(updated_at, worker_heartbeats.started_at)`, so a freshly-restarted lane gets a fair drain window before the same backlog is re-raised as red. (Incident: 8 spec-test rows enqueued at 10:45, box offline until 11:45, monitor at 12:00 incorrectly aged them 75m even though the queue had only existed for 15m of worker uptime ‚Äî now stays green as the lane drains.)

Owner-only ‚Äî both the page (client `role` guard) and the API (`workspace_members.role='owner'`, 403 otherwise).
