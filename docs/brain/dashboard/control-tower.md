# dashboard/control-tower

The single "is the machine healthy?" screen ([[../specs/control-tower]] Phase 1 + Phase 2; [[../specs/control-tower-agent-coverage]] adds the inline AI agents; [[../specs/control-tower-complete-coverage]] P1 adds full cron coverage + the reactive event-driven agents, P3 adds the department rollups) — a green/amber/red tile per autonomous loop, rolled up by org function. The objective-owner's window from the north star: where the CEO sees whether every proxy-optimizing loop is still alive and doing its job.

**Route:** `/dashboard/developer/control-tower` (client poller, owner-only)
**Sidebar:** **Developer** section (owner-only) → **Control Tower** (right under [[roadmap|Build box]]).

## Surfaces

- **Summary bar** — counts of healthy / warning / alerting tiles + "updated Ns ago". Includes the **error panels** (the API folds their colors into the count, [[../specs/error-feed-honest-panels]]) so an unconfigured/amber panel is never counted as healthy.
- **Department health (CEO-glance rollups)** ([[../specs/control-tower-complete-coverage]] P3) — the view **leads** with one rollup health tile per org function (**Platform / Growth / Retention / CS / CMO Health**, [[../functions]] · [[../goals/ceo-mode]]). Every loop carries an `owner` function in [[../libraries/control-tower|the registry]]; each tile is the **worst-of** color across that function's loops (a single red loop turns its department amber/red) with a **healthy/total** count + **open-alert** count — the CEO glance "which org function is healthy?". Below the rollups, each department is a **collapsible drill-in** (`<details>`, auto-open when not green) listing its loops sub-grouped by kind. A loop's owner = the function that owns the system it monitors (renewal/dunning/portal → retention; the meta + social + creative loops → growth/cmo; ticket-handler/triage → cs; the build/spec-test/migration-fix agents + the data-infra/monitoring crons → platform). `buildControlTowerSnapshot` returns `departments` (worst-of rollups from `OWNER_FUNCTIONS`) alongside `loops`.
- **Tiles, grouped by kind** (Worker · Crons · **Reactive agents** · Agent lanes · **Inline AI agents**), shown inside each department drill-in — each tile shows the loop label + description, its expected cadence, a colored status dot + `statusText`, **last ran**, **last produced** (compacted from the heartbeat's `produced` jsonb), any **open alert** (with how long it's been open + the violation detail), and a **history strip** (last ~10 runs, green/red ticks). The Crons group now covers the **full Inngest cron set** (every `inngest.createFunction` cron), and **Reactive agents** are the event-driven Inngest functions — `unified-ticket-handler` (THE inbound ticket pipeline), `dunning-payment-failed`, `returns-process-delivery`, `journey-session-completed`, `agent-todo-execute`, `chargeback-received` — evaluated like inline agents (idle = green; red on liveness-while-work / error-rate). ([[../specs/control-tower-complete-coverage]] P1.)
  - **green** — healthy, or genuinely idle (no work to do). **amber** — warning (cron genuinely awaiting its first run, worker mid self-update, a not-ok cron beat). **red** — an active violation that has paged the owners (a P1 liveness/freshness/stuck silence, the [[../specs/control-tower-complete-coverage]] **never-fired** check — a registered cron with 0 beats whose deploy is older than its cadence+grace, "registered but has never run", distinct from a genuinely-fresh cron's amber "awaiting first run" — an **inline-agent** liveness-while-work / error-rate failure ([[../specs/control-tower-agent-coverage]]), **or** a Phase 2 output-assertion failure — escalation idle-while-work, spec-test false-success, renewal integrity — where the loop ran but silently did nothing/wrong).
  - **Inline AI agents** (`ai:ticket-analyzer`, `ai:journey-delivery`, `ai:fraud-detector`, `ai:orchestrator`) — event-driven, no cadence: green when healthy or genuinely idle; red on **liveness-when-work-exists** (upstream work waited but 0 successful runs in the window) or **error-rate** (>50% of ≥5 runs errored). `statusText` carries the window counts ("healthy · N ok", "silent while N awaits", "failing: E/T errored"). `ai:orchestrator` ([[../specs/control-tower-agent-coverage]] Phase 2) is the per-ticket decision agent (`callSonnetOrchestratorV2`): its work-exists probe is inbound customer messages in the 2h window, and a run beats ok:false when it threw or returned a degraded/fallback decision — so a model that errors / parse-fails on every ticket trips error-rate even though it "ran".
- **Errors section** ([[../specs/error-feed-monitoring]] Phase 1 + Phase 2; [[../specs/client-error-capture]]) — five panels (**Vercel errors · Inngest failures · Supabase errors (app-layer) · Supabase errors (DB logs) · Client errors**) of grouped error incidents from the hidden surfaces. The fourth (Phase 2) is DB-level Supabase errors our app never saw — Postgres/auth/API — polled from the [[../integrations/supabase-management-logs]] API by [[../inngest/supabase-log-poll]] (`source='supabase-logs'`; green until the owner pastes the access token). The fifth is **client-side JS errors** on the storefront (PDP/customize/checkout/thank-you) + both portals — browser crashes our server logs can't see, POSTed to `/api/client-errors` ([[../libraries/client-error-reporter]], `source='client'`; amber "awaiting first event" until the first report/heartbeat lands). Each panel is **connection-aware** ([[../specs/error-feed-honest-panels]]): green only when the feed is **configured + receiving deliveries + 0 errors** ("connected · 0 errors, last delivery Ns ago"); **amber "not configured"** (+ a how-to-wire hint) when its secret/token is unset, or **amber "awaiting first event"** when configured but no delivery seen yet — never a misleading green "0 errors" while disconnected. With errors it shows the recency color (red ≤1 h, amber ≤24 h), the active-signature + total-occurrence counts, and the recent incidents (title, `×count`, last-seen). Liveness comes from `feed:<source>` beats in [[../tables/loop_heartbeats]] (Vercel drain POST / Supabase-logs poll); Inngest is proxied by the freshness of Inngest itself (latest cron beat). A new signature or a re-firing spike pages the owners (rate-limited: a burst of the same error = one page).
- **Coverage self-audit section** ([[../specs/control-tower-complete-coverage]] Phase 2, [[../libraries/control-tower-self-audit]]) — the watchdog auditing its own coverage. Amber **"Unregistered loop: X"** tiles for every cron `createFunction` in the serve route with no monitored-loop tile (the CODE↔REGISTRY diff — surfaces a coverage gap the moment a new cron lands without a registry row, instead of waiting for someone to notice), plus **"Not registered with Inngest: X"** tiles for serve-route fns Inngest Cloud has no record of (the CODE↔INNGEST diff — the deploy-didn't-re-sync gap). Empty = full coverage (green "every cron in code has a tile"). The in-code↔Inngest diff is best-effort and shows an "unverified" note without an `INNGEST_SIGNING_KEY`; the never-fired cron check covers the same gap from the heartbeat side regardless. These are amber (folded into the summary count), **not** a page — the paging signal for the same gap is the cron tile's **red `never_fired`** ("registered but has never run").
- **Coverage registration section** ([[../specs/coverage-auto-register-agent]], [[../libraries/coverage-register-agent]]) — the [[../libraries/coverage-register-agent|coverage-register agent]]'s surfaced proposals: each unregistered loop becomes an **unregistered loop → proposed `MONITORED_LOOPS` entry** card showing the inferred owner-function + cadence/window. **Phase 4 (approval-routing-engine):** this is now **monitoring only** — the **Register / Intentionally-unmonitored / Dismiss** decision moved to the routed [[agents|inbox]] (the section shows a **Decide in inbox →** link; the inbox's rich-approval modal still calls `POST /api/developer/control-tower/coverage-register` → `queued_resume` → the box materializes the chosen registry fix spec + queues its `build` — the owner-gate; the registry is never silently edited). Rendered just below the Coverage self-audit; hidden when there's no proposal waiting. A `coverage-register` [[../tables/agent_jobs]] in `needs_approval` drives it.
- **Repair feed section** ([[../specs/repair-agent]], [[../libraries/repair-agent]]) — the [[../libraries/repair-agent|Repair Agent]]'s surfaced triage items: each is **error → proposed fix [[spec]]** (a `needs_approval` repair shows the authored fix spec; a `needs_attention` repair shows a **needs human** note). **Phase 4 (approval-routing-engine):** **monitoring only** — the **Build / Dismiss** decision moved to the routed [[agents|inbox]] (a **Decide in inbox →** link; the inbox modal still calls `POST /api/developer/control-tower/repair` → the box queues the feature `build` job + **Dismiss** resolves the originating [[../tables/error_events]] row — the owner-gate the North star keeps). Auto-queued (allow-listed mechanical) + transient-resolved repairs complete silently and never appear here. Empty = nothing waiting on you.
- **DB Health section** ([[../specs/db-health-agent]], [[../libraries/db-health]]) — the [[../libraries/db-health|DB Health Agent]]'s read-only Postgres watch: **top tables by size/growth** (latest [[../tables/db_table_size_history]] snapshot) + **slowest queries** root-caused via `EXPLAIN` (cause + impact, from the slow-query pass beat) + **proposed fixes** — each a **finding → proposed fix [[spec]]** card. **Phase 4 (approval-routing-engine):** the size/slow-query panels stay (pure monitoring); the proposed-fix **Build / Dismiss** decision moved to the routed [[agents|inbox]] (a **Decide in inbox →** link; the inbox modal still calls `POST /api/developer/control-tower/db-health` → the box commits the spec to `main` + queues the `build` job — the owner-gate; **zero** DDL/deletes are applied without it). The two passes (`db-health-slow-query` ~hourly · `db-health-size-sweep` daily) are their own green/red monitored-loop tiles above. Empty proposals = nothing over threshold or already actioned.
- **Spec drift section** ([[../specs/spec-drift-agent]]) — phases whose code is verifiably on `main` but whose emoji is still ⏳/🚧 with **no merged build on record**: the residue the [[../libraries/spec-drift|Spec-Drift Agent]] won't auto-flip. Each open [[../tables/spec_drift]] row shows the spec + `P{n}` + the drift detail, with a **Mark P{n} ✅** one-tap flip (rewrites the phase emoji on `main` via `POST /api/roadmap/spec-drift` → the shared `flipPhaseToShipped` writer, then resolves the row + enqueues a spec-test if now shipped) and a **Dismiss** (resolve the row, leave the markdown). Empty = no unresolved drift. (The agent auto-flips the confident cases — merged build + code on main — without surfacing them; this section is only the ambiguous residue.)
- Polls `GET /api/developer/control-tower` every ~15s. Owner-gated (re-checks `workspace_members.role='owner'`; non-owners see an owner-only notice).

## Data source

- `GET /api/developer/control-tower` (`src/app/api/developer/control-tower/route.ts`) → `buildControlTowerSnapshot()` + `buildErrorFeedSnapshot()` (merged as `errorFeed`) + `getOpenSpecDrift()` (merged as `specDrift`, [[../libraries/spec-drift]]) + `getDbHealthPanel()` (merged as `dbHealth`, [[../libraries/db-health]]) ([[../libraries/control-tower]]). **Read-only** — the dashboard never opens/resolves alerts or pages; that's the [[../inngest/control-tower-monitor]] cron's job (loops) + the feeders' `recordError` (errors). Same snapshot the monitor evaluates, so the screen and the alerting agree.
- Reads [[../tables/worker_heartbeats]] (box), [[../tables/loop_heartbeats]] (crons + agent kinds + inline agents), [[../tables/loop_alerts]] (open incidents), [[../tables/agent_jobs]] (stuck detection + Phase 2 enqueue checks), plus [[../tables/tickets]] + [[../tables/subscriptions]] for the Phase 2 output assertions, [[../tables/tickets]] + [[../tables/journey_sessions]] + [[../tables/orders]] + [[../tables/ticket_messages]] for the inline-agent work-exists probes, [[../tables/error_events]] for the Errors section, [[../tables/agent_jobs]] (`kind='repair'`, `needs_approval`/`needs_attention`) for the Repair feed, and [[../tables/spec_drift]] for the Spec drift section, plus [[../tables/db_table_size_history]] (latest size snapshot) + [[../tables/agent_jobs]] (`kind='db_health'`, `needs_approval`/`needs_attention`) for the DB Health section.

> **Stuck-jobs false-positive eliminated at the source.** The "N jobs stuck in building past 60m" check used to fire on jobs *orphaned by a worker restart* — in-flight rows the dead instance left in `building` that the new worker never re-claimed (a real signal that read as a false alarm). The box worker's [[../recipes/build-box-setup|startup orphan-reaper]] (`reapOrphans()`) now clears those on boot — re-runnable kinds back to `queued`, work-producers to `failed` — so a restart no longer leaves stuck-in-`building` rows for this check to flag.

## Permissions

Owner-only — both the page (client `role` guard) and the API (`workspace_members.role='owner'`, 403 otherwise).
