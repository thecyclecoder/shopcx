# ad_avatar_candidates

The saved avatar-FACE library. Every face generated by Higgsfield Soul text-to-image (from the four attributes: gender, age, health level, ethnicity) is persisted here so the operator can reuse a look instead of regenerating it — each "Generate 3 faces" run costs ~3 Soul credits per face, so persistence avoids re-spending. The avatar-creation screen lists the existing library before offering to generate more. Faces can be deleted. A chosen face is minted into a persistent character ([[ad_avatars]]) on Create.

**Primary key:** `id`

## Common queries

### List available faces for a workspace (library)
```ts
const { data } = await admin.from("ad_avatar_candidates")
  .select("id, gender, age_range, health_level, ethnicity, storage_path, status, created_at")
  .eq("workspace_id", workspaceId)
  .neq("status", "discarded")
  .order("created_at", { ascending: false })
  .limit(200);
// re-sign each storage_path before returning to the client
```

### Mark a face used (when minted into a character)
```ts
await admin.from("ad_avatar_candidates")
  .update({ status: "used", used_avatar_id: avatarId })
  .eq("id", candidateId);
```

## Gotchas

- Enum values are **lowercase** (`status`, `health_level`, `ethnicity`).
- **Faces live in the private `ad-tool` bucket.** The API returns short-lived signed URLs, re-signed on each `GET` — never persist the signed URL in the row; only `storage_path` is durable.
- `DELETE` removes both the row and the underlying storage object.
- A unique storage path per face (`{stamp}_{ts}_{i}.png`) means library entries never collide even for identical attribute tuples.
