# agent_jobs

The build queue for the [[../specs/roadmap-build-console]] "do it" button. One row per job. The dashboard inserts a `queued` row; the box worker ([[../recipes/build-box-setup]]) claims it via `claim_agent_job()`, runs `claude -p` on Max, and drives it to a `claude/*` PR. Distinct from [[agent_todos]] (the ticket-driven queue) ŌĆö same shape, different driver.

**Fourteen `kind`s off one queue + RPC:** `'build'` (default ŌĆö build a spec to a PR), `'plan'` (the [[../specs/goal-decomposition-engine|goal-decomposition engine]] ŌĆö run the `plan-goal` skill against a goal ŌåÆ propose a milestoneŌåÆspec tree ŌåÆ on approval, auto-author the specs + queue their builds), `'fold'` (the [[../specs/fold-build-batching|fold-build batcher]] ŌĆö fold **every** [[pending_folds|pending-fold]] spec for the workspace into the brain in **one** PR), `'product-seed'` (the [[../specs/box-product-seeding|box product-seeding]] pipeline ŌĆö drives ONE product `none ŌåÆ published` via a **top-level `claude -p` on Max with web search** running the `seed-product` skill: web-research ŌåÆ reviews ŌåÆ triangulated benefits ŌåÆ content ŌåÆ Nano Banana Pro hero ŌåÆ self-QA ŌåÆ publish, all on Max with **no `ANTHROPIC_API_KEY`** and no PR; `spec_slug` = the `product_id`, `instructions` = JSON `{product_id, angle_override?}`), `'ticket-improve'` (the [[../specs/box-ticket-improve|ticket Improve agent]] ŌĆö ONE turn of a ticket-bound, resumable Max session: a top-level `claude -p` (no `ANTHROPIC_API_KEY`, web search on) that investigates the ticket **read-only** and either replies or proposes an approval-gated action plan parked in [[ticket_improve_chats]]`.pending_plan`; `spec_slug` = the improve-session id, `instructions` = JSON `{ticket_id, session_id, mode:'turn', user_message}`; **own concurrency-1 lane** (`MAX_TICKET_IMPROVE`), **no PR**, never mutates ŌĆö the plan is executed server-side by the Improve route on approval, [[../libraries/improve-plan-executor]]), `'spec-chat'` (the [[../specs/box-spec-chat|box spec-chat]] ŌĆö ONE turn of the roadmap authoring chat as a long-running, resumable **`claude -p` on Max** running the `spec-chat` skill: full working-tree reads + `WebSearch`, accumulated session context; `spec_slug` = the [[roadmap_chats]] thread id, `instructions` = JSON `{mode, chat_id?, slug?, seedSlug?, queueBuild?}` with `mode Ōłł {turn, finalize, verify}`), and `'triage-escalations'` (the [[../specs/box-escalation-triage|box escalation triage]] ŌĆö an **hourly sweep** of routine-owned escalated tickets ([[../inngest/triage-escalations]] enqueues one job per workspace per tick); the worker (`runEscalationTriageJob`) claims it on its **own concurrency-1 lane** (`MAX_TRIAGE=1`) and processes up to `TRIAGE_CAP` (default 5, env `AGENT_TODO_TRIAGE_CAP`) tickets, each through a **solverŌåÆskepticŌåÆquorum** loop run as 2ŌĆō4 separate top-level `claude -p` Max sessions (no `ANTHROPIC_API_KEY`, web search on, keeps DB/crypto secrets). On quorum it materializes via `src/lib/agent-todos/triage.ts` (customer fix ŌåÆ `pending` [[agent_todos]] ┬Ę rule ŌåÆ `proposed` [[sonnet_prompts]] ┬Ę code/analyzer fix ŌåÆ committed `docs/brain/specs/{slug}.md`); every ticket per sweep writes a [[triage_runs]] audit row. It opens **no PR** itself (specs commit straight to main, like the planner)), and `'spec-test'` (the [[../specs/spec-test-agent|box spec-test QA agent]] ŌĆö a **daily** non-destructive QA pass over ONE shipped-but-unverified spec's `## Verification` checklist; the [[../inngest/spec-test-cron]] cron enqueues one job per shipped-but-not-archived spec, the worker (`runSpecTestJob`) claims it on its **own concurrency-1 lane** (`MAX_SPEC_TEST=1`) and runs the `spec-test` skill on Max ŌĆö repo/`tsc`, `gh` CI, `vercel` deploy+logs+env, read-only DB probes, GET endpoints ŌĆö keeping DB/crypto secrets, no `ANTHROPIC_API_KEY`; `spec_slug` = the spec slug. It writes one [[spec_test_runs]] row with an `agent_verdict` stamp; it opens **no PR**, **never mutates prod**, and **never** marks a spec verified/archived), and `'migration-fix'` (the [[../specs/migration-fix-agent|migration-fix agent]] ŌĆö fired by an **event, not a cron**: the moment a [[migration_audits]] row transitions to `failed`, [[../libraries/migration-audit]] `verifyMigration` enqueues one job `{audit_id, subscription_id}` (`spec_slug` = the audit id) via [[../libraries/migration-fix]] `enqueueMigrationFixJob`. The worker (`runMigrationFixJob`) claims it on its **own concurrency-1 lane** (`MAX_MIGRATION_FIX=1`) and runs the `migration-fix` skill on Max (no `ANTHROPIC_API_KEY`, keeps DB/crypto/Appstle secrets) to DIAGNOSE the failing checks read-only and PROPOSE a typed billing repair that parks in `pending_actions` (`type:'migration_fix'`, `fix_kind Ōłł price_reconcile|variant_backfill|appstle_cancel`); the owner approves on [[../dashboard/migrations]] (the same `/api/roadmap/approve` ŌåÆ `queued_resume` gate), the worker executes it via [[../libraries/migration-fix]] `applyMigrationFix` and re-runs `verifyMigration` ŌĆö only a re-`passed` clears the row, unfixable stays `failed` with the box's written diagnosis. **No PR**, never mutates without approval), and `'dev-ask'` (the [[../specs/developer-message-center|Developer Message Center]] ŌĆö ONE turn of a founder-facing, **read-only** "ask the box anything" thread: a long-running, resumable **`claude -p` on Max** that carries the whole brain + full repo + **read-only prod DB AND `WebSearch`** (no `ANTHROPIC_API_KEY`, keeps DB/crypto secrets), run in a per-thread git worktree recreated on `origin/main` each turn. It investigates ("does {feature} work right now?"), pulls ad-hoc DB analytics (via **throwaway, never-committed `scripts/_*.ts`** query scripts), and plans specs ŌĆö all read-only and silent. It is **report-back, never a builder**: every proposed DB write / migration / spec handoff stops at a `pending_actions` approval card the owner must click, and only deterministic worker code (`mode:'approve_action'`) executes it. `spec_slug` = the [[dev_message_threads]] thread id, `instructions` = JSON `{thread_id, mode}` with `mode Ōłł {turn, approve_action}`; **own concurrency-1 lane** (`MAX_DEV_ASK=1`), **no PR**), and `'pr-resolve'` (the [[../specs/dirty-pr-resolver-agent|dirty-PR resolver agent]] ŌĆö fired by an **event, not a cron**: the [[../integrations/github-webhook|GitHub webhook]] (`push` to `main` / `pull_request`) detects a dirty (`CONFLICTING`) `claude/*` build PR and enqueues one job via [[../libraries/github-pr-resolve]] `enqueuePrResolveJob` (`spec_slug` = `pr-{number}`, `spec_branch` = the PR branch, `pr_number` set, `instructions` = JSON `{pr_number, branch, reason}`). The worker (`runPrResolveJob`) claims it on its **own concurrency-1 lane** (`MAX_PR_RESOLVE=1`) and runs a `claude -p` on Max (git available, prod secrets stripped ŌĆö **no prod creds, no `gh`/push**) that **`git merge origin/main`** into the PR branch + resolves the (usually additive) conflicts in a throwaway worktree; the **worker** then runs the **tsc GATE** + merge-correctness checks (origin/main is an ancestor ┬Ę no unmerged paths ┬Ę no conflict markers) and **pushes** ŌĆö never a non-compiling merge. On a heavy parallel-rewrite divergence or a merge that can't compile, it does NOT force a wrong merge: it **rebuilds-on-main** (close the PR + re-queue the originating `build` off a clean main) or, if it can't identify the spec, **surfaces** "PR #N needs a human merge: {why}" to the owner via [[../libraries/notify-ops-alert]] (job ŌåÆ `needs_attention`). **No PR of its own**; idempotent + deduped (one resolve per PR)), and `'repair'` (the [[../specs/repair-agent|Repair Agent]] ŌĆö fired by an **event, not a cron**: the moment the Control Tower records a NEW problem, [[../libraries/repair-agent]] `enqueueRepairJob` inserts one job (`spec_slug` = the [[error_events]] signature or `loop:<id>`, `instructions` = JSON `{source, signature, title, error_event_id?, loop_alert_id?}`). Both [[../libraries/control-tower]] `recordError` (new `error_events` signature) and `runControlTowerMonitor` (a newly-opened [[loop_alerts]]) call it, deduped by signature. The worker (`runRepairJob`) claims it on its own concurrency-limited **`repair` lane** (`MAX_REPAIR`, default 2) and runs a `claude -p` on Max (web search on, no `ANTHROPIC_API_KEY`, keeps read-only DB/crypto secrets) to INVESTIGATE the signature read-only ŌåÆ CLASSIFY (`real-bug`’Į£`monitor-false-positive`’Į£`foreign-app-noise`’Į£`transient`’Į£`needs-human`) ŌåÆ ACT: author a single-phase fix spec to main + **surface** it for one-tap owner Build (`needs_approval` + a `repair_build` `pending_actions` entry), or **auto-queue** the build for an allow-listed mechanical class (`REPAIR_AUTOBUILD_KINDS`), or no-op-**resolve** a transient [[error_events]] row, or **surface needs-human** (`needs_attention`, no spec). It NEVER edits product code / opens a PR / applies a migration ŌĆö the build does that, owner-gated. **No PR of its own**; deduped by signature), and `'regression'` (the [[../specs/regression-agent|Regression Agent]] ŌĆö fired by an **event, not a cron**: the moment the box's spec-test agent records a regression on a shipped spec (a Ō£ģ verification that now fails), `runSpecTestJob` calls [[../libraries/regression-agent]] `enqueueRegressionJob` (`spec_slug` = the regression signature `regression:{slug}:{hash}`, `instructions` = JSON `{signature, spec_slug, title, failing[], run_at}`), deduped by signature (a dismissed break never re-surfaces). The worker (`runRegressionJob`) claims it on its **own concurrency-1 lane** (`MAX_REGRESSION=1`) and runs a `claude -p` on Max (web search on, no `ANTHROPIC_API_KEY`, read-only DB) to REVIEW the regression read-only ŌåÆ either **dismiss** it (`transient’Į£foreign’Į£false-positive’Į£already-fixed`, recorded reasoning, no spec) or **author the fix spec DIRECTLY to main** (no "propose" step) + route via [[../libraries/approval-router]] `resolveApproverLive('platform')` ŌĆö a live+autonomous director auto-queues the build within its leash, else surface for the CEO inbox (`needs_approval` + a `regression_build` action). Loop-guard: Ōēź2 authored attempts for one spec ŌåÆ escalate to CEO (`needs_attention`). Every detect/dismiss/author/escalate writes a [[director_activity]] row. **No PR of its own**), and `'storefront-optimizer'` (the [[../specs/storefront-optimizer-agent|Storefront Optimizer agent]], M4 ŌĆö enqueued by a **daily cron** ([[../inngest/storefront-optimizer]] `enqueueDueCampaigns`), one job per DUE (product ├Ś lander-type ├Ś audience). The worker (`runStorefrontOptimizerJob`) claims it on its **own concurrency-1 lane** (`MAX_STOREFRONT_OPTIMIZER=1`) and runs a `claude -p` on Max (no `ANTHROPIC_API_KEY`, read-only DB) that reads the funnel + [[../libraries/storefront-lever-memory|lever map]] + [[../libraries/storefront-ltv-metrics|LTV proxy]] and PROPOSES ONE atomic reversible-lever hypothesis. The worker stands up the [[storefront_experiments|M1]] experiment vs holdout via [[../libraries/storefront-optimizer-agent]] `materializeCampaign` ŌĆö immediately when the [[storefront_optimizer_policy|policy]] auto-runs reversible levers, else surfaced as a `storefront_campaign` `pending_actions` card for one-tap owner Approve. A missing capability (offer / structural / new component) ŌåÆ author a scoped spec to main + surface a `storefront_build` Build card (never faked, never auto-built). `spec_slug` = the surface key `product_id:lander_type:audience` (Ōēż1 active campaign per surface). **No PR** except the missing-capability spec). The worker branches on `kind` (`runJob` / `runPlanJob` / `runFoldJob` / `runProductSeedJob` / `runTicketImproveJob` / `runSpecChatJob` / `runEscalationTriageJob` / `runSpecTestJob` / `runMigrationFixJob` / `runDeveloperMessageJob` / `runPrResolveJob` / `runRepairJob` / `runRegressionJob` / `runStorefrontOptimizerJob`). For a plan job, `spec_slug` holds the **goal** slug; for a fold job it's the `'fold-batch'` sentinel (the real spec list lives in [[pending_folds]]); for a ticket-improve job it's the improve-session id; for a spec-chat job it's the chat thread id (or `verify:{slug}`); for a triage-escalations job it's enqueued per-workspace by the cron.

**Primary key:** `id`

## `status` enum

`queued` ŌåÆ `building` (claimed) ŌåÆ `completed` (PR open) ┬Ę or pauses at ŌåÆ `needs_input` (product questions) / `needs_approval` (gated prod actions in `pending_actions`) ŌåÆ `queued_resume` (owner answered/approved; worker executes approved actions then `--resume`s) ŌåÆ `building` ŌĆ” ┬Ę or is dismissed by a director ŌåÆ `dismissed` ([[../specs/director-dismiss-park-and-short-circuit-spec]] Phase 1 ŌĆö a `dismiss-park` action from the spec's owning director flips a `needs_attention` row to `dismissed` + stamps `needs_attention_class='dismissed_by_director'` + writes a `dismissed_park` [[director_activity]] row. **Terminal but reversible** ŌĆö the CEO clicks Re-open from the activity feed (`POST /api/developer/director-activity/reopen-park`) and the row flips back to `needs_attention` for normal handling) ┬Ę or parks at ŌåÆ `blocked_on_usage` ([[../specs/box-multi-account-failover]] Phase 1 ŌĆö **every** Max account is at its 5-hour usage wall; **non-terminal**, the worker's `requeueBlockedOnUsage()` flips it back to `queued`/`queued_resume` once an account resets ŌĆö no manual rebuild) ┬Ę or is parked at ŌåÆ `held` ([[../specs/goal-milestone-build-sequencing]] Phase 3 ŌĆö the director's **sequence reconcile** ([[../libraries/platform-director]] `holdPrematureBuild`) CANCELS an out-of-order milestone build that fanned out before its prerequisites; `held` is **terminal for this job** ŌĆö it leaves needs_input/needs_approval and is never re-claimed (`claim_agent_job` only takes `queued`/`queued_resume`), and `specBuildState` ignores it so the goal escort re-releases the spec FRESH once its newly-applied `**Blocked-by:**` clears) ┬Ę terminal failures: `failed`, `needs_attention` (pushed but PR failed). Active = `queued|claimed|building|needs_input|needs_approval|queued_resume|blocked_on_usage` (one active build per spec). Status is free text (no CHECK), so `blocked_on_usage` + `held` needed no migration.

**`needs_input` MUST carry a renderable question (no empty parks).** A `needs_input` park is an *answerable* pause: the owner inbox / build console renders the open `questions` (and migration-fix renders them on [[../dashboard/migrations]]), so a park with an EMPTY `questions` (and empty `pending_actions`) is an unanswerable, lane-blocking **orphan** ŌĆö `log_tail="(awaiting owner answer)"` to a question that was never recorded, sitting forever blocking its single-flight lane with nothing any UI can surface. Observed live on a `migration-fix` job (`pending_actions=[]`, empty `questions`) that sat 9h blocking the concurrency-1 migration-fix lane on a non-broken audit ŌĆö because [[../libraries/migration-fix]] `enqueueMigrationFixJob` dedups on the active-status set (which **includes `needs_input`**), so the orphan blocked every future fix job for that audit. **Guard:** the worker's single `update()` chokepoint (`scripts/builder-worker.ts`) rejects any write that sets `status='needs_input'` with no `questions` AND no `pending_actions` ŌĆö it repairs the park to terminal `needs_attention` with a clear `error` so the lane frees and the outcome is visible, instead of a silent orphan. The migration-fix path (`runMigrationFixJob`) also guards at the call site: an agent that returns `needs_input` with empty `questions` is terminated `needs_attention` (the lane frees ŌåÆ the next integrity sweep re-enqueues a fresh fix job) rather than parked. A LEGITIMATE `needs_input` (a real question recorded) parks + surfaces normally. Same class as the spec-review "park-on-empty" fix.

**Orphan-reaping on worker restart.** A restart (self-update / deploy / crash) leaves the old instance's in-flight rows (`building`/`claimed`/`queued_resume`) stranded ŌĆö the new worker only claims `queued`, so they'd sit in `building` forever (tripping the [[../dashboard/control-tower]] "stuck in building" alert). On boot, `reapOrphans()` ([[../recipes/build-box-setup]] ┬¦ Startup orphan-reaper) reaps every job `claimed_at < WORKER_STARTED_AT` by kind: re-runnable kinds (`spec-test`/`triage-escalations`/`migration-fix`/`dev-ask`/`pr-resolve`) ŌåÆ back to `queued`; work-producers (`build`/`plan`/`fold`/`product-seed`/`spec-chat`/`ticket-improve`) ŌåÆ `failed` `error='orphaned by worker restart'` (never blindly re-run ŌĆö could double-push; recover a pushed branch via [[build-recover-pr-create]]). Cutoff-gated so a job the *live* instance owns is never killed.

**Stale-session reaping (mid-run death).** The startup reaper above only fires on a clean RESTART ŌĆö it misses a session that dies *while the worker keeps running*: a Max usage cap, a session crash/disconnect, or a box crash-loop that never reaches `reapOrphans`. That leaves a permanent `building` **zombie** (observed live: a `plan` job for goal `growth-director` sat `building` ~5h, `updated_at` frozen at `claimed_at`) ŌĆö the lane is held and the work stalls indefinitely. The fix ([[../specs/stale-session-reaper]]) is a **heartbeat + an in-loop reaper**, both in `scripts/builder-worker.ts`, **lane-agnostic** (covers every session-bearing kind ŌĆö build/plan/spec-test/director/ŌĆ” ŌĆö because they all run through `runBoxSession`):
- **Heartbeat (M = 2 min):** `runBoxSession` bumps `last_heartbeat_at` on each stream-json line, throttled to once per M; `launch` stamps it at claim. A live session is at most M stale; this is how "alive but long" is distinguished from "dead".
- **Reaper (N = 20 min):** `reapStaleSessions()` runs ~every minute in the poll loop (before the claim block, so a freed lane re-fills the same tick). It finds in-flight rows (`building`/`claimed`/`queued_resume`) whose `last_heartbeat_at` (fallback `updated_at` for a pre-migration row) is stale > N **and** that this process is **not** actively running (exempted via the in-memory `active` Map ŌĆö a live session is never killed). N >> M + the 10-min idle-kill window, so a long-but-alive build is safe. A reaped job is re-queued (`status='queued'`, `claimed_at`/`claude_session_id`/`claude_session_config_dir`/`log_tail` cleared) so it re-runs FRESH on whatever account the dispatcher picks next, and `reap_count` is incremented.
- **Retry cap (K = 3):** a job with `reap_count Ōēź K` is **escalated to `needs_attention`** (a CEO-visible park) instead of re-queued, so a structurally-doomed job can't infinite-loop. Every reap/escalate is logged (`[reaper] REAP ŌĆ”` / `[reaper] ESCALATE ŌĆ”`) ŌĆö never silent.

## `claim_agent_job(p_kinds text[] default null)`

`returns public.agent_jobs`. Grabs the oldest `queued`/`queued_resume` row `FOR UPDATE SKIP LOCKED`, flips it to `building`, returns it ŌĆö atomic claim safe for concurrent workers. `p_kinds` filters by kind (NULL = any). **build-gate cooldown** (build-gate-durable-review-signal): the claim also skips any queued job whose `claimed_at` is **in the future** (`claimed_at is null or claimed_at <= now()`) ŌĆö a hold-until back-off. When the claim-time build gate HOLDS a build (re-queue ŌĆö Vale hasn't passed it / a blocker isn't shipped), the worker re-queues it with `claimed_at` stamped `BUILD_GATE_HOLD_COOLDOWN_MS` (~90s) into the future instead of clearing it, so the held build backs off rather than being re-picked + re-evaluated on EVERY ~5s poll tick (the churn the gate author flagged). A normal queued job (`claimed_at=null`) is unaffected; the escort/sequence-reconciler still re-release on their slower cadence. The worker runs **per-kind concurrency** ([[../specs/fold-build-batching]] Phase 2): it claims `['build','plan']` into a 5-lane pool, `['fold','goal-fold']` into a **concurrency-1** lane (so a fold never races a feature build on the now-generated index files ŌĆö `goal-fold` joins this lane because folding a goal is doc-only + touches the brain index/cross-link pages exactly like a spec fold; [[../specs/goal-fold-from-db-row]]), `['product-seed']` into its own **2-lane** pool (the `seed-product` skill ŌĆö drives DB/storage via deterministic tools, not a code build / PR; [[../specs/box-product-seeding]]), `['ticket-improve']` into its own **concurrency-1** interactive lane (one resumable Max turn per ticket, [[../specs/box-ticket-improve]] ŌĆö reads brain/`src/` in the main checkout read-only, no worktree/PR), `['spec-chat']` into its own **concurrency-1** lane ([[../specs/box-spec-chat]] ŌĆö an interactive authoring-chat turn must not occupy a feature-build lane, nor block on one; commits the spec straight to main on finalize, no PR), `['triage-escalations']` into its own **concurrency-1** hourly-sweep lane (`MAX_TRIAGE=1`, [[../specs/box-escalation-triage]] ŌĆö solverŌåÆskepticŌåÆquorum over the escalated queue, materializes on agreement, no PR), and `['spec-test']` into its own **concurrency-1** daily-QA lane (`MAX_SPEC_TEST=1`, [[../specs/spec-test-agent]] ŌĆö non-destructive verification checks over one shipped spec, writes a [[spec_test_runs]] row, no PR, never mutates), and `['migration-fix']` into its own **concurrency-1** event-fired lane (`MAX_MIGRATION_FIX=1`, [[../specs/migration-fix-agent]] ŌĆö diagnose a failed migration read-only + propose a gated billing repair, no PR, mutates only via the approval gate), and `['dev-ask']` into its own **concurrency-1** interactive lane (`MAX_DEV_ASK=1`, [[../specs/developer-message-center]] ŌĆö a read-only "ask the box anything" turn with read-only DB + `WebSearch`; serialized so a turn never races the per-thread worktree's `git reset --hard origin/main`, and its own pool so it never starves the 5 build/plan lanes), and `['pr-resolve']` into its own **concurrency-1** event-fired lane (`MAX_PR_RESOLVE=1`, [[../specs/dirty-pr-resolver-agent]] ŌĆö merge `origin/main` into a dirty `claude/*` PR + resolve conflicts in a throwaway worktree; the worker runs the tsc GATE + pushes, or rebuilds-on-main / surfaces to the owner; serialized so one resolve owns a shared branch at a time), and `['repair']` into its own **`MAX_REPAIR` (default 2) event-fired lane** ([[../specs/repair-agent]] ŌĆö diagnose a NEW Control Tower problem read-only + author/surface a fix spec; a few lanes so N errors at once drain a few at a time without a pileup, no PR, mutates only via the owner-gated Build). Each non-build lane is a **top-level `claude -p` on Max** (no `ANTHROPIC_API_KEY`). Replaced the old zero-arg overload (dropped first ŌĆö a defaulted overload would be ambiguous on a no-arg call).

## `enqueue_fold(p_workspace, p_slug, p_user)`

`returns public.agent_jobs`. The atomic coalesce behind "Mark verified & archive": marks the spec [[pending_folds|pending-fold]] and ensures **exactly one** `queued` `kind='fold'` job for the workspace exists (reuses it if present ŌĆö the spec joins that batch ŌĆö else inserts one), returning it. Serialized per-workspace by `pg_advisory_xact_lock` so two simultaneous verifies can't open a double fold job; a partial unique index `agent_jobs_one_queued_fold_idx (workspace_id) where kind='fold' and status='queued'` is the belt-and-suspenders. Specs verified while a fold is mid-flight ride the **next** queued job, never the in-flight snapshot.

## Plan jobs (goal-decomposition engine)

A `kind='plan'` job runs the `plan-goal` skill in `runPlanJob`:
- **First run** ŌĆö propose a milestoneŌåÆspec tree. The planner writes nothing; its `needs_approval` output becomes `pending_actions` (one `type:'spec'` per branch, each carrying its `spec` payload). No PR. Proposals missing owner/parent are dropped (no-orphan rule). Each branch also declares its **prerequisites** ŌĆö a `blocked_by: [<slug>, ŌĆ”]` list naming the sibling proposed specs (or already-existing specs) it depends on (e.g. the agent spec is `blocked_by` the framework + memory + metric specs); acyclic, each slug must reference another proposed branch or a real spec. `blocked_by` (malformed entries dropped) rides on the `ProposedSpec` interface + the pending-action `spec` object. This makes a goal plan **self-sequencing** ŌĆö without it the engine's first real run ([[../goals/storefront-optimizer]]) proposed a correct tree but **no dependencies**, so approving all branches fanned out every build at once and a dependent (the agent) could build before its framework existed.
- **Resume** (owner approved/declined via `/api/roadmap/approve` ŌåÆ `queued_resume`) ŌĆö author the **approved** specs to `docs/brain/specs/`, wikilink them into the goal doc's `## Decomposition`, commit those docs **straight to main** via the GitHub Contents API (the `chat/finalize` pattern ŌĆö so the builds queued next find their specs on `origin/main`), record declines as ŌØī, then insert a `kind='build'` row per approved spec. When a spec declares blockers, the authoring step writes the **`**Blocked-by:** [[slug]], [[slug]]`** header line (the exact format [[../libraries/brain-roadmap]] `parseSpec` parses) right after the Owner/Parent line ŌĆö but from an **approved-blockers-only** list (any blocker slug the owner declined is dropped, so a spec is never permanently blocked by a branch the owner rejected). Enforcement is unchanged: `queueRoadmapBuild` ([[../libraries/roadmap-actions]]) refuses an uncleared blocker, so only the **unblocked** foundation specs queue a build immediately; the dependents **auto-queue as their blockers ship** (`autoQueueUnblockedBy`, [[../specs/spec-blockers]] Phase 2) ŌĆö approving the whole tree builds it in dependency order with no manual waving. One terminal `completed`; no planning PR (builds open their own `claude/*` PRs). See [[../specs/goal-decomposition-engine]].

## Gotchas

- **One active build per spec / one active plan per goal** ŌĆö `POST /api/roadmap/build` and `/api/roadmap/plan` refuse a new job if an active one (`queued|claimed|building|needs_input|needs_approval|queued_resume`) exists for that slug.
- **Status changes don't move the card live** on the board until reload; `BuildButton` polls `/api/roadmap/build?slug=` while active.
- A build that pauses (`needs_input`) keeps `claude_session_id`; answering (`/api/roadmap/answer`) sets `queued_resume` and the worker resumes that exact session.
- **Un-draft on completion is hardened** ([[../specs/build-lifecycle-hardening]]): a paused build opens its PR as a **draft**; on the resumeŌåÆ`completed` path the worker's `markReady` (GraphQL `markPullRequestReadyForReview`) fetches the PR `node_id` fresh **after the final push**, confirms the mutation returned non-draft, **logs + retries once** on failure (no longer swallows), and the completion path does a final `isDraft` re-check before flipping to `completed`. Fixes PR#75/#76 staying draft.
- **No re-request of an already-applied gated action** ([[../specs/build-lifecycle-hardening]]): the resume prompt lists `Gated actions executed: ŌĆ”` **and** `Already-applied gated actions ŌĆ” do NOT re-request`, so the build treats them as settled. Worker backstop ŌĆö if a resumed build re-emits a `needs_approval` whose `cmd` matches one already `done` on the job, it auto-marks it settled (carried in `pending_actions` with an `autoSettled` count) instead of re-pausing the owner; if that leaves no genuinely-new action it sets `queued_resume` and the build finishes. A repeat past `AUTO_SETTLE_MAX` (2) trips a loop guard ŌåÆ `needs_attention`. `build-spec`/`write-migration` also `probe-db` before requesting a migration and skip if the change already exists.

## Founder-prompted out-of-leash requests

`kind='ceo-authorized-out-of-leash'` ([[../specs/ceo-authorized-out-of-leash-actions]] Phase 1) ŌĆö a founder-prompted, CEO-routed Approval Request Ada raises from the [[director_coach_threads|Ask-Ada chat]] when she independently agrees a founder ad-hoc ask outside her leash is sound + the right call. The row lands directly in `status='needs_approval'` (never `queued`) carrying (a) her REASONING (why it's right + why it's out of leash + reversibility note) on `log_tail` + the pending action's `summary`+`preview`, and (b) a CONCRETE executable `pending_actions` entry (`type: 'run_prod_script' | 'apply_migration'`, `cmd`) + `out_of_leash: true` + `authorized_by: 'ceo-pending'` markers on the action. `instructions` is JSON `{origin:'director-coach', thread_id, director_function, action_type, cmd, reasoning, reversibility, irreversible, leash_rail, founder_ask, out_of_leash:true, authorized_by:'ceo-pending', dedupe_key}`; the dedupe key `{thread_id}::{actionType}::{cmd}` makes re-asking the same thing idempotent. `spec_slug` is `out-of-leash:{thread_id}` (there is no target spec). The kind is deliberately UNMAPPED in [[../libraries/approval-inbox]] `KIND_TO_FUNCTION` ŌåÆ the request routes to the CEO (Ada never authorizes her own out-of-leash action).
