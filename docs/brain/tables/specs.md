# specs

The card row for every spec — title, summary, owner, parent, blocked_by, priority/critical, deferred, intended_status, the rolled-up `status`, and the `milestone_id` FK link. ONE row per `(workspace_id, slug)`. The body lives in [[spec_phases]] (one row per phase, a child table). Authored by [[../specs/spec-body-table-and-backfill]] (M1 of [[../goals/db-driven-specs]]).

**Today** the spec body is still parsed from `docs/brain/specs/{slug}.md` by [[../libraries/brain-roadmap]] `parseSpec` and [[spec_card_state]] is a status-only mirror. This table holds the BODY — once [[../specs/spec-readers-from-db-retire-parser]] flips the readers (M3) the markdown is no longer authoritative; until then this is the secondary copy that the M3 cutover will lean on.

**Workspace-scoped** (mirrors [[spec_card_state]]). RLS: any authenticated user reads; service role does all writes (the writers hold the creds). No client-side spec writes.

**Primary key:** `id`

## Upsert spine

`specs_ws_slug` — a **unique index** on `(workspace_id, slug)`. The backfill and every future writer go through this `onConflict` key (insert on first write, update on repeat).

## Derived status

`specs.status` is no longer trigger-maintained — the `spec_phases_rollup` trigger on [[spec_phases]], the `specs_deferred_rollup` column trigger on this table, and `roll_up_spec_status` were all dropped in `derive-rollup-status` P3 (migration `20260725160000`). The stored column is **OVERRIDE-ONLY** and, since `specs-status-overrides-only` (migration `20260907130000`), holds ONLY the two NON-DERIVABLE lifecycle overrides (`deferred` / `folded`) or `NULL`. The CHECK now **REJECTS** every other value: a DERIVED state (`planned`/`in_progress`/`shipped`) — and, newly, `in_review` — is **NEVER persisted**; the writers clear it to `NULL` so the readers compute it from the phases. (`NULL` is the canonical "no override → purely derived" value.) The full axis is DERIVED at read time by [[../libraries/brain-roadmap]] `deriveSpecCardStatus` / `rollupPhaseStatus`:

- `in_review` is DERIVED, not stored (`specs-status-overrides-only`): a spec reads `in_review` iff NO build has started (the phase rollup is still `planned`) AND Vale has not durably passed it (`vale_review_passed_at IS NULL` — the SAME durable signal the claim-time build gate reads, so the board and the gate can never disagree). Because it can only appear while the rollup is `planned`, **a built spec can NEVER read `in_review`** — this kills the drift bug where a stored `in_review` that was never cleared pinned an all-phases-shipped spec in the In Review column. A send-back ("re-review this") is expressed by NULLing `vale_review_passed_at` (`markSpecCardBackToReview`), never a stored status.
- `folded` is terminal: the deriver returns it as-is (the fold worker sets it explicitly).
- `deferred=true` wins over phase progress.
- Multi-phase: any phase `in_progress` or any `shipped` (but not all) → `in_progress`; all (ignoring `rejected`) `shipped` → `shipped`.
- **Zero-phase (one-shot spec): the status derives from MERGE PROVENANCE, never the stored column.** `merged_pr` OR `last_merge_sha` set → `shipped`; both null → `planned` (then the `in_review` rule above applies if Vale hasn't passed it). The one-shot merge hook (`stampSpecMergeProvenance`) stamps `merged_pr`/`last_merge_sha` but NEVER advances `specs.status`, so reading the stored column here would leak a stale `planned` for a truly-shipped one-shot spec — the deriver reads provenance instead.

Because the deriver always prefers the phase rollup (or, for a one-shot, the merge-provenance derivation), a stale value on the stored column is never displayed. `specs-status-overrides-only` closes it at the WRITE side too: the single dual-write choke point ([[../libraries/spec-card-state]] `dualWriteSpecRow`, via `isOverrideStatus`), `setSpecStatus`, and `upsertSpec` all map any non-`{deferred,folded}` destination to `NULL`, so a derived state can no longer be persisted at all — and the tightened CHECK makes an accidental derived/`in_review` write a hard error rather than silent drift. The disposition lane ([[../libraries/spec-card-state]] `applyAdaDisposition`) flips a spec's board lane by CLEARING the override to `NULL` on `planned` (purely derived — it then reads `planned`/`in_progress`/`shipped` per the phases) and storing `'deferred'` only when the real `deferred` override is chosen. The bug this fixed: `noop-pipeline-test-4` carried a stored `status='planned'`; the analogous `in_review` drift pinned all-phases-shipped specs in the In Review column.
