# integrations/vercel-log-drain

The **Vercel Log Drain → `/api/webhooks/vercel-logs`** feed ([[../specs/error-feed-monitoring]] Phase 1): prod runtime errors / 500s pushed to us in real time, grouped + rate-limited into the Control Tower "Vercel errors" panel ([[../dashboard/control-tower]]).

**Endpoint:** `src/app/api/webhooks/vercel-logs/route.ts` · records via [[../libraries/control-tower]] (`error-feed.ts` → `recordError`, `source='vercel'`)

## How it works

1. **The owner creates the drain** via the Vercel API with our token (the lone Vercel setup — done **after** the endpoint ships): JSON drain, `delivery: batch`, filtered to error / 500-level runtime logs, URL = `https://shopcx.ai/api/webhooks/vercel-logs`, with a generated secret. *(Token has the `log-drains` scope — verified.)*
2. **Vercel POSTs batches** of log objects, signed `x-vercel-signature` = **HMAC-SHA1(rawBody)** with the drain secret.
3. The endpoint **verifies the signature** (timing-safe) against `VERCEL_LOG_DRAIN_SECRET`, **re-filters** to error/500-level entries (belt-and-suspenders over the drain's own filter — `level==='error'|'fatal'` or `statusCode/proxy.statusCode >= 500`), **groups the batch** by `(path, status, normalized message)`, **ACKs 200 immediately** with `{ received, incidents: groups.size }`, and then does the per-group `recordFeedDelivery('vercel')` + `recordError` fan-out inside `after(...)` from `next/server` — same Lambda invocation, off the response path.
   - **Why the split** — `recordError` chains extra work on a NEW signature: a `workspace_members` query + a Slack ops-alert per workspace admin + an `agent_jobs` insert to enqueue the repair agent. During a real error storm with many brand-new signatures, awaiting that fan-out inside the response path pushed a single POST past 300s and Vercel killed it with a Runtime Timeout — which is itself a `level:'error'` log the drain then re-delivered to this same endpoint (**the failure self-fed**, `vercel:c15ba2da4950bb3c` fired ~9× in ~30min). The pre-response work is now only sig-verify + JSON.parse + the in-memory group-by, so response time is bounded regardless of batch size; the recordError loop keeps running on the same Lambda after the 200 is sent. See [[../specs/vercel-logs-webhook-ack-fast-move-processing-after-response]].
   - **The `after()` fan-out is wall-clock bounded** — `after()` runs on the SAME Lambda invocation as the HTTP response, so it still counts against Vercel's 300s `maxDuration`. Moving the fan-out off the response path only got it off the DRAIN's response deadline; a real error storm with a large `groups` map could still push the invocation past 300s, Vercel killed it with a Runtime Timeout, and the drain re-delivered that log to this endpoint (`vercel:c15ba2da4950bb3c` again, ~12× over four days after the after() move). The loop now captures `const started = Date.now()` right after `recordFeedDelivery('vercel')` and breaks if `Date.now() - started > 250_000` (250s — 50s under Vercel's 300s cap), emitting a `console.warn` with the deferred count so the follow-up is observable. Deferred groups are dropped for this invocation — the next batch that re-fires the same signature will re-open them via `recordError`'s dedup key. See [[../specs/vercel-logs-webhook-bound-after-fanout-under-lambda-max-dura]]. The durable answer is an off-Lambda enqueue (Inngest fan-out over the same `groups` map) — out of scope for this fix, can layer on top of the deadline guard later.
   - The re-filter also **drops bare Lambda lifecycle/REPORT wrappers** (`isBareLifecycle`, factored into [[../libraries/control-tower]] `error-feed.ts` so it's reusable + unit-testable — see `error-feed.test.ts` / `npm run test:error-feed`): a 5xx whose entire message is `START`/`END`/`REPORT RequestId` scaffolding + the bare `[METHOD] path status=NNN` proxy line carries no error body — it's the non-actionable platform wrapper around a failure the function *already* logged via `console.error` (which has its own stable signature + repair spec). Capturing it too minted a second, redundant signature per failure (`vercel:ebdf493a37c60c34`). The proxy-summary matcher is **not** `$`-anchored after `status=NNN`: the real Vercel proxy line carries trailing tokens (duration/region/bytes, e.g. `[POST] /api/portal?route=removeLineItem status=502 669ms`), and the original `$`-anchored regex never matched it — so `.every()` failed and the wrapper was captured anyway, reopening `vercel:ebdf493a37c60c34` on a healthy, ticketed Appstle-502 loop (the `/api/portal` route's only 502 is the deliberate `handleAppstleError` response, already surfaced separately as a `portal-action-failed` ticket). See [[../specs/error-feed-drop-bare-502-proxy-wrapper]]. A lifecycle block that *also* carries a real message/stack (e.g. "Task timed out") is **not** bare and is still captured.
   - …and **drops Node Web-Streams client-abort teardown noise** (`isAbortedStreamNoise`, factored into [[../libraries/control-tower]] `error-feed.ts` so any feed can reuse it): when a visitor aborts an SSR stream mid-flight (client disconnect, `status:0` — the response never completed), Node core's Web-Streams teardown can race and emit a `level:'error'` log with **no frame in our code** — the documented TransformStream race (nodejs/node#62040) surfaced by Next.js streaming. Gated tightly so it only swallows this signature: requires `status === 0` **and** a message in the abort family (`transformAlgorithm is not a function`, `Invalid state: Controller is already closed`, `ERR_STREAM_PREMATURE_CLOSE`/`aborted`) **and** an `at ignore-listed frames`-only stack (zero frames in our code). Minting an open incident for it paged owners on a healthy PDP (`vercel:801aa4e3922198d3`). A real error with a frame in our code is **kept**. See [[../specs/error-feed-drop-aborted-stream-noise]].
   - …and **drops Inngest's built-in `LoggerMiddleware` bare "Inngest step error" log** (`isBareInngestStepErrorMiddlewareLog`, factored into [[../libraries/control-tower]] `error-feed.ts`): the Inngest SDK's `onStepError` fires on EVERY step throw (transient + final) with `proxyLogger.error({ err: arg.error }, 'Inngest step error')`. The error object lives in the JSON `err` context — Vercel's drain serializes only the bare Pino `msg` field, so what reaches us is the literal label `"Inngest step error"` with no body. Terminal failures are *already* authoritatively captured on `source='inngest'` by [[../inngest/inngest-failure-capture]] (`inngest/function.failed`), so the bare middleware log on `/api/inngest` is duplicate noise on a healthy retry loop — minting an OPEN incident for it paged Platform owners on a function that already self-heals (`vercel:b1daa612f563f5e9`). Gated tightly: requires `path === '/api/inngest'` AND the exact bare `"Inngest step error"` message; a log with the same label *plus* a real body Vercel managed to surface is **not** bare and is still captured. See [[../specs/error-feed-drop-bare-inngest-step-error-middleware-log]].
   - …and **drops the stack-form sibling — the Inngest step-wrapped non-Error log** (`isInngestStepWrappedNonErrorLog`, factored into [[../libraries/control-tower]] `error-feed.ts`): when a step handler throws a non-Error value (e.g. `throw {foo: 'bar'}` or any plain object), the SDK's `buildStepErrorOp` wraps it as `new Error(String(error))` — and `String({…})` is the literal `[object Object]`. Pino's `LoggerMiddleware.onStepError` logs the wrapped Error, so Vercel surfaces `Error: [object Object]` with a STACK whose every frame is inside the compiled Inngest chunk (`M.buildStepErrorOp` / `M.tryExecuteStep` / `steps-found` / `M._start`) — zero application frames because the wrapping happened in the SDK. Terminal failures are already captured on `source='inngest'` with the real function id + error class, so the vercel-side variant is duplicate noise — minting an OPEN incident for it paged Platform owners on a self-healing loop (`vercel:d48a64ae867f66dd`). Gated tightly: requires `path === '/api/inngest'` AND a message beginning with `Error: [object Object]` AND a stack containing a `buildStepErrorOp` frame AND **no** `at …` frame naming `src/` or `app/` — a real application throw carrying the same literal string ships a stack with a frame in our code and stays captured. See [[../specs/error-feed-drop-inngest-step-wrapped-non-error-noise]].
4. `recordError` records into [[../tables/error_events]] and **pages owners on a new signature or a re-firing spike**, rate-limited to one page per incident per 30 min (a burst of the same 500 = one page).

## Credentials / env

| Env | What | Who sets it |
|---|---|---|
| `VERCEL_LOG_DRAIN_SECRET` | the drain's signing secret (HMAC-SHA1) — **required**; without it the endpoint returns `503` (can't verify) | owner, at drain creation |
| `VERCEL_LOG_DRAIN_VERIFY` | the `x-vercel-verify` ownership token (optional) — echoed on `GET` + when Vercel presents it, so the drain's wiring check passes | owner, if Vercel requires it |

## Endpoints (this app)

- **`GET /api/webhooks/vercel-logs`** — Vercel's drain-ownership probe; echoes the `x-vercel-verify` token (presented header or `VERCEL_LOG_DRAIN_VERIFY`), else `200 ok`.
- **`POST /api/webhooks/vercel-logs`** — the signed log batch. `503` if no secret configured, `401` on bad signature, `400` on bad JSON, else `{ received, incidents }`.

## Gotchas

- **Signature is SHA1, not SHA256** (Vercel Log Drains) — over the **raw** request body (read via `request.text()` before any JSON parse).
- **Grouping happens twice** — once client-side per batch (so 500 identical entries in one POST = one `recordError` call with `occurrences: 500`), then again on `(source, signature)` in `error_events`.
- **Setup is the owner's**, but only the drain creation — the endpoint + grouping/alerting ship in the build. Until the secret is set the endpoint is live but refuses (`503`).
- **One failure used to land twice** — the app's `console.error` (actionable, own signature) *and* a bare Lambda lifecycle/REPORT block (non-actionable). `isBareLifecycle` now suppresses the latter so a single failed request mints one signature, not two. See [[../specs/vercel-capture-strip-lambda-lifecycle]].
