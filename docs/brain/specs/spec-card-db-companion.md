# Spec-card DB companion — instant PM state + deploy-pending flag ✅

**Owner:** [[../functions/platform]] · **Parent:** extends the roadmap/build-console; **supersedes + retires `roadmap-reads-specs-from-git`** (the live-GitHub approach that burned the API quota — its spec is deleted; this solves the same "instant status" goal from our own DB). Relates to [[spec-drift-agent]] (the reconciler keeps markdown ↔ DB in sync).

Today a card's status is parsed from the spec markdown's phase emojis (⏳🚧✅) **as bundled in the deployed build** — so a status change (a merge, a drift flip, an owner mark) doesn't show until a markdown edit + commit + **Vercel deploy**. That deploy lag is why the board feels stale, and why `roadmap-reads-specs-from-git` tried (wrongly) to read GitHub live. Move the *live* project-management state into a DB companion the board reads instantly.

## Model
- **`spec_card_state` table** (per workspace × spec_slug): `status`, `phase_states` (jsonb per phase), `flags` (e.g. `deploy_pending`, `building`, `blocked`), `last_merge_sha`, `updated_at`. The board reads this **DB-first** (instant), falling back to the markdown-parsed status when no row exists.
- **Canonical-source rule (the brain stays canonical):** the **markdown is canonical for spec *content* + the durable phase record**; `spec_card_state` is the **live mirror for the *board*** + transient flags that don't belong in committed markdown. On a true status conflict, **markdown wins** (it's what folds into the brain). The [[spec-drift-agent]] reconciler + the fold keep the two in sync — and become DB writes (instant) instead of markdown-edit-then-deploy.
- **Writers (instant):** a build merging → `status=shipped` for that phase + `deploy_pending=true` + `last_merge_sha`; a build starting → `building`; the drift reconciler / owner "mark verified" → flips; spec-blockers → `blocked`. All write the DB the moment the event happens.
- **`deploy_pending` (the "shipped · deploying" flag) — clean signal, no webhook:** the deployed app exposes its own `VERCEL_GIT_COMMIT_SHA`. If the live deployment's SHA is older than the card's `last_merge_sha`, the merged code isn't live yet → show **`shipped · deploying`**; once a deployment carrying that SHA is live → **`shipped · live`** (clear the flag). Generalizes the board's existing active-build "live overlay."

## Verification
*(Phase 1 — run after applying `scripts/apply-spec-card-state-migration.ts` and the deploy carrying this PR.)*

- **Migration landed.** Run `npx tsx scripts/_verify-spec-card-state-schema.ts` (or probe) → expect a `public.spec_card_state` table with columns `status, phase_states, flags, last_merge_sha`, a unique index on `(workspace_id, spec_slug)`, and RLS enabled.
- **Merge flips shipped instantly.** On `/dashboard/roadmap`, merge a `claude/*` build PR for a still-Planned/In-progress spec from `/dashboard/branches` → on the next board load the card jumps to the **Shipped — awaiting verification** column with a **`shipped · deploying`** chip, *before* the Vercel redeploy of the flipped markdown. In the DB, `select status, last_merge_sha, flags from spec_card_state where spec_slug='<slug>'` → `status='shipped'` (or in_progress if phases remain), `last_merge_sha` = the merge commit SHA, `flags->>'deploy_pending'='true'`.
- **Deploying → live.** Once the deployment carrying that merge SHA is live (`VERCEL_GIT_COMMIT_SHA` equals `last_merge_sha`, or a later deploy carries the flipped emoji) → the chip reads **`shipped · live`** (no stuck `deploying`).
- **Owner flip reflects immediately.** On a card, use the per-phase / status control (or the Control Tower one-tap drift flip) to set a phase ✅ → `POST /api/roadmap/status` (or `/api/roadmap/spec-drift`) commits the emoji AND writes `spec_card_state`; the board shows the new status on the next load, not after the next deploy.
- **Graceful fallback.** A spec with no `spec_card_state` row renders its markdown-parsed status unchanged (delete the row → the card falls back, no error). `resolveBoardStatus` never demotes below a markdown that's already ahead (markdown stays canonical).
- **Zero GitHub calls for status.** The board's status path touches only `spec_card_state` + the bundled markdown — `getRoadmap`/`getArchive`/`getSpec` read `fs` only (the `roadmap-reads-specs-from-git` git machinery is deleted from `src/lib/brain-roadmap.ts`).
- **Negative.** A card mid-build shows **In progress** (live-job overlay), never a false `shipped`; a merged card whose SHA is already the live deploy shows `shipped · live`, not a stuck `deploying`.

## Phase 1 — the table + instant writers + deploy-pending read ✅
`spec_card_state` migration (`supabase/migrations/20260623130000_spec_card_state.sql`, apply `scripts/apply-spec-card-state-migration.ts`); a new lib `src/lib/spec-card-state.ts` (read + writers + the `resolveBoardStatus` / `deploymentState` resolvers); written instantly from the merge path ([[../libraries/agent-jobs]] `reconcileMergedJobs` → `markSpecCardMergeShipped` with the PR `merge_commit_sha`), the drift reconciler ([[../libraries/spec-drift]] `reconcileSpecDrift`), and the owner flips (`/api/roadmap/status`, `/api/roadmap/spec-drift`). The board ([[../dashboard/roadmap]]) reads DB-first with markdown fallback (`resolveBoardStatus` — forward-merge, markdown-wins-if-ahead) + the `deploy_pending` SHA comparison (`deploymentState` vs `VERCEL_GIT_COMMIT_SHA`) and renders the `shipped · deploying` / `shipped · live` chip. The dead `roadmap-reads-specs-from-git` machinery is deleted from [[../libraries/brain-roadmap]] (the readers now read bundled `fs` only) and its spec file removed. Brain: [[../tables/spec_card_state]] (new) · [[../libraries/spec-card-state]] (new) · [[../libraries/brain-roadmap]] · [[spec-drift-agent]].

**Implementation note — `building` stays on the live-job overlay.** The board already promotes a Planned card with an active `agent_jobs` build to In progress (instant, DB-backed, self-clearing on terminal status) — this spec calls that the "active-build live overlay" being generalized. Rather than persist a separate `building` flag (which would go stale on a build failure, since there's no clean post-failure hook), Phase 1 keeps `building` on that live-job overlay and has `spec_card_state` persist the *shipped* progression + deploy flag — the parts that actually had deploy-lag. The `flags` jsonb still carries `blocked` (spec-blockers) and is open for future transient flags.
