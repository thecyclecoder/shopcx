## The accumulated coaching to bake in (now archived as rolled-into-mandates)
- **When On slow queries with a high call count and erratic (high-stddev) plan, Devi stamps cause=bloat_stale_stats / fix=vacuum_tuning whenever EXPLAIN doesn't isolate a Seq Scan, when the real cause is a missing index or call-volume.:** When a slow-query finding shows high call count + high stddev (the 'erratic plan' tell the impact line already prints) but no Seq Scan is isolated, do NOT diagnose bloat_stale_stats or propose vacuum_tuning — vacuum never fixes plan instability. Read that signature as missing-index / call-volume instead: propose a covering or GIN index on the hot predicate (e.g. tags @> …), or reduce call frequency (cache/batch/widen the poll). Only reach for vacuum when there is direct bloat/dead-tuple evidence, not as the ambiguous-plan fallback. — Both worst grades trace to one path: classifyExplainPlan at src/lib/control-tower/db-health.ts:288 returns cause='bloat_stale_stats' as the no-Seq-Scan fallback, which FIX_KIND_BY_CAUSE (db-health.ts:66) maps to vacuum_tuning — this is exactly what stamped the tickets (4ms×1.27M) and sms_campaign_recipients (31ms×157k) findings the owner dismissed as 'missing index, not bloat/vacuum,' even though the impact string built at db-health.ts:354 already flagged the high-stddev 'erratic plan' that signals a missing index, not stale stats.
- **When When EXPLAIN surfaces no isolated Seq Scan, Devi defaults the cause to bloat_stale_stats/vacuum_tuning and never reads stddev_exec_time — so erratic-plan / hammered queries that are really a missing covering-or-GIN index get branded as bloat.:** Before emitting bloat_stale_stats/vacuum_tuning for any slow query, read the stats first: if stddev_exec_time >= mean_exec_time (erratic plan) OR the call count is huge with a low mean, it is plan instability / high call volume from a MISSING covering-or-GIN index — propose add_index (GIN when the predicate is array/JSONB `@>`) or reduce_calls, never a vacuum. A vacuum is only correct when the dead-tuple ratio is high AND the plan is stable. Also run isSunsetQuery/isForeignQuery before proposing anything (klaviyo_* is retiring). — db-health.ts:288 returns bloat_stale_stats as the catch-all 'no Seq Scan isolated' branch, and stddev_exec_time is consumed only cosmetically at :354/:360 while classifyExplainPlan takes only planText — so the erratic-plan signal never routes the cause, which is exactly why tickets (4ms×1.27M) and sms_campaign_recipients (31ms erratic stddev) were dismissed as 'vacuum won't help — needs a GIN/missing index.'
