**Found in use 2026-07-01:** the **Devi** agent (`db_health`) sits at **4.6/10** after **2** coaching attempts that didn't stick. Rather than escalate to the CEO, roll the coaching into a permanent fix so the agent improves at the mandate level.

## The accumulated coaching to bake in (now archived as rolled-into-mandates)
- **When Three of the worst-graded slow-query findings (tickets g2, sms_campaign_recipients g3, plus the klaviyo g3) diagnosed the cause as `bloat_stale_stats` and proposed `vacuum_tuning`, when the query was actually fast-per-call/high-volume or a high-stddev equality/`@>` lookup — a missing/GIN index or call-reduction win, never a vacuum.:** When a slow-query EXPLAIN shows NO `Seq Scan` node, do NOT let it fall through to `bloat_stale_stats`/`vacuum_tuning`. First branch on mean_exec_time and the predicate: if mean < SLOW_PER_CALL_MEAN_MS (50ms) it is `high_call_volume` — reduce calls / cache / add a GIN index for an array-containment (`@>`) predicate like `tags @>` or a covering index for the hot WHERE (tickets was 4ms×1.27M, sms 31ms×157k). If it is a selective equality/containment lookup with high stddev (`message_sid = $1`), that erratic-plan variance is plan instability from a MISSING selective index (`add_index`), not bloat. Only ever emit `vacuum_tuning` when there is a real dead-tuple/bloat signal — 'erratic plan — high stddev' alone is not that signal. A vacuum is the wrong lever any time mean-per-call is already fast. — The mis-verdicts trace to db-health.ts:288, whose fallback returns `cause:'bloat_stale_stats'` ('no Seq Scan was isolated — likely stale stats') for any EXPLAIN lacking a textual Seq Scan (sms plan was `Aggregate → Limit`, no seq scan), and FIX_KIND_BY_CAUSE at db-health.ts:66 mechanically turns that into `vacuum_tuning`; the high-stddev impact string at db-health.ts:354 is what's being over-read as bloat. Owner dismissals confirm the class: job 6690475994378406674 'vacuum won't help… real lever is call-volume / a GIN index on tags' and job b208848b (sms_campaign_recipients) '30ms smells like a missing index, not vacuum/bloat.' The upgraded highCallVolume guard at db-health.ts:312 (threshold 50ms) is exactly the branch that should have caught both tickets (4ms) and sms (31ms) before the bloat fallback — the worker must apply that gate itself. (The klaviyo g3 sunset and set_config g5 foreign hits are a separate false-positive class already covered by isSunsetQuery/isForeignQuery at db-health.ts:301-304, so I did not fold them into this one rule.)
- **When Every dismissed/low-graded diagnosis defaulted to bloat/stale-stats → vacuum with zero dead-tuple evidence, when the pg_stat_statements shape actually meant high-call-volume (tickets 4ms×1.27M → GIN on tags), a missing index (sms_campaign_recipients 30ms with erratic stddev), or a sunset table (klaviyo_profile_events) — the vacuum fallback fired instead of reading mean/stddev/calls and table lifecycle.:** Before ever emitting a bloat_stale_stats/vacuum_tuning diagnosis, gate on the pg_stat_statements row shape first: (1) if mean_exec_time < ~50ms it is high_call_volume — the fix is reduce calls + a hot-predicate/GIN index (`@>` → GIN), NEVER vacuum, since per-call time is already fine; (2) if stddev > mean (erratic plan) with no Seq Scan isolated, treat it as a missing/covering index, not bloat; (3) if the query touches a sunset table (klaviyo_*), emit NO proposal. Only propose vacuum when you have a real bloat signal — dead-tuple ratio ≥0.2 AND stale autovacuum — which the slow-query path never measures, so vacuum should essentially never be its output. — classifyExplainPlan falls back to `bloat_stale_stats` whenever no Seq Scan is isolated (db-health.ts:288), mapped to vacuum_tuning (db-health.ts:66) with no dead-tuple check — exactly what produced the dismissed tickets/sms vacuum proposals; the SLOW_PER_CALL_MEAN_MS=50 volume gate (db-health.ts:122,312→319) and isSunsetQuery gate (db-health.ts:304) are precisely the checks that reclassify those rows to high_call_volume/skip, so applying them at the top of every job eliminates all three false-positive classes.
