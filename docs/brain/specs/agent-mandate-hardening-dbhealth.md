## The accumulated coaching to bake in (now archived as rolled-into-mandates)
- **When Devi diagnosed slow queries as bloat_stale_stats and prescribed vacuum_tuning whenever the EXPLAIN didn't isolate a Seq Scan, misreading 'erratic plan — high stddev' as bloat, even though the queries carried selective predicates (tickets.tags @> $3 at 4ms×1.27M; sms_campaign_recipients.message_sid = $1 at 31ms×157k) that a missing index / high call-volume — never a vacuum — explains.:** Before returning bloat_stale_stats/vacuum_tuning, look at the query's predicate and stats, not just whether a Seq Scan appears in the plan. If the WHERE has a selective equality (col = $1) or array/JSONB containment (tags @> $2) and/or high stddev ('erratic plan'), treat it as MISSING INDEX (add a covering/partial index; GIN for @>) — high stddev is plan instability from an absent index, not bloat. If mean is low (well under the 50ms per-call floor) but calls are huge, it's high_call_volume (reduce calls/cache + hot-predicate index), also never a vacuum. Only ever emit vacuum_tuning when you have an actual bloat signal (dead-tuple/vacuum-lag ratio), and never as the no-Seq-Scan fallback. — The wrong verdicts trace to the fallback at src/lib/control-tower/db-health.ts:288, which returns cause 'bloat_stale_stats' for any plan where no Seq Scan was isolated — exactly what caught the parameterized message_sid=$1 and tags @> $3 plans; combined with the stddev>mean 'erratic plan' string built at db-health.ts:354, the high-variance/missing-index symptom got mislabeled as bloat, so the vacuum fix (db-health.ts:734) could never address the real slow query.
- **When On the two hardest dismissals Devi diagnosed bloat_stale_stats→vacuum_tuning even though each EXPLAIN already showed an index-driven plan with low mean + high stddev: tickets q6690… (grade 2) used a Bitmap Index Scan on idx_tickets_tags_gin (4ms mean, stddev 5ms) and sms_campaign_recipients q7780… (grade 3) used an Index Scan on sms_campaign_recipients_message_sid_idx (31ms mean, stddev 66ms) — the owner killed both with 'vacuum won't help' / 'smells like a missing index, not vacuum/bloat.':** When the EXPLAIN plan already contains an Index Scan or Bitmap Index Scan and NO Seq Scan, NEVER emit cause=bloat_stale_stats / fix=vacuum_tuning — an index-driven plan does not get faster from VACUUM/ANALYZE. Read a low-mean + high-stddev index plan as plan instability from a missing COVERING/COMPOSITE index (or a GIN index when the predicate is `tags @> …`), or as pure high call volume, and propose that fix instead. — classifyExplainPlan falls through to `cause: "bloat_stale_stats"` at src/lib/control-tower/db-health.ts:288 whenever no Seq Scan token matches, so an already-index-driven plan (the tickets Bitmap Index Scan, the sms message_sid Index Scan) is mislabeled as bloat; that fallback must be gated on the ABSENCE of any Index/Bitmap Index Scan and, when one is present, routed to add_index/high_call_volume the way the mean<50ms path already does at db-health.ts:312 — exactly what the two owner dismissals demanded.
