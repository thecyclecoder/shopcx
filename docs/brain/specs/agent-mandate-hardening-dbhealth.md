# Harden the Devi agent — roll persistent coaching into its mandate

**Owner:** [[../functions/platform]] · **Parent:** [[platform-director-agent]] — graduate persistent coaching into a durable fix (director grades agents: low score → fix spec, never a CEO escalation)
**Found in use 2026-06-30:** the **Devi** agent (`db_health`) sits at **4.6/10** after **2** coaching attempts that didn't stick. Rather than escalate to the CEO, roll the coaching into a permanent fix so the agent improves at the mandate level.

## The accumulated coaching to bake in (now archived as rolled-into-mandates)
- **When Devi repeatedly jumps to proposing a fix (index, vacuum, or build) without first verifying table lifecycle status and without anchoring the diagnosis to the actual EXPLAIN output (plan type, bottleneck node, and stddev cause).:** When you see a slow-query alert, do three checks before proposing any fix: (1) confirm the table is not sunset/retiring — if it is, skip or flag as retirement-tracked; (2) run and document EXPLAIN (ANALYZE, BUFFERS) and identify the specific bottleneck node (seq scan, nested loop, etc.) rather than defaulting to bloat/stats assumptions; (3) verify the proposed index or fix directly addresses that bottleneck node — only then queue a build. — All three low-graded actions share the same upstream failure: Devi acted on a surface signal (slow query alert) without validating the context (table lifecycle) or grounding the diagnosis in actual plan evidence, causing either wasted work on retiring tables or the wrong fix (vacuum instead of a GIN index) on live ones. Enforcing these three sequential gates before any recommendation ensures the fix is both relevant and causally tied to what EXPLAIN actually shows.
- **When Devi interprets surface-level metrics (high stddev, elevated mean, sparse logs) as evidence of bloat or stale stats and recommends vacuum/statistics fixes without first reading EXPLAIN output to identify the actual execution plan bottleneck.:** When you see high stddev or a suspicious mean latency, do NOT assume bloat or stale stats — instead, run EXPLAIN (ANALYZE, BUFFERS) on the exact slow query first, confirm whether the plan shows a seq scan, missing index, or plan instability, then propose only the fix (e.g., a GIN/covering index, query rewrite) that directly addresses what the plan reveals; document the EXPLAIN output and your reasoning inline before queuing any build. — Bloat and stale stats produce predictably uniform slowdowns, not high stddev; high stddev almost always signals plan instability or missing index selectivity, which only EXPLAIN exposes. Skipping EXPLAIN and jumping to vacuum produces fixes that cannot address the real bottleneck, failing both the 'correct diagnosis' and 'fix addresses the slow query' rubric criteria simultaneously.

## Phase 1 — bake the coaching into the agent
- Make the above coaching PERMANENT behavior of the `db_health` agent — fold it into its prompt/mandate/code (its run-job + prompt in `scripts/builder-worker.ts` and the relevant `src/lib/agents/*`), not ephemeral appended `agent_instructions`. Once baked, the agent should follow it by default.
- Verify the agent's grade rollup recovers (≥ 7/10) over the next window of graded actions; if a coaching class still recurs, the bake-in missed it.
