# cancel_order direct action — let the orchestrator cancel a paid-but-unfulfilled one-time order before it ships ⏳

**Owner:** [[../functions/cs]] · **Parent:** CS mandate "Ticket-derived product fixes" · **Derived-from-ticket:** `21b8c370-d0c6-4c9b-9ad6-02fbba1c892c`

Ticket 21b8c370 escalated because a customer asked us to cancel a one-time web order placed by mistake (SC133086, paid, unfulfilled) and the orchestrator had no way to do it: cancelOrder() exists in src/lib/shopify-order-actions.ts but is wired only to the fraud-confirm and owner order-actions routes, never to directActionHandlers. The AI papered over the gap with a false 'no cancel-before-ship path exists' claim, promised a wasteful ship-then-return + full refund, then fired create_return on an unfulfilled order — which fails by definition because getReturnableItems only reads SUCCESS fulfillments — and escalated. Add cancel_order as a first-class, guarded direct action so this class of request resolves correctly instead of being routed into a doomed return.

## Problem (from escalated ticket `21b8c370-d0c6-4c9b-9ad6-02fbba1c892c`)
On a paid-but-unfulfilled one-time order, the orchestrator's only levers are partial_refund (order still ships — we eat COGS and the customer keeps unwanted product) and create_return (fails with 'No returnable items found on this order' because nothing is fulfilled — the literal escalation on this ticket). There is no cancel_order handler in src/lib/action-executor.ts directActionHandlers, and no sonnet_prompt covers one-time-order cancel-before-ship (all cancel rules route SUBSCRIPTION cancels to the journey). Result: a structurally-correct, common request ('cancel my order, I ordered it by mistake') has no safe automated path.

**Likely target:** `Add a `cancel_order` handler to directActionHandlers in src/lib/action-executor.ts wrapping cancelOrder() from src/lib/shopify-order-actions.ts. Required guards: (1) resolve the order on the customer's linked accounts and refuse if it isn't theirs (Validate-validatable-claims rule); (2) UNFULFILLED-only gate — if any fulfillment exists, do NOT cancel; route to create_return instead; (3) call with reason:'CUSTOMER', refund:true, restock:true, notify handled by the response message; (4) since Shopify orderCancel can run async, treat anything other than a clean orderCancelUserErrors-empty result as a failure and surface it honestly — never report a queued/failed job as success (extend cancelOrder() in src/lib/shopify-order-actions.ts:487 if it can return success on a queued job); (5) log the action + reasoning per the North Star (no silent money-moving execution). Add a Direct-actions-catalog tool_hint entry that sharply separates cancel_order (one-time order) from the cancel_subscription journey (subscription). Brain: update docs/brain/libraries/action-executor.md and docs/brain/libraries/shopify-order-actions.md (new exposed action), and add a 'cancel an unfulfilled one-time order' branch to docs/brain/lifecycles/ai-multi-turn.md and docs/brain/lifecycles/ticket-lifecycle.md, cross-referencing docs/brain/lifecycles/return-pipeline.md for the unfulfilled→cancel vs fulfilled→return split.`
