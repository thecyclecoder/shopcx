# Spec body in the DB — `specs` + `spec_phases` tables + one-time backfill

**Owner:** [[../functions/platform]] · **Parent:** [[../goals/db-driven-specs]] M1 — The spec body in the DB (phases are a TABLE, not jsonb)

The foundation step of [[../goals/db-driven-specs]]: stand up the two relations that hold a spec's BODY in the DB — `specs` for the card (title, summary, owner, parent, blocked_by, priority/critical, deferred, intended_status, status, milestone_id FK nullable) and `spec_phases` for ONE ROW PER PHASE (id, spec_id FK, position, title, body, status, pr, merge_sha, verification). Phases are a child TABLE (not a jsonb array on `specs`) specifically so a phase can MOVE between specs (lift P5 into a new deferred spec) via a single `UPDATE spec_phases SET spec_id=…, position=…` that preserves the phase's stable id + PR/SHA provenance + history — the gotcha [[spec-status-phase-pr-provenance]] already hit with jsonb. Status on `specs` ROLLS UP from the child phase rows (impossible to stick at `shipped` while only 1 of 4 phases is done — the [[spec-review-agent]] class of bug). One-time backfill: run the existing [[../libraries/brain-roadmap]] `parseSpec` parser ONE LAST TIME over `docs/brain/specs/*.md`, INSERT into `specs` + `spec_phases`. After this spec lands the rows are complete; the `.md` files stay authoritative until [[spec-readers-from-db-retire-parser]] cuts readers over. Closes the gap: today the spec body lives only as markdown ([[../libraries/brain-roadmap]] `parseSpec` L173) and [[../tables/spec_card_state]] is a status-only mirror without the body.

## Phase 1 — schema migration for `specs` + `spec_phases`
- Author `supabase/migrations/{YYYYMMDDNNNNNN}_specs_and_spec_phases.sql` creating two new `public` tables.
- **`public.specs`** columns: `id uuid pk default gen_random_uuid()`, `workspace_id uuid not null references workspaces(id) on delete cascade` (mirrors [[../tables/spec_card_state]]), `slug text not null` (the `docs/brain/specs/{slug}.md` key), `title text not null`, `summary text` (the first paragraph below the H1), `owner text not null` (a function slug — `growth｜cmo｜retention｜cfo｜logistics｜cs｜platform`), `parent text not null` (the mandate/milestone string, free-text — same shape parseSpec carries today), `blocked_by text[] not null default '{}'` (sibling spec slugs), `priority text` (`critical` or null — the **Priority:** flag, today on `spec_card_state.flags.critical`), `deferred boolean not null default false` (the parked flag, today on `spec_card_state.flags.deferred`), `intended_status text check (intended_status in ('planned','deferred'))` (the author-suggestion the director disposition lane uses, today on `spec_card_state.flags.intended_status`), `status text not null default 'in_review' check (status in ('in_review','planned','in_progress','shipped','deferred','folded'))` (the rolled-up status — including the [[spec-review-agent]] `in_review` and the [[../goals/db-driven-specs]] M4 `folded` state), `intended_status_set_by text`, `repair_signature text` (today parsed from the spec), `auto_build boolean default false` (today parsed from the spec), `milestone_id uuid` (nullable FK — set by [[goals-milestones-tables-and-backfill]]; a standalone spec has none), `created_at timestamptz not null default now()`, `updated_at timestamptz not null default now()`.
- **Unique index** on `(workspace_id, slug)` — the upsert spine, mirrors [[../tables/spec_card_state]].
- **`public.spec_phases`** columns: `id uuid pk default gen_random_uuid()` (STABLE across moves — the lift-a-phase-between-specs use case), `spec_id uuid not null references public.specs(id) on delete cascade`, `position int not null` (1-indexed, the ordering surface), `title text not null`, `body text not null` (the phase content as the brain renders it — bullet list / paragraph / code), `status text not null default 'planned' check (status in ('planned','in_progress','shipped','rejected'))`, `pr int` (the PR number that shipped the phase), `merge_sha text` (the merge commit SHA — provenance, [[spec-status-phase-pr-provenance]]), `verification text` (the per-phase ## Verification block when present), `created_at timestamptz not null default now()`, `updated_at timestamptz not null default now()`.
- **Unique index** on `(spec_id, position)`.
- RLS: enable on both; authenticated users read (mirrors [[../tables/spec_card_state]] `spec_card_state_select`); service role full access (`spec_card_state_service` shape). No client-side writes.
- The DB function `public.roll_up_spec_status(spec_id uuid)` reads the spec's `spec_phases` rows and writes the rolled-up status back to `specs.status` (same rule [[../libraries/brain-roadmap]] `deriveStatus` uses today: any `in_progress` → `in_progress`; all `shipped` → `shipped`; deferred flag overrides to `deferred`; in_review locked until cleared; otherwise `planned`). A row-level trigger on `spec_phases` (after insert/update of `status`) calls it — so the rollup happens in the DB, not in app code, and it's IMPOSSIBLE to commit a `specs.status='shipped'` with non-shipped phases. Apply via `scripts/apply-specs-tables-migration.ts` per [[../recipes/write-migration]].

## Phase 2 — the row writers + the spec/spec_phases lib
- New library `src/lib/specs-table.ts` (the future-canonical read/write surface, parallel to [[../libraries/spec-card-state]] until [[spec-readers-from-db-retire-parser]] retires the latter): `getSpec(workspaceId, slug)` (joins `specs` + `spec_phases` ordered by position → a `SpecRow` shape close to [[../libraries/brain-roadmap]] `SpecCard`), `listSpecs(workspaceId, filter?)`, `upsertSpec(workspaceId, row, phases)` (transactional — UPSERT the parent + REPLACE phases under the same `spec_id`, preserving phase ids by `position` where unchanged), `movePhase(phaseId, newSpecId, newPosition)` (the "lift P5 into a deferred spec" primitive — a single UPDATE keeping the phase's id + pr + merge_sha + history). `npx tsc --noEmit` clean.
- A thin `Phase` enum identical to [[../libraries/brain-roadmap]] `Phase` (planned/in_progress/shipped/rejected) — re-exported so callers don't churn imports.
- No reader cuts over in this spec — `getRoadmap` / `getSpec` still read markdown via [[../libraries/brain-roadmap]] until [[spec-readers-from-db-retire-parser]] flips them. This phase only ADDS the writer surface.

## Phase 3 — one-time backfill from `docs/brain/specs/*.md` → `specs` + `spec_phases`
- Author `scripts/backfill-specs-from-markdown.ts` (per [[../recipes/script-conventions]] — `_bootstrap.ts`, dry-run by default, `--apply` to write, idempotent + resumable). Loads every `docs/brain/specs/*.md`, runs [[../libraries/brain-roadmap]] `parseSpec` over each (the EXISTING parser, used ONE LAST TIME), and INSERTs `specs` + `spec_phases` rows.
- For each spec slug: look up the matching [[../tables/spec_card_state]] row (the live status mirror) and prefer its `status` / `flags.critical` / `flags.deferred` / `flags.intended_status` over the markdown-parsed values (the DB-mirror has been authoritative for status since [[spec-status-db-driven]]). Phase-level: prefer `spec_card_state.phase_states[i].{pr, merge_sha, status}` for any index the mirror knows, fall back to the markdown's emoji-derived phase status. Carry `priority/critical/deferred/intended_status` over too.
- Idempotent: re-running the backfill on an already-backfilled spec is a no-op (UPSERT `specs` by `(workspace_id, slug)`, REPLACE `spec_phases` only when the phase signature changed — same id-by-position rule the lib uses).
- Print a per-spec line: `backfilled {slug}: {N} phases, status={X}` (dry-run prints what WOULD insert). On `--apply`, after the loop completes, verify every `specs` row's rolled-up `status` matches the source `spec_card_state.status` (or the markdown parse when no mirror) — flag any mismatch for human review, do NOT silently overwrite.
- Out of scope: deleting the `docs/brain/specs/*.md` files (M4 fold writes them; [[spec-readers-from-db-retire-parser]] retires the parse). Out of scope: rewiring readers ([[spec-readers-from-db-retire-parser]]). Out of scope: rewiring writers ([[spec-authoring-writes-db-and-worker-materialize]]).

## Safety / invariants
- **Database is the spec.** Probe the live schema after `--apply` to confirm the columns + indexes + trigger match this doc (per [[../recipes/verify-schema]] — author `scripts/_verify-specs-schema.ts`). Never assume; the [[../tables/]]_brain page is the contract.
- **Workspace-scoped, RLS-protected.** Every read carries `workspace_id` (mirrors [[../tables/spec_card_state]]); every write goes through `createAdminClient()` (service role only, per [[../CLAUDE.md]]). No client-side spec writes.
- **Phases keep their id across moves.** `movePhase` is a single UPDATE preserving `spec_phases.id + pr + merge_sha + created_at`. A jsonb-style destroy+recreate would BREAK the [[spec-status-phase-pr-provenance]] provenance chain; trigger an alert if a backfill ever loses a `merge_sha` for a previously-shipped phase.
- **The rollup is a DB trigger, not app code.** If the trigger is ever dropped, the [[spec-review-agent]] "shipped with 1 phase" class returns instantly — call it out as a hard rail in the trigger comment.
- **Backfill is non-destructive.** The `.md` files stay in `docs/brain/specs/` until [[spec-readers-from-db-retire-parser]] retires the parse — leaving a rollback path if the row data is ever wrong. No `git rm` here.
- **Forward-merge guard preserved.** When backfilling a spec where the markdown is AHEAD of `spec_card_state.phase_states` (a fresh edit on disk the mirror hasn't caught), prefer the markdown — same rule [[../libraries/brain-roadmap]] `overlayDbStateOnSpec` enforces today (L389), so we don't backfill a STALE board state on top of a fresh disk edit.

## Completion criteria
- Migration applied; `scripts/_verify-specs-schema.ts` prints the expected columns/indexes/trigger/policies for `public.specs` + `public.spec_phases`. `npx tsc --noEmit` clean.
- `scripts/backfill-specs-from-markdown.ts --apply` populates a `specs` row + N `spec_phases` rows for EVERY file in `docs/brain/specs/*.md`. Spot-check three specs that exercise different shapes: a multi-phase shipped (e.g. [[regression-agent]]), a one-shot card (e.g. [[agent-mandate-hardening-ticket-improve]]), an `in_review` spec the [[spec-review-agent]] queue carries.
- The DB trigger fires: setting one phase's `status='shipped'` while siblings stay `planned` leaves `specs.status='in_progress'`; flipping the last sibling to `shipped` rolls `specs.status` to `shipped` in the same transaction. Try the inverse (an `UPDATE specs SET status='shipped'` directly while a phase is `planned`) and confirm the next phase write rolls it back.
- `movePhase(phaseId, newSpecId, newPosition)` lifts a phase between specs preserving `id + pr + merge_sha`; both `specs.status` values roll up correctly post-move (the source spec may drop from `shipped`→`in_progress`; the destination picks the phase up).
- No reader has been touched: `getRoadmap` / `getSpec` / the board / the Slack flows STILL read markdown via [[../libraries/brain-roadmap]] in this spec — the cutover is owned by [[spec-readers-from-db-retire-parser]]. Backfill double-write is acceptable here; readers stay markdown-first.
- Brain pages: NEW [[../tables/specs]] + [[../tables/spec_phases]] table pages, NEW [[../libraries/specs-table]] page, NEW [[../recipes/backfill-specs-from-markdown]] page (per the CLAUDE.md hard rule).
