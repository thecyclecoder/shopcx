# All spec readers from the DB — retire parser, overlay, reconciler, file-tracing

**Owner:** [[../functions/platform]] · **Parent:** [[../goals/db-driven-specs]] M2 — The board + all reads come from the DB
**Blocked-by:** [[spec-authoring-writes-db-and-worker-materialize]]

The cutover step. Once every spec write goes to the DB ([[spec-authoring-writes-db-and-worker-materialize]] keeps a mirror commit in place specifically so readers can swap atomically here), flip every reader off [[../libraries/brain-roadmap]] `parseSpec` (L173) + `overlayDbStateOnSpec` (L389) and onto `public.specs` + `public.spec_phases` (from [[spec-body-table-and-backfill]]). The roadmap board (`src/app/dashboard/roadmap/page.tsx`), the spec detail page (`src/app/dashboard/roadmap/[slug]/`), the board's box/chat APIs (`src/app/api/roadmap/box/route.ts`, `src/app/api/roadmap/chat/route.ts`), the Slack flows (`src/app/api/slack/events/route.ts`, `src/app/api/slack/interactions/route.ts`), the build dispatch's blocker gate (`src/app/api/roadmap/build/route.ts` → `queueRoadmapBuild` → `getSpecBlockers`), and the [[../specs/spec-test-deep-verification|spec-test agent]] all read `specs.status` + `spec_phases` rows directly (joined). With nothing reading markdown for spec content + state, retire `parseSpec`, `overlayDbStateOnSpec`, `mergePhaseStates`, the forward-merge guard ([[../libraries/brain-roadmap]]:389-432), and [[../libraries/spec-drift]] `reconcileSpecDrift` (`src/lib/spec-drift.ts:379`). Delete the specs entries from `next.config.ts` `outputFileTracingIncludes` (L24-29, 41, 45) — Vercel no longer needs to bundle `docs/brain/specs/**/*.md` for the dashboard routes. The `kind='mirror-spec-md'` worker lane from [[spec-authoring-writes-db-and-worker-materialize]] Phase 4 retires here. `specs.status`'s DB-trigger rollup (from [[spec-body-table-and-backfill]]) means the [[spec-review-agent]] "shipped with 1 of 4 phases" class CANNOT recur — the metric in [[../goals/db-driven-specs]] success metric. Closes the gap: today even with [[spec-status-db-driven]] shipped, `getRoadmap` still parses spec markdown for the BODY (`parseSpec` L173) and only overlays the DB for STATUS (`overlayDbStateOnSpec` L389); `reconcileSpecDrift` exists ENTIRELY to fight that disk-vs-DB drift; `next.config.ts` traces the specs dir into multiple route bundles for the same reason.

## Phase 1 — DB-first readers in `brain-roadmap`
- In `src/lib/brain-roadmap.ts`, replace `readSpecs()` (the function that reads + `parseSpec`s every `docs/brain/specs/*.md`) with `readSpecsFromDb(workspaceId)`: joins `specs` + `spec_phases` ordered by `position`, maps each row → a `SpecCard` shape (the public type stays the same — every downstream caller keeps compiling).
- `getRoadmap(workspaceId)` now returns DB-sourced cards directly (no overlay needed — the row IS the truth). `getSpec(slug, workspaceId)` is a single `select … where slug=$1` instead of a fs read + parseSpec.
- The `SpecCard.summary`/`title`/`phases`/`owner`/`parent`/`blockedBy`/`autoBuild`/`repairSignature` fields all come from columns (set by [[spec-body-table-and-backfill]] schema). Per-phase title/body/status/pr/merge_sha come from `spec_phases` rows.
- `listSpecSlugs()` becomes a `select slug from public.specs`. The few legacy no-arg callers ([[../libraries/agent-personas]] org-chart at startup, scripts) get a tiny shim `resolveDefaultWorkspaceId()` (already used in [[../libraries/brain-roadmap]]:446) that picks the single-tenant workspace.
- Verification — Phase 1: `getRoadmap()` against a workspace returns the same cards (deep-equal) as the pre-cutover overlay path for a snapshot of every spec. `getSpec(slug)` matches one-for-one. `npx tsc --noEmit` clean (the `SpecCard` shape is stable — only the SOURCE flips).

- **The blocker gate** ([[spec-blockers]]): `getSpecBlockers` resolves blocker slugs against `public.specs.status` directly — no markdown to parse. The gate stays — only its read source changes.
- Verification — Phase 2: every surface above renders the same content as pre-cutover for a representative set of specs (board view, spec page, Slack card, build dispatch, spec-test agent's read). `npx tsc --noEmit` clean. The new pages are NOT in `outputFileTracingIncludes` because they don't read the disk anymore (Phase 4 removes the entries).

## Safety / invariants
- **Strict deploy order.** Phase 1+2 (DB-first readers) MUST deploy BEFORE Phase 3 (delete the parser) — if the parser is gone but readers are still pointed at it, the board goes dark. Keep the mirror commit ([[spec-authoring-writes-db-and-worker-materialize]] Phase 4) running until Phase 2 ships to every reader.
- **The DB-trigger rollup is the rail.** With the parser gone, `specs.status` is the SOLE truth. The trigger from [[spec-body-table-and-backfill]] is what prevents the [[spec-review-agent]] "shipped with 1 phase" class — verify the trigger is still attached as part of this spec's completion checks (the [[../recipes/verify-schema]] script).
- **No reader reads the disk for spec content.** After Phase 3, `grep -rn "GOALS_DIR\|SPECS_DIR" src/` should return only the goal/lifecycle reads ([[goal-readers-from-db-retire-parsegoal]] still owns the goal-side cutover). A regression here re-introduces the very drift the goal kills.
- **Workspace-scoped reads.** Every `getRoadmap`/`getSpec` call passes a `workspaceId` from the auth context. The legacy no-arg fallback `resolveDefaultWorkspaceId()` is retained ONLY for single-tenant boot paths — flag any new no-arg caller as a regression.
- **The `.md` files stay on disk until M4 fold rewrites them.** Don't `git rm` here — [[spec-fold-from-db-row]] is the only path that writes/removes spec markdown. Removing them now would break the archive's git history surfaces.
- **No fallback to the parser.** The reader pages do NOT silently fall back to the markdown if the DB read fails — they fail loud (a 500 with a clear error) so an outage is visible, not papered over with a stale parse.

## Completion criteria
- `getRoadmap(workspaceId)` and `getSpec(slug, workspaceId)` are pure DB reads. `npx tsc --noEmit` clean. A representative cross-section of specs (multi-phase, in_review, one-shot, shipped, deferred) renders byte-for-byte identical on board + spec page vs the pre-cutover snapshot.
- Every reader surface listed in Phase 2 reads `public.specs` + `public.spec_phases` only. Slack cards, the build dispatch's blocker gate, the spec-test agent's read all match.
- `grep -rn "parseSpec\|overlayDbStateOnSpec\|reconcileSpecDrift\|mergePhaseStates" src/` returns zero matches. The `kind='mirror-spec-md'` worker lane is removed; no new mirror commits land on `main`.
- `next.config.ts` `outputFileTracingIncludes` no longer carries `"./docs/brain/specs/**/*.md"` for any route. The Vercel build is smaller (the specs corpus dropped from the bundle).
- The [[spec-review-agent]] "shipped with 1 of 4 phases" class is structurally impossible — manually set a `spec_phases.status='shipped'` while siblings stay `planned` and confirm `specs.status` rolls up to `in_progress` (the trigger from [[spec-body-table-and-backfill]] fires). Document the test in [[../recipes/verify-schema]] alongside the schema check.
- Brain pages: UPDATE [[../libraries/brain-roadmap]] (drop the parseSpec section; document the new DB-first reader path), DEPRECATE [[../libraries/spec-drift]] (or fold its remaining live use case under [[migration-drift-track-table-renames]]), UPDATE [[../tables/spec_card_state]] to note `phase_states` is superseded by `public.spec_phases` (the table itself parks for a follow-up retirement in [[spec-fold-from-db-row]]).
