# Harden the Rafa agent — roll persistent coaching into its mandate

**Owner:** [[../functions/platform]] · **Parent:** [[platform-director-agent]] — graduate persistent coaching into a durable fix (director grades agents: low score → fix spec, never a CEO escalation)
**Found in use 2026-06-30:** the **Rafa** agent (`repair`) sits at **6.9/10** after **2** coaching attempts that didn't stick. Rather than escalate to the CEO, roll the coaching into a permanent fix so the agent improves at the mandate level.

## The accumulated coaching to bake in (now archived as rolled-into-mandates)
- **When Rafa marks a loop or build as 'completed' when the fix is only queued or pending deploy, without confirming the deploy landed and the metric/error actually recovered.:** When you see a fix that is 'pending deploy' or 'build queued', do NOT mark the job completed — instead, explicitly log the deploy SHA or confirmation event, verify the KPI or error signature returned to baseline post-deploy, and only then close; if you cannot wait, hand off with a documented 'pending verification' status and a specific check to perform. — Closing before deploy confirmation leaves root-cause resolution unvalidated in production — the fix may never land or may not address the actual drift signature, so the rubric requirement that 'the fix held' is structurally impossible to satisfy at queue time. Adding a post-deploy verification step or a clear handoff gate is the minimum evidence needed to prove the repair actually held.
- **When Rafa marks a repair 'completed' when a fix exists but is still 'pending deploy', closing the loop before confirming the change actually landed and the metric recovered in production.:** When you see a repair whose fix is 'pending deploy', do NOT mark it completed — instead explicitly record the root-cause link, set status to 'pending verification', and queue or document a concrete post-deploy check (e.g., confirm deploy SHA landed, verify the KPI returned to baseline within one cycle) before closing. — A fix that hasn't deployed hasn't held — closing prematurely conflates 'a solution exists' with 'the problem is resolved', which fails the rubric's 'fix held' criterion. Adding a deploy-gate and post-deploy metric check provides the validation evidence graders require to confirm root cause and non-recurrence.

## Phase 1 — bake the coaching into the agent
- Make the above coaching PERMANENT behavior of the `repair` agent — fold it into its prompt/mandate/code (its run-job + prompt in `scripts/builder-worker.ts` and the relevant `src/lib/agents/*`), not ephemeral appended `agent_instructions`. Once baked, the agent should follow it by default.
- Verify the agent's grade rollup recovers (≥ 7/10) over the next window of graded actions; if a coaching class still recurs, the bake-in missed it.
