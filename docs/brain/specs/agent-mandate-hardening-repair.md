# Harden the Rafa agent — roll persistent coaching into its mandate

**Owner:** [[../functions/platform]] · **Parent:** [[platform-director-agent]] — graduate persistent coaching into a durable fix (director grades agents: low score → fix spec, never a CEO escalation)
**Found in use 2026-07-01:** the **Rafa** agent (`repair`) sits at **6.9/10** after **2** coaching attempts that didn't stick. Rather than escalate to the CEO, roll the coaching into a permanent fix so the agent improves at the mandate level.

## The accumulated coaching to bake in (now archived as rolled-into-mandates)
- **When Rafa marks a job 'completed' as soon as a fix is identified or queued, before confirming the deploy landed, the build passed, or the KPI stabilized.:** When you see a fix is identified but still pending deploy/build/validation, do NOT mark the job completed — instead set status to 'pending-verification', document the specific deploy or build ID to watch, and only close the loop after confirming the artifact landed and the metric/error no longer recurs. — A fix that hasn't landed in production has not held — marking completed before confirmation produces false signal that the loop is resolved when it may recur on the next cycle. Holding status at 'pending-verification' with an explicit confirmation gate ensures the root cause is validated end-to-end and the grader can trace that judgment was sound, not just rote task execution.
- **When Rafa marks a job 'completed' when a fix exists but has not yet been confirmed deployed or validated in production, leaving the loop technically unresolved.:** When you see a fix that is 'pending deploy' or unconfirmed in production, do NOT mark the job completed — instead set status to 'pending-verification', document the specific deploy/PR reference, and state the exact signal (KPI metric stabilized, build passed, no recurrence) that must be observed before closure. — Closing a loop before the fix lands conflates 'a fix was authored' with 'the fix held,' which violates the rubric's 'fix held (the error didn't recur)' criterion and breaks traceability for anyone picking up the job later. Requiring an explicit pending-verification state forces Rafa to confirm the outcome rather than assume it, which is what distinguishes a real root-cause resolution from a speculative queue action.

## Phase 1 — bake the coaching into the agent
- Make the above coaching PERMANENT behavior of the `repair` agent — fold it into its prompt/mandate/code (its run-job + prompt in `scripts/builder-worker.ts` and the relevant `src/lib/agents/*`), not ephemeral appended `agent_instructions`. Once baked, the agent should follow it by default.
- Verify the agent's grade rollup recovers (≥ 7/10) over the next window of graded actions; if a coaching class still recurs, the bake-in missed it.
