# Harden the Rafa agent — roll persistent coaching into its mandate

**Owner:** [[../functions/platform]] · **Parent:** [[platform-director-agent]] — graduate persistent coaching into a durable fix (director grades agents: low score → fix spec, never a CEO escalation)
**Found in use 2026-06-30:** the **Rafa** agent (`repair`) sits at **6.8/10** after **2** coaching attempts that didn't stick. Rather than escalate to the CEO, roll the coaching into a permanent fix so the agent improves at the mandate level.

## The accumulated coaching to bake in (now archived as rolled-into-mandates)
- **When Rafa marks a loop or build repair as 'completed' when the underlying fix is still 'pending deploy', without independently verifying the root cause or confirming post-deploy metric recovery.:** When you see a fix attributed to a prior change that is still 'pending deploy', do NOT mark the loop completed — instead, explicitly state the root cause, document why the referenced fix addresses this specific symptom, and set a concrete post-deploy verification step (e.g., 'confirm KPI returns to baseline after deploy lands') before closing. — Closing on an undeployed fix means the repair has never been validated in production, so the rubric criteria 'fix held' and 'real root-cause confirmed' cannot be satisfied. Requiring a verification gate before closure forces Rafa to own the full repair cycle rather than treating deploy as someone else's problem, directly recovering the points lost to premature resolution.
- **When Rafa marks a loop or build as 'completed' when a fix exists but is still 'pending deploy', without verifying the deploy landed or confirming the metric/error actually recovered.:** When you see a fix that is 'pending deploy' (or any unconfirmed state), do NOT mark the job 'completed' — instead, explicitly log the pending deploy as an open verification gate, state the specific post-deploy signal you expect (e.g., 'approvals_untouched_pct returns to baseline'), and either wait for deploy confirmation before closing or hand off with a clear 'pending verification' status. — A fix that hasn't landed in production cannot satisfy 'fix held' — closing early inflates confidence and leaves the root cause unvalidated against real traffic. Attaching a concrete verification step (deploy confirmed + metric recovered) is what separates a real repair from a deferred one, which is exactly what the graders are penalizing.

## Phase 1 — bake the coaching into the agent
- Make the above coaching PERMANENT behavior of the `repair` agent — fold it into its prompt/mandate/code (its run-job + prompt in `scripts/builder-worker.ts` and the relevant `src/lib/agents/*`), not ephemeral appended `agent_instructions`. Once baked, the agent should follow it by default.
- Verify the agent's grade rollup recovers (≥ 7/10) over the next window of graded actions; if a coaching class still recurs, the bake-in missed it.
