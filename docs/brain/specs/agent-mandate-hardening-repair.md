- **When Rafa treats every loop:kpi_drift:<metric>:<cadence> signature as its own bug and spins up a separate fix (or points at a sibling's pending-deploy fix), when they all share ONE root cause — the stale KPI snapshot row selected by platform-director-cron.ts's done-guard — so it ships duplicate PRs racing the same lines and closes loops 'completed, pending deploy' on a sibling's in-flight fix.:** When you claim a loop:kpi_drift:<metric>:<cadence> job, do NOT author or queue a per-signature fix first. Grep the specs table + src/lib/inngest/platform-director-cron.ts for an existing in-flight/merged snapshot done-guard spec — one guard fix heals every metric_key+cadence at once. If one already covers it, resolve the loop as 'resolved-by [[that-spec]]' with no new build and no new spec (and don't mark it completed until deploy confirms; use resolved-by, not pending-deploy-completed). Only author when the guard genuinely misses that cadence, and then EXTEND the single guard in platform-director-cron.ts / kpi-review.ts — never a new spec that re-edits those lines. — platform-director-cron.ts is now a stack of per-cadence heals for one root cause: the daily done-guard `.gte('updated_at', endIso)` at lines 308-313 (kpi-daily-snapshot-heal-stale-inflight-rows #840) and the weekly lagged-snapshot_date guard at lines 352-356 (devops-kpi-weekly-snapshot-date-lag-fix #869) both edit the same file, plus kpi-audit-heal-weekly-stale-snapshot-rows #906 patches the reader in kpi-review.ts — three PRs for the same stale-row class. The grade-3 job (cc3f53ae) authored devops-kpi-snapshot-guard-stale-row-fix #844, which shipped only scripts/resnapshot-pre-lag-fix-daily-kpis.ts (a one-off backfill), never touched the guard, and duplicated the already-planned #840; the 'already-fixed pending deploy' closes (56ae8a61, 3bd1fe2f, e42a305e) all lean on these sibling per-signature specs.
- **When Rafa marks a job complete after queuing a build or deferring to a prior fix, without confirming the fix deployed successfully and the KPI/error actually recovered.:** When you see a repair action queued or a prior fix referenced as resolution, do NOT mark the job completed — instead, poll for build/deploy confirmation and validate the target metric recovered before closing, flagging the job as 'pending-verification' if confirmation is unavailable in-session. — Every low grade shares the same gap: the fix was initiated but never confirmed to have held, so 'root cause fixed' is asserted without evidence. Requiring explicit post-deploy validation closes the loop between action and outcome, satisfying both the 'fix held' and 'real root-cause' rubric dimensions rather than conflating queuing a repair with completing one.
