# Post-launch hardening — review findings (optimizer + infra) 🚧

**Owner:** [[../functions/platform]] · **Parent:** hardens the autonomous build pipeline + [[storefront-optimizer-agent]] after the 2026-06-22 launch. · **Source:** two pre-A/B-approval review agents (2026-06-23). tsc clean, no active false-✅, customer-facing optimizer path confirmed SAFE; these are the 5 hardening findings. None block the A/B approval.

## Phase 1 — optimizer approval re-gate + JobStatus enum drift ✅ (highest value)
- **Re-assert scope+active on approval (TOCTOU, finding #3).** ✅ `runStorefrontOptimizerJob`'s approval-resume branch (`scripts/builder-worker.ts`) now re-loads the live policy (`loadOptimizerPolicy`) and re-asserts `isOptimizerActive` + `isProductInScope(surface.product_id)` **before** calling `materializeOptimizerCampaign` on an approved `storefront_campaign` action. If the optimizer was turned off or `product_scope` shrank between propose and approve, the action is refused (`status:"failed"` with a surfaced reason) and the campaign is NOT materialized — restoring the "every activation checks `product_id ∈ product_scope`" contract. (`evaluateProposalGate` still runs at propose time; this closes the TOCTOU window.)
- **`blocked_on_usage` enum drift (finding #2).** ✅ `src/lib/agent-jobs.ts` now includes `blocked_on_usage` in the shared `JobStatus` union *and* `ACTIVE_STATUSES`, mirroring the worker's own `ACTIVE_JOB_STATUSES`. A parked build now reads as live: `getLiveJobForSlug` returns it, the fold-guard 409s on it, and `cancelJobsForArchivedSpecs` treats it as cancellable-active — so a spec whose chained next-phase build is merely parked can no longer be folded/orphaned. (Surfaced `Record<JobStatus,…>` consumers `BuildButton`/`PlanButton` got a "Paused · usage" label + chip.)

## Phase 2 — auto-merge needs a real success gate (finding #1) ✅
`github-pr-resolve.ts` gated auto-merge on `mergeable_state==="clean"`, but the repo has **no CI workflows + no branch protection** — so GitHub reports `clean` for *any* non-conflicting `claude/*` PR. Real safety is only the build worker's pre-push `tsc --noEmit` (before the PR exists), not a verifiable PR status. A `claude/*` branch that reaches "open + mergeable" by another path (resumed build that pushed partial work, errored-post-push, manual push) auto-merged unreviewed.

✅ **Build-job success gate added.** New `getBranchBuildSuccess(branch)` in `src/lib/agent-jobs.ts`: looks up the NEWEST branch-owning `agent_job` (kinds `build`/`fold`/`pr-resolve` — the three that set `spec_branch` to the PR branch) and returns `ok:true` only when its status ∈ `SUCCESSFUL_BUILD_STATUSES` (`completed`/`merged`) — i.e. the worker drove it to `completed` only after its pre-push `tsc --noEmit` passed and the PR opened cleanly. `autoMergeReadyPrs` now requires this gate **in addition to** GitHub `clean`: a GitHub-clean PR whose owning job is `building`/`queued_resume` (partial push), `failed`/`needs_attention` (errored post-push), `needs_input`/`needs_approval` (paused), `blocked_on_usage` (parked), or that has **no owning job at all** (manual/untracked push) is refused (`buildGateBlocked++`, console-warned, surfaced in the Control Tower heartbeat) and left for the owner. Fails CLOSED (read error / missing job ⇒ refuse). The serialize guard keeps `merged===0`, so a gate-blocked PR never blocks a legitimately-built later PR in the same pass.

## Phase 3 — plan-resume busy-loop + policy RLS ⏳
- **Plan-resume repark/reclaim churn under partial cap (finding #4).** `requeueBlockedOnUsage` (`builder-worker.ts:~435`) un-parks **every** `blocked_on_usage` job when `!allAccountsCapped()`. A plan resume is pinned to its owning account (`canStartFresh=false`); if that account is capped but another is healthy → un-park → reclaim → re-park, every tick (DB churn + lane occupancy; **no token burn**). Fix: only un-park jobs whose *owning* account is now healthy (or null).
- **Optimizer policy RLS over-permissive (finding #5, needs-confirm).** The `storefront_optimizer_policy` SELECT policy is reportedly `using (auth.uid() is not null)` (any authed user reads any workspace's policy) despite a workspace-member comment. Low impact (no secrets; writes via service role; scope enforced in code) but a cross-workspace info-disclosure gap. Confirm the migration line, then scope SELECT to workspace members.

## Verification
- Shrink `product_scope` (or set `active=false`) AFTER a campaign is proposed, then approve it → it is **refused** (not materialized); with scope/active unchanged, approval materializes normally.
- A `blocked_on_usage` build counts as a live job: `getLiveJobForSlug` returns it; the fold-guard 409s on it; `cancelJobsForArchivedSpecs` cancels it.
- ✅ (Phase 2) Set a `claude/*` build job to `failed`/`needs_input`/`blocked_on_usage` (or have a `claude/*` PR with no owning `agent_job` at all) while its PR is open + GitHub-`clean`, fire a `push`/`pull_request` webhook (or call `autoMergeReadyPrs()`) → the PR is **NOT merged** (`buildGateBlocked` incremented, `[auto-merge] … build success gate refused it: …` logged), it stays open. With the owning build job `completed` and the PR clean → it auto-merges as before. Verify on the heartbeat: `produced.buildGateBlocked` reflects the refusals.
- Owner account capped + another healthy → a pinned plan-resume job is NOT repeatedly un-parked (no repark/reclaim churn); a build (start-fresh-able) still resumes.
- A non-member authed user cannot SELECT another workspace's `storefront_optimizer_policy`.
