# Harden the Vault agent — roll persistent coaching into its mandate

**Owner:** [[../functions/platform]] · **Parent:** [[platform-director-agent]] — graduate persistent coaching into a durable fix (director grades agents: low score → fix spec, never a CEO escalation)
**Found in use 2026-07-02:** the **Vault** agent (`security-review`) sits at **5.0/10** after **2** coaching attempts that didn't stick. Rather than escalate to the CEO, roll the coaching into a permanent fix so the agent improves at the mandate level.

## The accumulated coaching to bake in (now archived as rolled-into-mandates)
- **When Vault repeatedly validated code locally but missed that downstream consumers lacked the authority context binding the action or data to its exact command, tenant, audience, or lifecycle state.:** When a diff changes an approval action, persisted DB row, Inngest event, or scoped query, trace producer → storage → consumer/audience before emitting the parseable security-review JSON verdict persisted to the agent_jobs row; require exact command visibility, workspace/spec filters, allowlisted persisted diagnostics, and final authorization-state checks because validation at only one stage does not secure the resulting action. — scripts/builder-worker.ts:10533 stored cmd separately from preview while src/components/agents/role-inbox.tsx:832 rendered preview instead of cmd; src/lib/agent-jobs.ts:637 and :895 queried newer work without workspace/spec bindings. The same missing-boundary pattern appears where scripts/builder-worker.ts:6342 persisted raw stderr into the authenticated-readable policy at supabase/migrations/20260619140000_worker_heartbeats.sql:20 and where src/lib/agent-jobs.ts:597 omitted the in_review gate despite src/lib/spec-card-state.ts:618 firing before disposition.
- **When All 7 slips ended with no parseable verdict envelope — the security review was either capped/restarted before it could finish, or (account-matching-indexed-split, grade 2) finished the investigation but emitted prose/tooling-noise instead of one of the four required JSON verdict shapes, so nothing gradeable ever reached the worker.:** When you start a security-review, treat the JSON verdict envelope as the ONLY deliverable, not the prose: decide a provisional verdict FIRST, keep your FINAL message to exactly one JSON object — {"status":"clean|false-positive|needs-human|real-vuln",...} with no text/markdown before or after — and the instant completion is at risk (deep investigation, tool errors, budget/timeout or usage pressure) fall straight to {"status":"needs-human","review":"<one line why I couldn't finish>"} rather than continuing to investigate. A minimal parseable verdict always outscores thorough findings that trail off without an envelope. — builder-worker.ts:13201-13204 defines those four envelope shapes as the sole gradeable output and 13708-13713 fails/parks the job whenever none is parsed (resolveReviewVerdict fallback at 1654-1664); every one of the 7 slips hit exactly that no-verdict branch, and the account-matching 'tooling failure (agent produced no verdict)' case shows even the same-session repair re-prompt at 13224 never used its needs-human escape — emitting the envelope early and defensively is the one worker-controllable lever.

## Phase 1 — bake the coaching into the agent
- Make the above coaching PERMANENT behavior of the `security-review` agent — fold it into its prompt/mandate/code (its run-job + prompt in `scripts/builder-worker.ts` and the relevant `src/lib/agents/*`), not ephemeral appended `agent_instructions`. Once baked, the agent should follow it by default.
- Verify the agent's grade rollup recovers (≥ 7/10) over the next window of graded actions; if a coaching class still recurs, the bake-in missed it.
