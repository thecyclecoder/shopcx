# Spec-Test Agent (box QA over shipped-unverified specs) ✅

**Owner:** [[../functions/platform]] · **Parent:** Platform mandate "Autonomous build platform" (box-agent family with [[box-spec-chat]] · [[box-ticket-improve]] · [[box-escalation-triage]]). Fills the **Shipped → Verified** gap in [[../project-management]].

A box agent that **tests shipped-but-unverified specs against their own `## Verification` checklist** and reports what passed, what failed, and **what still needs a human** — so the owner's "Mark verified & archive" decision is fast and evidence-backed instead of a manual re-test of every shipped feature. It **never** auto-verifies (that human gate stays) and **never** mutates prod — it runs only non-destructive checks and surfaces the rest.

**Outcome:** a shipped spec arrives at the verify gate pre-tested: "8/10 auto-checks ✅, 1 ✗ (looks broken — here's the evidence), 1 👤 needs you to eyeball the page." The owner verifies in seconds, or sees a real regression before archiving.

## Why this fits
[[../project-management]]: **Shipped (✅)** = the build deployed the code (automated). **Verified** = the owner tested it in prod and it works (human, owner-only, *never* automated). The gap is a list of "live but not yet prod-verified" specs sitting in the board's "Shipped — awaiting verification" column, each with a `## Verification` checklist the build wrote. The agent executes the *automatable* bullets of that checklist so the human only does the parts that genuinely need them.

## Trigger + scope
- **Event trigger (primary, [[spec-test-on-ship]]):** the moment a spec transitions to **shipped**, its `spec-test` job is enqueued — no waiting for the daily sweep. Two ship paths, both hit the shared `enqueueSpecTestIfDue` guard in [[../libraries/agent-jobs]]: a **manual** status/phase flip (`POST /api/roadmap/status`, when the resulting `deriveStatus` is `shipped`) and a **build-merge** (`reconcileMergedJobs`, when a merged build's spec is now shipped).
- **Daily Inngest cron** (`spec-test-cron`, the [[../inngest/triage-escalations]] enqueue precedent) is now the **backlog catch-all** — it re-checks every **shipped-but-not-archived** spec (derived status `shipped` ([[../libraries/brain-roadmap]] `deriveStatus`) AND the spec file still in `docs/brain/specs/`, no `archive.d/{slug}.md`) and enqueues only those the event missed (box down / shipped before the event existed), via the same `enqueueSpecTestIfDue` guard.
- **Shared dedupe:** the guard skips a `(workspace, slug)` with an **in-flight** `spec-test` job **or** a **fresh** [[../tables/spec_test_runs]] row (~20h) — so a build-merge + a manual tweak + a cron tick in one window run **once**. Re-running is allowed once the spec changes again (a new ship state). Also on-demand: a **"Test now"** button on the shipped spec card.
- Concurrency-1 `MAX_SPEC_TEST` lane on the box.

## What it does (per spec, on Max)
A top-level `claude -p` on Max (web search on, no `ANTHROPIC_API_KEY`) in a repo checkout reads the spec + its `## Verification` bullets and, for each bullet, **classifies then acts**:
- **Auto-testable (non-destructive) → run + record ✅/✗ with evidence:** code/`Grep`/`Read` assertions ("route X exists / column Y selected"), **DB-state reads** (probe the table/row the bullet names), **GET / read-only API** calls, migration-applied probes, `tsc`/build/config presence, **role/RLS checks** (e.g. "as viewer → 403"). Each verdict carries the concrete evidence (the query result, the HTTP status, the file:line).
- **Needs human → flag, never execute:** anything **visual/UX** ("looks right", "renders the badge", "the page looks fantastic"), any check that would **mutate prod** (send a message, place an order, charge a card, fire a real email/SMS), or any **fault-injection / forced-failure** check ("force X to fail → expect the error handling") the read-only agent can't exercise — these are surfaced as "needs human testing," not run.
- **Inconclusive → flag** (couldn't determine without a side effect or missing fixture, or a genuinely ambiguous bullet).
- **`fail` requires positive breakage evidence ([[spec-test-classification]]):** a `fail` means the agent ran a non-destructive check and *observed* the feature doing the wrong thing — never "I couldn't verify this read-only." A real check that needs a forced fault / mutation / human eyes is `needs_human` (with a "code present at file:line; needs forced fault to confirm" note when the code plainly satisfies the bullet), not `fail`. Only evidence-backed `fail`s become regressions or flip the verdict to `issues`; `needs_human`/`inconclusive` never do. This keeps the Regressions list meaningful instead of filling with phantom regressions.

## 🚨 Guardrails (supervisable autonomy)
- **Read-only / non-destructive ONLY.** The agent never writes prod, never sends a customer message/email/SMS, never creates an order, never flips a spec to verified. It optimizes a bounded proxy — "the automatable checks pass" — and the **human owns the verify decision**. Reaches prod through read-only deterministic tools (DB reads, GET endpoints); any mutating verification path is auto-classified human-only.
- **It stamps, it doesn't gate.** The agent **never** marks a spec verified or archives it (that owner-only gate stays untouched). But it **applies its own "stamp of approval"** — an `agent_verdict` on the run: **`approved`** when zero auto-checks failed (its non-destructive coverage passed), **`issues`** when something auto-failed, plus a separate "human checks pending" flag. The stamp is a distinct **"Agent-tested ✅"** badge next to (not replacing) the human **Verified** state — a CEO→role→tool signal that "the bot checked the automatable parts and they hold," which the owner then confirms.
- **An auto-✗ on a shipped spec is high-signal** (the feature shipped but fails its own verification = a regression or an incomplete build) — surface it loudly; (stretch) propose a fix spec via the [[box-spec-chat]] finalize path rather than silently.

## Tooling on the box (this agent needs broad read access — runs ON the box)
The agent **must run on the remote box** (it's where the toolchain + creds live; never a Vercel function). Its non-destructive QA toolkit, all on the box:
- **Repo** — the working checkout for `Read`/`Grep`/`Glob` + `tsc`/build (does the route/column/component the bullet names actually exist + compile).
- **GitHub** (`gh` CLI + `GITHUB_TOKEN`, already on the box) — read PR/commit/**CI check** status for the spec's build, confirm the merge landed.
- **Vercel CLI** (`vercel` + a read-scoped `VERCEL_TOKEN`) — confirm the deploy that should carry the feature is **READY**, read **build + runtime logs** (catch a route 500ing in prod), check **env-var presence** (`vercel env ls`), and resolve the prod/preview URL to hit. **Provisioning:** install the Vercel CLI on the box + add `VERCEL_TOKEN` to the worker env (`/root/shopcx-worker.env`); `gh`/`GITHUB_TOKEN` already present ([[../recipes/build-box-setup]]).
- **Prod DB (read-only)** — service-role **reads** to probe the table/row/column a bullet asserts (never writes).
- **HTTP** — **GET / read-only** endpoint hits (prod + Vercel preview URL) for status/shape; role/RLS checks (expect 403 as a viewer).
- **WebSearch** when a check needs external context.
All read-only / non-destructive (see guardrails). The box keeps its secrets for this agent (unlike a code-build, which is secret-stripped) — it needs them to inspect prod — but the agent is constrained to reads.

## Data + surfacing
- **`spec_test_runs`** (new table): `id, workspace_id, spec_slug, run_at, agent_verdict (approved｜issues｜needs_human｜error), summary {auto_pass, auto_fail, needs_human, inconclusive}, checks jsonb [{text, verdict: pass｜fail｜needs_human｜inconclusive, evidence}], transcript`. One row per run; latest wins. The `error` verdict is the no-parseable-verdict terminal state added by [[spec-test-json-robustness]] (see below).
- **Primary surface — a `Developer` sidebar section with a `Spec Tests` (QA) page** (`/dashboard/developer/spec-tests`): the agent's report home — every shipped-unverified spec with its latest run: the **"Agent-tested ✅ / ⚠️ issues"** stamp, the auto pass/fail/needs-human counts, each check's verdict + evidence (expandable), and an aggregated **"Needs human testing"** list across all specs. A **"Test now"** button per spec. (New top-level `Developer` nav group — house future dev/QA tools here too.)
- **Roadmap board:** each "Shipped — awaiting verification" card gets a compact **test chip** (`✅ 8 · ✗ 1 · 👤 1`) + the **Agent-tested** stamp, linking to the Developer page.
- **VerificationCard** (`src/app/dashboard/roadmap/VerificationCard.tsx`, beside "Mark verified & archive"): render each checklist bullet with its auto-verdict + evidence inline + a distinct **"Needs human testing"** list — so the owner does only those, then verifies.

## Verification
- On the box, apply the migration (`npx tsx scripts/apply-spec-test-runs-migration.ts`) → expect `✓ spec_test_runs table present: true`; re-running it is a no-op (idempotent `IF NOT EXISTS`).
- In the dashboard sidebar (as owner), open **Developer → Spec Tests** (`/dashboard/developer/spec-tests`) → expect a page listing every shipped-but-unverified spec, each with a **Test now** button; non-owners don't see the nav item (owner-gated). A spec not yet tested shows "not yet tested".
- On a shipped spec's Spec Tests row, click **Test now** → expect a `POST /api/roadmap/spec-test` returning `{queued:true}` and a `kind='spec-test'` row inserted in `agent_jobs`; clicking again while it's in flight returns `{queued:true, already:true}` (no duplicate job).
- After the box runs the job (or `npx tsx scripts/builder-worker.ts` claims it), query `select agent_verdict, summary, jsonb_array_length(checks) from spec_test_runs order by run_at desc limit 1` → expect one row whose `summary` counts match the `checks` length and `agent_verdict` ∈ `approved|issues|needs_human` (re-derived from checks: `issues` iff any check is `fail`).
- On `/dashboard/roadmap`, a shipped card with a run → expect the compact **Agent-tested** stamp + `✅·✗·👤·?` chip linking to the Spec Tests page; on `/dashboard/roadmap/{slug}` the VerificationCard shows per-bullet verdicts with expandable evidence + a distinct **👤 Needs human testing** list.
- On the box, `npx tsx scripts/spec-test-db-probe.ts "select 1"` → expect JSON rows; `npx tsx scripts/spec-test-db-probe.ts "update spec_test_runs set agent_verdict='x'"` → expect it to refuse (read-only: rejects non-SELECT / mutating keywords) and exit non-zero.
- Confirm the cron is registered: `grep specTestCron src/app/api/inngest/route.ts` → present; the function id is `spec-test-cron` with cron `45 10 * * *`.
- Negative: a run never sends a message / creates an order / charges a card (mutating bullets are recorded `needs_human`, not executed); the agent never flips a spec to verified or writes to `archive.d/` — the owner's **Mark verified & archive** gate is untouched. API console stays flat (Max, no `ANTHROPIC_API_KEY`).

### Phase 2 — human-test queue + regression escalation
- On the box, apply the migration (`npx tsx scripts/apply-spec-test-human-checks-migration.ts`) → expect `✓ spec_test_human_checks table present: true`; re-running it is a no-op (idempotent `IF NOT EXISTS`).
- In the sidebar (as owner), **Developer → Human-test queue** (`/dashboard/developer/spec-tests/human-queue`) is present with an amber count badge = waiting `needs_human` checks + regressions; non-owners don't see it (owner-gated, `GET /api/developer/spec-test/human-queue` returns 403 for non-owners).
- On the Human-test queue, every `needs_human` check across shipped-unverified specs appears under **👤 Needs human testing** with its spec title (→ `/dashboard/roadmap/{slug}`) + expandable evidence. Click **✓ Tested** → expect a `POST /api/developer/spec-test/human-queue` `{ slug, check_key, check_text, resolution:"verified" }` returning `{ ok:true }`, a row upserted in `spec_test_human_checks`, and the check moving to the collapsed **Done** group. **Re-open** there → `{ clear:true }` deletes the row and the check returns to waiting.
- Integrity: `POST` with a `check_key` that isn't `checkKey(check_text)` → expect `400 check_key does not match check_text` (no arbitrary keys written).
- On a regressed spec (latest run `agent_verdict='issues'`), the queue's top **⚠️ Regressions** banner lists it with its failing checks + a **Propose fix spec** button (also on `/dashboard/developer/spec-tests` and the spec's VerificationCard). Click it → expect a `POST /api/roadmap/chat` `{ action:"propose_fix", slug }` returning `{ queued:true, chatId }`, a new `roadmap_chats` thread titled `Fix: {slug}` seeded with the failing checks, and a `kind='spec-chat'` `agent_jobs` turn row enqueued. `propose_fix` on a spec with no failing checks → expect `400 no regression to fix`.
- Negative: the resolve API only writes the owner's own `spec_test_human_checks` rows (never `spec_test_runs`, never a spec file); `propose_fix` only seeds a chat + enqueues a turn — it never authors/commits a spec itself (the owner finalizes). The agent never writes `spec_test_human_checks`.

## Phase 1 — sweep + auto-checks + report ✅
`spec-test` job kind + `MAX_SPEC_TEST` lane + `runSpecTestJob`; box provisioning (install Vercel CLI + `VERCEL_TOKEN` in the worker env; `gh`/`GITHUB_TOKEN` already there); the `spec-test` skill (classify + **non-destructive** execution of `## Verification` bullets using the box toolchain — repo/`tsc`, `gh` CI, `vercel` deploy+logs+env, DB reads, GET endpoints); `spec_test_runs` table with the `agent_verdict` stamp; the daily cron; the new **`Developer → Spec Tests`** page; and the board test-chip + Agent-tested stamp + VerificationCard per-bullet verdicts. "Test now" on-demand button.

## Robustness — strict JSON contract + parse-repair + `error` state ✅ ([[spec-test-json-robustness]])
The skill emits **only** the result JSON (fenced/last, no prose; explicit schema + one-shot example in `SKILL.md`). `runSpecTestJob` extracts it with `extractSpecTestResult` (whole-message → last fenced block → last balanced `{...}` scan, tolerating surrounding prose), **re-prompts once** on the same session for *only* the JSON on a parse failure, and — if it still can't parse, the agent reports `{"status":"error"}`, or the worker throws — writes the run with **`agent_verdict='error'`** (empty checks/summary, reason in `error`) instead of a silent 0-check `approved`/empty row. The Developer → Spec Tests page renders `error` as **"Run errored — retry"** with the raw output tail, and **Test now** re-runs it. Summary counts always derive from `checks[]`, so an empty `checks[]` can never read as a clean pass. See [[../tables/spec_test_runs]].

## Classification — `fail` needs breakage evidence, not "couldn't verify" ✅ ([[spec-test-classification]])
The verdict rules were tightened so the Regressions list stays meaningful. `fail` requires **positive evidence of breakage** — the agent ran a non-destructive check and *observed* the feature doing the wrong thing. A real check the agent can't run read-only (needs **fault injection / forcing a failure** like "force unparseable output → expect `error` state", a **mutation**, or **visual/UX** judgment) is `needs_human` — a human can verify it — **never `fail`**; when the implementing code plainly satisfies a bullet but the runtime path needs a forced fault, the agent prefers `needs_human` with a `"code present at file:line; needs forced fault to confirm"` note. Genuinely undeterminable → `inconclusive`. The Regressions list + the `issues` verdict are driven **only** by evidence-backed `fail`s; `needs_human`/`inconclusive` never appear as regressions or flip the verdict to `issues` (`normalizeSpecTest` derives `issues` iff `auto_fail>0`; `getHumanTestQueue` keys regressions off `fail` checks only). The empty/uncertain-run summary guard is unchanged (no `pass` → `needs_human`, never `approved`). Built in the `spec-test` `SKILL.md` (Step 1 classify + Step 2/3 rules) + the `runSpecTestJob` classification prompt.

## Maximize machine coverage — local-harness mode + narrowed `needs_human` + spec-file green-check ✅ ([[spec-test-maximize-machine-coverage]])
The founder's mandate — *"if a machine can test it, the machine does it"* — drove three changes so the agent stops over-deferring:
- **Third execution mode — non-destructive local harness/replay.** Beyond read-only repo/DB/HTTP probes, the agent MAY author a throwaway `_`-prefixed scratch script (never committed) that imports a pure function / parser / classifier / validator from `src/` and exercises it **locally**, including **fault injection** (malformed payload, forced unparseable output). No prod write, no network side-effect → it is `auto`, with the harness input+output as evidence. This converts the whole fault-injection bucket from `needs_human` to `auto` whenever the logic is reachable as a local unit.
- **Outcome-not-action framing.** For a *"do X (mutation) → expect observable Y"* bullet the agent verifies **Y** read-only (a prod row already in the expected state from real traffic, a populated column, a rendered context string) instead of deferring because **X** mutates. Only an observable that requires the agent itself to perform an irreversible prod mutation (no real-traffic instance) stays `needs_human`.
- **`needs_human` narrowed to two cases** — (a) genuine visual/aesthetic judgment, (b) an irreversible prod side-effect with no already-observable evidence AND no local-harness equivalent (a real SMS/email/charge actually hitting an external carrier/processor). The tie-breaker is now *"attempt a read-only outcome probe, then a local harness; defer ONLY if both are impossible."* The read-only-against-PROD ban and the `fail`-needs-breakage-evidence rule are untouched.
Built in the `spec-test` `SKILL.md` (Step 1 classify + Step 2 toolkit) + the `runSpecTestJob` classification prompt.

## Spec-file green-check writeback — green ✅ accumulates on the spec markdown ✅ ([[spec-test-maximize-machine-coverage]] Phase 3)
A green check used to live only in the DB (`spec_test_runs.checks[].verdict='pass'` for agent passes; `spec_test_human_checks` resolution='verified' for owner-marked) — the spec markdown never changed, so the founder never watched the file turn green before archive. Now:
- **A `## Verification` bullet is green** when its latest-agent check is `pass` OR the owner marked it `✓ Tested`. Derived per bullet by `deriveGreenBullets` (keyed by the existing `checkKey` hash so it survives re-runs) in [[../libraries/spec-test-runs]].
- **`reflectSpecGreenChecks` ([[../libraries/spec-green-writeback]])** annotates each green bullet in `docs/brain/specs/{slug}.md` with a **leading ✅** and strips it from any non-green bullet, then commits to `main` via the GitHub Contents API — a content writeback gated like other box commits (it only ever rewrites the leading ✅ of verification bullets, never the spec logic). It fires from **two triggers**: the box `runSpecTestJob` after a run lands (agent passes), and the owner-gated human-queue `POST` when the owner marks/clears a check.
- **The VerificationCard ([[../dashboard/roadmap]])** renders each bullet's live green state (a Verification-progress checklist + a per-row ✅/◻︎) so the founder watches it fill in, and shows an **"all checks green — ready to archive"** banner when every bullet is green — making "Mark verified & archive" a confident one-click.

## Headless-browser verification — the BROWSER check category ✅ ([[spec-test-deep-verification]] Phase 1)
A **fourth non-destructive execution mode** so RENDERED-UI bullets stop landing in `needs_human`. The agent can only `GET` endpoints otherwise, so "the page shows X / click → Y" bullets got punted; now a bullet asserting **rendered DOM** or a **non-mutating click-flow** ("the card shows the Agent-tested stamp + chip", "VerificationCard renders per-bullet verdicts", "composer textarea ≥5 rows / auto-grows", "a non-owner doesn't see the nav item / gets a 403", "open the tab → see X") is a **BROWSER check** — `auto`, `pass`/`fail` **with a screenshot as evidence** — not `needs_human`. A *rendered fact* is a browser check; only a *taste* judgment ("does it look good") stays visual `needs_human`.
- **The tool — `scripts/spec-test-browser-check.ts`** (the read-only browser power, companion to `scripts/spec-test-db-probe.ts`). It launches headless chromium (the box has it provisioned via `npx playwright install --with-deps chromium`; `playwright` is in `package.json`), **mints an OWNER Supabase session server-side** (the service-role admin `generateLink({type:'magiclink'})` → `verifyOtp` → a real session, encoded into @supabase/ssr's exact `base64-`/chunked cookie layout + the `workspace_id` cookie the middleware gate needs — **NO human creds / no password**), loads the owner-gated page authed, runs the requested assertions (`--assert-text` / `--assert-selector` / `--assert-redirect` / `--expect-status` / benign `--click`), and uploads a full-page screenshot to the private `spec-test-evidence` bucket. Stdout JSON carries `pass`, per-assertion results, and the screenshot's storage `path`; the agent copies that into the check's `screenshot` field.
- **🚨 Read-only firewall (the core safety rule, carried from the parent spec).** The browser session is **owner-authed but READ-ONLY on prod** — navigate + assert + benign clicks (expand a section / open a tab) ONLY. It NEVER fills or submits a form that mutates real customer/billing data (the tool has no input-typing/form-submit capability by construction); a bullet that can only be verified by a real customer/$/Meta/external mutation stays `needs_human` (or [[spec-test-deep-verification]] Phase 2's sandbox). Use `--role anon` for "unauthenticated/non-owner is gated → redirect to /login or 401/403".
- **Screenshot evidence rendering.** `spec_test_runs.checks[].screenshot` holds the **private-bucket storage path** (never a public URL — a dashboard screenshot can contain real customer data). The **Developer → Spec Tests page** signs it per-render (short-TTL signed URL via `signSpecTestScreenshot` in [[../libraries/spec-test-runs]]) and renders the image inline under the check's expandable evidence. See [[../dashboard/roadmap]] + [[../tables/spec_test_runs]].
## Sandboxed behavioral verification — the SANDBOX check category + the external firewall ✅ ([[spec-test-deep-verification]] Phase 2)
A **fifth non-destructive execution mode** so INTERNAL-ONLY behavioral bullets stop landing in `needs_human`. A bullet shaped *"fire event / call POST / approve → assert DB rows + `customer_events`/`triage_runs`"* whose **every side effect is internal** (a DB write or an Inngest event) is now a **SANDBOX check** — `auto`, pass/fail — driven on **dedicated `is_test` fixtures**, asserted, then cleaned up. Real-customer/$/external bullets stay `needs_human`.
- **🚨 External side-effect firewall (the core safety rule).** Drive a flow ONLY if you can PROVE every side effect stays internal. A flow that would call an **external API with real effect** — Amplifier fulfillment, a Braintree charge/refund, an Appstle mutation, a Resend/Twilio send, a live Meta ad pause — is **NOT run**; it stays `needs_human`. Three layers enforce it: the `INTERNAL_ONLY_FLOWS` allowlist (each flow declares the fixture precondition guaranteeing its internal-only branch), `assertTestWorkspace` (refuses any workspace where `is_test` is not true), and the `is_test` tenant carrying **NO external credentials** (defense in depth). Comp renewal is the canonical case: the **fail-closed** branch (comp sub + `comp_role` null → failed `type='comp'` txn + `subscription.comp_renewal_failed` event, no order, no advance) is fully internal → SANDBOX; the **happy** branch's **Amplifier handoff is the external edge** → assert up to it, flag the handoff itself human.
- **The fixtures — no new table.** A dedicated `is_test` workspace ([[../tables/workspaces]] `is_test` sentinel, migration `20260622120000`) plus a comp customer (`comp_role` null), a comp subscription, a ticket, and a migration_audit — all under that workspace, seeded idempotently by `scripts/seed-spec-test-fixtures.ts` (a **gated** owner-approved run — the box has no prod creds). Shared identity in [[../libraries/spec-test-sandbox]] `SPEC_TEST_FIXTURES`.
- **The toolkit — `scripts/spec-test-sandbox.ts`** (the behavioral power, companion to the read-only `spec-test-db-probe.ts` + `spec-test-browser-check.ts`): `info` (fixtures + the internal-only flow registry + forbidden external APIs), `isolation` (the non-test-workspace fingerprint), `fire <flowId>` (fire an allowlisted internal-only Inngest event — e.g. `comp-renewal-failclosed` — after it proves the precondition; asserts the rows/events and reports `isolation.zero_non_test_workspace_writes`), `post <flowId> [--body|--clear]` (call an allowlisted internal POST with **owner cookies** minted server-side, scoped to the `is_test` workspace — e.g. `human-queue-resolve`, `roadmap-answer`), `cleanup` (idempotent reset). Every mutation is gated on `assertTestWorkspace`.
- **Isolation proof.** `nonTestWorkspaceFingerprint` counts rows + newest timestamp per mutable table for everything that is NOT the `is_test` workspace; an unchanged fingerprint across a run proves **zero writes to non-test-workspace rows** (`isolation.zero_non_test_workspace_writes:true`). Built in [[../libraries/spec-test-sandbox]] + the `spec-test` `SKILL.md` (Step 1 classify + Step 2 toolkit) + the `runSpecTestJob` classification prompt.

## Phase 2 — human-test queue + regression escalation ✅
The **Human-test queue** (`/dashboard/developer/spec-tests/human-queue`, modeled on [[improve-queue]]) aggregates every `needs_human` check across the latest run of every shipped-unverified spec — the parts only the owner can run. Each waiting check has **✓ Tested** / **Dismiss**; resolutions persist in the new [[../tables/spec_test_human_checks]] table (keyed by a stable `check_key` hash of the bullet, so they survive re-runs), with a collapsed **Done** group + **Re-open**. The sidebar carries an amber count badge (waiting + regressions). **Regression escalation:** a shipped spec whose latest run is `agent_verdict='issues'` (it failed its own spec-test) surfaces loudly — a **Regressions** banner at the top of the queue + a **Propose fix spec** button (queue, Spec Tests page, and VerificationCard). It POSTs `{ action:"propose_fix" }` to `/api/roadmap/chat`, which seeds a box authoring chat (`Fix: {slug}`) with the failing checks + evidence as the brief and kicks off the first box turn ([[box-spec-chat]]); the owner reviews + finalizes it into a fix spec (the agent proposes, the owner finalizes — supervisable autonomy). New surfaces: `getHumanTestQueue` + the `check_key`/resolution helpers in `src/lib/spec-test-runs.ts`, `GET`/`POST /api/developer/spec-test/human-queue`, the `propose_fix` action, and the `ProposeFixButton` component.
