# `goals` + `goal_milestones` tables + `specs.milestone_id` FK + one-time backfill

**Owner:** [[../functions/platform]] · **Parent:** [[../goals/db-driven-specs]] M5 — Goals + milestones are data too (the top two tiers)
**Blocked-by:** [[spec-body-table-and-backfill]]

The foundation step for the goal side of [[../goals/db-driven-specs]]. Stand up the two relations that hold the top two tiers of the [[../project-management|Goal → Milestone → Spec → Phase]] hierarchy as data: `public.goals` (id, parent_goal_id self-ref nullable — a **SubGoal is just a goal with a `parent_goal_id`**, NOT a separate table, per the CEO-locked design contract; status `proposed｜greenlit｜complete｜folded`; title; body; owner; success_metric; outcome; proposer_function) and `public.goal_milestones` (id, goal_id FK, position, status, title, body). Specs gain a `milestone_id` FK column (already declared nullable in [[spec-body-table-and-backfill]]'s schema — this spec is where it gets a foreign-key constraint pointing at `public.goal_milestones(id)` and where the backfill populates it). Status at every tier ROLLS UP from children: `goal_milestones.status` from its child `public.specs.status` rows (which already roll up from `spec_phases`, per [[spec-body-table-and-backfill]]); `goals.status` from `goal_milestones.status` (with the explicit `proposed → greenlit` flip the CEO controls — see [[goal-greenlight-button-and-author-writes-db]]). Never the `**Status:**` markdown line or `⏳/✅` emojis. One-time backfill: parse `docs/brain/goals/*.md` (the existing [[../libraries/brain-roadmap]] `parseGoal` L868 — used ONE LAST TIME) into rows; `goal_milestones` rows come from the parsed `## Decomposition` sub-sections (one milestone per `### M{N} — title`). The `.md` files stay authoritative until [[goal-readers-from-db-retire-parsegoal]] cuts readers over. Closes the gap from [[../goals/db-driven-specs]] Outcome ("the CEO literally had no surface to approve the db-driven-specs goal; it had to be git-committed by hand") and Phase 3 of [[director-proposed-goals]] (today the proposed→greenlit flip is a markdown `setGoalStatusLine` write + commit + Vercel deploy); today `getGoals` ([[../libraries/brain-roadmap]] L1004) reads `docs/brain/goals/*.md` directly, and `parseGoal` is the goal-side twin of the spec-side `parseSpec` retired in [[spec-readers-from-db-retire-parser]].

## Phase 1 — schema migration for `public.goals` + `public.goal_milestones` + `specs.milestone_id` FK
- Author `supabase/migrations/{YYYYMMDDNNNNNN}_goals_and_goal_milestones.sql` creating two new `public` tables.
- **`public.goals`** columns: `id uuid pk default gen_random_uuid()`, `workspace_id uuid not null references workspaces(id) on delete cascade` (mirrors [[spec-body-table-and-backfill]] `specs`), `slug text not null` (the `docs/brain/goals/{slug}.md` key), `title text not null`, `body text not null` (the goal's outcome + why + model + target — the full body the goal markdown carries today), `outcome text` (the one-paragraph **Outcome:** line as a separate column for the board summary), `success_metric text` (the **Success metric:** line — the planner's gap-analysis anchor, per [[../skills/plan-goal]]), `owner text not null` (the function slug — `growth｜cmo｜retention｜cfo｜logistics｜cs｜platform`), `proposer_function text` (the **Proposed-by:** function — set by [[director-proposed-goals]] when a director authored the goal; null for CEO-authored goals), `parent_goal_id uuid references public.goals(id) on delete cascade` (NULLABLE self-ref — a SubGoal is just a goal with a parent, per the CEO-locked design contract), `status text not null default 'proposed' check (status in ('proposed','greenlit','complete','folded'))`, `created_at timestamptz not null default now()`, `updated_at timestamptz not null default now()`.
- **Unique index** on `(workspace_id, slug)` — the upsert spine.
- **Index** on `parent_goal_id` for the board's nested-goal render (CEO Mode ▸ Fully Autonomous CTO ▸ M1…M5 ▸ specs ▸ phases).
- **`public.goal_milestones`** columns: `id uuid pk default gen_random_uuid()` (STABLE across reorders — same lift-a-thing rule as [[spec-body-table-and-backfill]] `spec_phases`), `goal_id uuid not null references public.goals(id) on delete cascade`, `position int not null` (1-indexed), `title text not null` (e.g. "M1 — The spec body in the DB"), `body text` (the milestone description + sub-bullets the goal markdown's `### M{N}` block carries), `status text not null default 'planned' check (status in ('planned','in_progress','complete'))`, `created_at timestamptz not null default now()`, `updated_at timestamptz not null default now()`.
- **Unique index** on `(goal_id, position)`.
- **Add `milestone_id` FK constraint** on `public.specs` (the column already exists per [[spec-body-table-and-backfill]] schema; this migration adds `references public.goal_milestones(id) on delete set null`). A spec without a milestone (a standalone fix-spec or regression) keeps `milestone_id=null` — that's the intended shape.
- RLS: enable on both; authenticated users read; service role full access (mirrors [[../tables/spec_card_state]] / `public.specs`).
- DB function `public.roll_up_milestone_status(milestone_id uuid)`: reads all `public.specs` rows where `milestone_id=$1` and rolls up — any `in_progress` → `in_progress`; all `shipped|folded` → `complete`; otherwise `planned`. A row-level trigger on `public.specs` (after update of `status` or `milestone_id`) calls it. `public.roll_up_goal_status(goal_id uuid)`: reads all `public.goal_milestones` rows where `goal_id=$1` — all `complete` → consider the goal `complete` (but DO NOT auto-flip `proposed → greenlit` — that's the CEO action in [[goal-greenlight-button-and-author-writes-db]]); only flip `greenlit → complete` automatically (so a still-`proposed` goal doesn't sneak to `complete` without ever being greenlit, which would be a rail break). A trigger on `public.goal_milestones` after status update calls it.
- Apply via `scripts/apply-goals-tables-migration.ts` per [[../recipes/write-migration]]. Probe with `scripts/_verify-goals-schema.ts` per [[../recipes/verify-schema]].

## Phase 3 — one-time backfill from `docs/brain/goals/*.md` → `goals` + `goal_milestones`
- Author `scripts/backfill-goals-from-markdown.ts` (per [[../recipes/script-conventions]] — `_bootstrap.ts`, dry-run by default, `--apply` to write, idempotent + resumable). Loads every `docs/brain/goals/*.md`, runs [[../libraries/brain-roadmap]] `parseGoal` (the EXISTING parser, used ONE LAST TIME), and INSERTs `goals` + `goal_milestones` rows.
- For each goal slug: carry `parseGoal`'s parsed `title / outcome / proposedBy / status / body`. The `parent_goal_id` self-ref is set per the goal markdown's wikilink shape ("Reports to [[ceo-mode]]" → parent goal slug `ceo-mode`; the backfill does a second pass after the initial INSERT to resolve parent slug → parent uuid).
- `goal_milestones` rows come from `parseGoal`'s milestone list (the `### M{N} — title` sub-sections under `## Decomposition`). `body` is the markdown block under that heading. `position` is the order parsed. `status='complete'` if every child spec is `shipped`/`folded` at backfill time, `in_progress` if any is in progress, else `planned`.
- `specs.milestone_id` backfill: a SECOND pass after `goal_milestones` is populated — walk each `public.specs` row, look up its `parent` text against the parsed goal milestones, set `specs.milestone_id` where there's an unambiguous match. The director/Pia-authored specs already carry a `**Parent:** ../goals/{slug}|{milestone-title}` shape — the backfill matches on the milestone title or "M{N}" prefix. Specs that don't match any milestone (a standalone fix-spec under a function mandate, e.g. [[agent-mandate-hardening-ticket-improve]]) get `milestone_id=null` (the intended shape — they hang off a function mandate, not a milestone).
- Idempotent: re-running on an already-backfilled goal is a no-op (UPSERT by `(workspace_id, slug)`, REPLACE `goal_milestones` only when the milestone signature changed — same id-by-position rule as [[spec-body-table-and-backfill]]).
- Print a per-goal line: `backfilled {slug}: {N} milestones, status={X}, {M} specs attached`. Dry-run prints what WOULD insert. On `--apply`, after the loop completes, verify every `goals` row's status matches the markdown parse — flag a mismatch for human review.
- Out of scope: deleting the `docs/brain/goals/*.md` files. Out of scope: rewiring readers ([[goal-readers-from-db-retire-parsegoal]]). Out of scope: the CEO greenlight UI ([[goal-greenlight-button-and-author-writes-db]]). Out of scope: fold ([[goal-fold-from-db-row]]).

## Safety / invariants
- **Database is the spec.** Probe the live schema after `--apply` via `scripts/_verify-goals-schema.ts` (per [[../recipes/verify-schema]]). Never assume; the [[../tables/]]_brain page is the contract.
- **Workspace-scoped, RLS-protected.** Every read carries `workspace_id`; every write goes through `createAdminClient()` (service role). Mirrors [[spec-body-table-and-backfill]].
- **`parent_goal_id` is acyclic.** A goal CANNOT be its own ancestor. Add a CHECK trigger that walks the parent chain on INSERT/UPDATE and rejects a cycle — the design contract calls out optional-subgoals + reassignment ("a goal CAN be (re)assigned under another goal at any time"), so the move is one UPDATE, but cycles must be rejected at the rail.
- **Milestone IDs are stable.** Reordering or retitling a milestone in the parsed source UPSERTs by `(goal_id, position)` preserving `id`. A jsonb-style destroy+recreate would break any `public.specs.milestone_id` FKs pointing at the milestone — the FK is `on delete set null`, so a destructive write would silently unattach specs.
- **Rollup triggers are the rail.** Same shape as [[spec-body-table-and-backfill]]: a milestone can't be `complete` if any child spec is non-shipped; a goal can't auto-flip to `complete` while still `proposed` (must be `greenlit` first — the [[goal-greenlight-button-and-author-writes-db]] action is the only path to `greenlit`).
- **Backfill is non-destructive.** The `.md` files stay in `docs/brain/goals/` until [[goal-readers-from-db-retire-parsegoal]] retires the parse — leaving a rollback path. No `git rm` here.
- **Specs without a milestone keep `milestone_id=null`.** A standalone fix-spec / regression / function-mandate spec is the explicit zero-milestone shape; do not invent a placeholder milestone for them.

Spot-check three goals: a top-level greenlit goal ([[../goals/db-driven-specs]] itself), a SubGoal candidate (parent set to `ceo-mode` per the goal text), a still-proposed goal.
- The rollup triggers fire: setting one `goal_milestones.status='complete'` while siblings stay `planned` leaves `goals.status='greenlit'` (unchanged); flipping the last sibling to `complete` rolls `goals.status` to `complete` (only if currently `greenlit`; a `proposed` goal stays `proposed`). Manually setting `specs.status='shipped'` for every spec under a milestone rolls the milestone to `complete` via the spec-side trigger.
- `parent_goal_id` cycle protection rejects an `UPDATE goals SET parent_goal_id=$child WHERE id=$parent` attempt that would close a loop.
- `public.specs.milestone_id` is populated for every spec whose `**Parent:**` line matches a parsed milestone; standalone specs keep `null`. The board's milestone view (a follow-up surface in [[goal-readers-from-db-retire-parsegoal]]) can already join specs to milestones via the FK.

## Verification (Phase 1)

- Apply the migration: `npx tsx scripts/apply-goals-tables-migration.ts` → expect `✓ tables present: goal_milestones, goals`, `✓ functions present: 5`, `✓ triggers present: 3`, `✓ specs.milestone_id FK constraint present: true`.
- Probe the live schema: `npx tsx scripts/_verify-goals-schema.ts` → expect `public.goals` columns (`id, workspace_id, slug, title, body, outcome, success_metric, owner, proposer_function, parent_goal_id, status, created_at, updated_at`) and `public.goal_milestones` columns (`id, goal_id, position, title, body, status, created_at, updated_at`); indexes `goals_ws_slug`, `goals_parent_idx`, `goals_ws_status_idx`, `goal_milestones_goal_position`, `goal_milestones_goal_idx`, `goal_milestones_status_idx`; triggers `goals_no_cycle`, `goal_milestones_rollup`, `specs_milestone_rollup`; FK `specs_milestone_id_fkey :: FOREIGN KEY (milestone_id) REFERENCES goal_milestones(id) ON DELETE SET NULL`.
- `npx tsc --noEmit` → exit 0.
- Acyclicity rail (post-backfill / via psql): INSERT two goals A + B with `parent_goal_id` null; `UPDATE goals SET parent_goal_id = A WHERE id = B`; then `UPDATE goals SET parent_goal_id = B WHERE id = A` → expect a `parent_goal_id cycle` error (`SQLSTATE 23514`), and A's `parent_goal_id` stays null.
- Self-parent rail: `UPDATE goals SET parent_goal_id = id WHERE id = $A` → expect the same cycle error.
- Goal rollup (greenlit → complete): seed a `greenlit` goal G with two `goal_milestones` (M1, M2). Set `M1.status='complete'`; M2 stays `planned`. Expect `G.status` to remain `greenlit`. Then set `M2.status='complete'` → expect `G.status` to flip to `complete` via the trigger.
- Goal rollup (proposed never auto-greenlights): seed a `proposed` goal G with one milestone M; set `M.status='complete'` → expect `G.status` to stay `proposed` (the rail; only the CEO greenlight button moves `proposed → greenlit`).
- Milestone rollup: seed a milestone M with one `specs` row S (`milestone_id=M`, `status='planned'`). Expect `M.status='planned'`. Set `S.status='in_progress'` (via the [[spec_phases]] rollup or directly) → expect `M.status='in_progress'`. Set `S.status='shipped'` → expect `M.status='complete'`.
- FK behavior: DELETE a `goal_milestones` row that has a spec attached → expect the spec to survive with `milestone_id` reset to null (the standalone-spec shape).
