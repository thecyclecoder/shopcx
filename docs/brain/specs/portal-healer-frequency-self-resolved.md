# Portal healer — auto-dismiss frequency tickets when the local sub already matches

**Owner:** [[../functions/cs]] · **Parent:** CS mandate "Ticket-derived product fixes" · **Derived-from-ticket:** `c4bf5455-f271-487a-895c-6275631aff2a`

Add a frequencySelfResolved helper to src/lib/portal/remediation.ts that mirrors the existing cancelSelfResolved / changedateSelfResolved patterns. When a portal-action-failed ticket's route is 'frequency' and the local subscriptions row for the contract is already at the requested billing_interval + billing_interval_count, auto-dismiss the ticket with an internal note instead of escalating. Two cases this catches: (a) legacy pre-5ae738ac backlog tickets where a prior attempt landed end-to-end and a later retry surfaced a same-value 400 from Appstle — ticket c4bf5455-f271-487a-895c-6275631aff2a (Bekki Hopkins) is the first observed instance; her first portal click landed (local DB updated to WEEK/8 at 05:56:49), her retries hit a 504 and then a same-value 400, and the healer escalated because it can't tell the customer's intent is already satisfied. (b) Defensive backstop for the narrow race where the local DB matches the request but Appstle still 400s — e.g. a webhook updated the local row seconds before Appstle's billing-policy cache caught up. After 5ae738ac the create-the-ticket pattern is prevented at the Appstle call site by the noop guard short-circuiting same-value requests; this helper is the healer-side equivalent that handles the backlog + the residual race.

## Problem (from escalated ticket `c4bf5455-f271-487a-895c-6275631aff2a`)
src/lib/portal/remediation.ts has cancelSelfResolved (for CANCEL_ROUTES) and changedateSelfResolved (for 'changedate'/'change_date'), both gated in remediatePortalTicket. There is no equivalent for the 'frequency' route. When Appstle returns a 400 on a frequency update — most commonly because the change already landed and Appstle is rejecting the same-value re-submission — classifyPortalFailure returns 'human' because the error string and the portal-wrapped 502 status don't match any dismiss or transient signature. The ticket escalates even though the customer's intent (frequency = X) is already realized in the database. The 5ae738ac noop guard prevents new tickets of this shape at the Appstle call site (skips the PUT when local DB already matches), but does not address (1) legacy escalated tickets created before it deployed and (2) the narrow race where the local DB does match the request but the Appstle call still 400s for cache-coherency reasons. Add the helper + gate it for the 'frequency' route alongside the existing self-resolved checks.

**Likely target:** `src/lib/portal/remediation.ts — new async function frequencySelfResolved(admin, workspaceId, ctx, ticket) that reads subscriptions.billing_interval + billing_interval_count by shopify_contract_id (ctx.payload.contractId) and returns { resolved: true, reason } when they match ctx.payload.interval / ctx.payload.intervalCount (case-insensitive on interval); add a gate near the existing CANCEL_ROUTES.has(ctx.route) block in remediatePortalTicket for ctx.route === 'frequency' that calls it before classifyPortalFailure's 'human' disposition fires, closes the ticket with an [Auto-resolve] sysNote + auto-dismissed tag and returns { action: 'dismissed', reason }.
