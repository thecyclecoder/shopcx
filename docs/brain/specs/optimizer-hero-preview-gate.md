# Optimizer hero preview gate — see/reject-with-notes before a generated hero goes live ✅

**Owner:** [[../functions/growth]] · **Parent:** hardens [[storefront-optimizer-agent]] + [[storefront-optimizer-proposal-cards]]. · **Owner directive 2026-06-23:** for an `image` (hero) campaign, the owner approves the *concept* today and the hero is generated **post-approval and served to live shoppers sight-unseen**. Add a **preview gate**: generate the candidate first, the owner **sees the actual image** and either approves it live or **rejects with notes** that feed a better regeneration — iterate until it's right.

## Flow (insert a generate→preview→approve loop before the experiment goes live)
1. **Approve concept → generate candidate (don't go live yet).** On approving a hero-lever campaign, the worker generates the candidate hero via Nano-Banana ([[../libraries/gemini]]) and stores it as a **pending preview** on the campaign (NOT a live variant) — the job returns to `needs_approval` with the image surfaced on the card.
2. **Owner previews the real image** on the [[storefront-optimizer-proposal-cards|proposal card]]:
   - **Approve → goes live** — `materializeCampaign` stands up the experiment with *that* image as the variant arm.
   - **Reject with notes** — a free-text box ("warmer light", "show the pouch facing forward", "less busy background"). The notes **augment the prompt**, a new candidate is generated, and it previews again. Loop until approved (or the owner cancels the campaign). Keep the rejected attempts + notes on the campaign (the gen learns within the loop; nothing serves to a shopper until the owner approves an image).
3. Only the **final owner-approved image** is ever served. Nothing reaches live traffic on a prompt alone.

## Grounding the generation (owner's explicit asks — the gen MUST know these)
- **Composite the real product from the isolated bag image.** Source the product cutout from **`product_variants.isolated_image_url`** (set for Amazing Coffee — the clean isolated pouch) and composite it into the lifestyle scene. NEVER hallucinate/redraw the packaging — use the real isolated asset so the pouch/label is exact. (Pass it as the Nano-Banana Pro *combine* reference image.)
- **Generate at the correct hero dimensions.** Target the lander's actual hero slot, not a guessed size: for the **PDP** use the variant's stored **`hero_width × hero_height`** (`product_variants` — what `HeroSection` renders); for other landers use the hero aspect of that section (advertorial `16:10` / `4:5`, before/after `3:4`). Pick the closest Nano-Banana `aspectRatio` + a high resolution so the result fits the slot with no distortion/awkward crop.
- Honor the existing **locked-hero guard** — this writes an EXPERIMENT VARIANT overlay, never the canonical/locked Amazing Coffee hero asset (the control stays the locked hero; the variant is a separate render-time overlay, reversible).

- On `/dashboard/storefront/optimizer` (owner), tap **Approve concept → generate preview** on a pending hero proposal → expect the card to drop, then re-appear within ~30s showing the **actual generated image** + a "preview · approve to go live" badge. `select status, lander_type, lever from storefront_experiments where product_id='<pid>' order by created_at desc limit 1` → **no new `running` row** for this surface yet; `select status, pending_actions from agent_jobs where id='<jobId>'` → `needs_approval` with the action `stage:'preview'` + a `preview_image_url`.
- Inspect that `preview_image_url` → the hero **composites the real pouch** from `product_variants.isolated_image_url` (label/packaging is the actual product, not redrawn) and its aspect matches the PDP hero slot (`product_media slot='hero'` width×height → nearest Nano aspect; advertorial 3:2, before/after 3:4) — fits with no distortion.
- On the preview card, open **Reject with notes**, type e.g. "warmer light, pouch facing forward", tap **Reject & regenerate** → `POST /api/roadmap/approve {decision:'reject', notes}` → the card re-appears with a NEW candidate reflecting the notes; the prior image shows under **Rejected attempts** with its note; still **no** `running` experiment.
- On the preview card, tap **Approve & go live** → `POST /api/roadmap/approve {decision:'approve'}` → after the worker resumes, `select status, lever from storefront_experiments where product_id='<pid>' order by created_at desc limit 1` → a `running` row whose variant arm `patch.heroImageUrl` equals the **approved** `preview_image_url` (not a fresh render); the canonical locked hero is untouched (control); rollback reverts to it.
- Negative — a **content** (copy/chapter) proposal shows **Approve** (no "concept"/preview) and materializes the experiment on the first approve, skipping the gate (no image to preview). `POST /api/roadmap/approve {decision:'reject'}` on a non-`preview` action → 409 "this action has no image preview to reject". No image is ever served to a shopper without an explicit owner image-approval.

### What landed
- **Two-stage `storefront_campaign` approval** (`scripts/builder-worker.ts` `runStorefrontOptimizerJob`). The surfaced pending_action now carries a `stage`: a **hero** lever is surfaced `stage:'concept'`; a **content** (copy/chapter) lever carries no stage and skips the gate entirely (materializes on the first approve, as before).
  - **Concept-approve** (stage `concept`) → the worker re-gates (TOCTOU active/scope), then **generates the candidate hero** (`generateCampaignHero`) and re-surfaces the job as `needs_approval` with `stage:'preview'` + `preview_image_url`. It does **NOT** call `materializeCampaign` — no `storefront_experiments` row yet.
  - **Image-approve** (stage `preview`, `approveRoadmapAction` decision `approve`) → `materializeOptimizerCampaign(..., approvedHeroUrl=preview_image_url)` stands up the M1 experiment with **that exact image** as the variant arm (the approved URL is used verbatim — never a fresh regen).
  - **Reject-with-notes** (decision `reject` + `notes`, only valid at stage `preview`) → marks the action `reject_regen` with `reject_notes`; on resume the worker pushes `{url, notes, at}` onto `preview_attempts`, **regenerates** with the accumulated notes augmenting the prompt, and re-surfaces for preview. Loops until the owner approves or cancels (decline). Each candidate uploads to a distinct path (`…-a{N}.webp`) so rejected attempts retain their own images.
- **Grounded generation** (`src/lib/storefront/optimizer-agent.ts` `generateCampaignHero` + `resolveHeroAspect`/`nearestNanoAspect`): composites the real isolated pouch (`product_variants.isolated_image_url`) as the Nano-Banana *combine* reference with a preamble forbidding redrawing the packaging; targets the lander's real hero slot — **PDP** reads the product's `slot='hero'` `product_media` width×height (what `HeroSection` renders) → closest Nano `aspectRatio`; advertorial→`3:2` (16:10), before/after→`3:4`, listicle→`4:5`.
- **Approval plumbing** (`src/lib/roadmap-actions.ts` `approveRoadmapAction`): added a third decision `reject` (+ `notes`), gated to a `storefront_campaign` action at `stage:'preview'`; threaded through `POST /api/roadmap/approve`. No new approval route — same owner-gated `queued_resume` path. `GatedActionType` widened to include the storefront/repair action types it already carried at runtime.
- **Proposal card** (`src/app/dashboard/storefront/optimizer/page.tsx` + the proposals API): at `stage:'preview'` the card renders the **actual generated image**, a "preview · approve to go live" badge, the rejected-attempt thumbnails + their notes, and **Approve & go live / Reject with notes (free-text) / Cancel campaign**. The concept stage shows the hero prompt with **Approve concept → generate preview**. After a concept-approve/reject the card optimistically drops and the page polls for the regenerated preview to re-appear.
