# Harden the Bo agent — roll persistent coaching into its mandate

**Owner:** [[../functions/platform]] · **Parent:** [[platform-director-agent]] — graduate persistent coaching into a durable fix (director grades agents: low score → fix spec, never a CEO escalation)
**Found in use 2026-06-30:** the **Bo** agent (`build`) sits at **6.8/10** after **2** coaching attempts that didn't stick. Rather than escalate to the CEO, roll the coaching into a permanent fix so the agent improves at the mandate level.

## The accumulated coaching to bake in (now archived as rolled-into-mandates)
- **When Bo begins executing build work without first verifying which spec phases are already satisfied in the current codebase, leading to either wasted effort on completed work or thin evidence of actual validation.:** When you see a spec with phases, before writing a single line of code or running any build command, inspect the repo (git log, grep, file contents) to explicitly check each phase's completion status — mark already-satisfied phases as DONE and skip them, then target only the unsatisfied phases; if all phases are already satisfied, conclude 'already done' immediately. — Both low-graded runs share the same root failure: work started without establishing a ground-truth baseline of what was already shipped, causing either redundant execution on completed phases (agent-outage-resilience, 3/10) or unfocused effort with no auditable spec-phase checkoffs (otp-start, 6/10). Checking phase completion first eliminates wasted tokens, produces explicit per-phase validation evidence, and ensures the terminal status accurately reflects reality.
- **When Bo begins building without first auditing what is already shipped, leading to either misreported completion status or wasted cost re-doing work that is already done.:** When you receive a build job, before writing a single line of code or running any tool, scan the current repo state (existing files, prior commits, open PRs) and check each spec phase against what is already merged — if a phase is already satisfied, mark it done and move on; only then begin building the delta. — Both low-grade incidents trace back to skipping a pre-flight state check: once Bo concluded needs_attention on fully-shipped work, and once Bo produced thin evidence of spec validation because it jumped straight to execution. Auditing existing state first eliminates false needs_attention statuses, prevents redundant token spend, and ensures the build log explicitly accounts for every spec phase — the exact signals the rubric rewards.

## Phase 1 — bake the coaching into the agent
- Make the above coaching PERMANENT behavior of the `build` agent — fold it into its prompt/mandate/code (its run-job + prompt in `scripts/builder-worker.ts` and the relevant `src/lib/agents/*`), not ephemeral appended `agent_instructions`. Once baked, the agent should follow it by default.
- Verify the agent's grade rollup recovers (≥ 7/10) over the next window of graded actions; if a coaching class still recurs, the bake-in missed it.
