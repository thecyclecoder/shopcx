**Priority:** critical

**Owner:** [[../functions/platform]] · **Parent:** [[../goals/devops-director]] — hardens the goal→decompose→approve→build pipeline ([[../specs/goal-decomposition-engine|Pia]] + the director escort)
**Found in use 2026-06-24:** Pia decomposed [[platform-department-scorecard]] + [[grow-surface-platform-agent-team]] at 03:32 with NO `Blocked-by:` deps on the milestone specs. On approval, 5 milestone builds (platform-scorecard-surface/weekly/monthly, fleet-spend-governor, fleet-cost-metering) fanned out CONCURRENTLY at 03:45, out of order → 4 jammed in `needs_input`, fleet-cost-metering in `needs_approval` to the director (unapprovable — prerequisites don't exist). Two root causes: Pia emits no dependency DAG, and the approval path enqueues milestone builds directly instead of through the director's dependency-aware escort.

## North star — ordered, gated, never a free-for-all

A goal is a DAG of milestones; building them concurrently without deps corrupts the build (later specs reference earlier ones' outputs). The director sequences; Bo builds only the unblocked frontier. The CEO greenlights the GOAL; the director sequences the BUILDS.

## Phase 1 — Pia always emits the dependency DAG (decomposition integrity) ✅
- ✅ shipped — the [[../specs/goal-decomposition-engine|goal-decomposition planner]] declares dependencies: every proposed branch carries a `blocked_by` slug list (planner prompt in `runPlanJob`, `scripts/builder-worker.ts`), authored on resume as a `**Blocked-by:** [[slug]], …` header per dependent milestone spec, so the existing [[../specs/spec-blockers|spec-blockers]] chokepoint (`queueRoadmapBuild` refuses a blocked build; auto-queues the dependent on the blocker's merge) staggers them.
- ✅ shipped — a decomposition that emits a FLAT, blocker-less multi-milestone list is now REJECTED before it reaches the CEO. The PROPOSE round runs a deterministic gate (`isFlatBlockerlessTree` in `scripts/builder-worker.ts`: ≥2 proposed specs with not a single `blocked_by` edge) and, on a hit, does ONE bounded in-session re-plan — the planner must either declare the real build-order edges OR explicitly affirm independence in each spec's `intent`. A still-flat re-plan stands as the planner's deliberate, logged parallel decision (never the silent default). The corrective round reuses the same parse (`parsePlannerSpecs`) so the rejected proposal never persists to `needs_approval`.
- A genuinely-parallel milestone (no dep) is allowed but must be explicit, not the default.
- Brain: the plan flow trace in [[../lifecycles/roadmap-build-console]] · the plan-goal skill · [[../specs/spec-blockers]] · [[../libraries/brain-roadmap]] (the `Blocked-by` parse).

## Phase 2 — approved milestones route through the director, not a direct fan-out ✅
- ✅ shipped — greenlighting a goal marks it approved-to-build but enqueues NOTHING: the greenlight resume path (`runProposedGoalJob`, `scripts/builder-worker.ts`) only flips the goal `**Status:**` to `greenlit` via `setGoalStatusLine` — no `agent_jobs` build inserts. The director's `escortApprovedGoals` ([[../libraries/platform-director]]) is the SOLE enqueue route for a goal's milestones; it walks `isApprovedInProgress` goals and releases only the UNBLOCKED frontier — `if (card.blockedBy.some((b) => !b.cleared)) continue;` skips any spec with an unshipped blocker — the rest auto-queue as their blockers merge (the existing reactive auto-queue) and on the next escort pass.
- No path auto-enqueues all milestone builds on greenlight (verified absent in code). The greenlight → director-escort handoff is the only route to a build.
- Brain: [[../libraries/platform-director]] (`escortApprovedGoals`) · the approval/greenlight path · [[../specs/spec-blockers]].

## Phase 3 — reconcile the current out-of-order pileup ✅
- ✅ shipped — the director's standing pass now runs a **sequence reconcile** ([[../libraries/platform-director]] `findMilestoneSequenceViolations` + `holdPrematureBuild` + `ensureBlockedByLine`; box lane `reconcileMilestoneSequence`, the FIRST standing-pass step before the escort). It detects every milestone build that fanned out before its prerequisites — a goal-member spec with an UNMET (unshipped) declared blocker MISSING from its own `**Blocked-by:**` line and an ACTIVE build — then recovers: **HOLD** the premature build (parks it at the `held` status, cancelling the jammed needs_input/needs_approval fan-out), **apply** the AUTHORED `Blocked-by` order transcribed from the goal doc's per-spec `*(blocked by [[…]])*` annotations (`parseGoalSpecBlockers` — NEVER a guessed DAG), and let the existing reactive auto-queue + the goal escort **re-release** it in sequence as its blockers ship (`specBuildState` ignores a `held` job so the escort re-queues it fresh once unblocked, without tripping the loop-guard). Idempotent (a re-sequenced spec is no longer a candidate: its file carries the blocker, its build is `held`, and a recent `reconciled_sequence` ledger row dedups against fs-lag); bounded by `PLATFORM_DIRECTOR_SEQUENCE_CAP`; dormant until Platform is live+autonomous. Logs one `reconciled_sequence` [[../tables/director_activity]] row per re-sequenced build.
- One-shot for the current mess + a standing guard against a future blocker-less fan-out slipping through. (The 03:45 cluster's spec files already carry their `**Blocked-by:**` lines + the foundation specs shipped, so the reconcile is a no-op for it today — but it WOULD have caught it, and now guards every future fan-out.)
