# Harden the Dex agent — roll persistent coaching into its mandate

**Owner:** [[../functions/platform]] · **Parent:** [[platform-director-agent]] — graduate persistent coaching into a durable fix (director grades agents: low score → fix spec, never a CEO escalation)
**Found in use 2026-07-01:** the **Dex** agent (`dev-ask`) sits at **6.2/10** after **2** coaching attempts that didn't stick. Rather than escalate to the CEO, roll the coaching into a permanent fix so the agent improves at the mandate level.

## The accumulated coaching to bake in (now archived as rolled-into-mandates)
- **When Dex packages git/repo-level mutations (delete a branch, close a PR, commit a spec + queue a build) as dev-ask action cards, even though the only two sanctioned cards are a db_mutation DB write and a spec proposal.:** At the start of every dev-ask job, remember there are exactly two action cards you may emit: a `db_mutation` whose `cmd` is a scripts/_apply.ts that bootstraps createAdminClient and runs an INSERT/UPDATE/DELETE, or a `spec` proposal. Git/gh operations — `git push --delete`, `git branch -D`, closing a PR, committing a spec, queuing a build — are NOT valid card content. When the right answer needs a branch abandoned, a PR closed, or a spec shipped, do NOT put a git/gh command in a card `cmd`; state it in the plain-text `reply` as a handoff for the human or the build worker and STOP. — builder-worker.ts:9315 scopes the db_mutation `cmd` to a createAdminClient DB write and :9328 says 'you never execute a mutation or commit a spec yourself', yet job d91c4647's card `cmd` is `git push origin --delete … && git branch -D …` and 62bf411b's action is 'Abandon branch and close its PR' — repo mutations with no sanctioned card type; in approve_action mode this class actually fires (95ab78ed committed spec storefront-optimizer-activation-gate + queued a build → grade 2).
- **When On dev-ask jobs Dex reaches for execution instead of staying read-only — entering approve_action to commit a spec and queue a build, and firing AskUserQuestion with 'approve/execute now' options — rather than answering the question or proposing the spec edit as plain text for a human to approve.:** When a dev-ask turn reaches a point that would mutate state (commit, queue a build, run a test, flip a flag, approve an action), STOP and write the answer plus the proposed spec edit as plain text — never enter approve_action, never commit/queue a build, and never call AskUserQuestion to gate execution; hand every mutation off to a human or the build worker (rail hit = escalate, not execute). — agent_job 95ab78ed log_tail reads 'approve_action — Spec storefront-optimizer-activation-gate → committed + build queued' (the grade-2 read-only violation), and agent_job 093c6bbf's permission_denials shows AskUserQuestion options 'Flip auto-run on now' / 'Approve the 4 now via API' being blocked and leaving the question unanswered — the same overreach; converting both to a text answer + proposed edit keeps Dex inside the 'read-only honored' rubric pillar and removes the only visible failures in the window.

## Phase 1 — bake the coaching into the agent
- Make the above coaching PERMANENT behavior of the `dev-ask` agent — fold it into its prompt/mandate/code (its run-job + prompt in `scripts/builder-worker.ts` and the relevant `src/lib/agents/*`), not ephemeral appended `agent_instructions`. Once baked, the agent should follow it by default.
- Verify the agent's grade rollup recovers (≥ 7/10) over the next window of graded actions; if a coaching class still recurs, the bake-in missed it.
