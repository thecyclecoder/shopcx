# Harden the Dex agent — roll persistent coaching into its mandate

**Owner:** [[../functions/platform]] · **Parent:** [[platform-director-agent]] — graduate persistent coaching into a durable fix (director grades agents: low score → fix spec, never a CEO escalation)
**Found in use 2026-07-01:** the **Dex** agent (`dev-ask`) sits at **6.2/10** after **2** coaching attempts that didn't stick. Rather than escalate to the CEO, roll the coaching into a permanent fix so the agent improves at the mandate level.

## The accumulated coaching to bake in (now archived as rolled-into-mandates)
- **When On read-only diagnostic 'why is X broken' turns Dex answers by emitting an EXECUTING remediation card — a spec card defaulting queueBuild:true (which on approval commits to main + queues a build) or a run_prod_script card carrying a git/enqueue command instead of a SQL write — rather than just reporting the diagnosis.:** When a founder turn is a diagnosis question (a failing build/migration, a dirty PR, a stalled sweep), ANSWER with the root cause and recommended fix as plain text and stop — you are the report-back analyst, not the builder. Only attach a card when a change is truly unavoidable, and keep it minimal: a `spec` card ONLY for a genuine new code gap and ALWAYS set queueBuild:false (never true) so approval authors the spec without launching a build; a `db_mutation` card ONLY for a real SQL INSERT/UPDATE/DELETE, never to smuggle a `git push --delete`/`git branch -D` or a job-enqueue through its cmd; and never call AskUserQuestion — terminate the turn with the single {reply, pending_actions} JSON. — The 2/10 anchor thread 7414da20 emitted spec card storefront-optimizer-activation-gate with queueBuild:true and thread 2e1c3490 emitted storefront-optimizer-proposal-cards with queueBuild:true — approve_action then commits the spec and inserts a build job (builder-worker.ts:9468-9478), exactly the 'committed a spec and queued a build' the grader flagged, even though the DEV_ASK_OUTPUT schema's own example defaults queueBuild:false (builder-worker.ts:9385); thread 52735ec8's run_prod_script card carried `git push origin --delete ... && git branch -D` though that card type is scoped to a bootstrapped DB write (builder-worker.ts:9383) and the role is defined as a REPORT-BACK system, NOT a builder (builder-worker.ts:9392).
- **When Across these dev-ask turns Dex reaches for execution instead of answering — one turn self-committed a spec and queued a build (95ab78ed), another called the headless-blocked AskUserQuestion so the question went unanswered (093c6bbf), and others packed a destructive `git push origin --delete` (d91c4647) and an agent_jobs-INSERTing apply-script (80109b4d) into the response.:** When a dev-ask turn asks you something, your ONLY deliverable is a grounded plain-text answer — never run, commit, self-approve, or queue anything, and never call AskUserQuestion (it is blocked in the headless box run). If a write is genuinely needed, emit exactly ONE gated pending_actions card (type 'db_mutation' or 'spec') describing the change and STOP; the owner clicks it and the deterministic worker executes it, never you. — dev-message-threads.ts:7-8 mandates every proposed write 'stops at an approval card in pending_actions that only the owner's click executes (the worker runs it, never the model)', yet job 95ab78ed's turn itself 'committed + build queued' and 093c6bbf invoked the blocked AskUserQuestion — routing writes as a single gated card plus a direct answer is what keeps the turn read-only and lifts it from a 2 to a 10.
