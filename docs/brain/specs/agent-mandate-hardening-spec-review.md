# Harden the Vale agent — roll persistent coaching into its mandate

**Owner:** [[../functions/platform]] · **Parent:** [[platform-director-agent]] — graduate persistent coaching into a durable fix (director grades agents: low score → fix spec, never a CEO escalation)
**Found in use 2026-06-30:** the **Vale** agent (`spec-review`) sits at **6.6/10** after **2** coaching attempts that didn't stick. Rather than escalate to the CEO, roll the coaching into a permanent fix so the agent improves at the mandate level.

## The accumulated coaching to bake in (now archived as rolled-into-mandates)
- **When Vale consistently logs only a tally (e.g., '✅0 ⚠1') without naming the reviewed spec, the defect category, or the specific field/section that failed, making every finding unverifiable.:** When you emit a needs_fix or pass result, always include in the log: (1) the exact spec name, (2) the defect category from the rubric (mangled phase · missing owner/parent/blockers · missing Verification), (3) the specific field or section that is defective, and (4) a one-sentence rationale tying that field's state to the rubric criterion — never emit only a count. — Graders cannot distinguish a sound diagnosis from a lucky flag when the log is tally-only; every unexplained finding scores as unverifiable regardless of correctness. Attaching field-level evidence to every call lets the director confirm the judgment was substantively right, which is the sole path from 6 to 10 on this rubric.
- **When Vale consistently records only a pass/flag tally (e.g., '⚠1') in the log without naming the spec, the defective field, or the rubric category (mangled phases, missing owner/parent/blockers, missing Verification) that triggered the call.:** When you see yourself about to log a needs_fix or pass verdict, do instead write a one-line field-level diagnosis in the log for every spec reviewed — e.g., 'SPEC-42: needs_fix — missing Verification section and no owner assigned' — so each call is traceable to a named spec and a named rubric defect. — Graders cannot distinguish a correct diagnosis from a lucky guess when the log contains only a count; full field-level citations prove the rubric was actually applied and let the director act without re-examining the source markdown. Sparse logs are the sole reason every recent job landed at 6/10 despite correct pass/flag outcomes.

## Phase 1 — bake the coaching into the agent
- Make the above coaching PERMANENT behavior of the `spec-review` agent — fold it into its prompt/mandate/code (its run-job + prompt in `scripts/builder-worker.ts` and the relevant `src/lib/agents/*`), not ephemeral appended `agent_instructions`. Once baked, the agent should follow it by default.
- Verify the agent's grade rollup recovers (≥ 7/10) over the next window of graded actions; if a coaching class still recurs, the bake-in missed it.
