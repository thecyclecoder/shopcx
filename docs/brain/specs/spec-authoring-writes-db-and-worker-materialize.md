# Spec authoring writes to the DB — worker materializes for Bo

**Owner:** [[../functions/platform]] · **Parent:** [[../goals/db-driven-specs]] M3 — Authoring + building write/read the DB
**Blocked-by:** [[spec-body-table-and-backfill]]

Cut every spec-CREATION surface off `git commit docs/brain/specs/{slug}.md` and onto `INSERT INTO public.specs + public.spec_phases` (the tables stood up by [[spec-body-table-and-backfill]]). The planner ([[../libraries/agents]] `goal-proposals`, the [[../skills/plan-goal]] runner), the box worker's triage + fix-spec lanes ([[../recipes/build-box-setup]]), the [[director-judgment-lanes-fold-author-dismiss|director split/author]] paths, [[ada-slack-routed-approvals|Ada]] in chat, and [[spec-review-agent|Vale]] when she fixes a malformed spec all WRITE THE ROW. Editing a spec = `UPDATE public.specs` (or `UPDATE public.spec_phases` for a phase edit). Bo (the [[../skills/build-spec]] skill) READS THE ROW: the box worker materializes `specs` + `spec_phases` to a temp `{cwd}/.box/spec-{slug}.md` for the build-spec skill to consume — Bo never needs the `.md` on disk under `docs/brain/specs/`, and the worker never commits a spec body to `main`. The [[spec-review-agent|build hard-stop]] (`queueRoadmapBuild` refuses `status='in_review'`), the `in_review`→`planned`/`deferred` flow, and per-phase PR tagging ([[spec-status-phase-pr-provenance]]) all operate on the row. After this spec ships, the system runs as DUAL-WRITE (`.md` + DB) — every new write goes to the DB, but the markdown still gets a mirror commit so [[../libraries/brain-roadmap]] `parseSpec` keeps serving the still-markdown-first readers. The reader cutover (and the deletion of the mirror commit) is owned by [[spec-readers-from-db-retire-parser]]. Closes the gap: today every authoring path commits `.md` (e.g. [[director-proposed-goals]] commits a goal, the same shape; the planner skill commits a spec; the [[spec-review-agent]] Phase 3 governance still operates on markdown); `scripts/builder-worker.ts` feeds the build-spec skill the on-disk `.md` not a row.

## Phase 1 — author surfaces INSERT a row (every creation path)
- The [[../skills/plan-goal]] runner: on owner-approval of a proposed leaf, the box worker authoring path calls `upsertSpec` ([[spec-body-table-and-backfill]] `src/lib/specs-table.ts`) — INSERT `public.specs` row with `status='in_review'`, `intended_status=` author's suggestion, owner/parent/blocked_by, `summary=` the one-paragraph intent, plus N `public.spec_phases` rows (one per `## Phase` block the planner emitted). NO `git add docs/brain/specs/{slug}.md`. The legacy commit path is replaced by a DB write + a mirror commit (Phase 4 — kept until [[spec-readers-from-db-retire-parser]] retires it).
- The director's [[director-judgment-lanes-fold-author-dismiss|author lane]] (`scripts/builder-worker.ts` `runDirectorCoachJob` author action): same `upsertSpec`, same `in_review` default. The [[director-proposed-goals]] Phase 3 `goal` action sister flow stays goal-side ([[goal-greenlight-button-and-author-writes-db]]).
- The triage + fix-spec lanes ([[../recipes/build-box-setup]] worker's `runTriageEscalationsJob`, `runRegressionJob` `authorRegressionFixSpec`): both INSERT a row instead of committing a `.md`. `regression-agent.md` already authors the fix as a row in this spec; the regression's `**Regression-of:**` + `**Regression-signature:**` carry as DB columns on `specs` (`regression_of_slug text`, `regression_signature text` — added in [[spec-body-table-and-backfill]] schema or here as a small additive migration).
- [[ada-slack-routed-approvals|Ada]] when she emits a spec card from a chat: same `upsertSpec`.
- [[spec-review-agent|Vale]] when she fixes a malformed spec (restructures mangled phases, adds an owner): same `UPDATE` — `upsertSpec` with the corrected phases array (preserving phase ids via the `position` rule from [[spec-body-table-and-backfill]]).
- The `flags.intended_status` slot on [[../tables/spec_card_state]] becomes `specs.intended_status` (canonical); writers stop touching the mirror's `flags` and write the column directly.
- Verification — Phase 1: every author surface, given the same input, produces a `public.specs` row + `spec_phases` children, `status='in_review'`, `intended_status` set; NO new git commit for `docs/brain/specs/{slug}.md` is initiated from the path's primary write (the mirror commit in Phase 4 is the only commit, and it's a separate worker step). Run the planner against a stub greenlit goal in dev → expect a row + phases + zero git noise from the planner itself.

## Phase 2 — the worker materializes the row for Bo
- New library `src/lib/build-spec-materializer.ts`: `materializeSpec(workspaceId, slug, dir)` joins `specs` + `spec_phases` ordered by position → renders the EXACT brain-spec format (H1 title; Owner/Parent/Blocked-by metadata line; summary paragraph; `## Phase N — name` blocks; `## Safety / invariants`; `## Completion criteria`) → writes `{dir}/spec-{slug}.md`. Returns the path. NO status emoji on the H1 or phases — content-only, mirroring the [[spec-status-db-driven]] rule.
- `scripts/builder-worker.ts` `runBuildJob` (the lane that dispatches the [[../skills/build-spec]] skill): replace its `docs/brain/specs/{slug}.md` lookup with `materializeSpec(workspaceId, slug, '{cwd}/.box/spec-{slug}')` and hand the build-spec skill the temp path. The skill's prompt template gets a one-line note: "the spec body comes from the DB row; the file you read is a worker-materialized copy". The build-spec skill itself doesn't change — it just consumes whichever path the worker hands it.
- Phase progress writes ([[spec-status-phase-pr-provenance]] — the build skill records `pr + merge_sha` per phase on merge) move from `spec_card_state.phase_states[i]` UPSERTs to `public.spec_phases` UPDATEs (by `spec_id + position`). The `spec_card_state.phase_states` slot stays double-written until [[spec-readers-from-db-retire-parser]] retires it. The DB trigger from [[spec-body-table-and-backfill]] rolls `specs.status` up automatically when the phase status flips — no app-side rollup.
- The build hard-stop ([[spec-review-agent]] `queueRoadmapBuild` refusing `status='in_review'`) keeps the same check but reads `specs.status` directly (DB-authoritative — both before and after this spec).
- Verification — Phase 2: a queued build job picks the temp path, the build-spec skill consumes it, opens the `claude/*` PR per the existing recipe; on merge the phase row flips `status='shipped'` + `pr + merge_sha` recorded; the trigger rolls `specs.status` to `in_progress`/`shipped` per the rollup rule.

## Phase 3 — `in_review` flow, phase-PR tagging, editing all operate on the row
- The [[spec-review-agent|Vale]] verdicts (`approve → planned`, `defer → deferred`, `needs_fix → in_review` with corrections) all become `UPDATE public.specs SET status=…` writes (already true for status since [[spec-status-db-driven]] — this spec just confirms there's NO markdown commit left in Vale's lane). The corrections themselves write to `spec_phases` (lifting/reordering/retitling phases via `movePhase`/`upsertSpec` from [[spec-body-table-and-backfill]]).
- The board's spec-status writers ([[../libraries/spec-card-state]] `flipSpecStatus`, the build merge writer, the drift reconciler when it still fires, owner flips): converge on `specs.status` UPDATEs (the rollup trigger does the work for child phase writes). [[../tables/spec_status_history]] keeps recording the audit row per transition (the existing actor + reason shape, unchanged).
- The phase-PR provenance writer ([[spec-status-phase-pr-provenance]]) moves from `spec_card_state.phase_states[i].{pr,merge_sha}` upserts to `UPDATE spec_phases SET pr=$1, merge_sha=$2 WHERE spec_id=$3 AND position=$4` — preserving the [[spec-status-phase-pr-provenance]] gotcha around forward-merge safety.
- Editing a spec in [[../dashboard/roadmap]] (the future surface) or via a board action: a single `UPDATE public.specs` (for the body) or `UPDATE public.spec_phases` (for one phase) — no `git commit`, no Vercel deploy.

## Phase 4 — the markdown mirror commit (transitional — retired in [[spec-readers-from-db-retire-parser]])
- After every author/edit DB write in Phase 1-3, the worker fires a `kind='mirror-spec-md'` `agent_jobs` row that materializes the row to `docs/brain/specs/{slug}.md` and commits it to `main` (mirroring is necessary because [[../libraries/brain-roadmap]] `parseSpec` still serves `getRoadmap` / `getSpec` — readers don't cut over until [[spec-readers-from-db-retire-parser]]). The mirror commit is idempotent (same body in → same file out → `git diff` empty → skip).
- The mirror commit is the ONLY writer of `docs/brain/specs/*.md` after this spec lands. The legacy commit-from-author-path is gone.
- Verification — Phase 4: an author UPDATE that touches the DB → within one box poll, a single mirror commit appears on `main` carrying the rendered body. A no-op UPDATE (the same body re-saved) → no commit (idempotency). [[spec-readers-from-db-retire-parser]] is the spec that retires this whole phase.

## Safety / invariants
- **Database is the spec.** Every write goes through `createAdminClient()` (service role, [[../CLAUDE.md]]); no client-side spec/phase writes. RLS on `specs` + `spec_phases` enforces it server-side.
- **Workspace-scoped.** Every author surface threads `workspace_id` from the call site through to `upsertSpec`. A spec is owned by a workspace — never global (mirrors [[../tables/spec_card_state]]).
- **Phase IDs are stable.** The author path NEVER blows away a `spec_phases` row when re-saving — it UPSERTs by `(spec_id, position)` so the phase's `id + pr + merge_sha + created_at` survive an edit. Anything that destroys-and-recreates a phase row breaks the [[spec-status-phase-pr-provenance]] provenance and the lift-between-specs use case (the whole reason phases are a TABLE).
- **No bypass of the `in_review` build hard-stop.** [[spec-review-agent]] is the only writer that can flip a spec out of `in_review`; the build dispatch checks `specs.status` (not a cached value) on every queue.
- **The mirror commit is transitional.** This spec leaves a `kind='mirror-spec-md'` worker lane in place; [[spec-readers-from-db-retire-parser]] is the spec that deletes it. Until then, dual-write is mandatory — a missing mirror commit silently regresses the markdown-first readers.
- **No spec markdown is committed from the author path itself.** Verify: `grep` the author surfaces after this spec ships — no `git commit … docs/brain/specs/` in `goal-proposals.ts`, the director-coach `runDirectorCoachJob`, the planner runner, the regression `authorRegressionFixSpec`, Vale's fix path. The mirror is the ONE commit path, owned by the worker.

## Completion criteria
- Every author surface (planner / triage / fix-spec / regression / director split-author / Ada / Vale) writes `public.specs` + `public.spec_phases` rows on creation, with NO direct `git commit` of `docs/brain/specs/{slug}.md` from the author code path. `npx tsc --noEmit` clean.
- The build lane materializes the row to a temp file for the [[../skills/build-spec]] skill; a queued build picks the materialized path, opens the `claude/*` PR per the existing recipe, and on merge the matching `public.spec_phases` row flips to `shipped` with `pr + merge_sha` recorded.
- Editing a spec body or a single phase is a one-`UPDATE` operation that the DB trigger rolls up to `specs.status` — no Vercel deploy required.
- The `kind='mirror-spec-md'` lane appears on the box worker's lane list and commits each authored/edited spec to `main` as `docs/brain/specs/{slug}.md` — keeping the markdown-first readers ([[../libraries/brain-roadmap]] `parseSpec`) green. The commit is idempotent across re-saves.
- [[spec-review-agent|Vale]]'s in_review→planned/deferred verdicts are pure DB writes; `spec_status_history` still records the audit row per transition (no regression of the existing audit ledger).
