# PDP edge-served experiments ‚Äî variant at the edge, cached per arm ‚úÖ

**Owner:** [[../functions/platform]] ¬∑ **Parent:** the P2 of [[pdp-experiment-wiring]] (P1 shipped: PDP experiments apply, dynamic-when-testing). ¬∑ **Why:** P1 makes a tested PDP **fully dynamic** (no ISR cache) ‚Äî fine for now, but the PDP is the hot, high-traffic page. Edge-served A/B keeps it **edge-cached per variant** (fast *and* tested) instead of paying server render on every request. The owner's pattern: assign the variant at the **Vercel edge**, cache-key by it.

## The 5 pieces
1. **Active-experiment manifest in Edge Config.** Edge middleware can't afford a Supabase round-trip per request ‚Äî publish the running experiments (per `product √ó lander`: `variant_id`s, weights, `holdout_pct`, `promoted_variant_id`) to **Vercel Edge Config** (low-latency edge KV). The optimizer **re-publishes the manifest** on every experiment state change (`materializeCampaign` start, promote, kill, rollback) ‚Äî so the edge always has the current assignment set without a DB call. (If Edge Config isn't wired, a periodically-refreshed cached JSON blob the middleware fetches is the fallback.)
2. **Edge middleware** (`middleware.ts`, matched to the PDP route): read the manifest for this product/lander; **sticky-assign** by a deterministic hash of the visitor (the `sid` cookie) √ó experiment id, holdout-aware (same logic as `assignVariant`); set a `sx_variant=<experimentId>:<variantId>|holdout` cookie if absent. Internal/bot (`sx_internal`) ‚Üí no assignment.
3. **Cache-keyed-by-variant.** The middleware **rewrites** the request to a variant-keyed internal URL (e.g. add `?_sxv=<variantId>`), so each arm is a **distinct cacheable render** ‚Äî control = the real PDP (its existing cache), each variant its own edge-cached entry. (Next App Router: rewrite + the param participates in the cache key; or a `Vary`-by-cookie cache tag.)
4. **Page reads the assigned variant** (from the rewrite param / cookie) instead of assigning inline; applies the variant patch over `media_by_slot["hero"]` (reuse P1's apply); **stays cacheable** (no per-request `cookies()` assignment in the page ‚Äî the edge already did it). Exposure still emits for the assigned arm (internal/bot excluded), reusing the pixel path.
5. **Cache purge on content change** ‚Äî when a variant's hero is (re)approved/regenerated (the [[optimizer-hero-preview-gate]] flow) or an experiment promotes/ends, **purge that variant's cached PDP render** (revalidateTag / path) so the new content serves; on experiment end, drop the variant cookie/rewrite.

## Verification
- On the live storefront custom domain, with a PDP experiment running, `curl -s -D- 'https://{domain}/{handle}'` directly (no cookie) ‚Üí expect a `Set-Cookie: sx_variant=<expId>:<variantId>[:h]` header from the **edge middleware**; re-curl echoing that cookie ‚Üí expect the SAME `sx_variant` (sticky, no re-assign).
- `curl` the served-arm URL `https://{domain}/{handle}` with two different sticky `sx_variant` cookies that map to different variants ‚Üí expect a **different hero `<img>` URL** in the HTML per arm (the variant's `heroImageUrl` in the `hero` slot), and after warm-up `x-vercel-cache: HIT` on each variant URL (cached **per `_sxv`**, not one shared render).
- `GET https://{domain}/api/storefront/experiment-manifest` ‚Üí expect JSON `{ "<storefrontSlug>/<handle>": { experiments:[{ id, status, holdout_pct, variants:[{id,is_control}] }] } }` for each running/promoted PDP experiment, with `cache-control: ‚Ä¶s-maxage=15‚Ä¶`.
- Visit the PDP with `?sx_internal=1` (sets the cookie) ‚Üí expect **no** `sx_variant` assignment, no `_sxv` rewrite, and no `experiment_exposure` row for that session.
- A real (non-internal) visitor on a served arm ‚Üí expect one `experiment_exposure` (emitted client-side from the `sx_variant` cookie) and the [[storefront-test-detail-page|test detail page]]'s **PDP** arm session count to increment; a control/holdout visitor (cookie ends `‚Ä¶` / `:h`, no `_sxv`) also logs an exposure for the control arm.
- Start an experiment (`materializeCampaign`) ‚Üí within ~15s (blob fallback) / sub-second (Edge Config) the manifest route reflects it and a fresh visit gets assigned; promote ‚Üí the promoted variant's hero serves all non-holdout; kill/rollback ‚Üí `republishExperimentManifest` purges and every visitor reverts to the real cached PDP hero. A re-approved hero (going live via `materializeCampaign`) ‚Üí the PDP render purges and serves the new image.
- Negative: with **no** running PDP experiment, the manifest route returns `{}`, the middleware adds no `_sxv`/cookie, and the param-less PDP serves its normal static prerender (no per-request cost). The existing advertorial/listicle/before-after experiments (`?variant=‚Ä¶`) are untouched ‚Äî the middleware skips assignment when `?variant=` is present.

The optimizer re-publishes + purges on every PDP state change ‚Äî `materializeCampaign` (stand-up / approved-hero-goes-live) and `experiment-refresh` (promote / kill / rollback) call `republishExperimentManifest(admin, [productId])`.
