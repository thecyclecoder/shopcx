# Dashboard layout — wrap uncached auth (cookies/supabase.auth/workspace fetch) in <Suspense> so cacheComponents prerendering unblocks the build (round 4)

**Owner:** [[../functions/platform]] · **Parent:** extends [[../specs/regression-agent]] · **Regression-of:** [[pdp-edge-served-variant-use-cache-strip-dynamicparams]]
**Regression-signature:** `regression:pdp-edge-served-variant-use-cache-strip-dynamicparams:41a0f5dabbb9`

PR #618 stripped the last `dynamicParams = true` exports and the build no longer errors on 'dynamicParams is not compatible with cacheComponents' — but the SAME post-spec deploy still ERRORS, one layer deeper, with 'Route "/dashboard/...": Uncached data was accessed outside of <Suspense>' on every parallel prerender batch (different /dashboard/* pages reported each run because Next bails after the first 2 of 374 routes). Root cause: the dashboard layout does top-level uncached/runtime IO (supabase.auth.getUser + `cookies()` via getActiveWorkspaceId + getUserWorkspaces) and Next 16's cacheComponents (turned on in PR #607) refuses to statically prerender that — the migration guide explicitly directs to wrap runtime data access in <Suspense>. Move the layout's async auth gate into an inner component wrapped in `<Suspense>` so the static shell prerenders and the chain's verification (deploy lands READY, per-arm + bare PDP HITs restored) can finally hold.

## Offending change
Commit c620d402 / PR #607 'pdp-edge-served-variant-use-cache' enabled `cacheComponents: true` in next.config.ts. Under cacheComponents (per node_modules/next/dist/docs/01-app/02-guides/migrating-to-cache-components.md:57), top-level runtime/uncached IO in a layout or page on a static path blocks the static prerender and raises 'Route ...: Uncached data was accessed outside of <Suspense>'. The dashboard layout at src/app/dashboard/layout.tsx is exactly that pattern — line 17 `await supabase.auth.getUser()`, line 21 `await getActiveWorkspaceId()` (which calls `cookies()`), line 24 `await getUserWorkspaces(user.id)`. Every /dashboard/* prerender (374 routes, 29 parallel workers) inherits it; vercel inspect on shopcx-d42f2ujuj logs /dashboard/analytics/roas + /dashboard/settings/email-filters, on shopcx-7oxpsnc7f logs /dashboard/analytics/revenue + /dashboard/settings/text-marketing — different pages each batch because Next bails after the first 2 errors, but the issue is layout-level. The dashboard/brain/layout.tsx is the same pattern (await getBrainTree() — uncached fs read) and would break the very next prerender once the outer dashboard layout is fixed.

## Phase 1 — restore it
Refactor src/app/dashboard/layout.tsx: keep the existing imports, but split the default export into (1) a sync outer `DashboardLayout({ children })` that returns `<Suspense fallback={null}><DashboardLayoutInner>{children}</DashboardLayoutInner></Suspense>`, and (2) an `async function DashboardLayoutInner({ children })` that holds the existing await chain unchanged (createClient → auth.getUser → redirect('/login') if !user → getActiveWorkspaceId → redirect('/workspace/select') if !workspaceId → getUserWorkspaces → redirect if !current → return WorkspaceProvider/SectionNavProvider/Sidebar/PullToRefresh tree). Redirect() works fine inside a Suspense boundary during streaming. Same-PR sweep: src/app/dashboard/brain/layout.tsx — convert the same way (Suspense-wrap an async BrainLayoutInner that calls `await getBrainTree()`), OR add `'use cache'` + `import { cacheLife } from 'next/cache'; cacheLife('max');` at the top of the layout body (the brain tree is content-driven and slow-changing; the migration guide § '`dynamic = "force-static"`' notes 'use cache' is the recommended pattern for uncached data access in a layout). No other server layouts in src/app at static paths do uncached IO — confirmed: src/app/layout.tsx and src/app/(storefront)/layout.tsx are sync, src/app/widget/[workspaceId]/layout.tsx is sync, src/app/portal/[slug]/layout.tsx is dynamic-segment (not statically prerendered).
