# Positive-close gate must respect escalated_at; partial_refund must support native SHOPCX orders

**Owner:** [[../functions/cs]] · **Parent:** CS mandate "Ticket-derived product fixes" · **Derived-from-ticket:** `6ad65422-bbf6-48b4-b432-a881a026942f`

Close two cascading system bugs that today silently false-closed a score-2 ticket for a 14-renewal customer.

## Problem (from escalated ticket `6ad65422-bbf6-48b4-b432-a881a026942f`)
Ticket 6ad65422 was escalated by the action-executor (escalateTicket() set escalated_at + escalation_reason) after two action failures: (1) partial_refund on order_number SHOPCX33 returned "Could not resolve order" because src/lib/shopify-order-actions.ts:155-181 only knows how to refund through Shopify's REST API, but SHOPCX-prefixed native shopcx orders have shopify_order_id=NULL (see src/lib/order-number.ts — these are first-class native orders, not Shopify-imported); (2) {type:"escalate"} arrived as a directActionHandlers entry, which doesn't exist, so it returned "Unknown action type: escalate". When the customer next wrote "Thanks.", the deterministic positive-close gate at src/lib/inngest/unified-ticket-handler.ts:1481-1486 selected ONLY `agent_intervened, assigned_to` — it never checks `escalated_at` or `escalated_to` or `escalation_reason` — so suppression missed and the AI fired generatePositiveClose() with "thrilled I could help—enjoy your Superfoods products!" on a ticket whose two real problems (loyalty redemption + pricing concern) were 100% unaddressed. The Sonnet 'Escalated/agent-assigned ticket — hard gate, every turn' rule cannot save this — the positive-close branch runs deterministically BEFORE the orchestrator. As native shopcx orders proliferate, more refund attempts will route through partialRefundByAmount with a SHOPCX number, fail, escalate, and then false-close on the customer's next thank-you. We need to fix both bugs in one spec because they always co-occur.

**Likely target:** `Phase 1 (highest priority): in src/lib/inngest/unified-ticket-handler.ts:1481-1486, expand the positive-close suppression gate to additionally select `escalated_at, escalated_to, escalation_reason`; if `escalated_at` is non-null (regardless of agent_intervened/assigned_to), suppress positive-close and emit a system note: "Positive close suppressed — ticket is escalated (escalated_at=…, reason=…). AI must not close on the customer's behalf.". This single line of defense would have saved this ticket and every future one shaped like it. Phase 2: in src/lib/shopify-order-actions.ts:155-181 (partialRefundByAmount), when the resolved order row exists but `shopify_order_id` is NULL AND `braintree_transaction_id` (or equivalent native-payment column — probe the orders table) is non-null, fall through to a Braintree-direct refund path instead of returning "Could not resolve order". Add a unit test covering the SHOPCX-prefix native-order case. Phase 3 (optional polish): make directActionHandlers tolerant of {type:"escalate"} by routing it to the same escalateTicket() helper, so a model emitting the wrong shape still escalates cleanly instead of returning "Unknown action type". Verification: replay Jill's exact decision shape against the unified-ticket-handler in a test workspace; assert (a) positive_close is suppressed on a freshly-escalated ticket, (b) partial_refund on a SHOPCX-prefixed native order with a Braintree transaction succeeds end-to-end (or fails for a real Braintree reason, never for 'Could not resolve order'). Owner: function customer-experience. Parent: goal AI-first support reliability.`
