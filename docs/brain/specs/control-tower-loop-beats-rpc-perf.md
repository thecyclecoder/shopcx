# control_tower_loop_beats RPC — stop the 500 (bound the scan via lateral join) ⏳

**Owner:** [[../functions/platform]] · **Parent:** fixes the RPC shipped by [[control-tower-monitor-accuracy]]; complements [[control-tower-beats-read-failure-guard]] (which makes the snapshot *tolerate* this failure — this fixes the failure itself).

The Supabase error feed shows **`500 POST /rest/v1/rpc/control_tower_loop_beats` ×15** — the bounded-read RPC from [[control-tower-monitor-accuracy]] is **itself timing out**. Root cause (from the SQL): it computes `row_number() OVER (partition by loop_id order by ran_at desc)` **and** `count(*) OVER (partition by loop_id)` across **every** `cron`/`agent-kind` heartbeat row with **no time bound**, then filters `rn <= limit`. That's a full scan + sort/partition of the entire (ever-growing) cron+agent beat history on every snapshot call — the exact full-sort degradation it was built to replace (`GET /rest/v1/loop_heartbeats` 500). [[control-tower-beats-read-failure-guard]] (#203) makes the snapshot stay amber instead of false-red when this 500s — graceful, but the snapshot then has **no real beat data**, so the monitor flies blind. Fix the RPC.

## Fix — lateral join + drop the expensive all-time count
- **Latest N beats per loop via a lateral join, not a global window.** `select distinct loop_id … , lateral (select … from loop_heartbeats where loop_id = l.loop_id and kind not in ('inline-agent','reactive') order by ran_at desc limit p_history_limit)` — each lateral rides the `(loop_id, ran_at desc)` index and reads ≤N rows per loop, instead of scanning + sorting the whole table.
- **Drop `count(*) OVER partition` (the costly part).** The monitor only needs **"has this loop EVER beaten?"** (the never_fired signal). A loop with zero beats simply **doesn't appear** in the distinct-loop_id set → absent ⇒ 0 beats ever ⇒ never_fired candidate; present ⇒ has beaten. No all-time count needed. (If a count is ever wanted, get it from a separate cheap `group by` — never a per-row window.)
- **Optional time floor** on the latest-N read (e.g. `ran_at > now() - interval '90 days'`) to cap the per-loop index scan, since the monitor only cares about recent freshness anyway — without breaking the ever-beaten signal (presence in the distinct set is computed table-wide / cheaply).

## Verification
- `POST /rest/v1/rpc/control_tower_loop_beats` returns in well under the statement timeout on the live table; the Supabase feed shows **no new `control_tower_loop_beats` 500s**.
- `buildControlTowerSnapshot` gets real per-loop beats again (not the #203 amber-fallback): each loop shows its true latest beat + history strip; never_fired still fires only on 0-beats-ever loops.
- `EXPLAIN` shows the lateral using the `(loop_id, ran_at desc)` index (index scan, not a full sort over all rows).
- Negative: a loop with zero beats is still correctly absent (never_fired candidate); a loop with history is present (not never_fired).
