# Platform/DevOps Director agent

**Owner:** [[../functions/platform]] · **Parent:** M4 — Platform/DevOps Director agent
**Blocked-by:** [[approval-routing-engine]], [[directors-board-gamified]]

The **first live director** of the [[../goals/devops-director]] goal — the agent that takes the CEO out of platform operations. A new [[../tables/agent_jobs]] `kind` (`platform-director`) that **investigates → auto-approves** its routed inbox **within the leash**, **escorts approved goals through their milestones**, **watches the platform**, and **reports up in human terms** — escalating only the genuinely high-stakes calls. It supervises the **mature tools we already have** ([[../specs/repair-agent|repair]], [[db-health-agent]], [[coverage-auto-register-agent|coverage-register]], the builder chain + auto-ship + fold, the box, [[../dashboard/control-tower]]) — it **orchestrates, it does not rebuild**. It flips Platform's M2 `live + autonomous` flag on, so the [[approval-routing-engine|routing engine]] silently re-routes platform approvals from the CEO inbox to it. Today [[../functions/platform]] owns those tools but **nothing supervises them as a director**, [[../tables/agent_jobs]] has no `platform-director` kind, and the leash in the goal is unenforced. Success metric served: **% of platform approvals the CEO never touches** (auto-handled with audited history) trending up, and **goals escorted to completion without CEO babysitting**.

## Phase 1 — the `platform-director` job kind + investigate→auto-approve (within the leash)
- ✅ shipped
- A new [[../tables/agent_jobs]] `kind='platform-director'` with its own box lane (`scripts/builder-worker.ts` `runPlatformDirectorJob`, mirroring `runCoverageRegisterJob`/the repair runner), claimed via `claim_agent_job`. Runs as a Max `claude -p` box session (read-only prod DB + brain), like dev-ask/spec-chat. The cheap throttled enqueuer `enqueuePlatformDirectorJobs` (poll-loop background check, ~1 min) queues one director job per open Platform-routed Approval Request; **dormant (a no-op) until Platform is flipped `live + autonomous` in Phase 4** — `resolveApprover` routes nothing here until then. Logic lives in [[../libraries/platform-director]]; no migration (`agent_jobs.kind` has no CHECK).
- For each Approval Request routed to **Platform** ([[approval-routing-engine]]): read the *cause + proposed fix*, confirm it is **sound + low-risk + within the leash**, then approve via the existing approve path — writing an [[../tables/approval_decisions]] row with its reasoning. **Never rubber-stamps**: a request it cannot confirm sound, or that falls outside the leash, **escalates to the CEO** (Phase 3) rather than approving.
- **The leash — auto-approves (no CEO):** error fixes · db indexes/health · **additive / reversible** migrations · **milestone progression of an already-approved goal** · platform-monitoring fixes (the [[../goals/devops-director]] § leash + the standing autonomy rule in [[../operational-rules]]).

## Phase 2 — escort approved goals through their milestones
- ✅ shipped
- The chain-driving done by hand becomes the director's job: read goal/milestone state via [[../libraries/brain-roadmap]] `getGoals()`/`specCompletion`, and drive each approved goal's unblocked specs through **self-sequence → build → merge → fold** — leaning on the existing [[../specs/spec-blockers|blocked_by auto-queue]] (`autoQueueUnblockedBy` / `reconcileMergedJobs`) and the builder chain rather than reimplementing it. As a milestone's last blocker ships, the next unblocked spec auto-queues; the director confirms each landed clean and advances the goal.
- Milestone progression of an **already-approved** goal is inside the leash (auto). **Starting a new goal is not** (Phase 3 escalation).
- **Shipped:** [[../libraries/platform-director]] `escortApprovedGoals(admin)` + a throttled box sweep (`scripts/builder-worker.ts`, ~5 min, dormant until Phase 4 like the approval enqueuer). For each **approved** goal the director owns (`GoalCard.owner === platform` — new `Owner:` parse in [[../libraries/brain-roadmap]] `parseGoal` — with progress `0 < pct < 100`; a zero-progress goal is *unstarted* and escalates, never auto-starts), it queues every **unblocked, unshipped, never-queued** spec (`created_by=null`, same shape as `autoQueueUnblockedBy`/`queueNextChainedPhase` — chain/auto-ship/fold carry it from there), skipping blocked specs (the reactive auto-queue handles those on a blocker's merge) and `**Auto-build:** off` opt-outs, and logs an `escorted_goal` [[../tables/director_activity]] row per advance. Idempotent (a spec with a build job is *confirmed*, never re-queued). It never reimplements the build/merge/fold path.

## Phase 3 — loop-guard + CEO escalation (the high-stakes calls)
- ✅ shipped
- **Loop-guard:** the escort tracks build attempts/failures per spec ([[../libraries/platform-director]] `specBuildState` — counts `failed`/`needs_attention` `agent_jobs` build rows for the slug in the recent window). A build that **repeatedly fails** (≥ `PLATFORM_DIRECTOR_LOOP_GUARD_MAX = 2`) and has nothing in-flight → the director **stops**, diagnoses "likely a deeper issue," and **escalates to the CEO** to approve modifying the approach — **never an infinite resubmit loop** (a single prior failure under the cap is a bounded retry; the escort re-queues at most until the cap, then escalates).
- **Escalate UP to CEO** (route to the CEO inbox via M2, with the director's written diagnosis): a **repeatedly-failing build** (modify the spec/approach) · **destructive / irreversible** actions (a data-dropping migration, deleting infra) · **out-of-leash / unconfirmable** requests · **starting a NEW goal** (a zero-progress owned goal — only the CEO greenlights goals). Mirrors the standing autonomy rule (autonomous for low-risk/reversible; gate high-stakes/irreversible).
- **Shipped:** all escalations reuse the **existing M2 inbox** (a routed Approval Request notification — the inbox API shows an item to a role iff `metadata.routed_to_function === role`), never a parallel inbox:
  - `escalateApprovalRequestToCeo(admin, target, diagnosis)` — the runner's escalate branches ([[../libraries/platform-director|box lane]]) now **re-route** the declined Approval Request to the CEO (flip `routed_to_function` → `ceo`, prepend Ada's diagnosis to the body), instead of leaving it untouched. Creates a CEO-routed request if the reconciler hadn't emitted one yet (idempotent on `agent_job_id`).
  - `escalateDiagnosisToCeo(admin, {…})` — a standalone CEO-routed Approval Request + an `escalated` [[../tables/director_activity]] row for a high-stakes call with **no approvable target** (a loop-guard "deeper issue," or a zero-progress owned goal). **Notification-first, error-checked** (director-escalations-must-surface-to-ceo Phase 1): inserts the notification, checks its `error`, and only on success writes the `escalated` row + returns `{ emitted:true }`; a failed insert returns `{ emitted:false, error }` with no phantom `escalated` row (no more logged-but-invisible escalation). Deduped on a `dedupe_key` against an **existing notification** (not the `director_activity` ledger), so a never-surfaced escalation retries; once surfaced it pings once. Carries **no `agent_job_id`** so the inbox reconciler never reaps it.
- **Modifying or abandoning an approved goal** is never something the escort attempts autonomously — it only ever *advances* a goal within the leash, so there is no auto-path to gate; any such change remains a CEO call by construction.

## Phase 4 — watch the platform + report to the board
- ✅ shipped
- Read [[../dashboard/control-tower]] health (via its snapshot library) and post **human-readable** updates as 🛠️ Ada to the M3 board ([[directors-board-gamified]]) — what it squashed, what it's escorting, what it escalated — and contribute its slice of the **EOD recap**. Answers "why?" replies via the M3 dev-ask wiring.
- Flip Platform's [[approval-routing-engine|`live + autonomous`]] flag on as the activation switch (owner-confirmed), so platform approvals route to the director instead of the CEO.
- **Shipped:** [[../libraries/platform-director]] `postPlatformWatchUpdate(admin)` reads the **existing** [[../dashboard/control-tower]] snapshot (`buildControlTowerSnapshot` → the platform department rollup + its red loops) + today's [[../tables/director_activity]] (squashed = `approved_approval`, escorting = `escorted_goal`, escalated = `escalated`) and posts ONE conversational `update` as 🛠️ Ada to the [[../tables/director_messages]] board (`composePlatformWatchBody`). **Idempotent** per (workspace, UTC day) on `metadata.watch_date`; skips a fully-quiet, all-green day (no empty-board spam); **dormant** until live+autonomous like the rest of the leash. Driven on the daily beat by [[../inngest/platform-director-cron]] → the box **standing pass** (`scripts/builder-worker.ts` `runPlatformDirectorStandingPass` — the no-`target_job_id` branch of `runPlatformDirectorJob`, which now also runs `escortApprovedGoals` on the reliable cron beat).
- The other two Phase-4 surfaces **reuse what already exists** (supervise, don't rebuild): **"answers why?"** is the [[directors-board-gamified]] Phase-2 board→dev-ask wiring (`routeBoardReply` defaults the answer brain to Platform), and the **EOD-recap slice** is the [[directors-board-gamified]] Phase-4 [[../libraries/director-recap]] — Platform is a director, so its `approved_approval` / `escorted_goal` / `escalated` activity already rolls into the daily standup (`goalsAdvanced` ← `escorted_goal`, `bugsFixed`/`approvalsHandled` ← the director's `approval_decisions`). No parallel recap path.
- **Activation — applied 2026-06-23 20:35 (by `ceo`):** `scripts/apply-platform-live-autonomous.ts` upserted the [[../tables/function_autonomy]] `platform` row to `live + autonomous` (idempotent; reversible from the Agents hub). **Owner-confirmed prod action** — now that it has run, platform approvals route to the director and every surface above is live. (The "dormant until live+autonomous" notes above describe the conditional guard — what each surface does if the flag is toggled back off — not Platform's current state.)

## Safety / invariants
- **Never rubber-stamps.** Auto-approval requires the director to confirm cause + fix are sound, low-risk, and within the leash, with the reasoning logged to [[../tables/approval_decisions]] — an unconfirmable request escalates, it does not auto-approve ([[../operational-rules]] § North star).
- **The leash is hard.** Destructive/irreversible actions, modifying/abandoning a goal, a repeatedly-failing build, and starting a new goal **always** escalate to the CEO — the director can widen its own envelope only via M5 grading, never unilaterally.
- **No infinite loops.** The loop-guard stops a build that repeatedly fails on the same error and escalates a diagnosis — never blind resubmission.
- **Supervise, don't rebuild.** The director orchestrates repair / db-health / coverage-register / the builder chain / control-tower — it adds **no** parallel implementation of work those tools already do.
- **Auditable + supervisable.** Every decision (auto-approve or escalate) is logged with reasoning the CEO can read after the fact — the supervisable-autonomy contract (CEO → Director → tool).

## Completion criteria
- A `platform-director` [[../tables/agent_jobs]] kind + box lane exists; it processes Platform-routed Approval Requests, auto-approving only within the leash with logged reasoning.
- It escorts approved goals through milestones (self-sequence → build → merge → fold), reusing the blocked_by auto-queue + builder chain.
- The loop-guard stops repeated same-error builds and escalates; all high-stakes calls (destructive, goal-modify, new goal, repeatedly-failing build) escalate to the CEO.
- It posts human-readable updates + an EOD-recap slice to the M3 board and answers "why?" replies.
- Platform's `live + autonomous` flag flips on so the routing engine sends platform approvals to it; brain pages written + cross-linked from [[../goals/devops-director]].

## Verification
- Route a low-risk Platform approval (e.g. an additive migration / a db-index fix) → expect the `platform-director` job to auto-approve it within one pass, the underlying job to flip `queued_resume`, and an [[../tables/approval_decisions]] row with `decided_by='director'`, `autonomous=true`, and the director's reasoning.
- Route a **destructive** action (a data-dropping migration) → expect the director to **escalate to the CEO**, never auto-approve: the `platform-director` job completes with an `escalated` [[../tables/director_activity]] row, and the target's routed Approval Request notification flips to `metadata.routed_to_function='ceo'` with Ada's diagnosis prepended to the body — so it now appears in the **CEO** inbox (`GET /api/developer/agents/inbox?role=ceo`) and **disappears** from Platform's (`role=platform`). The target job stays `needs_approval` (never auto-approved).
- Make a build fail repeatedly on the same error (≥ 2 `failed` build rows for an escorted spec, nothing in-flight) → on the next escort sweep expect the loop-guard to **stop** resubmitting (no new `kind='build'` row) and instead emit a **CEO-routed** standalone Approval Request (`metadata.routed_to_function='ceo'`, `escalation_kind='loop_guard'`, deep-linking the spec) **and** an `escalated` `director_activity` row (`metadata.dedupe_key='loopguard:{slug}'`, `failed_attempts ≥ 2`). Re-run the sweep → expect **no duplicate** escalation (deduped on the existing notification). A spec with exactly **one** prior failure and nothing in-flight → expect a single bounded **retry** build (`instructions` containing `re-attempt #2`), not an escalation.
- With Platform live+autonomous, an **owned goal at 0%** (approved-but-unstarted) → on the escort sweep expect a **CEO-routed** "Greenlight needed" Approval Request (`escalation_kind='new_goal'`, deep-linking the goal) + an `escalated` `director_activity` row, and **no** auto-started build; re-run → no duplicate (deduped on `newgoal:{slug}`).
- Advance an approved goal (Phase 2): with Platform live+autonomous, an approved goal it owns (`0 < pct < 100`) that has an **unblocked, unshipped, never-queued** spec → on the next ~5-min escort sweep expect a `kind='build'` `agent_jobs` row for that spec (`created_by=null`, instructions starting `Escorted by the Platform/DevOps Director:`) **and** a `director_activity` row `action_kind='escorted_goal'`, `director_function='platform'`, `metadata.autonomous=true` naming the goal + queued slug(s). Re-run the sweep → expect **no duplicate** build row (the spec is now confirmed in-flight, not re-queued). A spec still blocked by an unshipped prerequisite is **not** queued (the reactive auto-queue fires when its last blocker ships). With Platform **not** live+autonomous → expect the sweep to be a no-op (zero rows). The board post describing the advance lands in Phase 4.
- **(Phase 4)** With Platform live+autonomous, on the daily [[../inngest/platform-director-cron]] beat (or Invoke it) the box claims the no-`target_job_id` standing `platform-director` job and runs the standing pass: after a day with director activity (a `director_activity` `approved_approval` / `escorted_goal` / `escalated` row), expect ONE 🛠️ Ada `kind='update'` [[../tables/director_messages]] post on the **Messages** board (`author='director'`, `author_function='platform'`, `metadata.source='platform-watch'`, `metadata.watch_date`=today) reading `🛠️ Platform watch — {health} . Today: {squashed N fixes · escorted M goals · escalated K to you}.` — the health half drawn from the Control Tower snapshot's platform department (`all N platform loops green` / `K red (…)` / `X/N green, M degraded`). Re-run the standing pass the same day → **no** duplicate post (idempotent on `watch_date`). A fully-quiet, all-green day → **no** post (`reason='quiet'`). With Platform **not** live+autonomous → the watch is a no-op (`reason='dormant'`, zero posts).
- On the M3 board, reply "why?" under Ada's watch post → expect the directors-board-gamified Phase-2 dev-ask wiring to route it (a `kind='dev-ask'` `agent_jobs` row, an "Ada is investigating…" indicator) and Ada's answer to post back as a threaded `reply`.
- Trigger `director-recap-cron` after a day Platform was active → expect Platform's slice in the EOD recap (a 🛠️ Ada `kind='recap'` post + Daily Summary) reflecting its `goalsAdvanced` (from `escorted_goal`) + `bugsFixed`/approvals (from its `approval_decisions`), and Platform summed into the CEO roll-up — no new recap code, the existing director-recap covers it.
- **(Activation, owner-confirmed prod)** Run `npx tsx scripts/apply-platform-live-autonomous.ts` → expect the [[../tables/function_autonomy]] `platform` row at `live=true, autonomous=true`. Confirm new platform approvals then route to the director (not the CEO inbox), appearing in CEO **Decision history** with `decided_by='director'`, `autonomous=true`.
