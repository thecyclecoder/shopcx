# Persist-to-renewal offer lever (storefront optimizer) ✅

**Owner:** [[../functions/growth]] · **Parent:** M6 — dynamic pricing-rules for persist-to-renewal offers
**Blocked-by:** [[storefront-dynamic-renewal-offers]]

The **approval-gated persist-to-renewal offer lever** for the [[storefront-optimizer-agent|M4 agent]] — the highest-stakes lever in the offer space. A *first-order* offer stays a coupon ([[../tables/coupons]], low-risk, autonomous); an offer that **persists to renewal** must become a dynamic [[../tables/pricing_rules]] offer (the model lands in the parent's P1) and is **always owner-approved** because it bleeds margin on every renewal. This spec is that lever, wired into the optimizer behind the build-approval gate.

## Phase 1 — the approval-gated offer lever
- ✅ shipped (2026-06-24)
- ✅ Added an `offer` variant kind to the [[storefront-optimizer-agent|M4 agent]] / [[storefront-experiment-bandit-framework|M1 framework]] (`OptimizerVariantPlan.kind:'offer'` carrying a typed `OfferPlan` = `offer_type ∈ subscribe_discount_pct｜fixed_renewal_price`, the delta, `ends_at`, optional rationale).
- ✅ On a `lever_class:'offer'` propose, the worker calls [[../libraries/storefront-optimizer-agent]] `proposeRenewalOfferCampaign` → inserts the [[../tables/pricing_rule_offers]] row (`status='proposed'`, inactive), the [[../tables/storefront_experiments]] shell (`status='draft'`), and the control + offer arms; surfaces a `storefront_offer` pending action carrying the offer id + plan. NEVER auto-activates — bleeds margin on every renewal.
- ✅ On owner approval, the worker calls `activateRenewalOfferCampaign` → flips the offer `proposed → active` (+ `approved_by/at` + `activated_at`) and the experiment `draft → running` (+ `started_at`). The M1 bandit then runs it vs the holdout and attributes outcomes on the predicted-LTV proxy exactly like any other lever (the offer's renewal-margin cost flows through [[../libraries/pricing]] `resolveSubscriptionPricing`, so a margin-bleeding offer that wins first-order but loses on LTV is caught).
- ✅ Checkout wiring ([[../libraries/storefront-optimizer-agent]] `resolveSubscriptionOfferId` → `src/app/api/checkout/route.ts`): when a checkout's session is on the offer ARM of a running offer experiment whose product is in the cart, the matching sub bucket carries `pricing_offer_id` (a REFERENCE, not a baked price) → renewals apply the delta automatically while the offer is active + in-window.
- ✅ Lever taxonomy: seeded `persist_to_renewal_offer` chapter + the two component levers (`renewal_discount_pct`, `renewal_fixed_price`) in [[../tables/storefront_levers]] so `nextLeverToTest` ranks the offer lever alongside the others. First-order-only discounts stay on the autonomous [[../tables/coupons]] path; only persist-to-renewal offers hit this gate.

## Phase 2 — guardrails, expiry + rollback
- ✅ shipped (2026-06-24)
- ✅ Margin floor: [[../tables/storefront_optimizer_policy]]`.min_renewal_margin_pct` (default 0.40, per migration `20260624140000_renewal_offer_lever.sql`) is the workspace rail. `proposeRenewalOfferCampaign` calls `computeModeledRenewalMargin`; below the floor, the proposal is REFUSED and the worker writes a [[../tables/director_activity]] row (`director_function='growth'`, `action_kind='escalated_margin_breach'`) with the modeled-margin diagnostic + offer terms — escalated to Growth + CFO, never surfaced as a normal proposal. COGS source missing on this workspace today (no `product_variants.cogs` column yet) → `cogs_source_missing=true` on the offer row SOFT-passes the floor and the audit honestly records the gap; the floor will bite once COGS lands.
- ✅ Auto-expire: `expireRenewalOffers` is called every refresh in [[../inngest/storefront-experiments]] (`refreshStorefrontExperiments`). Every `active` offer past `ends_at` → `status='expired'`, `expired_at=now()`, `deactivation_reason='auto_expire'`, and every `subscriptions.pricing_offer_id` referencing it is NULL'd — the sub reverts to base pricing on its next renewal automatically.
- ✅ Rollback: `rollbackRenewalOffersForExperiment` runs in the M1 refresh's rollback branch (LTV regression / refund spike) AND the kill branch — every linked offer (`proposed`/`approved`/`active`) is expired (`deactivation_reason='experiment_rollback: <reason>'`) and every `pricing_offer_id` referencing it is NULL'd, so a persist-to-renewal offer that touched real renewals is cleanly un-touched within the next refresh cycle.

## Safety / invariants
- **Never autonomous.** A persist-to-renewal offer is ALWAYS owner-approved before activation (it bleeds margin on every renewal). The agent proposes; the owner disposes. First-order coupons stay autonomous; persist-to-renewal offers do not.
- **Margin-floor hard rail.** No offer below the modeled renewal-margin floor reaches the owner; breaching escalates to Growth + CFO via [[../tables/director_activity]] ([[../operational-rules]] § North star).
- **LTV-accounted.** The offer's renewal-margin cost flows into the M3 predicted-LTV proxy ([[../libraries/pricing]] `resolveSubscriptionPricing`), so a first-order win that churns is caught by the reconciler, not rewarded.
- **Compliance.** No misrepresented pricing; honest offer labeling (brand voice + supplement compliance, the goal § hard rails).
