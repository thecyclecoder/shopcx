# Goal fold writes the brain from a DB row + flips the row to `folded`

**Owner:** [[../functions/platform]] · **Parent:** [[../goals/db-driven-specs]] M5 — Goals + milestones are data too (the top two tiers)
**Blocked-by:** [[goal-readers-from-db-retire-parsegoal]]

The endgame for the goal arc, parallel to [[spec-fold-from-db-row]] on the spec side. Once nothing READS goal markdown ([[goal-readers-from-db-retire-parsegoal]] retired `parseGoal`) and nothing WRITES it from the author path ([[goal-greenlight-button-and-author-writes-db]] moved authoring to row writes), only ONE thing should ever touch `docs/brain/goals/*.md` again: **goal fold**, and only on a COMPLETE goal (every child `goal_milestones` row `status='complete'`, which by the [[goals-milestones-tables-and-backfill]] rollup means every child `public.specs` row is `shipped` or `folded`). Fold extracts the goal's permanent record into the brain (a fully-folded narrative under `goals/{slug}.md` — the canonical historical artifact — plus cross-links updated on the lifecycle/dashboard/function pages it touches), flips the `public.goals` row to `status='folded'` (PRESERVED — audit + the board's archive view), and removes the legacy `docs/brain/goals/{slug}.md` only if it was a backfilled leftover (newly-authored goals never had a `.md` post-[[goal-readers-from-db-retire-parsegoal]], so the `git rm` is a no-op for them). Retire the last of the goal-status mirror — `setGoalStatusLine` is already gone post-[[goal-readers-from-db-retire-parsegoal]]; this spec confirms there's no markdown-status writer left anywhere. Closes the gap from [[../goals/db-driven-specs]] M5 ("Fold writes the goal's brain record on completion"): today the [[../skills/fold-to-brain]] skill folds SPECS only — there's no goal-fold mechanism, so completed goals sit as `.md` forever without ever being folded into a permanent brain artifact.

## Phase 1 — goal fold reads the complete `goals` row, writes the brain, flips to `folded`
- Extend the [[../skills/fold-to-brain]] skill (or author a sibling `fold-goal-to-brain` skill — the author calls it; the simpler shape is one skill that handles both, dispatching on the row's table) to read `public.goals` + `public.goal_milestones` + the joined child `public.specs` rows for the goal slug via [[../libraries/goals-table]] `getGoal` (from [[goals-milestones-tables-and-backfill]]).
- Guard: refuse to fold a goal whose `goals.status` is anything other than `complete`. The [[goals-milestones-tables-and-backfill]] DB rollup ensures `complete` ≡ every milestone is `complete` ≡ every spec is `shipped` or `folded` — so the guard is the same guarantee written twice (app + DB rail). A `greenlit` goal with all-shipped children is `complete` automatically per the rollup; a `proposed` goal can never be `complete` (the rollup refuses to auto-flip `proposed→complete` per the [[goals-milestones-tables-and-backfill]] rail).
- Render the goal row as the in-memory text the fold-agent reads (the H1 / **Owner:** / **Outcome:** / **Why now:** / **Model:** / **Target:** / **Success metric:** / **Decomposition** shape — derived from the row + its `goal_milestones` children + the joined `specs` per milestone). NO `.md` file is touched in this step.
- The fold-agent extracts the goal's permanent record into the brain: a goals/{slug}.md REWRITTEN as the final folded narrative (what this goal accomplished, the milestones that landed, the specs that shipped, the residual brain pages the work created/changed), cross-links updated on the [[../lifecycles/]] / [[../dashboard/]] / [[../functions/]] / [[../tables/]] / [[../libraries/]] pages the work touched. Mirrors the [[../skills/fold-to-brain]] shape for specs ([[../skills/fold-to-brain]] today writes the lifecycle/library/etc. pages); a folded GOAL gets ITS OWN final goals/{slug}.md as the canonical historical artifact (the goal's body is too large to dissolve only into the lifecycle pages — the goal is itself a multi-spec narrative).
- After the brain commits succeed, the worker flips `public.goals` row: `UPDATE public.goals SET status='folded', updated_at=now() WHERE id=$1`. Every column (title, body, outcome, success_metric, owner, parent_goal_id, milestones via the FK) stays — the board's archive view + audit history can render the folded goal unchanged.
- Cross-link the goal's archive entry: append to `docs/brain/archive.md` and (per the existing fold convention) `docs/brain/archive.d/goal-{slug}.md` for the git-history surface.
- Verification — Phase 1: fold a `complete` goal end-to-end; expect the final goals/{slug}.md committed as the canonical folded narrative, the lifecycle/library/dashboard/function brain pages cross-linked, `goals.status` flipped to `folded`, the row otherwise intact, and the archive index updated.

## Phase 2 — the board's archive view reads `folded` goal rows; retire any goal-status mirror residue
- The board's archive surface (the goals board's "Archive" filter, the goal detail page when `goals.status='folded'`) reads `goals.status='folded'` rows directly — no markdown parse, no overlay (already true post-[[goal-readers-from-db-retire-parsegoal]]).
- Any goal-status mirror residue ([[../libraries/brain-roadmap]] `GoalCard.status` overlay if any survives, the `flags.*` slots on [[../tables/spec_card_state]] that pointed at a goal's parent — already moved to `public.specs.intended_status` in [[spec-authoring-writes-db-and-worker-materialize]]) retires here. `grep -rn "GoalCard.*overlay\|setGoalStatusLine" src/` returns zero matches.
- The `outputFileTracingIncludes` entries for archive.md / archive.d (`next.config.ts` L24) STAY (they back the [[../specs/spec-test-agent]] read of shipped-unverified specs and the goal-archive page if it reads the markdown for any legacy use). Out of scope to retire those.
- Verification — Phase 2: a folded goal renders identically on the board's archive view as on the live `complete` view pre-fold (the row keeps all its data); the folded goal markdown is the canonical narrative anyone reading the brain finds.

## Safety / invariants
- **Fold ONLY a `complete` goal.** The guard is in the fold entrypoint; refuse if `goals.status !== 'complete'`. The [[goals-milestones-tables-and-backfill]] rollup ensures `complete` ≡ every milestone is `complete` ≡ every spec is `shipped` or `folded` — so a partial-progress goal can never sneak through.
- **Preserve the row.** Never `DELETE FROM public.goals` in fold. The row is the audit + the board's archive card + the historical record of the company's BHAGs. Use `status='folded'` exclusively. Same rail as [[spec-fold-from-db-row]] on the spec side.
- **The folded goals/{slug}.md is the canonical historical artifact.** Unlike spec fold (which dissolves the spec's body into lifecycle/library/etc. pages and `git rm`s the spec file), goal fold REWRITES `docs/brain/goals/{slug}.md` as the final narrative — a goal is a multi-spec story that deserves its own permanent page. The lifecycle/library/dashboard/function pages get cross-links; the goal page gets the narrative.
- **Workspace-scoped + service-role writes.** The status flip and the brain commits all go through `createAdminClient()`; RLS keeps direct-from-browser writes off.
- **Idempotent re-fold is a no-op.** Re-running fold on an already-`folded` goal is rejected (`goals.status='folded' → already folded`); the brain pages are NOT re-written.
- **The fold-agent NEVER writes outside its commit scope.** Verify each fold commit contains exactly the brain-page edits expected (the goal page + the cross-linked lifecycle/library/etc. pages + the archive index) — a fold commit with stray edits is flagged as a regression.
- **No goal-status markdown writers survive.** `setGoalStatusLine` is gone post-[[goal-readers-from-db-retire-parsegoal]]; any new writer is a regression of the whole arc. Test by attempting to apply a `goals.status` flip via a markdown commit — the worker should reject it (no such code path exists; the `kind='mirror-goal-md'` lane is gone).

## Completion criteria
- The fold skill reads from `public.goals` + `public.goal_milestones` (joined to child `public.specs`). A real `complete` goal, folded end-to-end, produces the canonical folded `docs/brain/goals/{slug}.md`, cross-links the lifecycle/library/dashboard/function pages it touches, flips `goals.status` to `folded`, preserves the row, and appends to the archive index. `npx tsc --noEmit` clean.
- The fold guard refuses to fold a non-`complete` goal (manually set `goals.status='greenlit'` → expect a clear "goal is not complete" error; no brain edits made).
- The board's goal-archive view reads `goals.status='folded'` rows directly; the folded goal card renders the goal's milestones + the specs that shipped under each, sourced from the row + its joined children.
- `grep -rn "setGoalStatusLine\|deriveGoalStatus\|parseGoal" src/` returns zero matches — confirming the goal-status markdown machinery is fully retired across the goal arc.
- The success metric from [[../goals/db-driven-specs]] holds end-to-end: zero markdown-vs-DB drift for goals (the rollup triggers are the rails); a goal is greenlit with a one-click button (a DB flag, never a git commit); fold writes the brain page + flips the row to `folded` (preserved).
- Brain pages: UPDATE [[../skills/fold-to-brain]] SKILL.md (handles a goal row OR a spec row, dispatching on table), UPDATE [[director-judgment-lanes-fold-author-dismiss]] (the director's fold lane can fold a goal too), UPDATE [[../goals/db-driven-specs]] (this goal itself will be the first goal-fold target once every milestone ships — the test case for the whole arc).
