### What this query actually is
Statement `-7821780334453251234` is **PostgREST's per-request RLS context setup** — `set_config('search_path' | 'role' | 'request.jwt.claims' | 'request.method' | 'request.path' | 'request.headers' | 'request.cookies', …, true)` — issued by `@supabase/ssr` / PostgREST **once for every REST request** that carries auth context (cookies/JWT). It touches no table (EXPLAIN is a bare `Result`, cost 0.00..0.03), so there is **no index to add** and **no VACUUM to run** — confirming the `high_call_volume` diagnosis. The only lever is **fewer authenticated REST round-trips**. `calls = 38,888,209`, `mean = 0ms`, `total = 1,291,950ms` — pure volume.

### The dominant source + the fix
The dashboard left **sidebar** (`src/app/dashboard/sidebar.tsx`) is mounted for **every user on every dashboard page** and its `fetchCounts` effect fires **~11 authenticated API requests per tick** (ticket-views, 3× escalation ticket counts, fraud-cases, reviews, todos, escalated, improve-queue, branches, spec-test human-queue) — each API route makes several PostgREST round-trips, every one re-running this `set_config` statement. It was polling at **10s** despite its own comment reading "poll every 30s".

**Fix:** widened the sidebar poll `10000ms → 30000ms` (`setInterval(fetchCounts, 30000)`), aligning the code to its documented intent. This cuts the always-on, authenticated REST round-trip rate from this component **~3×** (≈6 ticks/min → 2 ticks/min × ~11 requests × multiple PostgREST calls each × N online users) — the single largest, lowest-risk reduction of statement `-7821780334453251234`. Badge counts tolerate 30s staleness. No DDL, no migration, no data change. Other pollers (notification-bell 60s; import/product-intel 3s) are either already conservative or transient/page-scoped, not always-on, so out of scope for this fix.
