# All goal readers from the DB ‚Äî retire `parseGoal` + goals file-tracing

**Owner:** [[../functions/platform]] ¬∑ **Parent:** [[../goals/db-driven-specs]] M5 ‚Äî Goals + milestones are data too (the top two tiers)
**Blocked-by:** [[goal-greenlight-button-and-author-writes-db]]

The goal-side cutover. Once every goal write goes to the DB ([[goal-greenlight-button-and-author-writes-db]] keeps a mirror commit in place specifically so the reader cutover is atomic here), flip every goal READER off [[../libraries/brain-roadmap]] `parseGoal` (L868) + `getGoals` (L1004) + `getGoal` (L1013) + `deriveGoalStatus` (L779) and onto `public.goals` + `public.goal_milestones` (from [[goals-milestones-tables-and-backfill]]). The roadmap goals board (`src/app/dashboard/roadmap/goals/page.tsx`), the goal detail page (`src/app/dashboard/roadmap/goals/[slug]/page.tsx`), the goals taxonomy view (`src/app/dashboard/roadmap/map`), the function detail page (`src/app/dashboard/roadmap/functions/[slug]/page.tsx` ‚Äî surfaces a function's owned/contributed goals), the agents-hub CEO profile (`getOrgChart` in `src/lib/agents/org-chart.ts` ‚Äî carries `ceo.goals` with `status + proposedBy`, [[director-proposed-goals]] Phase 2 surface), the [[../skills/plan-goal]] runner (reads `docs/brain/goals/{slug}.md` per the skill description), the goal-decomposition engine ([[goal-decomposition-engine]]), the decomposition gate (`src/app/api/roadmap/plan/route.ts`), the escort ([[../libraries/platform-director]] `escortApprovedGoals`), and the Slack flows all read directly from the rows. With nothing reading goal markdown, retire `parseGoal`, `getGoals`, `getGoal`, `deriveGoalStatus`, and `setGoalStatusLine` (the [[director-proposed-goals]] markdown-status writer ‚Äî already orphaned post-[[goal-greenlight-button-and-author-writes-db]]; this spec removes it entirely). Delete the goals entries from `next.config.ts` `outputFileTracingIncludes` (L24, 25, 27, 28). Retire the `kind='mirror-goal-md'` worker lane from [[goal-greenlight-button-and-author-writes-db]] Phase 4. Closes the gap: today `next.config.ts` traces `./docs/brain/goals/**/*.md` into multiple route bundles; the [[../skills/plan-goal]] skill description still says "read `docs/brain/goals/{slug}.md`"; and `parseGoal` is the goal-side twin of the spec-side `parseSpec` retired in [[spec-readers-from-db-retire-parser]].

## Phase 1 ‚Äî DB-first goal readers in `brain-roadmap`
- In `src/lib/brain-roadmap.ts`, replace `getGoals(workspaceId)` (L1004 ‚Äî `Promise.all(slugs.map(parseGoal(fs.read‚Ä¶)))`) with `getGoalsFromDb(workspaceId)`: joins `public.goals` + `public.goal_milestones` ordered by `position`, joins `public.specs` (which already roll up from `spec_phases`, per [[spec-body-table-and-backfill]]) for per-milestone progress, maps each row ‚Üí a `GoalCard` shape (the public type stays the same ‚Äî every downstream caller keeps compiling).
- `getGoal(slug, workspaceId)` (L1013) becomes a single join: `public.goals` + its `public.goal_milestones` + (for the existing `specs: Record<string, SpecCard>` field on the return value) the `public.specs` rows whose `milestone_id` belongs to those milestones ‚Äî plus any specs whose `parent` text matches the goal but `milestone_id IS NULL` (legacy / standalone).
- The `GoalCard.status` field maps directly from `goals.status` ‚Äî no `deriveGoalStatus` overlay needed. The `proposedBy` field maps from `goals.proposer_function`. The `pct` derived field is computed from the joined `specs` rows (same shape as today ‚Äî the spec-side already rolls up).
- `listGoalSlugs()` becomes a `select slug from public.goals`. Legacy no-arg callers get the `resolveDefaultWorkspaceId()` shim, same shape as [[spec-readers-from-db-retire-parser]] Phase 1.
- Verification ‚Äî Phase 1: `getGoals()` against a workspace returns the same cards (deep-equal) as the pre-cutover markdown path for a snapshot of every goal. `getGoal(slug)` matches one-for-one. `npx tsc --noEmit` clean (the `GoalCard` shape is stable ‚Äî only the SOURCE flips).

## Safety / invariants
- **Strict deploy order.** Phase 1+2 (DB-first readers) MUST deploy BEFORE Phase 3 (delete `parseGoal`) ‚Äî if the parser is gone but readers are still pointed at it, the board's goal view goes dark. Keep the mirror commit ([[goal-greenlight-button-and-author-writes-db]] Phase 4) running until Phase 2 ships.
- **The rollup triggers from [[goals-milestones-tables-and-backfill]] are the rail.** With `parseGoal` gone, `goals.status` is the SOLE truth for goal status; the trigger prevents a `complete` goal with a non-complete milestone, and the `setGoalStatus` route from [[goal-greenlight-button-and-author-writes-db]] is the only path that touches `proposed‚Üígreenlit`.
- **No reader reads the disk for goal content.** After Phase 3, `grep -rn "GOALS_DIR\|docs/brain/goals" src/` should return only the [[goal-fold-from-db-row]] writer and the [[../skills/plan-goal]] / [[../skills/build-spec]] worker-materialization paths (which write the temp file to a non-`docs/brain/goals/` dir). A regression here re-introduces the very drift the goal kills.
- **Workspace-scoped reads.** Every `getGoals`/`getGoal` call passes a `workspaceId` from the auth context.
- **No fallback to the parser.** The reader pages do NOT silently fall back to the markdown if the DB read fails ‚Äî they fail loud (a 500 with a clear error). Mirrors [[spec-readers-from-db-retire-parser]].
- **The `.md` files stay on disk until M5d fold rewrites them.** Don't `git rm` here ‚Äî [[goal-fold-from-db-row]] is the only path that writes/removes goal markdown.
- **The plan-goal skill description follows.** Update `.claude/skills/plan-goal/SKILL.md` to say "Read the goal ‚Äî `public.goals` + `public.goal_milestones` rows via [[../libraries/goals-table]] `getGoal`" (or "read the worker-materialized goal at `{path}` ‚Äî sourced from the DB row"). A skill description that still says "read `docs/brain/goals/{slug}.md`" is a documentation regression.
