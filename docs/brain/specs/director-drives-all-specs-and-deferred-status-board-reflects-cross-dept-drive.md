# The board + activity reflect cross-dept drive

**Owner:** [[../functions/platform]] · **Parent:** [[platform-director-agent]] + [[board-grooming]] — generalizes the auto-build lanes to all owners under [[../goals/devops-director]]

## Phase 1 — the board + activity reflect cross-dept drive
- Activity rows + the daily board-watch note when the platform director drives another department's spec (owner shown, driver = me), so you can see the keystone covering for not-yet-live directors. Surfaces on the [[Platform Department Scorecard]].
- **Shipped:** Every Platform-director **drive** lane that may cover a non-platform spec stamps `metadata.owner_function` on its [[../tables/director_activity]] row (driver stays `director_function='platform'`; the owner names the OWNING function). The stamp lands at: `escortFixSpecs` (`escorted_fix`) · `initiatePlatformSpecs` (`escorted_init`) · `groomBoard` (`groomed_continue` / `groomed_split` — both branches) · `applyDirectorFoldNow` (`groomed_fold_now` — reads `GroomCandidate.owner`) · `applyDirectorAuthorFollowup` (`groomed_authored_spec` / `init_authored_spec` — the lane passes the candidate's owner; `repair_authored_spec` skips, no spec owner) · `applyDirectorDismissCandidate` (`groomed_dismissed` / `init_dismissed`). Repair-dismissal kinds (`dismissed_repair` / `kept_repair` / `repair_authored_spec`) and `approved_approval` do **NOT** carry it — they aren't spec-on-the-board drives.
  - The daily watch (`composePlatformWatchBody`) reads today's rows, counts those whose `metadata.owner_function` is non-platform (filtered to `CROSS_DEPT_DRIVE_KINDS` so an `escalated` row that happens to be stamped doesn't double-count as a cover), and appends *"Drove N specs for other departments (cs: 2, growth: 1)."* on any day she covered ≥1 cross-dept spec. Per-owner breakdown sorted by count desc → name asc, capped at 4 named owners with a "+K more" tail. Omitted on a day with zero cross-dept drives (quiet days stay tidy).
  - `PlatformWatchActivity` extended with `crossDeptDrives` + `crossDeptByOwner` (the rollup the line reads); recorded into `director_messages.metadata.activity` so the post's structured record carries the same KPI the [[../goals/platform-department-scorecard|Platform Department Scorecard]] will trend.

## Verification
- On the box's standing pass, drive a CS or Growth spec (e.g. queue a `groomed_continue` / `escorted_init` / `escorted_fix` for a spec whose `owner` is `cs`/`growth`/`cmo`) → expect a [[../tables/director_activity]] row with `director_function='platform'` AND `metadata.owner_function='cs'` (or the relevant owner), recorded by the same single write the lane already made.
- On the next daily Platform watch post (`scripts/builder-worker.ts` `runPlatformDirectorStandingPass` → `postPlatformWatchUpdate`), inspect the board post body in `director_messages` → expect a trailing sentence "Drove N specs for other departments (cs: K, growth: J)." naming the owners + per-owner counts, on any day ≥1 cross-dept drive landed.
- On a quiet day (zero cross-dept rows in the UTC window), inspect the same post → expect the "Drove …" sentence to be **omitted** (no fabricated zero line).
- On the watch post's `metadata.activity` JSON, inspect `crossDeptDrives` + `crossDeptByOwner` → expect the totals match the per-owner sum (`Σ crossDeptByOwner = crossDeptDrives`) and the keys are the owners (`cs`, `growth`, `cmo`, …) excluding `platform`.
- Negative — drive a platform-owned spec (the normal case) → expect the activity row carries `metadata.owner_function='platform'` (the watch counter ignores it; no inflation of cross-dept counts) AND the watch sentence stays omitted unless a separate cross-dept drive also landed that day.
