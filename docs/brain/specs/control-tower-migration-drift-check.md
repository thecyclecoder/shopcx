# Control Tower — migration-drift check (schema vs migrations) ✅

**Owner:** [[../functions/platform]] · **Parent:** extends [[control-tower]] + [[error-feed-monitoring]]. · **Found in use 2026-06-22:** migration `20260618140000_meta_performance_tables.sql` was **silently skipped** in the prod apply pipeline — `meta_campaigns`/`meta_adsets`/`meta_ads`/`meta_insights_daily` never existed, every upsert hit `PGRST205`, and (pre-`meta-insights-ingest-empty-fix`) the error was swallowed → the iteration engine had **empty ROAS data for weeks**, found only by manual investigation. Nothing surfaced the drift.

A migration that doesn't apply is invisible: the code references a table, the table isn't there, and unless something fails loud (it usually doesn't), it degrades silently. This is exactly the silent-degradation class the Control Tower exists to catch — but it has no schema-vs-migrations check today.

## Model
- **The check:** parse every `supabase/migrations/*.sql` for `create table [if not exists] (public.)?<name>`, collect the set of tables the migrations *should* have created, and diff against the **live schema** (`information_schema.tables where table_schema='public'`). Any created-table that's **missing** = a silently-unapplied migration → surface it.
- **Where it runs:** the deployed Next runtime can't read the `.sql` files (not bundled), so run it where the repo + DB both exist — **the build box** (a periodic job, alongside its other scheduled work; it has the migration files in the working tree and an admin DB connection), OR a build-time generated `expected-tables` manifest the Control Tower cron diffs at runtime. Box-side is simplest (mirrors how this audit was just run by hand). Reuse the same parse+diff logic.
- **Surface:** on drift, record a Control Tower incident — a `loop_alert` / `error_events` row ("migration drift: table `X` from `<migration>` is missing") visible on `/dashboard/developer/control-tower`, and register the check as a monitored loop (so a *dead* check is itself visible, per the registry discipline).
- **Scope the noise:** ignore tables a later migration intentionally `drop`s (a `drop table` migration after the create), and tables created conditionally. Only a genuinely-expected-but-absent table alerts. Known-sunset systems can be allowlisted (e.g. don't alert on Klaviyo tables being retired).

## Verification
- On `/dashboard/developer/control-tower`, find the **Migration drift check** tile (Platform → Crons). With the schema in sync → green "ran Ns ago", `produced` shows `status:"ok"`, `expectedCount` (~188) `=` parsed migration tables, `liveCount` live tables, `missing:[]`. A stale/never-run check shows red (cron-freshness / never-fired) like any other loop — proof a *dead* check is visible.
- Point the box check at a DB missing a migration-created table (or `drop table <t>` on a non-prod DB whose migration creates it): within one ~30 min cycle the tile flips **red** "N migration tables missing from live schema", a [[../tables/loop_alerts]] incident opens with `reason='migration_drift'` (detail names the table + its migration), owners are paged, and a [[repair-agent]] `repair` job (signature `loop:migration-drift-check`) is enqueued. Confirmed in a harness: a live set minus `meta_campaigns` → `status:"drift"`, `missing:[{table:"meta_campaigns",migration:"20260618140000_meta_performance_tables.sql"}]`.
- A migration that `create`s then a later one `drop`s a table → NOT in `expected` → never flagged (harness: `001 create foo` + `002 drop foo` → `foo` absent from expected; a `003` re-create revives it).
- An allowlisted sunset table (`klaviyo_*`) missing → lands in `produced.allowlistedMissing` (surfaced for visibility), NOT `missing` → tile stays green, no alert. (Harness: `klaviyo_events` absent → allowlisted, silent.)
- Host without `SUPABASE_DB_PASSWORD` (can't read the live schema) → `status:"skipped"`, beat `ok:false` → tile amber "last run reported not-ok", never a false red.
- Negative: a live table with no migration (created ad-hoc) → not flagged — the diff is one-directional (migrations → schema; expected-but-absent only).

## Phase 1 — parse migrations, diff live schema, alert on missing ✅
Shipped. `src/lib/control-tower/migration-drift.ts` — the pure parse+diff: `parseExpectedTables` (every `supabase/migrations/*.sql` → net-of-drops created-table set, each mapped to its creating migration; drop-aware, temp/commented-DDL-safe, quoted/`public.`-qualified names) → `computeDrift` vs the live set → `runMigrationDriftCheck`. Runs as a periodic **box job** (`scripts/builder-worker.ts` → `runMigrationDriftJob`, ~30 min) because the deployed runtime can't read the `.sql` files; it reads the live `public` tables off the pooler (raw `information_schema.tables`) and writes the result into the `migration-drift-check` loop's heartbeat (`kind:'cron'`, `produced.missing`). The monitor's new **`migration-drift` output assertion** ([[../libraries/control-tower]] `monitor.ts`) reads that beat and flips the tile red → standard de-duped [[../tables/loop_alerts]] + page + repair path; cron-freshness keeps a dead check visible. Drop-aware + `MIGRATION_DRIFT_ALLOWLIST` (`klaviyo_*`) scope the noise. Registered in `registry.ts` (`MIGRATION_DRIFT_LOOP_ID`, owner platform).
