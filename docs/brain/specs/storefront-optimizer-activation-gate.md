## Model — per-campaign approval (the owner taps "Build" to run each test)
The control model the owner chose (2026-06-22): the agent **proposes** campaigns; **each proposed campaign is surfaced as a card with a one-tap "Build / Approve"**, and **tapping it is the owner's approval to run that test** (exactly like approving a spec build / a repair fix). Nothing runs on live traffic without that per-campaign tap — even reversible levers. This is *more* conservative than auto-run and is the starting mode; letting reversible levers auto-run (skip the tap) is a later Growth-director opt-in, not the default.

- **A `storefront_optimizer_policy` row** (per workspace): `active boolean`, `product_scope` (allowlist of `product_id`s — Amazing Coffee only to start), `auto_run_reversible boolean DEFAULT false` (the later opt-in above), plus editable guardrails (max concurrent experiments, min sample, holdout %, auto-rollback thresholds). Agent-legible + agent-writable so the **Growth director** operates it later — the engine **never writes its own policy**.
- **`active` = "the agent proposes campaigns at all."** When `active=true`: the loop runs, and each due `(product × lander × audience)` campaign is **proposed as a card** (hypothesis + cited funnel/lever reasoning + the variant it would test) in `needs_approval` — waiting for the owner's **Build/Approve** tap. On approval → the M1 experiment goes `running` (sticky variant + exposures). On the tap being withheld → nothing runs. When `active=false` → the agent doesn't even propose (fully idle).
- **Table default `active=false` (safe for any new workspace); SEED the Superfoods workspace `active=true`, `product_scope=[amazing-coffee]`, `auto_run_reversible=false`** — so the optimizer is **on in propose-and-approve mode, scoped to Amazing Coffee**, the moment M4 ships: it proposes, you tap to approve each test.
- **Scope is enforced, not narrative.** Every proposal + activation checks `product_id ∈ product_scope` — it *cannot* touch a non-scoped product even if a lander exists.
- **Control surface:** a dashboard on/off + per-product scope + the `auto_run_reversible` toggle (the Growth control surface). Offer/structural levers stay approval-gated regardless.

## Verification
- Migration converges on the stale table: after apply, `select count(*) from information_schema.columns where table_name='storefront_optimizer_policy'` = 16 (all 6 previously-missing columns present).
- After build + seed: `select active, product_scope, auto_run_reversible from storefront_optimizer_policy where workspace_id=<superfoods>` → `active=true`, `product_scope=[amazing-coffee]`, `auto_run_reversible=false` (on, in propose-and-approve mode, scoped).
- With `active=true` + `auto_run_reversible=false` (the seed): when M4 proposes a campaign, it lands as a **`needs_approval` card with a Build/Approve tap**; until tapped, `select count(*) from storefront_experiments where status='running'`=0 and no `experiment_exposure` events fire. Tapping **Build/Approve** → the experiment goes `running` (sticky variant + exposures begin).
- Scope enforced: a proposed/activated experiment for a non-`product_scope` product is refused (not just unscheduled), even if a lander exists.
- `active=false` → the agent doesn't propose at all (idle). An offer/structural lever stays approval-gated regardless of `auto_run_reversible`.
- Negative: with no approval tap, there is no path by which a campaign mutates live storefront content or assigns a live variant.
