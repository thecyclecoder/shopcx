# Harden the Fenn agent — roll persistent coaching into its mandate

**Owner:** [[../functions/platform]] · **Parent:** [[platform-director-agent]] — graduate persistent coaching into a durable fix (director grades agents: low score → fix spec, never a CEO escalation)
**Found in use 2026-06-30:** the **Fenn** agent (`fold`) sits at **6.0/10** after **2** coaching attempts that didn't stick. Rather than escalate to the CEO, roll the coaching into a permanent fix so the agent improves at the mandate level.

## The accumulated coaching to bake in (now archived as rolled-into-mandates)
- **When Fenn invokes the fold-batch job without first checking quota/infrastructure health, and produces no checkpointed or auditable progress mid-run, so any interruption (rate-limit, restart) results in zero recoverable output and any completion lacks verifiable evidence of correct page targeting, cross-links, and archival.:** When you see a fold-batch job starting, do two things before processing any item: (1) probe the API quota and surface a needs_attention status immediately if the weekly limit is at risk; then (2) after each folded item emit a structured checkpoint log line stating which brain page was updated, which cross-links were set, and that the item is archived — so that a restart can resume from the last confirmed checkpoint and graders can verify rubric compliance without reading the PR diff. — The recurring miss is that every failed run (rate-limit or restart) leaves zero recoverable work because there is no early-exit on quota and no per-item progress record; every 'completed' run still grades 6/10 because the log contains no auditable confirmation of page placement or cross-link integrity. Adding a quota preflight converts an opaque hard failure into a surfaced, retryable signal, and per-item checkpoint lines make both partial and full runs fully verifiable against the rubric's three criteria.
- **When Every low-graded run either hit an API quota/restart with zero recoverable output, or completed with no auditable trace of which pages were touched, cross-links verified, or archive confirmed — leaving graders unable to credit the work.:** When you see a fold-batch job starting, do two things instead of diving straight into folding: (1) run a pre-flight check against the API quota and surface a needs_attention status immediately if the budget is exhausted; (2) after each document is folded, emit a structured checkpoint line naming the brain page targeted, the cross-links set, and the archive status — so that if the job is interrupted, partial work is recoverable and every completed fold is auditable. — Rate-limit and restart failures score low not because Fenn caused them, but because there is no early detection and no durable partial output to show — fixing pre-flight catches quota exhaustion before wasted time, and per-fold checkpoints mean a restart never equals zero delivered work. On successful runs, the same checkpoint output gives graders the explicit page/cross-link/archive evidence needed to award a 10 rather than a 6.

## Phase 1 — bake the coaching into the agent
- Make the above coaching PERMANENT behavior of the `fold` agent — fold it into its prompt/mandate/code (its run-job + prompt in `scripts/builder-worker.ts` and the relevant `src/lib/agents/*`), not ephemeral appended `agent_instructions`. Once baked, the agent should follow it by default.
- Verify the agent's grade rollup recovers (≥ 7/10) over the next window of graded actions; if a coaching class still recurs, the bake-in missed it.
