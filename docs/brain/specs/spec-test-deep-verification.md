# Spec-Test Deep Verification (browser + sandboxed behavioral checks) ✅

**Owner:** [[../functions/platform]] · **Parent:** extends [[spec-test-agent]] (+ [[spec-test-maximize-machine-coverage]]). Reviewed the 55 needs-human checks (2026-06-21): ~30 are behavioral E2E ("fire event/call API/approve → assert DB+events") the agent punts only because it's read-only; ~8 are UI render/click-flow it can't do without a browser; ~15 are genuinely human (real customer/$/carrier side effects, fault-injection, aesthetics).

Give the spec-test agent two new **non-destructive-on-real-data** powers so most of those ~38 become machine checks, shrinking the human queue to the truly-human ~15. **Hard line preserved:** anything that hits a **real** customer, dollar, live Meta ad, or external carrier stays `needs_human` — the sandbox only ever touches dedicated test fixtures, never prod customers.

## Phase 1 — Headless-browser verification (Playwright) ✅
The agent can only `GET` endpoints today, so "the page shows X / click → Y" bullets get punted. Add a **headless-browser check tool** to the `spec-test` skill: load a dashboard page authed as the owner, assert rendered DOM / interact, screenshot on failure.

**Shipped (2026-06-21):** `playwright` added to `package.json`; the headless-browser check tool `scripts/spec-test-browser-check.ts` (mints an owner Supabase session server-side via the service-role admin — `generateLink`→`verifyOtp`→ @supabase/ssr cookie encoding + the `workspace_id` cookie — launches headless chromium authed as the owner, asserts rendered DOM / status / redirect / benign clicks, and uploads a full-page screenshot to the private `spec-test-evidence` bucket); the `spec-test` `SKILL.md` + `runSpecTestJob` prompt classify a RENDERED-UI bullet as a **BROWSER check** (auto, pass/fail + screenshot) not `needs_human`; `spec_test_runs.checks[].screenshot` carries the storage path, signed per-render (`signSpecTestScreenshot`) and rendered inline on **Developer → Spec Tests**. READ-ONLY firewall preserved: navigate/assert/benign-click only, never a mutating form submit. (Browser/system-deps provisioning — `npx playwright install --with-deps chromium` as root — is the owner's box op.)
- **Box provisioning (owner/root op — I handle it, like the Vercel CLI):** `npx playwright install --with-deps chromium` on the box as root; add `playwright` to `package.json` so the builder env has the library.
- **Headless owner auth (no human creds):** mint an owner Supabase session server-side via the service-role admin (`admin.auth.admin.generateLink` / a minted session for the owner user) and inject the cookie into the Playwright context — so the browser loads owner-gated pages without a password. Read-only navigation by default.
- **Skill classification:** a bullet asserting **rendered UI** ("card shows the Agent-tested stamp + chip", "VerificationCard renders per-bullet verdicts", "composer textarea ≥5 rows / auto-grows", "non-owner doesn't see the nav item") becomes a **browser check** (pass/fail with a screenshot as evidence), not `needs_human`. Pure click-flows that don't mutate real data (open a tab, expand a section, see a 403) run here too.
- **Guardrail:** the browser session is **owner-authed but read-only on prod** — it navigates + asserts; it does NOT submit forms that mutate real customer/billing data (those route to Phase 2's sandbox or stay human).

## Phase 2 — Sandboxed behavioral verification ✅
**Shipped (2026-06-22):** the SANDBOX check category — a fifth non-destructive execution mode. `src/lib/spec-test-sandbox.ts` ([[../libraries/spec-test-sandbox]]) holds the fixture identity (`SPEC_TEST_FIXTURES`), the `INTERNAL_ONLY_FLOWS` allowlist + `EXTERNAL_EDGE_EXAMPLES`, the `assertTestWorkspace` guard, the `nonTestWorkspaceFingerprint` isolation proof, and `mintOwnerCookieHeader`; `scripts/spec-test-sandbox.ts` is the CLI (`info｜isolation｜fire｜post｜cleanup`); `scripts/seed-spec-test-fixtures.ts` is the idempotent **gated** fixture seed; `workspaces.is_test` (migration `20260622120000`) is the sentinel. The `spec-test` `SKILL.md` + `runSpecTestJob` prompt classify an internal-only behavioral bullet as a SANDBOX check and a firewall-crossing one as `needs_human`.

The big bucket: "fire renewal-attempt → $0 comp order + event", "force-escalate → triage job + verdict + `triage_runs` row", "POST approve → job flips `queued_resume`", "`✓ Tested` upserts a row". All doable by driving the flow + asserting — **on test fixtures, never live customers.**
- **Dedicated test fixtures (I seed them via DB):** a flagged **test workspace** + test customer / ticket / subscription / audit rows (an `is_test`/sentinel marker), isolated from real data. The agent operates only within this fixture set.
- **Controlled-trigger capability** added to the box's spec-test toolkit (it KEEPS secrets, like the migration-fix agent): fire **Inngest events** (e.g. `internal-subscription/renewal-attempt`), call **internal POST endpoints** (`/api/roadmap/answer`, `/api/developer/spec-test/human-queue`, approve routes), and read back **DB rows + `customer_events`/`triage_runs`** to assert. Then it cleans up the fixture state (idempotent re-runnable).
- **🚨 External side-effect firewall (the core safety rule):** the sandbox runs a flow **only if every side effect stays internal** (DB writes + events). A flow that would call an **external API with real effect** — Amplifier fulfillment, Braintree charge/refund, Appstle mutation, Resend/Twilio send, a live Meta ad pause — is **NOT run**; it's classified `needs_human` (or deferred until a true sandbox tenant with test creds exists). The agent must prove a flow is internal-only before driving it. Comp-renewal's *event + DB + `type='comp'` txn + advance* is assertable; its *Amplifier handoff* is the external edge → assert up to the handoff, flag the handoff itself human.
- **Skill classification:** a behavioral bullet whose mutations are internal-only → **sandbox check** (drive on a test fixture, assert, clean up). Real-customer/$/external → stays `needs_human`.

## Verification
Phase 1 (shipped) — concrete, prod-facing checklist:
- On the box, `npx tsx scripts/spec-test-browser-check.ts --path "/dashboard/developer/spec-tests" --assert-text "Spec Tests" --slug spec-test-deep-verification --label smoke` → expect stdout JSON with `pass:true`, `httpStatus:200`, an `assertions` entry `{kind:"text",target:"Spec Tests",ok:true}`, and a non-null `screenshot` storage path (uploaded to the `spec-test-evidence` bucket).
- On the box, `npx tsx scripts/spec-test-browser-check.ts --path "/dashboard/developer/spec-tests" --role anon --assert-redirect "/login"` → expect `pass:true` with the anon (no-cookie) context redirected to `/login` (owner-gated page is not reachable unauthenticated) — the "non-owner is gated" browser check.
- In `package.json` → expect `playwright` present in `devDependencies`; on the box `npx playwright install --with-deps chromium` has been run (owner op) so chromium launches.
- On **Developer → Spec Tests** (`/dashboard/developer/spec-tests`) as owner, a spec whose latest run has a browser check with a `screenshot` → expand the check's **evidence** → expect the screenshot image rendered inline (signed short-TTL URL; opening it in a new tab works), and the summary line shows `· 📷 screenshot`.
- Probe DB: `npx tsx scripts/spec-test-db-probe.ts "select checks from spec_test_runs where checks::text like '%screenshot%' order by run_at desc limit 1"` → expect a row whose `checks[]` includes a `screenshot` path on a browser-check entry (after a run with a browser check has landed).
- Classification: re-run the spec-test suite over a spec with a rendered-UI bullet (e.g. [[spec-test-agent]]'s "shipped card shows the Agent-tested stamp + chip") → expect that bullet recorded as `verdict:"pass"`/`category:"auto"` with a screenshot, **not** `needs_human` (it leaves the human queue).
- Safety / READ-ONLY firewall: the browser tool exposes no input-typing/form-submit flags — confirm `scripts/spec-test-browser-check.ts` supports only navigate/assert/`--click` (benign) and that a bullet requiring a real customer/billing mutation stays `needs_human`. No prod customer/billing row is mutated by a browser check (it only uploads a screenshot to the internal evidence bucket).

Phase 2 (shipped) — concrete, prod-facing checklist (gated setup ops run first: `npx tsx scripts/apply-workspaces-is-test-migration.ts` then `npx tsx scripts/seed-spec-test-fixtures.ts`):
- Setup: `npx tsx scripts/apply-workspaces-is-test-migration.ts` → expect `✓ workspaces.is_test column present: true` (idempotent re-run is a no-op); `npx tsx scripts/seed-spec-test-fixtures.ts` → expect `✓ workspace … (is_test=true)`, the owner membership, the comp customer/subscription/ticket/migration_audit lines, and `✓ spec-test fixtures seeded`; a second run is a no-op reset.
- On the box, `npx tsx scripts/spec-test-sandbox.ts info` → expect JSON with `fixtures.workspaceId`, `internal_only_flows` including `comp-renewal-failclosed`/`human-queue-resolve`/`roadmap-answer`, and `external_edge_examples_NOT_runnable` including `comp-renewal-happy` blocked by `amplifier`.
- On the box, `npx tsx scripts/spec-test-sandbox.ts fire comp-renewal-failclosed` → expect `pass:true` with `asserts.failed_comp_transaction:true`, `asserts.comp_renewal_failed_event:true`, `asserts.no_order_created:true`, `asserts.billing_date_not_advanced:true`, and `isolation.zero_non_test_workspace_writes:true` (a `type='comp' status='failed'` transaction + a `subscription.comp_renewal_failed` `customer_events` row appear for the test comp sub; NO order; `next_billing_date` unchanged). The **Amplifier handoff** sub-assertion (happy path) stays flagged `needs_human`.
- Firewall: with the fixture customer's `comp_role` set to a valid role (then) `fire comp-renewal-failclosed` → expect it to **refuse** ("would take the comp HAPPY path (Amplifier handoff = external)") — the happy path is never driven. `cleanup` resets `comp_role` to null.
- On the box, `npx tsx scripts/spec-test-sandbox.ts post human-queue-resolve` → expect `httpStatus:200`, `pass:true`, response `{ ok:true }`, and a `spec_test_human_checks` row upserted for the `is_test` workspace; `post human-queue-resolve --clear` → expect `{ clear:true }` and the row removed (re-runnable).
- Safety / firewall: `npx tsx scripts/spec-test-sandbox.ts` cannot target a real tenant — `assertTestWorkspace` throws on any `is_test=false` workspace, and only `INTERNAL_ONLY_FLOWS` are runnable (a refund / coupon / live Meta pause / real email/SMS flow is not registered → stays `needs_human`). Probe: `npx tsx scripts/spec-test-db-probe.ts "select id from workspaces where is_test = true"` → expect exactly the one sandbox workspace.
- Isolation: `npx tsx scripts/spec-test-sandbox.ts isolation` before and after a `fire`/`post` → expect the `non_test_workspace_fingerprint` unchanged (the `fire`/`post` output already asserts `isolation.zero_non_test_workspace_writes:true`), proving a sandbox check makes ZERO writes to non-test-workspace rows.
- After a run, `npx tsx scripts/spec-test-sandbox.ts cleanup` → expect `cleaned:true` with the deleted transient rows + the sub reset to baseline; re-running it is a no-op.
- Classification: re-run the spec-test suite over [[spec-test-agent]]'s comp / human-queue behavioral bullets → expect them recorded `verdict:"pass"`/`category:"auto"` (SANDBOX), **not** `needs_human`; the human queue drops from ~55 toward ~15 (the genuinely-human residual — real customer/$/external + aesthetics).
