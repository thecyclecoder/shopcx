## The accumulated coaching to bake in (now archived as rolled-into-mandates)
- **When Pia ends the plan pass without a grounded, blocker-wired tree — either bailing to "no proposals" or emitting a flat decomposition with empty `blocked_by`, sometimes on an unverified capability premise.:** At the START of every plan job, before you emit terminal JSON: (1) read the materialized goal + every brain page you'd cite and VERIFY each capability premise you decompose against (don't assume a tool can't/can do something — confirm it); (2) for EACH proposed spec, explicitly enumerate its build-order prerequisites and set `blocked_by` to those sibling/existing slugs — never leave the whole tree at `blocked_by:[]`; if specs really are independent, say so in each spec's `intent`. Never conclude with `status:"completed"`/"no proposals" or a flat blocker-less tree — if the goal genuinely needs no new specs, or you're blocked, return `status:"needs_input"` with your documented reasoning so the null/parallel outcome is a deliberate, defensible choice rather than a silent omission. — Jobs mqqwxdie+mqqvwglo hit the null terminal exit at scripts/builder-worker.ts:6335 ("no proposals", no documented reasoning), while mqqvp52y+mqpf7t9d were superseded for flat/empty-`blocked_by` trees caught by isFlatBlockerlessTree at scripts/builder-worker.ts:6196-6198 — forcing verified premises + explicit `blocked_by` edges (or an escalation) up front is exactly what makes the pass survive both gates and satisfy the sound-decomposition/correct-blocked_by rubric.
- **When Pia repeatedly commits to a decomposition (or produces no decomposition) without first verifying upstream capabilities, existing blockers, and directives — resulting in null outputs, false premises, and missing blocked_by wiring that forces full replanning.:** When you see a new job, before writing any spec: (1) confirm what upstream tools/data actually provide, (2) identify all tasks that must complete first and wire them into blocked_by, and (3) verify any standing directives (e.g. store-everything) apply — only then emit specs; if genuine work is impossible, produce a documented rationale artifact instead of silent null output. — Every low-grade trace back to committing structure before validating reality: false assumptions about tool capabilities, empty blocked_by fields despite clear dependencies, and null outputs with no explanation. Forcing a verification pass before decomposition eliminates the false-premise supersession cycle and ensures blocked_by relationships are correct from the start, satisfying all three rubric criteria in the first attempt.
