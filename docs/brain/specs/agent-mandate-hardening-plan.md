# Harden the Pia agent — roll persistent coaching into its mandate

**Owner:** [[../functions/platform]] · **Parent:** [[platform-director-agent]] — graduate persistent coaching into a durable fix (director grades agents: low score → fix spec, never a CEO escalation)
**Found in use 2026-06-29:** the **Pia** agent (`plan`) sits at **5.4/10** after **2** coaching attempts that didn't stick. Rather than escalate to the CEO, roll the coaching into a permanent fix so the agent improves at the mandate level.

## The accumulated coaching to bake in (now archived as rolled-into-mandates)
- **When Pia repeatedly commits a decomposition (or produces nothing) before verifying tool/data capabilities and mapping upstream dependencies, resulting in plans that are superseded, missing blocked_by wires, or empty.:** When you are about to write any spec, first (1) confirm every assumed capability is real (e.g., does AdLibrary actually expose destination URLs?), (2) list every upstream output this task needs and encode each as a blocked_by entry, and (3) only then commit the decomposition — never emit specs with empty blocked_by when upstream work clearly must finish first. — All three low-scored runs share a single root cause: the decomposition was committed before the worker validated its assumptions and dependency graph, forcing the safety net to re-plan or discard the work entirely. Auditing capabilities and wiring blocked_by relationships before writing specs eliminates the false-premise and missing-dependency failure modes that caused every superseded or empty plan.
- **When Pia repeatedly commits a decomposition (or produces none) before verifying domain assumptions and wiring blocked_by relationships, forcing supersession or a near-zero output.:** When you are about to write any spec, first explicitly answer three gate questions: (1) What assumptions am I making about upstream capabilities or data availability — and have I verified them? (2) Which specs depend on outputs from other specs, and are those dependencies encoded in blocked_by? (3) Does every spec have at least one owner task, or is it an orphan? Only after all three are answered affirmatively should you commit the decomposition. — Every low grade traces to skipping one or more of these gates — unverified assumptions caused false-premise supersessions, empty blocked_by fields violated the core rubric, and orphaned or missing specs produced near-zero value. Enforcing the three-gate check before committing any spec eliminates the root cause of all three failure modes rather than patching symptoms.

## Phase 1 — bake the coaching into the agent
- Make the above coaching PERMANENT behavior of the `plan` agent — fold it into its prompt/mandate/code (its run-job + prompt in `scripts/builder-worker.ts` and the relevant `src/lib/agents/*`), not ephemeral appended `agent_instructions`. Once baked, the agent should follow it by default.
- Verify the agent's grade rollup recovers (≥ 7/10) over the next window of graded actions; if a coaching class still recurs, the bake-in missed it.
