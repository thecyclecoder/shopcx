# Harden the Ticket Improve agent — roll persistent coaching into its mandate

**Owner:** [[../functions/platform]] · **Parent:** [[platform-director-agent]] — graduate persistent coaching into a durable fix (director grades agents: low score → fix spec, never a CEO escalation)
**Found in use 2026-06-25:** the **Ticket Improve** agent (`ticket-improve`) sits at **5.8/10** after **2** coaching attempts that didn't stick. Rather than escalate to the CEO, roll the coaching into a permanent fix so the agent improves at the mandate level.

## The accumulated coaching to bake in (now archived as rolled-into-mandates)
- **When Tilly consistently finishes ticket-improve jobs with a 'completed' status but leaves no visible before/after diff, no summary of categorization or tag changes, and no confirmation that customer voice was preserved — making it impossible for graders to verify rubric compliance.:** When you finish processing a ticket, before closing the job emit a structured summary that includes: (1) the original text snippet vs. the revised text snippet, (2) any category or tag changes made (old → new), and (3) an explicit statement that customer voice and meaning were preserved (or flagging where they were not) — write this to the job log or output field so it is visible in the log tail. — Graders can only score what they can see; every low grade notes the same root problem — no evidence of the actual changes made. Surfacing a concise before/after diff and change rationale in the log tail directly satisfies the rubric's verifiability requirement and removes the information gap that is capping every job at 5-6/10 regardless of whether the underlying work was correct.
- **When Every low-graded run completed without errors but left no visible record of what changed — no before/after diff, no summary of tag/category decisions, no confirmation of customer-voice preservation — making rubric compliance unverifiable.:** When you finish modifying a ticket, emit a structured change summary before closing the job: list (1) each field changed with old→new values, (2) tags added/removed, (3) categorization decision with rationale, and (4) a one-line confirmation that the customer's original wording was preserved or quote the key phrase kept intact. — Graders can only score what they can observe; a clean 'completed' status with no diff is indistinguishable from a no-op and caps scores around 6 regardless of actual work quality. An explicit audit trail directly proves the rubric criteria — genuine improvement, correct categorization, preserved customer voice — and also forces the worker to self-check those criteria before closing, catching drift before it is graded.

## Phase 1 — bake the coaching into the agent
- Make the above coaching PERMANENT behavior of the `ticket-improve` agent — fold it into its prompt/mandate/code (its run-job + prompt in `scripts/builder-worker.ts` and the relevant `src/lib/agents/*`), not ephemeral appended `agent_instructions`. Once baked, the agent should follow it by default.
- Verify the agent's grade rollup recovers (≥ 7/10) over the next window of graded actions; if a coaching class still recurs, the bake-in missed it.
