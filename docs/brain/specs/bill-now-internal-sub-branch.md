# bill_now (+ change_next_date today-fallback) — route internal subs to the renewal-attempt Inngest event

**Owner:** [[../functions/cs]] · **Parent:** CS mandate "Ticket-derived product fixes" · **Derived-from-ticket:** `dd67f3c7-e520-4b4a-bc67-f02ac758aeb2`

When a customer like Angel asks to ship today and the sub is internal-source (Braintree-billed, shopify_contract_id starts with 'internal-'), the executor must fire `internal-subscription/renewal-attempt` instead of calling Appstle's attemptBilling. Today the executor's bill_now handler has no internal branch, so the Appstle helper synthesizes a double-prefixed billing-attempt id ('internal-internal-…') and Appstle 400s — what broke Angel's two-bag rush order across two retry turns. The recent f2e2f0aa short-circuit in appstleAttemptBilling silenced the 400 for the dunning cron path but now makes the executor silently claim success without billing. Adding the same is_internal branch that already exists for update_line_item_price (action-executor.ts:855-862) — and that the portal's order-now handler already uses (src/lib/portal/handlers/order-now.ts:22-30) — closes the executor gap. The appstle short-circuit stays in place as defense-in-depth for the dunning cron.

## Problem (from escalated ticket `dd67f3c7-e520-4b4a-bc67-f02ac758aeb2`)
src/lib/action-executor.ts directActionHandlers.bill_now (lines 494-503) calls appstleGetUpcomingOrders + appstleAttemptBilling unconditionally. For internal subs, appstleGetUpcomingOrders (src/lib/appstle.ts:308-321) returns a synthetic order id 'internal-${contractId}' which becomes 'internal-internal-<hash>' when contractId is itself 'internal-<hash>'. Before f2e2f0aa, that hit Appstle and 400'd (Angel's case). After f2e2f0aa, appstleAttemptBilling silently returns success=true without billing, so the AI will falsely confirm shipment. The change_next_date handler (lines 456-485) has the same bug in its 'date ≤ today → bill_now' fallback path. Internal-source subs have a working immediate-bill path used by the portal's order-now handler (src/lib/portal/handlers/order-now.ts:22-30) — it fires `internal-subscription/renewal-attempt` via inngest.send (guarded on resolved.status === 'active'), which the existing internalSubscriptionRenewalAttempt function (src/lib/inngest/internal-subscription-renewals.ts:95) drives through charge → order → Avalara → Amplifier. The executor never mirrored that path.

**Likely target:** `src/lib/action-executor.ts — in directActionHandlers.bill_now (lines 494-503) and in the change_next_date today-or-past fallback (lines 469-477), add an is_internal branch (use isInternalSubscription from src/lib/internal-subscription.ts, mirroring the pattern at action-executor.ts:855-862). When internal: look up subscription { id, status } by shopify_contract_id; if status !== 'active', return { success: false, error: 'Subscription is not active' } (mirror order-now.ts:23-25); otherwise fire `internal-subscription/renewal-attempt` with { subscription_id, workspace_id } via inngest.send (mirror order-now.ts:22-30) and return { success: true, summary: 'Triggered immediate internal-sub renewal' }. Leave the appstleAttemptBilling 'internal-' short-circuit at src/lib/appstle.ts:354-357 IN PLACE — it's defense-in-depth for the dunningPaydayRetryCron path (f2e2f0aa). Update docs/brain/libraries/action-executor.md to note bill_now now routes by sub source.`
