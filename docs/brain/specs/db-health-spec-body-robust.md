# DB Health — never author an empty fix spec (re-derive on empty body) ✅

**Owner:** [[../functions/platform]] · **Parent:** hardens [[db-health-agent]] spec authoring. · **Found in use 2026-06-23:** `db-index-orders`'s spec committed **empty (0 bytes)** → the build had no diagnostic and stalled `needs_input`. Root cause: the agent authored the body fine (`enqueueDbHealthProposal` stores `spec_body: buildFixSpecMarkdown(finding)` on the job's `instructions` JSON; `materializeDbHealthSpec` writes `instr.spec_body` at owner-Build), **but a manual approval overwrote the `instructions` field with a plain string**, so `JSON.parse(instructions)` yielded `{}` → `instr.spec_body` empty → `putFileMain` wrote an empty file. Whatever the cause (clobbered instructions, a genuinely-empty `buildFixSpecMarkdown`, a JSON-parse miss), the agent must **never commit an empty spec**.

## Fix
- **`materializeDbHealthSpec` refuses to write an empty/whitespace body.** If `instr.spec_body` is empty: (a) **re-derive** — re-`buildFixSpecMarkdown` from the finding payload if it's still on the job (carry the structured `finding`, not just the rendered `spec_body`, so it can re-render), or re-query `pg_stat_statements` + re-run `EXPLAIN` for the signature on the box; (b) if it genuinely can't reconstruct → set `needs_input` with a clear reason, **never `putFileMain` an empty file**.
- **Make the body un-clobberable.** Persist the diagnostic on a structured field that the approval path doesn't overwrite — store the `finding` (and/or `spec_body`) in the job's `pending_actions` `db_health_build` action (which approve mutates by *status only*), not in the free-text `instructions` an approval might replace. The owner-Build resume reads from there.
- **Defensive guard in the build, too** — `build-spec` (and the auto-merge gate) should treat a 0-byte / no-`## Phase` spec as a build failure, not a silent merge (belt-and-suspenders against any empty spec, db-health or otherwise).

## Verification
- On a fresh `db_health` proposal in `agent_jobs` (run a DB Health pass, or inspect a newly-enqueued row), the `pending_actions[0]` (`type:'db_health_build'`) carries both `spec_body` (non-empty markdown) and `finding` (the structured object) → expect the build-critical diagnostic to live on the action, and `instructions` to be panel metadata only (no `spec_body` key).
- On a proposal whose `instructions` were manually overwritten with a plain string (simulate the `db-index-orders` clobber) but whose action still carries `finding`, tap **Build** on `/dashboard` (or `POST /api/developer/control-tower/db-health` `{jobId,action:'build'}`) → expect `runDbHealthJob` → `resolveDbHealthSpecBody` to re-render from `finding` and commit a NON-empty `docs/brain/specs/{spec_slug}.md` to `main`; the job completes and queues the `build`. Never a 0-byte file.
- On a proposal with NO `spec_body` and NO `finding` anywhere (genuinely unreconstructable), tap Build → expect the job to flip to `needs_input` with `error` `db_health: … — refused to commit an empty spec` and a `questions` entry; no empty commit lands.
- On a `kind:'build'` job whose `docs/brain/specs/{slug}.md` is 0-byte or has no `## Phase` section, when the box claims it → expect the build to flip to `failed` with error `spec docs/brain/specs/{slug}.md is empty/0-byte/missing … — refusing to build an empty spec`, and NO PR opened (no silent empty merge). Resumes are exempt.
- Negative: on a normal `db_health` proposal with a populated `spec_body` on the action, tap Build → expect it to author + queue the build exactly as before (no behavior change).

## Phase 1 — empty-body guard + re-derive + un-clobberable storage ✅
**Shipped.** `enqueueDbHealthProposal` (`src/lib/control-tower/db-health.ts`) now carries the build-critical `spec_body` **and** the structured `finding` on the `db_health_build` `pending_actions` action (approve mutates it by status only) instead of the clobberable free-text `instructions` (which keeps panel metadata only). The owner-Build resume `runDbHealthJob` → new `resolveDbHealthSpecBody` (`scripts/builder-worker.ts`) reads the body from the action first, falls back to the older `instructions.spec_body`, and re-renders `buildFixSpecMarkdown(finding)` if it's empty; a genuinely unreconstructable proposal goes `needs_input` with a clear reason. `materializeDbHealthSpec` refuses to `putFileMain` a 0-byte/phaseless body, and the generic build path fails loudly (status `failed`) on a 0-byte/no-`## Phase` spec rather than merging an empty PR — mirrored in the `build-spec` skill step 1.
