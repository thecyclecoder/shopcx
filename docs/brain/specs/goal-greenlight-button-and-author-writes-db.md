# Greenlight button (one-click DB flag) + goal authoring writes the row

**Owner:** [[../functions/platform]] · **Parent:** [[../goals/db-driven-specs]] M5 — Goals + milestones are data too (the top two tiers)
**Blocked-by:** [[goals-milestones-tables-and-backfill]]

Solve the missing-surface problem the [[../goals/db-driven-specs]] Outcome calls out by name: "the CEO literally had no surface to approve the db-driven-specs goal; it had to be git-committed by hand". Make GREENLIGHT a ONE-CLICK DB FLAG with a UI BUTTON on the board's goal card: a single `UPDATE public.goals SET status='greenlit' WHERE id=$1` (via [[goals-milestones-tables-and-backfill]] `setGoalStatus`). And cut every goal-creation/edit SURFACE off the markdown-commit path and onto `INSERT/UPDATE public.goals + public.goal_milestones`: the [[director-proposed-goals]] paths (the box worker's `runProposedGoalJob`, the [[director-judgment-lanes-fold-author-dismiss|director coach]] `goal` action), [[ada-slack-routed-approvals|Ada]] when she emits a goal card in chat, the [[../skills/plan-goal]] planner when it proposes a SUBGOAL or seeds milestones, and the goal-decomposition engine. Editing a goal = `UPDATE public.goals` (or `UPDATE public.goal_milestones` for a milestone edit) — no git commit, no Vercel deploy. [[spec-review-agent|Vale]]'s planned/deferred decisions read `goals.status` from the DB; `spec_card_state.flags.intended_status` (already moved to `specs.intended_status` in [[spec-authoring-writes-db-and-worker-materialize]]) keeps the same shape against the goal/milestone the spec attaches to. After this spec, the system runs goal-side DUAL-WRITE (`.md` + DB) — every write goes to the DB, but a mirror commit still lands on `main` so [[../libraries/brain-roadmap]] `parseGoal` keeps serving the still-markdown-first readers; the goal-reader cutover is owned by [[goal-readers-from-db-retire-parsegoal]]. Closes the gap from [[../goals/db-driven-specs]] M5: today greenlighting a goal means a markdown `**Status:**` flip via [[../libraries/brain-roadmap]] `setGoalStatusLine` + a git commit + a Vercel deploy ([[director-proposed-goals]] Phase 1 verification), and [[director-proposed-goals]] [[../libraries/goal-proposals]] commits `docs/brain/goals/{slug}.md` rather than writing a row.

## Phase 1 — the Greenlight button on the goal card (one-click DB flag) ✅
- The goal card on the board (`src/app/dashboard/roadmap/goals/page.tsx` list + `src/app/dashboard/roadmap/goals/[slug]/page.tsx` detail — both today read the markdown via `getGoals`/`getGoal` until [[goal-readers-from-db-retire-parsegoal]] flips them) gets a green **Greenlight** button on every `goals.status='proposed'` card. Click = `POST /api/roadmap/goal/greenlight {goalId}` → server route calls [[goals-milestones-tables-and-backfill]] `setGoalStatus(goalId, 'greenlit', actorUserId)` → `UPDATE public.goals SET status='greenlit', updated_at=now() WHERE id=$1 AND status='proposed'` (idempotent — refuses to re-flip a non-proposed goal). Records the actor in [[../tables/director_activity]] as `action_kind='greenlit_goal'`.
- The button is ONE click — no confirmation modal, no "are you sure" — same shape as the existing [[approval-routing-engine]] Approve action. Greenlight is reversible (a still-no-progress greenlit goal can be flipped back to `proposed` with a `POST /api/roadmap/goal/ungreenlight`, also one click).
- The companion **Decline** action on a `proposed` goal: `UPDATE public.goals SET status='declined'` (added as a `'declined'` enum value to the status check, or absorbed into the existing `'folded'` value with `body` carrying the decline reason — author calls it). [[director-proposed-goals]] Phase 1 verification's "Decline → delete `.md` from main" path retires here: the row stays (audit), the markdown mirror commits the new status.
- AuthZ: only the CEO role can greenlight (per the [[director-proposed-goals]] North star — "the CEO owns objectives, directors own progress within approved ones"). The route checks the actor's role and refuses non-CEO greenlights with a 403, even for a director on its own function. Mirrors the [[approval-routing-engine]] CEO-only routing today.
- Verification — Phase 1: a `proposed` goal renders a Greenlight button; clicking it flips `goals.status='greenlit'` in a single DB write within 100ms (no Vercel deploy, no GitHub call), a [[../tables/director_activity]] row records the actor, and the existing [[director-proposed-goals]] Phase 2 decomposition gate (`POST /api/roadmap/plan` reads `goals.status`) now lets the planner run on the goal. Greenlighting the same goal a second time is a no-op (the WHERE clause guards). Demonstrate the missing-surface fix end-to-end: take a freshly-authored `proposed` goal, click Greenlight, observe the goal active without any commit having landed.

## Safety / invariants
- **Greenlight is CEO-only.** The Phase 1 route checks the actor's role server-side; no director can greenlight a goal even for its own function. Mirrors the [[director-proposed-goals]] North star and the [[approval-routing-engine]] CEO-only routing rail.
- **Greenlight is reversible until progress lands.** A `greenlit` goal with no specs shipped under it can be flipped back to `proposed` (one click, same shape as Greenlight) — guards against a misclick spending compute. Once any child spec is `shipped`, the rollup pins the goal forward (the Phase 1 trigger from [[goals-milestones-tables-and-backfill]] is the rail; the un-greenlight route refuses if `goal_milestones.status` is anything other than `planned`).
- **The author surfaces NEVER bypass `proposed`.** A new goal lands as `proposed` even if authored by the CEO directly — every greenlight is an explicit CEO action recorded in [[../tables/director_activity]]. Mirrors the [[director-proposed-goals]] North star ("propose, don't self-activate").
- **`parent_goal_id` cycle protection holds.** The [[goals-milestones-tables-and-backfill]] CHECK trigger enforces acyclicity; any `UPDATE goals SET parent_goal_id=…` from an author surface (re-parenting a goal under another) trips the trigger if it would close a loop.
- **Workspace-scoped + service-role writes.** Every write goes through `createAdminClient()`; RLS keeps direct-from-browser writes off. Mirrors the rest of the goal.
- **The mirror commit is transitional.** This spec leaves a `kind='mirror-goal-md'` worker lane in place; [[goal-readers-from-db-retire-parsegoal]] deletes it. Until then, dual-write is mandatory — a missing mirror commit silently regresses the markdown-first readers.
