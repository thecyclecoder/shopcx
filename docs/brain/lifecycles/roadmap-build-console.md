# Lifecycle: Roadmap Build Console (self-driving roadmap)

Describe a feature в†’ spec в†’ autonomous build on the **Max subscription** в†’ answer questions / approve prod actions в†’ merge вЂ” all from the dashboard (phone-friendly), no laptop or terminal. This is the end-to-end home for the shipped [[../specs/roadmap-build-console]] + [[../specs/build-approval-gates]] specs.

## Two layers (don't conflate)

- **Brain markdown (`main`)** вЂ” canonical, *static* spec definitions + final status (the `вЏі planned В· рџљ§ in progress В· вњ… shipped В· вќЊ cut` phase emojis). Changes only when a PR merges. The board parses it.
- **`agent_jobs` (DB companion)** вЂ” *live, actionable* state: build status, `needs_input` questions, `needs_approval` pending actions. The box worker polls it; the dashboard renders it as buttons. Pending actions live here (not on the unmerged branch) вЂ” that's why a DB companion is required.

## Two runtimes

- **Vercel app** вЂ” the dashboard surfaces + APIs. Never touches the box. A "Build" tap just **inserts an `agent_jobs` row** in Supabase.
- **The box** ([[../recipes/build-box-setup]]) вЂ” a Hetzner CCX33, **Tailscale-only inbound**, runs `systemd: shopcx-builder` as the **non-root `builder`** user. It **polls Supabase outbound** (no inbound; the firewall stays shut вЂ” the box dials out, the app never dials in), claims a job, and runs the build as a headless `claude -p` on **Max**.

## End-to-end trace

1. **Author.** `/dashboard/roadmap` в†’ **вњЁ New feature** opens the authoring chat (`POST /api/roadmap/chat`), now **box-hosted on Max** (box-spec-chat вЂ” no longer the Anthropic API). Each turn enqueues a `kind='spec-chat'` [[../tables/agent_jobs]] row that resumes a long-running `claude -p` session on the build box, so the model has the **whole working tree** (`Read`/`Grep`/`Glob` over `docs/brain/` + `src/`) and **`WebSearch`** every turn вЂ” far more grounded than the old brain-index + two-GitHub-tools call, and **$0 marginal**. The route appends the user turn + sets `turn_status='thinking'`; the box appends the reply; `AuthoringChat.tsx` polls `chat-session?id=` and shows "thinking on the boxвЂ¦ (a minute)". Talk it through в†’ **Save spec** (the box resumes the session, emits the full spec markdown, the worker commits `docs/brain/specs/{slug}.md` to `main` via the GitHub Contents API) or **Save & build** (also queues a `build` job). **Refine with Opus** on a spec's detail page passes `slug` в†’ the box Reads the current spec from the working tree as grounding. (Hand-written specs work too.)
2. **Dispatch.** **Build** on a card в†’ `POST /api/roadmap/build` в†’ inserts `agent_jobs` (`queued`). One active build per spec **for a plain Build tap** (no instructions): a second tap coalesces into the live job (`alreadyActive`). Variants: per-phase **build** ([[../dashboard/roadmap]] PhaseList) and **Report issue** both queue a build scoped via `instructions` (no spec edit, spec stays вњ…) вЂ” and because they carry **new, distinct instructions**, when a build is already live they do **not** coalesce-and-drop; they enqueue a **distinct follow-up `queued` row** (`queuedBehindActive`) that runs after the active build (per-spec serialization). The UI/Slack say "queued to run next" вЂ” the issue text is never lost. See [[../specs/fix-report-issue-dropped]].
3. **Claim.** The worker calls `claim_agent_job()` (atomic, `FOR UPDATE SKIP LOCKED`) в†’ `building`. It runs `claude -p --dangerously-skip-permissions` (bypass, no prompts) via the **`build-spec` skill**, as `builder`, with prod-write secrets **stripped from the build env** and no `ANTHROPIC_API_KEY` (stays on Max).
4. **Outcomes** (the build emits one final-status JSON):
   - `completed` в†’ worker runs `npx tsc --noEmit` gate, commits, opens a `claude/*` PR (`ensurePr` retries `gh pr create` with backoff first вЂ” most failures are transient).
   - **PR-create failed but branch pushed** ([[../specs/build-recover-pr-create]]) в†’ after the retries are exhausted the worker flags `needs_attention` `error="branch pushed but PR creation failed"`. This is **recoverable** (the build is done + on origin): the card shows **Create PR** (primary) в†’ `POST /api/roadmap/build { jobId, recoverPr:true }` в†’ `createPrForJob` opens/adopts the PR for the pushed branch and flips the job в†’ `completed` вЂ” *not* a Rebuild that throws the build away. (Re-validates the #185 case.)
   - `needs_input` (product question) в†’ worker records `questions`, draft PR, pauses; the card shows an **answer form** в†’ `POST /api/roadmap/answer` в†’ `queued_resume` в†’ worker `claude --resume`s.
   - `needs_approval` (gated prod action: `apply_migration` / `run_prod_script` / `merge_pr`) в†’ worker records `pending_actions` + pauses; the card shows **Approve & apply** with the command preview в†’ `POST /api/roadmap/approve` в†’ worker (which holds prod creds) **executes the approved action**, then `--resume`s. The build itself never touches prod (credential-enforced).
   - `failed` / `needs_attention` в†’ surfaced with `error` + `log_tail`. A build that finishes with **no file changes and no PR** is flipped to `needs_attention` carrying the agent's `no_changes_reason` (the build-spec contract emits it when it makes no edits) вЂ” never a bare `completed` with no PR that masquerades as done (the card shows Rebuild as a retry). See [[../specs/fix-report-issue-dropped]] Phase 3.
5. **Review + merge.** The card's **Squash & merge** (or [[../dashboard/branches]]) merges the `claude/*` PR вЂ” owner-only, server-revalidated.
6. **Status reflects reality.** Spec emojis drive the board columns; `agent_jobs` drives the live per-card build chip + buttons.
7. **Verify в†’ fold в†’ archive.** **Shipped (вњ…)** = built + deployed (automated). The owner-only, human gate **Verified** comes after: once Dylan confirms a shipped feature works in prod, **Mark verified & archive** (`BuildButton`) queues a **fold-build** (`POST /api/roadmap/build` `{ verify: true }` в†’ canonical fold instructions in the build route, not a spec rebuild). The build folds the spec into its permanent brain homes, appends a one-line entry to [[../archive]], `git rm`s `specs/{slug}.md`, and opens a PR. Merge в†’ the spec leaves the board into the collapsed **Archived** section (reads `archive.md`). The Shipped column is relabeled **"Shipped вЂ” awaiting verification"** so it stays a short, real to-do list. Convention: `shipped в†’ verified в†’ fold + delete + archive-index` ([[../project-management]]). Nothing is lost вЂ” knowledge in the brain, pointer in the archive, spec `git show`-recoverable.
8. **Re-hydrate (New spec from brain).** Revisiting an archived (or any reference) feature doesn't reactivate the stale spec: **New spec from brain** seeds the authoring chat with the *current* brain page (`POST /api/roadmap/chat` `{ seedSlug }`) and Opus drafts a *fresh* spec to extend/fix it в†’ normal Build flow. The authoring chat in reverse.

## Safety model

Bypass is safe because of four things, not prompts: (1) **PR-gate** вЂ” code never hits prod until you merge; (2) **powerless build env** вЂ” secrets stripped, so a runaway command can't reach prod (credential-enforced); (3) **gated prod actions need a tap** вЂ” `apply_migration`/`run_prod_script`/`merge_pr` only run after owner approval, executed by the worker; (4) **non-root** builder user. The worker is the *only* component with prod creds.

## Billing

- **Authoring chat** в†’ Anthropic API (Opus `claude-opus-4-8`), cheap conversation tokens.
- **Builds** в†’ **Max subscription** (box `claude -p`, no API key). Verified: a real build ran bypass-as-builder on Max and opened a PR (2026-06-18).
