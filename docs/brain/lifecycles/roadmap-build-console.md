# Lifecycle: Roadmap Build Console (self-driving roadmap)

Describe a feature ‚Üí spec ‚Üí autonomous build on the **Max subscription** ‚Üí answer questions / approve prod actions ‚Üí merge ‚Äî all from the dashboard (phone-friendly), no laptop or terminal. This is the end-to-end home for the shipped [[../specs/roadmap-build-console]] + [[../specs/build-approval-gates]] specs.

## Two layers (don't conflate)

- **Brain markdown (`main`)** ‚Äî canonical, *static* spec definitions + final status (the `‚è≥ planned ¬∑ üöß in progress ¬∑ ‚úÖ shipped ¬∑ ‚ùå cut` phase emojis). Changes only when a PR merges. The board parses it.
- **`agent_jobs` (DB companion)** ‚Äî *live, actionable* state: build status, `needs_input` questions, `needs_approval` pending actions. The box worker polls it; the dashboard renders it as buttons. Pending actions live here (not on the unmerged branch) ‚Äî that's why a DB companion is required.

## Two runtimes

- **Vercel app** ‚Äî the dashboard surfaces + APIs. Never touches the box. A "Build" tap just **inserts an `agent_jobs` row** in Supabase.
- **The box** ([[../recipes/build-box-setup]]) ‚Äî a Hetzner CCX33, **Tailscale-only inbound**, runs `systemd: shopcx-builder` as the **non-root `builder`** user. It **polls Supabase outbound** (no inbound; the firewall stays shut ‚Äî the box dials out, the app never dials in), claims a job, and runs the build as a headless `claude -p` on **Max**.

## End-to-end trace

1. **Author.** `/dashboard/roadmap` ‚Üí **‚ú® New feature** opens the authoring chat (`POST /api/roadmap/chat`), now **box-hosted on Max** (box-spec-chat ‚Äî no longer the Anthropic API). Each turn enqueues a `kind='spec-chat'` [[../tables/agent_jobs]] row that resumes a long-running `claude -p` session on the build box, so the model has the **whole working tree** (`Read`/`Grep`/`Glob` over `docs/brain/` + `src/`) and **`WebSearch`** every turn ‚Äî far more grounded than the old brain-index + two-GitHub-tools call, and **$0 marginal**. The route appends the user turn + sets `turn_status='thinking'`; the box appends the reply; `AuthoringChat.tsx` polls `chat-session?id=` and shows "thinking on the box‚Ä¶ (a minute)". Talk it through ‚Üí **Save spec** (the box resumes the session, materializes the full spec markdown into the worktree as a scratch buffer, and the worker **authors it to the DB** ‚Äî `public.specs` + `public.spec_phases` via the author-spec SDK; **no `.md` is committed** ‚Äî specs live in the DB now, spec-pm-markdown-purge / retire-md-reads) or **Save & build** (also queues a `build` job). Finalize SUCCESS = the spec row (+ ‚â•1 phase) now exists in `public.specs` (read back via `getSpec`), and the slug is guarded to a real kebab slug (a new-feature chat is keyed by a UUID ‚Äî never let a UUID become the spec slug). **Refine with Opus** on a spec's detail page passes `slug` ‚Üí the worker materializes the existing row back to the worktree (`renderSpecRow`) so the box Reads it as grounding. (Hand-written specs work too.)
2. **Dispatch.** **Build** on a card ‚Üí `POST /api/roadmap/build` ‚Üí inserts `agent_jobs` (`queued`). One active build per spec **for a plain Build tap** (no instructions): a second tap coalesces into the live job (`alreadyActive`). Variants: per-phase **build** ([[../dashboard/roadmap]] PhaseList) and **Report issue** both queue a build scoped via `instructions` (no spec edit, spec stays ‚úÖ) ‚Äî and because they carry **new, distinct instructions**, when a build is already live they do **not** coalesce-and-drop; they enqueue a **distinct follow-up `queued` row** (`queuedBehindActive`) that runs after the active build (per-spec serialization). The UI/Slack say "queued to run next" ‚Äî the issue text is never lost. See [[../specs/fix-report-issue-dropped]].
3. **Claim.** The worker calls `claim_agent_job()` (atomic, `FOR UPDATE SKIP LOCKED`) ‚Üí `building`. It runs `claude -p --dangerously-skip-permissions` (bypass, no prompts) via the **`build-spec` skill**, as `builder`, with prod-write secrets **stripped from the build env** and no `ANTHROPIC_API_KEY` (stays on Max).
4. **Outcomes** (the build emits one final-status JSON):
   - `completed` ‚Üí worker runs `npx tsc --noEmit` gate, commits, opens a `claude/*` PR (`ensurePr` retries `gh pr create` with backoff first ‚Äî most failures are transient).
   - **PR-create failed but branch pushed** ([[../specs/build-recover-pr-create]]) ‚Üí after the retries are exhausted the worker flags `needs_attention` `error="branch pushed but PR creation failed"`. This is **recoverable** (the build is done + on origin): the card shows **Create PR** (primary) ‚Üí `POST /api/roadmap/build { jobId, recoverPr:true }` ‚Üí `createPrForJob` opens/adopts the PR for the pushed branch and flips the job ‚Üí `completed` ‚Äî *not* a Rebuild that throws the build away. (Re-validates the #185 case.)
   - `needs_input` (product question) ‚Üí worker records `questions`, draft PR, pauses; the card shows an **answer form** ‚Üí `POST /api/roadmap/answer` ‚Üí `queued_resume` ‚Üí worker `claude --resume`s.
   - `needs_approval` (gated prod action: `apply_migration` / `run_prod_script` / `merge_pr`) ‚Üí worker records `pending_actions` + pauses; the card shows **Approve & apply** with the command preview ‚Üí `POST /api/roadmap/approve` ‚Üí worker (which holds prod creds) **executes the approved action**, then `--resume`s. The build itself never touches prod (credential-enforced).
   - `failed` / `needs_attention` ‚Üí surfaced with `error` + `log_tail`. A build that finishes with **no file changes and no PR** is flipped to `needs_attention` carrying the agent's `no_changes_reason` (the build-spec contract emits it when it makes no edits) ‚Äî never a bare `completed` with no PR that masquerades as done (the card shows Rebuild as a retry). See [[../specs/fix-report-issue-dropped]] Phase 3.
5. **Review + merge.** The card's **Squash & merge** (or [[../dashboard/branches]]) merges the `claude/*` PR ‚Äî owner-only, server-revalidated.
6. **Status reflects reality.** Spec emojis drive the board columns; `agent_jobs` drives the live per-card build chip + buttons.
7. **Machine spec-test pass ‚Üí fold ‚Üí archive.** **Shipped (‚úÖ)** = built + deployed (automated). The fold comes after **automatically** when the spec's **MACHINE spec-test passes** (agent-verdict `approved`, no open regression ‚Äî Gate B `autoFoldVerifiedSpecs`, fold-on-spec-test-pass task #29): it queues the same **fold-build** (`enqueue_fold` ‚Üí canonical fold instructions in the build route, not a spec rebuild) that folds the spec into its permanent brain homes, appends a one-line entry to [[../archive]], flips the [[../tables/specs]] row to `status='folded'` (preserved) while `git rm`ing the content-only `specs/{slug}.md`, and opens a PR. **Human QA is advisory and never gates this**; the owner can still **Fold to brain now** (`BuildButton` ‚Üí `POST /api/roadmap/build` `{ verify: true }`) manually as an override. Merge ‚Üí the spec leaves the board into the collapsed **Archived** section (reads `archive.md`). The Shipped column is relabeled **"Shipped ‚Äî awaiting fold"** so it stays a short, real to-do list. Convention: `shipped ‚Üí folded + archive-index` ([[../project-management]]). Nothing is lost ‚Äî knowledge in the brain, pointer in the archive, the row preserved + the md `git show`-recoverable.
8. **Re-hydrate (New spec from brain).** Revisiting an archived (or any reference) feature doesn't reactivate the stale spec: **New spec from brain** seeds the authoring chat with the *current* brain page (`POST /api/roadmap/chat` `{ seedSlug }`) and Opus drafts a *fresh* spec to extend/fix it ‚Üí normal Build flow. The authoring chat in reverse.

## Safety model

Bypass is safe because of four things, not prompts: (1) **PR-gate** ‚Äî code never hits prod until you merge; (2) **powerless build env** ‚Äî secrets stripped, so a runaway command can't reach prod (credential-enforced); (3) **gated prod actions need a tap** ‚Äî `apply_migration`/`run_prod_script`/`merge_pr` only run after owner approval, executed by the worker; (4) **non-root** builder user. The worker is the *only* component with prod creds.

## Billing

- **Authoring chat** ‚Üí Anthropic API (Opus `claude-opus-4-8`), cheap conversation tokens.
- **Builds** ‚Üí **Max subscription** (box `claude -p`, no API key). Verified: a real build ran bypass-as-builder on Max and opened a PR (2026-06-18).
