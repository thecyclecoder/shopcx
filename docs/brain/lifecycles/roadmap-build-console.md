# Lifecycle: Roadmap Build Console (self-driving roadmap)

Describe a feature → spec → autonomous build on the **Max subscription** → answer questions / approve prod actions → merge — all from the dashboard (phone-friendly), no laptop or terminal. This is the end-to-end home for the shipped [[../specs/roadmap-build-console]] + [[../specs/build-approval-gates]] specs.

## Two layers (don't conflate)

- **Brain markdown (`main`)** — canonical, *static* spec definitions + final status (the `⏳ planned · 🚧 in progress · ✅ shipped · ❌ cut` phase emojis). Changes only when a PR merges. The board parses it.
- **`agent_jobs` (DB companion)** — *live, actionable* state: build status, `needs_input` questions, `needs_approval` pending actions. The box worker polls it; the dashboard renders it as buttons. Pending actions live here (not on the unmerged branch) — that's why a DB companion is required.

## Two runtimes

- **Vercel app** — the dashboard surfaces + APIs. Never touches the box. A "Build" tap just **inserts an `agent_jobs` row** in Supabase.
- **The box** ([[../recipes/build-box-setup]]) — a Hetzner CCX33, **Tailscale-only inbound**, runs `systemd: shopcx-builder` as the **non-root `builder`** user. It **polls Supabase outbound** (no inbound; the firewall stays shut — the box dials out, the app never dials in), claims a job, and runs the build as a headless `claude -p` on **Max**.

## End-to-end trace

1. **Author.** `/dashboard/roadmap` → **✨ New feature** opens the authoring chat (`POST /api/roadmap/chat`), now **box-hosted on Max** (box-spec-chat — no longer the Anthropic API). Each turn enqueues a `kind='spec-chat'` [[../tables/agent_jobs]] row that resumes a long-running `claude -p` session on the build box, so the model has the **whole working tree** (`Read`/`Grep`/`Glob` over `docs/brain/` + `src/`) and **`WebSearch`** every turn — far more grounded than the old brain-index + two-GitHub-tools call, and **$0 marginal**. The route appends the user turn + sets `turn_status='thinking'`; the box appends the reply; `AuthoringChat.tsx` polls `chat-session?id=` and shows "thinking on the box… (a minute)". Talk it through → **Save spec** (the box resumes the session, materializes the full spec markdown into the worktree as a scratch buffer, and the worker **authors it to the DB** — `public.specs` + `public.spec_phases` via the author-spec SDK; **no `.md` is committed** — specs live in the DB now, spec-pm-markdown-purge / retire-md-reads) or **Save & build** (also queues a `build` job). Finalize SUCCESS = the spec row (+ ≥1 phase) now exists in `public.specs` (read back via `getSpec`), and the slug is guarded to a real kebab slug (a new-feature chat is keyed by a UUID — never let a UUID become the spec slug). **Refine with Opus** on a spec's detail page passes `slug` → the worker materializes the existing row back to the worktree (`renderSpecRow`) so the box Reads it as grounding. (Hand-written specs work too.)
2. **Dispatch.** **Build** on a card → `POST /api/roadmap/build` → inserts `agent_jobs` (`queued`). One active build per spec **for a plain Build tap** (no instructions): a second tap coalesces into the live job (`alreadyActive`). Variants: per-phase **build** ([[../dashboard/roadmap]] PhaseList) and **Report issue** both queue a build scoped via `instructions` (no spec edit, spec stays ✅) — and because they carry **new, distinct instructions**, when a build is already live they do **not** coalesce-and-drop; they enqueue a **distinct follow-up `queued` row** (`queuedBehindActive`) that runs after the active build (per-spec serialization). The UI/Slack say "queued to run next" — the issue text is never lost. See [[../specs/fix-report-issue-dropped]]. **Reactive/agent enqueues** (blocker-cleared auto-queue, Ada's escort lanes, the review-pass consumer) go through the single **[[../libraries/agent-jobs]] `enqueueBuildIfDue`** chokepoint ([[../specs/bo-reactive-gated-build-enqueue]] Phase 1) — one cheap `getSpec` read + gate on the same `specReviewDone` / not-deferred / not-shipped / `auto_build` / blockers-cleared eligibility predicate Ada's `isBuildableSpec` applies, plus a one-in-flight guard. The **reactive companion** to the `*/5` platform-director cron is [[../inngest/build-on-eligible]] ([[../specs/bo-reactive-gated-build-enqueue]] Phase 2) — a `build/spec-build-eligible` Inngest event fires from [[../libraries/spec-card-state]] `markSpecCardValePassed` and `applyAdaDisposition('planned')`, the consumer runs `enqueueBuildIfDue`, so a build row appears within seconds of the transition (and no row appears while the spec is still in_review awaiting Vale). The `*/5` cron becomes the gated backstop for dropped events / cold workspaces. Closes the CEO-visible bug where `autoQueueUnblockedBy` inserted a `queued` build row for an unblocked-but-un-Vale-passed dependent (the claim-gate then held it indefinitely — the "premature row on the board" symptom). The claim-time backstops (`claimHeldForUnreviewedSpec` + `evaluateClaimTimeBuildGate`) stay in place as the last line of defense; deliberate override paths (pre-merge-fix, priority-directive, needs-attention-authored child, chain continuation) carry an inline `// intentional override:` comment.
3. **Claim.** The worker calls `claim_agent_job()` (atomic, `FOR UPDATE SKIP LOCKED`) → `building`. It runs `claude -p --dangerously-skip-permissions` (bypass, no prompts) via the **`build-spec` skill**, as `builder`, with prod-write secrets **stripped from the build env** and no `ANTHROPIC_API_KEY` (stays on Max).
4. **Outcomes** (the build emits one final-status JSON):
   - `completed` → worker runs `npx tsc --noEmit` + the `next build` gate, commits + pushes to `claude/build-{slug}`. **The PR is DEFERRED until accumulation completes** (`defer-pr-until-accumulation-complete`): for a multi-phase spec still accumulating (a phase still un-built), the build pushes the branch but opens **NO PR** — it completes on the branch and chains the next phase. `ensurePr` (which retries `gh pr create` with backoff — most failures are transient) runs ONLY in the build that completes accumulation (the last phase) or for a one-shot / single-phase spec (trivially complete), so the PR + the squash-merge action surface together at `in_testing`, never mid-accumulation. The branch's own Vercel preview (keyed on the `^claude/build-` git ref, not a PR) drives the pre-merge spec-test + security in the meantime. See [[spec-goal-branch-pm-flow]] § 2.
   - **PR-create failed but branch pushed** ([[../specs/build-recover-pr-create]]) → after the retries are exhausted the worker flags `needs_attention` `error="branch pushed but PR creation failed"`. This is **recoverable** (the build is done + on origin): the card shows **Create PR** (primary) → `POST /api/roadmap/build { jobId, recoverPr:true }` → `createPrForJob` opens/adopts the PR for the pushed branch and flips the job → `completed` — *not* a Rebuild that throws the build away. (Re-validates the #185 case.)
   - `needs_input` (product question) → worker records `questions`, draft PR, pauses; the card shows an **answer form** → `POST /api/roadmap/answer` → `queued_resume` → worker `claude --resume`s.
   - `needs_approval` (gated prod action: `apply_migration` / `run_prod_script` / `merge_pr`) → worker records `pending_actions` + pauses; the card shows **Approve & apply** with the command preview → `POST /api/roadmap/approve` → worker (which holds prod creds) **executes the approved action**, then `--resume`s. The build itself never touches prod (credential-enforced).
   - `failed` / `needs_attention` → surfaced with `error` + `log_tail`. A build that finishes with **no file changes and no PR** is flipped to `needs_attention` carrying the agent's `no_changes_reason` (the build-spec contract emits it when it makes no edits) — never a bare `completed` with no PR that masquerades as done (the card shows Rebuild as a retry). See [[../specs/fix-report-issue-dropped]] Phase 3.
5. **Review + merge.** The card's **Squash & merge** (or [[../dashboard/branches]]) merges the `claude/*` PR — owner-only, server-revalidated.
6. **Status reflects reality.** Spec emojis drive the board columns; `agent_jobs` drives the live per-card build chip + buttons.
7. **Machine spec-test pass → fold → archive.** **Shipped (✅)** = built + deployed (automated). The fold comes after **automatically** when the spec's **MACHINE spec-test passes** (agent-verdict `approved`, no open regression — Gate B `autoFoldVerifiedSpecs`, fold-on-spec-test-pass task #29): it queues the same **fold-build** (`enqueue_fold` → canonical fold instructions in the build route, not a spec rebuild) that folds the spec into its permanent brain homes, appends a one-line entry to [[../archive]], flips the [[../tables/specs]] row to `status='folded'` (preserved) while `git rm`ing the content-only `specs/{slug}.md`, and opens a PR. **Human QA is advisory and never gates this**; the owner can still **Fold to brain now** (`BuildButton` → `POST /api/roadmap/build` `{ verify: true }`) manually as an override. Merge → the spec leaves the board into the collapsed **Archived** section (reads `archive.md`). The Shipped column is relabeled **"Shipped — awaiting fold"** so it stays a short, real to-do list. Convention: `shipped → folded + archive-index` ([[../project-management]]). Nothing is lost — knowledge in the brain, pointer in the archive, the row preserved + the md `git show`-recoverable.
8. **Re-hydrate (New spec from brain).** Revisiting an archived (or any reference) feature doesn't reactivate the stale spec: **New spec from brain** seeds the authoring chat with the *current* brain page (`POST /api/roadmap/chat` `{ seedSlug }`) and Opus drafts a *fresh* spec to extend/fix it → normal Build flow. The authoring chat in reverse.

## Safety model

Bypass is safe because of four things, not prompts: (1) **PR-gate** — code never hits prod until you merge; (2) **powerless build env** — secrets stripped, so a runaway command can't reach prod (credential-enforced); (3) **gated prod actions need a tap** — `apply_migration`/`run_prod_script`/`merge_pr` only run after owner approval, executed by the worker; (4) **non-root** builder user. The worker is the *only* component with prod creds.

## Billing

- **Authoring chat** → Anthropic API (Opus `claude-opus-4-8`), cheap conversation tokens.
- **Builds** → **Max subscription** (box `claude -p`, no API key). Verified: a real build ran bypass-as-builder on Max and opened a PR (2026-06-18).

The goal→milestone link is now bound at author time: [[../libraries/build-goal-materializer]] `renderGoalRow` emits each milestone's `M{position}` handle + `_milestone_id: {uuid}_`, and the worker resolves the planner's `milestone` handle → real `goal_milestones.id` → `specs.milestone_id`. Director-PROPOSED goals are DB-only too — `proposeGoal`'s row write is now AUTHORITATIVE (aborts on failure; no markdown fallback), and `runProposedGoalJob` no longer commits/deletes any `docs/brain/goals/*.md` (the dead `putFileMain` helper was removed). The `plan-goal` SKILL + the PROPOSE prompt no longer reference purged `docs/brain/goals/` or `docs/brain/specs/` reads — existing-spec mapping grounds on the materialized goal render + `public.specs`. Box-impacting (`scripts/builder-worker.ts`) — takes effect on the next foreman/worker self-update restart. See [[../libraries/author-spec]] · [[../libraries/goal-proposals]] · [[../libraries/build-goal-materializer]].

**Shipped (2026-06-27, claim-time build gate):** the box now gates the FIRST claim of every `kind='build'` job — it no longer builds whatever sits `queued`. Two CEO-reported escapes (a malformed/never-Vale-reviewed spec built; milestone fan-out built out of sequence because `reconcileMilestoneSequence` only RE-orders reactively) are closed PREVENTIVELY. `evaluateClaimTimeBuildGate` runs at the build dispatch entry in `scripts/builder-worker.ts` BEFORE any worktree/account/`claude` work, and refuses the claim unless the spec is **(1) AUTHORED** — a live `public.specs` row (missing → PARK `needs_attention(spec_row_missing)`); **(2) Vale-SPEC-REVIEW-PASSED** — `card.valePass===true` (read off `specs.vale_pass`, the EXISTING [[../specs/spec-review-agent]] verdict — `markSpecCardValePassed`); and **(3) every `blocked_by` SHIPPED** — no `card.blockedBy[].cleared===false`. Every check reads **DERIVED** status via the brain-roadmap rollup (`getSpec` → `card.status`/`card.phases`/`card.valePass`/`card.blockedBy`), NEVER the stored `specs.status` column. On a hold the build is **un-claimed** (`status→queued`, `claimed_at→null`) so it stays claimable and the director escort / `reconcileMilestoneSequence` re-release it as its blockers ship; when a spec isn't Vale-passed yet the gate also **proactively enqueues** a `spec-review` pass (`enqueueSpecReviewIfDue`, deduped) so Vale reviews the authored spec promptly rather than waiting on the next `spec-review-cron` tick — Vale's pre-build review is wired, not invented. A resume is exempt (its branch/PR + WIP already exist). This is the front-door complement to the [[../libraries/roadmap-actions]] `queueRoadmapBuild` ENQUEUE gate (which the planner/Slack/BuildButton route through but a directly-inserted/raced build can bypass). Box-impacting (`scripts/builder-worker.ts`) — takes effect on the next foreman/worker self-update restart. See [[../specs/claim-time-build-gate]] · [[../specs/spec-review-agent]] · [[../specs/spec-blockers]].

**Shipped (2026-06-27, Vercel previews for spec-builds ONLY — `vercel-skip-non-spec-build-refs`):** the per-build preview pipeline ([[../goals/preview-test-promote-pipeline]] M1) whitelisted ALL `claude/*` refs in the Vercel Ignored-Build-Step (`CLAUDE_PREVIEW_IGNORE_COMMAND` in [[../libraries/vercel-project]]), so **every** foreman lane built a preview — including FOLDS (`claude/fold-*`). A fold's worktree is `origin/main` + brain-markdown-only edits with no meaningful app change, so its Vercel preview build **FAILS**; the failing required-ish `Vercel` check left fold PRs `mergeable_state` `UNSTABLE`, and the **Gate A auto-merge** (which correctly only acts on `clean`) refused to merge them — fold PRs #816/#817 stuck. Fix: spec-builds (the ONLY lane that needs a preview, for pre-merge spec-testing) now carry a distinct **`claude/build-`** prefix (`runBuildJob` in `scripts/builder-worker.ts`: `claude/build-${slug}-${hash}`), and the override whitelist matches **`^claude/build-`** only. Every other `claude/*` lane (fold / goal-fold / plan / spec-chat / dev-ask / director-coach) now SKIPS its Vercel build, so those PRs go back to `clean` and the existing Gate-A auto-merge picks them up — **no fold-gating change was needed** (the foreman never gated folds on Vercel; the `clean` requirement of auto-merge was tripped by the spurious failing check). The `claim-time hasActiveBuildForSlug` PR-detection now matches the new `claude/build-${slug}-` prefix (+ the legacy `claude/${slug}-` form for in-flight pre-rename builds). The Vercel override's live value is bound to `CLAUDE_PREVIEW_IGNORE_COMMAND` by the box auto-heal (re-PATCHed every tick), so a bare API PATCH is transient — the durable flip lands when the box self-updates to this code. The `_check-vercel-ignore-step-rails.ts` rail now asserts the `^claude/build-` contract. **Box-impacting** (`scripts/builder-worker.ts` branch naming + the auto-heal command) — takes effect on the next foreman/worker self-update restart. See [[../integrations/vercel]] · [[../libraries/vercel-project]] · [[../recipes/build-box-setup]].

**Shipped (2026-06-28, promote-on-green-merge-gate Phase 1):** the auto-merge gate (`autoMergeReadyPrs` in [[../libraries/github-pr-resolve]]) now requires BOTH pre-merge signals green for the branch (spec-test + security) on top of the existing mergeable+clean+build-gate; a built-but-not-both-green `claude/*` PR is left `in_testing` (matches `applyInTestingOverlay` on the board). Spec-test green = agent-verdict `approved`/`needs_human` + `auto_pass>=1` + 0 unresolved auto-`fail` regressions (mirrors the post-ship fold gate's Rail 2 via [[../libraries/spec-test-runs]] `isSpecTestGreenForBranch`); security green = `completedClean` signal (a `completed` security-review job, no live/surfaced sibling) via [[../libraries/security-agent]] `isSecurityGreenForBranch`. **Absence ≠ clean** — a branch with no pre-merge run defers to the next pass (the signals settle as the M2 pre-merge runs complete). All guardrails preserved: kill-switch (`auto_merge_enabled`), sync-aware, `claude/*`-only, serialize one-merge-per-pass, build-success gate, post-merge job-advance (`handleAutoMergedBuildBranch`), and Reva post-merge (@@ `openDeployWatch`; see [[../libraries/deploy-guardian]]) — defense-in-depth: pre-merge tests catch most, Reva catches regressions real traffic exposes post-prod. Fails CLOSED on tests-gate read errors (never auto-merges past an unevaluable gate). Test-block counts surfaced in Control Tower heartbeat + webhook logs (`.testsGateBlocked` + per-PR `.testsGateOk`).

**Shipped (2026-06-28, plan-job + foreman gates on spec row existence — [[../specs/planner-gates-build-queue-on-authored-specs]]):** the plan job now re-queries `public.specs` after authoring (via `getSpec` from [[../libraries/specs-table]]) and **only queues builds for specs that actually landed**, blocking phantom builds when an author transaction silently fails. Missing specs are listed in the job error and marked in the `acted` array so the inbox reflects reality. The foreman dispatch-time gate adds a second barrier — every call site that invokes `materializeSpec` now re-checks `getSpec()` before the build, routing a miss to `needs_attention` with `needs_attention_class='spec_row_missing'` (the spec row is missing; author lane failed silently upstream). This pair of gates closes the escape where an authored spec never reached the DB but a build queued anyway. See [[../libraries/specs-table]] § Callers / Build queue gates.

**Shipped (2026-07-11, planner authoring survives large multi-spec results):** `runPlanJob` no longer loses a completed planner result to a single-turn ingestion drop. Job `d5999907` fully authored 6 specs into a 62KB envelope but the primary `extractJson` parse returned **0** specs, and the whole job failed with "planner did not return a spec body for: …" — the spec bodies were on-disk in the session's transcript jsonl but were never merged back. Two layers close it. **Phase 1 (transcript fallback):** after `runClaude` returns, when `extractJson(resultText).specs.length < approved.length` the worker calls `recoverSpecsForSession(session)` (`scripts/planner-transcript-recover.ts`) — the recovery walks the session's on-disk transcript newest → oldest, returns the last assistant text that parses as `{status,specs:[...]}`, and the merge fills missing approved slugs. The primary parse wins on collision; the fallback only fills gaps. The failure message now distinguishes **genuinely missing** (planner never authored the slug — not in `resultText` AND not in the transcript) from **ingestion-dropped** (transcript had it but re-parse still failed). Byte size + primary-parse count are logged so any future truncation is loud. **Phase 2 (bounded per-result size — chunked authoring):** rather than one giant planner turn for all N approved specs, the worker now iterates in batches of at most `PLANNER_AUTHOR_BATCH_SIZE=2` (`scripts/planner-chunk-authoring.ts`), one `runClaude` per batch. The session chains across batches (batch N+1 resumes batch N's session so brain-page reads stay warm), and each spec is committed to `public.specs` the moment `authorSpecRowStructured` returns — so a mid-run batch failure leaves chunks 1..k landed in review, never rolled back to zero. Per-batch `resultText.length` is logged and a bounded-size summary line reports the max across batches. All prior gates carry through per batch (per-phase verification, per-spec intent, milestone resolution, `notReturned`/`transcriptRecovered` distinction, planner-gates-build-queue re-check via `getSpec`) — Phase 2 only bounds the request; failure modes and gates are unchanged. Box-impacting (`scripts/builder-worker.ts`) — takes effect on the next foreman/worker self-update restart. See [[../libraries/author-spec]].
