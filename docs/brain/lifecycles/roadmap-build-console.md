# Lifecycle: Roadmap Build Console (self-driving roadmap)

Describe a feature â†’ spec â†’ autonomous build on the **Max subscription** â†’ answer questions / approve prod actions â†’ merge â€” all from the dashboard (phone-friendly), no laptop or terminal. This is the end-to-end home for the shipped [[../specs/roadmap-build-console]] + [[../specs/build-approval-gates]] specs.

## Two layers (don't conflate)

- **Brain markdown (`main`)** â€” canonical, *static* spec definitions + final status (the `âŹł planned Â· đźš§ in progress Â· âś… shipped Â· âťŚ cut` phase emojis). Changes only when a PR merges. The board parses it.
- **`agent_jobs` (DB companion)** â€” *live, actionable* state: build status, `needs_input` questions, `needs_approval` pending actions. The box worker polls it; the dashboard renders it as buttons. Pending actions live here (not on the unmerged branch) â€” that's why a DB companion is required.

## Two runtimes

- **Vercel app** â€” the dashboard surfaces + APIs. Never touches the box. A "Build" tap just **inserts an `agent_jobs` row** in Supabase.
- **The box** ([[../recipes/build-box-setup]]) â€” a Hetzner CCX33, **Tailscale-only inbound**, runs `systemd: shopcx-builder` as the **non-root `builder`** user. It **polls Supabase outbound** (no inbound; the firewall stays shut â€” the box dials out, the app never dials in), claims a job, and runs the build as a headless `claude -p` on **Max**.

## Safety model

Bypass is safe because of four things, not prompts: (1) **PR-gate** â€” code never hits prod until you merge; (2) **powerless build env** â€” secrets stripped, so a runaway command can't reach prod (credential-enforced); (3) **gated prod actions need a tap** â€” `apply_migration`/`run_prod_script`/`merge_pr` only run after owner approval, executed by the worker; (4) **non-root** builder user. The worker is the *only* component with prod creds.
