4. Operation contract

The two op shapes every entry in §2 and §3 satisfies, plus the Gateway discriminator money-moving ops carry and the InternalOrAppstle<T> helper that binds each op to declaring both branches at the type level. Declared in src/lib/commerce/types.ts; zero implementations ship in this phase — Display impls arrive in M2b, Mutation impls in M2c.

DisplayOp<TInput, TView>

A read-only op: workspace + typed input → one canonical view (see §5).

type DisplayOp<TInput, TView> = (
  workspaceId: string,
  input: TInput,
) => Promise<TView>;

Example: const getSubscription: DisplayOp<string, SubscriptionView> — subscription id in, priced SubscriptionView out.

MutationOp<TInput, TResult>

A write op: workspace + typed input → { success, …result }. The callable signature is a function; the gateway metadata field is a property set on the same value when (and only when) money moves.

type MutationResult<TResult> = { success: boolean } & TResult;

interface MutationOp<TInput, TResult> {
  (workspaceId: string, input: TInput): Promise<MutationResult<TResult>>;
  /** Payment boundary. Undefined iff the op does not move money. */
  readonly gateway?: Gateway;
}

Example: const issueRefund: MutationOp<{ orderId: string; cents: number }, { refundId: string }>, with issueRefund.gateway = "braintree" for internal orders.

Gateway

The payment boundary a money-moving Mutation op routes through. Non-money mutations (a pause, a resume, a title enrichment) omit it.

type Gateway = "braintree" | "shopify";

Braintree owns internal-sub charges + refunds; Shopify owns Appstle-baked charges + refunds — the same routing the current renewal + refund paths use, now declared once at the type level so a caller cannot pick the wrong boundary.

InternalOrAppstle<T>

Every commerce op has TWO implementations — internal (engine + Braintree) and Appstle-baked (Appstle + Shopify). The SDK dispatcher takes an InternalOrAppstle<Op> and picks the branch at call time from sub.is_internal. Declaring only one branch is a type error — that is the point.

interface InternalOrAppstle<T> {
  internal: T;
  appstle:  T;
}

Example M2b/M2c wiring:

const getSubscriptionImpls: InternalOrAppstle<DisplayOp<string, SubscriptionView>> = {
  internal: async (ws, id) => { /* engine-priced */ },
  appstle:  async (ws, id) => { /* baked-priced  */ },
};
Why this shape. A caller who imports @/lib/commerce/subscription#getSubscription gets a DisplayOp<string, SubscriptionView> — one contract, both branches guaranteed to exist. The dispatcher (M2b) sits between the import and the two InternalOrAppstle branches, so the two-branch discipline is compiled in, not remembered.

7. Defect register (9)

The defects the SDK exists to close. #1 and #2 link to their M1 specs (defect-elimination milestone). #3–#9 are grounded in current codebase evidence — where fixed, links point at the merged fix; where open, at the code the Mutation op must guard.

#DefectWhereStatusHow the SDK closes it
1 Phantom refund — Braintree gateway reported a refund success before the settlement actually moved. 4 phantom "refunds" on SC128233; customer never paid back. src/lib/shopify-order-actions.ts (SC128233) M1 Mutation op issueRefund gates the write on settlement confirmation before writing note+tag; a caller cannot silently claim success. M1 spec: phantom-refund-braintree-settlement-gate
2 Coupon mis-fire — the wrong coupon (or a coupon at all) was applied to a sub because the mapping was hardcoded or the wrong shape was accepted. src/lib/coupons.ts + orchestrator apply M1 Mutation op applyCoupon gates on the DB-driven coupon-mapping table, normalizes both shapes, and rejects unknown codes. M1 spec: coupon-misfire-mapping-guard
3 Double-discount trap — when a variant wasn't in the catalog, base fell back to the item's already-post-discount baked price, then S&S applied again → double-discounted. src/lib/pricing.ts (fixed 2026-06) Fixed Dual-shape variant resolver (UUID + shopify_variant_id) in pricing.ts; SDK Mutation ops write UUID-only. See libraries/pricing § Gotchas.
4 Stale last_payment_status locks portal — a leaked last_payment_status='failed' on an internal sub permanently locks the customer out of every self-serve mutation (ticket efe0d2ad). portal handlers gate on last_payment_status Fixed Internal renewal + closeInternalDunningOnSuccess defence-in-depth writers; backfill at scripts/backfill-internal-sub-last-payment-status.ts. SDK Display op refuses to return the stale value once fixed. See tables/subscriptions § Gotchas.
5 Missing catalog line silently prices to $0 — Appstle-baked path used Number(price_cents || 0); an undefined price silently baked $0 into every display surface. src/lib/commerce/price.ts (fixed 2026-07, Phase 2) Fixed PriceInvariantError throws with sub id + line id when a non-gift, non-protection line yields undefined/NaN cents. Repro: npx tsx scripts/_probe-price-invariant.ts.
6 Tax quote staleness under dynamic pricing — cached Avalara quote used updated_at as freshness key; a catalog price / rule edit re-prices without touching the row, so updated_at didn't move and a stale quote leaked. ensureFreshSubscriptionTaxQuote (fixed 2026-06) Fixed avalara_quote_hash over engine-priced inputs (hashTaxInputs); SDK Display op reads via the freshness gate. Billing does its own commit=true call, so display staleness cannot mischarge. See libraries/pricing § Display layer.
7 NULL storefront attribution UTMs — native storefront orders (source_name=storefront) had NULL attributed_utm_*, so a Meta sale showed (direct). /api/checkout § 10b (fixed 2026-06-14) Fixed First-touch backfill from earliest storefront_sessions with a UTM; also stamps session_id, anonymous_id, advertorial_page_id, ad_campaign_id. SDK getOrder surfaces the resolved attribution directly. See tables/orders § Gotchas.
8 Comp sub renewal without allowlist — a comp sub whose customer lacked a valid comp_role would ship free if the renewal branch didn't gate first (would be free-inventory leak). internal-subscription-renewals (comp branch) Fixed Gate first — invalid comp_role → fail-closed with type='comp' failed transaction + subscription.comp_renewal_failed event, no shipment, no advance. SDK Mutation op preserves the gate. See tables/subscriptions § Gotchas.
9 Return refund on carrier first-scan — returns refunded when a carrier first scanned the package, not when EasyPost marked it delivered. Prone to lost-in-transit refund-then-nothing-arrives loss. returns pipeline (EasyPost webhook) Fixed Refund fires on EasyPost delivered. net_refund_cents is the contract, set at return creation — SDK issueRefund reads it, never re-derives. See tables/returns § Gotchas.

8. Build plan

  1. M2 — SDK core (this spec, commerce-sdk-scaffold-money-resolver)
    • Phase 1 — Scaffold src/lib/commerce + declare view shapes Shipped
    • Phase 2 — Relocate priceSubscription + lock money invariants Shipped
    • Phase 3 — Author this inventory HTML Shipped
    • Phase 4 — Publish the operation contract (DisplayOp / MutationOp / Gateway / InternalOrAppstle) Shipped
  2. M2b — Display implementations — implement getSubscription, listSubscriptions, getOrder, listOrders, getReturn, listReturns, getReplacement, getCustomer, getLoyalty, getChargeback, getFraudCase, getCrisis. Each op internal + Appstle branch behind one signature.
  3. M2c — Mutation implementations — implement every op in §3, wired to the correct gateway; every money move flows through priceSubscription.
  4. M2d — Surface migration — one PR per surface (§1), swap the import from @/lib/portal/helpers/enrich-pricing or @/lib/appstle / @/lib/subscription-items to @/lib/commerce. Deprecated shims stay until every surface is off them.
  5. M2e — Delete shims — remove the deprecated re-exports; delete the standalone appstle.ts / subscription-items.ts; regenerate brain library pages.
  6. Post-M2 — fold this HTML page into the relevant lifecycle pages ([[../lifecycles/customer-portal]], [[../lifecycles/subscription-billing]]); the inventory then survives as historical reference in the brain archive.
