Origin

Two one-off bugs turned out to be one systemic gap

A ticket action set a customer's next order via a raw Shopify GraphQL mutation instead of the internal-aware dispatcher; a dashboard rendered a correct date a day early. Both are instances of the same structural problem: customer data is read and written through three independent stacks that share almost nothing on the read side, and each re-implements — or skips — the internal-vs-Appstle branch. Internal subscriptions (billed by our own engine) and Appstle subscriptions carry different price and identity shapes; every surface that forgets the difference produces a wrong number or a dead action.

Structural finding

Three parallel data stacks, unified only halfway

Mutations are ~70% converged — the AI, Improve, and Triage paths all bottom out on the same directActionHandlers, and portal + AI both dispatch through the appstle.ts / subscription-items.ts wrappers. Reads are not unified at all, and the pricing-enrichment that prevents $NaN is applied inconsistently across the three. That inconsistency is the bug class.

Internal subs carry no baked items[].price_cents (engine-derived), and the dashboard routes never call the live pricing engine. Battle-testing

Performance tenets — baked in, not bolted on
SQL / RPC for anything list-or-aggregate. The 3h→8s precedent is the rule: list and rollup operations are set-based Postgres functions, not JS loops over paginated fetches. First conversions: Funnel (raw event/session tables) ≫ MRR (nested day-bucketing) > Portal-analytics (unbounded on “all time”) > Dunning.
Never silently truncate. Every listX paginates past Supabase's 1000-row cap via range or RPC. (Live example this pass: loyalty stats summed a 250-row sample and labeled it program-wide.)
Column-selective + batched enrichment. The customer-360 fans out to ~13 parallel fetches today; detailCustomer collapses that into one batched call to cut egress.
One money resolver. priceSubscription is the only path to a line/total price; internal subs can never emit undefined cents, so no surface can render $NaN.

Sequencing

Build plan — customer surfaces migrate last

The SDK is built and proven in isolation before a single customer-facing line changes. Each surface migrates behind an unchanged UI, verified by diffing new-SDK output against current output.

0

Contract lock

This document — the operation set, view shapes, defect register, and naming map become the SDK's acceptance spec.

1

SDK core — src/lib/commerce/, zero consumers

Display (listX/detailX + view shapes + priceSubscription everywhere) and Mutation (entity-named, internal-aware, gateway-aware refund). Nothing imports it yet → production unchanged.

2

Battle-test in isolation

Differential harness diffs SDK output vs current portal + dashboard output across internal / Appstle / grandfathered samples; NaN assertions; egress + latency + row-count benchmarks; SQL RPCs for the four aggregation candidates.

3

Migrate — dashboard lists → detail + ticket → AI

Lowest blast-radius first. Agent-facing surfaces (you see them before any customer). Fix the missing-action gaps here.

4

Customer portal — last

Migrates only after the SDK has run in production on every internal surface and is diff-verified against the portal's own current output.
